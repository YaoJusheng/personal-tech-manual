    一、简介¶

    OAuth 2.0 （RFC 6749）是一个关于授权的框架协议，而不是一个严格的身份认证协议。它的核心思想是：让第三方应用（客户端）在资源所有者（用户）的许可下，有限制地访问其在资源服务器上的受保护资源，而无需直接获取用户的密码。

    1.核心理解¶

    在设计和对接接口前，必须清晰理解 OAuth 2.0 中的四个核心角色：

    1. 资源所有者：通常指最终用户，拥有数据的所有权。
    2. 客户端：第三方应用，需要访问用户数据。核心原则：客户端永远不应该接触用户的密码。
    3. 授权服务器：负责认证用户并颁发访问令牌（Access Token）的核心服务。
    4. 资源服务器：存放用户数据的服务端（API），验证令牌的有效性并返回数据。

    核心抽象：令牌¶

    OAuth 2.0 的精髓在于用令牌代替密码。

    • Access Token：用于访问 API，通常设置短期有效（如 1 小时）。
    • Refresh Token：用于获取新的 Access Token，通常设置长期有效（如 30 天），且仅在与授权服务器通信时使用。

    四种授权模式¶

    接口设计取决于你选择哪种模式。最常用的是前两种：

    1. 授权码模式（Authorization Code Grant）：最安全、最常用的模式（适用于有后端服务器的 Web 应用）。
    2. 流程：客户端引导用户到授权服务器，用户登录并授权后，授权服务器返回授权码（code），客户端用授权码向令牌端点换取访问令牌。
    3. 优势：令牌不经过浏览器，降低了泄露风险。
    4. 特点：最安全，适用于有后端服务的机密客户端，访问令牌不经过浏览器。
    5. 现状：推荐使用的模式。
    6. 客户端凭证模式（Client Credentials Grant）：适用于机器对机器（M2M）或后台任务（没有用户参与的场景）。
    7. 流程：客户端直接用 Client ID 和 Secret 向令牌端点请求访问令牌，无用户参与。
    8. 特点：适用于机器间通信、后端服务调用或定时任务。
    9. 现状：广泛使用，适用于没有资源所有者的场景。
    10. 隐式模式（Implicit Grant）
    11. 流程：客户端直接通过浏览器重定向从授权端点获取访问令牌，无授权码环节。
    12. 特点：省略了授权码交换步骤，但令牌暴露在浏览器 URL 中，安全性较低。
    13. 现状：已不推荐使用（OAuth 2.0 Security Best Current Practice 建议废弃），应由授权码模式 + PKCE 替代。
    14. 密码模式（Resource Owner Password Credentials Grant）
    15. 流程：客户端直接收集用户的用户名/密码，并将其发送给授权服务器换取访问令牌。
    16. 特点：简单但客户端会接触到用户凭据，违背了 OAuth 的核心原则。
    17. 现状：已不推荐使用，仅适用于极度信任的遗留系统或第一方应用。

    二、接口契约¶

    1.接口契约与设计¶

    在设计 OAuth 2.0 接口时，核心在于定义授权服务器提供的三个核心端点，以及资源服务器对令牌的校验机制。

    1. 授权端点 (Authorization Endpoint)¶

    作用：用户登录并授权的地方。这是一个前端重定向接口。

    请求设计（GET）¶
    GET /authorize?
  response_type=code
  &client_id=YOUR_CLIENT_ID
  &redirect_uri=https://yourapp.com/callback
  &scope=read write
  &state=xyzABC123
    参数 契约说明
    response_type 固定为 code。如果是简化模式则为 token，但现已不推荐。
    client_id 公开标识符，非密钥。
    redirect_uri 必须严格校验。必须与客户端注册时预埋的地址完全匹配（或域名匹配），防止开放重定向攻击。
    scope 权限边界。遵循最小权限原则，不要请求超过需要的权限。
    state 强烈建议。用于防止 CSRF（跨站请求伪造）攻击，回调时会原样返回，用于校验请求一致性。
    响应（成功回调）¶
    HTTP 302 Found
Location: https://yourapp.com/callback?code=AUTH_CODE&state=xyzABC123
    响应（错误回调）¶
    Location: https://yourapp.com/callback?error=access_denied&state=xyzABC123

    常见错误：

    • invalid_request：缺少参数
    • unauthorized_client：客户端无权使用此方式
    • access_denied：用户拒绝了授权
    • invalid_scope：请求的 scope 无效

    2. 令牌端点 (Token Endpoint)¶

    作用：用授权码换取令牌，或用刷新令牌换取新令牌。这是一个后端接口（Postman 调用或服务器间调用）。

    请求设计（POST）：

    POST /token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=AUTH_CODE_HERE
&redirect_uri=https://yourapp.com/callback
&client_id=YOUR_CLIENT_ID
&client_secret=YOUR_CLIENT_SECRET

    响应设计：

    {
  "access_token": "2YotnFZFEjr1zCsicMWpAA",
  "token_type": "Bearer",
  "expires_in": 3600,
  "refresh_token": "tGzv3JOkF0XG5Qx2TlKWIA",
  "scope": "read write"
}
    字段 契约说明
    token_type 通常为 Bearer。客户端在调用 API 时必须将此值放入 Authorization 头。
    expires_in 有效期（秒）。客户端必须在此时间前主动刷新，而不是等到接口报错。
    refresh_token 仅在首次授权或刷新时下发。注意：刷新令牌的端点通常使用 grant_type=refresh_token，此时会返回新的 access_token 和新的 refresh_token（取决于安全策略）。

    3. 刷新令牌流程¶

    当访问令牌过期，客户端使用刷新令牌请求新的令牌。

    POST /token
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token
&refresh_token=REFRESH_TOKEN_HERE
&client_id=YOUR_CLIENT_ID
&client_secret=YOUR_CLIENT_SECRET

    响应结构与获取令牌相同（返回新的 access_token 和可选的 refresh_token）。

    4. 资源接口¶

    作用：业务 API 接口。这是资源服务器的职责。

    请求设计：

    GET /api/user/info
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...

    响应设计：

    • 成功 (200)：返回业务数据。

    • 令牌无效 (401 Unauthorized)：

    {
  "error": "invalid_token",
  "error_description": "Token expired or invalid signature"
}

    • 注意：不要返回 302 重定向到登录页，API 是无状态的，应返回明确的 401 状态码。

    • 权限不足 (403 Forbidden)：令牌有效但 scope 不足。

    2.关键设计细节¶

    在实际设计接口和契约时，以下几个点容易被忽视但又至关重要：

    1. 状态码与错误处理¶

    授权服务器在回调时，错误信息通过 URL 参数传递，而不是 HTTP 状态码。

    错误回调示例：

    https://yourapp.com/callback?error=access_denied&state=xyz
    错误码 场景
    invalid_request 缺少必填参数
    unauthorized_client 客户端无权使用此方式
    access_denied 用户拒绝了授权
    invalid_grant 授权码无效、过期或已被使用（授权码是一次性的）

    2. Redirect URI 白名单¶

    • 授权服务器必须维护每个客户端的 redirect_uri 白名单。
    • 匹配规则：严格匹配完整 URL（包括路径），或使用域名匹配时不能过于宽松。
    • 严禁允许 redirect_uri 包含通配符或任意子域名（除非明确注册）。

    3. state 参数¶

    • 必须要求客户端传递 state，并在回调时校验其值与发起时一致。
    • 防止 CSRF 和回调劫持。

    4. client_secret 保护¶

    • client_secret 仅能在后端使用，绝不能在浏览器端或移动端代码中硬编码。
    • 对于无法保证 secret 机密性的公开客户端（如纯前端应用），标准 OAuth 2.0 没有强制要求使用 PKCE，但实践中应避免使用授权码模式（或配合 PKCE 使用）。

    5. 令牌存储与传输¶

    • 所有涉及令牌传输的端点必须使用 HTTPS。
    • 访问令牌不应存储在浏览器本地存储（localStorage）中（易受 XSS 攻击），应优先存放在后端或 HttpOnly Cookie 中（需配合后端代理）。

    6. 令牌格式¶

    OAuth 2.0 规范本身不规定令牌格式，可以是：

    • 随机字符串（Opaque）：资源服务器需通过授权服务器的内省端点（Introspection）校验令牌。
    • 结构化令牌（如 JWT）：资源服务器可本地校验签名，无需调用授权服务器，但需注意撤销问题。

    选择哪种格式取决于系统架构，但标准 OAuth 2.0 并未强制使用 JWT。

    7. 权限范围（Scope）¶

    • 应设计细粒度的 scope，如 user:read、user:write、files:delete。
    • 授权服务器需在用户授权时明确展示 scope 列表，用户同意后颁发的令牌应包含 scope 信息。
    • 资源服务器应校验令牌中的 scope 是否满足接口所需。

    3.模式对比：授权码模式 vs 客户端凭证模式¶

    特性 授权码模式 客户端凭证模式
    适用场景 有用户参与的第三方应用 后端服务间调用、定时任务、没有用户场景
    是否需要用户登录 是 否
    是否涉及 redirect_uri 是 否
    使用的 grant_type authorization_code client_credentials
    是否返回 refresh_token 通常返回 通常不返回
    典型客户端类型 机密客户端（有后端） 机密客户端

    客户端凭证模式示例：

    POST /token
Content-Type: application/x-www-form-urlencoded
Authorization: Basic base64(client_id:client_secret)

grant_type=client_credentials
&scope=read

    响应仅包含 access_token，无 refresh_token。
