    1. HTTP 协议基础¶

    1.1 HTTP 简介¶

    HTTP (HyperText Transfer Protocol) 是一个应用层协议，用于在Web浏览器和Web服务器之间传输数据。它是无状态的、基于请求-响应模式的协议。

    1.2 HTTP 版本演进¶

    • HTTP/0.9 (1991): 最初版本，只支持GET方法
    • HTTP/1.0 (1996): 添加了POST、HEAD方法和状态码
    • HTTP/1.1 (1997): 持久连接、管道化、分块传输
    • HTTP/2 (2015): 多路复用、服务器推送、头部压缩
    • HTTP/3 (2022): 基于QUIC协议，更快的连接建立

    1.3 HTTP 请求结构¶

    GET /api/users HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0
Accept: application/json
Authorization: Bearer token123

[请求体]

    1.4 HTTP 响应结构¶

    HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 123
Cache-Control: max-age=3600

{"users": [...]}

    2. HTTP 方法详解¶

    2.1 GET 方法¶

    // 获取用户列表
fetch('/api/users', {
    method: 'GET',
    headers: {
        'Accept': 'application/json'
    }
})
.then(response => response.json())
.then(data => console.log(data));

    2.2 POST 方法¶

    // 创建新用户
fetch('/api/users', {
    method: 'POST',
    headers: {
        'Content-Type': 'application/json'
    },
    body: JSON.stringify({
        name: 'John Doe',
        email: 'john@example.com'
    })
});

    2.3 PUT 方法¶

    // 更新用户信息
fetch('/api/users/123', {
    method: 'PUT',
    headers: {
        'Content-Type': 'application/json'
    },
    body: JSON.stringify({
        name: 'Jane Doe',
        email: 'jane@example.com'
    })
});

    2.4 DELETE 方法¶

    // 删除用户
fetch('/api/users/123', {
    method: 'DELETE'
});

    2.5 PATCH 方法¶

    // 部分更新用户
fetch('/api/users/123', {
    method: 'PATCH',
    headers: {
        'Content-Type': 'application/json'
    },
    body: JSON.stringify({
        email: 'newemail@example.com'
    })
});

    3. HTTP 状态码¶

    3.1 1xx 信息性状态码¶

    • 100 Continue: 客户端应继续发送请求
    • 101 Switching Protocols: 协议切换

    3.2 2xx 成功状态码¶

    • 200 OK: 请求成功
    • 201 Created: 资源创建成功
    • 202 Accepted: 请求已接受，正在处理
    • 204 No Content: 请求成功，无返回内容

    3.3 3xx 重定向状态码¶

    • 301 Moved Permanently: 永久重定向
    • 302 Found: 临时重定向
    • 304 Not Modified: 资源未修改
    • 307 Temporary Redirect: 临时重定向（保持方法）

    3.4 4xx 客户端错误¶

    • 400 Bad Request: 请求语法错误
    • 401 Unauthorized: 未授权
    • 403 Forbidden: 禁止访问
    • 404 Not Found: 资源不存在
    • 429 Too Many Requests: 请求过多

    3.5 5xx 服务器错误¶

    • 500 Internal Server Error: 服务器内部错误
    • 502 Bad Gateway: 网关错误
    • 503 Service Unavailable: 服务不可用
    • 504 Gateway Timeout: 网关超时

    4. HTTP 头部详解¶

    4.1 请求头部¶

    const headers = {
    'Accept': 'application/json, text/plain, */*',
    'Accept-Language': 'zh-CN,zh;q=0.9,en;q=0.8',
    'Accept-Encoding': 'gzip, deflate, br',
    'User-Agent': 'Mozilla/5.0 (Windows NT 10.0; Win64; x64)',
    'Authorization': 'Bearer eyJhbGciOiJIUzI1NiIs...',
    'Content-Type': 'application/json',
    'Cache-Control': 'no-cache',
    'X-Requested-With': 'XMLHttpRequest'
};

    4.2 响应头部¶

    // 服务器响应头部示例
const responseHeaders = {
    'Content-Type': 'application/json; charset=utf-8',
    'Content-Length': '1234',
    'Cache-Control': 'public, max-age=3600',
    'ETag': '"33a64df551425fcc55e4d42a148795d9f25f89d4"',
    'Last-Modified': 'Wed, 21 Oct 2023 07:28:00 GMT',
    'Access-Control-Allow-Origin': '*',
    'X-Frame-Options': 'DENY',
    'X-Content-Type-Options': 'nosniff'
};

    5. HTTPS 安全协议¶

    5.1 HTTPS 简介¶

    HTTPS (HTTP Secure) 是HTTP的安全版本，通过TLS/SSL加密传输数据，提供：

    • 数据加密: 防止数据被窃听
    • 身份认证: 验证服务器身份
    • 数据完整性: 防止数据被篡改

    5.2 TLS/SSL 握手过程¶

    1. 客户端 -> 服务器: Client Hello
2. 服务器 -> 客户端: Server Hello + 证书
3. 客户端验证证书
4. 客户端 -> 服务器: 预主密钥
5. 双方生成会话密钥
6. 开始加密通信

    5.3 SSL 证书类型¶

    • DV (Domain Validated): 域名验证证书
    • OV (Organization Validated): 组织验证证书
    • EV (Extended Validation): 扩展验证证书

    5.4 配置 HTTPS¶

    Nginx 配置示例¶

    server {
    listen 443 ssl http2;
    server_name example.com;

    ssl_certificate /path/to/certificate.crt;
    ssl_certificate_key /path/to/private.key;

    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-RSA-AES128-GCM-SHA256:ECDHE-RSA-AES256-GCM-SHA384;
    ssl_prefer_server_ciphers off;

    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 10m;

    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

    location / {
        proxy_pass http://backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

    Apache 配置示例¶

    <VirtualHost *:443>
    ServerName example.com
    DocumentRoot /var/www/html

    SSLEngine on
    SSLCertificateFile /path/to/certificate.crt
    SSLCertificateKeyFile /path/to/private.key
    SSLCertificateChainFile /path/to/chain.crt

    SSLProtocol all -SSLv3 -TLSv1 -TLSv1.1
    SSLCipherSuite ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256
    SSLHonorCipherOrder off

    Header always set Strict-Transport-Security "max-age=31536000; includeSubDomains"
</VirtualHost>

    6. HTTP/2 特性¶

    6.1 多路复用¶

    HTTP/2 允许在单个连接上并行发送多个请求，解决了HTTP/1.1的队头阻塞问题。

    // HTTP/2 自动启用多路复用
Promise.all([
    fetch('/api/users'),
    fetch('/api/posts'),
    fetch('/api/comments')
]).then(responses => {
    // 所有请求并行处理
});

    6.2 服务器推送¶

    // 服务器可以主动推送资源
// 在响应HTML时推送CSS和JS文件

    6.3 头部压缩 (HPACK)¶

    HTTP/2 使用HPACK算法压缩头部，减少传输开销。

    6.4 二进制分帧¶

    HTTP/2 使用二进制格式传输数据，提高解析效率。

    7. HTTP/3 新特性¶

    7.1 基于 QUIC 协议¶

    HTTP/3 基于QUIC协议，提供：

    • 更快的连接建立: 0-RTT连接
    • 更好的移动网络支持: 连接迁移
    • 内置加密: 默认加密传输

    QUIC（Quick UDP Internet Connections）是 Google 主导、IETF 标准化的下一代传输协议，基于 UDP 实现。

    一句话总结:

    QUIC = UDP 的灵活性 + TCP 的可靠性 + TLS 1.3 的安全性 + 应用层的可迭代性。 它不是为了"更快"，而是为了在复杂网络环境下（移动切换、弱网、中间盒干扰）提供更稳定的用户体验，并让协议演进不再受操作系统内核束缚。

    1.为什么需要 QUIC？TCP 的痛点¶

    TCP/HTTP/2 问题 根本原因 QUIC 解决方案
    连接建立慢 TCP 三次握手 + TLS 握手 = 2-3 RTT 0-RTT 或 1-RTT 建立连接
    队头阻塞（Head-of-Line Blocking） TCP 流共享同一连接，丢包阻塞所有流 基于 UDP，每个流独立传输
    协议僵化 TCP/TLS 实现在操作系统内核，升级困难 用户态实现，可快速迭代
    连接迁移困难 四元组（IP+Port）绑定，NAT/切换 WiFi 断连 连接 ID 与 IP/Port 解耦
    安全性滞后 TLS 1.2 握手复杂，TLS 1.3 部署慢 内置 TLS 1.3，强制加密

    2.QUIC 核心架构¶

    ┌───────────────────────────────────────┐
│           HTTP/3 (应用层)              │
│    语义层：请求/响应、头部压缩(QPACK)     │
├───────────────────────────────────────┤
│           QUIC (传输层)                │
│  ┌───────────┐ ┌─────────┐ ┌────────┐ │
│  │ 流多路复用  │ │ 拥塞控制 │ │ 错误恢复│ │
│  │ (Streams) │ │ (CC)    │ │ (FEC)  │ │
│  └───────────┘ └─────────┘ └────────┘ │
│  ┌─────────────────────────────────┐  │
│  │        TLS 1.3 (内置加密)        │  │
│  │   握手与数据加密一体化，0-RTT       │  │
│  └─────────────────────────────────┘  │
├───────────────────────────────────────┤
│           UDP (网络层)                 │
│    无连接、无可靠性、无顺序保证            │
└───────────────────────────────────────┘

    关键洞察： QUIC 不是"更快的 TCP"，而是在 UDP 上重新实现了 TCP 的可靠性 + TLS 的安全性 + HTTP/2 的多路复用，且三者深度融合。

    3.核心机制详解¶

    3.1. 连接建立：0-RTT vs 1-RTT¶
    TCP + TLS 1.2:        QUIC (首次):
Client                Client
  | ---- SYN -------->  |  | ---- ClientHello + 数据 ---->
  | <--- SYN+ACK -----  |  | <--- ServerHello + 数据 ----
  | ---- ACK -------->  |  | (已完成，开始传输)
  | ---- ClientHello -> |
  | <--- ServerHello -- |
  | <--- Finished ----- |
  | ---- Finished ----> |
  | (共 2-3 RTT)        |  (1-RTT，若用 0-RTT 令牌则 0-RTT)

QUIC (0-RTT，重连):
Client
  | ---- ClientHello + 早期数据 + 应用数据 ---->
  | <--- ServerHello + 数据 -------------------
  | (0-RTT，基于之前会话的 PSK 和令牌)

    0-RTT 的前提：

    • 客户端之前与该服务器建立过连接
    • 缓存了服务器的 NewSessionTicket（包含 PSK 和传输参数）
    • 使用相同的源地址令牌（防止重放攻击）

    安全风险： 0-RTT 数据可能被重放攻击（Replay Attack），因此 QUIC 规定 0-RTT 数据不应包含非幂等操作（如 POST 请求修改状态）。

    3.2. 流多路复用（Stream Multiplexing）¶

    QUIC 在单一连接内支持多个独立的双向流（Stream），每个流有独立的流 ID、流量控制和顺序保证。

    QUIC Connection
├── Stream 0: 控制流（连接级设置、SETTINGS）
├── Stream 1: 客户端 → 服务器 请求 A
├── Stream 2: 服务器 → 客户端 响应 A
├── Stream 3: 客户端 → 服务器 请求 B
├── Stream 4: 服务器 → 客户端 响应 B（部分数据丢失）
└── Stream 5: 客户端 → 服务器 请求 C（不受影响！）

    与 HTTP/2 的关键区别：

    特性 HTTP/2 over TCP HTTP/3 over QUIC
    流实现 应用层帧复用，共享 TCP 连接 传输层原生支持，独立流
    队头阻塞 TCP 丢包阻塞所有流 UDP 丢包只影响该流
    流 ID 空间 31 位 62 位（更大）
    流量控制 连接级 + 流级 连接级 + 流级 + 更精细

    队头阻塞消除原理：

    • TCP 层：Packet 1 丢失，Packet 2-10 即使到达也必须等待重传，阻塞所有流。
    • QUIC 层：Stream A 的 Packet 1 丢失，只阻塞 Stream A；Stream B 的 Packet 2 正常到达，立即递交给应用层。QUIC 层：Stream A 的 Packet 1 丢失，只阻塞 Stream A；Stream B 的 Packet 2 正常到达，立即递交给应用层。
    3.3. 连接迁移（Connection Migration）¶
    场景：手机从 WiFi 切换到 4G
TCP:                    QUIC:
IP1:Port1 ----->        Connection ID: 0x1234
  | 切换网络，IP 变为 IP2:Port2
  | 连接重置（RST），应用感知断开
IP2:Port2 ----->        Connection ID: 0x1234（不变）
  | 新路径验证（PATH_CHALLENGE / PATH_RESPONSE）
  | 连接继续，应用无感知！

    实现机制：

    • 每个 QUIC 包携带 Destination Connection ID（DCID），标识逻辑连接而非四元组。
    • 路径变更时，发送 PATH_CHALLENGE 帧探测新路径 MTU 和可达性。
    • 服务器回复 PATH_RESPONSE，验证通过后数据流无缝切换。
    3.4. 内置 TLS 1.3¶

    QUIC 的加密不是"在 QUIC 外面套 TLS"，而是TLS 1.3 握手逻辑直接嵌入 QUIC 帧：

    QUIC 包结构：
┌──────────────┬───────────────┬─────────────────────────────┐
│  QUIC Header │  QUIC Frame(s)                              │
│  (部分加密)   │                                             │
│              │  ┌─────────┐ ┌─────────┐ ┌───────────┐      │
│              │  │CRYPTO   │ │ STREAM  │ │ ACK       │      │
│              │  │(TLS握手) │ │(应用数据)│ │(确认)      │      │
│              │  │         │ │         │ │           │      │
│              │  └─────────┘ └─────────┘ └───────────┘      │
└──────────────┴─────────────────────────────────────────────┘

    特点：

    • 除少量 Header 外，所有内容加密，包括流 ID、包号、ACK 信息（防止中间盒窥探和篡改）。
    • 没有明文 TCP/UDP 端口号暴露应用类型（但 SNI 仍可能暴露域名，ESNI/ECH 正在解决）。
    3.5. 拥塞控制与可靠性¶

    QUIC 在用户态重新实现了 TCP 的核心机制：

    机制 QUIC 实现 对比 TCP
    ACK 机制 SACK（选择性确认）+ ACK Delay 类似，但更精确
    重传 基于包号（Packet Number）单调递增 TCP 基于序列号，重传时序列号复用，RTT 测量混乱
    RTT 测量 包号单调递增，精确区分首传与重传 TCP 序列号复用导致 RTT 估计不准
    拥塞控制 可插拔：Cubic、BBR、BBRv2 内核实现，升级困难
    前向纠错（FEC） 可选，早期版本实验，现多放弃 无原生支持

    关键创新：单调递增的包号（Packet Number）:

    • TCP：重传时复用相同序列号，接收方无法区分是首传还是重传，导致 RTT 估计偏差。
    • QUIC：每个发送的包（即使重传）都有新的包号，ACK 明确对应具体发送时刻，RTT 测量精确。

    4.HTTP/3：QUIC 之上的应用层¶

    HTTP/3 不是"HTTP/2 的简单升级"，而是基于 QUIC 的完全重写。

    特性 HTTP/2 HTTP/3
    传输层 TCP + TLS QUIC (UDP + 内置 TLS)
    流控制 应用层 HPACK 头部压缩 QPACK（避免队头阻塞）
    服务器推送 支持（复杂且少用） 移除（被 WebTransport 替代）
    连接建立 2-3 RTT 0-1 RTT
    连接迁移 不支持 原生支持
    中间盒干扰 TCP 被深度检测/限速/重置 UDP 可能被防火墙丢弃（当前最大障碍）

    QPACK 头部压缩：

    • HTTP/2 的 HPACK 依赖有序传输，TCP 队头阻塞时头部解压停滞。
    • HTTP/3 的 QPACK 使用独立的单向流传输动态表更新，与应用数据流解耦。

    9.3 认证与授权¶

    // JWT Token 认证
const jwt = require('jsonwebtoken');

function authenticateToken(req, res, next) {
    const authHeader = req.headers['authorization'];
    const token = authHeader && authHeader.split(' ')[1];

    if (!token) {
        return res.sendStatus(401);
    }

    jwt.verify(token, process.env.ACCESS_TOKEN_SECRET, (err, user) => {
        if (err) return res.sendStatus(403);
        req.user = user;
        next();
    });
}
