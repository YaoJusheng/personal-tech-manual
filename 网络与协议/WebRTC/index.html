    1.介绍¶

    WebRTC（Web 实时通信）是一个免费开源项目，通过应用程序编程接口(API) 为Web 浏览器和移动应用程序提供实时通信(RTC) 。 它允许音频和视频通信和流媒体在网页内进行，允许直接进行点对点通信，无需安装插件或下载本机应用程序。

    1.1.ICE¶

    交互式连接创建（Interactive Connectivity Establishment，ICE）是一个允许你的浏览器和对端浏览器建立连接的协议框架。在实际的网络当中，有很多原因能导致简单的从 A 端到 B 端直连不能如愿完成。这需要绕过阻止建立连接的防火墙，给你的设备分配一个唯一可见的地址（通常情况下我们的大部分设备没有一个固定的公网地址），如果路由器不允许主机直连，还得通过一台服务器转发数据。ICE 通过使用以下几种技术完成上述工作。

    1.2.STUN¶

    NAT 的会话穿越功能Session Traversal Utilities for NAT (STUN) (缩略语的最后一个字母是 NAT 的首字母) 是一个允许位于 NAT 后的客户端找出自己的公网地址，判断出路由器阻止直连的限制方法的协议。

    客户端通过给公网的 STUN 服务器发送请求获得自己的公网地址信息，以及是否能够被（穿过路由器）访问。

    1.3.NAT¶

    网络地址转换协议Network Address Translation (NAT) 用来给你的（私网）设备映射一个公网的 IP 地址的协议。一般情况下，路由器的 WAN 口有一个公网 IP，所有连接这个路由器 LAN 口的设备会分配一个私有网段的 IP 地址（例如 192.168.1.3）。私网设备的 IP 被映射成路由器的公网 IP 和唯一的端口，通过这种方式不需要为每一个私网设备分配不同的公网 IP，但是依然能被外网设备发现。

    一些路由器严格地限定了谁能连接内网的设备。这种情况下，即使 STUN 服务器识别了该内网设备的公网 IP 和端口的映射，依然无法和这个内网设备建立连接。这种情况下就需要转向 TURN 协议。

    1.4.TURN¶

    一些路由器使用一种“对称型 NAT”的 NAT 模型。这意味着路由器只接受和对端先前建立的连接（就是下一次请求建立新的连接映射）。

    NAT 的中继穿越方式Traversal Using Relays around NAT (TURN) 通过 TURN 服务器中继所有数据的方式来绕过“对称型 NAT”。你需要在 TURN 服务器上创建一个连接，然后告诉所有对端设备发包到服务器上，TURN 服务器再把包转发给你。很显然这种方式是开销很大的，所以只有在没得选择的情况下采用。

    1.5.SDP¶

    会话描述协议Session Description Protocol (SDP) 是一个描述多媒体连接内容的协议，例如分辨率，格式，编码，加密算法等。所以在数据传输时两端都能够理解彼此的数据。本质上，这些描述内容的元数据并不是媒体流本身。

    从技术上讲，SDP 并不是一个真正的协议，而是一种数据格式，用于描述在设备之间共享媒体的连接。

    记录 SDP 远远超出了本文档的范围。但是，这里有几件事值得注意。

    结构体¶

    SDP 由一行或多行 UTF-8 文本组成，每行以一个字符的类型开头，后跟等号（“ =”），然后是包含值或描述的结构化文本，其格式取决于类型。以给定字母开头的文本行通常称为“字母行”。例如，提供媒体描述的行的类型为“m”，因此这些行称为“m 行”。

    2.工作原理¶

    参考文档：

    1.WebRTC｜“浅入深出”的工作原理详解

    2.WebRTC 总体架构概述

    2.1.核心组件与协议¶

    • RTCPeerConnection：表示两个对等方之间的连接，用于传输媒体流和数据。
    • MediaStream：表示媒体流，由多个媒体轨道（如音频、视频）组成。
    • RTCDataChannel：用于在对等方之间交换任意二进制数据。
    • ICE（Interactive Connectivity Establishment）：用于在NAT和防火墙后面建立P2P连接的协议。
    • STUN/TURN服务器：STUN服务器帮助对等方获取公共IP地址，TURN服务器在无法直接建立P2P连接时作为中继。

    1. 双方先调用 getUserMedia 打开本地摄像头；
    2. 向信令服务器发送加入房间请求；
    3. Peer B 接收到 Peer A 发送的 offer SDP 对象，并通过PeerConnection的SetLocalDescription方法保存 Answer SDP 对象并将它通过信令服务器发送给 Peer A。
    4. 在 SDP 信息的 offer/answer 流程中，Peer A 和 Peer B 已经根据 SDP 信息创建好相应的音频 Channel 和视频 Channel，并开启Candidate 数据的收集，Candidate数据（本地IP地址、公网IP地址、Relay服务端分配的地址）。
    5. 当 Peer A 收集到 Candidate 信息后通过信令服务器发送给 Peer B。同样的过程 Peer B 对 Peer A 也会再发送一次。

    这样 Peer A 和 PeerB 就相互交换了媒体信息及网络信息，如果能达到一致(找到交集)，就可以开始通讯了。

    1.什么是WebRTC信令？¶

    WebRTC信令是使两个对等点（peer）之间能够建立、控制和终止实时通信会话的过程。

    信令的主要作用是交换会话描述和网络配置等信息，以便双方能够正确地建立连接和传输数据。

    信令并不直接处理媒体数据流，而是作为建立连接的初始引导过程。

    2.信令的主要功能¶

    • 交换会话描述（SDP）：会话描述协议（SDP）包含媒体类型、编解码器、带宽等信息，用于配置媒体会话。信令通道用于交换这些SDP描述符。
    • 交换网络信息（ICE候选者）：ICE（Interactive Connectivity Establishment）协议用于收集和交换网络候选者信息，以便在NAT和防火墙后面建立P2P连接。
    • 控制会话生命周期：包括会话的初始化、修改、终止等操作。
    • 用户管理：如用户加入房间、离开房间等。

    3.信令的工作流程¶

    1. 获取媒体流：使用浏览器的getUserMedia API获取用户的音视频流。
    2. 创建PeerConnection对象：在浏览器端创建RTCPeerConnection实例，生成包含媒体和网络信息的SDP（Session Description Protocol）描述符。
    3. 交换SDP描述符：通过信令服务器（如WebSocket）将SDP描述符发送给对端。对端收到后创建自己的RTCPeerConnection对象并回复自己的SDP描述符。
    4. ICE协议寻找最佳路径：使用ICE协议在多个网络路径中寻找最佳连接，可能涉及STUN和TURN服务器。
    5. 传输数据：连接建立后，双方开始传输音视频数据，WebRTC处理实时编码和解码。
    6. 关闭连接：通信结束后，关闭RTCPeerConnection对象，释放资源。

    4.信令服务器的实现¶

    • 选择传输协议：信令服务器可以使用HTTP、WebSocket、XMPP等协议进行消息交换。
    • 消息格式：通常使用JSON格式的消息，包含类型、发送者、接收者、SDP描述符等信息。
    • 房间管理：信令服务器可以管理多个房间，每个房间内的用户可以进行实时通信。

    5. 信令的安全性¶

    • 加密传输：信令数据应使用HTTPS或WSS等加密协议传输，以确保数据安全。
    • 身份验证：对参与会话的用户进行身份验证，防止未授权访问。

    2.2.关键技术与优化¶

    • 媒体协商（SDP）：双方交换支持的音视频格式。
    • 网络协商（ICE候选）：交换网络信息以建立连接。
    • UDP传输：使用UDP协议提高传输速度，允许部分数据丢失。
    • 编解码器选择：根据需求选择合适的编解码器，如H.264、VP8等。
    • 带宽适应：动态调整码率以适应网络条件。

    核心技术与优化实践:

    • P2P连接建立（信令与ICE）：WebRTC连接始于通过信令服务器（WebSocket/MQTT）交换SDP和ICE候选地址。STUN/TURN服务器则负责NAT穿透，如TURN可在60-80%的企业网络中作为中继，确保复杂网络环境下的连接。
    • 智能传输与优化：WebRTC持续监控网络并自适应码率。其拥塞控制算法有Google的GCC（低延迟）或BBR（高带宽），部分应用甚至会混合使用。通过MP-TCP等多路径传输技术，可并发使用WiFi和蜂窝网络，优化后平均延迟降低42%，可靠性达99.99%。
    • 硬件加速编解码：调用硬件（如NV12）编码器能极大节省资源。测试显示，与软件编码相比，CPU占用率可降低约64%，编码延迟降低达75%，对移动端尤为关键。
    • 通用数据通道：RTCDataChannel 提供类似TCP（可靠有序）或UDP（低延迟无序）的P2P数据通道，非常适合游戏同步、文件传输等场景。

    2.3.应用场景¶

    • 视频通话：如远程会议、在线教育等。
    • 实时直播：低延迟的直播应用。
    • 文件传输：利用数据通道实现文件共享。
    • 游戏互动：实时同步游戏状态。

    3.API¶

    接口文档

    WebRTC 的核心 API 主要包含三个模块，分别处理媒体流、对等连接和数据传输：

    1. MediaStream：访问音视频设备并管理媒体流
    2. RTCPeerConnection：建立和管理点对点连接
    3. RTCDataChannel：实现双向数据传输

    3.1.获取媒体设备（MediaDevices）¶

    1. 功能与作用¶

    • 用于访问摄像头、麦克风等设备，捕获音视频流。
    • 管理媒体轨道（如启用/禁用、添加/移除轨道）。

    2. 核心方法¶

    • getUserMedia() 请求用户授权并获取媒体流：
    navigator.mediaDevices.getUserMedia({ video: true, audio: true })
  .then(stream => {
    const video = document.getElementById('video');
    video.srcObject = stream; // 将流绑定到视频元素
  })
  .catch(error => console.error('获取媒体失败:', error));
    • enumerateDevices() 枚举可用的媒体设备（如摄像头、麦克风）：
    navigator.mediaDevices.enumerateDevices()
  .then(devices => devices.forEach(device => console.log(device.label)));
    • getTracks() / addTrack() / removeTrack() 操作媒体流中的轨道：
    const stream = new MediaStream();
const videoTrack = stream.getVideoTracks()[0];
stream.addTrack(newAudioTrack); // 添加音频轨道
stream.removeTrack(videoTrack); // 移除视频轨道

    3. 事件处理¶

    • onaddtrack / onremovetrack：监听轨道变化。
    • onactive / oninactive：监听流的活动状态13。

    4.Example¶

    // 使用promise
const constraints = {
    'video': true,
    'audio': true
}
navigator.mediaDevices.getUserMedia(constraints)
    .then(stream => {
        console.log('Got MediaStream:', stream);
    })
    .catch(error => {
        console.error('Error accessing media devices.', error);
    });

// 使用async/await
const openMediaDevices = async (constraints) => {
    return await navigator.mediaDevices.getUserMedia(constraints);
}

try {
    const stream = openMediaDevices({'video':true,'audio':true});
    console.log('Got MediaStream:', stream);
} catch(error) {
    console.error('Error accessing media devices.', error);
}

    3.2.对等连接（RTCPeerConnection）¶

    1. 功能与作用¶

    • 建立和维护点对点连接，处理 SDP 协商、NAT 穿透（通过 ICE）9。
    • 支持音视频传输与网络状态监控。

    2. 核心方法¶

    • 创建实例 配置 ICE 服务器（STUN/TURN）：
    const pc = new RTCPeerConnection({
  iceServers: [{ urls: "stun:stun.l.google.com:19302" }]
});

    • SDP 交换流程

    • 创建 Offer：

      pc.createOffer()
  .then(offer => pc.setLocalDescription(offer))
  .then(() => sendOfferToRemote(offer)); // 通过信令服务器发送

    • 处理 Answer：

      pc.setRemoteDescription(remoteAnswer)
  .then(() => console.log("Answer 设置成功"));

    • ICE 候选处理 监听并交换网络候选地址：

    pc.onicecandidate = event => {
  if (event.candidate) sendCandidateToRemote(event.candidate);
};
// 接收远程候选
pc.addIceCandidate(remoteCandidate);

    3.事件监听¶

    • ondevicechange: 监测媒体设备的变化，WebRTC提供了devicechange事件和ondevicec-hange事件句柄，与navigator.mediaDevices结合即可实时监控媒体设备的热插拔。
    • onaddstream: 监听通道中新加入的流，当addstream事件被触发时，会调用此处理程序。
    • ondatachannel: 当数据通道事件被触发时，会调用此处理程序。将RTCDataChannel添加到此连接时发送此事件。
    • onicecandidate: 当RTCIceCcandidate对象添加时，会发送此事件。
    • oniceconnectionstatechange: 当iceConnectionState的值发生更改时，会发送此事件。
    • onnegotiationneeded: 当触发需要协商的事件时，会调用此处理程序。此事件由浏览器发送，
    • onremovestream: 当信号状态signalingState更改事件被触发时，会调用此处理程序。
    • onsignalingstatechange: 当removestream事件被激发时，会调用。此事件是在从该连接中删除MediaStream时发送的。
    • ontrack: 当远程方将MediaStream添加到此连接时，会发送此事件。

    4. 媒体流关联¶

    • 添加本地流到连接：
    pc.addStream(localStream); // 旧版 API
localStream.getTracks().forEach(track => pc.addTrack(track, localStream)); // 新版推荐
    • 接收远程流：
    pc.ontrack = event => {
  const remoteVideo = document.getElementById('remoteVideo');
  remoteVideo.srcObject = event.streams[0];
};

    5.Example¶

    const configuration = {
    iceServers: [{ urls: 'stun:stun.l.google.com:19302' }]
};
const peerConnection = new RTCPeerConnection(configuration);

// 监听ICE候选者事件
peerConnection.addEventListener('icecandidate', event => {
    if (event.candidate) {
        signalingChannel.send({'new-ice-candidate': event.candidate});
    }
});

// 创建数据通道
const dataChannel = peerConnection.createDataChannel('chat');

// 监听数据通道消息
dataChannel.addEventListener('message', event => {
    console.log('Message from remote:', event.data);
});

    3.3.数据通道（RTCDataChannel）¶

    1. 功能与作用¶

    • 在已建立的连接上传输任意数据（文本、二进制），支持低延迟通信36。

    2. 核心方法¶

    • 创建数据通道 初始化时指定可靠性参数：
    const dc = pc.createDataChannel('chat', {
  ordered: true,   // 保证数据顺序
  maxRetransmits: 3 // 最大重传次数
});

    • 数据传输

    • 发送数据：

      dc.send(JSON.stringify({ type: 'message', content: 'Hello!' }));

    • 接收数据：

      dc.onmessage = event => console.log('收到消息:', event.data);

    3. 事件监听¶

    • onopen / onclose：监听通道状态变化。
    • onerror：处理传输错误。

    4.Example¶

    // 创建数据通道
const peerConnection = new RTCPeerConnection(configuration);
const dataChannel = peerConnection.createDataChannel('chat');

// 监听数据通道打开和关闭事件
dataChannel.addEventListener('open', event => {
    console.log('Data channel opened');
});

dataChannel.addEventListener('close', event => {
    console.log('Data channel closed');
});

// 发送消息
dataChannel.send('Hello, remote!');

// 接收消息
peerConnection.addEventListener('datachannel', event => {
    const dataChannel = event.channel;
    dataChannel.addEventListener('message', event => {
        console.log('Message from remote:', event.data);
    });
});

    3.4.信令（Signaling）¶

    信令是WebRTC中用于协调通信过程的机制，用于在浏览器之间传递控制信息，如会话的建立、媒体的兼容性和网络配置。WebRTC标准本身并不定义信令协议，开发者可以根据需求选择合适的协议，如WebSocket、SIP或是基于HTTP的长轮询等。

    • Example:
    // 使用WebSocket作为信令服务器
const signalingServer = new WebSocket('wss://signaling-server.com');

signalingServer.addEventListener('message', async message => {
    const data = JSON.parse(message.data);
    if (data.sdp) {
        await peerConnection.setRemoteDescription(
            new RTCSessionDescription(data)
        );
    } else if (data.iceCandidate) {
        await peerConnection.addIceCandidate(data.iceCandidate);
    }
});

// 发送信令消息
signalingServer.send(JSON.stringify({
    type: 'offer',
    sdp: peerConnection.localDescription
}));

    3.5. ICE候选者（ICE Candidates）¶

    ICE（Interactive Connectivity Establishment）协议用于收集和交换网络候选者信息，以便在NAT和防火墙后面建立P2P连接。ICE候选者是可以让对等点相互连接的潜在地址。

    • Example:
    // 收集本地ICE候选者
peerConnection.addEventListener('icecandidate', event => {
    if (event.candidate) {
        signalingChannel.send({
            type: 'ice-candidate',
            candidate: event.candidate
        });
    }
});

// 添加远程ICE候选者
signalingChannel.addEventListener('message', async message => {
    if (message.type === 'ice-candidate') {
        try {
            await peerConnection.addIceCandidate(message.candidate);
        } catch (e) {
            console.error('Error adding received ice candidate', e);
        }
    }
});

    3.6.会话描述（Session Description）¶

    会话描述协议（SDP）包含媒体类型、编解码器、带宽等信息，用于配置媒体会话。在WebRTC中，会话描述用于协商双方的媒体能力和网络配置。

    • Example:
    // 创建offer
const offer = await peerConnection.createOffer();
await peerConnection.setLocalDescription(offer);
signalingServer.send(JSON.stringify(offer));

// 处理远程answer
signalingServer.addEventListener('message', async message => {
    if (message.type === 'answer') {
        await peerConnection.setRemoteDescription(
            new RTCSessionDescription(message)
        );
    }
});

    4.1.Coturn¶

    Coturn是一个开源的STUN/TURN服务器，用于在网络地址转换（NAT）环境下实现实时通信。它支持STUN、TURN和ICE协议，帮助解决NAT穿越问题，使得位于不同NAT之后的设备能够进行通信。

    1.安装Coturn¶

    在Linux系统上，可以使用包管理器安装Coturn。例如，在Ubuntu系统上，运行以下命令：

    sudo apt-get install coturn

    对于其他系统，可以从Coturn的官方网站下载源码，并按照官方文档进行编译和安装。

    2.配置Coturn¶

    安装完成后，需要配置Coturn服务器。主要配置文件是/etc/coturn/turnserver.conf。使用文本编辑器打开该文件，并根据需求进行配置。常见的配置项包括：

    • listening-ip：指定Coturn服务器监听的IP地址。
    • external-ip：指定Coturn服务器的公网IP地址。
    • realm：指定realm值，用于标识服务器。
    • server-name：指定服务器名称。
    • listening-port：指定监听的端口号，默认为3478。
    • tls-listening-port：指定TLS监听的端口号，默认为5349。
    • log-file：指定日志文件路径。
    • verbose：启用详细日志记录。
    • user：指定TURN认证的用户。
    • lt-cred-mech：启用长期凭证机制。

    配置示例：

    listening-ip=服务器内网IP地址
external-ip=服务器公网IP地址
realm=yourdomain.com
server-name=turnserver
listening-port=3478
tls-listening-port=5349
log-file=/var/log/turnserver.log
verbose
user=test:test123
lt-cred-mech

    完成配置后，保存并退出文件，然后重新启动Coturn服务器以应用更改：

    sudo service coturn restart

    3.使用Coturn¶

    在应用程序中，可以通过设置STUN和TURN服务器的URL来使用Coturn。例如，在WebRTC应用中，配置iceServers选项：

    const configuration = {
    iceServers: [
        { urls: 'stun:your_turn_server_ip:3478' },
        { urls: 'turn:your_turn_server_ip:3478', credential: 'test123', username: 'test' }
    ]
};
const peerConnection = new RTCPeerConnection(configuration);

    4.测试Coturn¶

    可以使用turnutils_uclient工具来测试Coturn服务器是否正常工作。例如：

    turnutils_uclient -u test -w test123 -r yourdomain.com turn:your_turn_server_ip:3478

    如果服务器响应正常，说明Coturn配置成功。
