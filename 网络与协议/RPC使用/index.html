    1.RPC理解¶

    RPC（Remote Procedure Call，远程过程调用）是一种计算机通信协议，允许程序在不同的计算机上调用彼此的程序或服务，就像调用本地函数一样。RPC 的目的是使网络通信透明化，使开发者能够透过网络轻松地调用远程服务。

    1.1.RPC 的基本原理¶

    1. 请求-响应模型：客户端发起请求，服务器处理请求并返回响应。
    2. 序列化：客户端将请求数据序列化为可传输的格式，服务器接收到后反序列化。
    3. 网络传输：请求通过网络发送到服务器，服务器处理后将结果返回给客户端。
    4. 调用：客户端接收到响应后，反序列化得到的结果并继续执行。

    1.2.RPC 的优点¶

    • 透明性：开发者无需关心底层网络通信，调用接口就像调用本地方法一样简单。
    • 语言无关性：不同编程语言可以通过 RPC 进行通信。
    • 模块化：可以将服务拆分为多个模块，便于管理和扩展。

    1.3.RPC 的缺点¶

    • 性能开销：网络延迟和序列化/反序列化的开销可能影响性能。
    • 错误处理：网络请求可能失败，需要额外的错误处理机制。
    • 调试困难：远程调用的调试通常比本地调用复杂。

    1.4.主流 RPC 框架推荐¶

    1. gRPC
    2. 语言支持：支持多种编程语言，包括 Go、Java、Python、C++ 等。
    3. 特点：基于 HTTP/2，支持流式传输，具有高性能和低延迟。
    4. 使用场景：适合微服务架构，尤其是在需要高吞吐量的场景。
    5. Apache Thrift
    6. 语言支持：支持多种语言，如 Java、C++、Python、PHP 等。
    7. 特点：提供了 IDL（接口定义语言）来定义服务接口和数据类型，支持多种传输和协议。
    8. 使用场景：适合需要跨语言服务的应用。
    9. JSON-RPC 和 XML-RPC
    10. 语言支持：几乎所有编程语言都支持。
    11. 特点：基于 JSON 或 XML 格式进行数据传输，简单易用。
    12. 使用场景：适合轻量级的服务调用，常用于 Web 应用。
    13. Apache Dubbo
    14. 语言支持：主要支持 Java，但也有其他语言的实现。
    15. 特点：提供服务治理、负载均衡、容错等特性。
    16. 使用场景：适合大型分布式系统，尤其是在微服务架构中。
    17. RabbitMQ (使用 RPC 模式)
    18. 语言支持：支持多种编程语言。
    19. 特点：基于消息队列的 RPC 实现，支持异步处理。
    20. 使用场景：适用于需要高可用性和解耦的系统。
    框架 优点 缺点 适用场景 性能
    gRPC - 高性能（基于 HTTP/2 和 protobuf） - 学习曲线较陡：需要熟悉 protobuf 和 gRPC 的概念 - 高性能应用：微服务架构、实时通信、双向流 - 优秀：低延迟、高吞吐量
    - 多语言支持：Python、C++、Java、Go 等 - 调试难度较高：二进制协议使调试和监控更复杂 - 需要跨语言通信的分布式系统
    - 支持双向流式通信
    Thrift - 多语言支持：支持多种编程语言 - 配置和使用复杂：需要处理多种传输层和协议 - 需要灵活配置传输层和协议的应用 - 优秀：性能接近 gRPC
    - 灵活的传输和协议选项：满足多种性能和安全需求 - 社区活跃度不如 gRPC：生态系统较 gRPC 略逊一筹 - 需要跨语言通信的大型分布式系统
    RPyC - 易用性高：专为 Python 设计，符合 Pythonic 风格 - 仅支持 Python：无法与其他语言互操作 - 纯 Python 项目、小型项目或快速原型开发 - 一般：适合低并发场景
    - 透明的远程对象访问：可以直接调用远程对象的方法 - 性能较低：高并发和大规模分布式系统中可能瓶颈明显 - 开发强调易用性、无需跨语言支持的场景

    备注：

    • gRPC 是性能和功能的最佳选择，但需要时间学习和设置，适合大中型系统。
    • Thrift 在需要高度自定义传输协议时表现出色，适合对传输层要求严格的应用。
    • RPyC 则是纯 Python 项目中快速实现 RPC 的轻量化选择，但性能和扩展性相对有限。

    2.gRPC使用¶

    2.1.简介¶

    gRPC (gRPC Remote Procedure Calls[2]) 是一个跨平台的开源高性能远程过程调用（RPC）框架。

    gRPC最初由Google创建，它使用一个通用的RPC基础设施Stubby来连接其数据中心内外运行的大量微服务，始于2001年。[3]2015年3月，Google决定构建下一个版本的Stubby并将其开源，于是就有了gRPC。现在除了Google之外，许多组织都在使用它来支持从微服务到计算的“最后一英里”（移动、Web和物联网）的用例。

    gRPC基于HTTP/2协议传输数据，使用Protocol Buffers作为接口描述语言，并提供认证（authentication）、双向串流（bidirectional streaming）和流量控制、阻塞或非阻塞绑定以及取消和超时（Deadlines）等功能。它为许多语言生成跨平台的客户端和服务器绑定。最常见的使用场景包括在微服务风格的架构中连接服务，或将移动设备客户端连接到后端服务。[4]

    gRPC对HTTP/2的复杂使用使得在浏览器中无法实现gRPC客户端，而需要使用代理。[5]

    最常见的应用场景是[6]：

    • 微服务框架下，多种语言服务之间的高效交互。
    • 将手机服务、浏览器连接至后台
    • 产生高效的客户端库

    2.1.1.核心概念¶

    • 客户端和服务端：gRPC 是客户端和服务端之间的远程过程调用框架。
    • Protocol Buffers (Protobuf)：用来定义数据结构和服务接口。
    • 通信协议：
    • 基于 HTTP/2，支持双向流、压缩和多路复用。
    • 四种调用模式：
    • Unary RPC（单向 RPC）
    • Server Streaming RPC（服务端流式响应）
    • Client Streaming RPC（客户端流式请求）
    • Bidirectional Streaming RPC（双向流式通信）

    2.1.2.gRPC原理¶

    gRPC基于标准的HTTP/2协议，使用Protocol Buffers序列化协议进行数据交换。其原理如下：

    1. 定义gRPC服务：定义服务接口和消息类型，使用Protocol Buffers语言编写。定义完毕后，可以通过Protocol Buffers编译器生成对应的服务和消息类型的代码。
    2. 实现gRPC服务：在服务器端实现定义的服务接口，并使用gRPC提供的API和工具将服务注册到gRPC服务器中。
    3. 调用gRPC服务：在客户端使用生成的服务代码，构造请求消息并发送给服务器，服务器接收到请求后处理并返回响应消息。
    4. 处理gRPC响应：客户端接收到响应消息后进行处理，得到最终结果。

    gRPC的性能高、延时低的原因有以下几点：

    1. HTTP/2协议：gRPC基于HTTP/2协议进行通信，该协议支持多路复用、二进制帧传输等特性，可以更高效地利用网络资源，提高通信性能。
    2. Protocol Buffers序列化协议：与JSON、XML等常见的序列化协议相比，Protocol Buffers具有更高的压缩率和更快的序列化/反序列化速度，从而提高了数据传输效率。
    3. 代码生成：gRPC使用Protocol Buffers编译器自动生成服务接口和消息类型的代码，避免了手动编写、维护代码的工作量，同时也提高了代码的质量和性能。
    4. 流式传输：gRPC支持流式传输，即可以同时发送多个请求或响应，从而提高了并发性能。
    5. 客户端和服务端的异步调用：gRPC支持异步调用，客户端可以通过Future或Callback等方式进行异步调用，从而提高了系统的响应速度和吞吐量。

    2.1.3.Protobuf语法¶

    1.简介¶

    Protocol Buffers 一种与语言无关、与平台无关的可扩展机制，用于序列化结构化数据。

    它类似于 JSON，但体积更小、速度更快，并且会生成本机语言绑定。您只需定义一次数据的结构，然后就可以使用专门生成的源代码轻松地将结构化数据写入各种数据流并使用各种语言读取这些结构化数据。

    Protocol Buffers 定义语言（在 .proto文件中创建）、proto 编译器生成的与数据接口的代码、特定于语言的运行时库、写入文件（或通过网络连接发送）的数据的序列化格式以及序列化数据的组合。

    Protocol Buffers 为大小高达几兆字节的类型化、结构化数据包提供了一种序列化格式。该格式既适用于短暂的网络流量，也适用于长期的数据存储。Protocol Buffers 可以扩展为新的信息，而不会使现有数据失效或需要更新代码。广泛用于服务器间通信以及磁盘上数据的存档存储。

    其一些优点包括：

    • 紧凑的数据存储
    • 快速解析
    • 适用于多种编程语言
    • 通过自动生成的类优化功能

    编译器 protoc 直接支持以下语言：

    • C++
    • C#
    • Java
    • kotlin
    • Objective-C
    • PHP
    • Python
    • Ruby
    2.语法¶

    定义.proto文件时，您可以指定字段为optional 或 repeated（proto2 和 proto3），或者在 proto3 中将其设置为默认的隐式存在。required（proto3 中没有将字段设置为的选项，proto2 中强烈建议不要使用此选项。有关更多信息，请参阅 指定字段规则中的“必填是永久的” 。）

    设置字段的可选性/可重复性后，您可以指定数据类型。协议缓冲区支持常见的原始数据类型，例如整数、布尔值和浮点数。有关完整列表，请参阅 标量值类型。

    字段还可以是：

    • 一种message类型，以便您可以嵌套定义的各个部分，例如重复的数据集。
    • 一种enum类型，因此您可以指定一组可供选择的值。
    • 一种oneof类型，当一条消息有多个可选字段，并且最多同时设置一个字段时可以使用该类型。
    • 一种map类型，用于将键值对添加到您的定义中。

    在 proto2 中，消息可以允许扩展定义消息本身之外的字段。例如，protobuf 库的内部消息架构允许扩展自定义、特定于用途的选项。

    有关可用选项的更多信息，请参阅 proto2或 proto3的语言指南。

    2.1.定义消息类型¶
    syntax = "proto3";

// SearchRequest 搜索请求
message SearchRequest {
  string query = 1;           // 查询字符串
  int32 page_number = 2;      // 页码
  int32 results_per_page = 3; // 每页条数
}

// SearchResponse 搜索响应
message SearchResponse {
    ...
}
    • 该文件的第一行指定您正在使用 protobuf 语言规范的 proto3 修订版。
    • edition(或对于proto2 syntax/proto3) 必须是文件的第一个非空、非注释行。
    • 如果没有指定 edition或，协议缓冲区编译器将假定您正在使用proto2。syntax
    • 消息SearchRequest定义指定了三个字段（名称/值对），每个字段对应一个要包含在此类消息中的数据。每个字段都有一个名称和一个类型。
    2.2.基本规范¶

    • 文件以.proto做为文件后缀，除结构定义外的语句以分号结尾

    • 结构定义可以包含：message、service、enum

    • rpc方法定义结尾的分号可有可无

    • Message命名采用驼峰命名方式，字段命名采用小写字母加下划线分隔方式

      message SongServerRequest {
      required string song_name = 1;
  }
    • Enums类型名采用驼峰命名方式，字段命名采用大写字母加下划线分隔方式
      enum Foo {
      FIRST_VALUE = 1;
      SECOND_VALUE = 2;
  }
    • Service与rpc方法名统一采用驼峰式命名
    2.3.字段规则¶
    • 字段格式：限定修饰符 | 数据类型 | 字段名称 | = | 字段编码值 | [字段默认值]
    • 限定修饰符包含:
    • singular: 单一字段, 默认规则.
    • required: 表示是一个必须字段 （已废弃），必须相对于发送方，在发送消息之前必须设置该字段的值，对于接收方，必须能够识别该字段的意思。发送之前没有设置required字段或者无法识别required字段都会引发编解码异常，导致消息被丢弃
    • optional：表示是一个可选字段，可选对于发送方，在发送消息时，可以有选择性的设置或者不设置该字段的值。对于接收方，如果能够识别可选字段就进行相应的处理，如果无法识别，则忽略该字段，消息中的其它字段正常处理。---因为optional字段的特性，很多接口在升级版本中都把后来添加的字段都统一的设置为optional字段，这样老的版本无需升级程序也可以正常的与新的软件进行通信，只不过新的字段无法识别而已，因为并不是每个节点都需要新的功能，因此可以做到按需升级和平滑过渡
    • repeated：表示一个重复字段, 该字段可以包含0~N个元素。其特性和optional一样，但是每一次可以包含多个值。可以看作是在传递一个数组的值

    • oneof: 表示一对一字段, 用于定义一组字段，其中只能有一个字段被设置。字段是互斥的，设置了其中一个字段后，其他字段将被清除。

    • 数据类型

    • Protobuf定义了一套基本数据类型。几乎都可以映射到C++\Java等语言的基础数据类型
    • N 表示打包的字节并不是固定。而是根据数据的大小或者长度
    • 关于 fixed32 和int32的区别。fixed32的打包效率比int32的效率高，但是使用的空间一般比int32多。因此一个属于时间效率高，一个属于空间效率高
    • 字段名称
    • 字段名称的命名与C、C++、Java等语言的变量命名方式几乎是相同的
    • protobuf建议字段的命名采用以下划线分割的驼峰式。例如 first_name 而不是firstName
    • 字段编码值
    • 有了该值，通信双方才能互相识别对方的字段，相同的编码值，其限定修饰符和数据类型必须相同，编码值的取值范围为 1~2^32（4294967296）
    • 其中 1~15的编码时间和空间效率都是最高的，编码值越大，其编码的时间和空间效率就越低，所以建议把经常要传递的值把其字段编码设置为1-15之间的值
    • 1900~2000编码值为Google protobuf 系统内部保留值，建议不要在自己的项目中使用
    • 字段默认值
    • 当在传递数据时，对于required数据类型，如果用户没有设置值，则使用默认值传递到对端
    • 在 proto3 中，默认值的行为如下：
      • 数值类型（如 int32, int64, uint32, uint64, sint32, sint64, fixed32, fixed64, sfixed32, sfixed64, double, float）：默认值为 0。
      • 布尔类型（bool）：默认值为 false。
      • 字符串类型（string）：默认值为空字符串 ""。
      • 字节类型（bytes）：默认值为空字节串 b""。
      • 枚举类型：默认值为枚举类型中定义的第一个枚举值（通常是 0）。
      • 消息类型：默认值为字段的默认实例，即所有字段都设置为默认值。
    2.4.定义service¶
    • 如果想要将消息类型用在RPC系统中，可以在.proto文件中定义一个RPC服务接口，protocol buffer编译器会根据所选择的不同语言生成服务接口代码
    • 例如，想要定义一个RPC服务并具有一个方法，该方法接收SearchRequest并返回一个SearchResponse，此时可以在.proto文件中进行如下定义：
    service SearchService {
    rpc Search (SearchRequest) returns (SearchResponse) {}
}
    • 生成的接口代码作为客户端与服务端的约定，服务端必须实现定义的所有接口方法，客户端直接调用同名方法向服务端发起请求，比较麻烦的是，即便业务上不需要参数也必须指定一个请求消息，一般会定义一个空message

    • --python_out

    • 作用：这个选项指定生成的普通 Protocol Buffers Python 代码的输出目录。这些代码包括消息类型的定义，但不包括 gRPC 服务的实现。

    • 使用细节：

      • 你可以指定一个目录路径作为参数，例如 --python_out=/path/to/output。
      • 如果不指定路径，默认是当前目录（.）。
      • 生成的文件名会与 .proto 文件名相同，但扩展名变为 .py。

    • --grpc_python_out

    • 作用：这个选项指定生成的 gRPC Python 服务代码的输出目录。这些代码包括服务接口的定义和服务的实现。

    • 使用细节：

      • 同样，你可以指定一个目录路径作为参数，例如 --grpc_python_out=/path/to/output。
      • 如果不指定路径，默认也是当前目录（.）。
      • 生成的文件名会与 .proto 文件名相同，但会加上 _grpc 后缀，例如 example_pb2_grpc.py。

    • 相互影响和调用

    • 独立性：这两个选项是独立的，它们生成的代码互不依赖。--python_out 生成的代码包含消息类型定义，而 --grpc_python_out 生成的代码依赖于这些消息类型来定义服务接口和实现。

    • 调用关系：

      • --grpc_python_out 生成的代码会导入 --python_out 生成的代码，因为服务的实现需要使用消息类型。
      • 因此，确保 --grpc_python_out 指定的目录在 Python 路径中可以访问到 --python_out 生成的文件，或者将它们放在同一个目录下。

    2.2.交互¶

    下面是 gRPC 的交互流程：

    1. 客户端调用方法：
    2. 客户端调用服务上的方法，发送请求（Unary、Streaming 等）。
    3. 服务端接收请求：
    4. 服务端接收到请求后，执行对应的服务方法。
    5. 服务端返回结果：
    6. 服务端将结果封装后返回给客户端。
    7. 客户端接收结果：
    8. 客户端接收到结果，完成调用。
    sequenceDiagram
    participant Client
    participant gRPC_ClientStub
    participant gRPC_ServerStub
    participant Server

    Client->>gRPC_ClientStub: 调用 gRPC 方法 (e.g., SayHello)
    gRPC_ClientStub->>gRPC_ServerStub: 通过 HTTP/2 发送请求
    gRPC_ServerStub->>Server: 执行服务逻辑 (e.g., SayHello)
    Server->>gRPC_ServerStub: 返回结果
    gRPC_ServerStub->>gRPC_ClientStub: 返回响应
    gRPC_ClientStub->>Client: 返回结果给客户端

    客户端调用服务：客户端通过 gRPC_ClientStub 调用 SayHello 方法。

    请求发送到服务端：gRPC_ClientStub 将请求通过 HTTP/2 发送到服务端。

    服务端处理请求：服务端处理请求，执行业务逻辑。

    服务端返回结果：服务端将结果返回给客户端。

    2.3.使用¶

    2.3.1.基本使用示例¶

    1.安装gRPC和protobuf：¶
    # 安装 gRPC 工具
$ pip install grpcio grpcio-tools protobuf
    2.Step 1: 定义 .proto 文件¶

    gRPC 使用 .proto 文件来定义服务和消息格式，以下是一个简单的示例：

    syntax = "proto3";

service Greeter {
  // Unary RPC
  rpc SayHello (HelloRequest) returns (HelloResponse);
}

message HelloRequest {
  string name = 1;
}

message HelloResponse {
  string message = 1;
}
    2.Step 2: 生成 gRPC 代码¶

    使用 protoc 工具 - protobuf编译器生成客户端和服务端的代码。

    python -m grpc_tools.protoc -I. --python_out=. --grpc_python_out=. greeter.proto

    这会在当前目录下生成 greeter_pb2.py 和 greeter_pb2_grpc.py 两个文件。

    3.Step 3: 实现服务端¶

    创建服务端，继承生成的 GreeterServicer 类，并实现方法。

    from concurrent import futures
import grpc
import greeter_pb2
import greeter_pb2_grpc

class GreeterService(greeter_pb2_grpc.GreeterServicer):
    def SayHello(self, request, context):
        return greeter_pb2.HelloResponse(message=f"Hello, {request.name}!")

# 启动 gRPC 服务
server = grpc.server(futures.ThreadPoolExecutor(max_workers=10))
greeter_pb2_grpc.add_GreeterServicer_to_server(GreeterService(), server)
server.add_insecure_port('[::]:50051')
server.start()
print("gRPC server is running...")
server.wait_for_termination()
    4.Step 4: 实现客户端¶

    创建客户端，连接到 gRPC 服务并调用接口。

    import grpc
import greeter_pb2
import greeter_pb2_grpc

class AccessLogClientInterceptor(grpc.UnaryUnaryClientInterceptor):
    def intercept_unary_unary(self, continuation, client_call_details, request):
        logger.info(f"Sending call to {client_call_details.method}")
        response = continuation(client_call_details, request)
        return response

def run():
    # 连接到 gRPC 服务
    with grpc.insecure_channel('localhost:50051') as _channel:
        channel = grpc.intercept_channel(_channel, AccessLogClientInterceptor())
        stub = greeter_pb2_grpc.GreeterStub(channel)

        # 调用 SayHello 方法
        response = stub.SayHello(greeter_pb2.HelloRequest(name="Alice"))
        print(f"Greeter Response: {response.message}")

if __name__ == "__main__":
    run()

    输出结果应该为：

    Hello, Alice!

from concurrent import futures
import grpc
import example_pb2
import example_pb2_grpc

class LoggingInterceptor(aio.ServerInterceptor):
    async def intercept_service(
        self,
        continuation: Callable[
            [grpc.HandlerCallDetails], Awaitable[grpc.RpcMethodHandler]
        ],
        handler_call_details: grpc.HandlerCallDetails,
    ) -> grpc.RpcMethodHandler:
        start_time = timeit.default_timer()
        method_name = handler_call_details.method
        response = await continuation(handler_call_details)
        end_time = timeit.default_timer()
        logger.info(f"Received request: {method_name} OK {(1000 * (end_time - start_time)):.3f} ms.")
        return response

class HelloService(example_pb2_grpc.HelloServiceServicer):
    async def SayHello(self, request, context):
        return example_pb2.HelloResponse(message=f"Hello, {request.name}!")

        ThreadPoolExecutor(max_workers=workers),
        interceptors=[LoggingInterceptor()]

    server.add_insecure_port("[::]:50051")
    await server.start()
    print("Server started on port 50051")
    await server.wait_for_termination()

if __name__ == "__main__":
    asyncio.run(serve())

    2.3.3.使用场景¶

    gRPC 优势：

    • 高性能：基于 HTTP/2 和 Protobuf。
    • 多语言支持：支持多种语言的客户端和服务端实现。
    • 四种调用模式满足不同场景。

    适用场景：

    • 微服务之间的高效通信。
    • 多语言服务交互。
    • 实时数据流。

    对比:

    指标 grpc.aio（官方） grpclib betterproto
    性能 高（基于 C++ 核心） 较高（纯 Python 实现） 较高（基于 asyncio）
    依赖 需要 grpcio（C++ 二进制） 无需二进制依赖 无需二进制依赖
    代码生成风格 原生 protobuf 风格 接近原生 protobuf 更 Pythonic
    易用性 较复杂，代码较冗长 简单轻量 开发友好
    适用场景 高性能分布式应用 轻量级异步服务 快速开发、现代化应用

    选择建议:

    • 高性能：优先使用官方库 grpc.aio。
    • 轻量化、无二进制依赖：选择 grpclib。
    • 现代化开发：选择 betterproto。

    2.3.4.grpc 相关库¶

    以下是一些常用的Python gRPC库，以及它们的优缺点：

    1.grpcio¶

    grpcio 是gRPC官方提供的Python库，支持同步和异步API，以及流式RPC。它基于C核心实现，性能较高，并且具有广泛的社区支持。

    • 性能较高
    • 支持同步和异步API

    缺点：

    • 安装较慢（需要编译C核心）
    • API使用较为复杂
    2.grpclib¶

    grpclib 是一个基于asyncio的gRPC库，提供了比官方grpcio更简单的API，支持流式RPC，并且兼容Python 3.5及以上版本。

    优点：

    • 使用简单，API易于理解
    • 支持流式RPC
    • 兼容Python 3.5及以上版本

    缺点：

    3.JSON-RPC 使用¶

    3.1.简介¶

    JSON-RPC，是一个无状态且轻量级的远程过程调用（RPC）传送协议，其传输內容透過 JSON 為主。

    基于 HTTP 或 WebSocket 进行通信。它是一种无状态、无约束的协议，适合简单的 RPC 调用。

    相較於一般的 REST 透過網址（如 GET /user）呼叫遠端伺服器，JSON-RPC 直接在內容中定義了欲呼叫的函式名稱（如 {"method": "getUser"}），這也令開發者不會陷於該使用 PUT 或者 PATCH 的問題之中。

    它允许运行在基于 Socket、HTTP 等诸多不同消息传输环境的同一进程中。其使用 JSON（RFC 4627）作为数据格式。

    核心概念:

    • 协议特点：

    • 基于 JSON 格式，简单易理解。

    • 无状态：每次请求都独立。

    • 双向通信：支持客户端调用服务端，也支持服务端调用客户端（通过 WebSocket 实现）。

    • 结构：

    • 请求：

      {
  "jsonrpc": "2.0",
  "method": "methodName",
  "params": {
    "param1": "value1",
    "param2": "value2"
  },
  "id": 1
}
      • jsonrpc: 协议版本（固定为 "2.0"）。
      • method: 调用的方法名。
      • params: 调用方法的参数（可选，支持数组或对象）。
      • id: 请求 ID，用于匹配响应。

    • 响应：

      {
  "jsonrpc": "2.0",
  "result": {
    "key": "value"
  },
  "id": 1
}

      • result: 方法调用的返回值（如果成功）。

      • id: 与请求中的 ID 对应。

      • 如果调用失败，则返回 error字段：

      {
  "jsonrpc": "2.0",
  "error": {
    "code": -32601,
    "message": "Method not found"
  },
  "id": 1
}

    3.2.交互¶

    3.2.1.JSON-RPC 交互流程¶

    交互步骤:

    1. 客户端通过 HTTP 或 WebSocket 发送 JSON-RPC 请求。
    2. 服务端解析请求，执行对应的方法。
    3. 服务端返回 JSON-RPC 格式的响应，包含结果或错误信息。
    4. 客户端解析响应，获取结果。

    交互流程图:

    sequenceDiagram
    participant Client
    participant Server

    Client->>Server: JSON-RPC 请求 (method, params, id)
    Server->>Server: 解析请求并执行方法
    Server->>Client: JSON-RPC 响应 (result 或 error, id)

    3.3.使用¶

    3.3.1.基本使用步骤¶

    1.Step 1: 实现服务端¶

    以下是基于 Python 的 JSON-RPC 服务端实现，使用 json-rpc 库：

    from jsonrpcserver import method, serve

@method
def add(a: int, b: int) -> int:
    return a + b

@method
def subtract(a: int, b: int) -> int:
    return a - b

if __name__ == "__main__":
    serve()

    服务端会监听 HTTP 或 WebSocket 请求，解析 JSON-RPC 请求并返回响应。

    2.Step 2: 实现客户端¶

    客户端通过发送 JSON-RPC 格式的请求调用服务端的接口：

    import requests
import json

# 定义服务端地址
URL = "http://127.0.0.1:5000"

# 发送 JSON-RPC 请求
payload = {
    "jsonrpc": "2.0",
    "method": "add",
    "params": {"a": 5, "b": 3},
    "id": 1
}
headers = {"Content-Type": "application/json"}
response = requests.post(URL, data=json.dumps(payload), headers=headers)

# 打印结果
print(response.json())

    3.3.2.使用场景¶

    JSON-RPC 与 gRPC、REST 对比:

    特性 JSON-RPC gRPC REST
    协议 自定义协议（基于 JSON） 基于 HTTP/2 和 Protocol Buffers 基于 HTTP/1.1
    数据格式 JSON Protobuf JSON、XML
    性能 中等 高 中等
    双向通信支持 支持（通过 WebSocket） 支持 不支持
    流式传输支持 不支持 支持 不支持
    工具支持 少 丰富 丰富
    使用场景 简单的 RPC 调用 高性能、跨语言的微服务通信 面向资源的 RESTful API

    场景建议:

    • JSON-RPC：
    • 适合简单的 RPC 调用。
    • 适合客户端和服务端都在同一语言生态中。
    • 适合需要快速实现的项目。
    • gRPC：
    • 适合高性能、跨语言的微服务架构。
    • 需要双向通信或流式传输支持。
    • 对性能要求高的分布式系统。
    • REST：
    • 适合基于资源的系统（如 RESTful API）。
    • 适合与第三方系统集成。
    • 更广泛的工具和社区支持。

    选择框架和协议时，需要根据项目的需求来决定：

    • 如果追求简单和快速实现，使用 JSON-RPC。
    • 如果需要高性能和跨语言支持，使用 gRPC。
    • 如果需要标准化且与前端交互多，使用 REST。
