    Redis 是一个开源的内存数据结构存储，用作数据库、缓存、消息代理和流引擎。 Redis 提供了五种主要的数据类型：String（字符串）、List（列表）、Set（集合）、Hash（哈希）、具有范围查询的ZSet（有序集合）以及位图、超日志、地理空间索引和流。 Redis 具有内置复制、Lua 脚本、LRU、事务和不同级别的磁盘持久性，并通过 Redis Sentinel 和 Redis Cluster 自动分区提供高可用性。每种数据类型在性能、存储结构和使用场景上有所不同，了解它们的特性有助于选择合适的数据结构，以达到最优性能。

    参考: Redis文档

    〇、安装:¶

    # linux
$ wget http://download.redis.io/releases/redis-6.0.8.tar.gz
$ tar xzf redis-6.0.8.tar.gz
$ cd redis-6.0.8
$ make

# or
$ sudo apt update
$ sudo apt install redis-server

    一、常用数据结构¶

    参考: Redis数据类型

    数值操作:

    # 将 key 中储存的数字值增一
INCR key
# 将 key 中储存的数字值减一
DECR key
# key 所储存的值减去给定的减量值（decrement）
DECRBY key decrement
# key 所储存的值增加给定的增量值
INCRBY key increment
# 增加指定浮点数
INCRBYFLOAT key increment

    1. String（字符串）¶

    性能特点：¶

    • Redis 的基本数据类型，底层实现是动态字符串（类似 C 语言的 char[]）。
    • 字符串操作的性能非常高，读写时间复杂度为 O(1)，即常数时间操作。
    • 适合存储简单的数据，如整数、浮点数、二进制数据、文本等。

    使用场景：¶

    • 缓存功能：可以将查询到的数据作为字符串存储，典型的应用是缓存数据库中的简单键值对数据。
    • 计数器：利用 INCR、DECR 操作，可以实现高效的分布式计数器。
    • 会话管理：常用来存储用户会话信息，如 Token 或 Session。

    示例：¶

    SET key "value"       # 设置字符串
GET key               # 获取字符串
INCR counter          # 递增
    • 其他
    # 设置
SET key value
MSET key value [key value ...]
# 获取指定 key 的值
GET key
# 将值 value 关联到 key ，并将 key 的过期时间设为 seconds (以秒为单位)
SETEX key seconds value
# 只有在 key 不存在时设置 key 的值
SETNX key value
MSETNX key value [key value ...]
# 返回 key 所储存的字符串值的长度
STRLEN key
# 查询
SCAN cursor [MATCH pattern] [COUNT count]
# 判断
exists key
# 删除
del key

    2. List（列表）¶

    性能特点：¶

    • Redis 的 List 是双向链表，支持在列表两端进行快速插入、删除操作。
    • 操作复杂度为：头尾插入/删除操作的时间复杂度为 O(1)，随机访问和遍历则为 O(n)。

    使用场景：¶

    • 消息队列：由于列表具有顺序，可以使用 LPUSH 和 RPOP 实现简单的消息队列功能，保证 FIFO（先进先出）。
    • 任务队列：可以利用 BLPOP 和 BRPOP 实现阻塞式任务队列，常用于生产者-消费者模型。
    • 时间线功能：如微博、朋友圈等可以使用 Redis List 来存储时间线数据。

    示例：¶

    LPUSH queue "task1"    # 从左边插入
RPOP queue             # 从右边弹出
LRANGE list 0 10       # 获取列表范围
    • 其他
    # 通过索引获取列表中的元素
LINDEX key index
# 获取列表长度
LLEN key
# 获取列表 start 和 stop 之间 的元素
LRANGE KEY start stop
# 将一个或多个值插入到列表头部
LPUSH key value1 [value2]
# 将一个值插入到已存在的列表头部
LPUSHX key value
# 在列表中添加一个或多个值
RPUSH key value1 [value2]
# 为已存在的列表添加值
RPUSHX key value
# 通过索引设置列表元素的值
LSET key index value
# 移出并获取列表的第一个元素
LPOP key
# 移除列表的最后一个元素，返回值为移除的元素
RPOP key
# 从列表中移除 count 个值与 value 相等的元素
LREM key count value
# 移除列表的最后一个元素，并将该元素添加到另一个列表并返回
RPOPLPUSH source destination
# 移出并获取列表的最后一个元素， 如果列表没有元素会阻塞列表直到等待超时或发现可弹出元素为止
BRPOP key1 [key2 ] timeout
# 移出并获取列表的第一个元素， 如果列表没有元素会阻塞列表直到等待超时或发现可弹出元素为止
BLPOP key1 [key2 ] timeout
# 对一个列表进行修剪(trim)，就是说，让列表只保留指定区间内的元素，不在指定区间之内的元素都将被删除
LTRIM key start stop

    3. Hash（哈希）¶

    性能特点：¶

    • Redis 的 Hash 是一个键值对集合，类似于 Map 或 Dict，适合存储对象类型的数据。
    • 单个字段的读写操作复杂度为 O(1)，适合存储较小的键值对。
    • Hash 结构非常适合在 Redis 中节省内存，因为它在存储小型字段时非常高效。

    使用场景：¶

    • 对象存储：适合存储具有多个属性的对象，如用户信息（姓名、年龄、邮箱等），比将每个属性作为单独的字符串存储更节省内存。
    • 计数器系统：可以通过 HINCRBY 来对某个字段的值进行累加操作，适合存储多个统计数据。

    示例：¶

    HSET user:1001 name "Alice"    # 设置字段
HGET user:1001 name            # 获取字段
HGETALL user:1001              # 获取整个哈希表
    • 其他
    # 删除一个或多个哈希表字段
HDEL key field1 [field2]
# 查看哈希表 key 中，指定的字段是否存在
HEXISTS key field
# 将哈希表 key 中的字段 field 的值设为 value
HSET key field value
HMSET key field1 value1 [field2 value2 ]
# 只有在字段 field 不存在时，设置哈希表字段的值
HSETNX key field value
# 获取存储在哈希表中指定字段的值
HGET key field
HMGET key field1 [field2]
HGETALL key
# 获取所有哈希表中的字段
HKEYS key
# 获取哈希表中所有值
HVALS key
# 获取哈希表中字段的数量
HLEN key
# 迭代哈希表中的键值对
HSCAN key cursor [MATCH pattern] [COUNT count]

    4. Set（集合）¶

    性能特点：¶

    • Redis 的 Set 是一个无序集合，底层使用哈希表实现，支持去重操作。
    • 添加、删除、判断元素是否存在的时间复杂度为 O(1)。
    • 可以高效地执行集合的交集、并集、差集等运算，时间复杂度为 O(N)，其中 N 是集合中元素的数量。

    使用场景：¶

    • 去重功能：适合存储不允许重复的元素集合，如用户 ID、IP 等。
    • 社交应用中的共同好友、共同关注：可以用 Redis Set 来存储用户的好友列表，快速计算用户之间的共同好友。
    • 标签管理：可以使用集合来存储文章的标签，并进行集合运算如交集、并集等。

    示例：¶

    SADD set1 "value1"     # 添加元素
SREM set1 "value1"     # 移除元素
SINTER set1 set2       # 求交集
    • 其他
    # 向集合添加一个或多个成员
SADD key member1 [member2]
# 获取集合的成员数
SCARD key
# 判断 member 元素是否是集合 key 的成员
SISMEMBER key member
# 返回集合中的所有成员
SMEMBERS key
# 移除并返回集合中的一个随机元素
SPOP key
# 移除集合中一个或多个成员
SREM key member1 [member2]
# 返回第一个集合与其他集合之间的差异
SDIFF key1 [key2]
# 返回给定所有集合的交集
SINTER key1 [key2]
# 返回给定所有集合的交集并存储在 destination 中
SINTERSTORE destination key1 [key2]
# 返回所有给定集合的并集
SUNION key1 [key2]
# 将 member 元素从 source 集合移动到 destination 集合
SMOVE source destination member
# 所有给定集合的并集存储在 destination 集合中
SUNIONSTORE destination key1 [key2]
# 迭代集合中的元素
SSCAN key cursor [MATCH pattern] [COUNT count]

    5. ZSet（有序集合）¶

    性能特点：¶

    • Redis 的 ZSet 是一个带权重的有序集合，底层实现是跳表（SkipList）。
    • 支持根据元素的分数（Score）进行排序，插入、删除、更新和根据分数范围查询的时间复杂度为 O(logN)，N 为集合的元素数量。

    使用场景：¶

    • 排行榜：可以用 ZSet 来存储分数并进行排序，适合实现类似积分排行榜、排名系统等功能。
    • 按时间排序的任务系统：ZSet 可以根据时间戳作为分数，来管理时间敏感的任务。
    • 延时队列：可以用 ZSet 结合分数来表示任务的延时时间，在特定时间点触发。

    示例：¶

    ZADD leaderboard 100 "user1"   # 添加用户及分数
ZRANGE leaderboard 0 10        # 获取排行榜前10名
ZRANK leaderboard "user1"      # 获取用户排名
    • 其他
    # 向有序集合添加一个或多个成员，或者更新已存在成员的分数
ZADD key score1 member1 [score2 member2]
# 获取有序集合的成员数
ZCARD key
# 计算在有序集合中指定区间分数的成员数
ZCOUNT key min max
# 在有序集合中计算指定字典区间内成员数量
ZLEXCOUNT key min max
# 有序集合中对指定成员的分数加上增量 increment
ZINCRBY key increment member
# 通过索引区间返回有序集合指定区间内的成员
ZRANGE key start stop [WITHSCORES]
# 通过字典区间返回有序集合的成员
ZRANGEBYLEX key min max [LIMIT offset count]
# 通过分数返回有序集合指定区间内的成员
ZRANGEBYSCORE key min max [WITHSCORES] [LIMIT]
# 返回有序集合中指定成员的索引
ZRANK key member
# 返回有序集中，成员的分数值
ZSCORE key member
# 移除有序集合中的一个或多个成员
ZREM key member [member ...]
# 移除有序集合中给定的字典区间的所有成员
ZREMRANGEBYLEX key min max
# 移除有序集合中给定的排名区间的所有成员
ZREMRANGEBYRANK key start stop
# 移除有序集合中给定的分数区间的所有成员
ZREMRANGEBYSCORE key min max
# 返回有序集中指定区间内的成员，通过索引，分数从高到低
ZREVRANGE key start stop [WITHSCORES]
# 返回有序集中指定分数区间内的成员，分数从高到低排序
ZREVRANGEBYSCORE key max min [WITHSCORES]
# 返回有序集合中指定成员的排名，有序集成员按分数值递减(从大到小)排序
ZREVRANK key member
# 计算给定的一个或多个有序集的并集，并存储在新的 key 中
ZUNIONSTORE destination numkeys key [key ...]
# 计算给定的一个或多个有序集的交集并将结果集存储在新的有序集合 destination 中
ZINTERSTORE destination numkeys key [key ...]
# 迭代有序集合中的元素（包括元素成员和元素分值）
ZSCAN key cursor [MATCH pattern] [COUNT count]

    Redis 数据类型性能对比总结：¶

    数据类型 基本操作 时间复杂度 特点 适用场景
    String 读写操作 O(1) 高效简单 缓存数据、计数器、会话管理等
    List 头尾插入 O(1) 双向链表，支持阻塞 消息队列、时间线数据、任务队列
    Set 添加/删除 O(1) 支持集合运算，无序去重 去重、共同好友、标签管理
    Hash 读写单字段 O(1) 节省内存，适合小型键值对 对象存储、计数器系统
    ZSet 插入/删除 O(logN) 有序，适合排序 排行榜、延时队列、任务调度

    总结：

    • String 适合存储单一数据的缓存、计数器和简单的键值对数据，操作非常高效。
    • List 适合需要顺序存储的场景，尤其适合队列和消息系统。
    • Set 适合需要去重的集合数据，尤其是社交系统中的好友关系、标签系统等。
    • Hash 适合存储复杂对象、多个字段的数据，内存使用效率高。
    • ZSet 适合需要排序、排名的场景，常用于排行榜、调度系统。

    二、常见问题¶

    1. 缓存穿透¶

    定义：¶

    缓存穿透指的是查询一个不存在的数据，因为缓存中没有数据，并且数据库中也没有对应的记录，导致每次查询都要访问数据库。这种情况下，缓存无法发挥作用，所有请求都会落到数据库，可能会导致数据库的负载骤增。

    场景：¶

    用户频繁请求一些数据库中不存在的记录，例如，查询用户ID为负数或极大的ID等非法数据。这些请求不会命中缓存，每次都直接请求数据库。

    解决方案：¶

    1. 缓存空值： 当查询结果为空时，将空值存入缓存，并设置一个较短的过期时间。这样后续请求相同的不存在数据时，可以直接从缓存中获取空值，避免频繁查询数据库。
    # 如果数据库中不存在该数据，缓存中存储一个空值
SET key "" EX 60  # 缓存空值1分钟

    1. 布隆过滤器（Bloom Filter）： 在请求到达缓存之前使用布隆过滤器来判断请求的数据是否存在。布隆过滤器是一个高效的概率性数据结构，可以快速判断某个数据是否一定不存在，但不能保证数据一定存在。通过布隆过滤器过滤掉不存在的数据请求，避免直接查询数据库。

    2. 布隆过滤器会在系统启动时根据已有的数据库记录进行初始化，将存在的数据ID加载到布隆过滤器中。

    3. 参数校验： 对请求参数做合法性校验，防止无效请求直接落到缓存或数据库。

    2. 缓存雪崩¶

    定义：¶

    缓存雪崩指的是大量缓存在同一时间失效，从而导致大量请求同时到达数据库，瞬间增加数据库的压力，甚至导致数据库崩溃。

    场景：¶

    假设系统中的大量缓存设置了相同的过期时间，且它们同时失效，那么在缓存失效后，所有查询都会绕过缓存直接请求数据库，瞬间给数据库带来巨大的压力。

    解决方案：¶

    1. 缓存数据的过期时间分散化： 设置缓存时，过期时间可以随机化或加一个随机的偏移值，避免大量缓存同时失效。
    # 设置过期时间时加一个随机偏移量，避免大量缓存同时失效
EXPIRE key EXPIRE_TIME + RANDOM(60)  # 多加60秒以内的随机时间

    1. 双缓存机制： 设置一个主缓存和一个备份缓存。当主缓存过期时，依旧允许访问备份缓存，从而减少数据库的压力。在备份缓存过期之前，可以异步更新主缓存。

    2. 热点数据预热： 在系统启动或大促活动等高峰期之前，预先加载部分热点数据到缓存中，避免在高峰期缓存集中失效导致的雪崩。

    3. 限流与降级： 在数据库压力增大的情况下，进行限流，保证数据库不会被瞬间压垮。此外，可以对某些非核心功能进行降级，减少对缓存和数据库的依赖。

    3. 缓存击穿¶

    定义：¶

    缓存击穿指的是某些热点数据在缓存中失效后，大量并发请求直接打到数据库。与缓存雪崩不同的是，缓存击穿通常是针对某个特别热点的键，缓存失效后，这个键的大量并发请求直接进入数据库，造成数据库压力过大。

    场景：¶

    某个热点数据（如商品详情页数据）被频繁访问，当缓存的过期时间到期后，突然大量请求同时查询该数据，因为缓存已经失效，这些请求直接打到数据库，可能导致数据库压力过大。

    解决方案：¶

    1. 互斥锁（锁机制）： 当一个请求发现缓存失效时，不直接查询数据库，而是通过加锁机制（如 SETNX）让第一个请求去加载数据并更新缓存，其他请求等待第一个请求加载完成后再从缓存中读取数据。
    # 获取锁，只有第一个请求能操作数据库
if (SETNX lock_key 1 EX 5):
    # 查询数据库并更新缓存
    SET cache_key value EX 3600
else:
    # 等待锁释放后查询缓存
    GET cache_key

    1. 热点数据永不过期： 对于特别热点的数据，可以考虑将其缓存设置为永不过期。这样就避免了缓存失效后瞬时大量并发请求打到数据库的问题。定期异步刷新缓存中的数据。

    2. 逻辑过期： 将缓存的过期时间控制在缓存值内部，缓存数据本身不设置过期时间，但标记一个逻辑上的过期时间。当数据被访问时，判断逻辑时间是否过期，过期则异步更新数据，但还是返回旧数据，避免数据库瞬时压力过大。

    总结

    • 缓存穿透：大量请求访问不存在的数据，导致每次都查询数据库。
    • 解决方案：缓存空值、使用布隆过滤器、参数校验。
    • 缓存雪崩：大量缓存同时失效，导致数据库承载大量请求。
    • 解决方案：设置不同的过期时间、双缓存机制、数据预热、限流与降级。
    • 缓存击穿：某个热点数据在缓存失效后，大量请求同时查询该数据，直接打到数据库。
    • 解决方案：互斥锁机制、热点数据永不过期、逻辑过期策略。

    三、Redis 高频面试题¶

    1. Redis 的五种数据类型是什么？¶

    • String（字符串）：最常用的数据类型，可以存储文本或二进制数据。
    • List（列表）：链表结构，支持左右两端的快速插入和删除。
    • Set（集合）：无序集合，支持去重和交集、并集等操作。
    • Hash（哈希）：键值对的集合，类似于 map 或 dict。
    • ZSet（有序集合）：带权重的有序集合。

    2. Redis 为什么这么快？¶

    • 基于内存：数据存储在内存中，读写速度极快。
    • 单线程模型：避免了多线程的上下文切换和竞争。
    • 非阻塞 I/O：使用 epoll 实现了高效的事件处理机制。

    3. Redis 持久化的方式有哪些？¶

    • RDB（Redis DataBase）：在指定的间隔时间内生成内存快照，保存到磁盘。
    • AOF（Append Only File）：记录每次写操作，将命令追加到日志文件中。

    4. Redis 的过期策略有哪些？¶

    • 定时删除：设置定时器，到期立即删除。
    • 惰性删除：访问键时检查是否过期，过期则删除。
    • 定期删除：周期性随机抽取一部分键进行过期检查和删除。

    5. Redis 分布式锁的实现原理是什么？¶

    • Redis 通过 SETNX（SET if Not eXists）命令实现分布式锁，结合 EXPIRE 命令设置锁的过期时间，防止死锁。
    • 常用操作:
    • redis-cli --scan --pattern "csrf_*" | xargs redis-cli del: 清除 Redis 中的 CSRF Token

    6. 什么是 Redis 的主从复制？¶

    • Redis 的主从复制是通过将数据从主节点复制到从节点，从而实现数据冗余和负载均衡。通过配置 slaveof 命令设置从节点。

    7. Redis 的哨兵模式是什么？¶

    • 哨兵模式是 Redis 的高可用解决方案，通过监控主节点的状态，自动完成主从切换，并通知客户端新的主节点。

    8. Redis 缓存穿透、缓存雪崩、缓存击穿的区别？¶

    • 缓存穿透：大量请求访问不存在的数据，导致每次都查询数据库。
    • 解决方案：缓存空值、使用布隆过滤器、参数校验。
    • 缓存雪崩：大量缓存同时失效，导致数据库承载大量请求。
    • 解决方案：设置不同的过期时间、双缓存机制、数据预热、限流与降级。
    • 缓存击穿：某个热点数据在缓存失效后，大量请求同时查询该数据，直接打到数据库。
    • 解决方案：互斥锁机制、热点数据永不过期、逻辑过期策略。
