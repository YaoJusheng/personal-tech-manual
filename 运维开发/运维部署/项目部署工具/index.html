    0.部署工具选择指南¶

    0.1 快速决策¶

    场景 推荐方案 说明
    单机临时运行 nohup / tmux 快速验证可用，不建议长期生产
    单机长期托管 systemd / supervisor 可控性和可靠性更好
    Python Web 服务 gunicorn + uvicorn worker 当前较主流 ASGI 生产组合
    多机应用发布 Ansible + CI/CD 标准化发布流程，易审计
    容器化规模部署 Kubernetes + Helm/GitOps 声明式、可回滚、易扩容

    0.2 常见组合¶

    • Web/API：nginx/traefik -> gunicorn/uvicorn -> app
    • 异步任务：supervisor/systemd -> celery/worker
    • 平台化：CI (Jenkins/GitHub Actions) -> 镜像仓库 -> Kubernetes

    0.3 最小上线标准¶

    1. 进程托管：异常退出可自恢复
    2. 反向代理：超时、限流、TLS 配置明确
    3. 可观测性：日志、指标、告警具备
    4. 回滚能力：版本可追踪，可快速回退
    5. 安全基线：最小权限、密钥不落盘

    0.4 分层部署思路¶

    flowchart LR
    Code[代码仓库] --> CI[CI 构建]
    CI --> Artifact[制品或镜像仓库]
    Artifact --> Deploy[部署层
    systemd/supervisor/K8S]
    Deploy --> Gateway[流量入口
    nginx/traefik]
    Gateway --> App[应用服务]
    App --> Obs[监控告警与日志]

    0.5 环境分级建议¶

    环境 目标 推荐策略
    dev 快速迭代 简化部署，保留最小监控
    staging 验证发布质量 尽量贴近生产配置
    prod 稳定与安全优先 灰度发布、严格权限、完整审计

    0.6 工具选型关键问题¶

    1. 服务规模是多少（单机/多机/跨地域）
    2. 是否需要弹性扩容与自愈
    3. 团队是否具备容器平台运维能力
    4. 是否有合规审计与回滚时效要求

    3.nohup/pm2示例¶

    3.1.nohup¶

    nohup python3 -u app.py > nohup.log &

    3.2.pm2¶

    3.2.1.简单命令行部署¶

    pm2 list
pm2 start app.py [--no-daemon] --name <alias> -x --interpreter python3 -o ./logs/out.log -e ./logs/error.log -i max
pm2 stop app_id
pm2 delete app_id
pm2 monit  # 监控项（cpu、内存）

    注：
    --name <alias>：起别名；
    -i：表示开启多进程，max由系统自动配置，可以指定具体进程数。

    3.2.2.配置文件部署¶

    • 首先，在项目下执行：pm2 init，生成 ecosystem.config.js 配置文件；
    • 修改配置文件：

    apps: [

            args: ' --port=8001',
            watch: true,
            ignore_watch: ['node_modules', 'build', 'logs'],
            // exec_mode: 'cluster',  // fork模式多进程无法复用端口, cluster模式js对python兼容差
            instances: 1,
            max_memory_restart: '1G',
            autorestart: true,

                "PORT": 81,

    1. nginx配置

    实现nginx会话保持:

    • ip_hash
    • sticky_cookie_insert（需要sticky模块）
    • jvm_route
    • 第一次请求过来，是没有带session信息的，jvm_route就根据round robin的方法，发到一台tomcat上面。
    • tomcat添加上session 信息，并返回给客户。
    • 用户再此请求，jvm_route看到session中有后端服务器的名称，它就把请求转到对应的服务器上。

    在 nginx.conf 文件中：

    worker_processes  4;
daemon off; # nginx不能以daemon模式启动
user nobody;

http {
    upstream fulllink_web {
        # 方法1
        ip_hash;

        server 127.0.0.1:8081;
        server 127.0.0.1:8082;
        server 127.0.0.1:8083;
        server 127.0.0.1:8084;

        # 方法2
        sticky_cookie_insert srv_id expires=1h domain=localhost path=/;
    }
    server {
        listen       8080;
        server_name  localhost;
        location / {
           proxy_set_header Host $http_host;
           proxy_set_header X-Real-IP $remote_addr;
           proxy_set_header X-Forwarded-For  $remote_addr;
           proxy_set_header X-Scheme $scheme;
           proxy_read_timeout 120s;
           proxy_pass http://fulllink_web;
        }
    }
}
    1. supervisor操作
    # ----------------- [supervisord 操作] -----------------
#1.启动：指定配置文件
supervisord -c /etc/supervisord.conf
#2.关闭 Supervisor
ps -ef | grep supervisord
kill -s SIGTERM pid
3.管理 Supervisor
#1）进入客户端
supervisorctl
#2）停止运行某个服务器进程
supervisor> stop fulllink_web:8001
#3）查看状态
supervisor> status
#4）停止运行整个tornado服务器进程组
supervisor> stop tornadoes:
#5）启动某个进程
supervisorctl start program_name
#6）重启某个进程
supervisorctl restart program_name
#7）停止全部进程
supervisorctl stop all
#8）载入最新的配置文件，停止原有进程并按新的配置启动、管理所有进程
supervisorctl reload
#9）启动新配置或有改动的进程
supervisorctl update

    7.Kubernetes 与 GitOps¶

    • 镜像构建后推送到镜像仓库（Harbor/ECR/GCR 等）
    • 使用 Helm 或 Kustomize 管理部署清单
    • 通过 Argo CD/Flux 从 Git 同步到集群，实现声明式发布与回滚

    8.工具选型建议¶

    • 新项目优先考虑容器化与声明式交付，降低环境差异成本
    • 存量项目可先从 supervisor/systemd + nginx 标准化，再逐步迁移到 Kubernetes
    • 不同环境（dev/staging/prod）建议使用同一套部署模板，仅覆盖变量

    9.发布策略与回滚方案¶

    9.1 发布策略¶

    策略 优点 成本 适用场景
    直接替换 流程简单 风险较高 低风险内部系统
    滚动发布 连续可用 回滚粒度一般 常规业务系统
    金丝雀发布 风险可控 依赖流量治理 核心线上服务
    蓝绿发布 回滚快 资源占用高 高可用系统

    9.2 回滚原则¶

    • 代码、配置、镜像版本都要可追踪。
    • 回滚动作尽量自动化，避免人工临场拼接命令。
    • 回滚后必须做健康检查和数据一致性校验。

    10.上线验收清单（通用）¶

    • 服务可用：健康检查通过，核心接口响应正常
    • 性能基线：关键接口延迟、错误率在目标阈值内
    • 安全基线：密钥托管、最小权限、对外端口最小暴露
    • 可观测性：日志、指标、告警三者齐全
    • 应急能力：回滚路径、值班人、故障 SOP 已明确

    11.常见问题与处理建议¶

    问题 典型原因 建议处理
    部署后服务起不来 配置遗漏或依赖缺失 增加启动前校验步骤
    发布过程超时 镜像拉取慢或探针配置不合理 预热镜像并优化超时参数
    回滚失败 版本资产不完整 强制执行制品与配置归档
    线上偶发 502/504 网关超时与上游负载不匹配 调整连接池、超时与扩容策略

    12.实践路线建议¶

    1. 先做部署标准化：统一目录、日志、配置模板
    2. 再做自动化：CI 打包、自动发布、自动回滚
    3. 最后做平台化：统一监控、统一审计、统一发布入口
