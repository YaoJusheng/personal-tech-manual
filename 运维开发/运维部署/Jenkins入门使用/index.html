    0.主流实践速览¶

    0.1 当前定位¶

    • Jenkins 仍是中大型企业里最常见的可自托管 CI/CD 平台之一。
    • 新项目常与 GitHub Actions、GitLab CI、Argo CD 组合使用：Jenkins 负责构建，GitOps 工具负责发布。

    0.2 推荐基线¶

    • 版本建议：优先使用 Jenkins LTS（长期支持版），并统一到 JDK 17 运行时。
    • 架构建议：Controller/Agent 分离，避免在 Controller 上直接跑构建任务。
    • Agent 建议：优先容器化临时 Agent（Kubernetes 或 Docker），减少脏环境问题。

    0.3 Pipeline 设计建议¶

    • Pipeline as Code：统一使用 Jenkinsfile，尽量避免在页面中手写脚本。
    • 共享能力：将通用逻辑沉淀为 Shared Library（如构建、制品上传、通知、质量门禁）。
    • 安全治理：凭据统一托管在 Credentials 中，禁止在脚本里硬编码账号密码。
    • 质量门禁：在构建后增加测试、SAST、依赖漏洞扫描、镜像扫描等步骤。

    0.4 学习顺序（建议）¶

    1. 自由风格任务（理解参数、环境变量、工作空间）
    2. Pipeline 语法（Declarative 为主）
    3. 多分支流水线（按分支自动触发）
    4. 凭据、共享库与权限模型
    5. 与容器平台、制品库、部署平台联动

    0.5 术语说明¶

    • 本文保留了若干可直接复用的历史示例脚本（含 apt/pip/compile 流程）。
    • 示例中的镜像、插件、凭据 ID、仓库地址请按实际环境替换。

    0.6 企业级最小架构（建议）¶

    flowchart LR
    DEV[开发者提交代码] --> SCM[GitHub/GitLab/Bitbucket]
    SCM --> MBP[Multibranch Pipeline]
    MBP --> AGENT[构建 Agent
    Docker/Kubernetes]
    AGENT --> TEST[单元测试/集成测试]
    TEST --> SCAN[代码与依赖扫描]
    SCAN --> IMAGE[镜像/制品构建]
    IMAGE --> REPO[制品库
    Nexus/Harbor]
    REPO --> DEPLOY[部署系统
    Helm/Argo CD]
    DEPLOY --> OBS[监控与告警]

    0.7 流水线分层思路¶

    • 触发层：Webhook、定时、手工触发，统一进入 Jenkinsfile。
    • 编译层：语言依赖安装、构建、测试、静态检查。
    • 制品层：镜像或压缩包生成、签名、上传制品仓库。
    • 发布层：由 Jenkins 触发部署工具，生产环境建议加人工审批。
    • 观测层：构建耗时、成功率、失败类型、回滚频率可视化。

    0.8 阶段门禁建议（可直接套用）¶

    阶段 目标 必做检查 失败处理
    Build 产出可部署包 依赖完整、构建成功 失败即终止
    Test 保证功能稳定 单测覆盖、关键集成测试 禁止进入下阶段
    Security 降低供应链风险 SAST、依赖漏洞、镜像扫描 记录缺陷并阻断高危
    Package 固化版本资产 版本号、签名、元数据 清理不完整产物
    Deploy 安全发布 变更审批、灰度策略、回滚预案 自动触发回滚流程

    0.9 凭据与权限治理建议¶

    • 凭据最小暴露：仅在需要的 stage 注入，避免全局环境泄漏。
    • 权限分离：管理员、流水线维护者、只读查看者分离。
    • 审计可追踪：谁在何时触发了哪次发布，必须可查询。
    • 禁止明文密钥：密码、Token、证书统一由 Jenkins Credentials 管理。

#!/bin/bash

# /* Prepare install */
function preinstall() {
    sed -i 's/deb.debian.org/mirrors.aliyun.com/g' /etc/apt/sources.list \
    && sed -i 's/security.debian.org/mirrors.aliyun.com/g' /etc/apt/sources.list \
    && apt-get update -y > /dev/null\
    && apt-get --no-install-recommends install -y \
        build-essential \
        bzip2 \
        clang \
        coreutils \
        gcc \
        shc \
        libgl1-mesa-glx \
        libmysqlclient-dev \
        libglib2.0-0 \
        libglib2.0-dev \
        libpcre3 \
        libpcre3-dev \
        libssl-dev \
        libwww-perl \
        python3-dev \
        python3-pip \
        python3-venv  > /dev/null \
    && rm -rf /var/lib/apt/lists/*

    pip3 install --no-cache-dir --upgrade pip wheel setuptools > /dev/null\
    && pip3 install --no-cache-dir -r requirements.txt  > /dev/null\
    && pip3 install --no-cache-dir PyJWT==2.3.0 nuitka==1.9.2 > /dev/null
}

# /* Compile module with nuitka */
function _compile_nuitka() {
    modules=("app" "migrates")
    singles=("config.py" "runserver.py")

    # TODO: (BE) If binary file needed for deployment, please uncomment the above.
    #  Module compilation is used here for more flexibility and convenience.

    compile_opts=" --quiet \
    --show-progress \
    --show-modules \
    --warn-unusual-code \
    --warn-implicit-exceptions \
    --python-flag=-S,-O,no_docstrings \
    --plugin-enable=pylint-warnings \
    --disable-ccache \
    --nofollow-imports \
    --remove-output \
    --no-pyi-file \
    --clang "

    # Compile by module
    for item in "\${modules[@]}"
    do
        echo -e "Start to compile module: \${item}"
        command="python3 -m nuitka --module \${item} --include-package=\${item} \${compile_opts}"
        eval "\${command}"
        mv "\${item}".*.so "\${item}".so
        rm -rf "\${item}"
        awk 'BEGIN {for(i=0;i<88;i++) printf "—"}' && echo ""
    done

    for file in "\${singles[@]}"
    do
        echo -e "Start to compile file: \${file}"
        command="python3 -m nuitka --module \${file} \${compile_opts}"
        eval "\${command}"
        mv "\${file%%.*}".*.so "\${file%%.*}".so
        rm "\${file}"
        awk 'BEGIN {for(i=0;i<88;i++) printf "—"}' && echo ""
    done
}

# /* Build and compile the privatization. */
function private_compile() {
    private_version=${FLASKX_PRIVATE_VERSION:-"MVP-beta"}
    srd=\${PWD}

    echo -e "Prepare for compilation: \n\t\${srd}"
    awk 'BEGIN {for(i=0;i<88;i++) printf "—"}' && echo ""
    [ -f "\${srd}/FlaskxCloud_\${private_version}.tar.bz2" ] && rm "\${srd}/FlaskxCloud_\${private_version}.tar.bz2"

    # Compile with Nuitka
    _compile_nuitka

    [ -d static ] || mkdir static
    curl ftp://180.167.218.82:201/pub/flaskx/flaskx_cloud/fe_assets/"${FLASKX_FE_VERSION:-"mvp"}"/dist.zip -u ftp_example:ftp_example -o ./static/dist.zip

    echo -e "Clear files and dirs..."
    awk 'BEGIN {for(i=0;i<88;i++) printf "—"}' && echo ""
    mv uwsgi_template.ini uwsgi.ini
    mv PRIVATE.md README.md
    rm todo ./*.tmpl src/*.whl  || echo ""
    rm -rf src/flaskx_serving src/flaskx_worker

    # Compile sh scripts. (manage)
    exp_msg="Please contact your provider: <jusheng.yao@example.com>"
    # shellcheck disable=SC2015
    [ -z "${FLASKX_PRIVATE_DATE}" ] && shc -r -f manage -o manage || shc -r -e "${FLASKX_PRIVATE_DATE}" -m "\${exp_msg}" -f manage -o manage
    rm ./*.c

    echo -e "Packaging and privately deploy code..."
    awk 'BEGIN {for(i=0;i<88;i++) printf "—"}' && echo ""
    # Mirror resource packaging to be determined.
    tar -jcf "FlaskxCloud_\${private_version}.tar.bz2" --exclude='compile' --exclude='deploy' --exclude='test' ./*  || echo ""
    echo -e "Packaged and compiled successfully."
}

preinstall || exit 1
private_compile && exit 0 || exit 1

    3.生产发布 Runbook¶

    3.1 一次标准发布流程¶

    1. 代码合并到目标分支并触发流水线
    2. 构建与测试通过后生成制品并归档
    3. 通过审批节点进入部署阶段
    4. 先发布到 staging 验证，再执行生产灰度
    5. 观测核心指标 15-30 分钟后完成全量

    3.2 回滚策略建议¶

    • 应用回滚：回退到上一个稳定制品（镜像 tag/构建号）。
    • 数据回滚：数据库变更采用前向兼容优先，避免强依赖逆向迁移。
    • 配置回滚：发布配置与代码同版本管理，支持一键回退。

    3.3 常见故障与处理¶

    现象 常见原因 排查重点 处理建议
    构建随机失败 Agent 环境不一致 基础镜像、依赖缓存 固定镜像版本，启用预热缓存
    拉取代码失败 凭据或网络问题 凭据权限、SCM 访问策略 更新凭据并验证最小权限
    部署后异常 配置不一致 环境变量、配置中心、密钥 发布前执行配置校验任务
    流水线变慢 串行步骤过多 stage 耗时、队列等待 拆分并行 stage，优化节点池

    3.4 可观测性指标建议¶

    • 构建成功率（按天/按周）
    • 平均构建时长与 P95 构建时长
    • 发布失败率与回滚率
    • MTTR（故障平均恢复时间）
