    0.阅读指引¶

    本文包含 DevOps、云计算与 OpenStack 的基础材料。为便于快速学习，建议按以下顺序阅读：

    1. 先看第 3 章 OpenStack 核心能力与组件
    2. 再回看第 2 章云计算服务模型（IaaS/PaaS/SaaS）
    3. 最后结合第 1 章 DevOps 工具链理解工程落地

    0.1 OpenStack 当前定位¶

    • OpenStack 仍是私有云和行业云（政企、运营商、科研）中主流 IaaS 方案之一。
    • 在现代平台中，常见形态为 OpenStack 提供资源底座 + Kubernetes 提供应用编排。
    • 典型目标：多租户隔离、可审计合规、资源池化与统一运维。

    0.2 生产环境常见组合¶

    • 计算虚拟化：KVM 为主，按需混合裸金属（Ironic）
    • 网络：Neutron + OVN/SDN 方案
    • 存储：Cinder（块）+ Swift/Ceph（对象）
    • 身份与权限：Keystone 对接企业身份体系（OIDC/LDAP/AD）
    • 监控告警：Prometheus/Grafana/集中日志平台

    0.3 建设建议¶

    • 优先确定可用区、网络平面、存储策略，再做服务编排
    • 把容量规划与故障演练纳入日常，不仅关注“能部署”
    • API 优先，UI 次之：自动化运维与审计会更可持续

    0.4 OpenStack + Kubernetes 常见分工¶

    层级 OpenStack 负责 Kubernetes 负责
    资源层 计算、网络、存储资源池化 使用资源运行业务容器
    租户层 多租户、项目隔离、配额治理 命名空间、RBAC、网络策略
    交付层 提供 IaaS 能力接口 应用编排、灰度发布、弹性扩缩
    运维层 主机、虚拟网络、卷与镜像治理 应用健康、服务拓扑、版本回滚

    0.5 资源交付链路（简图）¶

    flowchart LR
    Tenant[租户申请资源] --> Keystone[Keystone 认证授权]
    Keystone --> Nova[Nova 分配计算]

    1.DevOps简介¶

    DevOps 是一个完整的面向IT运维的工作流，以 IT 自动化以及持续集成（CI）、持续部署（CD）为基础，来优化程式开发、测试、系统运维等所有环节。

    【DevOps的概念】 DevOps一词的来自于Development和Operations的组合，突出重视软件开发人员和运维人员的沟通合作，通过自动化流程来使得软件构建、测试、发布更加快捷、频繁和可靠。

    DevOps是为了填补开发端和运维端之间的信息鸿沟，改善团队之间的协作关系。不过需要澄清的一点是，从开发到运维，中间还有测试环节。DevOps其实包含了三个部分：开发、测试和运维。

    实现DevOps需要什么？ 工具链如下：

    • 代码管理（SCM）：GitHub、GitLab、BitBucket、SubVersion(SVN)
    • 构建工具：Ant、Gradle、maven
    • 自动部署：Capistrano、CodeDeploy
    • 持续集成（CI）：Bamboo、Hudson、Jenkins
    • 配置管理：Ansible、Chef、Puppet、SaltStack、ScriptRock GuardRail
    • 容器：Docker、LXC、第三方厂商如AWS
    • 编排：Kubernetes、Core、Apache Mesos、DC/OS
    • 服务注册与发现：Zookeeper、etcd、Consul
    • 脚本语言：python、ruby、shell
    • 日志管理：ELK、Logentries、SLS
    • 系统监控：Datadog、Graphite、Icinga、Nagios
    • 性能监控：AppDynamics、New Relic、Splunk
    • 压力测试：JMeter、Blaze Meter、loader.io
    • 预警：PagerDuty、pingdom、厂商自带如AWS SNS
    • HTTP加速器：Varnish
    • 消息总线：ActiveMQ、SQS
    • 应用服务器：Tomcat、JBoss
    • Web服务器：Apache、Nginx、IIS
    • 数据库：MySQL、Oracle、PostgreSQL等关系型数据库；cassandra、mongoDB、redis等NoSQL数据库
    • 项目管理（PM）：Jira、Asana、Taiga、Trello、Basecamp、Pivotal Tracker、禅道、PingCode、Worktile

    1. 云计算的优势

    2. 按需自助服务

    4. 资源池化
    5. 可计量服务

    6. 快速弹性伸缩

    7. 定义 云计算是一种模型，一种按使用量付费的模式，这种模式可以通过互联网实现随时随地、便捷地、随需应变地从可配置计算资源共享池中获取所需的资源（包括：计算、网络、服务器、存储、应用软件和服务），资源能够快速供应并释放，使管理资源的工作量和服务提供商的交互减少到最低限度。

    8. 特征

    9. 易于管理：无需基础设施维护

    10. 高灵活度：可伸缩，无地域限制
    11. 高资源利用率：负载平衡，资源共享
    12. 高可用性：服务质量保证，异地容灾，
    13. 低成本：无前期投入，按需购买

    14. 安全性：统一集中式数据管理，多种密钥技术保护

    15. 云计算服务模式

    16. SaaS --（ Software as a Service）：软件即服务，侧重于服务，通过网络提供软件程序服务.

    17. PaaS --（Platform as a Service）：平台即服务，侧重于服务，以服务平台或者开发环境提供服务.
    18. IaaS --（Infrastructure as a Service）：基础设施服务即服务，注重计算资源共享，消费者通过Internet可以从完善的计算机基础设施获得服务.

    1. 云计算中的不同部署模型

    2. 私有云：云端资源所有的服务不是供别人使用，而是供自己内部人员或分支机构使用。私有云可充分保障虚拟化私有网络的安全，私有云的部署比较适合于有众多分支机构的大型企业或政府部门。

    3. 公有云：云端资源开放给社会公众使用。云端的所有权、日常管理和操作的主体可以是一个商业组织、学术机构、政府部门或者它们其中的几个联合。如：微软的Windows Azure Platform、 亚马逊的AWS、Salesforce .com，以及国内的阿里巴巴、用友伟库等。
    4. 社区云：云端资源专门给固定的几个单位内的用户使用，而这些单位对云端具有相同诉求（如安全要求、云端使命、规章制度、合规性要求等）。

    5. 混合云：由两个或两个以上不同类型的云（私有云、社区云、公共云）组成，它们各自独立，但用标准的或专有的技术将它们组合起来，而这些技术能实现云之间的数据和应用程序的平滑流转。目前绝大多数混合云由企事业单位主导，以私有云为主体，并融合部分公共云资源，就是说，混合云的消费者主要来自一个或几个特定的单位组织。

    6. 云计算中弹性与可扩展性的区别 弹性：首先，整合计算资源，将计算资源池化，通过虚拟机按需使用计算资源；其次，按量计费，让用户能够根据使用量按月按时甚至按秒来进行付费。强调启用和停用庞大的资源容量这一概念。 可扩展性：只要相应增加资源容量，就可以处理增加的工作负载。用户可以利用应用软件的快速部署条件来更为简单快捷的将自身所需的已有业务以及新业务进行扩展。

    3.OpenStack简介¶

    术语说明：本章中的部分表述属于入门化说明，实际生产部署请结合发行版文档与目标版本发布说明。

    OpenStack是一个开源的云计算管理平台项目，是一系列软件开源项目的组合。由NASA(美国国家航空航天局)和Rackspace合作研发并发起，以Apache许可证（Apache软件基金会发布的一个自由软件许可证）授权的开源代码项目。

    目前最流行的开源操作系统内核。OpenStack是IaaS(基础设施即服务)组件，让任何人都可以自行建立和提供云端运算服务。

    3.1.OpenStack概述¶

    1. 资源抽象：OpenStack将各类硬件资源，通过虚拟化与软件定义的方式，抽象成资源池。
    2. 资源分配与负载调度：OpenStack根据管理员/用户的需求，将资源池中的资源分配给不同的用户，承载不同的应用。
    3. 应用声明周期管理：OpenStack已经可以提供初步的应用部署/撤除、自动规模调整能力。
    4. 系统运维：OpenStack已经可以提供一定的系统监控能力。
    5. 人机交互：OpenStack提供人机接口，外界可以通过API、命令行或图形界面的方式参与OpenStack进行交互。

    3.2.OpenStack架构¶

    OpenStack 是一个开源云计算操作系统，通过一个数据中心管理大量的计算、存储、网络资源（三大核心组件）。管理员通过dashboard高效地管控这些资源，用户则通过web接口使用这些资源。

    OpenStack支持多种类型的云环境，试图打造一个部署简单、操作灵活、可大规模扩展、标准统一的云计算管理平台。OpenStack通过各种各样的服务灵活地提供基础设施即服务(IaaS)解决方案，并且每种服务都提供了API接口以便更加灵活的集成。

    OpenStack的服务主要包括：

    服务 项目名称 描述
    Dashboard Horizon UI模块。提供了一个基于web的自主服务门户，与OpenStack底层服务交互，诸如启动一个实例，分配IP地址以及配置访问控制。
    Computer Nova 计算服务组件。在OpenStack环境中计算实例的生命周期管理。按需响应包括生成、调度、回收虚拟机等操作。
    Network Neutron 网络服务组件。确保为其它OpenStack服务提供网络连接即服务，比如OpenStack计算。为用户提供API定义网络和使用。基于插件的架构其支持众多的网络提供商和技术。
    Objecet Storage Swift 对象存储服务模块。通过一个 RESTful,基于HTTP的应用程序接口存储和任意检索的非结构化数据对象。它拥有高容错机制，基于数据复制和可扩展架构。它的实现并像是一个文件服务器需要挂载目录。在此种方式下，它写入对象和文件到多个硬盘中，以确保数据是在集群内跨服务器的多份复制。
    Block Storage Cinder 块存储服务模块。为运行实例而提供的持久性块存储。它的可插拔驱动架构的功能有助于创建和管理块存储设备。
    Identity Service Keystone 身份服务模块。为其他OpenStack服务提供认证和授权服务，为所有的OpenStack服务提供一个端点目录。
    Image Service Glance 镜像服务组件。存储和检索虚拟机磁盘镜像，OpenStack计算会在实例部署时使用此服务。
    Telemetry Service Ceilmeter 监控服务组件。为OpenStack云的计费、基准、扩展性以及统计等目的提供监测和计量。
    Orchestration Service Heat 部署编排。既可以使用本地模板格式，亦可使用AWS CloudFormation模板格式，来编排多个综合的云应用，通过OpenStack本地REST API或者是CloudFormation相兼容的队列API。

    3.3.KVM简介¶

    kvm (开放虚拟化技术) 是虚拟化技术的一个典型实现，功能非常强大，使用很方便。kvm是基于内核的虚拟机， 本身主要实现对 CPU 的虚拟化，内存和IO的虚拟化使用了开源软件 qemu，qemu 是纯软件层面的虚拟化，其实就是个模拟器。kvm 要求 cpu 必须支持硬件虚拟化，可以通过 Linux 命令查看。

    OpenStack与KVM相互辉映。OpenStack几乎支持所有的虚拟化管理程序，不论是开源的（Xen与KVM）还是厂商的（Hyper-V与VMware）。但在以前，OpenStack是基于KVM开发的，KVM常常成为默认的虚拟机管理程序。两者都使用相同的开放源理念与开发方法。

    3.3.1.对比¶

    • Vmware的功能全面，设置全面，速度相对最慢；
    • VirtualBox的效率比Vmware高一些，中文用户最多；
    • KVM整体效率最高。

    3.3.2.系统要求¶

    检查/proc/cpuinfo，如果在cpu flags字段看到了vmx(Intel)或svm(AMD)，那么你的处理器就支持KVM。

    # 查看
egrep '(vmx|svm)' /proc/cpuinfo
# 获得KVM (ubuntu为例)
apt://qemu-kvm

    3.3.3.安装和配置 kvm 环境¶

    1. 安装 kvm 虚拟化相关软件包
    [root@kvm ~]# yum install -y kvm virt-* libvirt bridge-utils qemu-img
    1. 查看 kvm 模块是否加载到内核
    [root@kvm ~]# lsmod | grep kvm_intel
kvm_intel              53484  0
kvm                   316506  1 kvm_intel

    4.生产部署建议¶

    4.1 控制平面高可用¶

    • API 服务建议至少双实例并接入负载均衡。
    • 数据库和消息队列采用高可用部署，避免单点。
    • 关键组件升级前先在预生产演练。

    4.2 网络规划建议¶

    • 提前拆分管理网络、业务网络、存储网络。
    • 明确南北向与东西向流量路径，避免跨网段绕行。
    • 核心网络策略保持最小开放原则并可审计。

    4.3 存储与备份建议¶

    • 镜像、卷、对象存储分别制定容量与备份策略。
    • 关键业务卷开启定期快照并验证恢复时效。
    • 备份要有恢复演练，不能只做“备份成功”检查。

    5.运维指标与容量治理¶

    5.1 建议持续观测的指标¶

    • 计算：虚拟机启动成功率、调度耗时、超卖比
    • 网络：网络创建成功率、南北向延迟、丢包率
    • 存储：卷创建时延、IOPS、容量利用率
    • 平台：API 错误率、消息堆积、数据库连接占用

    5.2 容量治理建议¶

    1. 建立项目配额基线，避免租户抢占资源
    2. 预测 1-3 个月增长量，提前准备扩容窗口
    3. 大促或峰值场景前进行压测与应急演练

    6.常见故障与排查思路¶

    故障现象 可能原因 首查方向
    实例创建失败 调度资源不足、镜像异常 Nova 调度日志、配额与镜像状态
    网络不通 路由/安全组/网络节点异常 Neutron 网络拓扑与策略
    卷挂载失败 存储后端异常或权限问题 Cinder 服务状态与后端连通
    API 请求报错率上升 控制平面压力过高 API 网关、数据库、消息队列负载

    7.实施路径建议¶

    • 第 1 阶段：完成最小可用私有云（计算+网络+存储）
    • 第 2 阶段：补齐监控、告警、审计、备份恢复
    • 第 3 阶段：与 Kubernetes 或 PaaS 平台联动，形成统一交付体系
