    1.Nginx¶

    官网: Nginx文档

    Nginx 是一个高性能的 HTTP 和反向代理服务器，它常用于负载均衡、缓存、静态资源服务以及反向代理等场景。它的配置文件采用模块化结构，核心配置文件通常位于 /etc/nginx/nginx.conf，其中定义了全局设置、HTTP 服务配置、事件处理等内容。下面是 Nginx 配置的详解。

    1. 基本配置结构¶

    Nginx 的配置文件主要由以下三部分组成：

    • 全局块（Global block）：定义与整个 Nginx 服务相关的设置，如工作进程数量、用户和组权限等。
    • 事件块（Events block）：设置 Nginx 如何处理连接，如最大连接数、I/O 模型等。
    • HTTP 块（HTTP block）：配置 HTTP 服务相关的内容，包含多个服务器块，每个服务器块可以处理不同的域名或请求。
    # 全局块
user nginx;
worker_processes auto;
error_log /var/log/nginx/error.log warn;
pid /var/run/nginx.pid;

# 事件块
events {
    worker_connections 1024;
}

# HTTP 块
http {
    include       /etc/nginx/mime.types;
    default_type  application/octet-stream;

    log_format  main  '$remote_addr - $remote_user [$time_local] "$request" '
                      '$status $body_bytes_sent "$http_referer" '
                      '"$http_user_agent" "$http_x_forwarded_for"';

    access_log  /var/log/nginx/access.log  main;

    sendfile        on;
    tcp_nopush      on;
    keepalive_timeout  65;
    gzip  on;

    # 服务器块
    server {
        listen       80;
        server_name  localhost;

        location / {
            root   /usr/share/nginx/html;
            index  index.html index.htm;
        }

        error_page  500 502 503 504  /50x.html;
        location = /50x.html {
            root   /usr/share/nginx/html;
        }
    }
}

    2. 全局块¶

    • user：指定 Nginx 运行的用户和组。默认是 nginx 用户。
    • worker_processes：设置 Nginx 的工作进程数，可以设置为 auto，根据 CPU 核数自动调整。
    • error_log：指定 Nginx 的错误日志路径和日志级别（例如 warn, error, info 等）。
    • pid：设置 Nginx 的 PID 文件路径，用于存储主进程的 PID。

    3. 事件块¶

    • worker_connections：每个工作进程能够同时处理的最大连接数。与工作进程数量相关，最大连接数为 worker_processes * worker_connections。
    • multi_accept：开启后，工作进程可以同时接收多个连接请求。

    4. HTTP 块¶

    HTTP 块用于配置 HTTP 相关的全局设置和服务。

    • include：包含其他文件（如 mime.types）来指定文件类型。

    • log_format：定义日志格式，可以自定义字段，例如 remote_addr（客户端 IP）、time_local（请求时间）、status（状态码）等。

    • access_log：访问日志的路径和格式。

    • sendfile：开启后，Nginx 使用 sendfile 系统调用，优化文件发送的效率。

    • tcp_nopush：优化 TCP 性能，通常与 sendfile 配合使用。

    • keepalive_timeout：设置客户端连接的超时时间，通常设置为 60-65 秒。

    • gzip: 启用 Gzip 压缩以减少传输内容大小，提高页面加载速度。

    gzip on;
gzip_types text/plain text/css application/json application/javascript text/xml application/xml;
    • access_log 和 error_log: 配置访问日志和错误日志。
    access_log /var/log/nginx/access.log;
error_log /var/log/nginx/error.log;
    • client_max_body_size: 设置客户端请求体的最大大小，默认 1MB，用于限制文件上传大小。
    client_max_body_size 10M;

    5. 服务器块¶

    server 块用于定义虚拟主机，每个 server 块可以处理不同的域名或端口请求, 通常包含监听端口、域名、SSL 配置等。

    • listen: 定义服务器监听的端口，如 80（HTTP）或 443（HTTPS）。
    server {
    listen 80;
    server_name example.com;
}
    • server_name: 指定服务器的域名，如 example.com 或 localhost。支持通配符和正则表达式。
    server_name example.com *.example.org;
    • ssl: 用于 HTTPS 配置，开启 SSL 支持。
    listen 443 ssl;
ssl_certificate /etc/ssl/certs/example.com.crt;
ssl_certificate_key /etc/ssl/private/example.com.key;
    • location：定义如何处理不同路径的请求。/ 表示根路径。location 可以支持正则匹配和优先级匹配。
    • root：指定请求资源的根目录。
    • index：指定默认加载的文件，如 index.html。

    • 可以指定不同的处理规则：

      • location /: 匹配所有路径的请求，通常用于网站根路径。
      location / {
    root /var/www/html;
    index index.html index.htm;
}
      • location = /favicon.ico: 精确匹配路径，例如 favicon.ico。
      location = /favicon.ico {
    access_log off;
    log_not_found off;
}
      • location ~ .php$: 使用正则表达式匹配以 .php 结尾的路径，用于 PHP 解析。
      location ~ \.php$ {
    fastcgi_pass 127.0.0.1:9000;
    fastcgi_index index.php;
    include fastcgi_params;
}

    • 错误页面配置：

    • error_page：定义错误页面。例如，当服务器返回 500-504 状态码时，返回 /50x.html 页面。

    location 匹配规则

    location 用于匹配不同的 URL 路径，并根据路径采取不同的处理策略。Nginx 提供了几种常见的匹配方式：

    • 精确匹配：location = / 精确匹配根路径 /。
    • 前缀匹配：location /images/ 匹配以 /images/ 开头的路径。
    • 正则匹配：location ~* \.(jpg|png|gif)$ 使用正则表达式匹配图片扩展名，~* 表示忽略大小写。

    6. 反向代理配置¶

    Nginx 可以用作反向代理，将客户端请求转发到后端服务器。

    server {
    listen 80;
    server_name example.com;

    location / {
        proxy_pass http://backend_server;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
    • proxy_pass：将请求转发到后端服务器 http://backend_server。可以是一个 IP 或域名。
    • proxy_set_header：设置请求头，向后端服务器传递一些客户端信息（如原始 IP、请求的协议等）。
    • proxy_set_header Host $host;:
      • 作用：将原始请求的主机名（域名或IP）传递给后端。
      • 解释：Nginx 会将 $host 变量的值设置为客户端请求的 Host 头值（即用户访问的域名）。
      • 用途：确保后端服务器识别请求的原始域名或IP，通常在需要根据请求的主机名进行处理时有用（如多站点支持）。
    • proxy_set_header X-Real-IP $remote_addr;:
      • 作用：将客户端的真实 IP 地址传递给后端。
      • 解释：Nginx 将客户端的 IP 地址（即 $remote_addr）写入 X-Real-IP 头，以便后端服务器识别客户端的来源 IP。
      • 用途：后端服务器可以记录和识别用户的真实 IP 地址，便于日志记录和追踪客户端来源。
    • proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;:
      • 作用：保留请求的整个转发链，记录所有代理服务器的 IP 地址。
      • 解释：X-Forwarded-For 头的值是一个客户端 IP 地址链，表示经过的所有代理服务器的 IP 地址。使用 $proxy_add_x_forwarded_for 变量会在现有 X-Forwarded-For 值基础上添加当前客户端的 IP。
      • 用途：在经过多级代理的环境中记录整个代理链，使后端服务器能够识别请求的完整来源路径。
    • proxy_set_header X-Forwarded-Proto $scheme;:
      • 作用：传递客户端的原始请求协议（HTTP 或 HTTPS）。
      • 解释：$scheme 是 Nginx 内置变量，表示请求所用的协议（http 或 https）。通过 X-Forwarded-Proto 头将协议传递给后端。
      • 用途：后端应用程序可以根据协议做出相应的处理，通常用于生成绝对 URL 或确定是否在 HTTPS 下进行额外安全处理。

    7. 负载均衡配置¶

    Nginx 还支持多台服务器的负载均衡。

    http {
    upstream backend {
        server backend1.example.com;
        server backend2.example.com;
    }

    server {
        listen 80;
        server_name example.com;

        location / {
            proxy_pass http://backend;
        }
    }
}
    • upstream：定义多个后端服务器，Nginx 会将请求负载均衡到这些服务器。
    • proxy_pass：与负载均衡结合使用，将请求转发到 upstream 定义的后端。

    Nginx 支持的负载均衡策略有：

    • 轮询（Round-robin）：默认策略，按顺序将请求依次分配给后端服务器。
    • 最少连接数（least_conn）：将请求分配给连接数最少的后端服务器。
    • IP 哈希（ip_hash）：根据客户端 IP 进行哈希，使来自同一 IP 的请求始终分配到同一个服务器。

    8. HTTPS 配置¶

    Nginx 可以配置 HTTPS 来处理 SSL/TLS 加密。

    server {
    listen 443 ssl;
    server_name example.com;

    ssl_certificate /etc/nginx/ssl/example.com.crt;
    ssl_certificate_key /etc/nginx/ssl/example.com.key;

    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;

    location / {
        root /usr/share/nginx/html;
        index index.html index.htm;
    }
}
    • ssl_certificate 和 ssl_certificate_key：指定 SSL 证书和私钥文件。
    • ssl_protocols：指定允许的 SSL/TLS 协议版本。
    • ssl_ciphers：设置 SSL/TLS 加密套件。
    • ssl_session_cache 和 ssl_session_timeout：启用会话缓存以减少握手延迟。

    9. HTTP2 配置¶

    9.1. 基本 HTTP/2 配置¶

    在 HTTP 和 HTTPS 配置中，都可以通过在 listen 指令后添加 http2 参数来启用 HTTP/2。例如：

    server {
    listen 80 http2;
    listen 443 ssl http2;  # 仅在 HTTPS 上有效的 HTTP/2 配置

    server_name example.com;

    location / {
        root /var/www/html;
        index index.html;
    }
}

    9.2. HTTPS 下的 HTTP/2 配置（推荐）¶

    由于 HTTP/2 最常在 HTTPS 上启用，以下是完整的 HTTPS 配置示例。

    server {
    listen 443 ssl http2;  # 启用 SSL 和 HTTP/2
    server_name example.com;

    # SSL 证书和私钥路径
    ssl_certificate /etc/ssl/certs/example.com.crt;
    ssl_certificate_key /etc/ssl/private/example.com.key;

    # 启用 HTTP/2 和 SSL 的推荐设置
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;
    ssl_prefer_server_ciphers on;

    # 会话缓存，减少 SSL 握手时间
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 1d;

    # HSTS 头，防止降级攻击
    add_header Strict-Transport-Security "max-age=31536000" always;

    location / {
        root /var/www/html;
        index index.html;
    }
}

    9.3. HTTP 下的 HTTP/2 配置（非加密）¶

    尽管 HTTP/2 通常在 HTTPS 下使用，也可以在 HTTP 下启用，但这种方式不支持浏览器的 HTTP/2，因此主要用于内部服务：

    server {
    listen 80 http2;
    server_name example.com;

    location / {
        root /var/www/html;
        index index.html;
    }
}

    9.4. 完整的 HTTP 和 HTTPS 支持 HTTP/2 配置示例¶

    以下是同时支持 HTTP 和 HTTPS 的 HTTP/2 配置示例：

    # 全局配置
http {
    include       mime.types;
    default_type  application/octet-stream;

    sendfile        on;
    keepalive_timeout  65;

    # gzip 压缩，提升传输效率
    gzip on;
    gzip_types text/plain text/css application/json application/javascript text/xml application/xml application/xml+rss;

    server {
        # HTTP 配置
        listen 80 http2;
        server_name example.com;

        location / {
            root /var/www/html;
            index index.html;
        }
    }

    server {
        # HTTPS 配置
        listen 443 ssl http2;
        server_name example.com;

        ssl_certificate /etc/ssl/certs/example.com.crt;
        ssl_certificate_key /etc/ssl/private/example.com.key;

        # SSL 推荐配置
        ssl_protocols TLSv1.2 TLSv1.3;
        ssl_ciphers HIGH:!aNULL:!MD5;
        ssl_prefer_server_ciphers on;
        ssl_session_cache shared:SSL:10m;
        ssl_session_timeout 1d;
        add_header Strict-Transport-Security "max-age=31536000" always;

        location / {
            root /var/www/html;
            index index.html;
        }
    }
}

    关键配置解析:

    • listen 443 ssl http2：在 HTTPS 的 listen 指令上添加 http2 启用 HTTP/2。
    • ssl_protocols 和 ssl_ciphers：确保使用安全协议和密码套件，避免使用较弱的加密标准。
    • add_header Strict-Transport-Security：设置 HSTS 头，强制客户端仅通过 HTTPS 访问，增强安全性。
    • ssl_session_cache 和 ssl_session_timeout：启用 SSL 会话缓存，减少握手延迟。

    注意事项:

    • 浏览器支持：大部分浏览器仅在 HTTPS 下支持 HTTP/2，因此在生产环境中优先使用 HTTPS 配置。
    • HTTP/2 与 HTTP/1.1 的兼容性：Nginx 会自动回退到 HTTP/1.1 以兼容不支持 HTTP/2 的客户端。

    10. 性能优化配置¶

    10.1.worker_processes & worker_connections¶

    这两个参数直接影响并发性能，通常配置为 CPU 核心数和单进程最大连接数：

    worker_processes auto;  # 自动匹配 CPU 核心数
worker_connections 2048;  # 每个 worker 可处理的最大连接数

    10.2.keepalive_timeout¶

    定义客户端连接的空闲超时时间，避免空闲连接占用资源。值过低会增加重连次数，值过高可能增加资源占用。

    keepalive_timeout 65;

    10.3.sendfile¶

    用于加速静态文件传输，通常推荐启用：

    sendfile on;

    10.4.tcp_nopush 和 tcp_nodelay¶

    用于优化传输的 TCP 性能：

    • tcp_nopush：合并包发送（减少 TCP 报文数）。
    • tcp_nodelay：在请求开始后立即发送数据包，减少延迟。
    tcp_nopush on;
tcp_nodelay on;

    10.5.gzip¶

    启用 Gzip 压缩，提高页面加载速度，降低带宽消耗。设置 gzip_types 以指定要压缩的内容类型。

    gzip on;
gzip_types text/plain text/css application/json application/javascript text/xml application/xml application/xml+rss;
gzip_min_length 1000;  # 最小压缩字节数

    11. 缓存配置¶

    Nginx 可以作为缓存代理服务器，提升静态资源和 API 的响应速度：

    11.1.静态文件缓存¶

    location /images/ {
    expires 30d;
    add_header Cache-Control "public, no-transform";
}

    11.2.代理缓存¶

    proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=STATIC:10m inactive=24h max_size=1g;

server {
    location / {
        proxy_cache STATIC;
        proxy_cache_valid 200 1h;
        proxy_pass http://backend;
    }
}
    • proxy_cache_path：定义缓存路径和空间限制。
    • proxy_cache：启用缓存区域，STATIC 为缓存区域名。
    • proxy_cache_valid：指定缓存的状态码和时间，例如 200 状态缓存 1 小时。

    12. 安全配置¶

    12.1.限制请求速率¶

    防止暴力攻击或恶意请求：

    http {
    limit_req_zone $binary_remote_addr zone=one:10m rate=1r/s;

    server {
        location /login {
            limit_req zone=one burst=5 nodelay;
        }
    }
}
    • limit_req_zone：定义限速区域，以客户端 IP 作为标识。
    • limit_req：指定限速区域和允许的请求速率。

    12.2.隐藏版本信息¶

    避免泄露 Nginx 版本信息，减少攻击面：

    server_tokens off;

    12.3.限制请求体大小¶

    防止大文件请求影响服务器性能：

    client_max_body_size 10M;

    12.4.禁止访问敏感文件¶

    location ~ /\.ht {
    deny all;
}

    13. 日志配置¶

    配置 Nginx 日志有助于监控和排查问题：

    http {
    log_format main '$remote_addr - $remote_user [$time_local] "$request" '
                    '$status $body_bytes_sent "$http_referer" '
                    '"$http_user_agent" "$http_x_forwarded_for"';

    access_log /var/log/nginx/access.log main;
    error_log /var/log/nginx/error.log warn;
}
    • log_format：定义日志格式，如 main 格式包含客户端 IP、请求时间、状态码等。
    • access_log 和 error_log：分别记录访问日志和错误日志。

    2.Traefik¶

    官网: traefik-env

    Traefik 是一个开源的边缘路由器（reverse proxy），主要用于在容器化微服务架构中处理 HTTP、TCP、WebSocket 等请求流量。它支持自动发现服务并动态更新路由配置，具备强大的负载均衡、SSL/TLS 自动化、集成服务发现以及全面支持多种提供者（如 Docker、Kubernetes、Consul 等）。

    2.1.主要功能¶

    1. 自动化的服务发现：Traefik 可以从不同的服务提供者（如 Docker、Kubernetes、Consul 等）中自动发现服务，动态生成路由规则，无需手动编写大量配置文件。
    2. 反向代理：支持 HTTP/HTTPS、TCP 和 UDP 请求的反向代理，能够将外部请求路由到内部服务。
    3. 负载均衡：支持多种负载均衡算法，如轮询（Round Robin）、权重（Weight）等。
    4. SSL 自动化：通过集成 Let's Encrypt，Traefik 可以自动为你的服务获取并管理 SSL 证书，实现 HTTPS。
    5. API 和 Dashboard：提供 Web 界面（dashboard）和 API，可以直观地监控服务路由、健康状况、负载均衡策略等。
    6. 中间件（Middleware）：支持各种中间件功能，如身份认证、速率限制、重写 URL 等。

    2.2.基本架构¶

    Traefik 的核心是入口点（EntryPoints）、路由器（Routers）、服务（Services）*和*中间件（Middlewares）：

    • 入口点（EntryPoints）：定义了 Traefik 接收外部请求的端口（如 80、443）。每个入口点可以绑定一个或多个协议。
    • 路由器（Routers）：匹配规则（规则可以基于主机名、路径等），将请求路由到对应的服务。
    • 服务（Services）：指向你的实际应用（如 Docker 容器、Kubernetes Pod 或外部服务）。
    • 中间件（Middlewares）：可以对流量进行处理，如身份认证、重定向、修改请求头等。

    2.3.Traefik 基本使用示例¶

    下面是使用 Traefik 的基本步骤，包括使用 Docker Compose 的配置和简单的服务定义。

    1. Docker Compose 配置 Traefik¶

    假设我们有两个服务：一个是静态页面服务，另一个是 API 服务。我们可以通过 Traefik 进行代理和负载均衡。

    services:
  # Traefik 服务
  traefik:
    image: traefik:v2.9
    container_name: traefik
    command:
      - "--api.insecure=true"                  # 启用不安全的 API（生产环境禁用）
      - "--api.dashboard=true"                 # 启用 Traefik dashboard
      - "--entrypoints.web.address=:80"        # 定义 HTTP 入口点，监听 80 端口
      - "--providers.docker=true"              # 启用 Docker 提供者，自动发现服务
      - "--providers.docker.exposedbydefault=false"  # 仅处理显式定义的服务
    ports:
      - "80:80"      # 暴露 80 端口用于 HTTP 请求
      - "8080:8080"  # 暴露 8080 端口用于访问 Traefik dashboard
    volumes:
      - "/var/run/docker.sock:/var/run/docker.sock:ro"  # 授权访问 Docker API

  # 静态页面服务
  web:
    image: nginx:alpine
    container_name: web
    labels:
      - "traefik.enable=true"  # 启用 Traefik 路由
      - "traefik.http.routers.web.rule=Host(`example.com`)"  # 基于主机名匹配的路由
      - "traefik.http.services.web.loadbalancer.server.port=80"  # 指定服务的端口

  # API 服务
  api:
    image: hashicorp/http-echo:latest
    container_name: api
    command: ["-text", "Hello from API"]
    labels:
      - "traefik.enable=true"  # 启用 Traefik 路由
      - "traefik.http.routers.api.rule=Host(`api.example.com`)"  # 基于主机名匹配的路由
      - "traefik.http.services.api.loadbalancer.server.port=5678"  # 指定服务的端口

    2. 详细说明¶

    • Traefik 服务：
    • Traefik 作为反向代理，它监听 80 端口，并暴露 8080 端口用于访问 Traefik dashboard（http://localhost:8080）。
    • providers.docker=true 启用了 Docker 提供者，允许 Traefik 自动发现带有特定标签的 Docker 容器并代理它们。
    • 静态页面服务（web）：
    • 使用 Nginx 作为静态页面服务，服务的标签通过 traefik.http.routers.web.rule=Host('example.com') 定义了路由规则。所有访问 example.com 的请求将路由到这个 Nginx 服务。
    • API 服务：
    • 使用 HashiCorp 的 http-echo 镜像作为简单的 API 服务，路由规则通过 Host('api.example.com') 来定义。请求 api.example.com 将被转发到这个 API 服务。

    3. 启动并测试¶

    运行 Docker Compose：

    docker-compose up -d

    然后，配置本地的 /etc/hosts 文件，添加以下条目，以便 example.com 和 api.example.com 能够解析到你的本地机器：

    127.0.0.1 example.com
127.0.0.1 api.example.com
    • 访问 http://example.com，你将看到 Nginx 静态页面。
    • 访问 http://api.example.com，你将看到 Hello from API 的响应。
    • 访问 http://localhost:8080，可以查看 Traefik 的 Dashboard，展示当前的服务和路由。

    4. 跨网络¶

    代理多个 Docker Compose 项目（跨网络）

    有多个 Docker Compose 项目并希望让 Traefik 统一代理不同网络中的容器服务，可以参考以下配置:

    1. 定义全局网络：

    2. 创建一个跨多个 Compose 项目共享的 Docker 网络，使 Traefik 能够跨网络代理容器。

    3. 各 Compose 文件可以将各自的服务连接到这个共享网络，从而让 Traefik 能够访问它们。
    docker network create traefik_proxy

    1. 将 Traefik 加入共享网络：

    2. 在 Traefik 的 Docker Compose 文件中，将它连接到这个共享网络，以便它能够通过该网络访问其他服务。

    services:
  traefik:
    image: traefik
    networks:
      - traefik_proxy  # 添加共享网络
    ...
networks:
  traefik_proxy:
    external: true

    1. 各服务加入共享网络：

    2. 在每个 Docker Compose 项目的文件中，将需要代理的服务连接到 traefik_proxy 网络。这样，即使这些服务有各自的网络，只要它们都连接了共享网络，Traefik 就能找到它们。

    services:
  myservice:
    ...
    networks:
      - default
      - traefik_proxy  # 添加共享网络
networks:
  default:
    ...
  traefik_proxy:
    external: true

    1. 使用标签配置 Traefik 路由：

    2. 在各个服务上定义 traefik 标签，Traefik 将根据标签识别并代理这些服务。不同 Compose 项目中的服务可以配置不同的域名或路径前缀，从而为每个服务创建独立的路由规则。

    services:
  myservice:
    ...
    labels:
      - traefik.enable=true
      - traefik.http.routers.myservice.rule=Host(`service.example.com`)
      - traefik.http.services.myservice.loadbalancer.server.port=80

    1. 配置 Traefik 的 Docker Provider：

    2. 确保 Traefik 的 Docker Provider 启用了 exposedByDefault=false（默认不暴露），以确保仅代理被显式标记的服务。

    3. Traefik 将在共享网络内根据标签动态发现并代理服务。

    4. 主要限制和注意事项

    5. 外部网络访问限制：如果不想用共享网络，还可以通过 Traefik 的 HTTP 请求（而不是内部 Docker 网络）来代理外部网络中的服务。

    6. 安全和隔离：将 Traefik 暴露在多个网络中可能影响网络隔离，推荐对关键服务设置访问控制。
    7. 跨网络通信：对于需要高隔离的环境，可以使用 DNS 解析或 TCP 路由而不是共享网络，但这会增加配置复杂性。

    5.进阶功能¶

    5.1.HTTPS¶

    SSL 配置： 可以通过 Let's Encrypt 自动获取 SSL 证书，在 traefik 配置中添加以下内容：

    command:
  - "--entrypoints.websecure.address=:443"
  - "--certificatesresolvers.myresolver.acme.httpchallenge=true"    # 使用 HTTP-01 挑战
  - "--certificatesresolvers.myresolver.acme.httpchallenge.entrypoint=web"
  - "--certificatesresolvers.myresolver.acme.email=your-email@example.com"    # Let's Encrypt 注册邮箱
  - "--certificatesresolvers.myresolver.acme.storage=/letsencrypt/acme.json"  # 证书存储位置
    1. TLS-ALPN-01 (TLS 证书挑战)

    TLS-ALPN-01 挑战通过在目标服务器上使用 TLS ALPN（Application-Layer Protocol Negotiation）协议验证域名的所有权。验证过程涉及通过特定端口（443）建立 TLS 连接并发送特定的认证信息。

    工作原理:

    • Let's Encrypt 客户端会生成一个带有挑战响应的自签名证书。
    • 客户端将此证书通过 ALPN 协议与 CA 的服务器进行握手。
    • CA 会检查证书是否包含正确的挑战响应，以确认域名所有权。

    使用场景:

    • 适用场景：适用于无法在 80 端口上使用 HTTP 或 443 端口的 HTTPS 环境（例如一些严格的防火墙环境）。
    • 优点：不需要在公开的 HTTP 端口（80）上暴露服务，因此适合更严格的安全策略。
    • 缺点：不支持负载均衡器，因为 ALPN 验证要求直接对证书进行验证。

    实现方法:

    1. HTTP-01 (HTTP 文件挑战)

    HTTP-01 挑战通过在 HTTP 协议的 80 端口上创建一个特定的认证文件来完成域名验证。Let's Encrypt 的服务器会尝试访问该文件并验证内容来确认域名所有权。

    工作原理:

    • Let's Encrypt 客户端在目标服务器上生成一个挑战文件，并将其放置在 http://yourdomain.com/.well-known/acme-challenge/<token>。
    • Let's Encrypt 服务器会发送请求以验证此文件。
    • 如果请求成功且文件内容正确，证明域名所有者具有对该域名的控制权。

    使用场景:

    • 适用场景：适用于域名可以公开访问的 HTTP 服务，通常是中小型网站和简单架构。
    • 优点：配置简单，适合静态服务器环境，不依赖 DNS 配置。
    • 缺点：依赖 80 端口，如果域名后面是负载均衡器或 CDN，可能需要手动操作，且在严格的防火墙环境下可能无法使用。

    实现方法:

    配置 HTTP-01 挑战时，需要在服务器上确保 80 端口开放，并可以写入 .well-known 目录的挑战文件。自动化配置常见于 Certbot 等工具。

    1. DNS-01 (DNS TXT 记录挑战)

    DNS-01 挑战通过在域名的 DNS 记录中创建一个特定的 TXT 记录来完成验证。Let's Encrypt 服务器会验证该 TXT 记录的值是否符合预期以确定域名所有权。

    工作原理:

    • Let's Encrypt 客户端会生成一个挑战字符串。
    • 该字符串需要被写入域名的 TXT 记录，例如 _acme-challenge.yourdomain.com。
    • Let's Encrypt 服务器会查询此记录以验证字符串内容是否正确。

    使用场景:

    • 适用场景：适合不直接暴露 HTTP 服务或 HTTPS 443 端口的域名（如内网服务、泛域名证书）。尤其适合复杂网络结构，如 CDN 后的服务器或负载均衡后端。
    • 优点：支持泛域名验证（*.yourdomain.com），且不需要开放 80 或 443 端口。
    • 缺点：需要对域名 DNS 记录的管理权限和 API 支持。某些 DNS 服务商可能不支持快速更新 TXT 记录。

    实现方法:

    使用 DNS-01 挑战时，Let's Encrypt 客户端会请求特定的 API（如 Cloudflare 或 AWS Route 53）来自动更新 DNS 记录，Certbot 和 acme.sh 等工具可以很好地支持该过程。

    5.2.中间件¶

    中间件（Middlewares）： 你可以为某些路由添加中间件功能，例如为 api.example.com 添加速率限制：

  - "traefik.http.routers.api.middlewares=api-ratelimit"             # 绑定中间件
    5.3.比较总结¶

    各种挑战方式的对比与选择:

    挑战方式 工作端口 适用场景 优点 缺点
    TLS-ALPN-01 443 严格的 HTTPS 环境，无法开放 80 端口时使用 无需开放 80 端口 可能不支持负载均衡
    HTTP-01 80 通常的公开域名和简单的网络架构 配置简单，适合静态站点 必须开放 80 端口
    DNS-01 无 支持泛域名、CDN 后端、复杂网络架构等 适用性广，支持泛域名证书 依赖 DNS 提供商的 API 支持

    总结:

    Traefik 是一个强大的边缘路由器，适用于微服务环境，特别是在容器化和动态服务架构中。它通过自动化服务发现、SSL 管理和负载均衡简化了反向代理配置，具备丰富的中间件支持，用于流量管理、身份验证和请求修改。

    如果 公开域名且服务器上 80 端口可用，推荐使用 HTTP-01 挑战，配置简单。

    如果 需要泛域名证书或复杂网络环境（如 CDNs 或负载均衡后端），推荐 DNS-01 挑战，但需要支持自动更新 DNS 记录的 API。

    在特殊环境（如内部网络或无法开放 80 端口）下，可选择 TLS-ALPN-01 挑战，但需确保反向代理支持 ALPN。

    3.Openresty¶

    官网: Openresty文档

    OpenResty 是一个高性能的 Web 平台，它基于 Nginx 核心，并集成了 LuaJIT，使得开发者能够在 Nginx 中运行 Lua 脚本。OpenResty 允许开发者以更灵活的方式定制 Nginx 的行为，从而为复杂的 Web 应用场景提供极大的扩展能力。它广泛用于高并发应用，如 API 网关、Web 服务、代理服务器等。

    3.1.主要功能¶

    1. Lua 脚本扩展： OpenResty 允许在 Nginx 配置中嵌入 Lua 脚本，通过 LuaJIT 实现强大的动态处理能力，如自定义请求处理、复杂的访问控制、数据缓存和负载均衡等。
    2. 高性能和异步架构： 基于 Nginx 的事件驱动模型，OpenResty 能够处理大量并发连接，并且 Lua 模块可以以异步方式执行，保持极高的性能。
    3. 丰富的模块生态： OpenResty 集成了大量模块，支持 Redis、Memcached、PostgreSQL、MySQL 等后端系统，简化了分布式系统的开发。
    4. 灵活的 Web 应用： 通过 Lua 的嵌入式脚本，开发者可以灵活地实现自定义的 Web 应用逻辑，比如用户认证、请求重写、复杂的负载均衡策略等。
    5. 全面的 API 支持： OpenResty 可以用于构建高性能 API 网关，提供对 HTTP/2、WebSocket 等协议的支持，允许轻松处理复杂的业务逻辑。

    3.2.OpenResty 的基本架构¶

    OpenResty 的核心架构与 Nginx 相同，它依赖于事件驱动的模型，能够处理高并发的 HTTP 请求。在此基础上，OpenResty 增加了对 Lua 脚本的支持，使开发者可以编写高性能的 Lua 脚本来处理请求。其架构主要由以下部分组成：

    • Nginx 核心：负责处理 HTTP 请求、响应、反向代理等功能。
    • LuaJIT：Lua 脚本的 JIT 编译器，用于快速执行 Lua 代码。
    • 模块生态：包括各种第三方模块，如 Redis、Memcached、数据库连接模块等，用于扩展 Nginx 的功能。

    2. 基本配置示例¶

    下面是一个简单的 OpenResty 配置文件，它展示了如何通过 Lua 处理 HTTP 请求：

    worker_processes  1;

    worker_connections  1024;
}

    include       mime.types;
    default_type  application/octet-stream;

    server {
        listen       8080;
        server_name  localhost;

        location / {
            # 使用 Lua 脚本处理请求
            content_by_lua_block {
                ngx.say("Hello from OpenResty with Lua!")
            }
        }

        location /api {
            content_by_lua_block {
                -- 自定义 API 逻辑
                ngx.say("Welcome to API endpoint!")
            }
        }

        location /cache {
            content_by_lua_block {
                -- 与 Redis 交互
                local redis = require "resty.redis"
                local red = redis:new()
                red:set_timeout(1000)  -- 1 sec

                local ok, err = red:connect("127.0.0.1", 6379)
                if not ok then
                    ngx.say("failed to connect: ", err)
                    return
                end

                -- 从 Redis 获取数据
                local res, err = red:get("my_key")
                if res == ngx.null then
                    ngx.say("no data in Redis")
                else
                    ngx.say("Redis data: ", res)
                end
            }
        }
    }
}

    4. 动态代理场景：需要中间件功能或自定义插件处理请求时，尤其是在动态环境下需要快速变化的情况下, 适合频繁扩展、缩减或变更的环境，能够轻松实现热更新而无需重启服务，Traefik 能够提供强大的灵活性和性能。
    5. Nginx 的使用场景：
    6. 静态资源托管：如果你需要一个高性能的 Web 服务器来托管静态文件（如 HTML、CSS、JS 文件），Nginx 是最佳选择。
    7. 负载均衡和反向代理：Nginx 是一个成熟的反向代理和负载均衡解决方案，适合用于高并发、高稳定性需求的场景。
    8. 简单 Web 服务：对于不需要复杂动态处理的 Web 服务，Nginx 的配置相对简单、性能优越。在需要处理 WebSocket 连接或 HTTP/2 请求的场景下，Nginx 也提供了出色的支持。
    9. OpenResty 的使用场景：
    10. API 网关：如果你需要处理复杂的 API 请求（如身份验证、限流、缓存等），并且希望能够灵活地编写自定义逻辑，OpenResty 是一个强大且灵活的选择。
    11. 高性能 Web 应用：需要构建高度动态、高并发的 Web 应用，OpenResty 结合 Lua 的灵活性可以实现 Nginx 难以实现的功能。
    12. 缓存与动态服务：在需要高度定制缓存或处理复杂业务逻辑时，如项目需要频繁与 Redis、Memcached 等后端系统交互，同时又希望轻松编写缓存逻辑，OpenResty 提供了通过 Lua 轻松实现这种功能的途径。OpenResty 的异步 Lua 支持使其成为一种高效的解决方案。

    3.5.总结¶

    • 如果你需要自动化的微服务发现和动态代理，并且希望减少手动配置的工作量，Traefik 是最佳选择。
    • 如果你需要一个简单、高性能的反向代理或静态资源服务器，并且不需要复杂的动态逻辑，Nginx 是较好的选择。
    • 如果你需要处理复杂的动态业务逻辑，并希望通过脚本自定义处理请求（如 API 网关、缓存、身份验证等），OpenResty 提供了 Nginx 基础上更强大的 Lua 扩展能力，是构建高性能 Web 应用和 API 服务的理想选择。
    • 其他:
    • Traefik 适合微服务架构、动态发现和 SSL 自动化。
    • Nginx 是处理静态资源、高性能反向代理的行业标准。
    • OpenResty 适合需要高度定制和脚本化处理复杂动态请求的应用场景。
