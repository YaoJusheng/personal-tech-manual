    2.1.1.--mount文件系统挂载¶

    可以用于创建构建可以访问的文件系统挂载。支持的挂载类型有:

    Type Description
    bind (default) Bind-mount context directories (read-only).
    cache Mount a temporary directory to cache directories for compilers and package managers.
    tmpfs Mount a tmpfs in the build container.
    secret Allow the build container to access secure files such as private keys without baking them into the image or build cache.
    ssh Allow the build container to access SSH keys via SSH agents, with support for passphrases.
    2.1.1.1.--mount=type=bind¶

    此安装类型允许将文件或目录绑定到构建容器。默认情况下，绑定挂载是只读的。

    2.1.1.2.--mount=type=cache¶

    此安装类型允许构建容器缓存编译器和包管理器的目录。

    # syntax=docker/dockerfile:1
FROM golang
RUN --mount=type=cache,target=/root/.cache/go-build \
  go build ...


# e.g.
# syntax=docker/dockerfile:1
FROM ubuntu
RUN rm -f /etc/apt/apt.conf.d/docker-clean; echo 'Binary::apt::APT::Keep-Downloaded-Packages "true";' > /etc/apt/apt.conf.d/keep-cache
RUN --mount=type=cache,target=/var/cache/apt,sharing=locked \
  --mount=type=cache,target=/var/lib/apt,sharing=locked \
  apt update && apt-get --no-install-recommends install -y gcc
    2.1.1.3.--mount=type=tmpfs¶

    此挂载类型允许在构建容器中挂载 tmpfs。

    2.1.1.4.--mount=type=secret¶

    这种安装类型允许构建容器访问秘密值，例如令牌或私钥，而无需将它们烘焙到映像中。

    # syntax=docker/dockerfile:1
FROM python:3
RUN pip install awscli
RUN --mount=type=secret,id=aws,target=/root/.aws/credentials \
  aws s3 cp s3://... ...
    2.1.1.5.--mount=type=ssh¶

    此安装类型允许构建容器通过 SSH 代理访问 SSH 密钥，并支持密码短语。

    # syntax=docker/dockerfile:1
FROM alpine
RUN apk add --no-cache openssh-client
RUN mkdir -p -m 0700 ~/.ssh && ssh-keyscan gitlab.com >> ~/.ssh/known_hosts
RUN --mount=type=ssh \
  ssh -q -T git@gitlab.com 2>&1 | tee /hello
# "Welcome to GitLab, @GITLAB_USERNAME_ASSOCIATED_WITH_SSHKEY" should be printed here
# with the type of build progress is defined as `plain`.

    2.1.2.--network¶

    RUN --network 允许控制命令运行的网络环境。

    Type Description
    default (default) Run in the default network.
    none Run with no network access.
    host Run in the host's network environment.

    2.1.3.LABEL¶

    LABEL 指令将元数据添加到图像中。标签是一个键值对。要在 LABEL 值中包含空格，请像在命令行解析中一样使用引号和反斜杠。一些使用示例：

    LABEL "com.example.vendor"="ACME Incorporated"
LABEL com.example.label-with-value="foo"
LABEL version="1.0"
LABEL description="This text illustrates \
that label-values can span multiple lines."

LABEL multi.label1="value1" \
      multi.label2="value2" \
      other="value3"

    要查看图像的标签，请使用 Docker image inform 命令。您可以使用 --format 选项仅显示标签；

    docker image inspect --format='{{json .Config.Labels}}' myimage

    2.1.4.ADD¶

    ADD 有两种形式。包含空格的路径需要后一种形式。

    ADD [OPTIONS] <src> ... <dest>
ADD [OPTIONS] ["<src>", ... "<dest>"]

ADD file1.txt file2.txt /usr/src/things/

ADD https://example.com/archive.zip /usr/src/things/
ADD git@github.com:user/repo.git /usr/src/things/

    2.1.5.COPY¶

    COPY 有两种形式。包含空格的路径需要后一种形式。

    COPY file1.txt file2.txt /usr/src/things/

FROM golang AS build
WORKDIR /app
RUN --mount=type=bind,target=. go build -o /myapp ./cmd

COPY --from=build /myapp /usr/bin/

    2.1.6.HEALTHCHECK¶

    HEALTHCHECK 指令有两种形式：

    • HEALTHCHECK [OPTIONS] CMD 命令（通过在容器内运行命令来检查容器运行状况）

    • HEALTHCHECK NONE（禁用从基础镜像继承的任何健康检查）

    HEALTHCHECK 指令告诉 Docker 如何测试容器以检查它是否仍在工作。这可以检测诸如 Web 服务器陷入无限循环并且无法处理新连接等情况，即使服务器进程仍在运行也是如此。

    The options that can appear before CMD are:

--interval=DURATION (default: 30s)
--timeout=DURATION (default: 30s)
--start-period=DURATION (default: 0s)
--start-interval=DURATION (default: 5s)
--retries=N (default: 3)

    HEALTHCHECK --interval=5m --timeout=3s \
  CMD curl -f http://localhost/ || exit 1

ARG CUDA_VERSION=11.3.1
ARG PYTHON_VERSION=3.8.12
ARG NMDATA_VERSION=1.0.0
ARG TZ="Asia/Shanghai"
ARG PIP_SOURCE="https://pypi.tuna.tsinghua.edu.cn/simple"
ARG APT_SOURCE="mirrors.aliyun.com"

ARG USED_FOR_EXTERNAL=true
ARG UID=1000
ARG DGID=999
ARG USER=example
ARG PASSWORD=123456

FROM ubuntu:$BASE_VERSION AS package_builder

LABEL description="This is the base image for packages"
LABEL version="1.0"
LABEL maintainer="jusheng.yao@example.cn"

ARG TZ
ARG UID
ARG USER
ARG NMDATA_VERSION
ARG PYTHON_VERSION
ARG PIP_SOURCE
ARG APT_SOURCE

ENV DEBIAN_FRONTEND=noninteractive
ENV NMDATA_VERSION=${NMDATA_VERSION} PACKAGE_DIR=/data/packages

    && apt-get update > /dev/null \
    && apt-get --no-install-recommends install -d -y -o=Dir::Cache::Archives=${PACKAGE_DIR} \
        apt-utils \
        apt-transport-https \
        build-essential \
        curl \
        make \
        iproute2 \
        gdb \
        lcov \
        libbz2-dev \
        libffi-dev \
        libgdbm-compat-dev \
        libgdbm-dev \
        libgl1-mesa-glx \
        libglib2.0-0 \
        libglib2.0-dev \
        liblzma-dev \
        libmpdec-dev \
        libncurses5-dev \
        libpcre3 \
        libpcre3-dev \
        libreadline6-dev \
        libsqlite3-dev \
        libssl-dev \
        lzma \
        lzma-dev \
        moreutils \
        pkg-config \
        tzdata \
        tk-dev \
        unzip \
        util-linux \
        uuid-dev \
        vim \
        wget \
        xz-utils \
        zlib1g-dev > /dev/null \
    && dpkg -i ${PACKAGE_DIR}/*.deb || apt-get -f --no-install-recommends install -y \
    && wget -B -c -np -nv -N -nH -q --no-check-certificate https://www.python.org/ftp/python/${PYTHON_VERSION}/Python-${PYTHON_VERSION}.tgz -P ${PACKAGE_DIR}/ \
    && ln -fs /usr/share/zoneinfo/$TZ /etc/localtime \
    && dpkg-reconfigure -f noninteractive tzdata \
    && apt-get clean \
    && rm -rf /var/lib/apt/lists/*

    4.常见问题¶

    1. WSGI 和 ASGI 服务器的启动工具有哪些？比较并给出使用建议¶

    常见的 WSGI 服务器¶

    • Gunicorn：是最常用的 Python WSGI 服务器之一，支持多进程和多线程，具有良好的性能，适合中小型的同步应用。常用于搭配 Django 和 Flask 等 WSGI 框架。
    • uWSGI：是一种功能非常强大的服务器，支持多种语言，包括 Python。uWSGI 提供了更多的高级配置和功能选项，适合复杂的生产环境部署。通常用于部署大型 Python 应用，特别是需要细粒度控制时。
    • mod_wsgi：这是 Apache 服务器的 WSGI 模块。mod_wsgi 集成在 Apache 中，可以将 Python 应用直接运行在 Apache 上，但在灵活性和性能上不如 Gunicorn 和 uWSGI。

    使用建议：

    • 如果是小型或中型应用，使用 Gunicorn 是最佳选择，因为配置简单，且具有较好的性能和多进程支持。
    • 对于需要大量定制和更复杂的应用场景，uWSGI 提供更多配置选项，适合需要复杂管理的生产环境。
    • 如果环境中已经在使用 Apache 服务器，可以考虑使用 mod_wsgi 进行集成。

    常见的 ASGI 服务器¶

    • Uvicorn：轻量级、高性能的 ASGI 服务器，支持异步任务和 HTTP/2，非常适合使用 FastAPI 或 Starlette 等框架。
    • Daphne：是 Django Channels 项目的 ASGI 服务器，适合需要 WebSocket 和异步支持的 Django 应用。通常用于需要实时通信的应用，如聊天应用。
    • Hypercorn：另一个 ASGI 服务器，支持 HTTP/2 和 HTTP/3，功能更丰富，适合需要多协议支持的异步应用场景。

    使用建议：

    • 对于一般的 ASGI 应用（如 FastAPI、Starlette），Uvicorn 是首选，轻量且性能优秀。
    • 对于需要 Django Channels 支持的项目（如需要 WebSocket 和异步支持），Daphne 更合适。
    • Hypercorn 在多协议支持上有优势（如 HTTP/2 和 HTTP/3），适合更现代的应用需求。

    总结建议¶

    • WSGI 服务器：Gunicorn 更适合一般应用，uWSGI 则适合大型项目和需要复杂配置的场景。
    • ASGI 服务器：Uvicorn 更适合一般异步应用，Daphne 专为 Django Channels 设计，Hypercorn 适合需要 HTTP/2 和 HTTP/3 支持的场景。

    2. 对于一个 Docker 容器，关于内部应用的进程是单个进程好还是多个进程好？¶

    在 Docker 容器中，通常只建议运行单个主进程。这样符合 Docker 的设计哲学，即“一次运行一个服务”的理念。以下是不同选择的分析：

    • 单个主进程（推荐）：Docker 的设计是单进程容器，将应用的主进程设置为容器的 PID 1。这样，当主进程退出时，容器会自动停止，并且资源可以被回收，符合容器生命周期管理原则。
    • 多进程：虽然可以在一个容器中运行多个进程（例如使用 supervisord 管理多个进程），但这种方法通常是为了特殊用途，例如复杂的多服务容器或一些无法拆分的应用（例如需要某些守护进程来支持主进程的运行）。但多进程的方案会破坏 Docker 容器的“一个容器一个服务”的设计原则，不推荐除非有特别的需求。

    建议：

    • 生产环境：每个容器运行一个主进程（例如应用服务器或数据库等），如果需要多个服务，应考虑分多个容器来部署，并通过 Docker Compose 或 Kubernetes 编排。
    • 开发环境：在一些开发环境中可以暂时容忍多进程容器，但在生产中应尽量避免。

    3. Docker 容器本身是单个主进程启动的，在容器内部使用多进程的作用和目的，与在主机上多进程部署应用有什么区别？¶

    • 容器内的多进程：在单个容器中使用多进程方式（例如 Gunicorn 多进程、Uvicorn 的 worker），是为了提高应用的并发处理能力。多进程通常可以充分利用多核 CPU 资源。
    • 主机上的多进程：在物理主机上多进程运行多个应用实例（例如多容器实例或多进程应用）也是为了并发和负载均衡。在主机上可以直接启动多个进程或容器，每个实例可以独立管理和监控。

    区别：

    • 隔离性：容器内的多进程相对独立，但共享同一个文件系统和网络空间，而主机上的多进程实例可以进一步隔离。
    • 资源管理：容器可以限制 CPU 和内存使用，在容器内部的多进程会受到这些资源限制。
    • 扩展性：在主机上可以通过水平扩展增加容器数量，而在容器内多进程扩展受到单个容器资源限制。

    4. Uvicorn、Gunicorn、uWSGI 本身支持多进程启动应用，为什么还要再加上 Supervisor？其必要性是什么？¶

    虽然 Uvicorn、Gunicorn 和 uWSGI 本身可以支持多进程或多线程启动应用，但是在生产环境中使用 Supervisor 或类似的进程管理工具仍然具有一些优势。其必要性和优势如下：

    • 自动重启：Supervisor 能够自动监控和重启进程，如果应用崩溃或意外退出，Supervisor 可以确保进程立即重启，保持服务的高可用性。
    • 日志管理：Supervisor 支持集中管理日志，可以对所有进程的日志进行收集、归档、轮转等操作。对于单独的 Gunicorn 或 Uvicorn 进程，日志可能更难管理。
    • 进程监控：Supervisor 提供的管理界面可以查看每个子进程的状态，例如是否在运行、进程 ID 等，方便运维管理。
    • 多服务管理：在一些复杂场景中，可能需要启动多个服务（例如 Web 服务器、队列处理、定时任务等），Supervisor 可以统一管理这些服务，简化启动和管理流程。
    • 与操作系统无关：对于不同操作系统环境，Supervisor 提供了一个一致的进程管理方式，适用于多平台部署。

    总结：

    • 在简单的应用场景中（单一应用，没有复杂的进程管理需求），Gunicorn 或 Uvicorn 自带的多进程支持已经足够。
    • 在需要更多控制、日志管理、自动重启以及多服务管理的场景中，引入 Supervisor 可以显著提升服务的可靠性和可维护性。
