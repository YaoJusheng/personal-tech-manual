    1.1.生成自签名证书¶

    在服务器上生成 SSL 自签名证书（假设域名为 yourdomain.com）。在 GitLab 目录中创建证书文件夹，并生成证书文件：

    mkdir -p ./gitlab/config/ssl
# --- 方式1 ---
openssl req -new -newkey rsa:2048 -days 365 -nodes -x509 \
    -subj "/CN=yourdomain.com" \
    -keyout ./gitlab/config/ssl/yourdomain.com.key \
    -out ./gitlab/config/ssl/yourdomain.com.crt

# --- 方式2 ---
# 生成指定位数的 RSA 私钥：ca.key
openssl genrsa -out ca.key 2048
openssl req -new -x509 -days 3650 -key ca.key \
    -subj "/C=CN/ST=Shanghai/L=Shanghai/O=AI Team/OU=Personal/CN=local-dev.gitlab.com/emailAddress=jushengyao@gmail.com" \
    -out ca.crt -utf8
# 生成 server.csr、server.key
# CN：设置你要使用的域名
# -utf8：支持中文
openssl req -newkey rsa:2048 -nodes -keyout server.key \
    -subj "/C=CN/ST=Shanghai/L=Shanghai/O=AI Team/OU=Personal/CN=local-dev.gitlab.com" \
    -out server.csr -utf8
# 生成 ca.srl、server.crt
# subjectAltName：设置 DNS、IP
openssl x509 -req -extfile <(printf "subjectAltName=IP:192.168.80.14") -days 3650 -in server.csr -CA ca.crt -CAkey ca.key -CAcreateserial -out server.crt

    上述命令生成 .crt 和 .key 文件，分别为自签名的证书和私钥。

    1.1.1. 生成 CA 私钥¶

    openssl genrsa -out ca.key 2048
    • openssl genrsa: 生成 RSA 私钥。
    • -out ca.key: 指定输出文件为 ca.key，这是生成的 CA 私钥文件。
    • 2048: 指定生成的密钥长度为 2048 位。

    1.1.2. 生成 CA 证书¶

    openssl req -new -x509 -days 3650 -key ca.key \
    -subj "/C=CN/ST=Shanghai/L=Shanghai/O=AI Team/OU=Personal/CN=local-dev.gitlab.com/emailAddress=xxx@gmail.com" \
    -out ca.crt -utf8
    • openssl req: 创建和处理证书请求。
    • -new: 生成一个新的证书请求。
    • -x509: 生成自签名证书，而不是证书请求。
    • -days 3650: 指定证书有效期为 3650 天（约 10 年）。
    • -key ca.key: 使用之前生成的 CA 私钥 ca.key。
    • -subj: 指定证书主题信息，包含：
    • C=CN: 国家（Country）为中国。
    • ST=Shanghai: 省份（State）为上海。
    • L=Shanghai: 城市（Locality）为上海。
    • O=AI Team: 组织名称（Organization）为 AI Team。
    • OU=Personal: 组织单位（Organizational Unit）为 Personal。
    • CN=local-dev.gitlab.com: 公共名称（Common Name）为 local-dev.gitlab.com。
    • emailAddress=xxx@gmail.com: 电子邮件地址。
    • -out ca.crt: 指定输出文件为 ca.crt，这是生成的 CA 证书文件。
    • -utf8: 使主题字符串以 UTF-8 编码。

    1.1.3. 生成服务器私钥和证书请求¶

    openssl req -newkey rsa:2048 -nodes -keyout server.key \
    -subj "/C=CN/ST=Shanghai/L=Shanghai/O=AI Team/OU=Personal/CN=local-dev.gitlab.com" \
    -out server.csr -utf8
    • openssl req: 创建和处理证书请求。
    • -newkey rsa:2048: 生成新的 RSA 私钥，密钥长度为 2048 位。
    • -nodes: 生成不加密的私钥文件（即不使用密码保护）。
    • -keyout server.key: 指定输出文件为 server.key，这是生成的服务器私钥文件。
    • -subj: 指定证书主题信息，包含国家、省份、城市、组织、组织单位和公共名称，类似于上面的 CA 证书。
    • -out server.csr: 指定输出文件为 server.csr，这是生成的证书请求文件。
    • -utf8: 使主题字符串以 UTF-8 编码。

    1.1.4. 签署服务器证书¶

    openssl x509 -req -extfile <(printf "subjectAltName=IP:192.168.80.14") -days 3650 -in server.csr -CA ca.crt -CAkey ca.key -CAcreateserial -out local-dev.gitlab.com.crt
    • openssl x509: 用于处理 X.509 证书。
    • -req: 处理证书请求。
    • -extfile <(printf "subjectAltName=IP:192.168.80.14"): 指定扩展文件，通过使用进程替代将 subjectAltName 直接传递。此处为证书添加了一个主题备用名称（Subject Alternative Name），允许通过 IP 地址访问。
    • -days 3650: 指定证书有效期为 3650 天（约 10 年）。
    • -in server.csr: 输入证书请求文件 server.csr。
    • -CA ca.crt: 使用 CA 证书 ca.crt 来签署此请求。
    • -CAkey ca.key: 使用 CA 私钥 ca.key 来签署此请求。
    • -CAcreateserial: 创建序列号文件（通常是 ca.srl），如果没有提供此参数且没有序列号文件，则会报错。
    • -out local-dev.gitlab.com.crt: 指定输出文件为 local-dev.gitlab.com.crt，这是生成的服务器证书文件。

    1.2.完整的证书链¶

    完整的证书链（包括服务器证书、中间证书和根证书）

    1. 生成根证书和中间证书¶

    假设您已经生成了根证书和中间证书，以下是示例命令：

    1.1.生成根证书¶
    openssl genrsa -out ca.key 2048

openssl req -new -x509 -days 3650 -key ca.key \
    -subj "/C=CN/ST=Shanghai/L=Shanghai/O=AI Team/OU=Personal/CN=RootCA" \
    -out ca.crt -utf8
    1.2.生成中间证书¶
    openssl genrsa -out intermediate.key 2048

openssl req -new -x509 -days 3650 -key intermediate.key \
    -subj "/C=CN/ST=Shanghai/L=Shanghai/O=AI Team/OU=Personal/CN=IntermediateCA" \
    -out intermediate.crt -utf8

    2. 生成服务器证书¶

    openssl genrsa -out server.key 2048

openssl req -newkey rsa:2048 -nodes -keyout server.key \
    -subj "/C=CN/ST=Shanghai/L=Shanghai/O=AI Team/OU=Personal/CN=local-dev.gitlab.com" \
    -out server.csr -utf8

openssl x509 -req -days 3650 -in server.csr -CA intermediate.crt -CAkey intermediate.key -CAcreateserial \
    -out local-dev.gitlab.com.crt

    3. 创建完整的证书链¶

    您需要将根证书、中间证书和服务器证书合并为一个文件，通常以 .crt 或 .pem 后缀保存：

    cat local-dev.gitlab.com.crt intermediate.crt ca.crt > local-dev.gitlab.com.pem

    4. 在服务器配置中使用证书链¶

    在您的服务器配置中（例如 Nginx 或 Apache），您需要指向这个完整的证书链文件和私钥文件：

    Nginx 示例

    listen 443 ssl;
    server_name local-dev.gitlab.com;

    ssl_certificate /path/to/local-dev.gitlab.com.pem;  # 完整证书链
    ssl_certificate_key /path/to/server.key;  # 服务器私钥

    # 其他配置...
}

    确保在 gitlab 服务器中运行gitlab-ctl reconfigure以重新加载新证书;

    • 容器部署环境:
    • 将新的自签名ca.crt添加到 gitlab-runner 容器的/usr/local/share/ca-certificates/下
    • 运行update-ca-certificates
    • gitlab-ctl restart
    • gitlab-runner register 重新注册runner
    • 服务器部署环境:
    • 添加ca证书受信任
    • gitlab更新ca证书, 重启服务
    • nginx
      • sudo systemctl restart nginx
    • apache2
      • sudo systemctl restart apache2

    6.证书续期¶

    6.1.设置证书的续期计划¶

    在生产环境中，自签名证书通常设置较长的有效期（如 10 年）。但是为安全起见，建议定期进行检查。可以使用 cron 或其他调度工具定期检查证书的有效性，并在过期前一个月内开始续期。

    例如，创建 check_cert_expiry.sh 脚本，每月检查一次证书是否即将过期：

    #!/bin/bash

# 检查 server.crt 过期时间
CERT_FILE="/path/to/local-dev.gitlab.com.crt"
DAYS_BEFORE_EXPIRY=30

# 获取证书的到期日期
EXPIRY_DATE=$(openssl x509 -enddate -noout -in "$CERT_FILE" | cut -d= -f2)
EXPIRY_TIMESTAMP=$(date -d "$EXPIRY_DATE" +%s)
CURRENT_TIMESTAMP=$(date +%s)
DAYS_LEFT=$(( (EXPIRY_TIMESTAMP - CURRENT_TIMESTAMP) / 86400 ))

# 如果证书即将过期，执行续期流程
if [ "$DAYS_LEFT" -le "$DAYS_BEFORE_EXPIRY" ]; then
  echo "Certificate is about to expire in $DAYS_LEFT days. Renewing..."
  # 调用证书续期脚本
  /path/to/renew_cert_chain.sh
else
  echo "Certificate is valid for another $DAYS_LEFT days."
fi

    然后设置 cron 任务，每月检查一次：

    0 0 1 * * /path/to/check_cert_expiry.sh
    6.2. 编写证书续期脚本¶

    创建一个 renew_cert_chain.sh 脚本，用于更新自签名证书链：

    #!/bin/bash

# 1. 重新生成根证书和中间证书
openssl req -new -x509 -days 3650 -key ca.key \
    -subj "/C=CN/ST=Shanghai/L=Shanghai/O=AI Team/OU=Personal/CN=RootCA" \
    -out ca.crt -utf8

openssl req -new -x509 -days 3650 -key intermediate.key \
    -subj "/C=CN/ST=Shanghai/L=Shanghai/O=AI Team/OU=Personal/CN=IntermediateCA" \
    -out intermediate.crt -utf8

# 2. 更新服务器证书
openssl req -newkey rsa:2048 -nodes -keyout server.key \
    -subj "/C=CN/ST=Shanghai/L=Shanghai/O=AI Team/OU=Personal/CN=local-dev.gitlab.com" \
    -out server.csr -utf8

openssl x509 -req -days 3650 -in server.csr -CA intermediate.crt -CAkey intermediate.key -CAcreateserial \
    -out local-dev.gitlab.com.crt

# 3. 生成新的完整证书链
cat local-dev.gitlab.com.crt intermediate.crt ca.crt > local-dev.gitlab.com.pem

# 4. 重启 Web 服务
echo "Restarting web server to apply the new certificates."
sudo systemctl restart nginx
    6.3. 验证续期流程¶

    • 测试脚本：在生产环境前，使用短期有效证书（如 7 天）验证 check_cert_expiry.sh 和 renew_cert_chain.sh 是否有效。

    • 验证证书：通过 openssl 工具检查证书链是否有效：

    openssl verify -CAfile ca.crt local-dev.gitlab.com.pem

    1.3.配置 GitLab 的 HTTPS 设置¶

    1.3.1.自签名配置¶

    1.Docker compose配置¶

    在 docker-compose.yml 文件中定义 GitLab 服务，配置 HTTPS 设置并挂载自签名证书。

    示例 docker-compose.yml

    version: "3.8"

services:
  gitlab:
    image: gitlab/gitlab-ee:latest   # 选择 GitLab 镜像，可用 `gitlab-ce` 或 `gitlab-ee`
    hostname: "yourdomain.com"
    environment:
      GITLAB_OMNIBUS_CONFIG: |
        external_url "https://yourdomain.com"
        nginx['redirect_http_to_https'] = true
        nginx['ssl_certificate'] = "/etc/gitlab/ssl/yourdomain.com.crt"
        nginx['ssl_certificate_key'] = "/etc/gitlab/ssl/yourdomain.com.key"
        nginx['ssl_protocols'] = "TLSv1.2 TLSv1.3"
        gitlab_rails['initial_root_password'] = "${GITLAB_ROOT_PASSWORD:-123456}"
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - "./gitlab/config:/etc/gitlab"
      - "./gitlab/logs:/var/log/gitlab"
      - "./gitlab/data:/var/opt/gitlab"
      - "./gitlab/config/ssl:/etc/gitlab/ssl" # 挂载自签名证书路径
    restart: always
    shm_size: 256m
    deploy:
        replicas: 1
      resources:
        limits:
          cpus: '2'
          memory: 4G
        reservations:
          cpus: '1'
          memory: 1G

    1. 在 GITLAB_OMNIBUS_CONFIG 环境变量中配置 GitLab 的 HTTPS 设置：

    2. external_url: 将 http://yourdomain.com 设置为 HTTPS 地址，即 https://yourdomain.com。

    3. nginx['redirect_http_to_https']: 启用 HTTP 到 HTTPS 的重定向。
    4. nginx['ssl_certificate'] 和 nginx['ssl_certificate_key']: 指定证书和密钥文件路径。

    5. nginx['ssl_protocols']: 只启用 TLS 1.2 和 TLS 1.3 协议，以提高安全性。

    6. Docker资源分配限制:

    7. 部署 1 个副本（replicas: 1）。

    8. resources: resources 允许您定义服务在运行时的资源限制和保留。

    9. limits：定义服务可以使用的最大资源量。例如，如果您设置了 CPU 和内存的限制，那么服务不能超出这些限制。

    10. reservations：定义服务在调度时所需的最低资源量。Docker 将确保在调度时为服务保留这些资源，即使服务当前不使用这些资源。

    11. cpus 和 memory

    12. cpus：可以是小数，表示服务可以使用的 CPU 核心数。例如，'2' 表示最多使用 2 个 CPU 核心，'0.5' 表示最多使用半个 CPU 核心。

    13. memory：设置内存限制。可以使用字母后缀，如 M（兆字节）或 G（千兆字节）。例如，4G 表示最多使用 4GB 内存。

    2.在 GitLab 容器中信任自签名证书¶

    为了让 GitLab 及其组件信任自签名证书，还需将根证书 (server.crt) 添加到 GitLab 容器的受信任证书存储中。

    docker cp server.crt <container_name>:/usr/local/share/ca-certificates/server.crt
    • 在容器内更新受信任的 CA 证书：
    docker exec -it <container_name> update-ca-certificates

    这样 GitLab 容器内的服务（如 GitLab Runner）将信任自签名证书。

    • 重新配置
    gitlab-ctl reconfigure
    • 更新证书
    sudo gitlab-ctl hup nginx
sudo gitlab-ctl hup registry

    1.4.配置客户端信任自签名证书¶

    由于使用的是自签名证书，访问 GitLab 服务时浏览器会提示不安全连接。为避免该提示，您可以在客户端手动信任证书：

    3.2. 资源需求分析¶

    a. 内存¶

    • GitLab 运行时需要较多的内存，特别是在使用 CI/CD 功能时，建议至少 8 GB 以上的内存，以确保平稳运行。

    b. CPU¶

    • CPU 核心数会影响并发处理能力。对于小团队，2 核心是最低要求；对于中型团队，推荐使用 4 核心或更多，尤其在使用 CI/CD 时，能提高构建速度。

    c. 磁盘¶

    • 磁盘空间需求取决于项目的数量和大小。默认情况下，GitLab 自身需要约 4-5 GB 的空间，但您可能需要为仓库和 CI/CD 产生的构建工件保留额外空间。建议为每个项目至少预留 1 GB 的空间。

    3.3. 其他注意事项¶

    • 网络带宽: 确保有稳定的网络连接，尤其在团队协作和 CI/CD 过程中。
    • 备份与恢复: 需要考虑定期备份的策略和存储需求。
    • 负载均衡: 如果有多个用户并发访问，可能需要考虑使用负载均衡器。

    2.GitLab Runner CI 流程步骤¶

    2.1.获取token方式¶

    1. 获取实例级别的 Registration Token

    2. 适用于希望注册为共享 Runner 的情况（即适用于整个 GitLab 实例的所有项目）。

    3. 步骤：
    4. 使用管理员账号登录 GitLab。
    5. 进入 Admin Area（点击头像，选择 Admin）。
    6. 在左侧菜单选择 Runners。

    7. 在页面顶部的 Instance Runners 区域下，找到 Registration Token。这个令牌可以用于注册全局可用的 Runner。

    8. 获取组级别的 Registration Token

    9. 适用于只允许在某个 GitLab 组中的项目使用 Runner。

    10. 步骤：
    11. 登录 GitLab，进入目标组页面。
    12. 在左侧菜单选择 Settings > CI/CD。
    13. 展开 Runners 设置部分。

    14. 在 Group Runners 区域下，可以找到该组的 registration_token，用它注册的 Runner 仅对该组内的项目可用。

    15. 获取项目级别的 Registration Token

    16. 适用于仅允许在特定项目中使用 Runner 的情况。

    17. 步骤：
    18. 登录 GitLab，进入目标项目页面。
    19. 在左侧菜单选择 Settings > CI/CD。
    20. 展开 Runners 设置部分。
    21. 在 Project Runners 区域下，找到 Registration Token。该令牌仅对该项目有效。
    2.2.CI流程¶
    1. Runner 注册
    2. 使用 GitLab 管理员提供的 registration_token 注册 Runner。
    3. Runner 调用 GitLab API 的 POST /api/v4/runners 接口，并传入 registration_token。
    4. GitLab 验证令牌并返回 runner_token，用于后续认证。
    5. 获取作业（Job Requesting）
    6. Runner 启动并循环地请求新作业。
    7. Runner 调用 POST /api/v4/jobs/request 接口，携带 runner_token。
    8. GitLab 验证 Runner 并返回可用的作业信息（包括项目的 CI 配置、环境变量、认证信息等）。
    9. 设置工作环境
    10. Runner 根据作业配置克隆项目代码库。
      • 使用 job_token 认证 HTTPS 克隆或 SSH 克隆（如果配置了 SSH）。
    11. Runner 下载作业依赖的工件（artifact）和缓存（cache）。
    12. 执行作业脚本
    13. Runner 读取作业配置中的 .gitlab-ci.yml 文件，并按照其中定义的步骤执行 CI/CD 作业。
    14. 执行过程中，Runner 会输出日志和状态。
    15. 上传工件与更新状态
    16. 如果作业生成了新工件（例如构建文件），Runner 会使用 job_token 调用 GitLab API 上传这些工件。
    17. Runner 将作业的执行结果和状态报告给 GitLab，包括成功、失败或取消等状态。
    18. 清理工作环境
    19. 作业执行完成后，Runner 清理工作目录，删除中间文件和工件缓存，确保下一次作业从干净的环境开始。

    • build
    • test
    • deploy
    • .post

    中项目的顺序stages定义了作业的执行顺序：

    • 同一阶段的作业并行运行。
    • 上一阶段的作业成功完成后，下一阶段的作业将运行。

  - deploy

    用于rules包含或排除管道中的作业。

    在创建管道时会评估规则，并按顺序进行评估。找到匹配项后，将不再检查其他规则，并且根据配置将作业包含或排除在管道之外。如果没有匹配的规则，则不会将作业添加到管道中。

    在 GitLab CI/CD 配置中，当在作业级别定义了 rules 时，该作业会优先使用作业本身的 rules，而不会继承 workflow 中的 rules。也就是说，作业级别的 rules 会覆盖 workflow 的规则，workflow 仅决定整个流水线是否应被创建和运行，但不会影响单个作业的触发条件。

    rules接受规则数组。每条规则必须至少包含以下一项：

    • if
    • changes
    • exists
    • when

    规则还可以选择性地与以下内容组合：

    • allow_failure
    • needs
    • variables
    • interruptible

    您可以将多个关键字组合在一起以形成复杂的规则。

    该作业已添加到管道：

    • 如果if、changes或exists规则匹配，并且配置为when: on_success(如果未定义则为默认)、 when: delayed、 或when: always。
    • 如果达到的规则仅仅是when: on_success，，when: delayed或when: always。

    该作业未添加到管道：

    • 如果没有规则匹配。
    • 如果某条规则匹配且有when: never。

    4.3.5.一些关键字的理解¶

    1.dependencies与needs¶
    1.1. needs¶

    needs 用于声明当前作业需要依赖的其他作业。它允许作业不按阶段顺序执行，而是根据依赖关系并行运行，显著提高了 CI/CD 流水线的并行化程度。适合希望尽早开始执行、减少等待时间的作业。

    关键特点:

    • 并行性：使用 needs 的作业可以跨阶段执行，而不必按顺序依赖前一个阶段。例如，一个 deploy 作业可以直接依赖于 build 而跳过 test 阶段。
    • 定义依赖关系：可以直接指定需要的作业名称，并能跨阶段引用。
    • 默认下载缓存：needs 仅下载指定依赖的作业的缓存和工件。

    配置示例:

    stages:
  - build
  - test
  - deploy

build_job:
  stage: build
  script:
    - echo "Building the project"
  artifacts:
    paths:
      - build/

test_job:
  stage: test
  script:
    - echo "Testing the project"
  needs:
    - job: build_job

deploy_job:
  stage: deploy
  script:
    - echo "Deploying the project"
  needs:
    - job: build_job
    - job: test_job

    在上面的配置中：

    • test_job 和 deploy_job 都依赖于 build_job，因此可以在 build_job 完成后并行开始。
    • deploy_job 还依赖于 test_job，因此会等待 test_job 完成。

    适用场景:

    • 当作业具有跨阶段依赖关系时，需要使用 needs 优化执行顺序。
    • 希望加快 CI/CD 流程，在确保依赖作业完成后立即开始执行。
    1.2. dependencies¶

    dependencies 用于指定当前作业可以下载哪些上游作业生成的工件（artifacts）。与 needs 不同的是，它并不改变作业的执行顺序，仅决定是否下载上游作业的工件。适合需要访问特定上游作业的工件的场景。

    关键特点:

    • 不改变执行顺序：dependencies 不会影响作业的执行顺序，它只在作业开始执行时下载指定的工件。
    • 工件的下载控制：默认情况下，一个作业会自动下载前一阶段所有作业的工件，但 dependencies 允许手动指定要下载的作业工件。
    • 工件管理：可以减少不必要的工件下载，优化 CI/CD 流程的资源使用。

    配置示例:

    stages:
  - build
  - test
  - deploy

build_job:
  stage: build
  script:
    - echo "Building the project"
  artifacts:
    paths:
      - build/

test_job:
  stage: test
  script:
    - echo "Testing the project"
  dependencies:
    - build_job

deploy_job:
  stage: deploy
  script:
    - echo "Deploying the project"
  dependencies:
    - build_job

    在此配置中：

    • test_job 只下载 build_job 的工件，而不是所有前一阶段作业的工件。
    • deploy_job 同样仅下载 build_job 的工件，避免下载不必要的工件。

    适用场景:

    • 当作业需要特定上游作业的工件，而不想下载所有上游作业工件时。
    • 希望避免不必要的下载操作，从而提升执行效率。

    总结:

    特性 needs dependencies
    执行顺序 允许作业跨阶段执行，按依赖关系并行 不影响执行顺序，仅指定工件下载
    主要用途 声明依赖关系、优化流水线并行性 控制工件的下载，减少不必要的下载操作
    跨阶段依赖 支持跨阶段依赖 不支持跨阶段依赖
    下载控制 默认下载所依赖的作业工件 只下载指定的上游作业工件

    使用时：

    • 若想加快流水线速度，允许作业并行执行，使用 needs。
    • 若只需指定工件下载而不改变执行顺序，使用 dependencies。

    2.变量安全

    在 GitLab CI/CD 流程中，为了安全地使用和保护用户名、密码等敏感信息，通常使用 GitLab CI/CD Secret Variables。这些变量在 CI/CD 管道中可以被访问但不会被明文打印或记录，以确保安全性。以下是详细说明如何安全地获取和使用这些信息：

    4.6.1. 配置 GitLab CI/CD Secret Variables¶

    • 转到项目页面的 Settings > CI/CD。
    • 展开 Variables，点击 Add variable。
    • 添加变量名和值，例如 DOCKER_USERNAME、DOCKER_PASSWORD 等，设置为 Masked 和 Protected，确保它们不会在 CI 日志中以明文显示，且只能用于受保护的分支或标签。

    4.6.2. 在 .gitlab-ci.yml 中使用 Secret Variables¶

    在 GitLab CI/CD 脚本中，可以直接使用这些变量，例如：

    stages:
  - build
  - deploy

build:
  stage: build
  script:
    - echo "Logging in to Docker"
    - docker login -u "$DOCKER_USERNAME" -p "$DOCKER_PASSWORD" $DOCKER_REGISTRY
    - docker build -t $DOCKER_IMAGE .
    - docker push $DOCKER_IMAGE
    - docker logout $DOCKER_REGISTRY

deploy:
  stage: deploy
  script:
    - echo "Deploying to server"
    - sshpass -p "$SERVER_PASSWORD" ssh "$SERVER_USER@$SERVER_HOST" <<EOF
        docker pull $DOCKER_IMAGE
        docker run -d $DOCKER_IMAGE
      EOF

    4.6.3. 最佳实践¶

    • 环境隔离：使用 Protected Variables 来确保敏感信息只在特定分支（如 main 或 release）中可用。

    • 掩码保护：启用 Masked 选项，防止敏感信息被日志暴露。

    • 限制权限：在 GitLab 的 Access Control 设置中，限制谁可以更改和查看 CI/CD 变量。

    • 使用 .git-credentials 文件：如果需要在多个命令中重复使用凭据，可以在 CI 中临时创建并使用 .git-credentials 文件（确保妥善清理）：

    before_script:
  - echo "https://${GIT_USERNAME}:${GIT_PASSWORD}@gitlab.com" > ~/.git-credentials
  - git config --global credential.helper store

    4.6.4. 外部凭据存储¶

    对于更高的安全需求，使用外部凭据存储，如 HashiCorp Vault 或 AWS Secrets Manager，并通过 GitLab CI/CD 集成插件读取凭据。

    stages:
  - secure

secure-stage:
  stage: secure
  script:
    - echo "Fetching credentials from Vault"
    - export PASSWORD=$(vault read -field=password secret/my-app)

    4.6.5. 附加安全措施¶

    • 防止输出：在脚本中避免通过 echo 或日志打印敏感信息。
    • 设置过期时间：使用 GitLab 的选项设置 CI/CD Variables 的过期时间或定期更新凭据。

    4.6.6.SSH密钥¶

    在 GitLab CI/CD 中使用 SSH 密钥

    1.工作原理¶
    1. 在本地创建一个新的 SSH 密钥对 ssh-keygen。
    2. 将私钥作为变量添加到您的项目中。
    3. 在作业期间运行 ssh-agent 来加载私钥。
    4. 如果您正在访问私有仓库，将公钥复制到您想要访问的服务器（通常在 ~/.ssh/authorized_keys 中）或将其添加为部署密钥。
    2.示例1: GitHub-ssh免密登录¶
    3.示例2¶
    1. 生成 SSH 密钥对¶

    在你的本地机器上生成 SSH 密钥对：

    ssh-keygen -t rsa -b 4096 -C "your_email@example.com"

    这将创建一对密钥，通常位于 ~/.ssh/id_rsa 和 ~/.ssh/id_rsa.pub。

    2. 将公钥添加到远程服务器¶

    将本机(runner所在)生成的公钥 id_rsa.pub 添加到远程服务器的 ~/.ssh/authorized_keys 文件中：

    ssh-copy-id user@remote-server

    或手动复制 id_rsa.pub 的内容到 authorized_keys 中。

    3. 配置 GitLab CI/CD¶

    将私钥添加为 GitLab 的 CI/CD 变量：

    1. 进入项目设置：
    2. 打开 GitLab 项目，转到 Settings > CI/CD。
    3. 添加变量：
    4. 展开 Variables，点击 Add variable。
    5. 变量名为 SSH_PRIVATE_KEY，值为生成的 id_rsa 私钥内容。
    6. 将变量设置为保护的（Protected），如果只想在受保护的分支上使用。
    4. 配置 .gitlab-ci.yml¶

    在 GitLab CI 配置文件中使用密钥进行部署：

    stages:
  - deploy

deploy_job:
  stage: deploy
  script:
    - mkdir -p ~/.ssh
    - echo "$SSH_PRIVATE_KEY" | tr -d '\r' > ~/.ssh/id_rsa
    - chmod 600 ~/.ssh/id_rsa
    - ssh-keyscan -H remote-server >> ~/.ssh/known_hosts
    - ssh user@remote-server 'bash -s' < deploy-script.sh
    • mkdir -p ~/.ssh：创建 .ssh 目录。
    • echo "$SSH_PRIVATE_KEY" | tr -d '\r' > ~/.ssh/id_rsa：将私钥写入文件。
    • chmod 600 ~/.ssh/id_rsa：设置私钥权限。
    • ssh-keyscan：将远程服务器的 SSH 主机公钥(对输出中的所有主机名和地址进行哈希处理)添加到 known_hosts，避免首次连接时的提示。
    • ssh user@remote-server 'bash -s' < deploy-script.sh：在远程服务器上执行部署脚本。

    4.7.Docker in Docker¶

    4.7.1.基本用法¶

    要在 GitLab CI/CD 中启用 DinD，通常需要将 docker:dind (Docker-in-Docker) 作为服务来使用，并确保作业中的 docker 客户端可以访问到 dind 容器。

    以下是一个典型的 .gitlab-ci.yml 配置文件示例：

    image: docker:latest  # 使用 Docker 官方镜像作为主容器

services:
  - docker:dind  # 启用 Docker-in-Docker 服务

variables:
  DOCKER_HOST: "tcp://docker:2375"  # 指定 Docker 服务主机和端口
  DOCKER_TLS_CERTDIR: ""  # 禁用 Docker 的 TLS 校验（在非生产环境中可用）

stages:
  - build
  - deploy

# 构建镜像的作业
build:
  stage: build
  script:
    - docker info  # 查看 Docker 信息
    - docker build -t my-image:latest .  # 构建 Docker 镜像
    - docker tag my-image:latest myregistry.com/my-image:latest  # 打标签
    - docker login -u "$DOCKER_USER" -p "$DOCKER_PASSWORD" myregistry.com  # 登录 Docker 注册表
    - docker push myregistry.com/my-image:latest  # 推送到私有仓库

# 部署作业（视需求定义）
deploy:
  stage: deploy
  script:
    - echo "部署操作，可以在此执行后续步骤。"

    关键点说明:

    • 主镜像 (image)：使用 docker:latest 或其他适合的 Docker 客户端镜像，确保可以在主容器中运行 docker 命令。
    • 服务 (services)：将 docker:dind 添加到 services 中，以启动一个 Docker-in-Docker 的守护进程。
    • DOCKER_HOST 变量：定义 DOCKER_HOST 环境变量，告诉 docker 客户端连接 dind 服务的地址。
    • DOCKER_TLS_CERTDIR=""：默认情况下，docker:dind 启用了 TLS。如果不使用 TLS，请将此变量设为空，以便跳过 TLS 校验（仅适合非生产环境）。

    注意事项:

    • 性能：DinD 模式会有一些额外的开销，特别是在多层构建过程中，可能会降低 CI/CD 作业的执行速度。
    • 安全性：DinD 的实现对安全性要求较高，因为在容器中启动 Docker 引擎实际上增加了宿主机暴露的风险，通常建议在非生产环境或受控环境中使用。
    • 缓存管理：由于每次运行都会重新拉取基础镜像，为了加快构建速度，可以通过缓存构建层或共享注册表来提高效率。

    应用场景:

    DinD 适合以下场景：

    1. 持续构建和推送镜像：自动构建并推送镜像到注册表。
    2. 集成测试：在 CI/CD 作业中创建和运行容器环境以进行测试。
    3. 多阶段构建：在一个构建中构建和测试镜像，然后将其用于后续的部署阶段。

    4.7.2.启用TLS¶

    默认情况下，docker:dind 会启用 TLS，以确保 Docker 守护进程与 Docker 客户端之间的通信是加密的。

    1. 配置 .gitlab-ci.yml 文件¶

    使用 TLS 时，需要将 DOCKER_TLS_CERTDIR 设置为默认目录 /certs（这是 docker:dind 容器内的默认路径）并确保客户端容器可以访问这些证书。

    image: docker:20.10  # 使用支持 TLS 的 Docker 镜像

services:
  - name: docker:dind
    alias: docker  # 方便在 DOCKER_HOST 中引用

variables:
  DOCKER_HOST: tcp://docker:2376  # TLS 端口
  DOCKER_TLS_CERTDIR: "/certs"  # 启用 TLS 证书目录

stages:
  - build
  - deploy

# 构建镜像的作业
build:
  stage: build
  script:
    - export DOCKER_CERT_PATH=/certs/client  # 指定客户端证书路径
    - export DOCKER_TLS_VERIFY=1  # 启用 TLS 校验
    - docker info  # 验证 Docker 配置
    - docker build -t my-image:latest .  # 构建 Docker 镜像
    - docker login -u "$DOCKER_USER" -p "$DOCKER_PASSWORD" myregistry.com  # 登录 Docker 注册表
    - docker push myregistry.com/my-image:latest  # 推送到注册表

    关键配置说明:

    • DOCKER_HOST：设置为 tcp://docker:2376，这是 Docker TLS 模式的默认端口。
    • DOCKER_TLS_CERTDIR：指定证书存储目录。docker:dind 默认启用 TLS，证书会自动生成在 /certs 目录中。
    • DOCKER_CERT_PATH：Docker 客户端的证书路径，设置为 /certs/client。
    • DOCKER_TLS_VERIFY：启用 Docker 客户端的 TLS 校验。

    证书的管理:

    在默认情况下，docker:dind 会生成 CA、服务器和客户端证书，并将这些证书放在 /certs 目录下。客户端通过该路径进行安全访问。

    2.启用 TLS 与禁用 TLS 的应用场景¶
    配置方式 优点 缺点 适用场景
    启用 TLS 安全性更高，数据通信加密 配置复杂度更高，依赖证书管理 生产环境、对安全性要求较高的环境
    禁用 TLS 配置简单，无需证书管理，速度更快 安全性较低，不适合公网环境 开发、测试环境或内部受控环境
    • 启用 TLS：推荐用于生产环境，特别是在公共云或存在敏感数据的项目中，以确保数据传输的安全性。
    • 禁用 TLS：适用于本地开发、测试或内部受控的 CI/CD 环境，简化了配置和执行流程，有助于加快测试。

    4.8.4.示例4: 打包发布¶

    在 GitLab CI 中实现打包压缩文件、上传发布、并生成永久链接和 latest 标识的流程，可以按照以下步骤实现。

    实现方案:

    1. 定义打包阶段：将代码或编译成果打包成压缩文件。
    2. 上传和发布：将压缩包上传到远程存储。
    3. 创建永久链接：为每次发布的压缩包创建一个独立链接。
    4. 更新 latest 标识：发布成功后将最新的压缩包标记为 latest，实现最新版本的快速访问。
    1..gitlab-ci.yml 配置¶
    stages:
  - build
  - package
  - upload
  - publish

variables:
  PACKAGE_NAME: "my_project"                   # 项目名称
  TIMESTAMP: "$(date +%Y%m%d%H%M%S)"           # 时间戳用于唯一标识
  PACKAGE_FILE: "$PACKAGE_NAME-$TIMESTAMP.tar.gz"
  UPLOAD_PATH: "/path/to/remote/storage"       # 远程存储路径，视环境调整

# Step 1: 打包
package_job:
  stage: package
  script:
    - echo "Creating package..."
    - tar -czvf $PACKAGE_FILE ./                # 压缩项目文件
    - mkdir -p dist                             # 创建本地存储目录
    - mv $PACKAGE_FILE dist/                    # 将压缩包移动到 dist 目录
  artifacts:
    paths:
      - dist/$PACKAGE_FILE                      # 打包后的文件作为 artifacts 供后续作业使用

# Step 2: 上传压缩包
upload_job:
  stage: upload
  dependencies:
    - package_job
  script:
    - echo "Uploading $PACKAGE_FILE..."
    - scp dist/$PACKAGE_FILE user@remote_server:$UPLOAD_PATH  # 上传到远程服务器
  only:
    - tags  # 只在有标签的发布中上传

# Step 3: 创建链接和 latest 标识
publish_job:
  stage: publish
  dependencies:
    - upload_job
  script:
    - |
      echo "Setting permanent link and latest tag..."
      # 创建永久链接
      ssh user@remote_server "ln -sfn $UPLOAD_PATH/$PACKAGE_FILE $UPLOAD_PATH/$PACKAGE_NAME-$TIMESTAMP"
      # 更新最新版本的 latest 标识
      ssh user@remote_server "ln -sfn $UPLOAD_PATH/$PACKAGE_FILE $UPLOAD_PATH/$PACKAGE_NAME-latest"
  only:
    - tags  # 只在有标签的发布中执行
    2.配置说明¶
    • 文件名和时间戳标识：

    • 通过变量 TIMESTAMP 生成一个唯一的时间戳后缀，使每个压缩包文件名都唯一，便于存储和回溯历史版本。

    • 发布和 latest 标识：

    • 在发布阶段使用 ln -sfn 命令创建符号链接，实现指向最新版本文件的 latest 标识，方便用户始终通过统一链接获取最新发布版本。

    • 安全性：

    • 使用 CI/CD 变量存储敏感信息（如服务器登录凭据、API 密钥等），避免将凭据直接写入脚本。

    实现方案概述:

    1. 打包阶段：将项目内容打包并作为 artifacts 上传，确保在后续步骤中可以访问该文件。
    2. 发布阶段：使用 GitLab API 创建 Release，并在 Release 中添加永久下载链接。通过 latest 标识管理最新版本。
    stages:
  - build
  - package
  - release

variables:
  PACKAGE_NAME: "my_project"                   # 项目名称
  TIMESTAMP: "$(date +%Y%m%d%H%M%S)"           # 时间戳用于唯一标识
  PACKAGE_FILE: "$PACKAGE_NAME-$TIMESTAMP.tar.gz"

# Step 1: 打包
package_job:
  stage: package
  script:
    - echo "Creating package..."
    - tar -czvf $PACKAGE_FILE ./               # 压缩项目文件
  artifacts:
    paths:
      - $PACKAGE_FILE                          # 将打包文件作为 artifact 存储
    expire_in: 1 week                          # 设置 artifacts 的有效期

# Step 2: 发布并添加下载链接
release_job:
  stage: release
  dependencies:
    - package_job
  script:
    - |
      echo "Creating release and adding download link..."
      # 获取项目的 artifacts 下载 URL
      ARTIFACT_URL="${CI_PROJECT_URL}/-/jobs/${CI_JOB_ID}/artifacts/download?file_type=package&filename=$PACKAGE_FILE"
      # 使用 GitLab API 创建 Release
      curl --request POST --header "PRIVATE-TOKEN: $CI_JOB_TOKEN" \
        --header "Content-Type: application/json" \
        --data '{
          "name": "Release '"$CI_COMMIT_TAG"'",
          "tag_name": "'"$CI_COMMIT_TAG"'",
          "description": "Release for '"$CI_COMMIT_TAG"'",
          "assets": {
            "links": [
              {
                "name": "'"$PACKAGE_NAME-latest"'",
                "url": "'"$ARTIFACT_URL"'"
              }
            ]
          }
        }' "${CI_API_V4_URL}/projects/${CI_PROJECT_ID}/releases"
  only:
    - tags   # 仅在有标签的情况下发布

    实现详解:

    1. 使用 GitLab Artifacts：
    2. artifacts 在 package_job 中保存了打包文件，并通过 $CI_JOB_ID 提供的链接允许下载。
    3. 设置 expire_in 可以指定 artifacts 的有效期。
    4. 创建 Release 并添加 latest 标识：
    5. release_job 阶段通过 GitLab API 创建一个带有下载链接的 Release。
    6. 通过 GitLab API 中的 Release assets，添加一个下载链接，并命名为 latest，用于指向最新版本的下载地址。
    7. 敏感信息安全管理：
    8. 使用 GitLab CI/CD 的 $CI_JOB_TOKEN 来代替手动 API token。这种方法无需显式存储或管理敏感信息，保障了安全性。

    5.问题检查¶

    1. 检查套接字文件¶

    确认套接字文件存在并且权限正确。运行以下命令：

    ls -l /var/opt/gitlab/gitlab-rails/sockets/gitlab.socket

    如果该文件不存在，可能是 gitlab-rails 没有成功启动。确保文件存在并且其拥有者是 git 用户，权限应为 770：

    sudo chown git:git /var/opt/gitlab/gitlab-rails/sockets/gitlab.socket
sudo chmod 770 /var/opt/gitlab/gitlab-rails/sockets/gitlab.socket

    2. 查看 GitLab 日志¶

    检查 GitLab 的日志以获取更多信息。重点查看以下日志文件：

    • gitlab-rails.log:
    sudo tail -f /var/log/gitlab/gitlab-rails/gitlab-rails.log
    • nginx_access.log 和 nginx_error.log:
    sudo tail -f /var/log/gitlab/nginx/gitlab_access.log
sudo tail -f /var/log/gitlab/nginx/gitlab_error.log
    • gitlab日志
    gitlab-ctl tail

    3. 重新配置 GitLab¶

    尝试重新配置 GitLab，这可以修复潜在的配置问题：

    sudo gitlab-ctl reconfigure

# 升级
gitlab-ctl upgrade

sudo gitlab-rake db:migrate

    7.CI流程无法访问 GitLab 服务器¶

    在使用自签名证书的 GitLab 服务器时，GitLab CI/CD Runner 可能会因为无法验证自签名证书而连接失败。为了解决这个问题，可以通过以下步骤配置 GitLab Runner 以信任自签名证书。

    1.解决方案 1：在 Runner 中添加自签名证书¶

    1. 提取自签名证书 获取 GitLab 服务器上的自签名证书并保存为 .crt 文件。例如，在 GitLab 服务器上运行：
    sudo openssl s_client -showcerts -connect local-dev.gitlab.com:2143 </dev/null 2>/dev/null | openssl x509 -outform PEM > gitlab-selfsigned.crt
    1. 将证书添加到 Runner 容器 在 GitLab Runner 配置中，将证书文件挂载到 Runner 容器内的信任目录中。 编辑 docker-compose.yml 文件（或 Runner 的配置文件）来添加证书挂载：
    gitlab-runner:
  image: gitlab/gitlab-runner:latest
  volumes:
    - /path/to/gitlab-selfsigned.crt:/etc/gitlab-runner/certs/local-dev.gitlab.com.crt

    • 将 /path/to/gitlab-selfsigned.crt 替换为证书实际路径。

    • 重新启动 Runner 重新启动 GitLab Runner 容器以应用新的挂载：

    docker-compose restart gitlab-runner

    2.解决方案 2：在 .gitlab-ci.yml 中禁用 SSL 验证（仅在测试中使用）¶

    在 GitLab CI 配置文件 .gitlab-ci.yml 中，可以在初始化脚本中禁用 SSL 验证。请注意，这个方法不推荐在生产环境中使用，因为会降低安全性。

    before_script:
  - git config --global http.sslVerify false

    3.解决方案 3：环境变量中禁用 SSL 验证¶

    在特定作业中，可以设置 GIT_SSL_NO_VERIFY 环境变量来忽略 SSL 验证错误。

    variables:
  GIT_SSL_NO_VERIFY: "true"
