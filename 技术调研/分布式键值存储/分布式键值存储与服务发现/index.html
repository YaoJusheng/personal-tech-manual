    一、分布式键值存储（Distributed Key-Value Store）¶

    分布式键值存储是将数据以 key-value 格式 存储在多个节点上，通过 副本复制、一致性协议和容错机制 实现高可用、高一致性的数据库。

    典型代表： Etcd、Consul KV、Zookeeper、Redis Cluster、TiKV

    1. 核心目标¶

    • 高可用：任意节点宕机系统仍可服务
    • 高一致性（通常为强一致性）：所有节点对同一个 key 的值一致
    • 水平扩展：通过增加节点提升容量或吞吐
    • 容错恢复：节点重启、网络分区可自动恢复

    2. 基本原理¶

    分布式 KV 必须解决两个关键问题：

    (1) 数据复制¶

    数据需要在多个节点之间复制，使其具有冗余和高可用性。

    复制方式可分为：

    • 同步复制
    • 异步复制
    • 半同步复制（常见）

    (2) 一致性协议¶

    最关键的能力是 分布式一致性（Consensus） 主流协议：

    • Raft（易实现，现代系统主流）
    • Paxos（复杂，经典，但实现困难）
    • ZAB（Zookeeper 专用）

    以 Raft 为例：

    • 一个 Leader + 多个 Follower
    • 写操作全部经过 Leader 排序与提交
    • 大多数节点（过半）确认后才生效 → 强一致

    (3) MVCC（多版本并发控制）¶

    高级的 KV Store（如 Etcd）支持：

    • 每次写入都生成新 Revision
    • 读操作可基于版本读取
    • 可 watch 任意版本之后的事件

    这让它适合配置管理、主从选举、调度系统等。

    (4) Watch 机制（实时监听）¶

    分布式 KV 通常提供 Watch，用于监听 key 变化事件：

    • 节点注册/注销
    • 配置变更
    • 锁状态变化

    二、服务发现（Service Discovery）¶

    服务发现是微服务架构、容器平台和分布式系统中用于 自动识别和定位服务节点 的机制。

    1. 为什么需要服务发现？¶

    在分布式架构中，服务实例会频繁发生变化：

    • 弹性扩容/缩容
    • 宕机或重启 -动态部署或滚动发布

    因此不能依赖固定配置的 IP/PORT，必须依赖 动态注册+查询 的机制。

    2. 服务发现的核心流程¶

    可以抽象为以下步骤：

    (1) 服务注册（Register）¶

    服务实例启动时，将自身信息写入注册中心，如：

    service_name = "user-service"
key = "/services/user-service/instance-1"
value = { ip: "10.0.1.2", port: 8080, status: "healthy" }

    利用 TTL / Lease 保证自动过期。

    (2) 心跳与健康检查（Health Check）¶

    服务端定期刷新 lease，使其 key 不会过期。 Consul、Nacos 还支持主动/被动健康检查。

    (3) 服务发现（Discovery / Lookup）¶

    调用方查询某个 service_name 所有可用实例：

    • 客户端直连注册中心
    • 或使用内置的代理/DNS

    (4) 负载均衡（Load Balancing）¶

    得到服务实例列表后，客户端或网关需要进行：

    • 轮询
    • 权重分配
    • 基于健康状态过滤

    三、分布式键值存储与服务发现的关系¶

    分布式服务发现体系本质上高度依赖 分布式 KV。

    服务发现 = 服务注册 + 服务状态管理 + 服务列表查询 → 都是 key-value 操作。

    因此行业中几乎所有的服务发现组件，都基于分布式 KV：

    服务发现组件 底层是否使用分布式 KV KV 组件
    Etcd Yes Etcd 自身
    Consul Yes Consul KV + Raft
    Zookeeper Yes ZNode + ZAB
    Nacos Yes，但一致性可调 Nacos DB + 临时节点

    服务发现需要：

    • 高可用 → 多副本
    • 强一致性（可选） → Leader 写
    • Watch + TTL → 节点状态跟踪
    • 事务支持（可选） → 并发注册时避免冲突

    而这些都是分布式 KV 的核心能力。

    四、相关组件¶

    1.组件比较¶

    ⚔️ 主流组件全方位对比¶

    下表从多个维度对比了主流的分布式协调和服务发现组件，助你直观了解其差异。

    组件 一致性协议 数据模型 健康检查 主要优势 主要局限 典型应用场景
    ETCD Raft 键值对（有层次） 基于Lease/TTL 强一致性、高性能 、Watch机制完善 纯KV，功能较基础 Kubernetes大脑、服务发现、配置管理
    Zookeeper Zab (类似Paxos) 树形文件系统 自定义心跳 功能丰富（内置分布式锁、队列等）、成熟稳定 选举时服务可能短暂不可用、扩展性较差 Hadoop, Kafka, Solr 等大数据生态
    Consul Raft 键值对 丰富多样（TCP/HTTP等） 功能全面（集成服务发现、配置、安全网格）、强一致性 服务注册相对较慢、Leader故障需重选 微服务架构中的服务发现与配置管理
    Nacos Raft (针对配置) 键值对、服务元数据 客户端上报/服务端检查 集注册中心与配置中心于一体、动态配置、易用性高 云原生应用 ，需要动态配置管理的微服务
    Eureka —— (最终一致) 服务实例数据 客户端心跳 高可用、注册快速、简单易用、AP特性 不保证强一致性、无配置管理功能 Spring Cloud生态 ，追求高可用和最终一致性的场景

    🎯 组件选型指南¶

    了解了各组件的特性后，你可以根据以下场景进行选择：

    • 首选ETCD的场景：
    • 作为 Kubernetes的元数据存储，这是ETCD的经典应用。
    • 需要 强一致性 且 高性能 的分布式键值存储，用于服务发现或配置共享。
    • 系统架构基于 gRPC，ETCD原生支持，交互高效。
    • 考虑Zookeeper的场景：
    • 已有系统（如Kafka、Hadoop）依赖其作为协调核心。
    • 需要用到其内置的高级分布式原语，如分布式锁、队列等。
    • 选择Consul的场景：
    • 需要 开箱即用、功能全面 的服务发现和配置管理解决方案，尤其在企业级微服务架构中。
    • 对 网络安全性 有较高要求，Consul提供了服务网格支持。
    • 青睐Nacos的场景：
    • 处于 云原生 环境，需要将 服务注册发现 和 动态配置管理 统一解决的场景。
    • 追求 易于使用和运维，Nacos在易用性和功能丰富度上平衡较好。
    • 使用Eureka的场景（请谨慎评估）：
    • 项目基于 Spring Cloud 传统套件，且不要求强一致性，可以接受最终一致性。
    • 注重点在于服务的高可用性和快速的实例注册。

    请注意，Eureka 2.0已停止开发，在新项目中选型时需充分考虑其未来风险。

    📊 关键决策因素¶

    综合来看，在做最终决策时，请重点考量以下几点：

    1. 一致性需求：是否需要强一致性（CP）？ETCD、Zookeeper、Consul（默认）是稳妥选择。若能接受最终一致性（AP），Eureka或Nacos（在某些模式下）可以考虑。
    2. 功能定位：是单纯需要分布式协调和元数据存储（ETCD、Zookeeper），还是需要一体化的服务发现与配置中心（Consul、Nacos）。
    3. 生态与社区：考虑与现有技术栈（如Kubernetes、Spring Cloud）的集成度，以及组件的社区活跃度和学习成本。ETCD在云原生领域势头强劲，而Zookeeper在大数据领域积淀深厚。

    2.组件分析¶

    2.1.ETCD¶

    ETCD 是基于Raft 一致性算法的分布式键值存储系统，主打强一致性、高可用、轻量级，核心用于云原生场景的服务发现、配置管理、分布式锁、集群选主等场景。

    2.1.1. 核心定位¶

    ETCD 的设计目标是为分布式系统提供可靠的分布式协调能力，本质是一个分布式的 “高可用数据库”，聚焦于存储少量关键的元数据（而非海量业务数据）。

    2.1.2. 架构设计¶

    ETCD 采用分层架构，ETCD的架构主要包括网络层、Raft一致性算法模块、复制状态机和存储模块。写请求必须通过由Raft算法选举出的Leader节点处理，这保证了数据的一致性。ETCD集群通常由3个或5个奇数节点构成，遵循Quorum机制（即 majority，如3节点集群需要至少2个节点可用）来维持服务和数据一致性。从下到上分为 5 层：

    • 网络层：处理节点间的 TCP 通信，实现 Raft 协议的网络交互（如领导者选举、日志复制）。
    • 存储层：分为内存索引（加速键值查询）和持久化存储（WAL 预写日志 + 快照，防止数据丢失）。
    • Raft 层：实现 Raft 一致性算法，负责领导者选举、日志复制、集群一致性保障。
    • API 层：提供 HTTP/JSON、gRPC 接口，封装键值操作、Watch、Lease（租约）等核心功能。
    • 客户端层：提供 SDK（Go/Java/Python）、命令行工具（etcdctl），支持负载均衡、自动重连。
    2.1.3. 核心数据模型¶

    ETCD采用层次化的键值对（Key-Value） 数据模型，并提供了几个关键机制：

    • Prefix机制：采用分层的目录结构（类似文件系统），例如/micro/services/user表示用户服务的配置。
    • Revision机制（版本化）：每个键值对有唯一的mod_revision（修改版本）和create_revision（创建版本），支持多版本查询。
    • Lease（租约）：为键值对设置 TTL 租约，租约过期自动删除键值，适用于服务注册的临时节点。
    • Watch 机制：客户端可监听键的变化，实现配置实时推送、服务发现的动态感知。
    2.1.4. Raft 一致性算法¶

    ETCD 的强一致性依赖 Raft 协议，角色模型（Raft）核心流程包括 3 个阶段：

    1. 领导者选举（Leader）：集群节点分为 Leader（领导者）、Follower（跟随者）、Candidate（候选者），通过投票选举出唯一 Leader，负责处理所有写请求。
    2. 日志复制（Follower）：Leader 将写请求写入本地日志，再同步到所有 Follower，多数节点确认后才提交日志。
    3. 安全性：通过 “任期号”“日志匹配原则” 保证集群在网络分区、节点故障时的一致性。
    2.1.5. 关键特性¶
    • 强一致性：基于 Raft 实现线性一致性，适合存储关键元数据（如 K8s 的集群状态）。
    • 高可用：集群支持 3/5/7 节点部署，容忍(N-1)/2个节点故障（如 3 节点集群可容忍 1 个节点宕机）。
    • 轻量级：Go 语言编写，二进制包仅几十 MB，资源占用低。
    • 原生云原生：支持 TLS 加密、认证授权、与 Kubernetes 深度集成。

    1. 命令行工具 etcdctl 基础使用¶

    etcdctl 是 ETCD 官方命令行工具，支持所有核心功能，需先指定 ETCDCTL_API=3（v3 是当前稳定版 API）。

    （1）键值操作（K-V 存储）

    # 1. 设置键值对（支持分层目录结构，类似文件系统）
etcdctl --endpoints=http://localhost:2379 put /config/minio/endpoint "http://192.168.1.10:9000"
etcdctl --endpoints=http://localhost:2379 put /config/minio/accessKey "minioadmin"
etcdctl --endpoints=http://localhost:2379 put /config/minio/secretKey "minioadmin123"

# 2. 获取单个键值
etcdctl --endpoints=http://localhost:2379 get /config/minio/endpoint

# 3. 前缀查询（获取 /config/minio 下所有键值）
etcdctl --endpoints=http://localhost:2379 get /config/minio/ --prefix

# 4. 删除单个键
etcdctl --endpoints=http://localhost:2379 del /config/minio/secretKey

# 5. 前缀删除（删除 /config/minio 下所有键）
etcdctl --endpoints=http://localhost:2379 del /config/minio/ --prefix

    （2）Watch 机制（实时监听键变化）

    # 终端 1：监听 /config/minio/endpoint 键的变化
etcdctl --endpoints=http://localhost:2379 watch /config/minio/endpoint

# 终端 2：修改该键值，终端 1 会实时输出变化
etcdctl --endpoints=http://localhost:2379 put /config/minio/endpoint "http://192.168.1.11:9000"
# 终端 1 输出：
# PUT
# /config/minio/endpoint
# http://192.168.1.11:9000

    （3）Lease 租约（临时键，自动过期）

    适用于服务注册（服务下线后，租约过期自动删除注册信息）：

    # 1. 创建一个 30 秒的租约（返回租约 ID，如 7b6d4b07d38b8005）
etcdctl --endpoints=http://localhost:2379 lease grant 30

# 2. 绑定键到租约（租约过期后，键自动删除）
etcdctl --endpoints=http://localhost:2379 put /service/dataset-upload/192.168.1.20:8080 "online" --lease=7b6d4b07d38b8005

# 3. 续约租约（延长 30 秒）
etcdctl --endpoints=http://localhost:2379 lease keep-alive 7b6d4b07d38b8005

# 4. 查看租约状态
etcdctl --endpoints=http://localhost:2379 lease timetolive 7b6d4b07d38b8005

    （4）分布式锁（基于 ETCD 实现分布式互斥）

    # 终端 1：获取锁（锁定 /lock/dataset-upload，超时时间 60 秒）
etcdctl --endpoints=http://localhost:2379 lock /lock/dataset-upload --ttl=60

# 终端 2：尝试获取同一把锁，会阻塞直到终端 1 释放锁或超时
etcdctl --endpoints=http://localhost:2379 lock /lock/dataset-upload --ttl=60

# 终端 1：按 Ctrl+C 释放锁，终端 2 立即获取锁

    （1）安装依赖

    （2）Go 代码示例：键值操作 + Watch

    package main

import (
    "context"
    "fmt"
    "time"

    "go.etcd.io/etcd/client/v3"
)

func main() {
    // 1. 连接 ETCD 集群（单机/集群均支持，多个节点用逗号分隔）
    cli, err := clientv3.New(clientv3.Config{
        Endpoints:   []string{"127.0.0.1:2379"}, // 集群节点地址
        DialTimeout: 5 * time.Second,            // 连接超时时间
    })
    if err != nil {
        panic(fmt.Sprintf("ETCD 连接失败：%v", err))
    }
    defer cli.Close()
    fmt.Println("ETCD 连接成功")

    // 2. 设置键值
    ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
    _, err = cli.Put(ctx, "/config/minio/endpoint", "http://192.168.1.10:9000")
    cancel()
    if err != nil {
        panic(fmt.Sprintf("Put 失败：%v", err))
    }

    // 3. 获取键值
    ctx, cancel = context.WithTimeout(context.Background(), 2*time.Second)
    resp, err := cli.Get(ctx, "/config/minio/endpoint")
    cancel()
    if err != nil {
        panic(fmt.Sprintf("Get 失败：%v", err))
    }
    for _, kv := range resp.Kvs {
        fmt.Printf("键：%s，值：%s\n", kv.Key, kv.Value)
    }

    // 4. Watch 键变化（异步监听）
    watchChan := cli.Watch(context.Background(), "/config/minio/endpoint")
    go func() {
        for watchResp := range watchChan {
            for _, event := range watchResp.Events {
                fmt.Printf("Watch 事件：%s，键：%s，值：%s\n", event.Type, event.Kv.Key, event.Kv.Value)
            }
        }
    }()

    // 5. 模拟修改键，触发 Watch
    time.Sleep(2 * time.Second)
    ctx, cancel = context.WithTimeout(context.Background(), 2*time.Second)
    cli.Put(ctx, "/config/minio/endpoint", "http://192.168.1.11:9000")
    cancel()

    // 保持程序运行
    select {}
}

    2.2.Consul¶

    Consul 是 HashiCorp 开发的 分布式服务发现与服务治理平台，属于工程实践中功能最完备的服务发现系统之一。

    2.2.1.核心功能¶
    1. 服务发现（Service Discovery）：自动注册和发现微服务实例，支持 DNS/HTTP 两种查询方式。
    2. 健康检查（Health Check）：实时监控节点和服务的可用性，自动剔除故障实例。
    3. KV 存储（Consul KV）：基于分布式 KV 存储实现配置的动态推送与更新。
    4. Service Mesh（Consul Connect）：提供服务间的加密通信、流量控制和访问策略。
    5. 多数据中心支持（Multi-Datacenter）：原生支持跨地域的集群同步，实现全球服务治理。
    6. ACL 与安全机制（JWT、TLS、Token）

    Consul 最强的领域是 微服务治理，尤其适合中大型分布式系统。

    2.2.2.核心原理¶
    1. 一致性保障：Raft 算法¶

    Consul 的 Server 集群通过Raft 一致性算法保证集群状态的强一致性（CP 模型），核心流程与 ETCD 一致，分为三个阶段：

    1. 领导者选举：Server 节点在任期内通过投票选举出唯一 Leader，Follower 若超时未收到 Leader 心跳则转为 Candidate 发起选举。
    2. 日志复制：所有写请求（如服务注册、KV 配置更新）均由 Leader 处理，写入本地日志后同步到 Follower，多数节点确认后提交日志并更新状态机。
    3. 故障恢复：若 Leader 宕机，Follower 重新发起选举；网络分区恢复后，通过日志匹配原则同步数据，保证集群状态一致。

    适用场景：Consul 的 Raft 仅用于 Server 集群的元数据（服务注册信息、KV 配置）同步，而非海量业务数据存储，确保轻量级与高性能。

    2. 集群通信：Gossip 协议¶

    Consul 采用Gossip 协议实现节点间的信息传播，分为两种类型：

    （1）LAN Gossip

    • 作用：同一数据中心内的节点通信，传播集群成员信息、健康检查状态、服务列表等。
    • 特点：基于 UDP 实现，去中心化，每个节点仅需与少数节点通信即可广播信息，降低网络开销。

    （2）WAN Gossip

    • 作用：跨数据中心的 Server 节点通信，传播数据中心拓扑、跨地域服务信息。
    • 特点：仅在数据中心的 Server 节点间运行，保证跨地域集群的轻量级通信。
    3. 服务发现原理¶

    Consul 的服务发现分为服务注册和服务发现两个核心流程，支持手动注册和自动注册两种方式：

    （1）服务注册

    1. 微服务实例通过 Consul Client 的 HTTP API/DNS 或 SDK，将自身信息（服务名、IP、端口、标签、健康检查配置）注册到 Consul 集群。
    2. Client 节点将注册请求转发给 Server Leader，Leader 通过 Raft 同步注册信息到所有 Server 节点。
    3. Consul 为服务生成唯一标识，并通过 LAN Gossip 将服务信息传播到集群所有节点。

    （2）服务发现

    Consul 支持两种服务发现方式，满足不同场景需求：

    • DNS 方式：Consul 内置 DNS 服务器，客户端通过 dig serviceName.service.consul 即可查询服务实例的 IP 和端口，无需开发适配，适配传统应用。
    • HTTP API 方式：客户端通过 GET /v1/catalog/service/{serviceName} 接口查询服务详情，支持过滤健康实例、按标签筛选，适配微服务开发。
    4. 健康检查机制¶

    Consul 内置多级健康检查能力，确保只将请求路由到健康的服务实例，核心检查类型包括：

    1. 节点检查：监控 Consul 节点（Server/Client）的存活状态，基于 Gossip 协议传播。
    2. 服务检查：监控微服务实例的可用性，支持多种检查方式：
    3. TCP 检查：检测端口是否可连接（如 tcp://127.0.0.1:8080）。
    4. HTTP 检查：检测健康接口返回码（如 http://127.0.0.1:8080/health，要求 200 状态码）。
    5. gRPC 检查：调用 gRPC 健康检查接口（基于 grpc.health.v1 协议）。
    6. 脚本 / 命令检查：执行自定义脚本（如 bash check.sh），根据退出码判断健康状态。
    7. 检查结果处理：若服务实例连续失败达到阈值，Consul 会将其标记为不健康，并从服务发现结果中剔除，恢复后自动重新加入。
    5. 配置管理：分布式 KV 存储¶

    Consul 的 KV 存储基于 Raft 实现强一致性，用于存储微服务的配置信息，核心特性包括：

    • 分层键值结构：支持类似文件系统的路径式键（如 /config/dataset-service/db），便于配置分类管理。
    • 配置监听：客户端通过 Watch 机制监听键的变化，实现配置的动态推送（如 GET /v1/kv/config/dataset-service?watch=true）。
    • 版本控制：每个键值对有唯一的修改版本号，支持多版本查询和回滚。
    6. 服务网格（Consul Connect）¶

    Consul Connect 是内置的服务网格解决方案，基于Sidecar 代理实现服务间的流量治理，核心原理：

    1. mTLS 加密：Sidecar 代理为服务实例自动生成 TLS 证书，实现服务间通信的加密与身份认证。
    2. 流量代理：所有服务间的请求均通过 Sidecar 代理转发，支持流量路由、负载均衡、熔断限流。
    3. 访问控制：通过意图（Intentions） 策略定义服务间的访问规则（如允许 user-service 访问 dataset-service），实现细粒度权限控制。
    2.2.3.节点与架构¶

    Consul 采用服务端 / 客户端（Server/Client） 集群架构，结合 gossip 协议和 Raft 算法实现高可用与一致性，整体架构分为控制平面和数据平面两层。

    • 角色细分：

    （2）Client 节点

    • 核心作用：轻量级代理，无状态，负责转发客户端请求（服务注册、发现、健康检查）到 Server 集群，同时参与 gossip 协议传播集群信息。
    • 部署建议：每个应用服务器部署一个 Client 节点，降低 Server 节点的负载，提升请求转发效率。

    （3）Sidecar 代理（Consul Connect）

    • 核心作用：服务网格的数据平面组件，部署在微服务实例旁，负责服务间的流量代理、加密通信（mTLS）、访问控制和流量监控。

    核心层 实现 Raft 一致性、服务注册 / 发现、健康检查、KV 存储 Raft 算法、状态机
    应用层 提供 DNS/HTTP API、服务网格策略、多数据中心同步、可视化 UI DNS 服务器、HTTP 网关
    3.📝 关键工作流程¶
    1. 服务注册：
    2. 服务启动时，通过 Consul Client 的 API 或配置文件进行注册。
    3. Client 将注册信息转发给某个 Server 节点。
    4. 该 Server 节点（如果是 Follower，会转发给 Leader）通过 Raft 协议将服务信息写入到集群的持久化存储中。
    5. 服务发现：
    6. 应用程序向本地的 Consul Client 发起查询（通过 DNS 或 HTTP API）。
    7. Client 将查询请求转发给某个 Server 节点。
    8. Server 节点从自己的状态存储中返回健康的服务实例列表。
    9. Client 将结果返回给应用程序。
    10. 健康检查：
    11. 健康检查定义在服务注册时提交。
    12. Client 节点负责执行其所在节点上服务的健康检查（如检查端口、调用 HTTP 接口、执行脚本等）。
    13. 检查结果由 Client 报告给 Server 节点。
    14. 如果服务不健康，Server 节点会通过 Raft 协议更新其状态，并将其从健康服务列表中移除。

    2.3.Nacos¶

    Nacos 是阿里巴巴开源的动态服务发现、配置管理和服务管理平台，主打 “一站式微服务治理”，原生支持 CP/AP 模式切换，适配 Spring Cloud、Dubbo、Go-zero 等主流微服务架构，在国内微服务生态中应用广泛。

    2.3.1.Nacos 核心定位¶

    Nacos 的设计目标是解决微服务架构中的服务发现与配置管理痛点，核心应用场景包括：

    1. 动态配置管理：支持配置的实时推送、灰度发布、版本回滚，适用于数据集平台的存储配置、微服务节点配置等动态调整场景。
    2. 服务注册与发现：提供 HTTP/DNS/RPC 多种发现方式，支持服务的健康检查与故障剔除，适配微服务集群的动态扩缩容。
    3. 服务管理：提供可视化控制台，支持服务元数据管理、流量管控、服务分组，便于数据集平台的微服务治理。
    4. 动态 DNS 服务：支持权重路由、容灾路由，适配多环境（开发 / 测试 / 生产）的服务访问策略。

    Nacos 核心优势在于易用性（中文文档、可视化控制台）、多模式适配（CP/AP 动态切换）和生态兼容性（支持多语言、多框架），特别适合国内中小团队的微服务落地。

    2.3.2.整体架构¶

    Nacos 采用服务端 / 客户端架构，服务端支持单机 / 集群部署，客户端提供多语言 SDK（Java/Go/Python 等），整体架构分为接入层、核心层、存储层三层，同时通过插件化设计支持扩展能力。

    1. 集群架构¶

    Nacos 服务端集群包含核心节点（Nacos Server）和存储组件（嵌入式 Derby / 外置 MySQL），客户端通过 SDK/HTTP API 与服务端交互：

    此外，Nacos 提供插件扩展层，支持自定义健康检查、认证插件、存储插件等，满足个性化业务需求。

    2.3.3.核心原理¶
    1. 一致性模型：CP/AP 动态切换¶

    Nacos 是少数同时支持CP 模型和AP 模型的服务治理组件，可根据业务场景灵活切换：

    （1）CP 模式（基于 Raft）

    • 适用场景：存储关键配置、服务元数据（如数据集平台的 MinIO 存储配置），要求强一致性。
    • 实现原理：通过 Raft 协议保证集群数据的强一致性，Leader 节点处理写请求，多数节点确认后才提交数据，容忍 (N-1)/2 个节点故障。
    • 切换方式：在 Nacos 配置文件中设置 nacos.core.protocol=raft，并指定集群节点列表。

    （2）AP 模式（基于 Distro）

    • 适用场景：服务发现（如微服务实例注册），要求高可用性，允许短暂的数据不一致。
    • 实现原理：基于 Distro 协议（阿里自研的去中心化一致性协议），每个节点均处理写请求并同步数据到其他节点，无需选举 Leader，容忍任意节点故障。
    • 切换方式：默认服务发现为 AP 模式，可通过 API PUT /nacos/v1/ns/operator/switches 动态切换。
    2. 配置管理核心原理¶

    Nacos 的配置管理实现了配置的发布、推送、灰度、回滚全生命周期管理，核心流程如下：

    （1）配置发布

    1. 客户端 / 控制台通过 HTTP API 将配置（如 dataset.storage.minio.endpoint=http://127.0.0.1:9000）发布到 Nacos Server。
    2. 若为 CP 模式，Leader 节点将配置写入 MySQL，通过 Raft 同步到 Follower 节点；若为 AP 模式，Distro 协议同步到所有节点。
    3. Nacos 为配置生成数据 ID（配置唯一标识）、分组（配置分类）、命名空间（多环境隔离），支持配置的精细化管理。

    （2）配置推送

    Nacos 采用长轮询（Long Polling） 实现配置的实时推送，替代传统的定时轮询，减少网络开销：

    1. 客户端向 Nacos Server 发起长轮询请求，携带当前配置的版本号。
    2. 若配置未变更，Server 挂起请求（默认 30 秒）；若配置变更，Server 立即返回新配置和版本号。
    3. 客户端接收新配置后，本地缓存并应用，同时更新版本号。

    （3）配置灰度与回滚

    • 灰度发布：支持将配置推送到指定微服务实例（如数据集平台的测试节点），验证无误后全量发布。
    • 版本回滚：Nacos 保存配置的历史版本，可通过控制台 / API 一键回滚到任意历史版本，解决配置发布故障。
    3. 服务发现核心原理¶

    Nacos 的服务发现支持服务注册、健康检查、服务查询三大核心能力，默认采用 AP 模式（Distro 协议）：

    （1）服务注册

    1. 微服务实例通过客户端 SDK，将自身信息（服务名、IP、端口、健康检查配置）注册到 Nacos Server。
    2. AP 模式下，任意 Nacos 节点接收注册请求后，将服务实例信息同步到其他节点（Distro 协议）；CP 模式下，由 Leader 节点统一处理并同步。
    3. Nacos 为服务实例添加元数据（如服务版本、权重、标签），支持按标签筛选实例（如数据集平台的 “分片上传服务” 标签）。

    （2）健康检查

    Nacos 内置服务级健康检查，确保只将请求路由到健康实例：

    • 客户端主动上报：微服务实例定期向 Nacos Server 发送心跳（默认 5 秒），超时未上报则标记为不健康。
    • 服务端主动探测：支持 TCP/HTTP/gRPC 探测，Nacos Server 主动调用实例的健康检查接口，判断是否存活。
    • 结果处理：不健康实例被从服务注册表中剔除，恢复后自动重新加入，支持手动下线实例。

    （3）服务查询

    客户端通过HTTP API或DNS查询服务实例：

    • HTTP API：GET /nacos/v1/ns/instance/list?serviceName=dataset-service，返回健康的服务实例列表，支持按权重、标签过滤。
    • DNS：Nacos 内置 DNS 服务器，客户端通过 dig dataset-service.nacos 即可解析服务实例的 IP 和端口，适配传统应用。
    4. 命名空间与配置隔离¶

    Nacos 通过命名空间（Namespace）、分组（Group）、数据 ID 实现配置与服务的多层隔离，适配多环境 / 多租户场景：

    • 命名空间：隔离不同环境（如开发 / 测试 / 生产），每个命名空间有唯一 ID，默认命名空间为 public。
    • 分组：隔离同一环境下的不同业务（如数据集平台的 “存储组”“计算组”），默认分组为 DEFAULT_GROUP。
    • 数据 ID：配置的唯一标识，通常按 “服务名 - 环境.yaml” 命名（如 dataset-service-prod.yaml）。

    例如，数据集平台的生产环境 MinIO 配置可定义为：Namespace=prod、Group=storage、DataID=dataset-minio.yaml。
