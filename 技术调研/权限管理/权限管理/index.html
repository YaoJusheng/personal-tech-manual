    • 《授权管理：RBAC、PBAC、ABAC设计与实现》（英文名：Authorization Systems: Design, Implementation, and Usage）作者是Tomasz Janowski: 该书介绍了RBAC、PBAC、ABAC等多种授权管理模型的设计和实现方法，包括如何应用这些模型来管理企业资源和数据安全等。适合想要了解如何在实际项目中应用这些模型的读者。

    1.权限模型¶

    权限校验是指系统在进行用户访问控制时，对用户的权限进行检查，以确保用户有权执行所请求的操作。 通常情况下，权限校验是在用户登录后进行的，系统根据用户所拥有的权限和角色来限制用户对系统资源的访问。 权限校验可以保证系统的安全性，防止未授权的用户访问系统资源，同时也可以控制用户对系统的访问权限，保护系统的稳定性。

    权限管理是 Web 应用程序中的一个重要部分，它允许对用户的访问进行精细的控制。

    在权限管理中，一个关键问题是如何表示权限模型。通常，权限模型分为以下几类：

    • RBAC(Role-Based Access Control):基于资源与角色的访问控制模型。该模型将权限分为对象权限和操作权限，对象权限定义了用户可以访问的资源，操作权限定义了用户可以对资源执行的操作。该模型定义了角色、用户、权限和资源之间的关系，并授予不同的角色不同的权限。优点是比较灵活，可以根据不同的场景进行配置和扩展，缺点是管理起来比较复杂。在 RBAC 模型中，有许多工具可用于实现权限管理，如 Apache Shiro、Spring Security 等。

    • ABAC(Attribute-Based Access Control)：基于属性的访问控制。这种模型将访问控制的主要对象划分为资源和策略，策略由属性和值组成，可以动态地计算用户对资源的访问权限。该模型将权限授予用户或用户组，但是需要定义一些属性来确定用户是否有权限进行某项操作。例如，用户只能访问某个资源的特定属性，或者只有在特定时间或地点才能访问某个资源。优点是非常灵活，可以根据不同的属性进行控制，缺点是需要定义大量的属性，管理起来较为复杂。与 RBAC 相比，ABAC 更为灵活和细粒度，可以更好地适应复杂的授权场景。常见的 ABAC 工具包括 Open Policy Agent、Casbin 等。

    • PBAC(Policy-Based Access Control)：基于策略的访问控制。该模型将访问策略定义为一组规则，规定了用户可以执行的操作、在哪些资源上执行操作以及何时执行操作。该模型将权限授予策略而不是用户或用户组。策略通常由多个条件组成，包括用户身份、资源、操作、时间、地点等等。当这些条件匹配时，用户就可以执行相应的操作。优点是非常灵活，可以根据不同的条件进行控制，缺点是管理起来比较复杂。

    • ACL(Access Control List)：访问控制列表。该模型是为每个资源定义一个访问控制列表，控制用户对资源的访问权限，访问控制列表可以是用户、组或角色。

    • MAC(Mandatory Access Control)：强制访问控制。MAC 是一种强制性访问控制模型，其中每个用户和对象都被分配一个安全级别。每个对象只能由拥有相同或更高安全级别的用户访问。这种模型通常用于高度安全的应用程序，例如政府和军事系统。

    • DAC(Discretionary Access Control)：DAC 是一种自主性访问控制模型，其中用户拥有对自己创建的对象的控制权。这种模型通常用于个人计算机上的文件和目录访问控制。

    除了权限模型,还可以通过其他方式来实现权限管理:

    • 二进制位操作: 利用二进制优化的好处是可以省去用户和权限之间的关系表，只需要在用户表添加字段就可以，同样的如果是RBAC，角色和权限关系，用户和角色关系都可以这样用。同时这样做的优点是二进制的运算快、性能高，也能减少很多连表查询次数。

    无论采用何种方式，权限校验都应该严格限制用户对系统资源的访问，并能够对用户的访问行为进行审计，以便及时发现和防止系统安全漏洞的出现。

    常见的权限校验工具包括：

    • Casbin：支持多种权限校验模型，可以在多种编程语言中使用。
    • Apache Shiro：Java平台的安全框架，支持多种认证和授权方式。
    • Spring Security：Spring框架的安全框架，支持多种认证和授权方式。
    • Django Guardian：Django框架的安全框架，支持基于角色和基于对象的权限控制。
    • Flask-Security：Flask框架的安全框架，支持基于角色和基于对象的权限控制。
    • AWS IAM：Amazon Web Services提供的身份和访问管理服务，支持RBAC和ABAC模型。

    1.1.ACL模型¶

    ACL模型（Access Control List，访问控制列表）使用列表来管理和控制用户对资源的访问权限。每个资源都关联一个ACL，其中包含了对该资源的访问权限列表。 ACL通常包含了允许或拒绝访问的用户或用户组。常用于配置哪些用户可以访问操作哪些资源。

    当权限系统体量小，用户有直接对应具体功能点即可满足系统诉求时，可以考虑使用ACL模型作为参考。

    实现原理: 每一项资源，都配有一个列表，这个列表记录的就是哪些用户可以对这项资源操作。当系统试图访问这项资源时，会首先检查这个列表中是否有关于当前用户的访问权限，从而确定当前用户可否执行相应的操作。

    ACL权限中，只有用户、资源这两个要素:

    • 用户: 是发起操作的主体，可以是1个人，也可以是1个用户群组。例如: 后台管理系统的用户、OA系统的内部员工、设计部门等。
    • 资源: 用户可以访问操作的客体，例如: 页面、文档、数据等。

    ACL常用于部门隔离场景，适用资源如人事页面、客户页面。

    ACL模型优劣:

    • 优势: 研发实现快速简单，不需要耗费较多开发成本。
    • 劣势: 当拥有大量用户和资源时，维护成本变得极高，每次有新用户&新资源加入时都需要重新挨个配置每个用户与每个资源的关系，不适用于频繁变更用户权限的场景。

    1.1.1.DAC自主访问¶

    DAC在ACL基础上，增加了权力下放的能力，允许拥有权限的用户再次将权限授予其他用户。

    DAC常用于文件管理场景，适用资源如培训文档。如下图所示，拥有文档编辑权限的用户可以将编辑权限再次授予其他用户。

    1.1.2.MAC强制访问¶

    MAC在ACL基础上，增加了双重验证的限制，要求用户和要访问的资源必须具备相应的安全等级关系才可获取权限。

    MAC常用于保密信息场景，适用资源如保密数据。如绝密级的财务数据只能被企业高级用户查看、访客级用户无法查看企业级的文档、机密级的部门架构信息仅能被部门负责人级别的账号查看。

    1.2.RBAC模型¶

    1.2.1.RBAC模型简述¶

    RBAC模型（Role-Based Access Control，基于角色的访问控制）基于角色来管理和控制用户对资源的访问权限。用户被分配到不同的角色，而每个角色都被赋予一组特定的权限。RBAC模型简化了权限管理，RBAC的基本思想是将访问控制的管理从用户直接分配到资源的方式转变为通过角色的方式进行。这样，当用户的职责或角色发生变化时，只需调整其角色而无需更改每个用户的权限。

    RBAC模型包括以下关键组件:

    • 用户（User）: 系统的最终用户，被分配到不同的角色。
    • 角色（Role）: 一组相关权限的集合。用户通过分配角色来获取相应的权限。
    • 权限（Permission）: 定义了对系统资源的访问规则。权限被分配给角色。
    • 操作（Operation）: 对资源执行的特定动作。权限与操作相关联。

    RBAC认为权限的过程可以抽象概括为: 判断 【Who 是否可以对 What 进行 How 的访问操作（Operator）】 这个逻辑表达式的值是否为True的求解过程。

    即将权限问题转换为 Who、What、How 的问题。who、what、how 构成了访问权限三元组。

    • Who: 权限的拥用者或主体（如 Principal、User、Group、Role、Actor等等）
    • What: 权限针对的对象或资源（Resource、Class）。
    • How: 具体的权限（Privilege,正向授权与负向授权）。
    • Operator: 操作。表明对What的How操作。也就是Privilege+Resource
    • Role: 角色，一定数量的权限的集合。权限分配的单位与载体,目的是隔离User与Privilege的逻辑关系.
    • Group: 用户组，权限分配的单位与载体。权限不考虑分配给特定的用户而给组。组可以包括组(以实现权限的继承)，也可以包含用户，组内用户继承组的权限。User与Group是多对多的关系。Group可以层次化，以满足不同层级权限控制的要求。

    RBAC的基本思想是将访问控制的管理从用户直接分配到资源的方式转变为通过角色的方式进行。这样，当用户的职责或角色发生变化时，只需调整其角色而无需更改每个用户的权限。

    RBAC支持公认的安全原则: 最小特权原则、责任分离原则和数据抽象原则。

    • 最小特权原则: 在RBAC模型中可以通过限制分配给角色权限的多少和大小来实现，分配给与某用户对应的角色的权限只要不超过该用户完成其任务的需要就可以了。
    • 责任分离原则: 在RBAC模型中可以通过在完成敏感任务过程中分配两个责任上互相约束的两个角色来实现，例如在清查账目时，只需要设置财务管理员和会计两个角色参加就可以了。
    • 数据抽象: 数据抽象是借助于抽象许可权这样的概念实现的，如在账目管理活动中，可以使用信用、借方等抽象许可权，而不是使用操作系统提供的读、写、执行等具体的许可权。但RBAC并不强迫实现这些原则，安全管理员可以允许配置RBAC模型使它不支持这些原则。因此，RBAC支持数据抽象的程度与RBAC模型的实现细节有关。

    1.2.2.RBAC模型分类¶

    RBAC96模型家族,其中包括了RBAC0~RBAC3四个概念模型。

    • RBAC0模型: 最简单的用户、角色、权限模型。
    • 用户和角色是多对一关系，即: 一个用户只充当一种角色，一种角色可以有多个用户担当。
    • 用户和角色是多对多关系，即: 一个用户可同时充当多种角色，一种角色可以有多个用户担当。
    • RBAC1模型: 相对于RBAC0模型，增加了子角色，引入了继承概念，即子角色可以继承父角色的所有权限，即角色上有了上下级的区别。
    • 使用场景: 如某个业务部门，有经理、主管、专员。主管的权限不能大于经理，专员的权限不能大于主管，如果采用RBAC0模型做权限系统，极可能出现分配权限失误，最终出现主管拥有经理都没有的权限的情况。
    • RBAC2模型: 基于RBAC0模型，增加了对角色的一些限制: 角色互斥、基数约束、先决条件角色等。
    • 角色互斥: 同一用户不能分配到一组互斥角色集合中的多个角色，互斥角色是指权限互相制约的两个角色。案例: 财务系统中一个用户不能同时被指派给会计角色和审计员角色。
    • 基数约束: 一个角色被分配的用户数量受限，它指的是有多少用户能拥有这个角色。例如: 一个角色专门为公司CEO创建的，那这个角色的数量是有限的。
    • 先决条件角色: 指要想获得较高的权限，要首先拥有低一级的权限。例如: 先有副总经理权限，才能有总经理权限。
    • 运行时互斥: 例如，允许一个用户具有两个角色的成员资格，但在运行中不可同时激活这两个角色。
    • RBAC3模型: 称为统一模型，它包含了RBAC1和RBAC2，利用传递性，也把RBAC0包括在内，综合了RBAC0、RBAC1和RBAC2的所有特点。

    对功能模块的操作，对上传文件的删改，菜单的访问，甚至页面上某个按钮、某个图片的可见性控制，都可属于权限的范畴。有些权限设计，会把功能操作作为一类，而把文件、菜单、页面元素等作为另一类。而在做数据表建模时，可把功能操作和资源统一管理，也就是都直接与权限表进行关联，这样可能更具便捷性和易扩展性。

    1.3.ABAC模型¶

    ABAC模型（Attribute-Based Access Control，基于属性的访问控制）是一种比 RBAC模型 更加灵活的授权模型，ABAC模型基于属性来管理和控制用户对资源的访问权限, 使用一组属性和策略来决定是否授予用户访问权限，它的原理是通过各种属性来动态判断一个操作是否可以被允许。属性包括用户属性、环境属性和资源属性。策略定义了属性之间的关系和操作。

    • 用户属性: 包括用户的姓名、角色、组织、ID 和安全许可等内容。
    • 环境属性: 包括访问时间、数据的位置和当前组织的威胁等级。
    • 资源属性: 包括诸如创建日期、资源所有者、文件名和数据敏感性。

    1.3.1.术语¶

    • Attributes: 属性, 用于表示 subject、object 或者 environment conditions 的特征，attributes 使用 key-value 的形式来存储这些信息，比如我在公司的 role 是 developer，role 是 key，developer 是 value，而我的小组昵称xx，key 是 team，value 是 xx。
    • Subject: 常常指代使用系统的人或者其他使用者（non-person entity，NPE），比如说客户端程序，访问 API 的 client 或者移动设备等等。当然一个 subject 可以有多个的 attributes，就像用户属性这些我们曾经用过的名词一样。
    • Object: 指代我们这个 ACM 需要管理的系统资源，比如文件、记录、表格、进程、程序、某台机器或者某个网站，它可以是资源或请求的实体，也可以是Subject可以执行Operation的任何事物，任何你需要进行访问控制的资源都可以称为 object，同样 object 也可以有多项属性，我们也常常使用 resource 来描述这些资源，但是在 ABAC 的环境下，我们称为 object。
    • Operation: 操作, 操作是根据 subject 的请求对 object 执行功能。有了 object 有了 subject，自然就有了 subject 需要做的事情，比如查看某条记录，登录某台服务器。往往包括我们常说的读、写、修改、拷贝等等，一般 operation 是会表达在 request 中的，比如 HTTP method。
    • Privileges: 特权, 代表Subject的授权行为；它们由权威定义并体现 在 policy 或 rules。就本文件而言，条款 privileges 和 authorizations 可以互换，因为它们旨在传达一个人的权威和隐含的批准访问一个或多个 更多object。
    • Environment Conditions: 表示目前进行的访问请求发生时的操作或情境的上下文。Environment conditions 常常用来描述环境特征，是独立于 subject 与 object 的，常用来描述系统的情况: 比如时间，当前的安全等级，生产环境还是测试环境等等。
    • Policy: 策略, 策略是规则或关系的表示，通过 subject、object 的 attribute 与 environment conditions 一起来判断 subject 的请求是否能够被允许，比如说: policy 可以用人类语言这样表达，只有袋熊组的人才能访问这几台服务器，或者只有在办公室才能访问这些资源，但对于机器来说，无非就是一个判断语句罢了。当然了，policy 可以是一堆这样 boolean 逻辑判断的组合，比如只有公司的正式员工、并且在公司的六楼区域的网络中，才能访问某个服务。你可以使用 Specification Pattern 来实现 policy，其实没那么复杂。

    1.3.2.ABAC 模型的原理¶

    在 ABAC模型 中，一个操作是否被允许是基于对象、资源、操作和环境信息共同动态计算决定的。

    • 对象: 对象是当前请求访问资源的用户。用户的属性包括 ID，个人资源，角色，部门和组织成员身份等
    • 资源: 资源是当前用户要访问的资产或对象，例如文件，数据，服务器，甚至 API
    • 操作: 操作是用户试图对资源进行的操作。常见的操作包括“读取”，“写入”，“编辑”，“复制”和“删除”
    • 环境: 环境是每个访问请求的上下文。环境属性包含访问的时间和位置，对象的设备，通信协议和加密强度等

    在 ABAC模型 的决策语句的执行过程中，决策引擎会根据定义好的决策语句，结合对象、资源、操作、环境等因素动态计算出决策结果。每当发生访问请求时，ABAC模型 决策系统都会分析属性值是否与已建立的策略匹配。如果有匹配的策略，访问请求就会被通过。

    ABAC模型 的描述: 基于“对象-资源-操作-环境”的权限控制，描述了“XX对象（人/应用/组织/角色等）对 XX资源（页面/菜单/按钮/数据等）在 XX环境/因素（城市=上海等）下拥有 XX操作类型（增删改查等）的权限。

    ABAC模型优劣:

    • 优势: 能够根据上下文动态执行，可以发挥权限控制最大的灵活性，满足更细颗粒度的场景需求。
    • 劣势: 学习成本比较高，随着规则（策略）的不断增多，管理和维护成本也会攀升。

    1.3.3.设计方案流程¶

    • 1.定义实体: 用户与资源
    • 2.定义属性
    • 用户属性：可以包括角色、部门、地理位置、职位等。
    • 资源属性：可以包括类型、所有者、标签、敏感级别等。
    • 环境属性：可以包括时间、位置、设备类型等。
    • 3.定义策略或规则: 指定哪些属性的组合允许或禁止访问
    • 如果用户角色是 "管理员"，则允许访问所有资源。
    • 如果资源类型是 "文档"，并且所有者是当前用户，那么允许访问。
    • 如果资源标签包含 "机密"，并且当前时间是工作时间段，那么允许访问。
    • 4.访问控制系统收到访问请求后,获取用户与资源的属性
    • 5.与策略进行匹配,判断是否匹配任一策略,实现鉴权

    1.4.总结¶

    尽管 ABAC 被广泛认为是 RBAC 的演进形式，但它并不总是正确的选择。根据公司或项目的规模、预算和安全需求，选择其中之一。

    1. 综合优劣对比

    2. ACL模型: 基于资源进行配置，适用于用户量少、更新频次低、无明确职能划分的小型组织，开发成本低但后期管理维护成本高，适合作为平台搭建初期时的临时过渡方案；

    3. RBAC模型: 基于角色进行配置，适用于用户量一般、更新频次一般、有明确职能划分、较少定制权限的中型组织，开发成本高但后期维护成本低，适合作为平台管理少量内外部用户时的方案；

    4. ABAC模型: 基于属性进行配置，适用于用户量多、更新频次高、职能划分复杂多变、有安全合规诉求、权限定制多样化的大型组织，能够在保障效率的同时降低安全风险，适合作为平台管理大批量复杂用户时的方案。

    5. ABAC 和 RBAC 的对比项:

    6. 复杂性和灵活性

    7. RBAC模型相对简单，适用于较小规模且权限结构相对稳定的系统。它将权限分配给角色，然后将角色分配给用户，易于管理和维护。如果系统的权限需求相对简单，且角色与权限之间的关系相对固定，RBAC是一个简单而有效的选择。
    8. ABAC模型更加灵活，可以使用属性和策略来动态决定访问权限。它适用于较大规模、复杂度高或权限需求频繁变化的系统。ABAC模型可以根据用户属性、资源属性和环境属性进行动态授权决策，提供更细粒度的权限控制。
    9. 细粒度控制
    10. 如果系统需要细粒度的权限控制，例如基于多个属性或上下文条件进行访问决策，ABAC模型更适合。ABAC模型可以使用属性、策略和规则来定义更复杂的访问控制逻辑。
    11. 如果系统的权限需求相对简单，只需要对角色进行粗粒度的控制，RBAC模型可能足够满足要求。
    12. 可扩展性
    13. 如果系统的用户角色和权限结构相对稳定，并且不太可能发生频繁的变化，RBAC模型具有良好的可扩展性。因为RBAC模型简化了权限管理和维护，当需要添加新的角色或权限时，只需对角色和权限的关系进行适当的调整。
    14. 如果系统的用户属性、资源属性或访问策略需要经常变化，ABAC模型更具有可扩展性。ABAC模型可以通过修改属性和策略来适应新的访问需求，而无需对整个角色和权限结构进行重大改变。
    15. 风险和合规性要求
    16. 如果系统需要满足特定的合规性要求，例如符合某个标准或法规，RBAC模型可能更受欢迎。RBAC模型在访问控制方面具有较好的审计和可追溯性，易于满足合规性要求。

    17. 如果系统需要根据更复杂的上下文条件和属性进行动态授权决策，ABAC模型可以更好地满足风险管理和合规性需求。

    18. 关于如何选择的对比参考:

    19. 如果满足以下条件，请选择 ABAC:

    20. 拥有适当实施 ABAC 的时间、资源和预算。
    21. 身处一个不断成长的大型组织中。ABAC 支持可扩展性。
    22. 拥有分布在不同地区的劳动力。ABAC 可以帮助您添加基于位置和时区的属性。
    23. 希望访问控制策略尽可能精细和灵活。

    24. 希望您的访问控制策略能够面向未来。世界在不断发展，RBAC 正在慢慢成为一种过时的方法。ABAC 为您提供了对安全控制的更多控制和灵活性。

    25. 如果满足以下条件，请选择 RBAC:

    26. 属于中小型组织。
    27. 在您的组织内拥有明确定义的组，并且应用广泛的、基于角色的策略是有意义的。
    28. 实施访问控制策略的时间、资源和/或预算有限。
    29. 不要有太多的外部贡献者，也不要期望加入很多新人。

    2.4.2.2.优缺点¶
    • 优点：基于 Django 框架，支持对象级别的权限控制，对于复杂的权限管理场景提供了扩展接口，灵活性高。
    • 缺点：需要手动实现扩展，使用门槛相对较高。

    2.4.3.Django Guardian Rest Framework¶

    基于 Django Guardian 和 Django REST Framework 的权限管理库，支持 RESTful API 的权限控制。

    2.4.3.1.Example¶
    • 安装
    pip install django-guardian django-rest-framework
    • 更新配置settings.py
    INSTALLED_APPS = [
    ...
    'rest_framework',
    'rest_framework.authtoken',
    'guardian',
]
# 配置 Guardian 和 DRF 的身份验证
AUTHENTICATION_BACKENDS = [
    'django.contrib.auth.backends.ModelBackend',
    'guardian.backends.ObjectPermissionBackend',
    'rest_framework.authentication.TokenAuthentication',
]

REST_FRAMEWORK = {
    'DEFAULT_AUTHENTICATION_CLASSES': [
        'rest_framework.authentication.TokenAuthentication',
    ],
}
    • 在urls.py文件中添加路由
    from django.conf.urls import include, url
from rest_framework.authtoken.views import obtain_auth_token

urlpatterns = [
    ...
    url(r'^api-token-auth/', obtain_auth_token, name='api_token_auth'),
    url(r'^api/', include('myapp.urls')),
]
    • 自定义的权限管理类
    from rest_framework import permissions
from guardian.shortcuts import get_objects_for_user

class MyPermission(permissions.BasePermission):
    def has_permission(self, request, view):
        qs = get_objects_for_user(request.user, 'myapp.can_view')
        return qs.exists()
    • 在 DRF 的视图中使用 MyPermission
    from rest_framework import viewsets
from myapp.models import MyModel
from myapp.serializers import MyModelSerializer
from myapp.permissions import MyPermission

class MyModelViewSet(viewsets.ModelViewSet):
    queryset = MyModel.objects.all()
    serializer_class = MyModelSerializer
    permission_classes = [MyPermission]
    2.4.3.2.优缺点¶
    • 优点：
    • 简单易用：DGRF 提供了简单易用的接口来管理权限，使开发人员能够更轻松地控制用户对资源的访问。
    • 灵活性：DGRF 允许开发人员在 Django Guardian 和 DRF 中自定义权限管理类，以满足特定应用程序的需求。
    • 高度集成：DGRF 与 Django Guardian 和 DRF 高度集成，使开发人员能够快速实现在 DRF 中使用 Guardian 的权限管理功能。

    • 可扩展性：DGRF 提供了可扩展的架构，使开发人员能够通过添加插件来增强其功能。

    • 缺点：

    • 学习曲线：DGRF 对于那些不熟悉 Django Guardian 和 DRF 的开发人员来说，可能需要一些时间来学习和掌握。
    • 复杂性：由于其集成了两个强大的框架，DGRF 可能会增加应用程序的复杂性。
    • 性能：DGRF 可能会影响应用程序的性能，因为它需要更多的计算资源来管理权限。然而，这种影响通常很小，并且可以通过适当的优化来减少。

    2.4.x.其他¶

    除了Flask-ACL，还有其他基于ACL模型的Python Web权限管理工具，常用的有以下几种：

    • Flask-Principal: Flask-Principal是一个Flask扩展，提供了基于ACL和RBAC的权限管理功能。可以使用Principal对象定义权限和角色，使用identity_changed信号管理用户身份信息，并使用identity_loaded信号在请求上下文中加载身份信息。
    • 优点: 易于集成到Flask应用程序中，支持多种权限管理模型，并提供了较为完善的文档和示例。

    • 缺点: 功能相对较为简单，不支持复杂的权限管理需求。

    • Flask-Security: Flask-Security是一个Flask扩展，提供了基于ACL和RBAC的权限管理功能。可以使用RoleMixin和UserMixin对象定义角色和用户，使用login_required和roles_required装饰器限制访问权限。

    • 优点: 易于集成到Flask应用程序中，支持多种权限管理模型，并提供了较为完善的文档和示例。

    • 缺点: 需要手动编写数据库模型和视图函数，可能需要一定的编码工作量。

    • Pyramid ACL: Pyramid ACL是一个Pyramid扩展，提供了基于ACL的权限管理功能。可以为每个资源定义访问控制规则，支持多种访问控制策略，如基于对象的访问控制、基于角色的访问控制和基于分组的访问控制。

    • 优点: 易于集成到Pyramid应用程序中，支持多种访问控制策略，并提供了较为完善的文档和示例。
    • 缺点: 需要手动编写视图函数和访问控制规则，可能需要一定的编码工作量。
