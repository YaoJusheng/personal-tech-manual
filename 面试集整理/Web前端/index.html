    2. WebSocket 双向通信¶

    <script>
// 建立连接
const ws = new WebSocket('ws://localhost:8080');
// 连接成功
ws.onopen = () => {
  ws.send('客户端已上线');
};
// 接收服务端消息
ws.onmessage = e => {
  console.log('收到：', e.data);

    4. Web Worker 后台线程¶

    核心理解：

    • 解决 JS 单线程阻塞问题：JS 默认只有主线程，大量计算会卡死页面、卡顿 UI。

    • Web Worker 开启浏览器后台子线程，专门跑耗时运算，不阻塞页面。

    2. 只能通过 postMessage 主线程和子线程通信

    4. 适合大数据计算、循环遍历、解析大文件

    适用场景：

    • 海量数据排序 / 统计
    • 大 JSON、Excel 前端解析
    • 复杂加密、数据预处理

    CSS¶

    CSS 是什么：

    CSS（Cascading Style Sheets，层叠样式表）是一种用于描述 HTML 文档外观和格式的语言。它是 Web 开发的三大基石之一（HTML 结构、CSS 样式、JS 行为）。

    核心特性：

    • 层叠性：多个样式规则可应用于同一元素，按优先级合并
    • 继承性：某些属性（如 color、font-family）会自动传递给子元素
    • 优先级：!important > 内联样式 > ID > 类/属性/伪类 > 元素/伪元素

    CSS 的痛点：

    • 无法定义变量（原生 CSS 变量支持较晚且不完善）
    • 没有嵌套、模块化、函数等编程特性
    • 重复代码多，难以维护
    • 浏览器兼容性需手动处理

    盒模型组成： content → padding → border → margin

    两种盒模型：

/* 宽度 = content */

/* IE盒模型/替代盒模型 */

    • 水平方向不会合并
    • 解决：overflow: hidden、display: flex、display: flow-root

    3. CSS选择器优先级？¶

    优先级计算（a, b, c）：

    • a：!important
    • b：行内样式
    • c：ID选择器
    • d：类、伪类、属性选择器
    • e：元素、伪元素

    8.1. SCSS/Sass¶

    背景： Sass（Syntactically Awesome Style Sheets）是最早的 CSS 预处理器，2006 年诞生于 Ruby 社区。有两个语法版本：

    • Sass（缩进语法）：用缩进代替大括号，省略分号
    • SCSS（Sassy CSS）：完全兼容 CSS3 语法，使用大括号和分号（更常用）

    核心特性：

    // 1. 变量（$ 开头）
$primary: #1890ff;
$spacing: 8px;

// 2. 嵌套
.nav {
  background: $primary;
  &__item {  // BEM 写法支持
    padding: $spacing;
    &:hover { color: red; }
  }
}

// 3. 混合（@mixin + @include）
@mixin flex-center($gap: 0) {
  display: flex;
  align-items: center;
  justify-content: center;
  gap: $gap;
}
.container { @include flex-center(16px); }

// 4. 继承（@extend）
.btn { padding: 8px 16px; border: none; }
.btn-primary { @extend .btn; background: $primary; }

// 5. 控制指令
@for $i from 1 through 5 {
  .col-#{$i} { width: 20% * $i; }
}

// 6. 内置函数
$dark: darken($primary, 10%);
$light: lighten($primary, 10%);

// 7. 模块化（@use / @forward）
@use 'variables' as v;
@forward 'mixins';

    编译工具： sass（Dart Sass，官方推荐）、node-sass（LibSass，已弃用）

    8.2. Less¶

    背景： Less（Leaner Style Sheets）2009 年诞生，最初用 JavaScript 实现，受 Sass 影响但语法更贴近 CSS。由 Alexis Sellier（Cloudhead）创建。

    核心特性：

    // 1. 变量（@ 开头，类似 CSS）
@primary: #1890ff;
@spacing: 8px;

// 2. 嵌套（同 SCSS）
.nav {
  background: @primary;
  &:hover { color: darken(@primary, 10%); }
}

// 3. 混合（类名直接作为 mixin）
.flex-center(@gap: 0) {
  display: flex;
  align-items: center;
  justify-content: center;
  gap: @gap;
}
.container { .flex-center(16px); }

// 4. 运算（自动单位转换）
@width: 100px + 50; // 150px

// 5. 函数
@base: #f04615;
@color: fade(@base, 50%);

// 6. 导入（可嵌套在其他选择器中）
.button {
  @import "button-mixins.less";
}

// 7. 作用域（就近原则查找变量）
@var: red;
.scope {
  @var: blue;
  color: @var; // blue
}

    编译工具： less（官方 JS 编译器）、less-loader（Webpack）

    背景： Stylus 2010 年诞生于 Node.js 社区，由 TJ Holowaychuk（Express/Koa 作者）创建，以极高的表达性和极简语法著称。

  .col-{i}

.button

// 7. 强大的内置库（图片处理、颜色处理等）

    8.4. 选型建议¶
    选 SCSS 的情况¶
    • 大型项目、多人协作（语法严格，规范性强）
    • 需要强大的内置函数库（颜色处理、数学运算）
    • 使用 BEM 等规范命名（&__element 语法友好）
    • 配合主流框架/组件库（Element Plus、Ant Design Vue 等都用 SCSS）
    选 Less 的情况¶
    • 团队习惯接近原生 CSS
    • 使用 Bootstrap（源码用 Less）
    • Webpack 环境（Less 集成简单）
    • 不需要太复杂的功能
    选 Stylus 的情况¶
    • 个人项目、追求书写效率
    • 喜欢简洁语法
    • 配合 Vue 项目（Vue 早期推荐 Stylus）
    • 不在意社区活跃度
    趋势¶
    • SCSS 已成为事实标准，大多数大型项目和组件库采用 SCSS
    • Less 在历史项目中仍广泛使用
    • Stylus 使用率逐步下降

    9. PostCSS：后处理器¶

    9.1. 简介¶

    除了预处理器，PostCSS 是另一种 CSS 处理方案，它不是预处理器，而是 一个CSS 转换引擎 / 后处理器工具。

    • 工作流：CSS → 解析成 AST → 插件修改 AST → 生成新 CSS

    • 本身零功能，全靠插件干活

    • 核心价值：统一处理、自动前缀、未来语法降级、压缩、模块化等

    核心区别：

    • 预处理器：自定义语法 → 编译为 CSS
    • PostCSS：标准 CSS → 通过插件转换

    常用插件：

    • Autoprefixer：自动添加浏览器前缀
    • postcss-nested：支持 SCSS 风格嵌套（写标准 CSS 未来语法）
    • postcss-preset-env：使用未来的 CSS 特性语法（嵌套、变量、容器查询），自动降级
    • postcss-import：解析 @import，合并文件
    • postcss-modules：CSS 模块化（类名哈希，防冲突）
    • cssnano：CSS 压缩、去重、优化

    其他：

    • postcss-purgecss：分析你的 HTML/JS，移除未使用的 CSS 样式。

    • Tailwind CSS：基于 PostCSS 的原子化框架

    趋势： PostCSS 与 Tailwind CSS 的组合正在挑战传统预处理器的地位，代表 CSS 工程化的新方向。

    一个经典的工作流：

    标准 CSS（或预处理器编译后的 CSS）
    ↓

    1. postcss-import        （合并 @import 文件）

    3. Autoprefixer          （添加厂商前缀）
    4. postcss-pxtorem       （单位转换）
    5. cssnano               （压缩）

    for (const key of Reflect.ownKeys(obj)) {

    return clone;

    9. JavaScript防抖和节流？¶

    防抖（Debounce）： 事件触发n秒后执行，n秒内再次触发则重新计时。

    function debounce(fn, delay) {
    let timer = null;
    return function(...args) {
        clearTimeout(timer);
        timer = setTimeout(() => {
            fn.apply(this, args);
        }, delay);
    };
}

// 使用
window.addEventListener('resize', debounce(handleResize, 300));

    节流（Throttle）： 事件触发后n秒内只执行一次。

    function throttle(fn, interval) {
    let lastTime = 0;
    return function(...args) {
        const now = Date.now();
        if (now - lastTime >= interval) {
            lastTime = now;
            fn.apply(this, args);
        }
    };
}

// 使用
window.addEventListener('scroll', throttle(handleScroll, 200));

    场景： 防抖（搜索输入、窗口调整）、节流（滚动、点击）。

    • import：ES Module 规范，静态加载，编译时解析，支持 Tree Shaking
    • Node.js 中需配置 "type": "module" 或使用 .mjs 扩展名

    补充：

    • 现代 Node 项目建议优先 ESM，并通过 exports 字段做入口治理
    • CJS/ESM 混用要注意默认导出互操作与双包加载差异

    2. 常用的 Node.js 全局对象有哪些？¶

    • __dirname、__filename（CommonJS 专属，ESM 中不可用）
    • process（进程信息、环境变量、stdin/stdout）
    • Buffer（处理二进制数据）
    • setTimeout、setInterval、setImmediate
    • global（全局命名空间）

    补充： Node 18+ 原生提供 fetch、URL、AbortController 等 Web API。

    4. 中间件的原理和洋葱模型？¶

    • 中间件是一个函数，接收 (req, res, next)，可访问请求对象、响应对象和调用下一个中间件
    • 洋葱模型：请求由外向内进入，响应由内向外返回（Koa 的实现）
    • Express 是线性模型，Koa 通过 async/await 实现真正洋葱模型

    5. 如何理解 Node.js 的单线程与高并发？¶

    • 主线程是单线程的 JavaScript 执行线程
    • 通过事件循环 + 非阻塞 I/O + libuv 线程池 实现高并发
    • 计算密集型任务应使用 Worker Threads 或拆分为微服务

    6. Buffer 的常见使用场景和注意点？¶

    • 场景：文件 I/O、网络传输、加密、二进制数据处理
    • 注意：Buffer.alloc(size) 会初始化填充（安全），Buffer.allocUnsafe(size) 可能包含旧数据但更快
    • Buffer 与字符串互转时需指定编码（utf-8、base64 等）

    7. Node.js事件循环机制？¶

    • 六个阶段：timers → pending callbacks → idle/prepare → poll → check → close callbacks
    • 微任务（process.nextTick、Promise）在每个阶段之间执行
    • process.nextTick 优先级高于 Promise

   └─────────────┬─────────────┘

   │    pending callbacks      │  I/O callbacks deferred
   └─────────────┬─────────────┘
   ┌─────────────▼─────────────┐
   │       idle, prepare       │  内部使用
   └─────────────┬─────────────┘

   │         (轮询阶段)         │  执行I/O回调

   └─────────────┬─────────────┘
   ┌─────────────▼─────────────┐
   │      close callbacks      │  socket.on('close')
   └───────────────────────────┘
*/

    8. process.nextTick 和 setImmediate 的区别？¶

    11. 进程间通信（IPC）方案对比？¶

    • child_process.fork()：内置 IPC 通道，适合简单场景
    • Redis/RabbitMQ：解耦、跨机器、持久化
    • 共享内存（SharedArrayBuffer）：高性能，但复杂
    • 推荐：轻量用 Redis Pub/Sub，重任务用消息队列

    12. Worker Threads 和 Cluster 如何选型？¶

    • cluster：多进程，适合多核扩展 Web 服务吞吐
    • worker_threads：同进程多线程，适合 CPU 密集计算与内存共享场景
    • 实战建议：I/O 密集优先 cluster，CPU 密集优先 worker_threads

    13. 大规模 I/O 密集型应用的优化策略？¶

    • 使用流式处理代替一次性加载
    • 连接池管理（数据库、Redis）
    • 合理设置 UV_THREADPOOL_SIZE
    • 避免阻塞事件循环（拆分 CPU 密集任务到 Worker Threads）
    • 使用缓存（Redis 多级缓存）减少 I/O

    基于Proxy：

    const reactive = (obj) => {
    return new Proxy(obj, {
        get(target, key) {
            const result = Reflect.get(target, key);
            track(target, key);  // 收集依赖
            return typeof result === 'object'
                ? reactive(result)  // 深层响应式
                : result;
        },
        set(target, key, value) {
            Reflect.set(target, key, value);
            trigger(target, key);  // 触发更新
            return true;
        }
    });
};

    6. React中key的作用？¶

    作用： 帮助React识别哪些元素改变了（同级元素唯一标识）。

    • key 帮助 React 的 diff 算法识别元素变化，优化渲染性能
    • 不推荐用 index 作为 key（列表顺序变化时会导致错误复用）

    原则：

    • 使用唯一且稳定的key
    • 不要使用index（会破坏性能，还可能导致状态问题）
    • 使用id等业务标识
    // 错误
{items.map((item, index) => (
    <Item key={index} />
))}

// 正确
{items.map(item => (
    <Item key={item.id} />
))}

    为什么不用index会有问题：

    // 如果items = ['a', 'b']，index = [0, 1]
// 删除第一项后，items = ['b']，index = [0, 1]
// React会认为第一个key=0没变，但实际上对应的是'b'而非'a'

    • Render 阶段：构建 Fiber 树，进行 diff，标记副作用（可中断）
    • Commit 阶段：将变更同步到 DOM（不可中断）
    • Before mutation → Mutation → Layout 子阶段

    8. useMemo 和 useCallback 的区别与使用场景？¶

    • useMemo：缓存计算结果，依赖不变返回缓存值
    • useCallback：缓存函数引用，避免子组件不必要的重渲染
    • 注意：不要无差别使用，仅在性能瓶颈处使用（配合 React.memo）

    9. React 18 的并发特性有哪些？¶

    • useTransition / useDeferredValue：标记低优先级更新
    • 自动批处理：setState 在异步回调中也自动合并
    • Suspense 支持 SSR 流式渲染

    10. Context 的性能问题及优化方案？¶

    • 问题：Context 值变化时所有消费者都会重渲染
    • 优化：拆分 Context、使用 useMemo 稳定引用、基于订阅的外部状态库（如 Zustand）

    11. React Fiber架构？¶

    Fiber 是 React 16 重新实现的协调引擎，数据结构为链表形式的 Fiber 节点树，支持双缓存机制。

    解决的问题：

    • CPU长任务分片，避免长任务阻塞主线程
    • 支持任务分片优先级调度
    • 支持可中断更新

    Fiber双缓冲：

    • current树（正在渲染）
    • workInProgress树（正在构建）

    优先级：

    优先级 场景
    Immediate synchronous
    UserBlocking onClick、onChange
    Normal 数据获取
    Low 动画
    Background 预渲染

    commit阶段： 同步执行DOM更新

    13. Redux数据流？¶

    单向数据流：

    Action → Dispatch → Reducer → New State → View

    核心概念：

    // Action
const addTodo = (text) => ({
    type: 'ADD_TODO',
    payload: { text, id: Date.now() }
});

// Reducer
const todoReducer = (state = [], action) => {
    switch (action.type) {
        case 'ADD_TODO':
            return [...state, action.payload];
        case 'DELETE_TODO':
            return state.filter(t => t.id !== action.payload);
        default:
            return state;
    }
};

// Store
import { createStore } from 'redux';
const store = createStore(todoReducer);

// 订阅
store.subscribe(() => console.log(store.getState()));

// 分发
store.dispatch(addTodo('Learn Redux'));

    RTK（Redux Toolkit）简化写法：

    import { createSlice, configureStore } from '@reduxjs/toolkit';

const todoSlice = createSlice({
    name: 'todos',
    initialState: [],
    reducers: {
        addTodo: (state, action) => {
            state.push(action.payload);
        }
    }
});

const store = configureStore({
    reducer: {
        todos: todoSlice.reducer
    }
});

    2. HTTP缓存机制？¶

    Cache-Control：

    Cache-Control: max-age=3600        # 最大缓存时间
Cache-Control: no-cache            # 每次都验证
Cache-Control: no-store            # 不缓存
Cache-Control: private             # 仅浏览器缓存
Cache-Control: public              # 可被CDN缓存

    Etag/If-None-Match：

    服务器响应: ETag: "abc123"
客户端请求: If-None-Match: "abc123"
匹配则返回304

    Last-Modified/If-Modified-Since：

    服务器响应: Last-Modified: Wed, 01 Jan 2025 00:00:00 GMT
客户端请求: If-Modified-Since: Wed, 01 Jan 2025 00:00:00 GMT
未过期则返回304

    缓存优先级：

    1. Cache-Control: no-store → 不缓存
    2. Cache-Control: no-cache → 每次验证
    3. Cache-Control: max-age=0 → 每次验证
    4. 过期检查 → ETag/Last-Modified

    3. HTTPS工作原理？¶

    TLS握手过程：

    1. Client Hello - 客户端支持的TLS版本、加密套件、随机数
    2. Server Hello - 服务端选择加密套件、发送证书、随机数
    3. 客户端验证证书 - 使用CA公钥验证
    4. Pre-Master Secret - 客户端生成随机数，用公钥加密发送
    5. 生成会话密钥 - 双方用三个随机数生成对称密钥
    6. 加密通信

    加密算法：

    • 非对称加密：RSA/ECDSA（握手阶段）
    • 对称加密：AES/ChaCha20（通信阶段）
    • 数字签名：ECDSA/RSA
    • 密钥交换：DH/ECDH

    证书验证：

    浏览器 → 操作系统信任库 → 中间证书 → 根证书

    5. TCP三次握手和四次挥手？¶

    三次握手：

    Client                  Server
  │                       │
  │──── SYN (seq=x) ─────▶│  1. 客户端发送SYN
  │                       │
  │◀─── SYN+ACK ─────────│  2. 服务器返回SYN+ACK
  │                       │
  │──── ACK (seq=y) ─────▶│  3. 客户端发送ACK
  │                       │
  │      建立连接          │

    四次挥手：

    Client                  Server
  │                       │
  │──── FIN ─────────────▶│  1. 主动方发送FIN
  │◀─── ACK ──────────────│  2. 被动方返回ACK
  │                       │
  │◀──── FIN ─────────────│  3. 被动方发送FIN
  │───── ACK ────────────▶│  4. 主动方返回ACK
  │                       │
  │      断开连接          │

    为什么是三次： 确认双方的发送和接收能力 为什么是四次： FIN和ACK分开，确保数据传完

    1. XSS攻击与防御？¶

    XSS（跨站脚本攻击）： 向页面注入恶意脚本

    类型：

    类型 触发方式
    存储型 持久化到数据库

    return str

        .replace(/"/g, '&quot;')

    2. CSRF攻击与防御？¶

    CSRF（跨站请求伪造）： 利用用户已登录状态发送恶意请求

    防御：

    // 1. CSRF Token
// 服务器生成随机token，表单携带

// 2. SameSite Cookie

    • Strict - 完全禁止跨站cookie
