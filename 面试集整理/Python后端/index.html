    一、基础¶

    基础语法与类型系统¶

    1. Python 解释型、动态类型是什么意思？¶

    解释型语言：

    • Python 代码由 Python 解释器逐行读取并转换为字节码后执行，无需像 C/C++ 那样提前编译成机器码
    • 优点：开发效率高、跨平台能力强、修改代码即可运行
    • 缺点：执行效率相对编译型语言较低（通常比 C++ 慢 2-10 倍）
    • 工业场景：适合业务迭代快的 AI 算法服务，不适合极致性能要求的核心链路

    动态类型语言：

    • 变量类型在运行时确定，使用时无需声明类型
    • 优点：代码简洁、开发灵活、适合快速原型开发
    • 缺点：运行时才能发现类型错误，增加调试成本；IDE 类型推断能力受限
    • 工业场景：接口参数建议使用类型提示（Type Hint）提高可维护性
    # 动态类型示例
x = 10        # int
x = "hello"   # str，运行时确定类型
x = [1, 2, 3] # list

# 类型提示（工业级代码推荐）
def predict(model_input: dict) -> dict:
    return {"prediction": 0.95}

    工程建议：

    • 面向后端项目时尽量加类型标注
    • 核心链路配合静态检查工具，如 mypy、pyright

    2. Python 常见数据类型有哪些？¶

    基本数据类型：

    类型 说明 示例
    int 整数 10, 0x10
    float 浮点数 3.14, 1e10
    bool 布尔 True, False
    str 字符串 "hello"
    bytes 字节 b"hello"

    容器类型：

    类型 说明 示例
    list 列表 [1, 2, 3]
    tuple 元组 (1, 2, 3)
    dict 字典 {"a": 1}
    set 集合 {1, 2, 3}
    frozenset 不可变集合 frozenset([1,2])

    特殊类型：

    • None - 空值
    • complex - 复数 3+4j
    • range - 范围
    • slice - 切片对象

    3. list、tuple、dict、set 的区别和选型？¶

    特性 list (列表) tuple (元组) dict (字典) set (集合)

    5. *args 和 **kwargs 的作用和使用场景¶

    1. *args：将不定数量的位置参数打包为元组，常用于需要传递可变数量参数的函数。
    2. **kwargs：将不定数量的关键字参数打包为字典，常用于配置项传递或装饰器。
    3. 应用场景：编写装饰器时，常使用(*args, **kwargs)来保证装饰器能适配任何函数的参数签名。

    有时，我们定义函数格式如： func(x, y, *, a, b, c)，其中 * 是“位置参数终结符”：

    • * 之前：可以有普通位置参数。
    • * 之后：后面的所有形参，都只能是仅限关键字参数。

    典型的好处是为了安全性，避免函数调用时参数错位。

    1. 浅拷贝、深拷贝和直接赋值有什么区别？¶

    方式 说明 影响
    b = a 引用赋值 两者指向同一对象
    copy.copy(a) 浅拷贝 仅拷贝第一层
    copy.deepcopy(a) 深拷贝 递归拷贝所有层级

    = 赋值（引用赋值）：

    a = [1, [2, 3]]
b = a       # b 和 a 指向同一对象
b[0] = 10   # 修改 b，a 也被改变
print(a)    # [10, [2, 3]]

    注意：数字、字符串、元组 是不可变对象（不可变 = 一旦创建，就不能被修改）：

    a = 10
b = a
b = 20

    不是把 a 指向的 10 改成 20，而是：

    1. 在内存里新建一个数字 20
    2. 把标签 b 从 10 撕下来，贴到 20 上
    3. a 依然贴着 10，完全没动

    所以：

    • 不可变对象（数字、字符串、布尔）：就算用赋值 =，修改新变量也永远不会影响原变量
    • 可变对象（列表、字典、集合）：用赋值 =，修改会互相影响

    copy() 浅拷贝：

    import copy
a = [1, [2, 3]]
b = copy.copy(a)  # 只拷贝顶层
b[0] = 10         # 修改 b 不影响 a
b[1].append(4)    # 修改嵌套对象，a 也会变
print(a)          # [1, [2, 3, 4]]

    deepcopy 深拷贝：

    import copy
a = [1, [2, 3]]
b = copy.deepcopy(a)  # 递归拷贝所有层级
b[1].append(4)        # 完全独立
print(a)              # [1, [2, 3]]
print(b)              # [1, [2, 3, 4]]

    工业场景：

    • 浅拷贝：拷贝顶层结构（如列表元素是基础类型）
    • 深拷贝：多层嵌套对象（如 AI 模型的超参数配置、神经网络的层结构）
    • 多进程/多协程共享数据时必须深拷贝，防止数据竞争

    并发和配置场景里，深拷贝尤其重要，否则嵌套结构被共享后容易引发脏写。

    注意： 不可变对象（int、str、tuple）不会有拷贝问题，因为它们不可修改。

        return self
    def __next__(self):

        raise StopIteration

    并发编程¶

    1. GIL 是什么？高并发场景下如何规避 GIL 的影响？¶

    GIL（Global Interpreter Lock，全局解释器锁）：

    • Python 解释器为了保证线程安全，在任意时刻只允许一个线程执行 Python 字节码
    • 即使多核 CPU，同一时刻只有一个线程在执行 Python 代码

    直接影响：

    • CPU 密集型任务，多线程无法真正并行
    • IO 密集型任务，多线程依旧有价值，因为等待 IO 时会释放 GIL

    GIL 确保同一时刻只有一个线程执行 Python 字节码，限制了 CPU 密集型任务的多线程并行。 规避方法：

    • 使用多进程（multiprocessing）代替多线程；
    • 对 I/O 密集型任务，仍可用 asyncio 或线程，因为 GIL 在 I/O 时会释放；
    • 将计算密集型部分用 C 扩展、NumPy 或 Numba 实现；
    • 使用 concurrent.futures.ProcessPoolExecutor。

    实战建议：

    • IO 密集型：FastAPI + asyncio
    • CPU 密集型：multiprocessing 或任务下沉到其他语言服务
    • 大模型推理：不要指望纯 Python 线程解决吞吐问题

    4. async/await 的运行机制是什么？¶

    async/await 基于事件循环（如 asyncio），实现单线程并发，适合高 I/O 场景（如 HTTP 请求、数据库查询）。

    核心模型：

    • 协程函数定义为 async def
    • await 把执行权交还给事件循环
    • 事件循环在多个可继续执行的任务之间切换

    优势：

    • 单线程可处理大量 IO 任务
    • 切换开销小
    • 不需要大量线程

    限制：

    • 无法直接利用多核做 CPU 并行
    • 全链路需要异步支持，数据库、HTTP 客户端都要异步版本

    1. Python 的垃圾回收机制是什么？¶

    Python的垃圾回收（GC）机制 主要使用引用计数为主，标记清除和分代回收为辅的机制。

    1. 引用计数：每个对象维护引用次数，为0时立即释放。但无法处理循环引用（如两个对象互相引用）。
    2. 标记清除：专门解决循环引用。从根对象出发，标记所有可达对象，未标记的即为垃圾。
    3. 分代回收：基于“大部分对象生存周期短”的假设。新对象在0代，存活越久代数越高、检查频率越低。

    原因：

    • 引用计数让对象可即时释放
    • 循环引用无法靠引用计数独立解决，因此需要 GC 辅助

    2. 引用计数机制¶

    原理：每个对象维护一个引用计数器，记录当前有多少引用指向该对象。

    import sys

a = [1, 2, 3]      # 引用计数 = 1
b = a              # 引用计数 = 2
c = [a]            # 引用计数 = 3

print(sys.getrefcount(a))  # 实际多1，因为参数传递临时+1

del b              # 引用计数 = 2
del a              # 引用计数 = 1
c.append(4)        # 列表本身还是同一个对象
del c              # 引用计数 = 0，对象被回收

    优点：实时性强，垃圾立即回收 缺点：无法处理循环引用

    3. 循环引用与 GC 垃圾回收¶

    为什么会有循环引用问题？

    两个对象互相引用时，即使外部不再访问它们，引用计数也不会变成 0。

    这就是标记清除存在的意义。

    循环引用示例：

    # 循环引用，引用计数无法回收
a = []
b = [a]
a.append(b)  # a 引用 b，b 引用 a，引用计数都为 1 但已不可达

# 手动删除后仍然无法回收
del a
del b  # 此时 a 和 b 引用计数 0，但互相引用形成循环

    GC 垃圾回收器：解决循环引用问题

    import gc

# 触发垃圾回收
gc.collect()

# 禁用自动 GC
gc.disable()

# 手动分代回收
gc.collect(generation=2)  # 回收所有代

    分代回收策略：

    代 回收频率 对象生命周期
    Generation 0 最高 新创建的对象
    Generation 1 中等 从 0 代存活下来的对象
    Generation 2 最低 从 1 代存活下来的对象 
    # 查看各代对象数量
print(gc.get_count())    # GC 计数
print(gc.get_threshold()) # 触发阈值 (700, 10, 10)

    6.3.内存溢出示例¶
    1.场景 1：全局列表无限追加导致内存溢出（最常见）¶

    问题代码（内存泄漏 / 溢出）：

    # global_list 一直占用内存，无法被GC回收

    问题根源：

    全局变量生命周期和进程一致，持续累加引用，对象永远不会被回收。

    修复方案：

    1. 改用局部变量，函数结束自动释放
    2. 用完手动清空 list.clear()
    3. 分批处理，不要一次性全加载

    # 局部变量，函数结束自动释放
    temp_list = []
    for i in range(100000):
        temp_list.append({"id": i, "data": "x" * 1024})
    # 手动清空（稳妥）
    temp_list.clear()

if __name__ == "__main__":
    task()
    2.场景 2：循环引用导致 GC 无法回收内存¶

    问题代码（循环引用泄漏）

    class A:
    def __del__(self):
        print("A 对象被销毁")

class B:
    def __del__(self):
        print("B 对象被销毁")

def create_cycle():
    a = A()
    b = B()
    # 互相循环引用
    a.b = b
    b.a = a

if __name__ == "__main__":
    create_cycle()
    print("函数执行完毕，但对象没立刻回收")

    现象：函数结束后 __del__ 迟迟不执行，内存占用降不下来。

    解决办法:

    1. 手动打破引用
    2. 主动调用 gc.collect() 强制垃圾回收
    3. 避免实例互相挂属性

    修复代码:

    import gc

class A:
    def __del__(self):
        print("A 对象被销毁")

class B:
    def __del__(self):
        print("B 对象被销毁")

def create_cycle():
    a = A()
    b = B()
    a.b = b
    b.a = a
    # 手动打破循环引用
    a.b = None
    b.a = None

if __name__ == "__main__":
    create_cycle()
    # 强制触发垃圾回收
    gc.collect()

    6.6.总结¶
    1.内存溢出 / 内存泄漏 典型表现¶

    核心表现：内存只涨不跌、越跑越慢、CPU/swap 飙高、程序卡死或被 OOM 杀死、抛 MemoryError，重启后临时恢复，过段时间复现。

    • 系统层级表现

    • 程序内存持续走高，只涨不跌

      运行时间越久，占用内存（RSS/VIRT）不断上涨，不会自动回落。

    • 系统卡顿、整机变慢

      Python 进程占用大量物理内存，挤占其他程序内存，系统整体卡顿、鼠标卡顿。

    • 被系统 OOM 杀死

      Linux 下触发 OOM Killer，进程直接被系统强制干掉，日志无正常退出痕迹。

    • 交换分区（Swap）占用飙升

      物理内存耗尽，大量使用磁盘 Swap，程序运行极慢、卡死假死。

    • 程序运行表现

    • 运行速度越来越慢

      内存不足触发频繁 GC 垃圾回收、缺页中断，业务接口 / 循环逻辑耗时越来越高。

    • 程序假死、卡死无响应

      主线程被 GC 阻塞、内存分配等待，请求卡住、接口超时、无日志输出。

    • 频繁 GC、CPU 莫名飙高

      GC 不停扫描回收对象，CPU 占用居高不下，业务逻辑没跑多少，CPU 却爆满。

    • Python 程序报错表现

    • 直接抛出 MemoryError

      内存彻底耗尽，申请新对象内存失败，代码直接抛错崩溃。

    • 子进程 / 线程莫名崩溃、无故退出

      内存溢出导致子进程内存分配失败，悄无声息退出。

    • 日志丢失、局部逻辑异常

      内存紧张时部分 IO、日志写入异常，报错零散、无规律。

    • 业务层面表现

    • 接口响应时间逐分钟变长

      服务跑的越久，RT 越来越大，重启后立刻恢复正常。

    • 批量任务越跑越慢，后期卡住

      循环任务、数据清洗任务前期正常，中后期内存累积，执行效率断崖式下跌。

    • 重启程序后瞬间恢复正常

      内存溢出 / 泄漏最典型特征：重启自愈，过一段时间又复现

    • 开发 / 运维观测特征

    • top/htop 看到 Python 进程 RES 内存持续上涨

    • 容器环境：Pod 内存水位持续拉高，触发 容器 OOM 重启
    • 日志无报错，但内存曲线单边上涨
    • 局部变量、全局容器无限膨胀，对象数量只增不减
    2.内存溢出通用排查步骤¶
    1. 监控内存：用 psutil/memory_profiler 查看内存走势
    2. 排查全局变量：检查是否有全局列表 / 字典持续累加
    3. 检查循环引用：用 gc 模块查看不可回收对象
    4. 优化数据读取：大文件、大数据集分批 / 逐行处理
    5. 及时断引用：不用的对象手动置 None、clear()
    6. 避免常驻对象：不用长期持有大量实例
    7. 慎用 del：不依赖析构做资源释放，改用上下文管理器
    3.通用避坑总结¶
    1. 尽量少用全局容器存大量数据；
    2. 大数据分批处理，不一次性加载到内存；
    3. 避免类实例循环互相引用；
    4. 不用的对象手动 del、置 None、清空容器；
    5. 线上用 psutil 定时监控进程内存，提前告警。

    1. Python 后端里如何选择多线程、多进程和协程？¶

    Web API、数据库、HTTP 调用 协程或多线程 IO 密集，等待多

    5. 幂等设计¶

    5.1.核心思路¶
    1. 唯一标识 + 去重记录 每次操作生成一个全局唯一的请求 ID（例如 UUID、业务流水号、Snowflake ID），服务端记录已处理过的 ID，重复请求直接返回之前的结果。
    2. 状态机控制 将业务对象设计为有限状态机，只允许从当前状态流转到特定下一状态。重复请求如果状态已经变更，则忽略或返回成功。
    3. 乐观锁 / 条件更新 利用数据库版本号（version）或时间戳（updated_at），更新时带上旧值条件。若条件不满足，说明已被其他请求更新，本次更新无效。
    4. 写时检查 + 原子操作 将“查询-判断-写入”合并为一个原子操作（如数据库唯一约束、Redis 的 SETNX、Lua 脚本），避免并发导致的重复写入。
    5. 业务层面的天然幂等 某些操作本身是幂等的，例如 DELETE 一个已不存在的资源、SET 一个固定值（SET key = value）、无副作用的只读查询。
    5.2.实现方式¶
    1. 全局唯一请求 ID（去重表 / Redis 幂等表）¶

    流程：

    • 客户端生成 requestId，或服务端根据业务参数（如订单号+时间戳）生成。
    • 服务端接收请求后，先去重存储中查找 requestId 是否存在。
    • 若不存在，则执行业务逻辑，并将 requestId 存入存储（设置过期时间）。
    • 若存在，直接返回上一次处理结果（或忽略）。

    实现选型：

    • 数据库唯一索引：创建一个幂等表，字段为 requestId、业务结果等。插入成功则处理，重复插入会报唯一冲突，捕获后直接返回。
    • Redis：使用 SETNX requestId value EX ttl，原子性设置并设置过期时间（避免无限堆积）。
    • 分布式缓存（如 Memcached）：需要保证高可用和持久化时慎用。

    优点：简单通用，适用于任何需要幂等的写操作（下单、支付、消息消费）。 缺点：需要额外存储，依赖存储的可靠性；跨多个服务时需要共享同一个去重中心。

    2. 数据库唯一约束（业务键）¶

    利用数据库表的唯一索引，直接阻止重复数据的写入。例如：

    • 支付流水表：对 (订单号, 支付类型) 建立唯一索引。
    • 消息消费记录表：对 (消息ID, 消费者) 建立唯一索引。

    优点：强一致性，无需额外查询。 缺点：只能防重复插入，不能防止重复更新（可结合版本号）。

    3. 乐观锁（版本号）¶

    适用于更新操作，例如：

    UPDATE account SET balance = balance - 100, version = version + 1
WHERE id = 1 AND version = 5;

    如果 version 不匹配，说明已被其他请求修改，当前更新失败。

    优点：无需额外存储，性能高。 缺点：需要业务方处理更新失败时的重试或回滚。

    4. 悲观锁（分布式锁）¶

    对于并发极高但业务要求强幂等的场景，可以使用分布式锁（Redis Redlock、ZooKeeper）。对业务唯一键加锁，确保同一时刻只有一个请求能执行。

    优点：简单粗暴，能避免任何并发重复。 缺点：性能损耗大，容易成为瓶颈，锁超时处理复杂。

    5. 状态机模式¶

    将业务流程设计为状态机，每个状态只允许特定的输入。例如订单状态：待支付 → 已支付 → 已发货。重复的“支付”请求检测到订单状态已经是“已支付”，直接返回成功。

    优点：业务清晰，天然防止重复流转。 缺点：需要完整设计状态和转换规则，不适合无状态的操作。

    6. 消息队列的幂等消费¶

    MQ 会保证 at-least-once 投递，消费者必须实现幂等。常用方案：

    • 使用业务表唯一键（如订单号）避免重复插入。
    • 利用 Redis 记录最近处理的消息 ID（滑动窗口）。
    • 使用数据库版本号处理更新。
    5.3.推荐策略¶
    场景 推荐方式 说明
    创建资源（如创建订单） 业务唯一键 + 数据库唯一约束 例如订单号、流水号作为唯一索引。
    更新资源（如扣减库存） 乐观锁（版本号）或 状态机 性能好，能防止 ABA 问题。
    HTTP API 调用 客户端 requestId + Redis 幂等表 适合前端防重复提交、重试场景。
    消息消费 业务唯一键 + 去重表 保证 at-least-once 下的精确一次。
    文件上传 / 触发外部通知 Token 机制（先获取 token，提交时携带 token，服务端删除 token） 类似于防重复提交。
    跨服务调用（分布式事务） 全局唯一 ID + 最终一致性 + 幂等接口 结合 TCC、Saga 模式。
