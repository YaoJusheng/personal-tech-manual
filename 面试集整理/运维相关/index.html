    一、基础¶

    运维方法论与 Linux¶

    技术栈与工具选择¶

    领域 主流方案 说明
    Linux 发行版 Ubuntu LTS / Rocky Linux / Debian 生产环境优先稳定版本
    服务管理 systemd / supervisor systemd 为主流
    远程管理 SSH / Mosh SSH 必备，Mosh 适合弱网
    批量运维 Ansible / SaltStack Ansible 上手快、生态广
    系统性能分析 perf / eBPF / sysstat 从系统调用到函数级定位

    Go / Python / Rust 落地建议¶

    • Go：服务排障优先结合 pprof 与系统指标联动，避免只看应用日志。
    • Python：重点关注 GIL、worker 模型（gunicorn/uvicorn）和内存增长趋势。
    • Rust：优先使用结构化日志和指标埋点，定位问题时结合 perf 与火焰图。

    通用面试题¶

    1. 运维（Ops）、DevOps、SRE 的区别是什么？¶
    概念 核心目标 关注点
    运维（Ops） 系统稳定运行 部署、监控、故障响应
    DevOps 缩短交付周期 研发运维协同、流水线自动化
    SRE 可靠性工程化 SLI/SLO、错误预算、自动化治理

    面试建议：先说三者关系是递进而不是替代，再结合你做过的流程改造或稳定性治理案例。

    2. Linux 线上排障标准顺序是什么？¶

    推荐顺序：

    1. 看整体负载：uptime、top、htop
    2. 看 CPU：mpstat、pidstat、top -H
    3. 看内存：free -h、vmstat 1
    4. 看磁盘：df -h、du -sh、iostat -x 1
    5. 看网络：ss -s、ss -lntp、iftop
    6. 看日志：journalctl、应用日志、中间件日志
    uptime
top -H -p <pid>
pidstat -p <pid> 1
perf top -p <pid>

    分析要点：

    • 单核满通常是热点线程/锁竞争/死循环。
    • 多核满通常是流量激增、批任务抢占、无上限并发。
    • 必须联动看 QPS、错误率、GC、下游超时。
    3. 内存问题怎么排查？¶
    free -h
vmstat 1
ps aux --sort=-%mem | head
pmap -x <pid> | head -n 30
cat /proc/meminfo

    常见根因：

    • 缓存膨胀（可回收）
    • 堆内存泄漏（持续增长）
    • 线程过多导致栈内存占用
    • OOM 前频繁抖动（swap/回收压力）
    4. 磁盘与 I/O 问题怎么排查？¶
    df -h
du -h --max-depth=1 /var | sort -h
iostat -x 1

高频考点：

- `df` 看的是文件系统，`du` 看的是目录实际占用。
- 删除日志后磁盘未释放，常见是文件句柄未关闭（`lsof +L1`）。
- I/O 高要看 `await`、`util`、队列长度，不只看带宽。

##### 6. 网络故障排查链路是什么？

```bash
ss -tulnp
ss -s
ping <ip>

    步骤建议：DNS -> 连通性 -> 端口监听 -> TLS 握手 -> 应用层响应。

    5. Linux 信号和 IPC 高频点有哪些？¶

    常用信号：SIGTERM（优雅停机）、SIGKILL（强制）、SIGINT、SIGUSR1/2。
    IPC 常见方式：管道、消息队列、共享内存、信号量、Socket。
    面试常追问：共享内存最快，但同步复杂度也最高。

    Docker 与容器基础¶

    技术栈与工具选择¶

    领域 主流方案 说明
    容器引擎 Docker / containerd / Podman 生产集群常见 containerd
    镜像构建 Docker BuildKit / Kaniko / Buildah CI 无特权构建常用 Kaniko
    镜像仓库 Harbor / ECR / ACR / GCR Harbor 常用于私有化

    Go / Python / Rust 落地建议¶

    • Go：优先多阶段构建，产出静态二进制镜像（体积小、启动快）。
    • Python：用 slim 基础镜像并固定依赖版本，避免运行时下载依赖。
    • Rust：构建阶段缓存依赖，运行阶段使用 distroless 或最小镜像。

docker volume prune -f
    3. Dockerfile 最佳实践有哪些？¶
    1. 多阶段构建，减少最终镜像体积。
    2. 固定依赖版本，保证可复现。
    3. 使用非 root 用户运行。
    4. 合并层并清理缓存。
    5. 增加健康检查和合理启动命令。
    4. Docker 网络和数据持久化怎么答？¶
    • 网络模式：bridge、host、overlay、none。
    • 持久化：匿名卷、命名卷、绑定挂载。
    • 生产建议：数据目录用命名卷或云盘挂载，配置用只读绑定挂载。

log() { echo "[$(date '+%F %T')] $*"; }

retry() {
    local n=0
    local max=3
    local delay=2
    while true; do
        "$@" && break || {
            n=$((n+1))
            if [[ $n -ge $max ]]; then
                log "command failed after $n attempts: $*"
                return 1
            fi
            sleep $delay
        }
    done
}
