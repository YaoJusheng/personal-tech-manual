    一、基础¶

    基础语法与类型系统¶

    1. Go 语言适合做后端的原因是什么？¶

    维度 特性 对后端的价值
    性能 编译型语言，接近 C/C++ 适合高并发 API、网关、调度服务
    部署 单二进制，无运行时依赖 发布简单，容器化友好
    并发 goroutine + channel 易于写出高并发程序
    工程性 标准库完整 http、JSON、sync、context 开箱可用
    稳定性 静态类型 + 编译期检查 线上类型错误少，维护成本低

    典型场景：

    • API 网关与鉴权服务
    • 消息消费者和调度中心
    • 数据采集、边缘服务
    • AI 推理接入层

    2. Go 有哪些常见基本类型？byte 和 rune 有什么区别？¶

    常见基础类型：

    • 布尔：bool
    • 整型：int、int8、int16、int32、int64、uint 等
    • 浮点：float32、float64
    • 复数：complex64、complex128
    • 字符串：string

    byte 和 rune：

    • byte 是 uint8 的别名，通常表示一个字节
    • rune 是 int32 的别名，表示一个 Unicode 码点，中文字符在 UTF-8 编码下占 3 个字节，用 rune 处理才能正确计数。
    s := "Go语言"
fmt.Println(len(s))         // 8，字节数
fmt.Println(len([]rune(s))) // 4，字符数

    3. new 和 make 的区别是什么？¶

    维度 new(T) make(T, args...)
    作用 分配内存 + 零值初始化 分配并初始化 slice、map、channel
    适用类型 任意类型（struct、int、slice、map、channel 等） 仅三种引用类型：slice、map、channel
    返回值 返回 *T（指针） 返回 T（值本身，不是指针）
    零值行为 分配内存后设为类型零值 完成内部数据结构初始化（如 slice 的 header、map 的哈希桶、channel 的缓冲区）

    结论：

    • 需要指针时优先考虑 new
    • 初始化 slice、map、channel 必须使用 make

    new：分配 + 零值，返回指针¶

    // 1. 基本类型
p := new(int)       // *int，值为 0
fmt.Println(*p)     // 0

// 2. 结构体
type User struct {
    Name string
    Age  int
}
u := new(User)      // *User，{Name: "", Age: 0}
u.Name = "Alice"    // 通过指针访问

// 3. 甚至可以用 new 分配 slice/map（但极少这样用）
s := new([]int)     // *[]int，指向一个 nil slice
m := new(map[string]int)  // *map[string]int，指向一个 nil map
// 注意：*s 和 *m 此时还不能直接 append 或赋值，需要再初始化

    make：初始化引用类型的内部结构¶

    // 1. slice：分配底层数组，返回 slice header
s := make([]int, 5, 10)   // len=5, cap=10，底层数组已分配，元素零值

// 2. map：初始化哈希表结构，返回可用 map
m := make(map[string]int) // 可直接 m["key"] = 1，不会 panic

// 3. channel：初始化 channel 结构体和环形缓冲区
ch := make(chan int, 5)   // 缓冲通道，容量 5

    4. Go 是值传递还是引用传递？¶

    Go 只有值传递。

    • 基础类型、数组、结构体作为参数时传递副本
    • 想修改原对象，需要传指针
    // 值传递 - 传递副本
func inc(a int) {
    a = 100 // 不影响原值
    a++
}

// 引用传递 - 传递指针
func incPtr(a *int) {
    *a = 100 // 影响原值
    *a++
}

    注意：

    • slice、map、channel、interface、func本身就是引用类型，传递的是"指针"
    • 数组作为参数是值传递（建议用slice）

    5. defer 的执行顺序和常见陷阱是什么？¶

    defer 是 Go 语言最核心的特性之一，它的设计哲学是："申请资源后立即声明释放，让申请和释放成对出现，即使中间发生 panic 也能保证执行"。

    • defer 语句立即注册一个延迟调用，不会马上执行
    • 被 defer 的函数在外层函数返回前执行
    • 执行顺序：多个 defer 遵循 LIFO 后进先出，像栈一样最后声明的先执行。
    • 参数求值：defer 语句声明时，参数值就已经被捕获，不是执行时才取值。
    • 返回值影响：如果函数使用命名返回值，defer 中可以修改该返回值；如果是匿名返回值，则不影响。
    func test() (i int) {  // 命名返回值
    defer func() { i += 1 }()
    return 0            // 最终返回 1
}

    规则：

    1. defer 在声明时就完成参数求值
    2. 在外层函数返回前执行
    3. 多个 defer 按后进先出执行
    4. 命名返回值可被 defer 修改

    常见陷阱：

    • 循环中直接捕获循环变量
    • 忘记理解“参数在注册时求值，不是在执行时求值”

    2. channel 有哪些用途和状态？如何区分有无缓冲？¶

    channel：

    • goroutine 间的通信机制
    • 遵循"不要通过共享内存来通信，而要通过通信来共享内存"
    • 类型：有缓冲/无缓冲、只读/只写/双向
    2.1. channel 的核心用途¶
    1. 传递数据
    2. 协程同步
    3. 并发限流
    4. 任务分发与流水线
    // 1. 通信：传递数据
ch <- value

// 2. 同步：协程间协调
done := make(chan struct{})
go func() {
    defer close(done)
    // 工作
}()
<-done  // 等待完成

// 3. 限流：令牌桶/漏桶
sem := make(chan struct{}, 10)  // 最多 10 并发
sem <- struct{}{}
go func() {
    defer func() { <-sem }()
    // 处理
}()

// 4. 任务分发
tasks := make(chan Task, 100)
go workerPool(tasks)  // 多个 worker 从同一 channel 消费
    2.2. 有缓冲和无缓冲区别¶
    • 无缓冲：发送和接收必须同时就绪，强调“握手同步”
    • 有缓冲：可先写入缓冲区，适合削峰填谷和异步流水线

done := make(chan struct{})

// 有缓冲：可存储 N 个元素再阻塞

sem := make(chan struct{}, 10)

// 广播：关闭 channel，所有接收者立即收到零值
close(done)

    使用建议：

    • 纯信号通知优先 chan struct{}
    • 需要强同步、确认交接时使用无缓冲 channel
    • 需要吞吐和生产消费解耦时使用有缓冲 channel
    2.3. channel 的几种状态¶

    正常接收 阻塞直到有数据 阻塞直到缓冲区有数据

    • nil channel 常被用来“动态关闭某个 case”
    • close 后只能接收，不能发送
    • 有缓冲 channel 的行为要结合容量和当前元素个数理解

    3. 如何避免 channel 死锁？¶

    常见死锁原因：

    • 无接收者却发送到无缓冲 channel
    • for range 读取一个永不关闭的 channel
    • 主协程阻塞等待，子协程也在互相等待
    • 没有为异常分支设计超时或退出机制

    常见规避方式：

    // ❌ 死锁：无缓冲 channel 且无人接收
ch := make(chan int)
ch <- 1  // 阻塞在此

// ✅ 解决方案1：使用缓冲
ch := make(chan int, 1)
ch <- 1

// ✅ 解决方案2：select + default
select {
case ch <- 1:
default:
    // 处理溢出
}

// ✅ 解决方案3：超时
select {
case ch <- 1:
case <-time.After(5 * time.Second):
    // 超时处理
}

// ✅ 解决方案4：关闭 channel
close(ch)  // 接收方收到零值

    额外建议：

    • 明确谁负责关闭 channel
    • 广播取消优先使用 close(done) 或 context.Context
    • 需要超时的逻辑一律带 select

    5. select 的作用是什么？¶

    select 用于监听多个 channel 操作，是 Go 处理并发 IO、超时和取消的核心语法。

    特性：

    • 等待多个channel就绪
    • 随机选择一个可执行的case
    • 如果所有case都阻塞，执行default（如果有）
    • 如果没有default，则阻塞直到某个case就绪

    示例：

    select {
case v := <-ch1:
    fmt.Println("ch1:", v)
case v := <-ch2:
    fmt.Println("ch2:", v)
case <-ctx.Done():
    return ctx.Err()
case <-time.After(time.Second):
    return errors.New("timeout")
default:
    fmt.Println("default")
}

    应用场景：

    • 多路复用
    • 超时控制
    • 优雅关闭
    • 避免阻塞

    Context 用于跨 goroutine 传递：

    • 取消信号

    • 截止时间
    • 请求级元数据

    使用场景：

    • HTTP 请求超时
    • 数据库查询取消
    • 服务间链路超时传播
    • goroutine 生命周期控制

    7. 什么是 goroutine 泄漏（Goroutine Leak）？如何避免？¶

    Goroutine 泄漏指 协程被创建后无法退出，持续占用调度器和内存资源。

    Goroutine 最怕“发出去收不回来”。典型泄漏场景：一个 Goroutine 在等待一个永远不会收到数据的 channel。

    典型原因：

    • 永远阻塞在 channel 收发，死锁
    • 忘记监听退出信号，channel忘记关闭
    • 发送数据到无接收者的channel，下游不可用，重试逻辑没有退出条件

    规避策略与方案：

    • 使用 context.Context 传递取消信号，select 时监听 ctx.Done() ，所有后台协程都接入 ctx.Done()
    • 对外部依赖调用统一设置超时
    • channel 生产者在结束时关闭 channel
    • 为消费者设计退出协议，使用 sync.WaitGroup 等待所有 Goroutine 完成再退出主函数
    // 使用context控制生命周期
func worker(ctx context.Context) {
    for {
        select {
        case <-ctx.Done():
            return  // 收到取消信号，退出
        default:
            // 正常处理
        }
    }
}

    8. sync.Map 的原理和适用场景是什么？¶

    sync.Map 内部主要由两个区域组成：

    • read：原子读区域，读多写少场景下可无锁读取
    • dirty：写入区域，需要加锁维护

    核心思想：read + dirty双字典:

    type Map struct {
    mu Mutex
    read atomic.Value // readOnly
    dirty map[interface{}]*entry
    misses int
}

    基本原理：

    1. 读操作先查read（无锁），未命中加锁查dirty
    2. 写操作只操作dirty
    3. miss 达到阈值后将 dirty 提升为新的 read
    4. 删除操作标记为expunged

    特点：

    • 适合读多写少场景
    • 正常路径无锁
    • 首次加载慢

    适用场景：

    • 读多写少
    • key 生命周期不稳定
    • 不方便手动维护锁粒度

    不适用场景：

    • 高频写入
    • 需要复合操作原子性

    9. Go 中的并发控制实现¶

    Go 不推荐使用共享内存 + 锁的方式（虽然也支持 sync.Mutex），而是倡导 “不要通过共享内存来通信，而要通过通信来共享内存”。

    sync.Mutex / sync.RWMutex – 共享内存加锁

    保护临界区，RWMutex 允许多读单写。

    type Counter struct {
    mu    sync.Mutex
    value int
}

func (c *Counter) Inc() {
    c.mu.Lock()
    c.value++
    c.mu.Unlock()
}

func (c *Counter) Value() int {
    c.mu.Lock()
    defer c.mu.Unlock()
    return c.value
}

    二、进阶¶

    调度与内存管理¶

    1. Go 的 GMP 调度模型是什么？¶

    三元素：

    • G（Goroutine）：协程，用户态的轻量级线程，栈初始仅 2KB，可动态增长。
    • M（Machine）：操作系统线程，通常 1MB，实际执行 Goroutine 的载体。
    • P（Processor）：逻辑处理器，管理 G 的队列，负责调度 Goroutine 到 M 上执行，每个 P 维护一个本地运行队列。

    调度机制：

    • P 的数量由 GOMAXPROCS 决定，默认等于 CPU 核心数, P 有自己的 G 队列（本地队列）。
    • M 从 P 获取 G 执行
    • P 可绑定 M 执行，也可空闲
    • G 阻塞时，M 绑定其他 P 继续执行
    • 当 M 阻塞（如系统调用）时，P 会与 M 分离并附着到其他空闲 M 上，保证 P 始终尽量忙碌。
    • 工作窃取（work stealing）：当某个 P 的本地队列为空时，会从其他 P 或全局队列偷取一半 Goroutine 执行。
    ┌─────────────────────────────────────────┐
│                   P                     │
│  ┌───┐ ┌───┐ ┌───┐ ┌───┐                │
│  │ G1│ │ G2│ │ G3│ │ G4│  (本地队列)     │
│  └───┘ └───┘ └───┘ └───┘                │
│        ↑      ↑                         │
│        │      └─── 正在运行              │
│        └── 可运行，等待 M                 │
└─────────────────────────────────────────┘
          │
          ▼
┌─────────────────────────────────────────┐
│                   M                     │
│  OS Thread (系统线程)                    │
└─────────────────────────────────────────┘

    优势：

    • 极低的上下文切换成本, 创建/销毁 Goroutine 开销极低（相比线程）。
    • 阻塞时自动切换，无需显式 await，编程简单。
    • 利用多核并行，适合 CPU 密集型与 I/O 密集型混合场景。

    它支撑高并发的原因：

    • goroutine 轻量
    • 用户态调度成本低
    • 阻塞切换自动化
    • 能充分利用多核

    2. Go 的 内存分配管理机制¶

    Go 的内存分配可以概括为“TCMalloc 风格 + 多级缓存分配”。

    • 栈优先：小对象直接分配在栈上
    • 逃逸分析：确定对象在栈还是堆
    • TCMalloc：多核并行分配，减少锁竞争

    TCMalloc算法 + 三级分配：

    层级 大小 说明
    mspan <32KB 对象分配
    mcache Per-P 每个 P 的本地缓存，分配最快
    mcentral 共享 全局缓存，按 size class 管理共享对象池
    mheap 全局 堆内存管理

    对象大小分类：

    • 微小对象（<16B）：使用sizeclass，多个对象打包到一个slot
    • 小对象（16B-32KB）：直接按sizeclass分配
    • 大对象（>32KB）：直接从heap分配

    面试里建议补充：

    • 小对象优先走本地缓存，减少锁竞争
    • 大对象会直接向堆申请
    • 频繁分配和释放会直接影响 GC 压力

    3. Go 的 GC 是如何工作的（垃圾回收）？¶

    现代 Go GC 主要基于：

    • 三色标记
    • 并发标记清扫
    • 混合写屏障

    目标：

    • 尽量降低 Stop The World 时间
    • 在吞吐和延迟之间平衡

    常见调优项：

    • GOGC=100：堆增长到上次 GC 后的 100% 时触发新 GC
    • GOMEMLIMIT：设置进程内存上限

    补充：

    • GC 并非越少越好，要结合延迟、吞吐、对象分配速率看
    • 高并发服务里对象池和复用能显著减轻 GC 压力
    2.1.垃圾回收机制¶

    Go 自 1.8 版本后采用 并发三色标记清除 + 混合写屏障，目标是 低延迟（STW 微秒~毫秒级） 并尽可能减少对 CPU 吞吐的影响。

    演进历史：

    版本 GC类型
    Go 1.3 串行标记+Sweep
    Go 1.5 并行标记+Sweep（写屏障）
    Go 1.8 混合写屏障
    Go 1.12 幅流式清扫
    Go 1.17 糕慢删除
    Go 1.21 收割器（Skyscower）

    调优参数：

    GOGC=100  // 默认，heap增长100%时触发GC
GOMEMLIMIT=1073741824  // 内存限制
    1. 三色标记流程¶

    将对象分为三种颜色（逻辑状态）：

    • 白色：尚未被访问的对象，回收的目标。
    • 灰色：已访问但未完成对其引用扫描的对象。
    • 黑色：已扫描完毕，确认可达的对象。

    过程（并发执行）：

    1. 初始标记 (Mark Setup)：短暂 STW，开启写屏障，启动并发标记。
    2. 并发标记 (Concurrent Mark)：从根对象（goroutine 栈、全局变量等）开始深度遍历，将可达对象标记为灰色或黑色。此阶段与用户代码并发执行，利用写屏障维护增量正确性。
    3. 重新标记 (Mark Termination)：短暂 STW，处理并发标记期间新增的引用变化，完成所有标记。
    4. 并发清除 (Concurrent Sweep)：回收白色对象内存，与用户代码并发。Go 使用空闲链表管理内存，清除仅将内存放回链表，无需立即归还 OS。
    2. 混合写屏障 (Hybrid Write Barrier)¶
    • 目的：保证在并发标记阶段，用户代码修改对象引用时不破坏标记结果的正确性。
    • 实现：当用户修改指针（如 *p = q）时，写屏障会捕获这个操作，并将新引用的对象（q）直接标记为灰色，防止其被误回收。
    • 优势：相比于插入写屏障或删除写屏障，混合写屏障避免了重新扫描大量栈内存，大幅缩短了 STW 时间。
    3. GC 触发条件与调参¶
    • 主动触发：runtime.GC() 强制执行。
    • 被动阈值：当堆内存大小相比上次 GC 结束时增长超过 GOGC 百分比（默认 100）时触发。例如上次 GC 后堆为 10MB，当堆达到 20MB 时触发下一轮 GC。
    • 其他触发：程序超过 2 分钟没有 GC 也会触发（防止内存无限增长）。
    4. Go GC 的内存分配优化¶
    • 栈分配：Go 编译器通过逃逸分析，将不会逃逸到堆上的对象直接在栈上分配，栈内存由函数调用自动回收，无 GC 压力。
    • 内存分配器：采用类似 tcmalloc 的多级缓存（P 与 M 绑定的小对象缓存），减少锁竞争并降低 GC 扫描范围。
    • 无分代、无压缩：Go 没有分代概念，也不移动对象（即不压缩内存），因为移动对象需更新大量指针，在并发标记环境下极其复杂。这可能导致内存碎片，但官方认为碎片程度可接受。
    5. 性能特征¶
    • STW 时间：早期 Go 版本 STW 可达百毫秒；现在通常在 微秒到几十微秒 级别（1.20 后甚至低于 100μs）。
    • CPU 开销：并发标记消耗约 25% 的 CPU 能力（可通过环境变量 GOMAXPROCS 调整标记阶段占用的线程数）。
    • 吞吐影响：对大多数网络服务而言，GC 造成的停顿几乎不可感知；但对极高吞吐的批处理任务（如高频内存分配），可通过调大 GOGC 减少 GC 频率，或使用 pprof 分析优化。
    2.2.GC 优势¶
    • 低延迟：增量式 GC（Concurrent Mark），暂停 < 1ms
    • 分代回收：新对象（Generation 0）频繁回收，老对象减少回收
    • 标记-清除：三色标记，非全程 STW

    对比 Java GC：

    • Java GC 暂停可能达 100ms-1s
    • Go GC 暂停 < 1ms，适合金融、实时系统

    4. 什么是逃逸分析（Escape Analysis）？¶

    逃逸分析决定变量分配在栈上还是堆上。

    逃逸： 变量从栈分配移动到堆分配。

    常见逃逸场景：

    • 返回局部变量指针
    • 接口装箱
    • 发送对象到 channel
    • 闭包捕获局部变量
    // 1. 返回指针
func foo() *int {
    a := 10
    return &a  // a逃逸到堆
}

// 2. 发送指针到channel
ch := make(chan *int, 1)
a := 10
ch <- &a  // a逃逸

// 3. slice/append导致重新分配
var s []int
s = append(s, 1)  // s可能逃逸

// 4. interface类型
var i interface{}
i = a  // 如果i是interface，a可能逃逸

    查看逃逸分析：

    go build -gcflags '-m' main.go

    优化思路：

    • 减少不必要的指针返回
    • 控制接口转换
    • 避免不必要闭包捕获
    • 高频路径减少临时对象创建

    8. 如何实现一个 Go 的 worker pool（协程池）？¶

    worker pool 本质上是固定数量的 goroutine 从任务队列里取任务，用于限制并发度和稳定吞吐。

    type Task struct {
    ID    int
    Data  interface{}
}

type WorkerPool struct {
    tasks    chan *Task      // 任务队列
    results  chan *Result   // 结果队列
    numWorker int
}

func NewWorkerPool(numWorker int, bufferSize int) *WorkerPool {
    pool := &WorkerPool{
        tasks:    make(chan *Task, bufferSize),
        results:  make(chan *Result, bufferSize),
        numWorker: numWorker,
    }

    for i := 0; i < numWorker; i++ {
        go pool.worker(i)
    }
    return pool
}

func (p *WorkerPool) worker(id int) {
    for task := range p.tasks {
        // 处理任务
        result := process(task)
        p.results <- result

        // 模拟处理
        time.Sleep(10 * time.Millisecond)
    }
}

func (p *WorkerPool) Submit(task *Task) {
    p.tasks <- task
}

func (p *WorkerPool) Close() {
    close(p.tasks)
    close(p.results)
}

// 使用
pool := NewWorkerPool(10, 1000)
for _, task := range tasks {
    pool.Submit(task)
}
pool.Close()

    适用场景：

    • 批量处理数据
    • 控制下游数据库或模型服务压力
    • 避免无限制 goroutine 膨胀

    工业场景：

    • 批量处理传感器数据
    • 并发模型推理请求
    • 图片/视频处理管道

    9. 是否需要在 Go 中使用线程池或进程池？¶

    先给出明确结论：绝大多数场景下，完全不需要使用传统意义上的线程池（即预先创建一批 OS 线程并分配任务）；使用进程池（类似多进程模型）也极少必要。 但是，在某些特殊情况下，可以使用 协程池（goroutine pool） 来限制并发数，这不是线程池。

    • 默认做法：直接 go func() 启动 goroutine，配合 WaitGroup、context、channel 进行同步和控制。无需担心线程开销。
    • 需要限制并发度时：使用有缓冲 channel 作为信号量，或启动固定数量的 worker goroutine（goroutine 池）。不要自己操作 OS 线程。
    • 绝对避免：手动创建并管理 M（无法直接操作），或引入线程池库去创建多个 M —— 这既无必要，也可能与 Go 调度器冲突。
    • 性能调优：若 goroutine 数量极大（> 百万），考虑将任务分片、减少锁竞争、使用 sync.Pool，或调整 GOMAXPROCS 和 runtime.GOMAXPROCS 参数。
    9.1. 为什么通常不需要线程池？¶
    • goroutine 足够轻量：一个线程（M）可以承载成千上万个 goroutine，创建 100 万个 goroutine 的开销（内存约 2GB）远小于 100 万个线程（不可行）。线程池的意义在于复用昂贵的 OS 线程，而 Go 的 M 已经由运行时管理，无需用户干预。
    • GMP 自动优化：当 goroutine 阻塞（如等待 channel、系统调用），调度器会自动切换，空闲的 M 可用于其他 G，线程不会因为一个 G 阻塞而浪费。用户手动实现线程池反而会破坏这种自适应能力。
    • 并发控制用 channel 更清晰：限制并发数量使用带缓冲 channel（工人模式）比维护线程池更简单、更 Go 风格。
    9.2. 什么时候需要“池化”思想（但仍是 goroutine 池）？¶

    虽然不需要线程池，但限制同时运行的 goroutine 数量是常见需求（本质是控制并发度），例如：

    • 防止下游数据库/API 被压垮。
    • 控制内存占用（虽然 goroutine 轻量，但数量过多（>10万）也会有调度开销和栈内存增长）。
    • 实现稳定的吞吐量（如批处理系统每批次固定 worker 数量）。

    此时我们实现的是 goroutine pool，例如：

    • 使用 ants 库（高性能 goroutine 池）。
    • 或自定义 worker pool 模式：启动固定数量的 worker goroutine，从中读取任务 channel。

    注意：这仍然是在用户态复用 goroutine，底层 OS 线程由 Go 运行时管理，与 Java/CPP 中创建固定线程池完全不同。Goroutine 池仅仅是为了限制并发度，而不是为了节省线程创建开销。

    9.3. 什么时候可能需要“进程池”？¶

    极少数场景下会用到进程池：

    • 隔离性要求极高：某些任务可能崩溃导致整个进程退出，如果使用子进程可以隔离开（如 Chrome 多进程架构）。Go 本身有 recover 捕获 panic，但仍不能解决 CGO 或外部库导致的 segfault，此时可考虑 exec 子进程。
    • 绕过 Go 的 GC 或内存限制：单个 Go 进程对内存管理有全局影响，若某块任务消耗巨大且不可控，可以拆分为独立进程，利用操作系统的进程级清理。
    • 使用多语言组件：需要调用 Python/Node 脚本，通过进程池（如 os/exec + 队列）复用进程。

    以上场景极为少见，在普通业务开发中几乎不会遇到，更常见的是使用 微服务 而非多进程。

    9.4. 不要混淆“线程/进程池”与“Goroutine + channel 并发模型”¶
    概念 Go 中的对应实现 是否需要
    线程池（复用 OS 线程） 不需要，GMP 运行时自动管理 M 否
    进程池（多个进程） 极少情况用 os/exec + 进程控制 基本不需要
    限制并发 goroutine 数 带缓冲 channel 或 worker pool 常见需求（但注意这不是线程池）
    限制 CPU 核心使用 设置 GOMAXPROCS 按需设置
    9.5.示例¶
    9.5.1.固定 Worker Pool（经典 Channel 实现）¶

    这是最基础、最可控的池化模型：启动固定 N 个 worker，从共享的 jobs channel 中取任务执行。

    package main

import (
    "fmt"
    "sync"
    "time"
)

// Task 是我们要执行的工作单元
type Task struct {
    ID int
}

// WorkerPool 管理固定数量的 worker
type WorkerPool struct {
    workers  int
    jobs     chan Task
    wg       sync.WaitGroup
}

// NewWorkerPool 创建池
func NewWorkerPool(workers int, queueSize int) *WorkerPool {
    return &WorkerPool{
        workers: workers,
        jobs:    make(chan Task, queueSize),
    }
}

// Start 启动所有 worker
func (p *WorkerPool) Start() {
    for i := 0; i < p.workers; i++ {
        p.wg.Add(1)
        go p.worker(i)
    }
}

// worker 实际处理任务
func (p *WorkerPool) worker(id int) {
    defer p.wg.Done()
    for task := range p.jobs {
        fmt.Printf("worker %d 处理任务 %d\n", id, task.ID)
        time.Sleep(100 * time.Millisecond) // 模拟耗时
    }
}

// Submit 提交任务（非阻塞，若队列满会阻塞）
func (p *WorkerPool) Submit(t Task) {
    p.jobs <- t
}

// Shutdown 优雅关闭：关闭通道，等待所有任务处理完
func (p *WorkerPool) Shutdown() {
    close(p.jobs)
    p.wg.Wait()
}

// 使用示例
func main() {
    pool := NewWorkerPool(10, 100) // 10 个 worker，队列容量 100
    pool.Start()

    // 模拟海量请求
    for i := 0; i < 1000; i++ {
        pool.Submit(Task{ID: i})
    }

    pool.Shutdown()
    fmt.Println("所有任务完成")
}

    适用场景：任务量已知、需要精准控制并发数、不希望过载。

    9.5.2.信号量模式（Goroutine 限流）¶

    如果不想维护长生命周期的 worker，只想限制同时运行的 goroutine 数量，可用带缓冲 channel 作信号量。

    func main() {
    var wg sync.WaitGroup
    maxConcurrency := 10
    sem := make(chan struct{}, maxConcurrency)

    urls := make([]string, 1000) // 模拟 1000 个请求
    for _, url := range urls {
        wg.Add(1)
        sem <- struct{}{} // 获取令牌
        go func(u string) {
            defer wg.Done()
            defer func() { <-sem }() // 释放令牌

            // 实际业务逻辑
            fmt.Println("处理:", u)
            time.Sleep(100 * time.Millisecond)
        }(url)
    }
    wg.Wait()
}

    适用场景：不需要持久 worker，只是偶尔需要限制瞬时并发数。

    9.5.3.生产级推荐：ants 协程池¶

    自己管理 worker 的生命周期、任务队列、动态扩容、优雅退出是比较复杂的。ants 是目前最优秀的 Go 协程池库，性能极高，内部复用 goroutine，支持定期清理、panic 恢复、超时等。

    package main

import (
    "fmt"
    "sync"
    "time"

    "github.com/panjf2000/ants/v2"
)

func main() {
    var wg sync.WaitGroup

    // 创建池，容量 10
    pool, _ := ants.NewPoolWithFunc(10, func(i interface{}) {
        defer wg.Done()
        // i 是传入的任务参数
        fmt.Println("处理任务:", i)
        time.Sleep(100 * time.Millisecond)
    })
    defer pool.Release()

    // 批量提交任务
    for i := 0; i < 1000; i++ {
        wg.Add(1)
        _ = pool.Invoke(i) // 阻塞直到有空闲 worker
    }

    wg.Wait()
    fmt.Println("所有任务完成")
}

    如果你不需要固定函数，也可以使用通用池：

    pool, _ := ants.NewPool(10)
defer pool.Release()

var wg sync.WaitGroup
for i := 0; i < 1000; i++ {
    wg.Add(1)
    task := i
    pool.Submit(func() {
        defer wg.Done()
        fmt.Println("处理:", task)
    })
}
wg.Wait()

    优势：

    • 自动复用 goroutine，减少创建销毁开销。
    • 支持动态调整池大小、超时、panic 捕获。
    • 线程安全，Benchmark 性能接近原生 goroutine。

func (c *Context) Next() {
    c.index++
    for c.index < len(c.handlers) {
        c.handlers[c.index](c)
        c.index++
    }
}

    7. Go语言如何实现TCP服务器？¶

    func main() {
    listener, _ := net.Listen("tcp", ":8080")
    defer listener.Close()

    for {
        conn, err := listener.Accept()
        if err != nil {
            continue
        }
        go handleConn(conn)
    }
}

func handleConn(conn net.Conn) {
    defer conn.Close()
    buf := make([]byte, 1024)
    for {
        n, err := conn.Read(buf)
        if err != nil {
            return
        }
        conn.Write(buf[:n])
    }
}
