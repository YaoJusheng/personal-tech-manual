    一、基础¶

    1. 关系型数据库¶

    1.1 MySQL¶

    1. MySQL存储引擎有哪些？区别？¶
    引擎 事务 锁粒度 外键 索引 适用场景
    InnoDB 支持 行级锁 支持 Clustered Index OLTP、默认
    MyISAM 不支持 表级锁 不支持 Non-Clustered OLAP、只读
    MEMORY 不支持 表级锁 不支持 Hash Index 临时表、缓存
    ARCHIVE 不支持 行级锁 不支持 - 日志、归档
    CSV 不支持 表级锁 不支持 - 数据交换

    InnoDB特性：

    • MVCC（多版本并发控制）
    • 支持ACID事务
    • 行级锁
    • 外键约束
    • Crash Recovery
    • Buffer Pool缓存

    MyISAM特性：

    • 全文索引
    • 表压缩
    • COUNT(*)优化
    • 空间函数（GIS）
    2. MySQL索引类型有哪些？¶
    索引类型 说明 示例
    主键索引 唯一非空 PRIMARY KEY
    唯一索引 唯一可空 UNIQUE
    普通索引 普通 INDEX idx_name(col)
    联合索引 多列 INDEX idx(a,b,c)
    全文索引 文本搜索 FULLTEXT(col)
    前缀索引 字符串前缀 INDEX idx(col(10))

    InnoDB索引结构： B+Tree

            [root]
        /    \
     [node]  [node]
     / | \   / | \
   ...  ...  ... ...

叶子节点：双向链表

    聚集索引 vs 非聚集索引：

    特性 聚集索引 非聚集索引
    表中数量 只能有1个 可有多个
    叶子节点 存储完整数据 存储主键值
    查找速度 快 需回表
    3. 什么是回表查询？如何避免？¶

    回表： 先查普通索引获取主键，再查主键索引获取完整数据。

    -- 创建普通索引
CREATE INDEX idx_name ON users(name);

-- 执行查询
SELECT * FROM users WHERE name = 'Tom';
-- 1. 先在idx_name中找到name='Tom'对应的主键id=5
-- 2. 再用id=5回表查主键索引获取整行数据

    避免回表：

    -- 方式1：覆盖索引（查询列都在索引中）
SELECT name, age FROM users WHERE name = 'Tom';
-- 只需idx_name索引，无需回表

-- 方式2：建立联合索引
CREATE INDEX idx_name_age ON users(name, age);
    4. 最左前缀原则是什么？¶

    联合索引的匹配规则： 从索引最左列开始，依次向右匹配。

    CREATE INDEX idx ON users(name, age, phone);

-- 可以使用索引的情况
SELECT * FROM users WHERE name = 'Tom';           -- 使用name
SELECT * FROM users WHERE name = 'Tom' AND age = 25;  -- 使用name, age
SELECT * FROM users WHERE name = 'Tom' AND age = 25 AND phone = '123'; -- 使用全部

-- 不能完全使用索引
SELECT * FROM users WHERE age = 25;              -- 不能使用
SELECT * FROM users WHERE name = 'Tom' AND phone = '123'; -- 只能使用name

    原理： B+Tree按照(name, age, phone)顺序构建，跳跃中间列会导致索引失效。

    5. MySQL事务特性（ACID）？¶

    ACID：

    特性 说明 实现机制
    Atomic（原子性） 事务要么全做，要么全不做 Undo Log
    Consistency（一致性） 事务执行前后数据一致 其他三个特性保证
    Isolation（隔离性） 并发事务互不干扰 MVCC + 锁
    Durability（持久性） 提交后数据永久保存 Redo Log

    隔离级别：

    级别 脏读 不可重复读 幻读
    Read Uncommitted 可能 可能 可能
    Read Committed 不可能 可能 可能
    Repeatable Read（MySQL默认） 不可能 不可能 可能
    Serializable 不可能 不可能 不可能

    1.2 PostgreSQL¶

    1. PostgreSQL特点？¶
    特性 PostgreSQL MySQL
    ACID 完全支持 InnoDB支持
    SQL标准 高度兼容 部分兼容
    事务隔离级别 4种+可序列化 4种
    并发控制 MVCC + 锁 MVCC + 锁
    索引类型 多种 较少
    JSON支持 JSON/JSONB JSON
    Full-text search 内置 需插件
    扩展性 强 较弱
    性能优化 复杂但强大 相对简单

    优势场景：

    • 复杂查询
    • 数据完整性要求高
    • GIS应用（PostGIS）
    • JSON处理（JSONB）
    • 自定义函数/类型
    • 复制复杂需求
    2. PostgreSQL索引类型？¶
    索引类型 命令 适用场景
    B-tree CREATE INDEX 默认，范围/等值查询
    Hash USING hash 等值查询
    GiST USING gist 地理空间、全文搜索

    7. 索引失效的场景有哪些？¶
    • 未遵循最左前缀：联合索引 (A, B, C)，查询条件中没有用到A，则索引会整体失效。
    • 类型转换：假设 name 是字符串类型，SQL 里写成 WHERE name = 123，会触发隐式类型转换导致索引失效。
    • 函数/运算操作：WHERE YEAR(create_time) = 2023，在索引列上使用函数会失效，应改为范围查询。

    8. 事务 ACID & 隔离级别¶
    1. 脏读

    定义：事务A读取到了事务B 已修改但尚未提交 的数据。如果事务B随后回滚，事务A读到的就是无效的“脏数据”。

    • 发生时机：事务B修改数据后，在提交前，事务A就读取了。

    • 后果：基于不存在的数据做决策，比如错误地发货。

    • 解决方案：将隔离级别至少设为 READ COMMITTED

    • 不可重复读

    定义：同一个事务内，两次读取 同一行数据，得到的结果 值不同。侧重在数据的 内容被修改或删除。

    • 发生时机：事务A两次读取之间，事务B修改了同一行并提交。

    • 后果：事务A内数据前后不一致，逻辑判断错乱。

    • 解决方案：

      • 路径一：使用 REPEATABLE READ 隔离级别

      • 原理：依靠 MVCC。事务开始时创建一个一致性快照，之后所有读操作都读这个快照，不理会外部的修改。MySQL 默认就是这个级别。

      • 路径二：使用乐观锁（应用程序层面控制）

      • 适用：当你保持在 READ COMMITTED 级别，但仍需防止关键数据被中途修改时。

      • 实现方式：在表中加一个 version 字段。
      -- 1. 先查出当前版本
SELECT balance, version FROM account WHERE id = 1;
-- 假设 balance=1000, version=1

-- 2. 更新时必须同时校验版本号，版本号不符合则更新失败
UPDATE account
SET balance = balance - 500, version = version + 1
WHERE id = 1 AND version = 1;
      • 优点：不需要锁定数据，只在提交更新时检查，性能高。适合读多写少的场景。

    • 幻读

    定义：同一个事务内，以相同条件执行两次查询，结果的 行数不同。侧重在 有新的数据插入或删除。

    • 发生时机：事务A两次范围查询之间，事务B 插入了新数据 并提交。
    • 后果：统计数据不准，或执行更新操作时漏掉新插入的数据。
    • 解决方案：
      • 通用方案：SERIALIZABLE 隔离级别
      • 原理：所有涉及的行范围都会被加上严格的锁，禁止其他事务插入、删除数据，保证前后查询结果集完全一致。
      • MySQL 特有方案：利用间隙锁
      • 原理：MySQL 在 REPEATABLE READ 级别下，对范围查询使用的 间隙锁，会锁定索引记录之间的间隙，阻止新数据插入。
      • 验证：在 MySQL 的 RR 级别下，一个事务执行 SELECT*FROM user WHERE ageBETWEEN 20 AND 30 FOR UPDATE; 会锁定 [20, 30] 这个区间，其他事务无法插入 age=25 的新数据，从而防止幻读
      • 注意：这只是锁住了被查询的间隙，如果新插入的数据落在这些间隙之外，仍可能产生幻读。
    隔离级别 脏读 不可重复读 幻读 适用场景
    读未提交 (READ UNCOMMITTED) 可能 可能 可能 追求性能可接受
    读已提交 (READ COMMITTED) ❌ 可能 可能 Oracle 默认
    可重复读 (REPEATABLE READ) ❌ ❌ 可能 MySQL 默认
    串行化 (SERIALIZABLE) ❌ ❌ ❌ 严格一致性 
    -- MySQL 可重复读 + MVCC
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
START TRANSACTION;
-- 事务内读取一致，不受其他事务影响
COMMIT;

    MySQL隔离级别:

    1. 默认级别：MySQL InnoDB 默认是 Repeatable Read (可重复读)。
    2. 解决的问题：通过 MVCC（多版本并发控制），解决了脏读和不可重复读；又通过 间隙锁（Gap Lock），在很大程度上避免了幻读。

    MySQL的行级锁：

    • 记录锁：锁定精确匹配到的行。
    • 间隙锁：锁定索引记录间的空隙，阻止其他事务在此插入，防止幻读。在 RR 级别下生效。
    • 临键锁：记录锁 + 间隙锁的组合，是 RR 级别下最常见的锁形态，锁住一个左开右闭的区间。
    • 意向锁：表级锁，表明事务打算在表中加行锁，用于帮助表锁和行锁快速判定冲突。

    分布式事务：

    • XA 事务：两阶段提交，强一致但性能差
    • TCC：Try-Confirm-Cancel，性能好但实现复杂
    • 最终一致性：消息队列 + 补偿事务

    10. MySQL 的主从复制原理？¶

    原理：

    1. 二进制日志：主库将所有数据变更写入 Binlog。
    2. I/O 线程：从库启动一个 I/O 线程连接主库，接收 Binlog 并写入本地的 Relay Log（中继日志）。
    3. SQL 线程：从库的 SQL 线程读取 Relay Log，将里面的操作在从库上重放一遍，实现数据同步。
    Master                    Slave
  │                          │
  │──写事务─────────────────▶│ binlog dump
  │                          │
  │◀───IO_thread────────────│ 拉取binlog
  │                          │
  │                          │───relay log
  │                          │
  │                          │───sql_thread执行
  │                          │

    复制方式：

    方式 原理 优点 缺点
    异步复制 主不等待从 性能高 可能丢失数据
    半同步复制 主等待至少一个从 数据安全 有延迟
    全同步复制 主等待所有从 最安全 性能差

    3.PostgreSQL EXPLAIN 详解¶

    PostgreSQL 的执行计划是树形节点结构，比 MySQL 更贴近底层算法。

    3.1.用法¶
    -- 基础（估算成本）
EXPLAIN SELECT * FROM users WHERE age > 20;

-- 实际执行 + 真实时间 + 缓冲区命中（PG 最强分析组合）
EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)
SELECT * FROM users WHERE age > 20;

-- 查看触发器、约束等额外开销
EXPLAIN (ANALYZE, VERBOSE, COSTS, BUFFERS, FORMAT TEXT)
SELECT * FROM orders WHERE user_id = 100;
    3.2.节点类型（核心）¶
    节点 含义 类比 MySQL
    Seq Scan 顺序扫描堆表 type=ALL
    Index Scan 索引扫描 + 回表取数据 type=ref/range + 回表
    Index Only Scan 仅索引扫描（覆盖索引） type=index + Extra=Using index
    Bitmap Index Scan → Bitmap Heap Scan 位图索引扫描 批量索引查询后按物理顺序回表，减少随机 IO
    Nested Loop 嵌套循环 JOIN 小表驱动大表
    Hash Join 哈希 JOIN 大表 JOIN，构建哈希表
    Merge Join 归并 JOIN 两表已排序，高效
    Sort 显式排序 Extra=Using filesort
    Aggregate 聚合 GROUP BY / 聚合函数
    Limit 限制返回行数 LIMIT
    Gather 并行查询收集结果 PG 9.6+ 并行特性
    3.3.成本解读¶
    EXPLAIN SELECT * FROM users WHERE age > 20;
-- 输出：
-- Seq Scan on users  (cost=0.00..35.50 rows=1000 width=64)
    字段 含义
    cost=0.00..35.50 启动成本..总成本（单位是"页面读取次数的估算"，非毫秒）
    rows=1000 估算返回行数
    width=64 估算每行平均字节数
    3.4.Buffers 详解（PG 独有且极重要）¶
    EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM big_table;

    输出中的：

    • Shared Hit：从共享缓冲区（内存）命中
    • Shared Read：从磁盘读取
    • Shared Written：写入磁盘
    • Temp Read/Written：临时文件 IO（通常意味着内存不足，使用了磁盘排序/哈希）

    调优目标： 让 Shared Hit 占比极高，Temp 为 0。

    3.5.MySQL vs PostgreSQL EXPLAIN 对比¶
    特性 MySQL PostgreSQL
    输出结构 扁平表格 树形节点
    真实执行 EXPLAIN ANALYZE（8.0.18+） EXPLAIN (ANALYZE)（长期支持）

    覆盖索引标识 Extra: Using index Index Only Scan

    6. PostgreSQL 的 MVCC 机制与 MySQL 有何不同？为什么说 PG 的 MVCC 更适合高并发读写？¶

    参考答案：

    • PG 的 MVCC：通过元组内的 xmin（插入事务 ID）和 xmax（删除/更新事务 ID）实现。更新 = 插入新元组 + 标记旧元组无效。无 undo log 回滚段，直接保留旧版本。
    • 优势：读操作永不阻塞写操作，写操作也尽量不阻塞读（仅在唯一索引检查时短暂锁）。没有回滚段争用，并发性能稳定。
    • 代价：需要定期 VACUUM 清理死元组，否则表膨胀。

    MySQL的MVCC：

    InnoDB 在每行记录后保存两个隐藏字段：trx_id（最近修改它的事务ID）和 roll_pointer（指向 Undo Log 版本链的指针）。读操作根据当前事务的 ReadView，沿着版本链找到一个对当前事务“可见”的快照数据，从而实现读不加锁，极大提升并发性能。

    与 MySQL InnoDB 对比：

    • InnoDB 使用 undo log 记录旧版本，回滚段可能成为瓶颈。
    • PG 的旧版本直接存在数据页中，读可见性判断快（仅比较事务 ID 快照），适合写入极频繁的场景。

    1. 预加载（Eager Loading）：

    2. Django: User.objects.prefetch_related('orders')

    2. 在高并发场景下，预加载（JOIN）与分两次查询哪种更好？为什么？¶

    参考答案：

    • JOIN 方式：
    • 优点：一次网络往返，适合数据量小且关联表索引好。
    • 缺点：当一对多关系时结果集会膨胀（用户数 × 订单数），传输了大量重复数据；复杂 JOIN 可能导致 MySQL/PostgreSQL 优化器选错索引。
    • 分两次查询 + IN 条件：
    • 优点：结果集紧凑，无重复数据；可以利用各自表的最佳索引；更容易分页。
    • 缺点：两次网络请求（但可并行）。

    推荐：

    • 大多数场景下，两次查询更可控（尤其是关联数据多时）。
    • 当 N 很小且关联表记录也少时（如每个用户只有 2~3 个 profile），JOIN 更简单。
    • 在微服务中，两次查询还可能跨服务，此时应使用 批量 gRPC 或 DataLoader 合并请求。

    1. Redis分布式锁？¶

    潜在风险：

    • 使用 SET NX EX 命令加锁，释放时用 Lua 脚本校验 value 保证原子性。
    • 风险：锁超时导致任务未完成被释放；主从切换导致锁丢失；时钟跳跃影响过期时间。
    • 改进方案：Redlock（有限场景）或使用 etcd / ZooKeeper。

    实现方式：

    import redis

def acquire_lock(redis_client, lock_name, timeout=30):
    """获取锁"""
    lock_key = f"lock:{lock_name}"
    lock_value = str(uuid.uuid4())

    # NX=不存在时设置，EX=过期时间
    if redis_client.set(lock_key, lock_value, nx=True, ex=timeout):
        return lock_value
    return None

def release_lock(redis_client, lock_name, lock_value):
    """释放锁"""
    lock_key = f"lock:{lock_name}"

    # Lua脚本保证原子性
    lua = """
    if redis.call("get", KEYS[1]) == ARGV[1] then
        return redis.call("del", KEYS[1])
    else
        return 0
    end
    """
    redis_client.eval(lua, 1, lock_key, lock_value)

lock_value = acquire_lock(r, "order:1001")
if lock_value:
    try:
        # 业务逻辑
        process_order()
    finally:
        release_lock(r, "order:1001", lock_value)

    Redlock算法（多节点）：

    def redlock_acquire(redis_clients, lock_name, timeout=30):
    lock_value = str(uuid.uuid4())
    acquired = 0
    timeout_ms = timeout * 1000

    for client in redis_clients:
        if client.set(lock_key, lock_value, nx=True, px=timeout_ms):
            acquired += 1

    return acquired >= len(redis_clients) // 2 + 1

    模式 特点 适用场景
    主从 一主多从，读写分离 读多写少
    哨兵 自动故障转移 高可用
    Cluster 分片（16384 slot） 海量数据 + 高可用

    参考答案：

    • 哨兵模式（Sentinel）：
    • 解决主从自动故障转移，不解决分片。
    • 适合数据量不大，但要求高可用的场景。
    • 客户端连接哨兵获取当前 master 地址。
    • Redis Cluster：
    • 原生分布式，数据自动分片到 16384 个槽位。

    4. 什么是 Redis 热 key 问题？如何发现和解决？¶

    参考答案： 定义：某个 key 被超高并发访问（如秒杀商品、微博热搜），导致单节点 CPU 飙升。

    发现方法：

    • 使用 redis-cli --hotkeys 扫描（需开启 LFU 淘汰策略）。
    • 在客户端 SDK 中统计访问频率并上报。
    • 代理层（如 Twemproxy）记录 key 请求次数。

    解决方案：

    • 本地缓存：在服务内存中缓存热 key，设置较短过期时间，并定时异步刷新。
    • 分散热点：在 key 后加随机后缀，映射到多个副本节点（如 product_1、product_2），客户端随机访问。
    • 读写分离：增加只读从节点分担读压力。
    • 使用 Redis Cluster：热 key 会落到一个槽位，可借助 Proxy 再分片。

    AI 辅助：利用机器学习预测访问模式，动态调整副本数量或拆分策略。

    5. Redis 缓存三大问题：穿透、击穿、雪崩及解决方法¶
    问题 核心原因 主要危害 推荐解决方案
    穿透 查询不存在的数据 DB 被打爆，可能被恶意攻击 布隆过滤器 + 缓存空值
    击穿 单个热点 key 失效瞬间 DB 瞬时压力骤升 互斥锁 / 逻辑过期异步重建

    1. 缓存穿透（请求不存在的数据）¶

    定义： 查询一个不存在的数据（如用户 id = -1），由于缓存中没有，请求会直接穿透到数据库。恶意攻击者可以利用这一点，大量请求不存在的数据，导致数据库压力过大。

    解决方案：

    • 缓存空值：

    • 当数据库查询结果为空时，也将这个 key 缓存起来（value 为 null），设置较短的过期时间（如 1~5 分钟）。

    • 优点：简单，防止同一不存在 key 反复打到 DB。

    • 缺点：占用少量内存，可能被恶意构造大量不同不存在的 key 打满缓存。

    • 布隆过滤器（Bloom Filter）：

    • 在缓存前面加一层布隆过滤器，存放所有可能存在的 key（如所有用户 ID）。

    • 请求到来时，先用布隆过滤器判断 key 是否存在。若不存在，直接拒绝访问；若存在，才走缓存→DB。
    • 优点：极省内存（1% 误差率下每个 key 约 1.2 字节），防止任意不存在 key 攻击。
    • 缺点：有微小误判（可能把不存在的判断为存在），需要定期重建过滤器。
    # 解决方案：布隆过滤器
from bloom_filter import BloomFilter
bf = BloomFilter(capacity=1000000, error_rate=0.01)

def get_device(device_id):
    if device_id in bf:
        return redis.get(f"device:{device_id}")
    return None  # 布隆过滤判断不存在就直接返回

# 伪代码
def get_user(uid):
    # 布隆过滤器判断
    if not bloom_filter.exists(uid):
        return None

    user = cache.get(uid)
    if user:
        return user

    user = db.query("SELECT * FROM users WHERE id = ?", uid)
    if user:
        cache.setex(uid, 3600, user)
    else:
        cache.setex(f"null:{uid}", 60, "NULL")  # 缓存空值
    return user

    • 增强参数校验：

    • 在 API 层拦截明显非法 ID（如负值、超大值）。

    • 限流与黑名单：

    • 对同一 IP 或用户的异常访问进行封禁。

    2. 缓存击穿（热点 key 过期，瞬间大量请求）¶

    定义： 某个热点 key（如秒杀商品）在缓存失效的瞬间，大量并发请求同时打到数据库，导致数据库瞬时压力过大甚至崩溃。

    区别：击穿是针对单个热点 key，而雪崩是大量不同 key 同时失效。

    解决方案：

    • 互斥锁（Mutex）：

    • 当缓存失效时，只有第一个请求去数据库查询并重建缓存，其他请求等待。

    • 实现方式：使用 Redis 的 SETNX 或 Redisson 的分布式锁。

    • 伪代码：

      # 示例1
value = redis.get(key)
if value is None:
    if redis.setnx(lock_key, "1", expire=10):
        value = db.query()
        redis.setex(key, ttl, value)
        redis.delete(lock_key)
    else:
        time.sleep(0.01)
        value = redis.get(key)  # 重试

# 示例2
def get_user_lock(uid):
    key = f"user:{uid}"

    user = cache.get(key)
    if user:
        return user

    lock = redis.lock(f"lock:{key}", timeout=30)
    if lock.acquire():
        try:
            user = db.query("SELECT * FROM users WHERE id = ?", uid)
            cache.setex(key, 3600, user)
        finally:
            lock.release()
    else:
        time.sleep(0.1)
        return cache.get(key)

    • 缺点：会略微增加延迟，可能锁重试风暴，可结合“逻辑过期”优化。

    • 逻辑过期 + 异步重建：

    • 不设置物理过期时间，而是存储一个逻辑过期时间字段。

    • 当读取时发现逻辑过期，则尝试获取锁，获取成功的线程异步去更新缓存（更新期间返回旧数据）。
    • 优点：对吞吐影响小，不阻塞读请求。

    • 缺点：实现复杂，可能返回短暂脏数据。

    • 永不过期 + 后台定时更新：

    • 对热点 key 不设 TTL，而是由后台任务定期刷新。

    • 适合可接受的弱一致性场景（如配置项）。
    3. 缓存雪崩（大量 key 同时过期）¶

    定义： 大量缓存 key 在同一时间段内集中失效，或者 Redis 实例宕机，导致所有请求都落到数据库，数据库压力骤增甚至挂掉。

    解决方案：

    • 随机化过期时间：
    • 在原本固定的 TTL 上加一个随机偏移（如 3600 秒 ± 300 秒），避免同时失效。
    • 代码示例：expire_time = base_ttl + random.randint(0, 300)。
    • 缓存高可用：
    • 采用 Redis Cluster 或主从 + Sentinel，避免单节点故障导致全缓存不可用。
    • 使用多级缓存：本地缓存（Caffeine） + Redis，本地缓存可分担部分压力。
    • 熔断、降级、限流：
    • 通过 Hystrix、Sentinel 等组件，当 Redis 或数据库出现高延迟时，直接返回默认值或错误，保护系统。
    • 限制单位时间访问数据库的线程数（如使用 Semaphore）。
    • 提前预热：
    • 对于已知的热点数据（如大促活动），在活动开始前手动将数据加载到 Redis，并分散过期时间。
    • 双写 + 旁路更新：
    • 写操作时同时更新 DB 和 Redis，并设置较短的过期时间，利用异步任务修复不一致，减少集中失效

    • 适用于：数据仓库 ETL、微服务数据同步、升级迁移（在线从 11 到 14）。
    • 限制：DDL 不会自动复制；大事务会延迟；序列不同步。

    3. 分库分表后ID生成方案？¶

    方案对比：

    方案 特点 适用场景
    UUID 无中心，简单 不适合作为主键
    Snowflake 有序，有中心 趋势递增
    雪花ID变种 多区域支持 全球化
    数据库自增 简单，有中心 单库单表

    Snowflake实现：

    import time

class SnowflakeIDGenerator:
    def __init__(self, worker_id=0, datacenter_id=0):
        self.worker_id = worker_id
        self.datacenter_id = datacenter_id
        self.sequence = 0
        self.last_timestamp = -1
        self.epoch = 1609459200000  # 2021-01-01

        # 位移
        self.worker_id_bits = 5
        self.datacenter_id_bits = 5
        self.sequence_bits = 12

        self.max_worker_id = -1 ^ (-1 << self.worker_id_bits)
        self.max_datacenter_id = -1 ^ (-1 << self.datacenter_id_bits)

        self.worker_id_shift = self.sequence_bits
        self.datacenter_id_shift = self.sequence_bits + self.worker_id_bits
        self.timestamp_left_shift = (self.sequence_bits +
                                      self.worker_id_bits +
                                      self.datacenter_id_bits)
        self.sequence_mask = -1 ^ (-1 << self.sequence_bits)

    def _current_millis(self):
        return int(time.time() * 1000)

    def next_id(self):
        timestamp = self._current_millis()

        if timestamp == self.last_timestamp:
            self.sequence = (self.sequence + 1) & self.sequence_mask
            if self.sequence == 0:
                while timestamp <= self.last_timestamp:
                    timestamp = self._current_millis()
        else:
            self.sequence = 0

        self.last_timestamp = timestamp

        return ((timestamp - self.epoch) << self.timestamp_left_shift |
                (self.datacenter_id << self.datacenter_id_shift) |
                (self.worker_id << self.worker_id_shift) |
                self.sequence)

generator = SnowflakeIDGenerator(worker_id=1, datacenter_id=1)
id = generator.next_id()

    1. 有一张上亿数据量的用户表，如何安全地进行水平分库分表？¶

    答案要点： 这考察架构设计能力，不要只回答“分表”两个字。

    1. 分表键的选择：通常选 user_id 作为分片键，确保用户的所有查询落在一个分片上。
    2. 迁移方案：不能停机！
    3. 第一步：通过数据库中间件（如 ShardingSphere）或业务代码双写，将存量数据慢慢单向同步到新表。
    4. 第二步：检查数据一致后，并行运行双写新老库或通过网关灰度切流。
    5. 第三步：老库数据不再有写流量后下线。
