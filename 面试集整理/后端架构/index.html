    一、微服务架构¶

    1. 微服务拆分原则¶

    • 按业务域拆分
    • 高内聚低耦合
    • 数据独立，服务自治

    2. 微服务核心组件¶

    • API 网关
    • 服务注册发现
    • 配置中心
    • 链路追踪
    • 熔断限流
    • 日志聚合
    • 监控告警

    3. RESTful API 设计原则中，如何设计一个"获取某设备最新 N 条数据"的接口？¶

    GET /devices/{device_id}/readings?limit=10

    返回状态码 200、404、400，使用 JSON 格式

    二、高并发架构¶

    • 读写分离

    • 缓存架构
    • 异步化（MQ）
    • 削峰填谷
    • 限流熔断
    • 水平扩容

    三、高可用架构¶

    • 多实例部署
    • 异地多活
    • 故障自动转移
    • 幂等设计
    • 降级兜底

    四、AI 系统通用架构¶

    1. 接入层：Go 网关（抗流量、鉴权、限流）
    2. 业务层：微服务集群（Python/Go）
    3. AI 模型层：模型服务集群（Python/Triton）
    4. 数据层：MySQL + Redis + 时序库 + 向量库 + 对象存储
    5. 消息层：Kafka/RabbitMQ 解耦
    6. 调度层：任务调度、模型更新、灰度发布
    7. 运维层：监控、日志、链路追踪、容器编排

    五、模型部署与工程化¶

    1. Docker 和 Kubernetes 的作用？¶

    • Docker：容器化应用，保证环境一致
    • K8s：容器编排，自动扩缩容、滚动更新、服务发现，适合部署多算法模型

    2. 微服务架构下，Python 算法服务（慢、GPU）和 Go 接入服务（快）如何通信？¶

    • 使用 gRPC 或 HTTP + 异步任务队列（如 RabbitMQ、Kafka）
    • Go 负责接收请求，将任务发到队列，Python worker 消费并返回结果
    • 优点：解耦、可独立扩缩 Python 推理服务

    3. 如何保证算法模型在工业现场长期运行的稳定性？¶

    • 健康检查与自动重启（K8s liveness/readiness probe）
    • 模型版本管理与灰度发布（如 MLflow + K8s）
    • 输入输出异常检测、降级策略（模型不可用时回退到规则引擎）
    • 监控关键指标：推理耗时、内存、GPU 使用率、预测分布漂移

    4. 设计一个支持 1000 个工厂、每厂每秒 1 万条数据的实时异常检测系统架构¶

    • 边缘端：采集 + 轻量模型（如孤立森林）
    • 云端：Kafka 接收数据 → Flink/Spark Streaming 实时聚合 → Redis 缓存 → 核心模型（如 LSTM-AE）→ 告警通知
    • 数据库：时序数据库存储原始数据，关系库存储规则/配置
    • 语言：Go 编写高并发接入层，Python 编写模型训练与推理服务

    5. 模型部署方式¶

    • Flask/FastAPI 封装
    • gRPC 服务
    • Triton Inference Server
    • TorchServe / TensorFlow Serving
    • ONNX + TensorRT 加速
    • 边缘端部署（TensorRT / OpenVINO）

    6. 模型服务高可用¶

    • 多实例负载均衡
    • 模型预热
    • 动态扩缩容
    • 健康检查 & 自动重启
    • 灰度发布、AB 测试

    7. MLOps 架构¶

    • 数据采集 → 预处理 → 特征存储 → 训练 → 验证 → 打包 → 部署 → 监控 → 迭代

    8. 如何基于 LLM（如 OpenAI API）快速搭建一个代码评审辅助机器人？请描述技术方案。¶

    参考答案：

    • 架构：Webhook 接收 Git 的 MR/PR 事件 → 发送到消息队列（RabbitMQ）→ Worker 读取 diff → 调用 LLM API（如 GPT-4）分析潜在问题 → 将评论发回 Git 平台。
    • 提示词工程：要求模型关注性能、安全、可读性、异常处理，并给出修改建议。
    • 优化：缓存已评审的相似代码片段，使用 embedding 检索历史评审案例。
    • 注意：敏感代码不上传外部 LLM，可私有化部署 LLaMA 或 CodeLlama。

    六、综合高频¶

    1 基础必答题¶

    1. 进程、线程、协程区别
    2. 进程：资源分配最小单位，独立空间，开销大。
    3. 线程：调度最小单位，共享进程资源，切换有开销。

    4. 协程：用户态轻量级线程，切换极快，适合高并发 I/O。

    5. 同步、异步、阻塞、非阻塞 - 理解 I/O 模型差异

    6. HTTP/HTTPS 流程 - 三次握手、TLS 握手过程

    7. RESTful API 设计原则 - 资源、动词、状态码

    8. 什么是幂等性，如何实现 - GET/POST/PUT/DELETE 幂等性分析

    2 中间件 & 数据库¶

    1. MySQL 索引失效场景 - 函数、隐式转换、!=、% 前缀模糊

    2. Redis 缓存三大问题 - 穿透、击穿、雪崩及解决方案

    3. 消息队列作用与选型（Kafka/RabbitMQ） - 解耦、削峰、异步

    4. 分库分表方案 - 垂直/水平分片，分片键选择

    5. 时序数据库为什么快 - 写入优化、时间分区、压缩算法

    3 并发与高可用¶

    1. 如何设计限流算法（令牌桶 / 漏桶） - 算法原理与实现

    2. 熔断与降级区别 - 保护系统稳定性

    3. 分布式锁实现方案 - Redis SETNX、Zookeeper

    4. 如何保证接口高可用 - 多维度高可用策略

    • 异步队列削峰

    4.2. 设计工业设备数据平台架构¶

    • 边缘采集（Go）
    • MQ 缓冲
    • 时序存储
    • 实时计算
    • AI 异常检测
    • 业务库存储
    • 可视化大屏
    • 告警系统

    4.3. Go + Python 混合架构优势¶

    • Go：高并发、稳定、网关、采集、调度
    • Python：AI 模型、数据分析、算法迭代
    • 优势：开发效率与运行效率兼顾

    4.4.设计一个支持多产品线复用的中台服务（如用户鉴权、配置中心），你会考虑哪些扩展性要点？¶

    参考答案：

    • 多租户隔离：通过 tenant_id 字段分离数据和配置。
    • 版本管理：配置支持灰度发布、回滚（如 etcd + 版本号）。
    • API 设计：面向资源，使用 gRPC Gateway 同时支持 HTTP/2 和 REST。
    • 扩展点：插件化（如插件加载 Python 动态模块或通过 sidecar 容器）。
    • 可观测性：OpenTelemetry 统一 trace，Prometheus + Grafana 监控。
    • 降级方案：本地缓存配置 + 配置中心故障时使用本地默认值。

    4.5. 设计一个SSO的架构系统，要求支持10个以上微服务，且日活100万+？¶

    1.定位¶

    目标：支撑日活 100W+、10 + 微服务集群，实现全业务单点登录、统一鉴权、跨端互通、高可用、高并发、安全合规

    协议标准：主用 OIDC 1.0 + OAuth2.1，兼容 SAML2.0，内置 RBAC+ABAC 混合权限模型

    部署形态：分布式集群部署，无状态服务 + 有状态缓存分层

    3.核心能力清单¶
    1. 多端统一登录：账号密码、短信、扫码、人脸、第三方登录（微信 / 支付宝 / 企业账号）
    2. 全场景授权：授权码模式、隐式模式、密码模式、PKCE 移动端、设备码 Device Code
    3. 单点登录 SSO：一次登录全站通行，跨域名、跨子系统无感跳转
    4. 令牌体系：AccessToken (短期) + RefreshToken (长期) + IdToken (OIDC 身份)
    5. 统一鉴权：网关统一鉴权 + 服务本地二次鉴权双重校验
    6. 会话管理：强制下线、多端互踢、会话过期、异地登录风控
    7. 高并发支撑：百万日活、高峰 QPS 抗压、令牌缓存轻量化
    8. 安全能力：令牌黑名单、HTTPS、签名验签、防重放、密码加盐加密、权限最小化
    9. 租户隔离：支持多租户体系，适配企业多部门 / 多产品线隔离
    4.核心服务拆分（满足 10 + 微服务接入）¶
    4.1. Auth-Server 认证核心集群（无状态水平扩容）¶
    • 负责登录校验、生成三类令牌、OIDC 协议对接
    • 集群部署，负载均衡分流，轻松承接百万日活登录流量
    • 对外暴露标准 /authorize /token /userinfo 接口
    4.2. 用户中心 User-Service¶
    • 账号增删改查、用户资料、部门组织架构
    • 统一用户唯一 ID，全业务体系用户唯一标识
    4.3. 权限中心 Permission-Service¶
    • 角色管理、菜单权限、接口权限、数据权限
    • 输出统一权限接口，所有微服务统一拉取权限
    4.4. 客户端管理 Client-Service¶
    • 管理所有接入微服务 ClientID/ClientSecret
    • 配置回调地址、授权范围、登录有效期、白名单
    4.5. 会话会话 Session-Service¶
    • 登录会话存储、在线用户统计、强制下线
    • 会话状态全量落 Redis，极速读写
    4.6. 安全风控 Security-Service¶
    • 异常登录拦截、IP 风控、频繁登录限流、验证码风控
    4.7. 日志审计 Audit-Service¶
    • 登录日志、授权日志、权限变更日志、操作留痕
    5.百万日活高并发架构设计¶
    5.1. 流量分层削峰¶
    • 入口层：APISIX 网关做限流、熔断、灰度、IP 黑名单
    • 登录接口做单机 QPS 限制，防止爆破登录
    • 高峰登录流量放入本地缓冲 + Redis 排队削峰
    5.2. 缓存架构（核心提速）¶
    • Redis 主从 + 哨兵集群
    • AccessToken 缓存：5~15 分钟短期有效
    • RefreshToken 缓存：7~30 天
    • 用户基础信息、角色权限预热缓存
    • 令牌黑名单（注销 / 下线令牌快速失效）
    • 热点用户登录信息本地 JVM 二级缓存，减少 Redis 穿透
    5.3. 数据库架构¶
    • MySQL 分库分表：用户表按 ID 哈希分片
    • 读写分离：主库写账号信息，从库查登录 / 权限
    • 登录流水、审计日志冷数据归档，减轻库压力
    5.4. 无状态设计¶

    所有 SSO 核心服务无本地会话，状态全部下沉 Redis

    支持任意节点扩缩容，轻松应对流量暴涨

    6.标准 SSO 登录主流程（OIDC 授权码模式）¶
    1. 微服务应用跳转 SSO 统一登录页
    2. 用户输入账号密码完成认证
    3. SSO 生成授权码回调业务应用
    4. 业务应用拿授权码换取：Access Token + Refresh Token + ID Token业务应用拿授权码换取：Access Token + Refresh Token + ID Token
    5. 业务应用携带 Token 请求网关
    6. 网关统一校验 Token 合法性、有效期、权限范围
    7. 校验通过放行至对应微服务
    8. Token 过期自动使用 RefreshToken 无感续登
    9. 退出登录清除本地 Token+SSO 注销全局会话
    7.微服务接入规范（10 + 服务统一标准）¶
    1. 所有微服务统一对接网关，不直连 SSO 服务
    2. 网关统一完成 Token 解析、身份注入、基础鉴权
    3. 微服务只做业务细粒度权限校验
    4. 统一用户上下文：Header 注入 X-User-Id、X-Roles、X-Tenant-Id
    5. 接入只需配置 ClientID、秘钥、跳转域名，零侵入改造
    8.安全架构设计¶
    1. 传输安全：全站 HTTPS，接口签名
    2. 令牌安全：AccessToken 不可明文存储，移动端强制 PKCE
    3. 退出机制：全局注销、单端注销、批量强制下线
    4. 防刷：验证码限流、设备绑定、异地登录提醒
    5. 秘钥管理：非对称 RSA/ECC 签名验签，密钥定期轮换
    6. 数据安全：用户密码 BCrypt 加密存储，禁止明文
    9.容量预估（日活 100W+）¶
    • 日均登录请求：300W~500W
    • 高峰并发 QPS：预估 8000~15000
    • Redis 部署：3 主 3 从哨兵集群，足够承载会话 + 令牌
    • Auth 服务集群：8~16 节点无状态集群轻松承载
    • 数据库：2 主 4 从 MySQL 集群满足账号与权限读写
    10.容灾与高可用方案¶
    1. SSO 服务多机房部署，异地多活
    2. Redis 集群故障自动切换
    3. 登录降级：核心登录可用，非核心登录渠道临时关闭
    4. 令牌缓存雪崩防护：过期时间随机打散
    5. 定时全量会话巡检、无效令牌清理
    11.可扩展能力¶
    1. 快速接入新增微服务，分钟级上线
    2. 可扩展企业组织架构、分级管理员
    3. 可对接 LDAP、企业微信、钉钉统一组织登录
    4. 支持私有化部署 + 云端 SaaS 双模式
    5. 支持对接 IoT 设备、后端服务 API 密钥授权
    12.技术栈推荐¶
    • 后端：Go + Python 微服务体系
    • Go 负责：核心认证鉴权、网关、令牌服务、高并发流量入口、风控限流（高性能、低延迟、扛高 QPS）
    • Python 负责：用户管理后台、权限配置管理、审计日志分析、运营配置、第三方登录对接、数据统计（业务逻辑快迭代、后台管理友好）
    • 网关：APISIX / gRPC
    • 认证协议：Security OAuth2 + OIDC
    • 缓存：Redis Cluster
    • 数据库：MySQL + Redis + MongoDB
    • 监控：SkyWalking + Prometheus + ELK
    • 部署：K8s 容器化编排
