    1.git多项ssh配置¶

    1.1.生成密钥文件¶

    ssh-keygen -t rsa -f ~/.ssh/id_rsa_xxx -b 4096 -C "xxx@example.com"

    1.2.添加到ssh-agent¶

    ssh-add ~/.ssh/id_rsa_xxx

# 查看
ssh-add -l

    1.3.配置多项ssh¶

    vi ~/.ssh/config

    在配置文件中写入:

    # github
Host github.com
HostName github.com
IgnoreUnknown UseKeychain
AddKeysToAgent yes
UseKeychain yes
User aaa@outlook.com
PreferredAuthentications publickey
IdentityFile ~/.ssh/id_rsa

# bitbucket
Host bitbucket.org
HostName bitbucket.org
IgnoreUnknown UseKeychain
AddKeysToAgent yes
UseKeychain yes
User xxx@example.com
PreferredAuthentications publickey
IdentityFile ~/.ssh/id_rsa_xxx

    2.git commit规范¶

    2.1.规范简介¶

    格式:

    <type>([scope]): <subject>
// 空一行
[body]
// 空一行
[footer]

    其中字段含义:

    • type 包含以下几种：
    • feat: feature，产品新功能，通常是能够让用户觉察到的变化，小到文案或样式修改。
    • fix: bug fix，修复 bug。
    • docs: documentation，更新文档或注释。
    • style: code formatting, missing semi colons, … 代码格式调整，对逻辑无影响：比如为按照 eslint 或团队风格修改代码格式。注意不是 UI 变更。
    • refactor: 重构、代码优化但不影响现有功能或添加功能。比如文件、变量重命名、代码抽象为函数，消除魔法数字等。
    • test: when adding missing tests 单测相关变更。
    • chore: 杂项, 其他无法归类的变更，比如代码合并。
    • perf: 性能提升变更。
    • ci: 持续集成脚本相关变更。
    • build: 代码构建相关变更：比如修复部署时的构建问题、构建脚本 webpack 或 gulp 相关变更。
    • temp: 临时代码：不计入 CHANGELOG，比如必须部署到某种环境才能测试的变更。如测试真机上 transparent title 启动参数是否设置成功。
    • scope: 可选。变更范围细粒度要合适，在一个项目中保持一致最好，比如页面名、模块名、或组件名。
    • subject: 此次提交目的的简短描述，仅一行，不超过 100 字符。提交信息要重点描述为何要改，内容反而不重要，其次如果是英语，则用命令式(决定了动词开头)、现在时、首字母不大写、行末无句号、冒号为英文半角，后面加空格。
    • body: 详细描述。
    • footer: 关闭 aone bug 请在此描述。 修复的 bug 必须要以 "Closes" 为前缀，后面接 aone issue id，这样可以自动将 bug 平台的问题关闭。比如：Closes #234

    参考: git-commit规范

    2.2.添加emoji¶

    在 type 前加 emoji 符号, 如: :rocket: feat(models): ...

    2.3.设置 git commit 模板¶

    • 建立模板文件: 在项目中建立 .git_template 文件
    # head: <type>(<scope>): <subject>
# feat/fix/docs/refactor/perf/test/style/revert/chore(scope/issue): changelog
:rocket: feat(models): description overview

# body: 72-character wrapped.
# Add/Modify/Details if necessary
details

# footer
# Fix issues if any
    • 设置模板
    git config commit.template .git_template           # 当前项目
git config --global commit.template ~/.gitmessage  # 全局设置
    • Commit message 的工具
    # Commitizen
npm install -g commitizen
# Change log 工具
npm install -g conventional-changelog

    3.免输入密码配置¶

    3.1.~/.netrc¶

    1.基本配置¶

    machine github.com
login your_username
password your_token_or_password

machine gitlab.com
login your_email@example.com
password your_password

# 最后（严格的文件权限）
chmod 600 ~/.netrc

    重要说明：

    • machine：Git 服务器域名（如 GitHub.com）
    • login：Git 用户名或邮箱
    • password：使用 GitHub Personal Access Token（推荐）或账户密码
    • 每部分之间需空一行分隔不同服务器的配置

    2.vscode 远程终端¶

    2.1.搭配vscode远程连接避免手动输入密码¶
    echo 'alias ssh="sshpass -f ~/.netrc ssh"' >> ~/.bashrc
source ~/.bashrc
    2.2.ssh config¶
    1. 使用 SSH 密钥认证（推荐）
    2. 在 SSH 配置文件中（如你已配置），指定 IdentityFile，并确保私钥已添加到 SSH Agent。
    3. 配置 AddKeysToAgent yes 和 UseKeychain yes（Mac 推荐），这样密钥会自动加载，无需每次输入密码。
    4. 如果密钥有密码，可用 macOS 钥匙串自动解锁。
    5. 确保 SSH 密钥已生成并添加到 SSH Agent：
      • 运行 ssh-add ~/.ssh/id_rsa（或你的密钥路径）。
      • macOS 推荐在钥匙串中保存密钥密码。
    6. 启用 SSH Agent 转发
    7. 设置 ForwardAgent yes，并在 VS Code Remote-SSH 设置中启用 agent 转发，方便多跳连接。
    8. 使用本地服务器模式
    9. VS Code Remote-SSH 支持本地服务器模式，减少密码输入次数。设置 remote.SSH.useLocalServer 为 true。
    10. 配置 SSH_ASKPASS
    11. 如果有外部认证设备（如 YubiKey），可通过 remote.SSH.externalSSH_ASKPASS 设置让 VS Code 不覆盖你的认证应用。

    3.配置 Git 使用 netrc¶

    # 禁用所有凭据助手
git config --global --unset credential.helper
git config --global credential.helper ""

# 清除所有缓存凭据
git credential-cache exit

# 或完全禁用缓存
git config --global --unset credential.helper

    4.将环境变量添加到 shell 配置文件¶

    echo 'export GIT_CONFIG_COUNT=1' >> ~/.bashrc
echo 'export GIT_CONFIG_KEY_0=credential.helper' >> ~/.bashrc
echo 'export GIT_CONFIG_VALUE_0=""' >> ~/.bashrc
source ~/.bashrc

    5.测试配置¶

    # 测试 GitHub 连接
git ls-remote https://github.com/user/repo.git

# 启用调试模式（查看详细过程）
GIT_TRACE=1 GIT_CURL_VERBOSE=1 git push

    4.多server配置进阶¶

    📋 方案对比¶

    方案 核心原理 适用场景 优点 缺点
    SSH 配置 创建不同的 SSH 密钥对，通过 ~/.ssh/config 指定不同主机使用不同密钥 绝大多数情况，特别是需要 SSH 认证的场景 配置一次，永久生效；安全灵活 需要生成和管理多个密钥对
    条件式 Git 配置 在 Git 配置中根据仓库路径、URL 或目录设置不同的用户名和邮箱 主要解决不同仓库使用不同 user.name 和 user.email 能精细控制每个仓库的提交者信息 不直接解决 SSH 密钥切换
    Git Credential Manager 使用 Git 凭据管理器存储多套凭据 主要适用于 HTTP/HTTPS 协议 自动匹配凭据，无需手动切换 Linux 原生支持较弱

    推荐使用 SSH 配置方案，它最可靠且适用于所有 Git 操作。

    🔐 方法一：SSH 配置（推荐）¶

    这是最主流和可靠的方式，通过为不同的 Git 服务配置不同的 SSH 密钥来实现。

    步骤 1：生成多对 SSH 密钥¶

    # 为 GitHub 生成密钥（默认用 GitHub 邮箱）
ssh-keygen -t ed25519 -C "your_github_email@example.com" -f ~/.ssh/id_ed25519_github

# 为 GitLab 生成密钥（默认用 GitLab 邮箱）
ssh-keygen -t rsa -b 4096 -C "your_gitlab_email@example.com" -f ~/.ssh/id_rsa_gitlab

# 为公司 GitLab 生成密钥（如有需要）
ssh-keygen -t ed25519 -C "your_company_email@example.com" -f ~/.ssh/id_ed25519_company

    步骤 2：将公钥添加到对应的 Git 服务¶

    # 查看并复制公钥内容
cat ~/.ssh/id_ed25519_github.pub
cat ~/.ssh/id_rsa_gitlab.pub

# 分别添加到：
# GitHub: https://github.com/settings/keys
# GitLab: https://gitlab.com/-/profile/keys

    步骤 3：配置 SSH Config 文件¶

    编辑或创建 ~/.ssh/config 文件：

    # ~/.ssh/config

# GitHub 账户配置（个人账户）
Host github.com
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519_github
    IdentitiesOnly yes
    # 可选：指定特定 GitHub 用户
    # User git-personal

# GitLab 个人账户（gitlab.com）
Host gitlab.com
    HostName gitlab.com
    User git
    IdentityFile ~/.ssh/id_rsa_gitlab
    IdentitiesOnly yes

# 公司内部 GitLab 服务器（示例）
Host gitlab.company.com
    HostName gitlab.company.com
    User git
    IdentityFile ~/.ssh/id_ed25519_company
    IdentitiesOnly yes
    Port 22
    # 如果公司使用非标准端口
    # Port 2222

# 为同一个服务的不同账户创建别名（如多个 GitHub 账户）
Host github-work  # 工作账户别名
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519_github_work
    IdentitiesOnly yes

Host github-personal  # 个人账户别名
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519_github_personal
    IdentitiesOnly yes

    步骤 4：测试 SSH 连接¶

    # 测试 GitHub 连接
ssh -T git@github.com
# 预期输出：Hi username! You've successfully authenticated...

# 测试 GitLab 连接  
ssh -T git@gitlab.com
# 预期输出：Welcome to GitLab, @username!

# 测试别名连接（如果配置了多个账户）
ssh -T git@github-work

    步骤 5：克隆和配置仓库¶

    现在你可以使用不同的 SSH 配置来克隆仓库：

    # 克隆 GitHub 仓库（会自动使用对应的 SSH 配置）
git clone git@github.com:username/repo.git

# 克隆 GitLab 仓库
git clone git@gitlab.com:username/repo.git

# 如果配置了别名，使用别名克隆
git clone git@github-work:company/project.git

    ⚙️ 方法二：条件式 Git 配置¶

    此方法主要解决不同仓库使用不同的 user.name 和 user.email 的问题。

    全局 Git 配置¶

    # 设置全局默认配置（作为回退选项）
git config --global user.name "Your Default Name"
git config --global user.email "default@example.com"

# 设置全局的 SSH 命令（可选）
git config --global core.sshCommand "ssh -F ~/.ssh/config"

    基于目录的条件配置¶

    编辑 ~/.gitconfig 文件，添加条件配置：

    # ~/.gitconfig
[user]
    name = Your Default Name
    email = default@example.com

# 对特定目录下的所有仓库使用特定配置
[includeIf "gitdir:~/work/"]
    path = ~/.gitconfig-work

[includeIf "gitdir:~/personal/"]
    path = ~/.gitconfig-personal

# 对特定仓库使用特定配置（使用 glob 模式）
[includeIf "gitdir:~/projects/github/*/"]
    path = ~/.gitconfig-github

[includeIf "gitdir:~/projects/gitlab/*/"]
    path = ~/.gitconfig-gitlab

    然后创建对应的配置文件：

    # ~/.gitconfig-work (工作配置)
[user]
    name = Your Work Name
    email = work@company.com
[core]
    sshCommand = "ssh -F ~/.ssh/config -i ~/.ssh/id_ed25519_work"

# ~/.gitconfig-personal (个人配置)
[user]
    name = Your Personal Name
    email = personal@example.com
[core]
    sshCommand = "ssh -F ~/.ssh/config -i ~/.ssh/id_ed25519_personal"

# ~/.gitconfig-github
[user]
    name = Your GitHub Name
    email = github@example.com

# ~/.gitconfig-gitlab
[user]
    name = Your GitLab Name
    email = gitlab@example.com

    基于 URL 的条件配置（Git 2.13+）¶

    # ~/.gitconfig
[user]
    name = Default Name
    email = default@example.com

# 对 GitHub 仓库使用特定配置
[url "git@github.com:"]
    insteadOf = gh:
[url "gh:"]
    pushInsteadOf = git@github.com:
[includeIf "hasconfig:remote.*.url:gh:**"]
    path = ~/.gitconfig-github

# 对 GitLab 仓库使用特定配置
[url "git@gitlab.com:"]
    insteadOf = gl:
[includeIf "hasconfig:remote.*.url:gl:**"]
    path = ~/.gitconfig-gitlab

    使用 URL 别名克隆：

    # 使用别名克隆，会自动应用对应配置
git clone gh:username/repo.git    # 对应 GitHub
git clone gl:username/repo.git    # 对应 GitLab

    🔄 方法三：自动化脚本¶

    创建自动化脚本简化多仓库操作：

    #!/bin/bash
# ~/bin/git-clone-multi
# 根据 URL 自动选择配置克隆仓库

clone_repo() {
    local url="$1"
    local target_dir="$2"

    case "$url" in
        *github.com*)
            echo "Cloning GitHub repository..."
            GIT_SSH_COMMAND="ssh -i ~/.ssh/id_ed25519_github" \
            git -c user.name="GitHub Name" \
                -c user.email="github@example.com" \
                clone "$url" "$target_dir"
            ;;
        *gitlab.com*)
            echo "Cloning GitLab repository..."
            GIT_SSH_COMMAND="ssh -i ~/.ssh/id_rsa_gitlab" \
            git -c user.name="GitLab Name" \
                -c user.email="gitlab@example.com" \
                clone "$url" "$target_dir"
            ;;
        *)
            echo "Cloning with default configuration..."
            git clone "$url" "$target_dir"
            ;;
    esac
}

# 使用示例
# clone_repo "git@github.com:user/repo.git" "./repo"

    5.常见问题¶

    5.1.中文显示问题¶

    核心配置：关闭 core.quotepath

    core.quotepath 是 Git 控制文件名编码显示的核心配置，默认值为 true（启用转义）。将其设置为 false 即可让中文直接显示。

    方法 1：全局设置（推荐，影响所有仓库）¶

    git config --global core.quotepath false

    方法 2：仅当前仓库设置（仅影响当前项目）¶

    git config core.quotepath false
