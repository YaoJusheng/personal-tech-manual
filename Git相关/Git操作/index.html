    GitHub文档

    1、安装¶

    sudo apt-get install git

    2、新建仓库¶

    #1.创建本地仓库目录:
mkdir dir
#2.进入目录:
cd dir
#3.初始化为空仓库:
git init
#4.查看空仓库:
ls -ah

    3、设定user和email¶

    git config --global user.name "yaojusheng"
git config --global user.email "jusheng_yao@outlook.com"

    4、版本操作¶

    4.1.工作区新建文件¶

    vim test.txt

    4.2.把文件加到暂存区（批量添加：git add -A）¶

    git add test.txt

    [!NOTE] 注意 在Git中，可以通过以下命令来显示当前你的Git中采取哪种对待换行符的方式： $ git config core.autocrlf

    此命令会有三个输出，"true"，"false"或者"input"：

    1. 为 true 时，Git会将你add的所有文件视为文本文件，将结尾的CRLF转换为LF，checkout时会再将LF转回CRLF。
    2. 为 false 时，line endings不做任何改变，文本文件保持其原来的样子。
    3. 为 input 时，add时Git会把CRLF转换为LF，而check时仍旧为LF，所以Windows操作系统不建议设置此值。

    Windows中，将core.autocrlf设为false

    4.3.提交到仓库¶

    git commit -m "添加操作的注释说明"

    4.4.查看当前仓库状态¶

    git status

    4.5.查看修改¶

    git diff test.txt

    4.6.查看日志，获取提交记录¶

    git log
git log --oneline
git log --pretty=format:"%an %ae %ad %cn %ce %cd %cr %s" --graph
git reflog

    4.7.重命名分支¶

    git branch -m old new
# 删除远程分支
git push --delete origin oldName
# 上传新命名的本地分支
git push origin newName
# 关联
git branch --set-upstream-to origin/newName

    4.8.版本回退¶

    git reset --hard HEAD^    --- 回退到上一个版本
git reset --hard HEAD^^   --- 回退到上上个版本
# ...
git reset --hard HEAD~100   --- 回退到之前100的版本
git reset --hard 版本号     --- 回退到指定版本
# revert
git revert HEAD~1
git revert -n  commit版本号
# 回退连续几个commit
git revert 2ad42a7f...26a60ff1 -m 1

    4.9.删除仓库的文件¶

    git rm test.txt
git commit -m "del"

    4.10.还原删除¶

    git checkout 其实是用版本库里的版本替换工作区的版本，无论工作区是修改还是删除，都可以“一键还原”

    4.11.暂存当前修改¶

    git stash会把所有未提交的修改（包括暂存的和非暂存的）都保存起来，用于后续恢复当前工作目录。

    # 1. stash 保存当前修改
master > git stash [save "message"]
master > git checkout develop

# 2. 重新应用缓存的stash
develop > git stash pop
# 注: 这个指令将缓存堆栈中的第一个stash删除，并将对应修改应用到当前的工作目录下
# 可以使用git stash apply命令，将缓存堆栈中的stash多次应用到工作目录中，但并不删除stash拷贝
develop > git stash apply

# 3.查看现有stash
git stash list

# 4.移除stash
git stash drop
git stash clear

# 5.查看指定stash的dif
git stash show

# 6.从stash创建分支
git stash branch test-dev

    参考: stash用法

    5.远程仓库GitHub¶

    5.1、注册¶

    5.2、连接远程¶

    git remote add origin https://github.com/YaoJusheng/mygit.git
# 或者
git remote add origin git@github.com:YaoJusheng/xxx.git

# 然后
git pull origin main
git branch --set-upstream-to=origin/main main

# 查看本地仓库与远程仓库的关联详情
git remote -v

# 取消关联：git remote rm origin

# 更改关联方式
git config remote.origin.url git@xxx

    5.3、创建SSH Key，使用ssh进行上传提交更新时需要公钥，生成获取pubkey¶

    ssh-keygen -t rsa -b 4096 -C "Jusheng_Yao@outlook.com"

# 生成的pubkey默认设置：C:/User/.ssh
# 打开id_rsa.pub,复制内容在github中settings中SSH and GPG key中添加SSH key
# 添加内容为复制的内容

    5.4、从远程仓库同步到本地仓库¶

    git pull -u origin master

    1.情况一：本地目录为空（或允许覆盖）¶

    cd /path/to/your/local-folder

# 1. 初始化 Git（如果目录还没初始化）
git init

# 2. 添加远程仓库
git remote add origin https://github.com/username/repo.git

# 3. 拉取远程代码并建立跟踪（假设远程默认分支是 main）
git pull origin main

# 4. 设置本地 main 分支跟踪远程
git branch --set-upstream-to=origin/main main

    2.情况二：本地已有文件/内容¶

    如果本地已有文件且未提交，直接 pull 会报错（refusing to merge unrelated histories），需要允许合并不相关历史：

    cd /path/to/your/local-folder
git init
git remote add origin https://github.com/username/repo.git

# 先提交本地现有内容，避免冲突丢失
git add .
git commit -m "init: local files"

# 拉取远程并强制允许不相关历史合并
git pull origin main --allow-unrelated-histories

# 设置跟踪
git branch --set-upstream-to=origin/main main

    合并时如果出现冲突，手动编辑冲突文件后：

    git add .
git commit -m "merge: sync with remote"

    3.情况三：本地目录已初始化过 Git¶

    5.7、设置多个远程仓库¶

    可参考：多server配置

    1.查看当前远程仓库¶

    git remote -v

    2.添加多个远程仓库¶

    你可以通过两种方式设置多个远程仓库：

    方法一：添加多个不同名称的远程仓库¶
    # 添加第一个远程仓库（通常命名为 origin）
git remote add origin https://github.com/yourname/repo.git

# 添加第二个远程仓库（可以命名为 gitlab、gitee 等）
git remote add gitlab https://gitlab.com/yourname/repo.git

# 添加第三个远程仓库
git remote add gitee https://gitee.com/yourname/repo.git

    推送时需要指定远程仓库名称：

    git push origin main    # 推送到 GitHub
git push gitlab main    # 推送到 GitLab
git push gitee main     # 推送到 Gitee
    方法二：为同一个远程名称添加多个 URL¶

    这种方式可以让你用一个命令同时推送到多个仓库：

    # 先添加一个主要远程仓库
git remote add origin https://github.com/yourname/repo.git

# 再添加其他仓库的 URL 到同一个远程名称
git remote set-url --add origin https://gitlab.com/yourname/repo.git
git remote set-url --add origin https://gitee.com/yourname/repo.git

    现在，推送时只需一个命令就能同步到所有仓库：

    git push origin main

    3.移除远程仓库¶

    如果需要移除某个远程仓库或 URL：

    # 移除整个远程仓库（如 gitee）
git remote remove gitee

# 移除某个远程仓库的特定 URL
git remote set-url --delete origin https://gitlab.com/yourname/repo.git

    7.子模组¶

    Git子模块允许你将一个Git仓库作为另一个Git仓库的子目录。它能让你将另一个仓库克隆到自己的项目中，同时还保持独立的提交。

    .gitmodules 文件结构:

    [submodule "lib/library"]
    path = lib/library
    url = https://github.com/user/library.git
    branch = main

[submodule "docs/theme"]
    path = docs/theme
    url = https://github.com/user/theme.git

    7.1.添加子模块¶

    repo根目录下.gitmodules文件中会定义子模块相关内容, 使用命令git submodule add添加:

    # 添加子模块
git submodule add <repository_url> <path>

# 示例：将bootstrap仓库作为子模块添加到项目的public/vendor/bootstrap目录
git submodule add https://github.com/twbs/bootstrap.git public/vendor/bootstrap
# or
git submodule add git@github.com:YaoJusheng/xxx.git <subname>
# 之后提交

# 删除
git submodule deinit <subname>
git rm <subname>

    7.2.克隆包含子模块的项目¶

    # 克隆主项目
git clone <repository_url>

# 初始化子模块（将子模块的配置文件从.gitmodules复制到.git/config）
git submodule init

# 更新子模块（克隆子模块仓库并检出指定的提交）
git submodule update

# 或者组合使用
git submodule update --init

# 或者递归克隆（如果子模块还有子模块）
git submodule update --init --recursive

# 克隆主项目时同时克隆子模块（一步到位）
git clone --recurse-submodules <repository_url>

    7.3.更新子模块¶

    # 更新子模块到最新提交（在主项目中运行）
git submodule update --remote

# 更新特定子模块
git submodule update --remote <path>

# 更新子模块到最新提交并提交主项目
git submodule update --remote
git add <submodule_directory>
git commit -m "Update submodule to latest version"

# 更新所有子模块（包括嵌套子模块）
git submodule update --init --recursive
git submodule update --remote --recursive

    7.4.删除子模块¶

    # 1. 删除子模块目录
git rm --cached <submodule_path>
rm -rf <submodule_path>

# 2. 删除.gitmodules中的子模块部分（或者整个文件如果只有一个子模块）
vim .gitmodules

# 3. 删除.git/config中的子模块部分（可选）
vim .git/config

# 4. 提交更改
git add .gitmodules
git commit -m "Remove submodule"

# 5. 删除子模块目录（如果之前没有删除）
rm -rf .git/modules/<submodule_name>

    7.5.查看子模块状态¶

    # 查看子模块状态
git submodule status

# 查看更详细的子模块状态
git submodule summary

# 查看子模块的远程仓库地址
git submodule foreach 'git remote -v'

    7.6.批量操作子模块¶

    # 对所有子模块执行同一个命令
git submodule foreach 'git checkout master'

# 递归地对所有子模块（包括嵌套子模块）执行命令
git submodule foreach --recursive 'git checkout master'

# 带条件的操作
git submodule foreach 'if [ -f package.json ]; then npm install; fi'

    7.7.修改子模块URL¶

    # 方法1：编辑.gitmodules文件
vim .gitmodules
# 修改 [submodule "path"] 下的 url
git submodule sync  # 同步到.git/config

# 方法2：使用git命令
git config --file .gitmodules submodule.lib/library.url https://new.url/repo.git
git submodule sync

    7.8.子模块分支管理¶

    # 设置子模块跟踪特定分支
git config -f .gitmodules submodule.lib/library.branch develop

# 让子模块检出分支而不是分离头指针
git submodule update --remote --checkout

# 在子模块中创建分支
cd lib/library
git checkout -b feature-branch
git push -u origin feature-branch

    9.问题汇总¶

    0.定位问题，查看日志¶

    GIT_TRACE=1 GIT_CURL_VERBOSE=1 git push  # 假设查看push操作问题

    1.git push 失败的问题¶

    刷新DNS缓存： ipconfig /flushdns

    2.remote: fatal: early EOF + index-pack failed¶

    处理：
    修改 GIT 的本地 config 文件，在 core 中加入：
    compression = -1

    3.error: RPC failed; result=55, HTTP code = 200¶

    原因：发生错误是同步数据过大导致
处理：
    git config http.postBuffer 524288000
    或在config文件中修改缓存大小

    4.error: RPC failed; curl 56 OpenSSL SSL_read: SSL_ERROR_SYSCALL, errno 10053¶

    原因：SSL错误，因为服务器的SSL证书没有经过第三方机构的签署，所以才报错
处理：
    git config http.sslVerify "false"
    git config --global http.sslVerify false

    5.error：fatal: The remote end hung up unexpectedly fatal: early EOF fatal: index-pack failed¶

    处理：
    git config --global core.compression 0

    6.fatal: fetch-pack: invalid index-pack output¶

    原因：因为文件太大导致报错

    7.对于git checkout HEAD -- {file}这种撤销操作后的恢复¶

    如果之前 git add 过（即改动进过暂存区），可用如下操作：

    • 定位未被引用的 Blob 对象:
    git fsck --lost-found
# 作用：扫描对象数据库，列出未被任何提交/分支引用的对象

    Git 会把“悬空 blob”（dangling blob）对象写在 .git/lost-found/other/ 里

    • 找对应文件内容：
    cd .git/lost-found/other
file *          # 看哪个 blob 像文本
grep -l "你记得的一段代码" *

    • 验证 Blob 内容, 并恢复: git show <blob-sha> 或 cp 恢复到原路径。

    • 其他:

    • 方案 A：恢复工作区: git checkout <blob-sha> -- pkg/ui/dashboard.go

    • 方案 B：直接恢复暂存区: git restore --source=<blob-sha> --staged pkg/ui/dashboard.go

    • 复杂场景处理： 若多次 add 和 checkout 交错，需结合 git reflog 排查操作时间线：

    git reflog show --all | grep 'dashboard.go'  # 查找相关记录

    Example:

    # 假设 git checkout HEAD -- pkg/ui/dashboard.go，撤销了修改操作
❯ grep -l "UnifiedRegionDrawer" .git/lost-found/other/*
.git/lost-found/other/6f3b18b3d9fc8911728dec5b4016bf05b3d1241e

❯ git show 6f3b18b3d9fc8911728dec5b4016bf05b3d1241e > pkg/ui/dashboard.go
