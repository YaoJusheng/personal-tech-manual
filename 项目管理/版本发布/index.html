    一、概述¶

    1.1 目的¶

    本文档旨在为软件研发团队提供一套标准化、可落地的版本发布管理规范，涵盖版本号管理、迭代发布流程、环境管理、发布检查清单、回滚机制及发布后监控等全生命周期内容。通过规范化发布流程，降低发布风险，提升交付效率与系统稳定性。

    1.2 适用范围¶

    适用于公司所有软件产品（含 Web 应用、移动应用、微服务、基础平台）的版本发布工作，团队可依据项目规模和技术特点进行适当裁剪。

    1.3 核心原则¶

    • 可重复性：发布过程标准化、脚本化，任何人按流程均可执行。
    • 可追溯性：每次发布留有完整记录，代码变更与版本一一对应。
    • 可回滚性：任何发布均应具备快速回滚能力，且经过演练验证。
    • 最小化影响：通过灰度、分批等策略，将发布对用户的影响降到最低。
    • 自动化优先：重复性操作通过 CI/CD 流水线自动完成，减少人为失误。

    二、版本号规范¶

    2.1 语义化版本（SemVer）¶

    遵循 语义化版本 2.0.0 规范，格式为：

    MAJOR.MINOR.PATCH[-PRE_RELEASE][+BUILD_METADATA]
    字段 说明 示例
    MAJOR 重大版本，包含不兼容的 API 变更 2.0.0
    MINOR 次版本，向下兼容的功能新增 1.3.0
    PATCH 修订号，向下兼容的 Bug 修复 1.3.5
    PRE_RELEASE 预发布标识（alpha / beta / rc） 1.4.0-rc.1
    BUILD_METADATA 构建元数据，通常为 CI 流水线号或 Git commit SHA（仅供标识，不参与版本排序） 1.3.5+build.202

    2.2 版本类型与发布场景¶

    版本类型 触发场景 示例
    Major 版本 架构重构、重大功能变更、API 不兼容 1.x.x → 2.0.0
    Minor 版本 迭代新功能交付、小版本特性上线 1.2.x → 1.3.0
    Patch 版本 Bug 修复、安全补丁、热修复 1.3.4 → 1.3.5
    Alpha 版本 内部测试版，功能不完整，仅供开发自测 1.4.0-alpha.1
    Beta 版本 外部测试版，功能基本完整，供测试团队 1.4.0-beta.2
    RC 版本 发布候选版，功能冻结，回归测试阶段 1.4.0-rc.1
    Hotfix 版本 生产环境紧急修复，基于当前线上版本 1.3.4 → 1.3.5

    2.3 版本号递增规则¶

    初始开发阶段：0.x.x（0.1.0 起步）
首次稳定发布：1.0.0
新增功能（兼容）：MINOR +1，PATCH 归零 → 1.3.0 → 1.4.0
问题修复（兼容）：PATCH +1 → 1.3.4 → 1.3.5
不兼容变更：MAJOR +1，MINOR/PATCH 归零 → 1.x.x → 2.0.0

    ⚠️ 注意：预发布版本（alpha/beta/rc）不应用于生产环境；同一迭代发布前应完成 RC 验证。

    三、分支策略与代码管理¶

    3.1 推荐分支模型（Git Flow 简化版）¶

    main / master      ← 生产环境代码，只接受 merge，严禁直接 push
  │
  ├─ release/x.x.x  ← 发布分支，功能冻结，仅做修复
  │     └─ hotfix/xxx ← 热修复分支，从 main 拉出，修完合回 main + develop
  │
develop            ← 集成分支，所有 feature 合入此处
  │
  ├─ feature/xxx    ← 功能开发分支，从 develop 拉出
  └─ bugfix/xxx     ← 问题修复分支，从 develop 拉出

    3.2 分支命名规范¶

    分支类型 命名格式 示例
    功能分支 feature/<issue-id>-<brief-desc> feature/1023-user-login
    修复分支 bugfix/<issue-id>-<brief-desc> bugfix/1058-login-crash
    发布分支 release/<version> release/1.4.0
    热修复 hotfix/<version>-<brief-desc> hotfix/1.3.5-payment-timeout
    实验性分支 experiment/<desc> experiment/new-cache-layer

    3.3 代码合并要求¶

    • 所有代码合并通过 Pull Request（PR）/ Merge Request（MR） 进行，不允许直接向 main/develop 推送。
    • PR 必须满足：
    • [ ] 至少 1 位 同伴进行 Code Review 并批准（核心模块 2 位）
    • [ ] CI 流水线全部通过（构建 + 单测 + 代码扫描）
    • [ ] 无未解决的 Review 评论
    • [ ] 关联对应的需求/缺陷 Issue
    • 合并策略：推荐使用 Squash Merge 保持 commit 历史整洁；release 分支使用 Merge Commit 保留完整记录。

    四、迭代发布流程（全生命周期）¶

    4.1 迭代发布总览¶

    以 双周 Sprint 为例，完整迭代发布流程如下：

    Sprint 开始
    │
    ├─ [D1-D8]  功能开发 & 单测 & Code Review
    │
    ├─ [D9]     代码冻结 / 功能切割 → 拉取 release/x.x.x 分支
    │
    ├─ [D9-D10] 集成测试 + UAT 验收（测试环境）
    │
    ├─ [D11]    发布前评审 & 发布申请审批
    │
    ├─ [D12]    灰度发布（预发布/Staging 环境验证）
    │
    ├─ [D13]    全量生产发布 & 观察期（2-4h）
    │

    4.7.2 发布策略选择¶

    根据风险等级和业务特点选择合适的发布策略：

    ① 蓝绿部署（Blue-Green Deployment）:

    流量
  │
  ▼
负载均衡
  │
  ├─── 蓝色环境（当前生产 v1.3.5）  ←── 切流前全量承接
  │
  └─── 绿色环境（新版本 v1.4.0）    ←── 部署完成后，切流验证

切流：负载均衡 100% 指向绿色环境
回滚：负载均衡切回蓝色环境（秒级）
    • 适用：无状态服务、快速切换需求、数据库 Schema 兼容的场景
    • 优点：发布/回滚速度快，无停机
    • 注意：需双倍资源，数据库需向前兼容

    ② 金丝雀发布（Canary Release）:

    全量流量 100%
    │
    ├── 95% → 旧版本实例（v1.3.5）
    │
    └──  5% → 新版本实例（v1.4.0）← 金丝雀节点
                   │
            观察 15-30 分钟
                   │
            ┌──────┴──────┐
           正常           异常
            │               │
        逐步扩容         立即回滚
      10% → 50% → 100%
    • 适用：新功能上线、高风险变更、大流量应用
    • 优点：风险可控，影响面小
    • 工具：Nginx（upstream 权重配置）、Kubernetes（Deployment 灰度）、Istio（流量管理）

    ③ 滚动发布（Rolling Update）:

    实例 1: [v1.3.5] → 停止 → 部署 v1.4.0 → 健康检查 → [v1.4.0] ✓
实例 2: 等待实例1就绪 → [v1.3.5] → ... → [v1.4.0] ✓
实例 3: 等待实例2就绪 → [v1.3.5] → ... → [v1.4.0] ✓
    • 适用：Kubernetes 标准部署场景，资源受限时
    • 配置示例（Kubernetes）：
    strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 1        # 最多允许超出期望副本数 1 个
    maxUnavailable: 0  # 发布期间不允许不可用

    ④ 功能开关（Feature Toggle）:

    代码中预置开关：
if (featureFlag.isEnabled("new-payment-flow", userId)) {
    // 新支付流程
} else {
    // 旧支付流程
}
    • 适用：功能逐步灰度、A/B 测试、紧急降级
    • 工具：LaunchDarkly、Unleash、自研配置中心
    • 优点：发布与功能上线解耦，运行时控制，回滚无需重新部署

    4.7.3 生产发布执行步骤¶

    1. 执行数据库迁移（如有）
    # Flyway
flyway migrate -url=jdbc:mysql://prod-db:3306/app -locations=db/migration
# Liquibase
liquibase update --changelog-file=db/changelog.xml

    ⚠️ 数据库迁移应向前兼容，支持新旧代码同时运行（兼容期间不删除旧列/表）。

    1. 触发生产部署
    # 方式一：手动触发 CI/CD
./deploy.sh --env=prod --version=1.4.0 --strategy=canary --canary-weight=5

# 方式二：在 CD 平台（如 ArgoCD）手动审批并同步
argocd app sync myapp --revision v1.4.0
    1. 健康检查与验证
    2. 观察部署过程中的健康检查（readinessProbe / livenessProbe）
    3. 确认新版本 Pod/实例 Ready
    kubectl rollout status deployment/myapp -n production
    1. 发布后冒烟验证
    2. 由测试人员/发布负责人在生产环境执行核心冒烟用例

    3. 观察监控大盘 15–30 分钟

    4. 逐步扩量（金丝雀场景）

    观察期 → 5% → 10% → 25% → 50% → 100%
每个阶段观察 10–15 分钟，指标正常后继续扩量

    4.7.4 发布观察期监控指标¶

    在发布后的 2–4 小时观察期内，重点关注：

    监控维度 关键指标 告警阈值（示例）
    服务可用性 成功率（HTTP 2xx/3xx 比例） < 99.9% 告警
    错误率 5xx 错误率 > 0.1% 告警
    响应时间 P95 / P99 延迟 较发布前上升 > 20% 告警
    吞吐量 QPS / TPS 异常波动 ± 30%
    资源使用率 CPU、内存、连接池使用率 CPU > 80%、内存 > 85%
    业务指标 核心业务转化率、订单量等 较基线下降 > 5%
    日志错误 ERROR / CRITICAL 级别日志量 明显突增告警

    工具推荐：Prometheus + Grafana、Datadog、SkyWalking、阿里云 ARMS

    五、热修复（Hotfix）发布流程¶

    5.1 适用场景¶

    • 生产环境出现 P0/P1 级别严重 Bug，影响核心业务正常运行
    • 安全漏洞需紧急修复
    • 数据异常需立即止损

    5.2 热修复发布流程¶

    [P0/P1 Bug 报告]
       │
[影响评估 & 决策：是否需要热修复？]
       │ 是
[从 main 分支拉取 hotfix 分支]
       │
[最小化修复 & 本地验证]
       │
[Code Review（快速通道，至少 1 人审查）]
       │
[部署至测试环境验证（15–30 分钟）]
       │
[生产发布（优先蓝绿/直接部署）]
       │
[验证修复效果 & 监控 30 分钟]
       │
[打 Tag + 合并回 main & develop]
       │
[事故复盘（24–48 小时内）]

    5.3 热修复操作步骤¶

    # 1. 从 main 拉取 hotfix 分支
git checkout main && git pull origin main
git checkout -b hotfix/1.3.5-payment-timeout

# 2. 修复代码，更新版本号（PATCH +1）
# ... 修复代码 ...
# 更新版本号 1.3.4 → 1.3.5

# 3. 提交，走快速 Code Review

# 4. 合并回 main
git checkout main
git merge --no-ff hotfix/1.3.5-payment-timeout

# 5. 打 Tag
git tag -a v1.3.5 -m "Hotfix v1.3.5: fix payment timeout"
git push origin main --tags

# 6. 同步到 develop
git checkout develop
git merge --no-ff hotfix/1.3.5-payment-timeout
git push origin develop

# 7. 删除 hotfix 分支
git branch -d hotfix/1.3.5-payment-timeout
git push origin --delete hotfix/1.3.5-payment-timeout

    5.4 热修复 SLA 要求¶

    事故级别 响应时间 修复上线时间
    P0 ≤ 15 分钟 ≤ 2 小时
    P1 ≤ 30 分钟 ≤ 4 小时
    P2 ≤ 2 小时 纳入下次迭代

    七、回滚机制¶

    7.1 回滚触发条件¶

    出现以下情况时，立即执行回滚：

    • 生产发布后，核心业务成功率 < 99%
    • 5xx 错误率 > 0.5%（持续 5 分钟以上）
    • 关键业务指标（如支付成功率、订单量）出现明显异常下跌
    • 系统出现 OOM、大量超时、服务不可用
    • 监控告警持续触发，且无法快速定位根因

    回滚决策人：发布负责人可直接决策执行回滚，无需等待审批。

    7.2 回滚方式¶

    ① 应用代码回滚:

    # Kubernetes 快速回滚（滚动发布场景）
kubectl rollout undo deployment/myapp -n production
kubectl rollout status deployment/myapp -n production

# 回滚到指定历史版本
kubectl rollout history deployment/myapp -n production
kubectl rollout undo deployment/myapp --to-revision=3 -n production

# 蓝绿部署回滚：切换负载均衡流量回蓝色环境
# 在 CD 平台或运维脚本中执行流量切换

    ② 数据库回滚:

    # Flyway 回滚（需提前准备 undo 脚本）
flyway undo -url=jdbc:mysql://prod-db:3306/app

# 手动恢复备份（极端情况）
mysqldump --restore backup_before_release.sql

    ⚠️ 数据库回滚风险高，优先采用向前修复（forward-fix）策略，即通过补丁脚本修正数据，避免直接回滚 Schema。

    ③ 配置回滚:

    # 配置中心（如 Nacos / Apollo）回滚到历史版本
# 在配置中心控制台找到上一个配置快照，点击回滚

# 或使用版本管理系统中的历史配置
git checkout v1.3.5 -- configs/prod/application.yml

    7.3 回滚 SLA¶

    发布策略 目标回滚时间
    蓝绿部署 ≤ 1 分钟
    Kubernetes 滚动发布 ≤ 3 分钟
    金丝雀发布 ≤ 5 分钟
    功能开关 ≤ 1 分钟（运行时关闭）

    7.4 回滚后处置¶

    1. 立即通知相关团队（研发、运维、PO、业务方）
    2. 保留现场：导出异常日志、APM 数据，禁止清理
    3. 根因分析：24 小时内完成初步定位
    4. 事后复盘：48 小时内召开复盘会议，输出复盘报告
    5. 修复验证：修复后走正常发布流程，禁止绕过测试直接发布

    八、环境管理规范¶

    8.1 标准环境划分¶

    环境 用途 数据来源 部署触发 访问权限
    开发（DEV） 开发自测、联调 Mock 数据 feature 分支合并自动触发 研发团队
    测试（TEST） 功能测试、集成测试 测试数据集 release 分支推送自动触发 研发 + 测试
    UAT 用户验收测试 脱敏生产数据 手动触发 PO + 业务方
    预发布（STAGING） 最终验证，与生产同构 脱敏生产数据 人工审批后触发 研发 + 测试 + 运维
    生产（PROD） 线上用户使用 真实数据 人工审批后触发 运维（研发只读）

    8.2 环境管理原则¶

    1. 环境隔离：各环境的网络、数据、配置完全隔离，禁止跨环境访问。
    2. 配置管理：使用配置中心（如 Nacos、Apollo）或 Kubernetes ConfigMap/Secret 管理各环境配置，禁止在代码中硬编码环境差异配置。
    3. 数据保护：生产数据不得流入测试/开发环境；UAT/Staging 使用经过脱敏处理的数据。
    4. 环境一致性：Staging 环境需与生产保持高度一致（镜像版本、配置模板、基础设施规格）。
    5. 环境版本标记：每个环境的当前部署版本应通过监控大盘或运维平台可视化查询。

    十一、发布度量与持续改进¶

    11.1 核心发布度量指标¶

    指标 说明 目标值
    部署频率 单位时间内的发布次数 ≥ 每两周一次
    变更前置时间（Lead Time） 代码提交到成功部署生产的时间 ≤ 3 天
    变更失败率 导致服务降级/回滚的发布比例 ≤ 5%
    平均恢复时间（MTTR） 生产故障从发现到恢复的平均时间 ≤ 1 小时
    发布成功率 成功完成且无需回滚的发布比例 ≥ 95%
    回滚时间 从触发回滚到完成回滚的时间 ≤ 5 分钟

    以上为 DORA（DevOps Research and Assessment）四项关键指标的工程实践参考。

    11.2 迭代发布复盘¶

    每次迭代发布结束后，在 Sprint 回顾会中包含发布质量回顾：

    1. 发布数据回顾：部署时长、测试通过率、遗留 Bug 数
    2. 问题识别：发布过程中遇到的阻塞、风险、流程缺陷
    3. 改进措施：针对问题提出 1–3 项可执行的改进行动项
    4. 行动跟踪：将改进措施记入下一 Sprint 的 Backlog
