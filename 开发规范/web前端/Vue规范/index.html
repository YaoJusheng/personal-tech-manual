    1.1 组件命名应该是多词¶

    细则映射示例¶

    // 文件: TodoItem.vue
<template>
  <div class="todo-item">{{ item.text }}</div>

    1.2 详细的prop定义¶

    在提交的代码中，prop 定义应始终尽可能详细，至少指定类型。

    详细的 prop 定义有两个好处：

    1. 它们记录了组件的 API，以便很容易看到组件的使用方式。
    2. 在开发过程中，如果组件提供了格式不正确的 props，Vue 会警告你，帮助你发现潜在的错误来源。
    // This is only OK when prototyping
const props = defineProps(['status'])
// Good
const props = defineProps({
  status: String
})

// Even better!
const props = defineProps({
  status: {
    type: String,
    required: true,

    validator: (value) => {
      return ['syncing', 'synced', 'version-conflict', 'error'].includes(
        value
      )
    }
  }
#### 细则映射示例（对应细则：1.1.4）

### 1.1.3 使用v-for时必使用key

```JavaScript
/* bad */
<ul>
  <li v-for="todo in todos">
    {{ todo.text }}
</ul>
/* Good */
<ul>
  <li
    v-for="todo in todos"
    :key="todo.id"
  >
    {{ todo.text }}
  </li>
</ul>

    细则映射示例¶

    • 推荐：使用稳定且唯一的 key，例如数据库 ID。
    <li v-for="todo in todos" :key="todo.id">{{ todo.text }}</li>
    • 反例：使用数组索引作为 key，会导致状态错位。
    <li v-for="(todo, idx) in todos" :key="idx">{{ todo.text }}</li>

    1.4 同一个节点，禁止同时使用 v-if 和 v-for¶

  <li
    v-for="user in users"
    v-if="user.isActive"
    :key="user.id"
  >

// Good
const activeUsers = computed(() => {
  return users.filter((user) => user.isActive)

<ul>

</ul>
    • 反例：在同一节点上同时使用 v-for 与 v-if。
    <li v-for="user in users" v-if="user.isActive">{{ user.name }}</li>

    1.5 使用组件范围的样式¶

    对于应用程序，顶级 App 组件和布局组件中的样式可以是全局的，但所有其他组件都应始终限定范围。

    <template>
  <button class="btn btn-close">×</button>
</template>
// Bad
<style>
.btn-close {
  background-color: red;
}
</style>
// Good
/* Using the `scoped` attribute */
<style scoped>
.button {
  border: none;
  border-radius: 2px;
}

.button-close {
  background-color: red;
}
</style>
<template>
  <button :class="[$style.button, $style.buttonClose]">×</button>
</template>

#### 细则映射示例（对应细则：1.1.5）

- 推荐：对大多数组件使用 `scoped` 或 CSS Modules，避免样式全局污染。

```vue
<style scoped>
.button { color: red }
</style>
    • 反例：在非顶级组件中使用全局样式，可能影响其他组件。
    /* 全局 style 影响范围过大 */
.btn-close { background: red }

    <!-- Using CSS modules -->
<style module>
.button {
  border: none;
  border-radius: 2px;
}

.buttonClose {
  background-color: red;
}
</style>
<template>
  <button class="c-Button c-Button--close">×</button>
</template>

<!-- Using the BEM convention -->
<style>
.c-Button {
  border: none;
  border-radius: 2px;
}

.c-Button--close {
  background-color: red;
}
</style>

    2 强烈推荐¶

    2.1 单文件组件¶

    每当构建系统可用于连接文件时，每个组件都应位于其自己的文件中。

    这有助于您在需要编辑组件或查看如何使用组件时更快地找到组件。

    app.component('TodoList', {
  // ...
})

app.component('TodoItem', {
  // ...
})
components/
|- TodoList.js
|- TodoItem.js

components/
|- TodoList.vue
|- TodoItem.vue

    2.2 单文件组件命名规则¶

    单文件组件的文件名应始终为 PascalCase 或 kebab-case。

    PascalCase 在代码编辑器中与自动补全配合得最好，因为它尽可能与我们在 JS（X） 和模板中引用组件的方式一致。但是，混合大小写的文件名有时会在不区分大小写的文件系统上产生问题，这就是为什么 kebab-case 也是完全可以接受的。

    // Bad
components/
|- mycomponent.vue
components/
|- myComponent.vue

// Good
components/
|- MyComponent.vue
components/
|- my-component.vue

    2.3 基础组件命名规则¶

    应用特定于应用程序的样式和约定的基本组件（也称为 presentational、dumb 或 pure components）都应以特定前缀开头，例如 Base、App 或 V。

    // Bad
components/
|- MyButton.vue
|- VueTable.vue
|- Icon.vue

// Good
components/
|- BaseButton.vue
|- BaseTable.vue
|- BaseIcon.vue

components/
|- AppButton.vue
|- AppTable.vue
|- AppIcon.vue

components/
|- VButton.vue
|- VTable.vue
|- VIcon.vue

    2.4 紧密耦合的组件名称¶

    与其父组件紧密耦合的子组件应包含父组件名称作为前缀。

    如果一个组件仅在单个父组件的上下文中有意义，那么这种关系应该在其名称中很明显。由于编辑器通常按字母顺序组织文件，因此这也会使这些相关文件彼此相邻。

    // Bad
components/
|- TodoList.vue
|- TodoItem.vue
|- TodoButton.vue

components/
|- SearchSidebar.vue
|- NavigationForSearchSidebar.vue
// Good
components/
|- TodoList.vue
|- TodoListItem.vue
|- TodoListItemButton.vue

components/
|- SearchSidebar.vue
|- SearchSidebarNavigation.vue

    2.5 组件名称中的单词顺序¶

    组件名称应以最高级别（通常是最通用）的词语开头，并以描述性修饰词语结尾。

    你也许会疑惑，为什么我们要强制组件名称使用较少的自然语言？

    在自然英语中，形容词和其他描述词通常出现在名词之前，而例外则需要连接词。例如：

    • Coffee with milk
    • Soup of the day
    • Visitor to the museum

    如果您愿意，您绝对可以在组件名称中包含这些连接器词，但顺序仍然很重要。

    另请注意，被视为“最高级”的内容将与你的应用相关。例如，假设一个应用程序具有搜索表单。它可能包括如下组件：

    components/
|- ClearSearchButton.vue
|- ExcludeFromSearchInput.vue
|- LaunchOnStartupCheckbox.vue
|- RunSearchButton.vue
|- SearchInput.vue
|- TermsCheckbox.vue

    您可能会注意到，很难看出哪些组件特定于搜索。现在让我们根据规则重命名组件：

    components/
|- SearchButtonClear.vue
|- SearchButtonRun.vue
|- SearchInputExcludeGlob.vue
|- SearchInputQuery.vue
|- SettingsCheckboxLaunchOnStartup.vue
|- SettingsCheckboxTerms.vue

    由于编辑器通常按字母顺序组织文件，因此组件之间的所有重要关系现在一目了然。

    您可能想以不同的方式解决这个问题，将所有搜索组件嵌套在“search”目录下，然后将所有设置组件嵌套在“settings”目录下。我们只建议在非常大的应用程序（例如 100+ 组件）中考虑这种方法，原因如下：

    • 浏览嵌套的子目录通常比滚动浏览单个 components 目录需要更多的时间。
    • 名称冲突（例如多个 ButtonDelete.vue 组件）使得在代码编辑器中快速导航到特定组件变得更加困难。
    • 重构变得更加困难，因为 find-and-replace 通常不足以更新对移动组件的相对引用。
    // Bad
components/
|- ClearSearchButton.vue
|- ExcludeFromSearchInput.vue
|- LaunchOnStartupCheckbox.vue
|- RunSearchButton.vue
|- SearchInput.vue
|- TermsCheckbox.vue
// Good
components/
|- SearchButtonClear.vue
|- SearchButtonRun.vue
|- SearchInputQuery.vue
|- SearchInputExcludeGlob.vue
|- SettingsCheckboxTerms.vue
|- SettingsCheckboxLaunchOnStartup.vue

    2.6 自闭合组件¶

    Bad:

    <!-- In Single-File Components, string templates, and JSX -->
<MyComponent></MyComponent>
<!-- In in-DOM templates -->
<my-component/>

    Good:

    <!-- In Single-File Components, string templates, and JSX -->
<MyComponent/>
<!-- In in-DOM templates -->
<my-component></my-component>

    2.7 模板中的组件名称大小写¶

    在大部分的项目中，组件名称经常是在单文件组件种驼峰命名，在模板中用kebab-case

    驼峰命名比kebab-case的优势如下：

    • 编辑器可以在模板中自动完成组件名称，因为 PascalCase 也用于 JavaScript。
    • MyComponent 与单个单词的 HTML 元素相比，在视觉上更明显，my-component 因为存在两个字符差异（两个大写字母），而不仅仅是一个字符（连字符）。
    • 如果你在模板中使用任何非 Vue 自定义元素，例如 Web 组件，PascalCase 确保你的 Vue 组件保持清晰可见。

    不幸的是，由于 HTML 不区分大小写，DOM 内模板仍然必须使用 kebab-case。

    还要注意，如果你已经在 kebab-case 上投入了大量资金，那么与 HTML 约定的一致性以及能够在所有项目中使用相同的大小写可能比上面列出的优势更重要。在这些情况下，到处使用 kebab-case 也是可以接受的。

    Bad

    <!-- In Single-File Components and string templates -->
<mycomponent/>
<!-- In Single-File Components and string templates -->
<myComponent/>
<!-- In in-DOM templates -->
<MyComponent></MyComponent>

    Good

    <!-- In Single-File Components and string templates -->
<MyComponent/>
<!-- In in-DOM templates -->
<my-component></my-component>

//OR
<!-- Everywhere -->
<my-component></my-component>

    2.8 JS/JSX 中的组件名称大小写¶

    JS/JSX 中的组件名称应始终为 PascalCase，尽管对于仅通过 app.component 使用全局组件注册的简单应用程序，它们在字符串中可能是 kebab-case。

    在 JavaScript 中，PascalCase 是类和原型构造函数的约定 - 本质上，任何可以具有不同实例的东西。Vue 组件也有实例，所以也使用 PascalCase 是有意义的。作为一个额外的好处，在 JSX（和模板）中使用 PascalCase 可以让代码的读者更容易区分组件和 HTML 元素。

    但是，对于仅通过 app.component 使用全局组件定义的应用程序，我们建议改用 kebab-case。原因是：

    • 在 JavaScript 中很少引用全局组件，因此遵循 JavaScript 的约定不太有意义。
    • 这些应用程序总是包含许多 DOM 内模板，其中必须使用 kebab-case。
    app.component('myComponent', {
  // ...
})
import myComponent from './MyComponent.vue'
export default {
  name: 'myComponent'
  // ...
}
export default {
  name: 'my-component'
  // ...
}
app.component('MyComponent', {
  // ...
})
app.component('my-component', {
  // ...
})
import MyComponent from './MyComponent.vue'
export default {
  name: 'MyComponent'
  // ...
}

    细则映射示例¶

    • 推荐：明确类型、必需性与 validator。
    const props = defineProps({
  status: {
    type: String,
    required: true,
    validator: v => ['ok','error'].includes(v)
  }
})
    • 反例：仅写字符串数组定义缺乏约束。
    const props = defineProps(['status'])

    2.9 组件名称命名为全称¶

    // Bad
components/
|- SdSettings.vue
|- UProfOpts.vue
// Good
components/
|- StudentDashboardSettings.vue
|- UserProfileOptions.vue

    2.10 prop 名称大小写¶

    在声明期间，prop 名称应始终使用 camelCase。在 DOM 模板中使用时，props 应该是 kebab 大小写的。单文件组件模板和 JSX 可以使用 kebab-case 或 camelCase 属性。大小写应该是一致的 - 如果你选择使用 camelCased props，请确保在你的应用程序中不使用 kebab-cased 的 props

    // Bad
const props = defineProps({
  'greeting-text': String
})
// for in-DOM templates
<welcome-message greetingText="hi"></welcome-message>


// Good
const props = defineProps({
  greetingText: String
})
// for SFC - please make sure your casing is consistent throughout the project
// you can use either convention but we don't recommend mixing two different casing styles
<WelcomeMessage greeting-text="hi"/>
// or
<WelcomeMessage greetingText="hi"/>

// for in-DOM templates
<welcome-message greeting-text="hi"></welcome-message>

    2.11 多属性元素¶

    具有多个属性的元素应跨越多行，每行一个属性。

    在 JavaScript 中，将具有多个属性的对象拆分为多行被广泛认为是一种很好的约定，因为它更容易阅读。我们的模板和 JSX 也值得同样的考虑。

    // Bad
<img src="https://vuejs.org/images/logo.png" alt="Vue Logo">
<MyComponent foo="a" bar="b" baz="c"/>
// Good
<img
  src="https://vuejs.org/images/logo.png"
  alt="Vue Logo"
>
<MyComponent
  foo="a"
  bar="b"
  baz="c"
/>

    2.12 模板中的简单表达式¶

    组件模板应仅包含简单表达式，更复杂的表达式应重构为计算属性或方法。

    模板中的复杂表达式使它们的声明性降低。我们应该努力描述应该出现什么，而不是我们如何计算该值。

    计算属性和方法还允许重用代码。

    // Bad
{{
  fullName.split(' ').map((word) => {
    return word[0].toUpperCase() + word.slice(1)
  }).join(' ')
}}

// Good
<!-- In a template -->
{{ normalizedFullName }}
// The complex expression has been moved to a computed property
const normalizedFullName = computed(() =>
  fullName.value
    .split(' ')
    .map((word) => word[0].toUpperCase() + word.slice(1))
    .join(' ')
)

    2.13 简单的计算属性¶

    复杂的计算属性应该被分割成尽可能多的更简单的属性。

    更简单、命名良好的计算属性有以下优势：

    • 更便于测试：当每个计算属性只包含一个非常简单的表达式，并且依赖项非常少时，编写测试来确认它是否正常工作要容易得多。
    • 更便于读代码：简化计算属性会强制您为每个值指定一个描述性名称，即使它没有被重用。这使得其他开发人员（以及未来的您）更容易专注于他们关心的代码并弄清楚发生了什么。
    • 更能适应不断变化的需求：任何可以命名的值都可能对视图有用。例如，我们可能决定显示一条消息，告诉用户他们节省了多少钱。我们也可能决定计算销售税，但可能会单独显示，而不是作为最终价格的一部分。

    小型、集中的计算属性对信息的使用方式所做的假设较少，因此随着需求的变化，需要较少的重构。

    // Bad
const price = computed(() => {
  const basePrice = manufactureCost.value / (1 - profitMargin.value)
  return basePrice - basePrice * (discountPercent.value || 0)
})

// Good
const basePrice = computed(
  () => manufactureCost.value / (1 - profitMargin.value)
)

const discount = computed(
  () => basePrice.value * (discountPercent.value || 0)
)

const finalPrice = computed(() => basePrice.value - discount.value)

    2.14 带引号的属性值¶

    非空 HTML 属性值应始终位于引号内（单引号或双引号，以 JS 中未使用的为准）。

    虽然在 HTML 中不需要没有任何空格的属性值必须有引号，但这种做法通常会导致避免使用空格，从而使属性值的可读性降低。

    <input type=text>
// Bad
<AppSidebar :style={width:sidebarWidth+'px'}>
// Good
<AppSidebar :style="{ width: sidebarWidth + 'px' }">

    2.15 指令简写¶

    指令简写 （： for v-bind：， @ for v-on： 和 # for v-slot） 应该总是使用或从不使用。(不要两者混用)

    // Bad
<input
  v-bind:value="newTodoText"
  :placeholder="newTodoInstructions"
>
// Good
<input
  :value="newTodoText"
  :placeholder="newTodoInstructions"
>
<input
  v-bind:value="newTodoText"
  v-bind:placeholder="newTodoInstructions"
>

// Bad
<input
  v-on:input="onInput"
  @focus="onFocus"
>
// Good
<input
  @input="onInput"
  @focus="onFocus"
>
<input
  v-on:input="onInput"
  v-on:focus="onFocus"
>


// Bad
<template v-slot:header>
  <h1>Here might be a page title</h1>
</template>
<template #footer>
  <p>Here's some contact info</p>
</template>
// Good
<template v-slot:header>
  <h1>Here might be a page title</h1>
</template>

<template v-slot:footer>
  <p>Here's some contact info</p>
</template>

<template #header>
  <h1>Here might be a page title</h1>
</template>

<template #footer>
  <p>Here's some contact info</p>
</template>

    3 补充：实践建议与工具¶

    • 使用 <script setup> 与 TypeScript（推荐）：在新项目中优先使用 <script setup lang="ts">，结合 defineProps/defineEmits 明确类型。
    • 静态检查/格式化：启用 eslint + plugin:vue/vue3-recommended 与 prettier，并在 CI 中运行以保持风格一致。
    • 组件粒度：如果组件超过 ~200 行或承担多种职责，应拆分为更小的组件并添加单元测试。
    • 性能注意：对于大型列表使用稳定 :key，并在适当场景下采用虚拟滚动以减少 DOM 操作。
    • 示例（props 校验）：请参考文中 defineProps 的最佳/反例；尽量提供 validator、required 与 default。

    以上补充旨在为团队提供可执行的工具链与快速提升代码质量的实践。
