    基础原则¶

    1. 单一职责：每个容器只运行一个主进程
    2. 无状态：容器本身不存储持久化数据，数据通过 Volume 挂载
    3. 不可变性：镜像构建后不应在运行时修改，通过重新构建更新
    4. 最小权限：以非 root 用户运行，只开放必要端口
    5. 可重现性：锁定依赖版本，确保每次构建结果一致
    6. 层数最小化：合并同类 RUN 指令，减少镜像层数
    7. 可观测性：内置健康检查端点，日志输出到 stdout/stderr，支持结构化日志
    8. 快速启动：容器应能在秒级内完成启动，避免启动时执行耗时初始化

    基础镜像选择¶

    优先级顺序¶

    scratch / distroless > alpine > slim > 完整发行版
    类型 大小 适用场景
    scratch 0MB 纯静态二进制（Go、Rust）
    gcr.io/distroless/* ~2MB 无 shell 的生产镜像
    alpine ~5MB 需要包管理的轻量镜像
    debian:slim / ubuntu:*-slim ~30-80MB 需要 glibc 兼容性
    完整发行版 >100MB 复杂依赖、开发调试

    规范¶

    • 必须指定具体版本标签，禁止使用 latest
    • 优先使用官方镜像（Docker Hub Official）
    • 生产镜像优先选择 alpine 或 distroless
    • 定期更新基础镜像以获取安全补丁
    # ✅ 推荐
FROM golang:1.22-alpine AS builder
FROM gcr.io/distroless/static-debian12:nonroot

# ❌ 禁止
FROM ubuntu:latest
FROM python

    指令规范¶

    FROM¶

    # 使用 AS 命名阶段（多阶段构建）
FROM node:20-alpine AS base
FROM node:20-alpine AS builder
FROM node:20-alpine AS runner

    LABEL¶

    LABEL maintainer="team@example.com" \
      version="1.0.0" \
      description="App description"

    RUN¶

    • 每个逻辑单元合并为一条 RUN 指令，用 && 连接，\ 换行
    • 每条命令一行，便于阅读和 diff
    • 安装完包后立即清理缓存
    # ✅ 推荐
RUN apt-get update && apt-get install -y --no-install-recommends \
    curl \
    ca-certificates \
    && rm -rf /var/lib/apt/lists/*

# Alpine 清理
RUN apk add --no-cache \
    curl \
    ca-certificates

# ❌ 避免：多层 RUN 增加层数
RUN apt-get update
RUN apt-get install -y curl
RUN rm -rf /var/lib/apt/lists/*

    COPY vs ADD¶

    • 优先使用 COPY，语义明确
    • ADD 仅用于需要自动解压 tar 包或从 URL 获取文件的场景
    # ✅ 推荐
COPY package*.json ./
COPY --from=builder /app/dist ./dist

# ADD 的合理用场
ADD https://example.com/file.tar.gz /tmp/

    WORKDIR¶

    • 始终使用绝对路径
    • 不要使用 RUN cd /some/path，使用 WORKDIR 替代
    WORKDIR /app

    CMD vs ENTRYPOINT¶

    指令 用途
    ENTRYPOINT 定义容器的主执行命令（不可被 docker run 覆盖）
    CMD 提供默认参数，可被 docker run 后的参数覆盖 
    # 推荐：JSON 数组格式（exec 形式），避免 shell 信号无法传递
ENTRYPOINT ["node", "server.js"]
CMD ["--port", "3000"]

# 组合使用
ENTRYPOINT ["/app/entrypoint.sh"]
CMD ["--config", "/etc/app/config.yaml"]

# ❌ 避免 shell 形式（PID 1 问题，信号无法传递）
CMD node server.js

    ENV¶

    • 只设置运行时必要的环境变量
    • 不要在 ENV 中存储敏感信息（密码、Token）
    ENV NODE_ENV=production \
    PORT=3000

    EXPOSE¶

    • 仅声明文档用途，不实际发布端口

    多阶段构建¶

    多阶段构建是减小镜像体积、分离构建环境与运行环境的核心手段。

WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -ldflags="-w -s" -o server ./cmd/server

# 阶段 2：运行
FROM gcr.io/distroless/static-debian12:nonroot AS runner
COPY --from=builder /app/server /server

docker build --target test -t myapp:test .

# 构建最终生产阶段
docker build --target runner -t myapp:prod .

    层缓存优化¶

    Docker 构建时，每一条指令对应一个镜像层。若某层内容未变化，Docker 直接复用缓存；一旦某层缓存失效，其后续所有层都必须重新执行。因此合理安排指令顺序和使用 BuildKit 高级缓存是提升构建效率的核心手段。

    缓存失效触发条件¶

    指令 失效条件
    FROM 基础镜像 digest 变化
    RUN 命令字符串变化，或上层缓存失效
    COPY / ADD 文件内容（checksum）变化，或上层缓存失效
    ENV / ARG 值变化（ARG 会污染后续所有层缓存）
    WORKDIR 路径变化

    关键：COPY . . 会将整个构建上下文（包括源码）纳入 checksum 计算，只要任意文件改动就触发缓存失效，因此必须将其拆分，先单独复制依赖描述文件。

    原则一：变化频率低的层放前面¶

    将指令按变化频率从低到高排列，使高频变化不影响低频层的缓存：

    操作系统/运行时层（极少变）
    ↓
系统依赖安装（较少变）
    ↓
应用依赖安装（package.json / go.mod / requirements.txt 变时）
    ↓
配置文件（偶尔变）
    ↓
源代码（频繁变）
    ↓
构建产物生成

    # ✅ 正确顺序
FROM node:20-alpine
WORKDIR /app

# 系统依赖（最少变）
RUN apk add --no-cache curl

# 依赖描述文件（package.json 变时才失效）
COPY package.json package-lock.json ./
RUN npm ci

# 源代码（最频繁变，放最后）
COPY . .
RUN npm run build

# ❌ 错误顺序：任何代码改动都导致 npm ci 重新执行
COPY . .          # ← 源码变化，后续全部失效
RUN npm ci        # ← 每次都重新安装
RUN npm run build

    原则二：精准 COPY，避免宽泛匹配¶

    # ❌ 复制整个目录，任意文件变化都触发缓存失效
COPY . .

# ✅ 只复制真正需要的文件
COPY package.json package-lock.json ./
COPY tsconfig.json ./
COPY src/ ./src/
COPY public/ ./public/

    配合 .dockerignore 排除测试文件、文档、.git 等，减少 checksum 计算范围和上下文传输体积。

    原则三：ARG 放在尽量靠近使用处¶

    ARG 值变化会使其后的所有层缓存失效：

    # ❌ ARG 放在最前面，值变化导致全部重建
ARG APP_VERSION=1.0.0
FROM golang:1.22-alpine
RUN go mod download      # 每次版本号变就重新执行
COPY . .
RUN go build -ldflags="-X main.version=${APP_VERSION}" .

# ✅ ARG 紧贴使用处
FROM golang:1.22-alpine
COPY go.mod go.sum ./
RUN go mod download      # 缓存不受 APP_VERSION 影响
COPY . .
ARG APP_VERSION=1.0.0    # ← 放在真正用到它的指令之前
RUN go build -ldflags="-X main.version=${APP_VERSION}" .

COPY src ./src
RUN --mount=type=cache,target=/root/.cargo/registry \
    --mount=type=cache,target=/root/.cargo/git \

    BuildKit 缓存挂载详解¶

    BuildKit --mount=type=cache 将宿主机目录挂载到构建步骤，不写入镜像层，跨多次构建持久保留。

    # syntax=docker/dockerfile:1

# 基本语法
RUN --mount=type=cache,target=<容器内路径>[,id=<缓存ID>][,sharing=<模式>] \
    <命令>

    关键参数说明：

    参数 说明 默认值
    target 容器内缓存挂载路径 必填
    id 缓存标识符，不同 stage 可共享同一缓存 基于 target 派生
    sharing 并发共享模式：shared（多构建共享）/ private（独占）/ locked（排队） shared
    uid / gid 挂载目录的所有者 0
    ro 只读挂载 false 
    # syntax=docker/dockerfile:1

# pip：所有使用 python:3.12 的阶段共享同一缓存
RUN --mount=type=cache,id=pip-cache,target=/root/.cache/pip \
    pip install -r requirements.txt

# apt：使用 locked 避免并发构建时索引损坏
RUN --mount=type=cache,target=/var/cache/apt,sharing=locked \
    --mount=type=cache,target=/var/lib/apt,sharing=locked \
    apt-get update && apt-get install -y --no-install-recommends \
    build-essential libssl-dev

# Go：模块缓存跨 builder/runner 阶段共享
RUN --mount=type=cache,id=gomod,target=/go/pkg/mod \
    --mount=type=cache,id=gobuild,target=/root/.cache/go-build \
    go build ./...

# npm：使用 private 避免并发写入冲突
RUN --mount=type=cache,target=/root/.npm,sharing=private \
    npm ci

    远程缓存（Remote Cache）¶

    将构建缓存存储到 Registry，在 CI 无本地缓存的环境中大幅提速。

    # 推送缓存到 Registry（inline 模式，缓存内嵌入镜像 manifest）
docker buildx build \
  --cache-from type=registry,ref=registry.example.com/myapp:cache \
  --cache-to   type=registry,ref=registry.example.com/myapp:cache,mode=max \
  -t registry.example.com/myapp:latest --push .

# mode=max：缓存所有中间层（推荐，命中率更高）
# mode=min：只缓存最终镜像层（节省存储）

    # GitHub Actions GHA 缓存（Actions Cache 服务，无需自建 Registry）
docker buildx build \
  --cache-from type=gha \
  --cache-to   type=gha,mode=max \
  -t myapp:latest .

    # 本地目录缓存（适合 self-hosted runner 或 CI 共享卷）
docker buildx build \
  --cache-from type=local,src=/tmp/docker-cache \
  --cache-to   type=local,dest=/tmp/docker-cache,mode=max \
  -t myapp:latest .

    多阶段并行构建¶

    BuildKit 支持多个无依赖关系的构建阶段并行执行：

    # syntax=docker/dockerfile:1

# 阶段 A 和 B 无依赖关系，BuildKit 自动并行
FROM node:20-alpine AS frontend-builder
WORKDIR /frontend
COPY frontend/package*.json ./
RUN npm ci
COPY frontend/ .
RUN npm run build

FROM golang:1.22-alpine AS backend-builder
WORKDIR /backend
COPY backend/go.mod backend/go.sum ./
RUN go mod download
COPY backend/ .
RUN go build -o /app/server .

# 最终阶段：合并两个并行阶段的产物
FROM gcr.io/distroless/base-debian12:nonroot
COPY --from=frontend-builder /frontend/dist /app/static
COPY --from=backend-builder  /app/server    /server
EXPOSE 8080
ENTRYPOINT ["/server"]

    镜像体积优化¶

    镜像体积直接影响拉取速度、存储成本和攻击面。以下从多个维度系统优化。

    一、选用最小基础镜像¶

    优先选用精简版基础镜像，这是体积优化最直接的手段：

    # Go / Rust 静态二进制 → scratch 或 distroless/static
FROM scratch
FROM gcr.io/distroless/static-debian12:nonroot

# 需要 glibc + 少量系统库 → distroless/base
FROM gcr.io/distroless/base-debian12:nonroot

# 需要 CA 证书、时区数据 → distroless/base-nossl / cc
FROM gcr.io/distroless/cc-debian12:nonroot

# 需要包管理 → alpine:3.x（指定具体版本）
FROM alpine:3.19

# 需要 glibc 兼容 → debian:bookworm-slim
FROM debian:bookworm-slim

    各基础镜像体积参考：

    镜像 压缩大小 解压大小
    scratch 0 0
    gcr.io/distroless/static-debian12 ~2MB ~4MB
    gcr.io/distroless/base-debian12 ~20MB ~54MB
    alpine:3.19 ~3.5MB ~8MB
    debian:bookworm-slim ~30MB ~80MB
    ubuntu:22.04 ~29MB ~78MB
    ubuntu:22.04（完整） ~29MB ~78MB
    node:20-alpine ~40MB ~130MB
    node:20-slim ~95MB ~280MB
    node:20 ~360MB ~1.1GB

    二、多阶段构建分离构建与运行环境¶

    多阶段构建是体积优化最有效的手段，最终镜像只包含运行时必需文件：

    # 未优化前（单阶段）：镜像包含编译器、源码、依赖等全部内容
FROM golang:1.22  # ~800MB
COPY . .
RUN go build -o server .
# 最终镜像：~800MB+

# 多阶段优化后：最终镜像只含编译好的二进制
FROM golang:1.22-alpine AS builder   # 构建阶段，不进入最终镜像
COPY . .
RUN CGO_ENABLED=0 go build -ldflags="-w -s" -o /server .

FROM scratch                          # 最终镜像：0MB 基础
COPY --from=builder /server /server
# 最终镜像：~8MB

    各语言多阶段构建体积对比：

    语言 单阶段 多阶段 + alpine 多阶段 + distroless/scratch
    Go ~800MB ~15MB ~5-10MB
    Rust ~1.8GB ~15MB ~5-10MB
    Node.js ~1.1GB ~80MB ~120MB（distroless/nodejs）
    Python ~1.0GB ~80MB ~90MB（distroless/python）
    Java 17 ~500MB ~200MB（JRE） ~100MB（jlink 定制 JRE）

    三、清理包管理器缓存¶

    包管理器会留下索引、缓存文件，必须在同一 RUN 指令中清理（清理放在单独 RUN 无效，因为前一层已固化）：

    # apt / apt-get
RUN apt-get update && apt-get install -y --no-install-recommends \
    curl \
    ca-certificates \
    libssl-dev \
    && apt-get clean \
    && rm -rf /var/lib/apt/lists/* /tmp/* /var/tmp/*

# apk（Alpine）：--no-cache 参数等价于 update + 使用后清理
RUN apk add --no-cache \
    curl \
    ca-certificates \
    openssl-dev

# yum / dnf（CentOS/RHEL）
RUN yum install -y curl openssl-devel \
    && yum clean all \
    && rm -rf /var/cache/yum

# pip
RUN pip install --no-cache-dir -r requirements.txt
# 或清理已有缓存
RUN pip install -r requirements.txt \
    && pip cache purge \
    && rm -rf /root/.cache/pip

# npm
RUN npm ci --omit=dev \
    && npm cache clean --force

# pnpm
RUN pnpm install --frozen-lockfile --prod \
    && pnpm store prune

# yarn 2+（berry）
RUN yarn workspaces focus --production \
    && yarn cache clean

# gem（Ruby）
RUN gem install bundler \
    && bundle install --without development test \
    && bundle clean \
    && rm -rf /usr/local/bundle/cache/*.gem

# composer（PHP）
RUN composer install --no-dev --optimize-autoloader \
    && rm -rf /root/.composer/cache

# cargo：最终阶段不保留编译缓存（通过多阶段构建自然隔离）

    四、按需安装，使用虚拟包¶

    Alpine 支持虚拟包（.build-deps），安装编译依赖后立即删除：

    # 安装 → 编译 → 删除编译依赖
RUN apk add --no-cache --virtual .build-deps \
    gcc musl-dev libffi-dev openssl-dev \
    && pip install --no-cache-dir cryptography bcrypt \
    && apk del .build-deps

# 只保留运行时必需的包
RUN apk add --no-cache \
    libffi openssl  # ← 运行时依赖

    Debian 对应方式：

    RUN apt-get update \
    && apt-get install -y --no-install-recommends \
       gcc python3-dev libffi-dev \
    && pip install --no-cache-dir cryptography \
    && apt-get purge -y --auto-remove gcc python3-dev libffi-dev \
    && rm -rf /var/lib/apt/lists/*

    # .github/workflows/docker.yml
name: Build and Push Docker Image

on:
  push:
    branches: [main]
    tags: ['v*']
  pull_request:
    branches: [main]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Log in to Registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Extract metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=ref,event=branch
            type=semver,pattern={{version}}
            type=semver,pattern={{major}}.{{minor}}
            type=sha,prefix=sha-

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          platforms: linux/amd64,linux/arm64
          push: ${{ github.event_name != 'pull_request' }}
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
          build-args: |
            BUILD_DATE=${{ github.event.repository.updated_at }}
            VCS_REF=${{ github.sha }}

      - name: Run Trivy vulnerability scanner
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ steps.meta.outputs.version }}
          format: 'sarif'
          output: 'trivy-results.sarif'
          severity: 'CRITICAL,HIGH'
          exit-code: '1'

    # 运行时调整（高并发服务）
docker run --ulimit nofile=65536:65536 myapp

# daemon.json 全局默认值
# { "default-ulimits": { "nofile": { "Name": "nofile", "Hard": 65536, "Soft": 65536 } } }

    容器内 DNS 解析问题¶

    # 指定 DNS 服务器
docker run --dns 8.8.8.8 --dns 8.8.4.4 myapp

# daemon.json 全局配置
# { "dns": ["8.8.8.8", "8.8.4.4"] }

    镜像标签策略¶

    # 推荐：语义化版本 + Git SHA + latest
myapp:1.2.3          # 语义化版本（不可变）
myapp:1.2            # 主次版本（滚动更新）
myapp:sha-abc1234    # Git SHA（可追溯）
myapp:latest         # 最新稳定（CI 自动更新）

# 分支构建
myapp:main-20240226-abc1234
