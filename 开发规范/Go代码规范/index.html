    1. 总体原则¶

    • 【强制】可读性优先于技巧。
    • 【强制】优先遵循官方习惯（Effective Go、Code Review Comments）。
    • 【强制】风格分歧以工具输出为准。
    // 示例
// 细则1：可读性优先，函数名完整表达意图
func CalculateTotalPrice(items []Item) int { return 0 }

// 细则2：优先 Go 官方习惯，错误显式返回
func LoadConfig(path string) (Config, error) { return Config{}, nil }

// 细则3：风格交给工具，不手动对齐空格
// 运行 gofmt 后统一格式

    // 推荐：命名清晰、逻辑直接
func CalculateTotalPrice(items []Item) int {
    totalPrice := 0
    for _, item := range items {
        totalPrice += item.Price
    }
    return totalPrice
}

// 反例：过度技巧化，难读
func calc(i []Item) (t int) {
    for _, v := range i {
        t += v.Price
    }
    return
}

    说明：命名与返回值应表达业务语义，避免省略式风格；读者应在 10 秒内理解函数目的。

    2. 工具链与工程基线¶

    2.1 必备工具¶

    • 【强制】格式化：gofmt
    • 【推荐】导入整理：goimports
    • 【推荐】静态检查：go vet、golangci-lint
    • 【推荐】漏洞扫描：govulncheck
    # 推荐：统一工具链
# 细则1：格式化
gofmt -w .
# 细则2：导入整理
goimports -w .
# 细则3：静态检查
go vet ./... && golangci-lint run
# 细则4：漏洞扫描
govulncheck ./...

    说明：gofmt/goimports 解决风格一致性，vet/lint 发现语义问题，govulncheck 覆盖供应链风险，四者缺一不可。

    2.2 CI 最低门槛¶

    • 【强制】go test ./...
    • 【强制】gofmt -w 后无差异
    • 【推荐】go vet ./...
    • 【推荐】golangci-lint run
    # 示例
name: go-check

on: [pull_request]

jobs:
    verify:
        runs-on: ubuntu-latest
        steps:
            - uses: actions/checkout@v4
            - uses: actions/setup-go@v5
                with:
                    go-version: '1.22'
            - run: go test ./...
            - run: test -z "$(gofmt -l .)"
            - run: go vet ./...
            - run: golangci-lint run

    说明：CI 门槛必须与本地开发命令一致，避免“本地通过、CI 失败”的双标准。

    2.2.1 golangci.yml 模板（示例）¶

    run:
    timeout: 5m

linters:
    enable:
        - govet
        - staticcheck
        - ineffassign
        - errcheck
        - revive
        - gocritic

issues:
    exclude-use-default: false

    2.3 版本管理¶

    • 【强制】使用 Go Modules。
    • 【推荐】go 指令与团队运行环境版本一致。
    • 【推荐】依赖升级采用“可回滚”的小步策略。
    // 示例
// 细则1：启用 Go Modules
module example.com/project

// 细则2：锁定团队版本
go 1.22

// 细则3：小步升级依赖（一次一个）
require github.com/gin-gonic/gin v1.10.0

    说明：依赖升级建议一批一库，并配合回归测试；出现问题能快速回滚到上个可用版本。

    3. 命名规范¶

    3.1 包与文件¶

    • 【强制】包名短、小写、无下划线、无复数歧义。
    • 【推荐】文件名用小写下划线，如 user_service.go。
    // 示例
// 细则1：包名短小写
package user

// 细则2：文件名示例 user_service.go
type Service struct{}

    说明：包名应反映领域能力而不是技术实现，如 user、order，避免 common、utils 泛化命名。

    3.2 标识符¶

    • 【强制】导出符号首字母大写，包内符号小写。
    • 【强制】缩写统一：ID、URL、HTTP。
    • 【推荐】接口命名体现行为，不强制 -er，但需可读。
    // 示例
// 细则1：导出/非导出命名
type UserService struct{}
func (s *UserService) getInternalState() {}

// 细则2：缩写统一
type HTTPClient struct{}
// 推荐
type UserID string

// 反例：缩写风格不统一
type UserId string

// 细则3：接口名体现行为
type UserRepository interface {
    FindByID(ctx context.Context, id string) (*User, error)
}

    说明：缩写统一能减少 API 心智负担，尤其在跨仓协作和代码生成场景中收益明显。

    4. 包结构与分层¶

    4.1 建议结构¶

    • cmd/：程序入口
    • internal/：内部实现
    • pkg/：可复用公共包（仅必要时）
    • api/：协议定义（OpenAPI/Proto）
    # 示例
# 细则1: cmd
cmd/server/main.go
# 细则2: internal
internal/user/service.go
# 细则3: pkg
pkg/clock/clock.go
# 细则4: api
api/openapi.yaml

    项目骨架（示例）¶

    .
├── cmd/
│   └── server/
│       └── main.go
├── internal/
│   ├── app/
│   ├── domain/
│   ├── repository/
│   └── transport/
├── pkg/
├── api/
└── go.mod

    // cmd/server/main.go
package main

func main() {
    _ = NewServer().Run()
}

    说明：入口层只做组装，不承载业务；业务逻辑放到 internal，降低耦合和启动复杂度。

    4.2 依赖方向¶

    • 【强制】上层依赖下层，禁止反向依赖。
    • 【强制】禁止循环依赖。
    • 【推荐】通过接口实现依赖倒置。
    // 示例
// 细则1：service 依赖 repo 抽象，而非 repo 反向依赖 service
type UserRepo interface { FindByID(context.Context, string) (*User, error) }

// 细则2：避免 A->B->A 循环引用
// package service imports repository; repository 不 imports service

// 细则3：依赖倒置
type UserService struct { repo UserRepo }

    说明：依赖倒置让上层面向抽象编程，便于单测替身注入和基础设施替换。

    5. 代码风格与设计¶

    5.1 函数设计¶

    • 【推荐】函数职责单一。
    • 【推荐】参数过多时引入参数对象。
    • 【推荐】优先返回值，不隐式修改外部状态。
    // 示例
// 细则1：只做“创建用户校验+构造”单一职责
func CreateUser(ctx context.Context, in CreateUserInput) (User, error) { return User{}, nil }

// 细则2：参数对象
type CreateUserInput struct { Name, Email string }

type CreateUserInput struct {
    Name  string
    Email string
}

// 细则3：通过返回值传递结果，不修改全局变量
func CreateUser(ctx context.Context, in CreateUserInput) (User, error) {
    if in.Name == "" {
        return User{}, errors.New("name is required")
    }
    return User{Name: in.Name, Email: in.Email}, nil
}

// 反例：参数过多且副作用不透明
func CreateUserBad(ctx context.Context, name, email, phone, addr, city, country string, enable bool) error {
    globalUserCount++
    return nil
}

    说明：参数对象提升可读性，且便于后续字段扩展；副作用应显式暴露在返回值或依赖中。

    5.2 Receiver 约定¶

    • 【强制】修改状态的方法使用指针接收者。
    • 【推荐】只读方法可使用值接收者，但需考虑拷贝成本。
    // 示例
type Counter struct {
    value int
}

// 推荐：修改状态使用指针接收者
func (c *Counter) Inc() {
    c.value++
}

// 推荐：只读可值接收者
func (c Counter) Value() int {
    return c.value
}

    说明：同一类型若混用值/指针接收者需谨慎，保持一致性能降低“是否拷贝/是否可变”的歧义。

    5.3 禁止项¶

    • 【强制】禁止在业务代码中使用全局可变变量。
    • 【强制】禁止隐式依赖初始化顺序。
    • 【推荐】单函数超过 80 行应考虑拆分。
    // 示例
// 细则1：反例-全局可变状态
var badConfig map[string]string

// 细则2：反例-init 顺序耦合
func init() { badConfig = map[string]string{"k":"v"} }

// 细则3：推荐-拆分为小函数
func HandleRequest(ctx context.Context, req Req) error {
    if err := validate(req); err != nil { return err }
    return process(ctx, req)
}

// 推荐：通过依赖注入
type Service struct {
    config map[string]string
}

    说明：全局可变状态会引入初始化顺序和并发安全问题，优先通过构造函数注入依赖。

    5.4 可见性控制¶

    • 【强制】最小导出原则。
    • 【推荐】包内辅助函数不导出。
    • 【推荐】避免“为测试而导出”。
    // 示例
// 细则1：仅导出必要类型
type UserService struct{}

// 细则2：辅助函数不导出
func buildUser(id string) *User { return &User{ID: id} }

// 细则3：测试通过同包测试访问私有方法，不强行导出
func (s *UserService) GetUser(ctx context.Context, id string) (*User, error) {
    return buildUser(id), nil
}

    说明：导出面越小，越容易维护兼容性；包内函数应默认私有，避免形成隐式公共 API。

    6. 错误处理¶

    6.1 基础规则¶

    • 【强制】错误立即处理，不忽略返回值中的 error。
    • 【强制】错误信息小写开头，不加句号。
    • 【强制】包装错误使用 %w。
    // 示例
f, err := os.Open("conf.yaml") // 细则1
if err != nil {
    return fmt.Errorf("open config failed: %w", err) // 细则2+3
}
defer f.Close()


// 反例：吞错
_ = os.Remove("temp.txt")

// 推荐：按场景判定并记录上下文
if err := os.Remove("temp.txt"); err != nil && !errors.Is(err, os.ErrNotExist) {
    return fmt.Errorf("remove temp file failed: %w", err)
}

    说明：错误处理要“可观测 + 可追踪 + 可恢复”，不能简单忽略。

    6.2 错误分类与传播¶

    • 【推荐】业务错误与系统错误分层。
    • 【推荐】在边界层统一映射错误码。
    • 【推荐】使用 errors.Is/As 判断类型。
    // 示例
var ErrUserNotFound = errors.New("user not found") // 细则1

func mapErrToCode(err error) int {
    if errors.Is(err, ErrUserNotFound) { // 细则3
        return 404 // 细则2
    }
    return 500
}

func (s *Service) GetUser(ctx context.Context, id string) (*User, error) {
    u, err := s.repo.FindByID(ctx, id)
    if err != nil {
        return nil, fmt.Errorf("find user by id failed: %w", err)
    }
    if u == nil {
        return nil, ErrUserNotFound
    }
    return u, nil
}

if errors.Is(err, ErrUserNotFound) {
    return http.StatusNotFound
}
return http.StatusInternalServerError

    6.3 Panic 使用边界¶

    • 【强制】业务逻辑中禁止以 panic 作为流程控制。
    • 【推荐】仅在不可恢复的启动失败或编程错误使用 panic。
    • 【推荐】HTTP/RPC 边界必须设置 recover 中间件。
    // 示例
func boot() {
    if err := initCriticalResource(); err != nil { // 细则2
        panic(err)
    }
}

// 细则1：业务流程使用 error 返回，不 panic
func CreateOrder(ctx context.Context, req CreateOrderReq) error { return nil }

func Recovery(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        defer func() {
            if rec := recover(); rec != nil {
                http.Error(w, "internal error", http.StatusInternalServerError)
            }
        }()
        next.ServeHTTP(w, r)
    })
}

    说明：recover 仅用于边界层兜底，业务分支应使用 error 明确表达失败路径。

    7. 并发规范¶

    7.1 Goroutine 生命周期¶

    • 【强制】每个 goroutine 都有退出路径。
    • 【强制】启动 goroutine 时定义取消机制。
    • 【推荐】泄漏风险通过测试或 pprof 验证。
    go func(ctx context.Context) {
    for {
        select {
        case <-ctx.Done():
            return
        default:
            work()
        }
    }
}(ctx)

    说明：所有后台任务都必须绑定取消信号，避免请求结束后 goroutine 继续运行造成泄漏。

    7.2 Context 约定¶

    • 【强制】context.Context 作为第一个参数。
    • 【强制】不要在结构体中持久保存 request context。
    • 【强制】不要把可选业务参数塞进 context。
    // 推荐
func (s *Service) Query(ctx context.Context, req QueryRequest) error {
    return s.repo.Query(ctx, req)
}

    说明：context 只承载取消、超时、链路信息，不承载业务参数。

    7.2.1 超时模板¶

    func (s *Service) Handle(ctx context.Context) error {
    ctx, cancel := context.WithTimeout(ctx, 3*time.Second)
    defer cancel()

    return s.repo.Do(ctx)
}

// 反例：无超时
func (s *Service) BadHandle(ctx context.Context) error {
    return s.repo.Do(ctx)
}

    说明：所有外部 I/O（DB、RPC、HTTP）必须配置超时，超时是系统自我保护边界。

    7.3 Channel 约定¶

    • 【强制】谁创建，谁关闭（默认规则）。
    • 【推荐】发送方关闭通道，接收方只消费。
    • 【推荐】用 select 处理超时、取消和退出信号。
    select {
case <-ctx.Done():
    return ctx.Err()
case msg, ok := <-ch:
    if !ok {
        return nil
    }
    _ = msg
}

// 推荐：发送方关闭
func producer(ch chan<- int) {
    defer close(ch)
    for i := 0; i < 3; i++ {
        ch <- i
    }
}

    说明：关闭规则固定后，可显著减少重复 close、发送到已关闭通道等并发 bug。

    7.4 共享状态¶

    • 【推荐】优先消息通信，其次互斥锁。
    • 【强制】锁保护的数据必须最小化锁范围。
    • 【推荐】sync.Map 仅用于特定并发读多写少场景。
    type Cache struct {
    mu   sync.RWMutex
    data map[string]string
}

func (c *Cache) Get(k string) (string, bool) {
    c.mu.RLock()
    v, ok := c.data[k]
    c.mu.RUnlock()
    return v, ok
}

    说明：加锁范围越小，锁竞争越低；先取数据副本再做耗时逻辑。

    7.5 WaitGroup 与 errgroup 约定¶

    • 【强制】并发任务集合必须等待完成，不允许主流程提前退出。
    • 【推荐】需要“首错返回 + 全量取消”时使用 errgroup.WithContext。
    • 【推荐】并发任务内优先返回错误，不在 goroutine 内直接日志终结流程。
    func fetchAll(ctx context.Context, ids []string) error {
    g, ctx := errgroup.WithContext(ctx)
    for _, id := range ids {
        id := id
        g.Go(func() error {
            return fetchOne(ctx, id)
        })
    }
    return g.Wait()
}

    说明：errgroup 可以在任一任务失败后统一取消其余任务，避免“部分失败仍继续消耗资源”。

    7.6 并发数量控制（背压）¶

    • 【强制】外部依赖调用必须有并发上限，防止瞬时流量打爆下游。
    • 【推荐】用有缓冲信号量（channel）或 worker pool 控制并发度。
    func processBatch(ctx context.Context, jobs []Job) error {
    sem := make(chan struct{}, 10)
    g, ctx := errgroup.WithContext(ctx)

    for _, job := range jobs {
        job := job
        sem <- struct{}{}
        g.Go(func() error {
            defer func() { <-sem }()
            return handleJob(ctx, job)
        })
    }

    return g.Wait()
}

    说明：并发上限就是背压阀门，防止系统在高峰时进入雪崩。

    7.7 Worker Pool 规范¶

    • 【推荐】CPU 密集型任务采用固定 worker 数，减少 goroutine 过度膨胀。
    • 【强制】worker 必须可优雅退出（监听 ctx.Done() 和任务通道关闭）。
    func startWorkers(ctx context.Context, n int, jobs <-chan Job) {
    for i := 0; i < n; i++ {
        go func() {
            for {
                select {
                case <-ctx.Done():
                    return
                case job, ok := <-jobs:
                    if !ok {
                        return
                    }
                    _ = process(job)
                }
            }
        }()
    }
}

    说明：固定 worker 数量可预测资源占用，便于容量评估和压测。

    7.8 定时任务与 ticker 约定¶

    • 【强制】time.Ticker 使用后必须 Stop()。
    • 【推荐】定时循环必须响应 ctx.Done()。
    func runCron(ctx context.Context) {
    ticker := time.NewTicker(30 * time.Second)
    defer ticker.Stop()

    for {
        select {
        case <-ctx.Done():
            return
        case <-ticker.C:
            syncData(ctx)
        }
    }
}

    说明：未停止的 ticker 会持续占用资源，是常见隐性泄漏点。

    7.9 竞态与死锁防护¶

    • 【强制】并发改动必须执行竞态检测：go test -race ./...。
    • 【推荐】避免锁嵌套；如必须嵌套，固定全局加锁顺序。
    // 推荐：固定锁顺序，避免 ABBA 死锁
func transfer(a, b *Account, n int) {
    first, second := a, b
    if a.ID > b.ID {
        first, second = b, a
    }

    first.mu.Lock()
    second.mu.Lock()
    defer second.mu.Unlock()
    defer first.mu.Unlock()

    a.Balance -= n
    b.Balance += n
}

    说明：死锁预防优先于事后排查，统一锁顺序是最实用的工程策略。

    7.10 并发代码评审补充项¶

    • 【强制】是否存在 goroutine 无退出路径？
    • 【强制】是否存在持锁调用外部 I/O？
    • 【推荐】是否有并发上限与降级策略？
    - [ ] goroutine 均绑定 ctx 或显式退出条件
- [ ] channel 关闭方唯一且明确
- [ ] 外部调用已配置 timeout
- [ ] 已执行 `go test -race ./...`

    说明：并发问题上线后代价高，评审阶段清单化是最有效的前置控制手段。

    8. 日志与可观测性¶

    8.1 日志规范¶

    • 【强制】日志包含关键上下文（request_id、user_id、resource_id）。
    • 【强制】避免记录敏感信息（密码、token、身份证号）。
    • 【推荐】结构化日志优先（JSON 字段）。
    // 示例
logger.Info("create order", // 细则3
    "request_id", reqID,      // 细则1
    "user_id", userID,        // 细则1
    "resource_id", orderID,   // 细则1
)
// 细则2：不打印 password/token

// 反例：结构化字段缺失，且日志不可检索
logger.Info(fmt.Sprintf("create order success: %s", orderID))

    说明：结构化日志便于按字段检索和聚合告警，禁止拼字符串日志作为主形态。

    8.1.1 日志字段约定¶

    • 请求字段：request_id、trace_id、span_id
    • 业务字段：user_id、tenant_id、resource_id
    • 错误字段：error、error_code
    // 示例
logger.Error("db timeout",
    "request_id", reqID,   // 细则1
    "trace_id", traceID,   // 细则1
    "user_id", userID,     // 细则2
    "resource_id", "u-1", // 细则2
    "error", err.Error(),  // 细则3
    "error_code", "DB_TIMEOUT", // 细则3
)

    8.2 指标与追踪¶

    • 【推荐】关键路径暴露延迟、错误率、吞吐指标。
    • 【推荐】统一指标命名与标签约定。
    • 【推荐】分布式调用链打点并透传 trace_id。
    // 示例
start := time.Now()
err := s.repo.Save(ctx, order)
metrics.Observe("order_save_latency_ms", float64(time.Since(start).Milliseconds())) // 细则1+2
metrics.Inc("order_save_total") // 细则1+2
if err != nil { metrics.Inc("order_save_error_total") } // 细则1+2
_ = traceIDFromContext(ctx) // 细则3

    说明：指标命名应稳定且可聚合（如 _total、_latency_ms），避免随意变更导致监控断档。

    9. API 与序列化¶

    • 【强制】对外 DTO 与内部领域模型解耦。
    • 【推荐】序列化字段显式 tag，不依赖默认行为。
    • 【推荐】响应结构保持向后兼容。
    // 示例
type UserDTO struct { // 细则1
    ID   string `json:"id"`   // 细则2
    Name string `json:"name"` // 细则2
    Age  *int   `json:"age,omitempty"` // 细则3
}

func ToUserDTO(u User) UserDTO {
    return UserDTO{ID: u.ID, Name: u.Name}
}

    说明：DTO 负责“边界契约”，领域模型负责“业务语义”；两者分离可降低外部协议变更冲击。

    9.1 DTO 规则¶

    • 【强制】禁止直接对外暴露数据库实体。
    • 【强制】返回 JSON 字段必须显式 json tag。
    • 【推荐】新增字段应保证后向兼容（可选字段）。
    // 示例
type UserEntity struct { ID, Password string } // 细则1（内部）
type UserResp struct {
    ID    string  `json:"id"`             // 细则2
    Alias *string `json:"alias,omitempty"` // 细则3
}

// 反例：直接暴露实体，泄漏敏感字段
func BadGetUser() UserEntity {
    return UserEntity{ID: "u1", Password: "secret"}
}

    说明：响应结构必须经过裁剪与映射，避免将内部字段（密码、审计信息）外泄。

    10. 数据库与外部资源¶

    • 【强制】SQL 参数化，禁止拼接。
    • 【强制】数据库操作必须有超时和取消。
    • 【推荐】事务作用域最小化。
    • 【推荐】幂等写操作有唯一键或去重策略。
    // 示例
ctx, cancel := context.WithTimeout(ctx, 2*time.Second) // 细则2
defer cancel()

tx, err := db.BeginTx(ctx, nil) // 细则3
if err != nil { return err }
defer tx.Rollback()

row := db.QueryRowContext(ctx, "SELECT id, name FROM users WHERE id = ?", userID)

_, err = tx.ExecContext(ctx, "INSERT INTO payments(order_id, idempotency_key) VALUES(?, ?)", orderID, idemKey) // 细则1+4
if err != nil { return err }
tx.Commit()

// 反例：字符串拼接 SQL，存在注入风险
query := "SELECT id, name FROM users WHERE id = '" + userID + "'"
_ = db.QueryRowContext(ctx, query)

    说明：所有 SQL 必须参数化；即使输入来自内部系统也不能信任。

    10.1 查询规范¶

    • 【强制】查询必须有分页或上限。
    • 【强制】禁止全表扫描接口直接对外。
    • 【推荐】慢查询需记录 SQL 摘要与耗时。
    // 示例
const maxPageSize = 100

func ListUsers(ctx context.Context, page, pageSize int) ([]User, error) {
    if pageSize <= 0 || pageSize > maxPageSize { // 细则1
        return nil, errors.New("invalid page size")
    }
    start := time.Now()
    rows, err := db.QueryContext(ctx,
        "SELECT id, name FROM users ORDER BY id DESC LIMIT ? OFFSET ?",
        pageSize, offset,
    ) // 细则2

    if err != nil {
        return nil, err
    }
    if cost := time.Since(start); cost > 200*time.Millisecond { // 细则3
        logger.Warn("slow query", "sql", "users_list", "cost_ms", cost.Milliseconds())
    }

    defer rows.Close()
    return scanUsers(rows)
}

    说明：分页上限是服务保护阀；缺少上限会导致单请求占用过多连接和内存。

    11. 测试规范¶

    11.1 单元测试¶

    • 【强制】测试文件以 _test.go 结尾。
    • 【推荐】使用表驱动测试覆盖边界。
    // 示例
// 文件名：user_service_test.go  // 细则1
func TestAdd(t *testing.T) {
    tests := []struct {  // 细则2
        name string
        a    int
        b    int
        want int
    }{
        {name: "positive", a: 1, b: 2, want: 3},
        {name: "zero", a: 0, b: 0, want: 0},
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            if got := Add(tt.a, tt.b); got != tt.want {
                t.Fatalf("Add() = %d, want %d", got, tt.want)
            }
        })
    }
}

    11.2 测试实践¶

    • 【推荐】抽取测试辅助函数并使用 t.Helper()。
    • 【推荐】避免 sleep 等待，优先事件同步。
    • 【推荐】关键路径覆盖失败分支。
    // 示例
func mustCreateUser(t *testing.T, repo UserRepo, u User) { // 细则1
    t.Helper()
    if err := repo.Create(context.Background(), u); err != nil {
        t.Fatalf("create user failed: %v", err)
    }
}

// 细则2：用 channel 同步代替 sleep
done := make(chan struct{})
go func() { defer close(done); _ = doAsync() }()
<-done

// 细则3：覆盖失败路径

    说明：测试辅助函数要通过 t.Helper() 标注，失败栈才能指向真实调用位置。

    11.3 测试命名与分层¶

    • 【推荐】测试名采用 Test<函数名>_<场景>。
    • 【推荐】单测聚焦纯逻辑，集成测试聚焦依赖交互。
    • 【推荐】关键模块建立基准测试与回归测试双轨。
    // 示例
func TestCreateUser_InvalidEmail(t *testing.T) {} // 细则1+2
func TestCreateUser_Success(t *testing.T) {}      // 细则1+2
func BenchmarkCreateUser(b *testing.B) {}         // 细则3

    11.4 基准测试¶

    • 【推荐】性能敏感函数编写 benchmark。
    • 【推荐】基准数据固定化，减少噪声。
    // 示例
func BenchmarkParsePayload(b *testing.B) { // 细则1
    payload := []byte(`{"id":"u-1","name":"alice"}`) // 细则2
    b.ResetTimer()
    for i := 0; i < b.N; i++ {
        _, _ = ParsePayload(payload)
    }
}

    说明：基准测试应固定输入数据并减少外部依赖，避免噪声影响性能结论。

    12. 性能与内存¶

    • 【推荐】热路径避免反射和频繁分配。
    • 【推荐】切片容量可预估时预分配。
    • 【推荐】对象池仅用于高频对象且确认收益。
    • 【强制】性能优化前先度量（pprof、benchmark）。
    // 示例
func BuildIDs(users []User) []string {
    ids := make([]string, 0, len(users)) // 细则2
    for _, u := range users {
        ids = append(ids, u.ID)
    }
    return ids
}
// 细则1：避免反射
// 细则3：对象池需压测后启用
// 细则4：先 benchmark/pprof 再改

    // 反例：未预分配，append 多次扩容
func BuildIDsBad(users []User) []string {
    var ids []string
    for _, u := range users {
        ids = append(ids, u.ID)
    }
    return ids
}

    说明：热路径预分配容量是低成本优化，能减少 GC 压力和内存拷贝。

    12.1 常见性能误区¶

    • 在热路径频繁使用 fmt.Sprintf。
    • 未预分配切片容量导致反复扩容。
    • 在高频路径使用反射处理通用逻辑。
    // 示例（反例）
func badLog(id int) string { return fmt.Sprintf("id=%d", id) } // 细则1
func badIDs(users []User) []string { var ids []string; for _, u := range users { ids = append(ids, u.ID) }; return ids } // 细则2
// 反例：热路径反射
func badField(v any, name string) any { return reflect.ValueOf(v).FieldByName(name).Interface() } // 细则3

    13. 安全规范¶

    • 【强制】输入校验（长度、格式、范围、白名单）。
    • 【强制】密钥和配置走环境变量或密管系统。
    • 【强制】HTTP 服务设置读写超时与空闲超时。
    • 【推荐】依赖库定期执行漏洞扫描。
    // 示例
func validateUserName(name string) error { // 细则1
    if len(name) < 2 || len(name) > 32 {
        return errors.New("invalid name length")
    }
    if !regexp.MustCompile(`^[a-zA-Z0-9_]+$`).MatchString(name) {
        return errors.New("invalid name format")
    }
    return nil
}
dbPassword := os.Getenv("DB_PASSWORD") // 细则2
_ = dbPassword
_ = &http.Server{ReadTimeout: 5 * time.Second, IdleTimeout: 60 * time.Second} // 细则3
// 细则4：CI 执行 govulncheck ./...

    说明：输入校验建议采用“长度 -> 字符集 -> 业务规则”的分层顺序，失败信息可审计但不泄露内部实现。

    13.1 HTTP 安全建议¶

    • 【强制】设置读写超时与最大请求体大小。
    • 【推荐】统一鉴权中间件并进行最小权限控制。
    • 【推荐】对外错误响应不暴露堆栈和内部路径。
    // 示例
srv := &http.Server{ // 细则1
    Addr:              ":8080",
    ReadHeaderTimeout: 3 * time.Second,
    ReadTimeout:       5 * time.Second,
    WriteTimeout:      10 * time.Second,
    IdleTimeout:       60 * time.Second,
    Handler:           http.MaxBytesHandler(router, 1<<20),
}
_ = AuthzMiddleware(router) // 细则2
http.Error(w, "internal error", http.StatusInternalServerError) // 细则3
_ = srv

    说明：超时与请求体大小限制是最基础的抗滥用手段，默认应在网关和应用层双重启用。

    14. 注释与文档¶

    • 【强制】导出符号必须有注释，且以符号名开头。
    • 【推荐】注释解释意图与约束，不翻译代码。
    • 【推荐】复杂并发或业务边界必须有设计说明。
    // 示例
// UserService 提供用户读写服务。 // 细则1
// 约束：所有写操作必须在事务中完成。 // 细则2
// 并发说明：异步任务必须绑定 ctx 取消。 // 细则3
type UserService struct{}

    说明：注释要解释“为什么有这个约束”，而不是重复字段和流程。

    15. 代码评审清单¶

    • 是否通过 gofmt/go test？
    • 错误是否可追踪、可判定、可恢复？
    • 并发是否存在泄漏、死锁或竞态？
    • 导出接口是否最小且稳定？
    • 是否覆盖关键边界与失败路径？
    • 是否存在资源泄漏（连接、文件、goroutine）？
    • 是否存在过大 API 面导致可维护性问题？

- [ ] 细则1：`gofmt` + `go test ./...` 通过

- [ ] 细则6：连接/文件/goroutine 有释放，并发逻辑有退出路径
