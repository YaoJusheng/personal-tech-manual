    一、背景描述¶

    团队之间的 PR（Pull Request）审查是保证代码质量、促进团队协作、提升开发效率的关键环节。制定清晰的审查规范和流程有助于提高代码的可维护性，减少错误，并确保代码符合团队的最佳实践和技术要求。

    二、PR 提交前的准备¶

    2.1.编写清晰的 PR 描述¶

    提交 PR 时，确保 PR 描述简洁明了。描述应包括：

    • 变更的目的：为什么要进行这次修改？解决了什么问题？
    • 实现方式：修改的具体内容和实现方式。
    • 关联的任务或 bug：PR 是否与特定的任务、bug、需求相关联？如果有，提供相关链接（如 Jira 或 GitHub Issue）。
    • 测试覆盖：说明是否已经进行了单元测试或集成测试，或者如何验证修改是否有效。

    2.2.分支管理¶

    • 确保 PR 是基于主干分支（如 main 或 develop）的最新版本创建的，以避免与主分支的冲突。
    • 分支命名应具有描述性，便于理解所作的更改。例如：
    • hotfix/<authen>-<task_id>
    • feat/<authen>-<task_id>
    • task/<authen>-<task_id>
    • release/X.Y.Z

    其中：

    • authen： 表示 提交者 的身份说明 或 项目代号， 比如：AI、BE、FE或 Cloud、Dataset、Train.
    • task_id: 表示 在项目管理平台上 建立任务的唯一 ID号。
    • X.Y.Z: 代表 SemVer 规范中的 MAJOR.MINOR.PATCH。

    带背景颜色的为推荐选项。

    2.3.测试¶

    • 提交PR 前，确保代码通过了单元测试、集成测试以及任何团队规定的其他自动化测试。
    • 如果是 UI或功能的更改，确保通过了相关的 UI 测试或手动验证。

    三、PR 审查流程¶

    3.1.自动化检查（CI/CD）¶

    • 在 PR 提交后，自动化构建、测试和静态分析工具（如 CI/CD 流程）应自动运行，确保提交不会引入破坏性的代码或错误。
    • 代码风格检查（如 flake8 或 black）
    • 单元测试和集成测试
    • 安全性扫描（如 bandit）
    • 依赖检查（如 safety）

    审查者应根据 CI/CD 提供的反馈，确保所有检查项通过。

    Example：

    # .github/workflows/pr-check.yml
jobs:
  validation:
    steps:
      - name: 代码格式化检查
        run: black --check .

      - name: 安全扫描
        uses: sast-scan@v2
        with:
          rules: security-rules.yml

      - name: 单元测试
        run: pytest --cov=src --cov-fail-under=80

      - name: 构建验证
        run: docker build -t app:pr-${{ github.run_id }} .

    3.2.审查者选择¶

    • 分配审查者：通常，PR 提交者的团队负责人或代码库的维护者会指定一到两位审查者。如果有特定领域的专家，他们也可以作为审查者。
    • 轮流审查：审查者应轮流承担审查责任，避免重复审查。

    3.3.代码审查¶

    1.审查标准¶

    1.1.明确目标¶
    • 代码审查不仅是为了找 bug，还要确保代码设计和可读性。
    1.2 小且频繁的 PR¶
    • 确保 PR 改动尽量小，过大的 PR 可能导致审查难度增加。
    1.3 自动化检查¶
    • 使用工具（如 lint、CI）自动化检测代码格式和基础错误。
    1.4.审查清单¶
    • 代码质量
    • 符合编码规范 (Black/ESLint 报告通过)
    • 无重复代码 (SonarQube重复率<5%)
    • 功能正确性
    • 关联JIRA需求验证通过
    • 边界条件处理完备
    • 安全合规
    • 无敏感信息硬编码
    • SQL注入/XSS防护措施
    • 代码可读性
    • 代码应清晰易懂，变量名和函数名要有描述性。
    • 保持代码简洁，避免复杂的嵌套逻辑或长函数。
    • 代码中的注释应简洁明了，必要时可以解释复杂的实现。
    • 可维护性
    • 关键代码添加注释
    • 更新相关文档
    • 性能考虑
    • 对性能敏感的代码，检查是否考虑了性能优化。例如，避免不必要的数据库查询、避免低效的循环等。
    • 大型的性能优化更改应该由团队中的性能专家进行审查

    2.1. 精简状态标签体系¶
    短标签 颜色 适用阶段 关键规则说明
    WIP 🔵 蓝 开发中（预创建Work in Progress） 未完成，禁止合并（PR 尚未准备好进行审查，正在开发中）
    TEST 🟢 绿 开发者自测 可在描述中附加测试截图/日志
    QA 🟡 黄 QA测试阶段 (Quality Assurance) 自动触发测试环境部署
    PROD-ACK 🟣 紫 产品经理验收 需@产品负责人确认需求匹配
    REV 🟡 黄 审查中（Review） PR 正在等待审查或审查者正在进行中。
    READY 🟢 绿 待合并 通过所有检查，所有问题解决，PR 准备好合并。

    BLOCK ⚫ 黑 阻塞状态 需在评论中注明阻塞原因
    DEAD ⚪ 白 废弃PR 超7天未更新自动标记

    3.4.发布与标签管理¶

    ​ 在开发项目的版本管理中，发布和标签管理（Release & Tag Management）是确保代码库有序、可追溯的关键环节。合理的标签和版本发布流程可以帮助团队更好地控制发布过程，确保稳定版本的追踪与管理。

    1. 标签的命名规范¶

    标签（Tag）是 Git 中用于标记特定版本的工具，通常用于标识一个发布版本。以下是一些推荐的标签命名规范：

    • 主版本号: vX.Y.Z-TAG.N（符合 语义版本控制 规范）
    • X（MAJOR）：主版本号，发生不兼容的 API 修改时增加。
    • Y（MINOR）：次版本号，添加功能时增加，且向后（下）兼容。
    • Z（PATCH）：修订号，为补丁版本，进行向后（下）兼容的 bug 修复时增加。
    • TAG.N：预发布版本，alpha/beta/rc阶段测试，如：v1.5.0-rc.2。
    • 标签类型：
    标签类型 格式 示例 创建场景
    正式发布 vMAJOR.MINOR.PATCH v2.3.1 生产环境发布
    预发布 vMAJOR.MINOR.PATCH-TAG.N v1.5.0-beta.1 测试验证阶段
    热修复 vMAJOR.MINOR.PATCH+hotfix.N v3.2.1+hotfix.1 紧急问题修复

    示例：

    • v1.0.0: 初始稳定版本
    • v1.1.0: 添加新功能，不破坏现有功能
    • v1.0.1: 修复小 bug
    • v2.0.0: 不兼容的重大改变
    • v1.0.0-alpha: Alpha 版本，开发阶段
    • v1.0.0-beta: Beta 版本，进行 QA 测试阶段
    • v1.0.0-rc: Release Candidate（候选版本），接近稳定版的版本

    2. 发布与标签管理流程¶

    a. 开发与标记流程¶
    1. 开发完成后：每次开发完一个新功能或修复完 bug，都会在 Git 中创建一个新的提交或分支，并进行单元测试或集成测试。
    2. 提交与合并：开发分支通过拉取请求（PR）合并到主分支（如 main 或 master）。所有变更都应通过代码审查和自动化测试验证。
    3. 创建 Release 标签：在主分支上完成发布前，创建一个新的 Git 标签（Tag）。标签应遵循语义化版本控制的规范。
    b. 版本发布管理¶

    参照标签类型

    1. 开发阶段：
    2. 使用 dev 或 alpha 标签标识开发版本。
    3. 示例：v1.0.0-dev、v1.0.0-alpha。
    4. 测试阶段：
    5. 在完成开发后，进行 QA 测试时，可以使用 beta 标签。
    6. 示例：v1.0.0-beta。
    7. 稳定版本发布：
    8. 当功能冻结且经过充分的测试后，发布稳定版本，创建 vX.Y.Z 标签。
    9. 示例：v1.0.0。
    10. 分支-标签联动策略
    分支类型 允许创建的标签类型 保护规则
    main 仅vX.Y.Z正式标签 需2人审核+CI全通过
    release/* vX.Y.Z-rc.N预发布标签 禁止强制推送
    hotfix/* vX.Y.Z+hotfix.N热修复标签 自动同步到所有旧版本分支
    1. 操作规范:
    # 创建附注标签（推荐）
git tag -a v1.2.3 -m "Release v1.2.3: 新增支付接口"

# 推送标签到远程
git push origin v1.2.3

# 删除错误标签
git tag -d v1.2.3 && git push origin :refs/tags/v1.2.3

# 查看标签关联内容
git show v1.2.3

# 验证标签签名
git tag -v v1.2.3
    c. 持续集成与发布¶
    1. 自动化发布流程：
    2. 通过 CI/CD 工具（如 GitHub Actions、GitLab CI、Jenkins 等）自动化版本号的管理和发布。每次 Git 标签创建时，自动执行构建、测试、打包和部署等流程。
    3. 在每次推送到 main 或 master 分支时，CI/CD 管道可以根据 Git 标签自动发布新版本。
    d.发布日志（Release Notes）¶

    每次发布版本时，应附带详细的发布日志。发布日志是记录版本更新内容的文档，帮助开发者和用户理解新版本的功能、修复以及潜在的 breaking change。

    • 发布日志的内容应包括：
    • 版本号：例如 v1.0.0
    • 发布日期：例如 2025-02-25
    • 更新内容：
      • 新功能：新增的功能或特性。
      • 改动：功能的改动或优化。
      • 修复：解决的问题、bug 修复等。
      • Breaking Changes：不兼容的更改，需要用户注意更新。
    • 示例：
    # v1.0.0 - 2025-02-25

### 新功能：
- 新增用户认证模块。
- 支持基于角色的权限管理。

### 修复：
- 修复了在特定条件下登录失败的问题。
- 优化了数据库查询的性能。

### Breaking Changes：
- `User` 类的 `is_active` 字段已从 `boolean` 改为 `enum` 类型，需更新数据库。

    发布日志可以手动维护，也可以通过 GitHub Releases 等工具自动生成。

    e. 最佳实践示例¶
    1. 版本升级场景
    2. 功能开发完成 → 创建 release/1.5.0 分支
    3. 测试通过 → 打标签 v1.5.0-rc.1 → 部署预发布环境
    4. 验收通过 → 合并到 main → 打正式标签 v1.5.0
    5. 紧急修复场景
    6. 从 v1.4.2 创建 hotfix/payment-bug
    7. 修复后打标签 v1.4.2+hotfix.1 → 同时合并到 main和release/*

    四、工具支持¶

    • GitHub、GitLab 和 Bitbucket 等 Git 平台通常提供丰富的 PR 审查工具，如自动化检查、评论、审批、合并策略等功能。
    • CI/CD 工具（如 Jenkins、CircleCI、Travis CI）能够自动触发构建、测试和检查，确保 PR 在合并前不会破坏代码。
    • SonarQube、CodeClimate 等工具帮助进行代码质量检查，自动检查代码中的潜在问题。
