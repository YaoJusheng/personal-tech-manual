    可以每一条规范前加上【强制】或【推荐】、【可选】的标签，让其他成员更有针对性地遵守规范内容

    1.基础规则与约定¶

    PEP 8 遵循：严格遵循 PEP 8 风格指南。诸如以下基础规则：

    • 【强制】使用 4 个空格缩进。
    • 【强制】每行代码长度不超过 79 (有些时候也可以放宽为120) 个字符。
    • 【强制】使用空行分隔逻辑块， 函数和类之间用两个空行分隔，方法之间用一个空行分隔。
    • 【强制】函数的命名使用小写字母和下划线分隔（snake_case）。
    • 【强制】类名使用驼峰命名法（CamelCase）。
    • ...

    2 命名规约¶

    1. 【强制】变量命名

    使用小写字母和下划线分隔（snake_case）。

    my_variable = 10
    1. 【推荐】常量命名

    使用大写字母和下划线分隔（SNAKE_CASE）。

    MY_CONSTANT = 42
    1. 【强制】函数命名

    使用小写字母和下划线分隔（snake_case）。

    顶级定义之间空两行, 方法定义之间空一行。

    def my_function1():
    pass


 def my_function2():
    pass
    1. 【强制】类命名

    使用驼峰命名法（CamelCase）。

    顶级定义之间空两行, 比如函数或者类定义. 方法定义, 类定义与第一个方法之间, 都应该空一行. 函数或方法中, 某些地方要是你觉得合适, 就空一行.

    class MyClass:

    def cls_func1(self):
        pass

    def cls_func2(self):
        pass

    3.常量定义¶

    • 【推荐】集中定义：将所有常量定义在一个模块中，如 constants.py。

    使用全大写命名，必须定义在模块顶部或专用配置类中。

    禁止魔法数值，需用常量替代。

    # constants.py
MY_CONSTANT = 42
ANOTHER_CONSTANT = "value"
    • 使用常量：通过模块导入常量。
    from constants import MY_CONSTANT

    4.控制语句¶

    • 【推荐】避免复杂条件嵌套（超过3层需重构）。

    • 使用 isinstance 做类型判断，禁用 type()。

    • if-else：使用清晰的条件语句。

    if condition:
    # do something
else:
    # do something else

# Good
if isinstance(value, (int, float)) and 0 < value < 100: ...

# Bad
if type(value) == int: ...
    1. for 循环：使用 enumerate 来同时获取索引和值。
    for index, value in enumerate(my_list):
    print(index, value)

    5.集合处理¶

    1. 【推荐】不可变性优先

    2. 只读场景使用 Tuple 替代 List。

    VALID_STATUSES = ("running", "stopped")

    1. 深拷贝与浅拷贝

    2. 修改嵌套集合时使用 copy.deepcopy。

    import copy
new_config = copy.deepcopy(DEFAULT_CONFIG)

    1. 【推荐】推导式优化

    2. 简单逻辑优先使用推导式，但避免嵌套超过2层。

    # 列表推导式(list)
squares = [x**2 for x in range(10) if x % 2 == 0]

# 集合推导式(set)
sets = {x**2 for x in range(10) if x % 2 == 0}

# 字典推导式(dict)
mapping = {x: x**2 for x in range(10) if x % 2 == 0}

    6.线程安全与线程池¶

    1. 线程安全

    使用锁：在多线程环境中，使用 threading.Lock 来确保线程安全。

    • 【强制】共享资源必须加锁：
    from threading import Lock
counter = 0
counter_lock = Lock()

def increment():
    global counter
    with counter_lock:
        counter += 1
    • 【强制】避免可变对象默认参数：
    # Bad: 默认列表在多次调用中共享
def process(items=[]): ...

# Good
def process(items=None):
    items = items or []

    1. 线程池

    2. 【推荐】使用线程池：使用 concurrent.futures.ThreadPoolExecutor 来管理线程。

    3. 【强制】禁止直接创建裸线程（threading.Thread）。
    from concurrent.futures import ThreadPoolExecutor

with ThreadPoolExecutor(max_workers=4) as executor:
    futures = [executor.submit(task, arg) for arg in args]
    results = [f.result() for f in futures]

    7.注释规范¶

    根据 PEP-484 来对 python3 代码进行注释,并使用诸如 pytype 之类的类型检查工具来检查代码.

    7.1.【推荐】Docstring¶

    • 【推荐】文档字符串：每个模块、类和函数都应有清晰的文档字符串。
    • 【推荐】公共函数/类 必须包含Google风格Docstring
    • 行内注释：在复杂的逻辑后添加简短的行内注释。
    def calculate_accuracy(predictions, labels):
    """Calculate the accuracy of model prediction.

    Args:
        predictions (np.ndarray): Model prediction results，shape(N,)
        labels (np.ndarray): Real tags, shape(N,)

    Returns:
        float: Accuracy, range[0.0, 1.0]
    """
    pass


x = x + 1  # Increment x

    7.2.【强制】类型注释¶

    1. 基本的类型注释

    每个函数参数应该有明确的类型标注。类型注释通常位于参数名称后，并通过冒号 : 来分隔。

    def greet(name: str, age: int) -> None:
    print(f"Hello, {name}. You are {age} years old.")
    • name: str 表示 name 参数是一个字符串。
    • age: int 表示 age 参数是一个整数。

    • -> None 表示该函数没有返回值。

    • 常见类型注释

    • 基本类型

    • int：表示整数类型。

    • float：表示浮动小数类型。

    • bool：表示布尔值类型。

    • str：表示字符串类型。

    • None：表示函数没有返回值。

      def multiply(x: int, y: int) -> int:
    return x * y

    • 复合类型

    • list：表示列表类型。

    • tuple：表示元组类型。

    • dict：表示字典类型。

    • set：表示集合类型。

      def process_list(data: list[int]) -> int:
    return sum(data)


def get_coordinates() -> tuple[float, float]:
    return (1.0, 2.0)

    • Union 类型：表示一个变量或返回值可以是多个类型之一。需要导入 typing 模块中的 Union 类。

      from typing import Union

def get_value(value: Union[int, str]) -> str:
    return str(value)

    • Optional 类型：是 Union[X, None] 的简写，表示值可以是类型 X 或者 None。

      from typing import Optional

def find_item(name: str) -> Optional[str]:
    # 如果找不到则返回 None
    return None

    • 其他参考typing模块

    • 类型注释的规范要求

    • 类型注释的位置

    • 对于函数参数和返回类型，应该在函数签名中明确标注。

    • 类型注释应当位于函数签名的最后，而不是在参数旁边。

    • 应该为所有公开函数提供类型注释，尤其是库和框架代码。

      def calculate_area(radius: float) -> float:  # 标准
    return 3.14 * radius ** 2

    • 清晰性优先

    • 如果类型复杂，考虑将其拆分成多个简单的类型别名（Type Alias）。

    • 保持类型注释简洁且直观，避免过于复杂的类型链。

    • 统一规范

    • 应该在项目中统一使用类型注释风格。例如，使用 list[int] 和 tuple[str, str] 而不是 List[int] 和 Tuple[str, str]（除非需要兼容较旧的 Python 版本）。从 Python 3.9 开始，推荐直接使用内置的容器类型（list、dict、tuple）进行类型注释，因为它们更加简洁且与 Python 的内置类型一致。

    • 避免在类型注释中使用 Any，除非无法推测类型。若使用 Any，则应注释说明。

    • 使用 mypy 或 Pyright 等工具检查类型

    pip install mypy

mypy your_script.py

    • 避免过度使用复杂的类型注释

    • 对于过于复杂或深层嵌套的类型，尽量避免冗长的注释，采用类型别名来提高可读性。

      from typing import List, Dict

# 不推荐
def process_data(data: Dict[str, Dict[str, List[int]]]) -> None:
    pass


# 推荐
DataType = Dict[str, Dict[str, List[int]]]

def process_data(data: DataType) -> None:
    pass

    8.异常处理¶

    • 捕获具体异常类型，向上抛出自定义异常

    • try-except：使用 try-except 块处理可能的错误，避免使用裸 except。

    • 捕获所有异常时，应该避免使用裸 except（例如：except:），它会捕获所有异常类型，包括系统退出和键盘中断等。

    • 推荐捕获特定类型的异常，或者使用 except Exception as e 来捕获所有常规异常。

    • 可以使用多个 except 来处理不同类型的异常，避免一个 except 捕获多种类型的异常。

    • 使用 elif 和 else 可以进一步控制异常流程。

    • else 块会在 try 块没有抛出异常时执行，它可以用来放置没有异常时应该执行的代码。

    • finally 块会在 try / except 语句后无论是否发生异常都会执行，通常用来进行资源释放或清理工作。

      # 不推荐
try:  
    # Some risky code
except:  
    # Too broad, catching everything including system exit or keyboard interrupts


# 推荐
try:
    result = 10 / 0
except ZeroDivisionError as e:
    print(f"Error: {e}")
    raise e


try:
    file = open("data.txt", "r")
except FileNotFoundError:
    print("File not found!")
else:
    data = file.read()
    print(data)
finally:
    file.close()

    • 使用 raise from 进行异常链的传递

    • Python 3 引入了 raise from 语法，它可以将一个异常引发的上下文传递给另一个异常。这有助于更好地捕获和调试异常链。

    • 通过异常链，可以保留原始异常的上下文信息。

      try:
    value = int("not_a_number")
except ValueError as e:
    raise CustomError("Custom error occurred") from e

    • 日志记录：使用 logging 模块记录错误和调试信息。

    • 使用 exc_info=True 参数来记录异常堆栈跟踪信息，它可以在日志中详细记录异常的上下文，帮助开发人员定位问题。

      import logging

logging.basicConfig(level=logging.DEBUG)
logging.debug("This is a debug message")

try:
    some_function()
except Exception as e:
    logging.error("Error in function execution", exc_info=True)

    10.依赖管理工具¶

    10.1.常用工具¶

    1. Conda

    • 功能：环境隔离完善，支持非 Python 包，适合数据科学项目。
    • 优点：强大的环境管理能力，适合需要多种依赖（如 NumPy、TensorFlow）的项目。
    • 局限性：环境迁移复杂，资源占用较大, 依赖解析速度较慢。
    • 商业用途许可问题
    • Anaconda 和 Miniconda 的许可限制：
      • Anaconda 和 Miniconda 是商业性质的产品，其许可协议明确禁止在未获得商业许可的情况下用于商业用途。
      • 自 2020 年 4 月起，Anaconda 更改了许可条款，明确禁止未经许可的商业使用。
      • 对于拥有 200 名或以上员工的组织，使用 Anaconda 产品需要购买商业版或企业版许可证。
    • 适用场景：机器学习和数据科学项目。
    • 推荐值：⭐⭐⭐⭐

    2. Pip-tools

    • 功能：依赖锁定精确，编译过程可控，团队协作友好。
    • 优点：精确控制依赖版本，适合中大型项目。
    • 局限性：需要额外工具链，学习成本略高。
    • 适用场景：中大型 Web 项目（如 Django、Flask）。
    • 推荐值：⭐⭐⭐⭐

    3. Pipenv

    • 功能：自动创建虚拟环境并管理依赖（Pipfile + Pipfile.lock）。
    • 优点：依赖管理规范，适合现代 Python 项目。
    • 局限性：Windows 安装较复杂，对大型项目性能一般。
    • 适用场景：全平台新手和团队协作项目。
    • 推荐值：⭐⭐⭐⭐

    4. Poetry

    • 功能：官方维护，专注于依赖管理和打包，生成 pyproject.toml 配置文件。
    • 优点：与 Python 生态深度集成，支持构建分发包。
    • 局限性：学习曲线较陡，命令行选项较多。
    • 适用场景：全平台现代化 Python 项目开发。
    • 推荐值：⭐⭐⭐⭐⭐

    5. Mamba

    • 功能：Conda 的改进版，用 C++ 实现，解决 Conda 的依赖解析和并行下载问题。
    • 优点：速度比 Conda 快很多，其他方面与 Conda 类似。
    • 局限性：与 Conda 兼容性有限。
    • 适用场景：需要快速依赖解析的项目。
    • 推荐值：⭐⭐⭐⭐

    6. Pixi

    • 功能：类似于 uv，针对 Conda 生态系统，用 Rust 编写，支持多平台。
    • 优点：速度快，单二进制文件，无外部依赖。
    • 局限性：与其他工具兼容性有限。
    • 适用场景：需要管理多个虚拟环境和定义文件的项目。
    • 推荐值：⭐⭐⭐⭐

    7. PDM

    • 功能：类似于 Poetry，但遵循 PEP 标准，可使用 uv 进行依赖解析和安装。
    • 优点：遵循 PEP 标准，可利用 uv 进行依赖管理。
    • 局限性：与 Poetry 类似，是 Python 工具，有较多依赖。
    • 适用场景：中大型 Python 项目。
    • 推荐值：⭐⭐⭐⭐⭐

    8. UV

    • 功能：一个用 Rust 编写的极快的 Python 包和项目管理器。
    • 优点：速度飞快，是普通pip的10-100倍， 性能与安全性方面较为优秀， 且提供跨平台支持。
    • 局限性：
    • 不支持虚拟环境管理： UV 目前不提供虚拟环境管理功能，这意味着用户需要结合其他工具（如 venv 或 virtualenv）来管理虚拟环境。
    • 不支持维护多个互斥环境： UV 无法像 pipx 那样维护多个互斥的环境，这在某些复杂项目中可能会是一个限制。
    • 社区活跃度较低： UV 的社区相对较小，活跃度不如 Poetry 或其他成熟工具。这意味着在遇到问题时，可能较难找到社区支持。
    • 适用场景：
    • 需要快速安装和管理依赖的开发者：UV 的并行下载和编译功能可以显著提升开发效率。
    • 希望简化 Python 工具链的用户：UV 集成了包管理、虚拟环境管理、脚本运行和 Python 版本管理等功能，简化了工具链。
    • 需要高性能依赖解析和包安装的大型项目：UV 的高性能依赖解析和包安装功能使其成为大型项目的理想选择。
    • 需要跨平台支持的项目：UV 支持 macOS、Linux 和 Windows，适合跨平台开发。
    • 推荐值：⭐⭐⭐⭐⭐

    10.2.最佳方案与工具选型分析¶

    1.核心选型维度¶

    维度 关键考虑点
    开源友好 工具许可证是否允许自由使用（MIT/BSD > Apache > GPL）
    商业合规 是否支持私有仓库、依赖来源审计、许可证白名单
    性能 依赖解析速度、缓存机制、安装并行度
    安全性 漏洞扫描（CVE）、依赖来源可信度（PyPI vs 自建源）、依赖哈希校验
    易用性 开发者上手成本、IDE支持、文档完善度

    2.主流工具比较¶

    2.1. 基础工具：pip + requirements.txt¶
    • 适用场景：
    • 小型项目、快速原型开发
    • 对依赖管理要求简单的场景
    • 优点：
    • Python官方原生支持，无需额外工具
    • 开源和商业项目通用
    • 缺点：
    • 无依赖锁定机制（需手动生成 requirements.txt）
    • 无法处理依赖冲突（需人工干预）
    • 安全性增强方案：
    # 依赖哈希校验（pip 8+）
pip install --require-hashes -r requirements.txt

# 漏洞扫描
pip install pip-audit
pip-audit
    2.2. 企业级工具：pip-tools + pipenv¶
    • 适用场景：
    • 需要兼容遗留项目（requirements.txt 模式）
    • 企业内网环境（需自建私有源）
    • 优点：
    • pip-tools 提供确定性依赖（pip-compile 生成 requirements.txt）
    • pipenv 结合虚拟环境管理（Pipfile + Pipfile.lock）
    • 支持Hash校验（pipenv check）
    • 缺点：
    • 工具链较复杂（需同时维护多个文件）
    • pipenv 性能问题（依赖解析速度慢）
    • 安全合规实践：
    # 生成带哈希的requirements.txt
pip-compile --generate-hashes requirements.in

# 检查许可证合规性
pip-licenses --format=json | jq '.[].License'
    2.3.科学计算场景：conda/mamba¶
    • 适用场景：
    • 依赖复杂科学计算库（NumPy/PyTorch）
    • 跨平台环境（需处理非Python依赖，如CUDA、MKL）
    • 优点：
    • 二进制依赖管理（避免源码编译）
    • 支持环境隔离（conda environments）
    • mamba 加速依赖解析（C++实现）
    • 缺点：
    • 商业使用需注意Anaconda仓库协议（2020年后企业用户需付费）
    • 镜像源同步延迟较高
    • 企业替代方案：
    • 自建Conda仓库（Artifactory + conda-mirror）
    • 使用社区版Miniconda + 替换默认源（如清华源）
    2.4.开源项目¶
    • 推荐工具：poetry
    • 理由：
    • 统一管理依赖与打包流程，简化贡献者上手成本
    • pyproject.toml 已成为PEP 621标准，未来兼容性好
    • 自动生成优雅的依赖声明（支持版本范围与排除冲突）
    • 典型配置：
    [tool.poetry]
name = "myproject"
version = "0.1.0"

[tool.poetry.dependencies]
python = "^3.8"
requests = { version = "^2.28", extras = ["security"] }
pandas = ">=1.5,<2.0"

[tool.poetry.group.dev.dependencies]
pytest = "^7.2"
black = "^23.3"

[build-system]
requires = ["poetry-core>=1.0.0"]
build-backend = "poetry.core.masonry.api"
    2.5.商业项目¶
    • 推荐组合：pip-tools + pip-audit + 私有仓库
    • 理由：
    • 完全控制依赖来源（仅允许从企业私有PyPI安装）
    • 通过 pip-compile 生成确定性的 requirements.txt，便于审计
    • pip-audit 集成CI/CD，阻断含CVE漏洞的依赖
    • 安全增强措施：
    • 依赖来源控制：
    • # pip.conf[global] index-url = http://internal-pypi.example.com/simple/ trusted-host = internal-pypi.example.com
    • 许可证白名单：
    • # 使用pip-licenses检查禁止的许可证（如GPL） pip-licenses | grep "GPL" && exit 1

    10.3.最终方案¶

    出于性能、安全和统一风格的考虑， 建议： uv + pip-audit + ruff +nuitka。

    其中：

    • uv： 依赖管理工具。
    • ruff：综合性的静态代码检查和代码风格格式化工具。
    • pip-audit：漏洞扫描。
    • nuitka：编译打包工具。
