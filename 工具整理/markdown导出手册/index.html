    一、需求拆解与约束¶

    当前目标可以拆成 4 个硬性要求：

    1. 文档源是一个包含多级嵌套目录的大型 Markdown 知识库。
    2. 需要在 PC 和平板上都能顺畅浏览，具备目录导航和全文检索能力。
    3. 不允许修改原始文档文件（零侵入）。
    4. 要有可落地的实施步骤、示例和后续维护方式。

    从工程视角，最终方案应满足下面的评估标准：

    • 零侵入：站点配置和构建产物必须放在源目录外。
    • 可读性：响应式布局、移动端导航、代码高亮、目录层级清晰。
    • 可维护性：新增文档后无需大规模改动配置。
    • 可分发性：可本地离线访问，也可部署到静态托管平台。

    二、方案对比分析¶

    方案 实现方式 优点 局限 适配结论
    1. 静态站点生成器（MkDocs / VitePress / Docusaurus） 在源目录外创建站点工程，读取原 Markdown 并构建静态站点 响应式好、搜索强、导航清晰、链接保真、可自动化部署 需安装 Python 或 Node 环境 最优，推荐
    2. HTML手册化导出（mdBook/脚本合并） 用 mdBook 生成手册站点，或将多个 Markdown 聚合为单文件 HTML 分发简单，可离线 需要维护目录文件或合并脚本，超大目录整理成本较高 可作为补充，不建议主方案
    3. EPUB/PDF（Pandoc） 批量转电子书或 PDF 便于归档与传阅 检索和跳转体验弱，多级导航丢失明显 适合归档，不适合主阅读场景
    4. 编辑器导出（Typora/Obsidian Publish） 用 GUI 工具导出 上手快 可定制性弱，部分方案收费或依赖云服务 仅适合轻量个人使用

    结论：如果强调“零侵入 + 多端体验 + 可持续维护”，优先选择静态站点生成器。

    三、方案一：静态站点生成器（MkDocs / VitePress）¶

    示例分析¶

    场景：当前仓库文档层级深、文件数多，且要求“零侵入 + 检索 + 多端浏览”。

    • 选择 MkDocs：适合 Python 环境、配置简单、对中文文档体验稳定。
    • 选择 VitePress：适合 Node.js 团队、前端定制能力更强、扩展生态丰富。

    结论：该方案仍是主方案，优先级最高。

    为什么是它?:

    • 零侵入：通过 docs_dir 指向原目录，不改任何源文件。

    • 多端友好：Material 主题对平板/手机响应式成熟。

    • 搜索与导航完整：内置全文检索、目录树、页内目录。

    • 部署成本低：构建后是纯静态文件，可本地开服务或一键部署。

    • 学习成本低：配置文件单一，运维门槛低。

    落地流程 A：MkDocs 实现¶

    1. 目录规划

    2. 原始文档目录：/path/project-config-doc

    3. 站点工程目录：/path/project-config-site-mkdocs

    4. 安装依赖

    python -m pip install --upgrade pip
pip install mkdocs mkdocs-material
    1. 初始化并配置

    进入配置目录:

    mkdocs new project-config-site-mkdocs
cd project-config-site-mkdocs

    配置示例:

    site_name: 技术文档手册
docs_dir: ../project-config-doc
site_dir: site
theme:
    name: material
    language: zh
plugins:
    - search

    配置参考：MkDocs

    1. 启动与构建
    mkdocs serve
mkdocs build

    落地流程 B：VitePress 实现¶

    1. 目录规划

    2. 原始文档目录：/path/project-config-doc

    3. 站点工程目录：/path/project-config-site-vitepress

    4. 初始化依赖

    mkdir project-config-site-vitepress && cd project-config-site-vitepress
npm init -y
npm install -D vitepress
ln -s ../project-config-doc docs
mkdir -p docs/.vitepress
    1. 编写配置（docs/.vitepress/config.mts）
    import { defineConfig } from "vitepress";

export default defineConfig({
    title: "技术文档手册",
    description: "基于 Markdown 知识库生成的多端浏览手册",
    lang: "zh-CN",
    themeConfig: {
        search: { provider: "local" }
    }
});
    1. 添加脚本并运行
    {
    "scripts": {
        "docs:dev": "vitepress dev docs",
        "docs:build": "vitepress build docs",
        "docs:preview": "vitepress preview docs"
    }
}

    npm run docs:dev
npm run docs:build

    落地流程 C：发布到 GitHub Pages（完整步骤）¶

    本节以 MkDocs（Python 栈）和 VitePress（Node 栈）为例，详细说明从仓库配置到站点上线的全流程，适用于"文档仓库即源仓库"的最常见场景。

    C-1. 前置准备¶

    1. GitHub 仓库检查

    确认目标仓库已创建并包含 Markdown 文档，且你对该仓库拥有 Settings → Pages 的配置权限（Owner 或 Admin）。

    1. 启用 GitHub Pages

    2. 打开仓库 → Settings → Pages（左侧菜单）。

    3. 在 Source 下拉框中选择 GitHub Actions（推荐，不需要额外的 gh-pages 分支）。

    4. 保存后页面会显示 Pages 的目标 URL，格式通常为： https://<用户名>.github.io/<仓库名>/

    5. 仓库权限配置

    6. 打开 Settings → Actions → General。

    7. 将 Workflow permissions 设置为 Read and write permissions，并勾选 Allow GitHub Actions to create and approve pull requests。

    8. 这是 actions/deploy-pages 正常运行的必要条件。

    9. 本地验证构建通过

    发布前必须确保本地构建零报错：

    # MkDocs
mkdocs build --strict

# VitePress
npm run docs:build

    --strict 模式会将所有警告（如链接失效）升级为错误，提前暴露问题。

    C-2. MkDocs 发布到 GitHub Pages¶

    步骤 1：配置 mkdocs.yml

    site_url 是 GitHub Pages 正确解析相对路径的关键，必须填写：

    site_name: 技术文档手册
site_url: https://<用户名>.github.io/<仓库名>/
docs_dir: .
site_dir: site
theme:
    name: material
    language: zh
    features:
        - navigation.tabs
        - navigation.top
        - search.highlight
        - content.code.copy
plugins:
    - search:
        lang: zh
    - awesome-pages          # 自动读取目录，无需手写 nav（需安装 mkdocs-awesome-pages-plugin）
extra:
    social:
        - icon: fontawesome/brands/github
          link: https://github.com/<用户名>/<仓库名>

    安装额外插件（如使用 awesome-pages）：

    pip install mkdocs-awesome-pages-plugin

    步骤 2：创建 GitHub Actions 工作流文件:

    在文档仓库根目录创建 .github/workflows/deploy-pages.yml：

    name: Deploy Docs to GitHub Pages

on:
  push:
    branches:
      - main          # 推送到 main 分支时自动触发
  workflow_dispatch:  # 支持手动触发

# 授权 Actions 写入 Pages
permissions:
  contents: read
  pages: write
  id-token: write

# 同一时间只保留一个部署任务，防止并发冲突
concurrency:
  group: pages
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4
        with:
          fetch-depth: 0          # 拉取完整历史，用于 git-revision-date 插件

      - name: Setup Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.12"
          cache: pip              # 缓存 pip 依赖，加速后续构建

      - name: Install MkDocs and plugins
        run: |
          pip install --upgrade pip
          pip install \
            mkdocs \
            mkdocs-material \
            mkdocs-awesome-pages-plugin

      - name: Build site (strict mode)
        run: mkdocs build --strict --verbose

      - name: Setup Pages
        uses: actions/configure-pages@v5

      - name: Upload site artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: site              # mkdocs 默认构建输出目录

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4

    步骤 3：推送并观察结果:

    将工作流文件推送到 main 分支后：

    1. 打开仓库 → Actions 选项卡，可以看到 Deploy Docs to GitHub Pages 工作流正在运行。
    2. 等待 build 和 deploy 两个 Job 都变绿后，打开 Settings → Pages 中显示的 URL。
    3. 如果出现 404，等待 1～2 分钟让 CDN 缓存刷新。

    C-3. VitePress 发布到 GitHub Pages¶

    步骤 1：配置 base 路径

    VitePress 发布到子路径时，必须在 docs/.vitepress/config.mts 中设置 base，否则 CSS/JS 路径会 404：

    import { defineConfig } from "vitepress";

export default defineConfig({
    title: "技术文档手册",
    description: "基于 Markdown 知识库生成的多端浏览手册",
    lang: "zh-CN",
    base: "/<仓库名>/",              // 关键：与 GitHub Pages 子路径保持一致

    C-6. 常见发布问题排查¶

    现象 原因 解决方法
    Actions 报 403 / 权限不足 Workflow permissions 未开启写权限 Settings → Actions → General 设为 Read and write
    页面空白 / CSS 丢失 VitePress base 路径未配置 在 config.mts 添加 base: "/<仓库名>/"
    图片 404 图片引用使用了绝对路径 改为相对路径或放到 public/ 目录
    中文路径 404 URL 编码问题 文件名避免使用中文，或在 MkDocs 开启 use_directory_urls: false
    推送后站点未更新 浏览器或 CDN 缓存 强制刷新（Ctrl+Shift+R），或等待 5 分钟
    build 步骤失败但本地正常 CI 环境缺少依赖或路径差异 检查 requirements.txt 或 package.json 版本是否锁定
    deploy job 被跳过 needs: build 的 build job 失败 先修复 build 错误，build 成功后 deploy 才会执行

    1）一键构建脚本¶

    在站点工程目录分别放置 MkDocs 和 VitePress 的构建脚本，统一命名为 build.sh，开发者只需执行该脚本即可完成本地构建与预览。

    MkDocs 版本（build.sh）

    #!/usr/bin/env bash
# 文档站点一键构建脚本（MkDocs）
# 用法：bash build.sh [serve|build|clean]
set -euo pipefail

# ── 常量配置 ──────────────────────────────────────────────
SITE_DIR="site"
LOG_FILE="build.log"
TIMESTAMP=$(date '+%Y-%m-%d %H:%M:%S')

# ── 工具函数 ──────────────────────────────────────────────
log()  { echo "[${TIMESTAMP}] $*" | tee -a "${LOG_FILE}"; }
die()  { log "ERROR: $*"; exit 1; }

check_deps() {
    command -v python3 >/dev/null 2>&1 || die "未找到 python3，请先安装 Python 3.9+"
    command -v mkdocs  >/dev/null 2>&1 || {
        log "未找到 mkdocs，正在自动安装依赖..."
        pip install --quiet mkdocs mkdocs-material mkdocs-awesome-pages-plugin
    }
}

clean() {
    log "清理上次构建产物..."
    rm -rf "${SITE_DIR}"
    log "清理完成"
}

build() {
    log "开始构建站点（严格模式）..."
    mkdocs build --strict 2>&1 | tee -a "${LOG_FILE}"
    local exit_code=${PIPESTATUS[0]}
    if [[ ${exit_code} -ne 0 ]]; then
        die "构建失败，请检查 ${LOG_FILE} 中的详细错误"
    fi
    log "构建成功：$(pwd)/${SITE_DIR}"
    log "共生成页面数：$(find "${SITE_DIR}" -name '*.html' | wc -l | tr -d ' ')"
}

serve() {
    log "启动本地预览服务（http://127.0.0.1:8000）..."
    mkdocs serve --dev-addr 127.0.0.1:8000
}

# ── 入口 ─────────────────────────────────────────────────
check_deps

case "${1:-build}" in
    serve)  serve  ;;
    build)  clean && build ;;
    clean)  clean  ;;
    *)      die "未知参数: $1（可选: serve | build | clean）" ;;
esac

    VitePress 版本（build.sh）

    #!/usr/bin/env bash
# 文档站点一键构建脚本（VitePress）
# 用法：bash build.sh [dev|build|preview|clean]
set -euo pipefail

DIST_DIR="docs/.vitepress/dist"
LOG_FILE="build.log"
TIMESTAMP=$(date '+%Y-%m-%d %H:%M:%S')

log()  { echo "[${TIMESTAMP}] $*" | tee -a "${LOG_FILE}"; }
die()  { log "ERROR: $*"; exit 1; }

check_deps() {
    command -v node >/dev/null 2>&1 || die "未找到 node，请先安装 Node.js 18+"
    command -v npm  >/dev/null 2>&1 || die "未找到 npm"
    if [[ ! -d "node_modules" ]]; then
        log "node_modules 不存在，正在安装依赖..."
        npm ci --silent
    fi
}

clean() {
    log "清理构建产物..."
    rm -rf "${DIST_DIR}"
    log "清理完成"
}

build() {
    log "开始构建站点..."
    npm run docs:build 2>&1 | tee -a "${LOG_FILE}"
    local exit_code=${PIPESTATUS[0]}
    if [[ ${exit_code} -ne 0 ]]; then
        die "构建失败，请查看 ${LOG_FILE}"
    fi
    log "构建成功：$(pwd)/${DIST_DIR}"
    log "共生成文件数：$(find "${DIST_DIR}" -type f | wc -l | tr -d ' ')"
}

case "${1:-build}" in
    dev)     check_deps && npm run docs:dev ;;
    build)   check_deps && clean && build ;;
    preview) check_deps && npm run docs:preview ;;
    clean)   clean ;;
    *)       die "未知参数: $1（可选: dev | build | preview | clean）" ;;
esac

    使用方式：在站点工程目录执行 bash build.sh build 构建，bash build.sh serve 本地预览，bash build.sh clean 清理产物。

    plugins:
  - search
  - awesome-pages

  links:
    absolute_links: warn        # 绝对链接降级为警告

    # 安装 htmltest

    - name: 🔍 Check links

    八、常见问题与规避¶

    1. 图片不显示： 检查图片路径是否相对当前 Markdown 文件；避免绝对本地路径。

    2. 链接跳转异常： 优先使用相对路径链接，避免混用大小写或平台特定路径分隔符。

    3. 文件名含空格/特殊字符： 建议统一命名风格，减少 URL 编码带来的导航不一致。

    4. 首次构建慢： 大目录首次构建耗时正常；后续增量修改预览会明显更快。

    5. 平板打不开页面： 确认电脑防火墙放行端口，且设备在同一局域网。

    九、最终效果预期¶

    • PC 与平板均可获得一致的浏览体验。
    • 左侧目录树可反映多级文件夹结构。
    • 支持全文检索、页内目录和代码高亮。
    • 原始 Markdown 文件保持完全不变。
    • 可离线局域网访问，也可在线托管统一访问。

    十、建议实施顺序¶

    1. 先用自动导航快速跑通（不写 nav）。
    2. 试运行 1 周，收集高频访问路径。
    3. 再固化 nav，提升目录结构稳定性。
    4. 最后接入 CI 自动发布，降低维护成本。

    十二、GitHub Actions 自动发布模板（GitHub Pages）¶

    如果你希望每次提交后自动生成并发布手册，可在仓库添加：

    • .github/workflows/manual-pages.yml

    示例内容：

    name: Build and Deploy Manual

on:
  push:
    branches:
      - main
      - develop
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: pages
  cancel-in-progress: true

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.11"

      - name: Install dependencies
        run: |
          python -m pip install --upgrade pip
          pip install mkdocs mkdocs-material

      - name: Build site
        run: mkdocs build --strict

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: site

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4

    启用要点：

    1. 仓库 Settings -> Pages -> Source 选择 GitHub Actions。
    2. 确保 mkdocs.yml 与文档路径配置正确，且本地 mkdocs build 能成功。
    3. 推荐把构建失败视为质量门禁，持续修复坏链和非法路径。
