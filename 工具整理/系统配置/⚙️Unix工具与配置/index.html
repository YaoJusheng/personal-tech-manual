    一、Linux命令行工具¶

    1.命令行速记工具¶

    Cheat命令行速记工具

    1.1.安装¶

    # amd64
cd /tmp \
  && wget https://github.com/cheat/cheat/releases/download/4.4.2/cheat-linux-amd64.gz \
  && gunzip cheat-linux-amd64.gz \
  && chmod +x cheat-linux-amd64 \
  && sudo mv cheat-linux-amd64 /usr/local/bin/cheat \
  && cd -

# arm64
cd /tmp \
  && wget https://github.com/cheat/cheat/releases/download/4.4.2/cheat-linux-arm64.gz \
  && gunzip cheat-linux-arm64.gz \
  && chmod +x cheat-linux-arm64 \
  && sudo mv cheat-linux-arm64 /usr/local/bin/cheat \
  && cd -

    1.2.配置¶

    # 手动创建并初始化配置
$ mkdir -p ~/.config/cheat && cheat --init > ~/.config/cheat/conf.yml
$ mkdir -p ~/.config/cheat/cheatsheets/community
$ mkdir -p ~/.config/cheat/cheatsheets/personal

# 首次执行并初始化(自动创建目录配置)

    4.远程连接¶

    4.1.基本配置¶

    ssh_config 和 sshd_config 是SSH（Secure Shell）客户端和服务器端的配置文件，它们分别用于配置SSH客户端和服务器的行为。以下是它们的区别以及详细分析和使用配置：

    1. ssh_config（SSH客户端配置文件）¶
    • 作用：用于配置SSH客户端的行为，影响本地机器上使用ssh命令连接到远程服务器时的默认行为。
    • 默认位置：在Linux系统中，通常位于/etc/ssh/ssh_config，也可以在用户主目录下创建~/.ssh/config文件进行个性化配置。
    • 配置内容：
    • Host：定义主机别名，可以简化远程连接命令。
    • HostName：指定远程主机的实际IP地址或域名。
    • User：指定远程连接时使用的默认用户名。
    • Port：指定远程SSH服务的端口号。
    • IdentityFile：指定用于认证的私钥文件路径。
    • ForwardAgent：是否转发SSH代理身份认证信息。
    • ForwardX11：是否启用X11转发，用于图形界面应用程序的远程显示。
    • ServerAliveInterval：客户端到服务器的存活检测间隔时间，单位为秒，用于防止连接超时。

    使用配置示例：

    # 在用户主目录下创建或编辑.ssh/config文件
Host myserver
    HostName 192.168.1.100
    User myuser
    Port 22
    IdentityFile ~/.ssh/id_rsa
    ForwardAgent yes
    ForwardX11 yes
    ServerAliveInterval 60
    2. sshd_config（SSH服务器端配置文件）¶
    • 作用：用于配置SSH服务器的行为，影响远程客户端连接到当前主机时的认证和访问控制等。
    • 默认位置：在Linux系统中，通常位于/etc/ssh/sshd_config。
    • 配置内容：
    • Port：指定SSH服务监听的端口号，默认为22。
    • ListenAddress：指定SSH服务监听的IP地址，默认监听所有地址。
    • PermitRootLogin：是否允许root用户远程登录。
    • PasswordAuthentication：是否允许密码认证。
    • PubkeyAuthentication：是否允许公钥认证。
    • AuthorizedKeysFile：指定公钥文件的存储路径。
    • MaxSessions：最大允许的SSH会话数。
    • LoginGraceTime：用户登录时的宽限时间，超过此时间未成功登录则断开连接。
    • MaxAuthTries：登录时允许的最大密码尝试次数。

    使用配置示例：

    # 编辑/etc/ssh/sshd_config文件
Port 22
ListenAddress 0.0.0.0
PermitRootLogin prohibit-password
PasswordAuthentication no
PubkeyAuthentication yes
AuthorizedKeysFile .ssh/authorized_keys
MaxSessions 10
LoginGraceTime 120
MaxAuthTries 3
    3.总结¶
    • ssh_config用于配置SSH客户端，影响本地机器连接远程服务器时的行为。
    • sshd_config用于配置SSH服务器，影响远程客户端连接到当前主机时的认证和访问控制等。

    方法一：修改用户级别的ssh_config文件¶

    修改~/.ssh/config文件， 添加：

    ServerAliveInterval 60
    ServerAliveCountMax 3
    方法二：使用命令行参数临时设置¶

    如果不想修改配置文件，也可以在连接时通过命令行参数临时设置保持活动状态：

    ssh -o ServerAliveInterval=60 -o ServerAliveCountMax=3 user@remote_host
    • ServerAliveInterval=60：每60秒向服务器发送一次保持活动状态的数据包。
    • ServerAliveCountMax=3：如果服务器在3个间隔内没有响应，SSH客户端将自动断开连接。

    ServerAliveInterval 60
    ServerAliveCountMax 3

    差异对比：

    维度 RDP（Remote Desktop Protocol） VNC（Virtual Network Computing）
    发明方 Microsoft AT&T 剑桥实验室 → 开源社区
    协议/标准 私有，持续更新（RDP 10.8） 开源，RFB 协议（3.8）
    工作模式 远程会话（单独桌面） 屏幕镜像（共享物理桌面）
    图形加速 支持 DirectX/OpenGL 重定向 纯帧缓冲，无 GPU 加速

    典型带宽 低（100 kbps 可流畅） 高（同屏 1080p ≈ 1–5 Mbps）

    典型端口 TCP 3389 TCP 5900+n
    一句话总结 “像本地一样开新桌面，省带宽” “把现场屏幕搬过来，简单通用”

    1. RDP 是“开新房”——服务器给你单独准备一套桌面，互不打扰；VNC 是“直播旧房”——所有人看同一块屏幕。
    2. Windows 远程桌面 = RDP 原生，开即用；Linux 远程桌面 = VNC 原生，装即用；Linux 想用 RDP 得装 xrdp。
    3. 带宽紧张选 RDP，跨平台/临时救急选 VNC；要安全都记得加 TLS 或 SSH 隧道。

    6.Web服务器¶

    6.1.WSGI 服务器¶

    6.1.1.uwsgi¶

    uwsgi 是一种功能非常强大的服务器，支持多种语言，包括 Python。uWSGI 提供了更多的高级配置和功能选项，适合复杂的生产环境部署。通常用于部署大型 Python 应用，特别是需要细粒度控制时。

    ps -ef | grep uwsgi | grep -v grep | awk '{print $2}' | xargs kill -9
ps -ef | grep uwsgi
netstat -anlp | grep 6006
lsof -i tcp:8888
uwsgi --ini uwsgi.ini
uwsgi --reload uwsgi.ini
uwsgi --stop uwsgi.ini
uwsgi --connect-and-read status uwsgi.ini

uwsgi --http 0.0.0.0:8888 --module runserver:app --master --processes 2 --threads 2 --enable-thread
    6.1.2.gunicorn¶

    gunicorn 是一个常用的 Python WSGI HTTP 服务器之一，支持多进程和多线程，具有良好的性能，适用于 Django、Flask 等 WSGI 应用。它支持多进程、多线程，并且可以与负载均衡工具结合使用，适合中小型同步应用和微服务架构。

    示例命令：

    gunicorn -w 4 -b 0.0.0.0:8000 myapp:app

    参数说明：

    • -w 4：指定 4 个工作进程。
    • -b 0.0.0.0:8000：绑定 IP 和端口，通常设置为本地端口，交由反向代理（如 Nginx）代理请求。

    最佳实践：将 gunicorn 与 Nginx 或 Apache 配合使用，让反向代理服务器负责处理静态资源、SSL/TLS 等。

    6.1.3.mod_wsgi¶

    mod_wsgi：这是 Apache 服务器的 WSGI 模块。mod_wsgi 集成在 Apache 中，可以将 Python 应用直接运行在 Apache 上，但在灵活性和性能上不如 Gunicorn 和 uWSGI。

    6.2.ASGI 服务器¶

    6.2.1.Uvicorn¶

    uvicorn 是一个轻量级的、高性能的 ASGI 服务器，适用于 FastAPI、Starlette 等 ASGI 应用。与 gunicorn 一样，通常用于生产环境，并且可以很好地支持异步操作，适合高并发的应用场景。

    示例命令：

    uvicorn myapp:app --host 0.0.0.0 --port 8000 --workers 4

    参数说明：

    • --workers 4：指定 4 个工作进程。
    • --host 和 --port：绑定主机和端口。

    最佳实践：与 Nginx 或 Traefik 等反向代理服务器结合，以实现更好的负载均衡和静态资源管理。

    6.2.2. gunicorn + uvicorn¶

    对于 ASGI 应用，可以将 gunicorn 和 uvicorn 组合使用，这样可以利用 gunicorn 的进程管理能力以及 uvicorn 的高性能异步处理能力。这种方式非常适合 FastAPI 等需要异步支持的框架。

    示例命令：

    gunicorn -w 4 -k uvicorn.workers.UvicornWorker myapp:app

    说明：

    • -k uvicorn.workers.UvicornWorker：指定使用 uvicorn 的 ASGI worker 来处理请求。
    6.2.3.Daphne¶

    daphne 是一个专为 Django Channels 设计的 ASGI 服务器，适合 WebSocket 连接的应用。daphne 通常用于实时通讯应用，如聊天、通知等场景。

    示例命令：

    daphne -b 0.0.0.0 -p 8000 myproject.asgi:application

    说明：

    • 与 Nginx 或 Apache 的反向代理结合使用，以处理静态文件和负载均衡。
    6.2.4. hypercorn¶

    hypercorn 是另一个高性能的 ASGI 服务器，支持 HTTP/2 和 HTTP/3，适合需要现代协议支持的 Web 应用。

    示例命令：

    hypercorn -b 0.0.0.0:8000 myapp:app

    7. 进程管理¶

    使用 supervisor 或 systemd , 在生产环境中，仅使用 gunicorn 或 uvicorn 启动服务器并不够，通常还需要进程管理工具如 supervisor 或 systemd 来确保服务的稳定性和自动重启。

    7.1.使用 supervisor 配置示例¶

    创建一个 supervisor 配置文件，例如 /etc/supervisor/conf.d/myapp.conf：

    [program:myapp]
command=/path/to/gunicorn -w 4 -b 0.0.0.0:8000 myapp:app
directory=/path/to/app
user=www-data
autostart=true
autorestart=true
stderr_logfile=/var/log/myapp.err.log
stdout_logfile=/var/log/myapp.out.log

    启动和管理服务：

    sudo supervisorctl reread
sudo supervisorctl update
sudo supervisorctl start myapp

    7.2.使用 systemd 配置示例¶

    创建一个 systemd 配置文件，例如 /etc/systemd/system/myapp.service：

    [Unit]
Description=MyApp Gunicorn Service
After=network.target

[Service]
User=www-data
Group=www-data
WorkingDirectory=/path/to/app
ExecStart=/path/to/gunicorn -w 4 -b 0.0.0.0:8000 myapp:app
Restart=always

[Install]
WantedBy=multi-user.target

    启动和管理服务：

    sudo systemctl daemon-reload
sudo systemctl start myapp
sudo systemctl enable myapp

function! FindPythonVenv() abort
  let l:current_dir = getcwd()
  let l:venv_dir = l:current_dir . '/.venv'
  if isdirectory(l:venv_dir)
    return l:venv_dir . '/bin/python'
  endif

  " 如果当前目录下没有 .venv，检查 ../.venv 目录
  let l:project_name = fnamemodify(l:current_dir, ':t')
  let l:global_venv_dir = expand('../.venv/' . l:project_name)
  if isdirectory(l:global_venv_dir)
    return l:global_venv_dir . '/bin/python'
  endif

  " 如果../.venv 目录下没有，检查 ~/.venv 目录
  let l:project_name = fnamemodify(l:current_dir, ':t')
  let l:global_venv_dir = expand('~/.venv/' . l:project_name)
  if isdirectory(l:global_venv_dir)
    return l:global_venv_dir . '/bin/python'
  endif

  " 默认返回系统的 Python 解释器
  return 'python'
endfunction

function! FindPythonVenv()
  let current_dir = getcwd()
  let venv_dir = current_dir . '/.venv'
  if isdirectory(venv_dir)
    return venv_dir . '/bin/python'
  endif

  " 如果当前目录下没有 .venv，检查 ~/CodeSpace/.venv 目录
  let project_name = fnamemodify(current_dir, ':t')
  let global_venv_dir = expand('$HOME/CodeSpace/.venv/' . project_name)
  if isdirectory(global_venv_dir)
    return global_venv_dir . '/bin/python'
  endif

  " 如果~/CodeSpace/.venv 目录下没有，检查 ~/.venv 目录
  let project_name = fnamemodify(current_dir, ':t')
  let global_venv_dir = expand('$HOME/.venv/' . project_name)
  if isdirectory(global_venv_dir)
    return global_venv_dir . '/bin/python'
  endif

  " 默认返回系统的 Python 解释器
  return 'python3'
endfunction

function! ProjectOpen(arg)
    let arg = a:arg
    let found = "false"
    for project in g:projects
        let project_name = get(project, 0)
        let project_path = get(project, 1)
        let project_python_libs = get(project, 2, [])

        if project_name == arg
            exe 'cd '.project_path.'/git/'
            silent! NERDTreeCWD
            python import vim; activate_this = '%s/bin/activate_this.py' % vim.eval('project_path')
            for lib in project_python_libs
                python import sys,vim; sys.path.append(vim.eval('lib'))
                echom 'Path '.lib.' has been added'
            endfor
            echom 'Project '.project[0].' has been activated - '.project_path
            let found = "true"
        endif
    endfor
    if found == "false"
       echom 'Project '.a:arg.' can not be found!'
    endif
endfunction

function! ProjectNames(prefix)
    let venvs = []
    for project in g:projects
       if len(a:prefix) == 0
            call add(venvs, project[0])
       elseif match(project[0], a:prefix) >= 0
            call add(venvs, project[0])
        endif
    endfor
    return venvs
endfunction

rg -v "Class" ./ -t py -C2
# 6.只显示匹配部分(-o)
rg -e "if.*debug" ./ -t py -o
# 7.忽略大小写(-i)
rg -ie "if.*debug" ./ -t py -o
# 8.只打印无匹配的文件名, --files-without-match
rg -e "(from|print|sys|os|re)" ./ -t py --files-without-match
# 9.打印所有将被搜索的文件 --files

    13.1.全能型工具：Peaclock¶

    特点：

    • 多模式支持：时钟、倒计时、秒表功能合一，可通过快捷键切换。
    • 视图丰富：支持 ASCII 艺术、数字、二进制等显示方式，适配终端大小并保持宽高比。
    • 国际化能力：基于 ICU 库，支持多语言时间格式（如中文），需设置环境变量LC_TIME=zh_CN.UTF-8。
    • 自定义深度：可调整颜色（4/8/24 位）、填充、边距，甚至在倒计时结束时执行指定 Shell 命令。

    安装：

    # 包管理器安装（FreeBSD等）

    2.3.3.输出指标¶
    • Latency（延迟）：
    • Latency 表示每个请求的平均延迟时间，通常以毫秒为单位。它是从发送请求到接收到响应的时间。
    • Requests/sec（每秒请求数）：
    • Requests/sec 表示每秒完成的请求数量。这个指标显示了服务器每秒能够处理的请求数量。
    • Transfer/sec（传输速率）：
    • Transfer/sec 表示每秒传输的数据量，通常以 MB/s 或者 KB/s 为单位。它表示服务器的带宽使用情况。
    • Socket errors（Socket错误）：
    • Socket errors 列出了在测试期间发生的各种 socket 错误的数量，比如连接错误、读取错误、写入错误、超时等。
    • Non-2xx or 3xx responses（非 2xx 或 3xx 响应）：
    • 显示了除了 2xx 和 3xx 状态码之外的响应数量，这些状态码通常代表着请求的失败或错误。
    • Errors：
    • Errors 列出了在测试期间发生的各种错误数量，可能包括超时、连接错误、HTTP 错误等。
    • Duration（持续时间）：
    • Duration 表示整个测试的持续时间，通常以秒为单位。
    2.3.4.特点与比较¶
    • 特点：
    • 高度可定制，支持多线程，适用于HTTP基准测试。
    • 提供较为详细的性能统计和结果。
    • 适用场景：
    • 需要更高级定制和更详细结果分析的场景。
    • 限制：
    • 需要一定的命令行技能和理解性能统计指标的知识。

    ab 和 siege 更适合简单的测试场景，对于快速进行基准测试或较简单的压力测试是不错的选择。 wrk 更适合对于 HTTP 请求和并发性能有更高要求的场景，需要更高级的定制和更详细的结果分析。
