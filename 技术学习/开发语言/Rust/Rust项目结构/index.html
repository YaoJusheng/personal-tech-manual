    思考:Package、Crate、Module 与目录含义?

    Rust 的项目结构设计核心是 “分层组织代码”，通过 Package（项目包）、Crate（编译单元）、Module（代码模块）三层结构实现代码的模块化、可复用和可维护。同时，src、target 等目录承担着源代码存储、编译产物输出等核心职责。本文将从核心概念到实际应用，全面拆解 Rust 项目结构。

    总结:

    Rust 项目结构的核心是 “三层组织”：

    • Package 管理项目配置和 Crate；
    • Crate 是编译最小单位（二进制 / 库）；
    • Module 拆分代码、控制可见性。

    一、核心概念：Package、Crate、Module 详解¶

    这三个概念是理解 Rust 项目结构的基础，从大到小层层嵌套：Package 包含 Crate，Crate 包含 Module。

    1. Package（项目包）¶

    定义¶

    Package 是 Rust 的 项目打包单位，是使用 cargo 工具管理的最小单元，用于组织一个或多个相关的 Crate，以及项目的依赖、构建配置等。

    核心特征¶

    • 必须包含一个 Cargo.toml 文件（项目配置文件，记录依赖、构建规则等）；
    • 可以包含多个 Crate，但 最多只能有 1 个库 Crate（Library Crate），可以有任意多个二进制 Crate（Binary Crate）；
    • 通过 cargo new <项目名> 创建（默认生成二进制 Crate），cargo new --lib <项目名> 创建库 Crate。

    示例：Package 的构成¶

    一个典型的 Package 结构（包含 1 个库 Crate + 2 个二进制 Crate）：

    my_project/          # Package 根目录
├── Cargo.toml       # 项目配置文件
├── Cargo.lock       # 依赖版本锁定文件（自动生成，无需手动修改）
├── src/             # 源代码目录（核心 Crate 存放处）
│   ├── lib.rs       # 库 Crate 的入口文件（Crate Root）
│   ├── main.rs      # 默认二进制 Crate（名为 my_project）的入口文件
│   └── bin/         # 额外二进制 Crate 存放目录（可选）
│       ├── cli.rs   # 第二个二进制 Crate（名为 cli）
│       └── tool.rs  # 第三个二进制 Crate（名为 tool）
└── tests/           # 集成测试目录（可选）

    2. Crate（编译单元）¶

    定义¶

    Crate 是 Rust 的 最小编译单位（Rust 编译器一次只编译一个 Crate），编译后会生成可执行文件（二进制 Crate）或库文件（库 Crate）。

    分类¶

    类型 作用 入口文件 编译产物
    二进制 Crate（Binary） 生成可执行程序（如命令行工具） src/main.rs 或 src/bin/xxx.rs 可执行文件（target/debug/xxx）
    库 Crate（Library） 提供可复用代码（供其他 Crate 依赖） src/lib.rs 库文件（如 libxxx.rlib）

    关键特性¶

    • Crate Root：每个 Crate 都有一个 “入口文件”（main.rs 或 lib.rs），编译器从这里开始解析代码，构建整个 Crate 的模块树（Module Tree）；
    • 一个 Package 中，src/main.rs 对应一个默认的二进制 Crate（Crate 名与 Package 名相同）；若需多个二进制 Crate，可在 src/bin 目录下创建多个 .rs 文件（每个文件对应一个二进制 Crate）；
    • 库 Crate 不能直接运行，必须被其他二进制 Crate 依赖使用（通过 Cargo.toml 中的 dependencies 声明）。

    3. Module（代码模块）¶

    定义¶

    Module 是 Crate 内部的 代码组织单元，用于拆分 Crate 中的代码（如按功能、模块拆分），控制代码的可见性（访问权限），避免命名冲突。

    核心作用¶

    • 拆分大型 Crate：将一个复杂的 Crate 拆分为多个逻辑模块（如 user、utils、network）；
    • 控制可见性：默认情况下，Module 内的成员（函数、结构体、枚举等）是私有的，仅能在 Module 内部访问；通过 pub 关键字可将成员暴露给外部；
    • 形成模块树：以 Crate Root（main.rs/lib.rs）为根节点，所有 mod 声明的模块构成一棵 “模块树”。

    模块树示例¶

    假设 src/lib.rs 中有如下代码：

    // 声明一个名为 user 的模块
mod user {
    // 模块内嵌套子模块
    mod profile {
        // 私有函数（默认）
        fn format_name() -> String {
            "Alice".to_string()
        }

        // 公开函数（pub 暴露）
        pub fn get_user_name() -> String {
            format_name()
        }
    }

    // 公开子模块（pub 暴露，外部可访问 user::profile）
    pub mod profile;
}

// 声明一个名为 utils 的模块（拆分到单独文件）
mod utils;

    对应的模块树结构：

    crate（根节点，对应 lib.rs）
├── user（模块）
│   └── profile（子模块，pub 暴露）
│       ├── format_name（私有函数）
│       └── get_user_name（公开函数）
└── utils（模块，拆分到 utils.rs 文件）

    二、项目目录结构详解¶

    Rust 项目的目录结构遵循约定优于配置（Convention over Configuration）原则，核心目录为 src 和 target，其他目录为辅助功能。

    1. src/：源代码目录（核心）¶

    所有可编译的源代码都存放在 src/ 下，编译器仅会处理该目录内的文件。

    1.1.核心文件 / 子目录¶

    路径 含义与作用
    src/main.rs 默认二进制 Crate 的入口文件（Crate Root），编译后生成与 Package 同名的可执行文件
    src/lib.rs 库 Crate 的入口文件（Crate Root），编译后生成库文件，供其他 Crate 依赖
    src/bin/ 额外二进制 Crate 的存放目录（可选）。目录下每个 .rs 文件都是一个独立的二进制 Crate，编译后生成对应的可执行文件（如 src/bin/cli.rs 编译为 cli）
    src/xxx/ 模块对应的目录（用于拆分多文件模块），通常配合 mod.rs 或模块名文件使用

    1.2.src/lib.rs：库 Crate 的入口¶

    src/lib.rs 是库 Crate 的 “根节点”，编译器从这里开始解析代码，构建库的模块树。它的核心价值是 封装可复用逻辑，作为项目的 “核心业务层”。

    1. 核心作用¶
    • 代码复用：将项目中通用的逻辑（如工具函数、数据结构、业务逻辑）放在 lib.rs 中，供本项目的多个二进制 Crate（如 src/main.rs 和 src/bin/*.rs）或外部项目复用。
    • 测试友好：库 Crate 中的代码可以直接编写单元测试（通过 #[cfg(test)] 模块），而二进制 Crate 的代码（如 main.rs）通常需要通过集成测试验证。
    • 对外暴露接口：如果项目需要作为库被其他 Rust 项目依赖（如发布到 crates.io），lib.rs 是对外暴露公共 API 的唯一入口。
    2. 实际使用场景¶
    场景 1：封装通用工具函数¶

    例如，一个处理字符串的工具库：

    // src/lib.rs
//! 一个简单的字符串处理库（文档注释，通过 `cargo doc` 生成文档）

/// 将字符串转换为大写（公共 API）
pub fn to_uppercase(s: &str) -> String {
    s.to_uppercase()
}

/// 内部辅助函数（私有，仅库内使用）
fn trim_whitespace(s: &str) -> &str {
    s.trim()
}

// 单元测试（直接在库中编写，通过 `cargo test` 运行）
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_to_uppercase() {
        assert_eq!(to_uppercase("hello"), "HELLO");
    }
}
    场景 2：定义核心数据结构与业务逻辑¶

    例如，一个用户管理系统的核心逻辑：

    // src/lib.rs
pub mod user {
    /// 用户结构体（公共 API）
    pub struct User {
        pub id: u64,
        pub name: String,
    }

    impl User {
        /// 创建新用户（公共构造函数）
        pub fn new(id: u64, name: &str) -> Self {
            User {
                id,
                name: name.to_string(),
            }
        }

        /// 获取用户名（公共方法）
        pub fn get_name(&self) -> &str {
            &self.name
        }
    }
}

/// 模拟用户存储（业务逻辑）
pub struct UserStore {
    users: Vec<user::User>,
}

impl UserStore {
    pub fn new() -> Self {
        UserStore { users: Vec::new() }
    }

    pub fn add_user(&mut self, user: user::User) {
        self.users.push(user);
    }

    pub fn get_user_by_id(&self, id: u64) -> Option<&user::User> {
        self.users.iter().find(|u| u.id == id)
    }
}
    3. 被依赖的方式¶

    库 Crate 可以被两种类型的代码依赖：

    （1）本项目内的二进制 Crate（如 src/main.rs）¶

    同一个 Package 中的二进制 Crate 可以直接通过 Package 名称 引用库 Crate 中的代码，无需在 Cargo.toml 中额外声明。

    例如，src/main.rs 使用上述 lib.rs 中的逻辑：

    // src/main.rs
// 引用同一个 Package 中的库 Crate（Package 名为 `user_system` 时）
use user_system::{user::User, UserStore};

fn main() {
    let mut store = UserStore::new();
    store.add_user(User::new(1, "Alice"));

    if let Some(user) = store.get_user_by_id(1) {
        println!("Found user: {}", user.get_name()); // 输出 "Found user: Alice"
    }
}
    （2）外部项目¶

    如果将库发布到 crates.io 或作为本地依赖，其他项目可在 Cargo.toml 中声明依赖后使用：

    # 外部项目的 Cargo.toml
[dependencies]
user_system = "0.1.0" # 假设库已发布到 crates.io

    外部项目的代码中引用：

    // 外部项目的 src/main.rs
use user_system::user::User;

fn main() {
    let user = User::new(2, "Bob");
    println!("User: {}", user.get_name());
}

    1.3.src/main.rs：默认二进制 Crate 的入口¶

    src/main.rs 是二进制 Crate 的 “根节点”，必须包含 main 函数（程序的入口点）。它的核心价值是 作为应用程序的 “启动器” 和 “调度器”，连接用户输入与库 Crate 的核心逻辑。

    1. 核心作用¶
    • 程序入口：通过 main 函数启动程序，是操作系统加载程序时执行的第一个函数。
    • 处理外部输入：负责解析命令行参数、读取配置文件、处理用户交互等 “外部交互逻辑”。
    • 调度核心逻辑：调用库 Crate（lib.rs）中的函数或模块，将外部输入转换为业务逻辑的调用，并处理输出结果。
    • 生成可执行文件：编译后直接生成可执行文件（如 target/debug/项目名），可独立运行。
    2. 实际使用场景¶
    场景 1：简单命令行工具¶

    例如，一个调用库中字符串处理函数的命令行工具：

    // src/main.rs
use std::env;
// 引用本项目的库 Crate（假设 Package 名为 `string_tool`）
use string_tool::to_uppercase;

fn main() {
    // 处理命令行参数（外部输入）
    let args: Vec<String> = env::args().collect();
    if args.len() < 2 {
        eprintln!("用法：{} <字符串>", args[0]);
        std::process::exit(1);
    }

    // 调用库中的核心逻辑
    let input = &args[1];
    let result = to_uppercase(input);

    // 输出结果
    println!("转换后：{}", result);
}

    运行方式：

    cargo run -- "hello world"
# 输出：转换后：HELLO WORLD
    场景 2：复杂应用的启动与调度¶

    例如，一个用户管理系统的命令行界面，调用 lib.rs 中的 UserStore：

    // src/main.rs
use std::io;
use user_system::{user::User, UserStore};

fn main() {
    let mut store = UserStore::new();
    loop {
        // 显示菜单（用户交互）
        println!("\n用户管理系统");
        println!("1. 添加用户");
        println!("2. 查询用户");
        println!("3. 退出");
        print!("请选择：");

        // 读取用户输入
        let mut choice = String::new();
        io::stdin().read_line(&mut choice).unwrap();
        let choice = choice.trim();

        // 调度库中的逻辑
        match choice {
            "1" => add_user(&mut store),
            "2" => query_user(&store),
            "3" => break,
            _ => println!("无效选择"),
        }
    }
}

// 辅助函数：添加用户（调用库中的 User::new 和 UserStore::add_user）
fn add_user(store: &mut UserStore) {
    print!("请输入用户 ID：");
    let mut id_str = String::new();
    io::stdin().read_line(&mut id_str).unwrap();
    let id: u64 = id_str.trim().parse().unwrap();

    print!("请输入用户名：");
    let mut name = String::new();
    io::stdin().read_line(&mut name).unwrap();
    let name = name.trim();

    store.add_user(User::new(id, name));
    println!("用户添加成功");
}

// 辅助函数：查询用户（调用库中的 UserStore::get_user_by_id）
fn query_user(store: &UserStore) {
    print!("请输入要查询的用户 ID：");
    let mut id_str = String::new();
    io::stdin().read_line(&mut id_str).unwrap();
    let id: u64 = id_str.trim().parse().unwrap();

    match store.get_user_by_id(id) {
        Some(user) => println!("找到用户：{}（ID：{}）", user.get_name(), user.id),
        None => println!("未找到 ID 为 {} 的用户", id),
    }
}

    这个例子中，main.rs 只负责处理用户交互（菜单、输入输出），而核心的用户创建、存储、查询逻辑全部封装在 lib.rs 中，实现了 “交互层” 与 “业务层” 的分离。

    3. 与其他二进制 Crate 的关系¶

    除了 src/main.rs，项目还可以通过 src/bin/ 目录创建多个二进制 Crate（如工具、子命令）。这些二进制 Crate 与 src/main.rs 地位平等，都可以依赖 src/lib.rs 中的库逻辑。

    例如，添加一个批量导入用户的工具 src/bin/import.rs：

    // src/bin/import.rs
use user_system::{user::User, UserStore};
use std::fs::File;
use std::io::BufReader;
use serde::Deserialize;

// 定义 JSON 导入格式（依赖 serde 和 serde_json）
#[derive(Deserialize)]
struct ImportUser {
    id: u64,
    name: String,
}

fn main() {
    let mut store = UserStore::new();
    // 读取 JSON 文件（简化示例）
    let file = File::open("users.json").unwrap();
    let reader = BufReader::new(file);
    let import_users: Vec<ImportUser> = serde_json::from_reader(reader).unwrap();

    // 调用库中的逻辑导入用户
    for u in import_users {
        store.add_user(User::new(u.id, &u.name));
    }

    println!("成功导入 {} 个用户", store.get_user_count());
}

    运行该二进制 Crate：

    cargo run --bin import

    这里，src/bin/import.rs 和 src/main.rs 都依赖 lib.rs 中的 User 和 UserStore，实现了代码复用。

    1.4.lib.rs 与 main.rs 的协作模式¶

    在实际项目中，lib.rs 和 main.rs 通常遵循 “业务逻辑下沉到 lib，交互逻辑放在 main” 的原则，这种模式有以下优势：

    1. 复用性：lib.rs 中的逻辑可被多个二进制 Crate（main.rs、src/bin/*.rs）复用，避免代码重复。
    2. 可测试性：lib.rs 中的代码可直接编写单元测试（通过 #[test]），而 main.rs 中的交互逻辑通常难以单元测试，可通过测试 lib 间接验证。
    3. 可扩展性：当需要添加新功能（如 CLI 子命令、Web 接口）时，只需新增二进制 Crate（如 src/bin/web.rs），直接复用 lib.rs 中的核心逻辑。
    4. 对外暴露：若未来需要将项目作为库供其他项目使用，lib.rs 已封装好所有必要的 API，无需修改核心逻辑。

    2. target/：编译产物目录¶

    cargo build、cargo run 等命令的编译产物（中间文件、最终可执行文件、库文件）都会存放在 target/ 下，该目录由 Cargo 自动管理，无需手动修改。

    核心子目录¶

    路径 含义与作用
    target/debug/ 开发模式（Debug 模式）的编译产物目录（默认编译模式）。特点：编译快、包含调试信息、无优化
    target/release/ 发布模式（Release 模式）的编译产物目录（通过 cargo build --release 生成）。特点：编译慢、无调试信息、有性能优化，适合生产环境
    target/deps/ 项目依赖的 Crate 的编译产物目录
    target/doc/ 文档目录（通过 cargo doc 生成项目文档，包含自身代码和依赖的文档）
    target/out/ 构建脚本（build.rs）的输出目录（若项目有构建脚本）

    3. 其他辅助目录 / 文件¶

    路径 / 文件 含义与作用
    Cargo.toml Package 的配置文件（必需），记录项目名称、版本、作者、依赖、构建规则等
    Cargo.lock 依赖版本锁定文件（自动生成），确保每次构建使用相同版本的依赖，避免版本冲突
    tests/ 集成测试目录（可选），目录下的 .rs 文件是独立的测试 Crate，通过 cargo test 运行
    examples/ 示例代码目录（可选），存放项目的使用示例，通过 cargo run --example <示例名> 运行
    benches/ 基准测试目录（可选），存放性能测试代码，通过 cargo bench 运行
    build.rs 构建脚本（可选），编译前执行（如生成代码、编译原生依赖），需在 Cargo.toml 中声明

fn generate_version_info() -> Result<(), Box<dyn std::error::Error>> {
    let out_dir = env::var("OUT_DIR")?;
    let dest_path = Path::new(&out_dir).join("version_info.rs");

    // 获取 Git 提交信息
    let git_hash = get_git_hash().unwrap_or_else(|_| "unknown".to_string());
    let git_tag = get_git_tag().unwrap_or_else(|_| "".to_string());

    // 获取构建时间
    let build_time = chrono::Utc::now().format("%Y-%m-%d %H:%M:%S UTC").to_string();

    // 获取 Cargo 包信息
    let version = env::var("CARGO_PKG_VERSION")?;
    let name = env::var("CARGO_PKG_NAME")?;

    // 生成 Rust 代码
    let content = format!(
        r#"
/// 自动生成的版本信息
pub const VERSION_INFO: VersionInfo = VersionInfo {{
    name: "{}",
    version: "{}",
    git_hash: "{}",
    git_tag: "{}",
    build_time: "{}",
    target: "{}",
}};

#[derive(Debug, Clone)]
pub struct VersionInfo {{
    pub name: &'static str,
    pub version: &'static str,
    pub git_hash: &'static str,
    pub git_tag: &'static str,
    pub build_time: &'static str,
    pub target: &'static str,
}}
"#,
        name,
        version,
        git_hash,
        git_tag,
        build_time,
        env::var("TARGET")?
    );

    fs::write(&dest_path, content)?;

    // 告诉 Cargo 当这些文件改变时重新构建
    println!("cargo:rerun-if-changed=.git/HEAD");
    println!("cargo:rerun-if-changed=.git/refs");

    Ok(())
}

fn get_git_hash() -> Result<String, String> {
    Command::new("git")
        .args(&["rev-parse", "HEAD"])
        .output()
        .map_err(|e| e.to_string())
        .and_then(|output| {
            if output.status.success() {
                Ok(String::from_utf8_lossy(&output.stdout).trim().to_string())
            } else {
                Err("git rev-parse 失败".to_string())
            }
        })
}

fn get_git_tag() -> Result<String, String> {
    Command::new("git")
        .args(&["describe", "--tags", "--always"])
        .output()
        .map_err(|e| e.to_string())
        .and_then(|output| {
            if output.status.success() {
                Ok(String::from_utf8_lossy(&output.stdout).trim().to_string())
            } else {
                Ok("".to_string()) // 没有标签是可以接受的
            }
        })
}

fn check_pkgconfig(package: &str) {
    if pkg_config::Config::new()
        .probe(package)
        .is_err()
    {
        println!("cargo:warning={} 库未找到", package);
    }
}

fn check_header(header: &str) {
    let test_code = format!("#include <{}>\nint main() {{ return 0; }}", header);
    let out_dir = env::var("OUT_DIR").unwrap();
    let test_file = Path::new(&out_dir).join("test_header.c");

    fs::write(&test_file, test_code).unwrap();

    let status = std::process::Command::new("cc")
        .arg(&test_file)
        .arg("-o")
        .arg(test_file.with_extension("out"))
        .status();

    if status.is_err() || !status.unwrap().success() {
        println!("cargo:warning=头文件 {} 未找到", header);
    }
}

        println!("cargo:rustc-cfg=has_avx2");
    }
}

");

    三、Module 的创建语法与可见性控制¶

    Module 的创建有两种方式：单文件内创建（适合简单模块）和 多文件 / 目录拆分（适合复杂模块）。

    1. 单文件内创建 Module¶

    直接在 Crate Root（main.rs/lib.rs）或其他模块中使用 mod 关键字声明模块，模块内可嵌套子模块。

    语法示例¶

    // src/main.rs（二进制 Crate 入口）
fn main() {
    // 访问模块内的公开函数（绝对路径：crate::user::profile::get_user_name）
    let name = crate::user::profile::get_user_name();
    println!("User name: {}", name);

    // 访问模块内的公开函数（相对路径：user::profile::get_user_name）
    let age = user::get_age();
    println!("User age: {}", age);
}

// 声明模块 user（单文件内）
mod user {
    // 公开函数（pub 修饰，外部可访问）
    pub fn get_age() -> u8 {
        25
    }

    // 嵌套子模块 profile
    mod profile {
        // 私有函数（默认，仅模块内可访问）
        fn format_name() -> String {
            "Bob".to_string()
        }

        // 公开函数（外部可通过 user::profile::get_user_name 访问）
        pub fn get_user_name() -> String {
            format_name() // 模块内可访问私有函数
        }
    }

    // 公开子模块 profile（必须显式 pub，否则外部无法访问子模块）
    pub mod profile;
}

    2. 多文件 / 目录拆分 Module¶

    当模块代码较多时，可将模块拆分为独立文件或目录，核心是通过 mod 声明模块与文件的关联。

    拆分规则（Rust 2018+ 支持两种方式）¶

    模块结构 方式 1（传统：使用 mod.rs） 方式 2（推荐：模块名文件）
    单个文件模块 mod utils; + src/utils.rs 同方式 1（无区别）
    含子模块的目录模块 mod user; + src/user/mod.rs（声明子模块） mod user; + src/user.rs（声明子模块）

    示例 1：单个文件模块（utils.rs）¶

    1. 创建文件 src/utils.rs（模块代码）：
    // src/utils.rs
// 公开函数（供外部访问）
pub fn add(a: i32, b: i32) -> i32 {
    a + b
}

// 私有函数（仅 utils 模块内可访问）
fn subtract(a: i32, b: i32) -> i32 {
    a - b
}
    1. 在 Crate Root 中声明模块：
    // src/main.rs
// 声明 utils 模块（关联 src/utils.rs 文件）
mod utils;

fn main() {
    let sum = utils::add(10, 20);
    println!("Sum: {}", sum); // 输出 30

    // 错误：subtract 是私有函数，外部无法访问
    // let diff = utils::subtract(10, 20);
}

    示例 2：含子模块的目录模块（user 模块）¶

    需求：创建 user 模块，包含 profile 和 auth 两个子模块。

    方式 1：使用 mod.rs（传统方式，兼容所有 Rust 版本）¶
    1. 创建目录结构：
    src/
├── main.rs
└── user/          # user 模块目录
    ├── mod.rs     # user 模块入口文件（声明子模块）
    ├── profile.rs # profile 子模块
    └── auth.rs    # auth 子模块
    1. 编写 src/user/mod.rs（核心：声明子模块并控制可见性）：
    // src/user/mod.rs
// 声明子模块 profile（关联 profile.rs）
mod profile;
// 声明子模块 auth（关联 auth.rs）
mod auth;

// 公开子模块（外部可访问 user::profile 和 user::auth）
pub mod profile;
pub mod auth;

// user 模块自身的公开函数
pub fn get_user_id() -> u64 {
    1001
}
    1. 编写子模块代码（src/user/profile.rs 和 src/user/auth.rs）：
    // src/user/profile.rs
pub fn get_nickname() -> String {
    "rust_dev".to_string()
}

    // src/user/auth.rs
pub fn is_login() -> bool {
    true
}
    1. 在 main.rs 中使用：
    // src/main.rs
// 声明 user 模块（关联 src/user/mod.rs）
mod user;

fn main() {
    println!("User ID: {}", user::get_user_id()); // 1001
    println!("Nickname: {}", user::profile::get_nickname()); // rust_dev
    println!("Is login: {}", user::auth::is_login()); // true
}
    方式 2：模块名文件（Rust 2018+ 推荐，无需 mod.rs）¶

    无需创建 src/user/mod.rs，而是将模块入口代码写在 src/user.rs 中，目录结构简化为：

    src/
├── main.rs
├── user.rs        # user 模块入口文件（替代 mod.rs）
└── user/          # 子模块目录（与 user.rs 同名）
    ├── profile.rs
    └── auth.rs

    编写 src/user.rs（声明子模块）：

    // src/user.rs
// 声明子模块（关联 user/profile.rs 和 user/auth.rs）
mod profile;
mod auth;

// 公开子模块
pub mod profile;
pub mod auth;

pub fn get_user_id() -> u64 {
    1001
}

    使用方式与方式 1 完全一致（main.rs 代码不变）。

    3. 可见性控制（pub 关键字）¶

    Rust 的可见性规则：默认私有，仅模块内及子模块可访问；通过 pub 暴露给外部。

    pub 的常见用法¶

    语法 含义
    pub mod xxx 公开模块，外部可访问该模块及其公开成员
    pub fn xxx() 公开函数，外部可通过模块路径访问
    pub struct XXX 公开结构体，外部可实例化和访问其公开字段（结构体字段默认私有）
    pub enum XXX 公开枚举，所有变体默认公开（枚举变体无额外可见性控制）
    pub(crate) 仅当前 Crate 内可访问（比 pub 限制更严格）
    pub(super) 仅父模块可访问
    pub(in path) 仅指定路径的模块可访问（如 pub(in crate::user) 仅 user 模块内可访问）

    示例：结构体的可见性¶

    // src/user/profile.rs
// 公开结构体（外部可访问）
pub struct User {
    pub name: String, // 公开字段
    age: u8,          // 私有字段（外部无法访问）
}

impl User {
    // 公开构造函数（外部需通过该函数创建实例）
    pub fn new(name: &str, age: u8) -> Self {
        User {
            name: name.to_string(),
            age,
        }
    }

    // 公开方法（访问私有字段）
    pub fn get_age(&self) -> u8 {
        self.age
    }
}

    在 main.rs 中使用：

    mod user;

fn main() {
    let user = user::profile::User::new("Alice", 25);
    println!("Name: {}", user.name); // 允许（name 是公开字段）
    println!("Age: {}", user.get_age()); // 允许（通过公开方法访问私有字段）
    // println!("Age: {}", user.age); // 错误：age 是私有字段
}

    4. use 语句：简化模块路径¶

    通过 use 可将模块成员引入当前作用域，避免重复写长路径。

    常见用法¶

    // src/main.rs
mod user;

// 引入模块（简化路径）
use user::profile;
use user::auth::is_login;

// 引入并起别名
use user::profile::User as UserProfile;

fn main() {
    // 简化前：user::profile::get_nickname()
    println!("Nickname: {}", profile::get_nickname());

    // 简化前：user::auth::is_login()
    println!("Is login: {}", is_login());

    // 别名使用
    let user = UserProfile::new("Bob", 30);
    println!("Name: {}", user.name);
}

    glob 导入（*）：导入模块所有公开成员¶

    use user::profile::*; // 导入 profile 模块所有公开成员

fn main() {
    let user = User::new("Charlie", 28); // 直接使用 User，无需写 profile::User
    println!("Nickname: {}", get_nickname());
}

    四、mod.rs 的作用与写法总结¶

    mod.rs 是 传统的模块入口文件，用于声明子模块、控制可见性、导出成员，核心作用如下：

    1. mod.rs 的核心作用¶

    • 声明当前目录对应的模块的子模块（如 mod profile; 关联 profile.rs）；
    • 控制子模块的可见性（pub mod profile; 公开子模块）；
    • 导出模块成员（通过 pub use 简化外部访问路径）；
    • 存放当前模块的核心逻辑（如模块级函数、结构体定义）。

    2. mod.rs 的典型写法¶

    // src/user/mod.rs
// 1. 声明子模块（关联当前目录下的 .rs 文件）
mod profile;
mod auth;

// 2. 公开子模块（外部可访问）
pub mod profile;
pub mod auth;

// 3. 存放当前模块的核心逻辑
pub fn get_user_id() -> u64 {
    1001
}

// 4. 导出成员（简化外部访问路径：user::User 替代 user::profile::User）
pub use profile::User;

    外部使用时可直接访问导出的成员：

    // src/main.rs
mod user;
use user::User; // 直接导入 User，无需写 user::profile::User

fn main() {
    let user = User::new("Dave", 35);
}

    3. 注意事项¶

    • Rust 2018+ 支持用 “模块名文件”（如 user.rs）替代 mod.rs，功能完全一致，且目录结构更清晰，推荐优先使用；
    • mod.rs 仅用于 “目录模块”（即模块对应一个目录），单个文件模块（如 utils.rs）无需 mod.rs；
    • 若目录中同时存在 mod.rs 和模块名文件（如 user.rs），Rust 会报错（冲突）。

    五、常见误区与注意事项¶

    1. Package 与 Crate 的区别：Package 是 “项目包”（含配置和 Crate），Crate 是 “编译单元”（含代码）；一个 Package 可包含多个 Crate，但最多一个库 Crate；
    2. mod.rs 的兼容性：Rust 2018+ 支持 “模块名文件” 替代 mod.rs，但 mod.rs 仍兼容，老项目可继续使用；
    3. 可见性的层级关系：父模块无法访问子模块的私有成员，子模块可访问父模块的私有成员；
    4. Crate 间的依赖：若需在二进制 Crate 中使用库 Crate，无需额外配置（Package 内的库 Crate 自动可见）；外部 Crate 需在 Cargo.toml 的 dependencies 中声明；
    5. src 目录的唯一性：编译器仅处理 src/ 内的代码，若需存放非编译文件（如配置、资源），应放在 Package 根目录（如 config/、assets/）。

    总结¶

    Rust 项目结构的核心是 “三层组织”：

    • Package 管理项目配置和 Crate；
    • Crate 是编译最小单位（二进制 / 库）；
    • Module 拆分代码、控制可见性。

    目录方面，src/ 存放源代码（Crate Root 和模块），target/ 存放编译产物，辅助目录（tests/、examples/）按需使用。模块创建支持单文件和多文件拆分，mod.rs 或 “模块名文件” 用于声明子模块，pub 关键字控制可见性。
