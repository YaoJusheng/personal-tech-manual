    🗂️ Cargo 命令分类¶

    根据功能，Cargo 的子命令大致可分为以下几类：

    类别 常用命令 主要用途
    项目创建 cargo new, cargo init 创建新的 Rust 项目或在工作区内初始化模块
    构建与编译 cargo build, cargo check, cargo run, cargo clean 编译项目、检查错误、运行可执行文件及清理构建产物
    测试与检查 cargo test, cargo bench, cargo clippy 执行单元测试、集成测试、性能基准测试及代码 lint 检查
    依赖管理 cargo add, cargo update, cargo tree 添加、更新依赖或查看依赖树
    文档生成 cargo doc 根据代码中的注释生成本地和依赖库的 API 文档
    包发布 cargo login, cargo publish, cargo yank 登录 crates.io、发布包或将特定版本标记为不可用
    辅助工具 cargo fmt, cargo fix 自动格式化代码或修复编译器警告

    运行 cargo --list 可查看所有已安装的子命令-link。

    ⚙️ 全局选项¶

    这些选项可以放在任何子命令之前，用于控制 Cargo 的整体行为。

    类别 选项 说明
    基础信息 -V, --version 显示版本信息
    -h, --help 显示帮助信息，后跟子命令可获取详细帮助
    --list 列出所有已安装的子命令
    --explain <CODE> 显示 Rust 编译错误的详细解释
    输出控制 -v, --verbose 启用详细输出，显示更多构建信息
    -q, --quiet 静默模式，不输出任何信息
    --color <WHEN> 控制彩色输出 (auto, always, never)
    网络与锁文件 --frozen / --locked 要求 Cargo.lock 文件是最新的，否则报错
    --offline 阻止 Cargo 访问网络，仅使用本地缓存的依赖
    工具链选择 +<toolchain> 指定 Rust 工具链，例如 +stable 或 +nightly
    -Z <flag> 使用 Nightly 通道的不稳定特性

    🔧 常用子命令详解¶

    以下是开发中最常用的几个子命令及其重要选项。

    cargo new / cargo init¶

    选项 说明
    --bin / --lib 创建二进制应用或库项目
    --edition <YEAR> 指定 Rust 版本 (2015, 2018, 2021, 2024)
    --name <NAME> 设置项目名称，默认为目录名
    --vcs <VCS> 初始化版本控制系统 (git, hg, none)
    --registry <REG> 限制包只能发布到特定注册表

    cargo add¶

    选项 说明
    --path <PATH> 从本地文件系统的指定路径添加依赖。
    --git <URL> 从指定的 Git 仓库拉取依赖。
    --branch <BRANCH> 从 Git 拉取时使用指定分支。
    --tag <TAG> 从 Git 拉取时使用指定标签。
    --rev <SHA> 从 Git 拉取时使用指定的提交哈希值。
    --registry <NAME> 指定使用的注册表（如私有注册表）。
    --base <PATH> （不稳定） 添加本地 crate 时的基准路径，仅限 Nightly 版本。
    --dev 或 -D 将依赖添加到 [dev-dependencies] 段，仅在测试、示例和基准测试中可用。
    --build 或 -B 将依赖添加到 [build-dependencies] 段，仅在构建脚本 build.rs 中可用。
    --target <TARGET> 为特定目标平台添加依赖，例如 --target 'cfg(unix)'。
    --features <FEATURES> 或 -F 启用依赖项的一个或多个可选特性。多个特性可用逗号分隔，并建议使用引号或直接连接以防止 shell 解析错误。
    --no-default-features 禁用依赖项的默认特性。
    --default-features （与前者相反） 重新启用依赖项的默认特性。
    --optional 将依赖项标记为可选，通常用于创建自己的 feature 时。
    --no-optional 将依赖项标记为必选（非可选）。
    --rename <NAME> 为依赖项在本地重命名，以避免命名冲突。
    --public / --no-public （不稳定，Nightly 可用） 分别标记依赖是否暴露在你的公共 API 中。
    --dry-run 预览修改结果，但不会实际修改 Cargo.toml 文件。
    -v / --verbose 输出更详细的构建信息。
    -q / --quiet 静默模式，不输出 Cargo 的日志信息。
    --color <WHEN> 控制是否使用彩色输出，可选 auto（默认）、always、never。
    --manifest-path <PATH> 指定 Cargo.toml 文件的路径。

    cargo build¶

    选项 说明
    --release 使用 release 模式构建，启用优化
    --target <TRIPLE> 为目标平台交叉编译
    --manifest-path 直接指向 Cargo.toml 文件，定义了所有 Cargo 操作的根上下文。
    -p, --package <SPEC> 仅构建工作区中指定的包
    --workspace 构建工作区中的所有成员
    --lib 仅构建库目标
    --bin <NAME> 构建指定的二进制目标
    --features <FEATURES> 激活指定的特性
    --all-features 激活所有特性
    --no-default-features 不激活默认特性

    cargo run¶

    选项 说明
    --release 以 release 模式构建并运行
    -p, --package <SPEC> 运行工作区中指定包的二进制文件
    --bin <NAME> 运行指定的二进制目标
    --example <NAME> 运行指定的示例程序
    --target <TRIPLE> 为目标平台编译并运行
    --features <FEATURES> 激活指定的特性

    特殊用法：使用 -- 分隔符将参数传递给程序本身。例如，cargo run -- --help 会将 --help 参数传递给被运行的程序。

    cargo test¶

    选项 说明
    --no-run 仅编译测试代码，不运行
    --no-fail-fast 运行所有测试，即使有失败的测试也不提前终止
    --test <NAME> 只运行指定的集成测试文件
    --lib 只运行单元测试
    --doc 只运行文档测试

    特殊用法：使用 -- 分隔符将参数传递给测试运行器。例如，cargo test -- --nocapture 可显示测试中的 println! 输出。

    cargo doc¶

    选项 说明
    --open 生成文档后自动在浏览器中打开
    --no-deps 不为依赖项生成文档
    --document-private-items 也为非公开项生成文档

    cargo install¶

    选项 说明
    --list 列出所有已安装的二进制 crate。
    --index 执行命令时临时指定其他源。
    --force 强制重新安装，即使已存在。
    --version <VERSION> 安装指定版本。
    --git <URL> 从 Git 仓库安装。
    --path <PATH> 从本地路径安装。
    --bin <NAME> 只安装指定的二进制文件。
    --example <NAME> 只安装指定的示例。

    其他¶

    cargo update¶

    选项 说明
    <spec> 只更新指定的包-。
    --dry-run 显示将要更新的内容，但不实际执行。
    --precise <VERSION> 将包更新到精确的版本。
    --workspace 更新工作区中的所有包。

    cargo clean¶

    选项 说明
    --doc 只删除 target/doc 目录下的文档。
    --release 只删除 target/release 目录下的 release 产物。
    --target-dir <DIRECTORY> 指定要清理的目录。
    -p, --package <SPEC> 只清理指定包的产物-。

    cargo search¶

    选项 说明
    --limit <LIMIT> 限制显示的结果数量（默认 10，最大 100）。
    --registry <REGISTRY> 指定要搜索的注册表。

    cargo publish¶

    选项 说明
    --dry-run 执行所有检查但不实际上传。
    --token <TOKEN> 用于认证的 API token-。
    --registry <REGISTRY> 指定要发布到的注册表。
    --allow-dirty 允许工作目录中有未提交的更改时发布。

    cargo yank¶

    选项 说明
    --vers <VERSION> 指定要 yank 的版本。
    --undo 撤销 yank 操作。
    --token <TOKEN> API token。
    --registry <REGISTRY> 指定注册表。

    cargo login¶

    选项 说明
    --registry <REGISTRY> 指定要登录的注册表。

    cargo logout¶

    选项 说明
    --registry <REGISTRY> 指定要登出的注册表。

    cargo fetch¶

    选项 说明
    --target <TRIPLE> 为指定目标架构获取依赖。

    cargo generate-lockfile¶

    选项 说明
    --manifest-path <PATH> 指定 Cargo.toml 的路径。
    --frozen / --locked 强制使用锁文件。
    --offline 离线模式。

    cargo locate-project¶

    选项 说明
    --workspace 定位工作区根目录的 Cargo.toml。
    --message-format <FMT> 指定输出格式（如 json）。

    cargo package¶

    选项 说明
    --list 列出将包含在包中的文件。
    --no-verify 打包后不进行验证。
    --allow-dirty 允许未提交的更改。

    cargo pkgid¶

    选项 说明
    <spec> 指定要获取 ID 的包-。

    cargo verify-project¶

    选项 说明
    --manifest-path <PATH> 指定 Cargo.toml 的路径。

    cargo version¶

    选项 说明
    -v, --verbose 显示更详细的版本信息。

    🔌 扩展命令¶

    Cargo 的设计允许用户添加自定义子命令。在 $PATH 中创建一个名为 cargo-something 的可执行文件，就可以通过 cargo something 来运行它。常用的扩展命令包括：

    • cargo edit: 提供 cargo add、cargo rm 等命令来直接编辑 Cargo.toml。
    • cargo tree: 以树形结构显示依赖关系。
    • cargo audit: 检查依赖项中的安全漏洞。
    • cargo expand: 展开宏以查看生成的代码。

    🏗️ 高级特性¶

    • 工作空间（Workspace）: 通过共享的 Cargo.lock 和输出目录来管理多个相关的包。
    • 特性（Features）: 一种条件编译机制，允许在编译时选择性地启用或禁用代码。
    • 自定义构建脚本: 通过在 build.rs 文件中编写 Rust 代码，在编译主库之前执行自定义逻辑，例如生成代码或链接系统库。

    🎨 配置与自定义¶

    配置文件 (config.toml)¶

    Cargo 的配置是分层合并的，允许在不同级别（项目、全局）进行设置。配置文件使用 TOML 格式，可在项目或用户目录的 .cargo/config.toml 中定义持久化设置。

    1. 读取配置位置¶

    Cargo 会按以下顺序读取配置，后读取的覆盖先读取的（就近原则）：

    层级 路径（Linux/macOS） 路径（Windows）
    系统级 /etc/cargo/config.toml %PROGRAMDATA%\cargo\config.toml
    用户级（全局） ~/.cargo/config.toml %USERPROFILE%\.cargo\config.toml
    项目级 <project>/.cargo/config.toml <project>\.cargo\config.toml

    注意：

    • 旧版文件名是 config（无后缀），新版推荐 .toml 后缀。两者同时存在时，Cargo 会报错。
    • 项目级配置应提交到 Git，用于统一团队构建行为（如交叉编译目标、私有源）。

    2. 配置项解析¶

    2.1. [source] —— 镜像源与替换（最常用）¶

    控制依赖从哪里下载，地址前缀 sparse+ 表示使用稀疏索引（Cargo 1.70+），速度远快于传统的 git 协议全量克隆。

    # 将官方 crates.io 替换为国内镜像
[source.crates-io]
replace-with = 'tuna'

# 定义 tuna 镜像（sparse 协议，推荐）
[source.tuna]
registry = "sparse+https://mirrors.tuna.tsinghua.edu.cn/crates.io-index/"

# 定义 ustc 镜像（备用）
[source.ustc]
registry = "sparse+https://mirrors.ustc.edu.cn/crates.io-index/"

# 定义 rustcc 镜像
[source.rustcc]
registry = "sparse+https://crates.rustcc.cn/crates.io-index/"

    关键规则：

    • replace-with 的值必须与 [source.xxx] 中的 xxx 严格一致
    • sparse+ 前缀表示稀疏索引（Cargo 1.70+），速度远快于传统 git 协议
    • 末尾必须带 /
    2.2. [net] —— 网络行为¶
    [net]
# 下载失败时的重试次数（默认 2）
retry = 3

# 并发下载的 crate 数量（默认根据 CPU 核心数）
# concurrency = 8

# 使用系统 git 命令拉取 git 依赖，而非 Cargo 内置的 libgit2
# 解决国内 libgit2 SSL/证书/代理兼容性问题的万能钥匙
git-fetch-with-cli = true

# 离线模式（仅使用本地缓存，不联网）
# offline = true

# 通用策略：为每个目标平台配置链接器
[target.x86_64-pc-windows-gnu] # 目标：Windows (MinGW)
linker = "x86_64-w64-mingw32-gcc" # 指定MinGW交叉编译器作为链接器

[target.aarch64-unknown-linux-gnu] # 目标：ARM64 Linux
linker = "aarch64-linux-gnu-gcc" # 指定ARM64 GCC交叉编译器

[target.x86_64-unknown-linux-musl] # 目标：使用musl libc的Linux（静态链接，兼容性极佳）
linker = "x86_64-linux-musl-gcc"

[target.armv7-linux-androideabi] # 目标：ARMv7 Android
linker = "arm-linux-androideabi-clang"
ar = "arm-linux-androideabi-ar"
    2.3. [http] —— HTTP/代理/SSL¶
    [http]
# HTTP 代理（支持 http:// 和 socks5://）
proxy = "http://127.0.0.1:7890"

# 多代理分流（部分 Cargo 版本支持）
# proxy = "http://proxy.example.com:8080"

# 禁用 SSL 证书校验（仅临时调试，生产环境极度危险）
# ssl-verify = false

# 检查证书吊销列表（CRL），某些内网环境需关闭
# check-revoke = false

# 连接/读取超时（秒）
# timeout = 30

# 多路复用（HTTP/2）
# multiplexing = true

    代理环境变量（优先级高于配置文件）：

    export HTTP_PROXY=http://127.0.0.1:7890
export HTTPS_PROXY=http://127.0.0.1:7890
export ALL_PROXY=socks5://127.0.0.1:7890
export NO_PROXY=localhost,127.0.0.1,.example.com
    2.4. [build] —— 默认构建行为¶
    [build]
# 默认构建目标（常用于嵌入式/交叉编译）
target = "x86_64-pc-windows-gnu"

# 并行编译的代码生成单元数（默认等于 CPU 核心数）
jobs = 8

# 传递给 rustc 的全局 flags（影响所有 crate）
rustflags = ["-C", "target-cpu=native"]

# 传递给 rustdoc 的 flags
rustdocflags = ["--cfg", "docsrs"]

# 增量编译开关
incremental = true

# 指定 Cargo 使用的 rustc 路径（非默认）
# rustc = "/path/to/rustc"

# 指定 target 目录（默认项目根目录的 target/）
# target-dir = "/path/to/global/target"
    2.5. [profile] —— 编译优化配置¶

    覆盖 dev（cargo build）和 release（cargo build --release）的默认行为。

    [profile.dev]
# 调试信息：0=无, 1=行表, 2=完整（默认 2）
debug = 2

# 优化级别：0=无, 1=基本, 2=正常, 3=激进, "s"=体积, "z"=极致体积
opt-level = 0

# 代码生成单元数：增大加速编译，减小提升优化（release 建议 1）
codegen-units = 256

# 开启 panic 时打印 backtrace（默认 true）
panic = 'unwind'

# 链接时优化（LTO）：false, true, "thin", "fat"
# release 建议开启以减体积提性能
# lto = false

[profile.release]
opt-level = 3
codegen-units = 1      # 单 CGU 以获得最佳优化
lto = true             # 全链接时优化
panic = 'abort'        # 移除 panic 展开代码，减小体积
strip = true           # 剥离符号表（Rust 1.59+）

    自定义 Profile：

    [profile.production]
inherits = "release"
opt-level = 3
lto = "fat"
codegen-units = 1

    使用：cargo build --profile production

    2.6. [registries] —— 私有 Registry¶

    用于企业内部私有 crate 仓库（如基于 GitLab、Cloudsmith、Artifactory）。

    [registries]
my-company = { index = "https://gitlab.example.com/rust/registry" }

# 指定默认 registry（除 crates.io 外）
[registry]
default = "my-company"

# 私有 registry 的认证令牌（通常由 cargo login 写入，不建议手动编辑）
[registry.my-company]
token = "xxxx"

    环境变量¶

    通过设置环境变量可以改变 Cargo 的行为-：

    • CARGO_HOME: 指定 Cargo 的缓存和配置目录-。
    • CARGO_TARGET_DIR: 更改构建产物的输出目录-。
    • RUSTFLAGS: 向 Rust 编译器传递额外的命令行参数。
    • RUST_BACKTRACE: 设置 panic 时是否打印详细回溯信息。

    💡 常见问题与技巧¶

    1. 如何快速获得帮助？ 使用 cargo help <command> 命令可以查看特定子命令的详细帮助文档。
    2. --release 和默认构建的区别？ 默认构建（cargo build）使用 dev 配置，追求快速编译；--release 模式则使用 release 配置，会进行大量优化，产生更高效的可执行文件，但编译时间更长。
    3. 工作区（Workspace）相关选项 在工作区根目录使用 --workspace 可作用于所有成员，或用 -p 选择特定包，用 --exclude 排除部分包。
    4. 处理多个二进制目标 当项目有多个二进制文件时，cargo run 需要配合 --bin 参数指定要运行哪一个。
    5. 依赖更新策略 cargo update 会更新 Cargo.lock 以匹配 Cargo.toml 中指定的版本范围，而 cargo install 用于安装二进制工具。
    6. 代码检查与修复 cargo check 快速检查错误而不生成二进制文件；cargo fix 可自动修复一些编译器警告。
