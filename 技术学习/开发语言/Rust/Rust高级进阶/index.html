    5.元编程¶

    5.1. 元编程是什么？¶

    元编程的本质是：编写能「生成 / 操作 / 扩展 Rust 代码」的代码。

    常规 Rust 代码是「告诉编译器做什么」，元编程代码是「告诉编译器如何生成代码」，最终编译器会把元编程生成的代码与你的业务代码合并，一起编译为二进制可执行文件。

    5.2. Rust 元编程的核心边界¶

    Rust 不支持「运行时反射 (Reflection)」，这是刻意的设计选择：

    • 反射会带来运行时性能损耗、二进制体积膨胀、类型安全弱化；
    • 反射的核心能力（获取类型元信息、动态调用方法等），在 Rust 中都通过「编译期元编程」实现，且做到了零运行时成本的等价替代。

    5.3. Rust元编程中的宏¶

    1.简介¶

    在 Rust 中宏分为两大类：声明式宏( declarative macros ) macro_rules! 和三种过程宏( procedural macros ):

    • 派生宏：#[derive]，可以为目标结构体或枚举派生指定的代码，如 #[derive(Debug, Clone)]

    • 类属性宏(Attribute-like macro)：用于为目标添加自定义的属性

    // 假设有一个路由属性宏
#[route(GET, "/users")]
fn get_users() { /* ... */ }
// 宏可以读取 GET 和 "/users"，并生成相应的路由注册代码
    • 类函数宏(Function-like macro)：看上去就像是函数调用
    // 定义
#[proc_macro]
pub fn sql(input: TokenStream) -> TokenStream { /* 解析 SQL 字符串并生成代码 */ }
// 使用
let query = sql!("SELECT * FROM users WHERE id = 1");
    方式 入口 展开阶段 能力 例子
    声明宏 macro_rules! 词法/模式匹配 编译前端 AST 前 文本级替换、可变参数 println! / vec!
    过程宏 #[derive(...)] 函数（操作 AST） 编译中段 AST → AST 类型级代码生成、复杂逻辑 #[derive(Serialize)] / #[tokio::main]

    宏展开 早于类型检查，因此可以「先拼代码，再让 rustc 检查」。

    2.声明宏 (macro_rules!)¶

    声明式宏（关键字 macro_rules!）是 Rust 内置、原生支持的元编程，无需任何外部依赖，是 Rust 程序员最先接触的元编程形式，也是标准库中 vec!/println!/vec![]/format! 等核心宏的实现方式。

    官方命名：Macro by Example (MBE) —— 基于「示例匹配 + 代码展开」的宏，这精准概括了它的工作原理。

    2.1.核心语法与定义方式¶
    // 标准声明式宏定义模板
#[macro_export] // 关键：将宏导出到当前crate的根命名空间，跨模块可用
macro_rules! 宏名称 {
    // 匹配规则1: 模式 => { 展开的代码 }
    (匹配模式1) => { 展开代码块1 };
    // 匹配规则2: 支持多规则重载（类似函数重载）
    (匹配模式2) => { 展开代码块2 };
}
    • #[macro_export] 作用：不加则宏仅在当前模块可见，加上后全局可见（如标准库的宏）；
    • 匹配模式的分隔符是括号 / 方括号 / 大括号，三者等价，只是语法习惯（如 vec![]、println!()）。

    示例:

    // 定义一个简单的宏：创建向量
macro_rules! my_vec {
    // 匹配：my_vec![1, 2, 3]
    ($($x:expr),*) => {
        {
            let mut temp_vec = Vec::new();
            $(temp_vec.push($x);)* // 对每个匹配到的表达式执行 push
            temp_vec
        }
    };
}
// 使用
let v = my_vec![1, 2, 3];
    2.2.核心工作原理：模式匹配 + 代码展开¶

    声明式宏的核心逻辑只有两步，无任何执行逻辑，纯静态匹配：

    1. 匹配阶段：编译器将你调用宏时传入的代码片段，与宏定义中的「匹配模式」做AST 语法树片段匹配（不是字符串匹配！）；
    2. 展开阶段：匹配成功后，编译器将宏的调用处，替换为模式对应的「展开代码块」。

    ✅ 关键区别：Rust 的声明式宏是 AST 片段匹配，而非 C/C++ 的字符串替换，这是 Rust 宏的安全性核心 —— 不会出现字符串替换的语法漏洞，匹配失败会直接编译报错。

    2.3.核心特性：重复匹配（最常用能力）¶

    声明式宏的核心威力来自 重复匹配语法：$(...) sep rep，这是实现「可变参数」的唯一方式，几乎所有实用的声明式宏都依赖它。

    • $(...)：包裹需要重复匹配的代码片段；
    • sep：可选，分隔符（如 , 逗号，最常用）；
    • rep：必须，重复修饰符，只有两种取值：*（零次或多次）、+（一次或多次）。
    #[macro_export]
macro_rules! my_vec {
    // 匹配规则1: my_vec![] —— 空调用，生成空Vec
    () => { Vec::new() };
    // 匹配规则2: my_vec![1, 2, 3] —— 带可变参数的调用，生成初始化Vec
    ($($elem:expr),*) => {{
        let mut v = Vec::new();
        $(v.push($elem);)* // 重复展开：每一个$elem都执行一次push
        v
    }};
}

// 调用
fn main() {
    let v1 = my_vec![]; // 展开为 Vec::new()
    let v2 = my_vec![1,2,3]; // 展开为 push三次的代码
    assert_eq!(v2, vec![1,2,3]);
}
    • $elem:expr：表示匹配「任意表达式 (expression)」，expr 是 Rust 内置的片段指定符，还有 ident(标识符)、ty(类型) 等；
    • 重复匹配是「双向的」：$(...) 匹配了多少个片段，$() 展开就会生成多少行代码。
    2.4.声明式宏的优缺点（选型必看）¶

    ✅ 优点：

    1. 零依赖，原生支持，无需任何 Cargo 配置；
    2. 语法简洁，编写成本低，适合简单场景；
    3. 编译速度快，无额外编译开销；
    4. 安全，AST 匹配无字符串替换漏洞。

    ❌ 缺点：

    1. 能力有限：仅能做「匹配 + 展开」，无逻辑判断、无类型检查、无代码修改能力；
    2. 嵌套复杂：多层匹配会导致宏定义可读性极差；
    3. 无类型感知：无法根据传入的类型做差异化展开。

    3.过程宏¶

    过程宏是 Rust 最强大、最核心的元编程能力，没有之一，也是 Rust 生态繁荣的核心驱动力（serde::Serialize、tokio::main、derive(Debug) 都是过程宏实现）。

    3.1.核心定义¶

    过程宏本质是：一个在编译期被编译器调用的、合法的 Rust 函数。

    它的核心逻辑是：编译器将你标记了宏注解的代码（如结构体、函数），解析为语法树 Token 流 (TokenStream)，传入这个 Rust 函数；函数对 TokenStream 做任意逻辑处理（解析 / 修改 / 生成），最后返回新的 TokenStream；编译器将返回的 TokenStream 直接注入到源代码中，完成编译。

    3.2.过程宏的三大「标配依赖」（缺一不可）¶

    过程宏需要解析 TokenStream（编译器的语法单元），Rust 官方提供了三个核心库，所有过程宏开发都基于它们，被称为「过程宏三板斧」，必须在 Cargo.toml 中引入：

    # 过程宏的Cargo依赖标配
[dependencies]
proc-macro2 = "1.0"  # 兼容标准库proc_macro的TokenStream，增强版
syn = "2.0"          # 核心：将TokenStream解析为Rust语法树(AST)，支持遍历/修改
quote = "1.0"        # 核心：将Rust语法树(AST)转回TokenStream，代码生成神器

    三者分工：syn 负责「读」语法树，quote 负责「写」语法树，proc-macro2 负责「桥接」编译器。

    3.3.硬性要求：过程宏必须是独立的 crate¶

    这是 Rust 编译器的强制规则，无任何例外：

    你的过程宏代码，必须放在一个独立的 Rust 库项目（crate）中，且在 Cargo.toml 中声明 proc-macro = true。

    原因：编译器需要将过程宏编译为「编译器插件」，与业务代码隔离编译，避免循环依赖和编译期污染。

    过程宏 crate 的 标准配置

    # 过程宏专属Cargo.toml，必须写这两行
[lib]
proc-macro = true  # 声明这是一个过程宏crate，编译器会特殊处理
    3.4.三类过程宏¶
    类型 签名 典型用途
    派生宏 #[derive(MyTrait)] fn derive(input: TokenStream) -> TokenStream 自动实现 Serialize / Clone / Debug
    属性宏 #[my_attr(args)] fn attr(args: TokenStream, input: TokenStream) -> TokenStream 注入代码、路由注册 #[get("/")]
    类函数宏 my_macro!(...) fn my_macro(input: TokenStream) -> TokenStream 任意语法块 → 新代码

    最小派生示例：

    use proc_macro::TokenStream;
use quote::quote;
use syn::{parse_macro_input, DeriveInput};

#[proc_macro_derive(Hello)]
pub fn derive_hello(input: TokenStream) -> TokenStream {
    let ast = parse_macro_input!(input as DeriveInput);
    let name = &ast.ident;
    let gen = quote! {
        impl #name {
            pub fn hello(&self) -> &'static str {
                concat!("Hello, I am ", stringify!(#name))
            }
        }
    };
    gen.into()
}

    使用：

    #[derive(Hello)]
struct Cat;

fn main() {
    println!("{}", Cat.hello());   // → Hello, I am Cat
}
    • syn 负责把 TokenStream 解析成结构化 AST
    • quote! 用模板生成新 AST
    • 全程在 编译期 完成，零运行时成本。
    1.派生宏 (Derive Macros) —— 最常用的过程宏¶

    注解语法：#[derive(自定义宏名)]，对标 Rust 原生的 #[derive(Debug, Clone)]。

    核心场景：为结构体 / 枚举 / 联合体，自动生成指定 trait 的实现代码。

    核心特点：标记在 struct/enum/union 上，输入是该类型的语法树，输出是该类型的 trait 实现代码。

    经典示例：实现一个自定义派生宏 #[derive(Hello)]

    作用：为任意结构体自动生成一个 hello() 方法，打印结构体名称。

    // 步骤1：在独立的proc-macro crate中编写派生宏
use proc_macro::TokenStream;
use quote::quote;
use syn::{parse_macro_input, DeriveInput};

// 核心：派生宏的入口函数，#[proc_macro_derive(宏名)] 是固定注解
#[proc_macro_derive(Hello)]
pub fn derive_hello(input: TokenStream) -> TokenStream {
    // 1. 将编译器传入的TokenStream解析为Rust语法树(结构体/枚举)
    let input = parse_macro_input!(input as DeriveInput);
    // 2. 获取结构体/枚举的名称
    let name = input.ident;
    // 3. 生成代码：为该类型实现Hello trait + hello方法
    let gen = quote! {
        pub trait Hello {
            fn hello(&self);
        }

        impl Hello for #name {
            fn hello(&self) {
                println!("Hello, I'm a {}!", stringify!(#name));
            }
        }
    };
    // 4. 将生成的代码转回TokenStream返回给编译器
    gen.into()
}

// 步骤2：在业务crate中调用
#[derive(Hello)]
struct Person { name: String, age: u8 }

fn main() {
    let p = Person { name: "Tom".into(), age: 18 };
    p.hello(); // 自动生成的方法，输出：Hello, I'm a Person!
}

    这就是 serde::Serialize/serde::Deserialize 的核心原理：通过派生宏为你的结构体自动生成序列化 / 反序列化的代码。

    2.属性宏 (Attribute Macros)¶

    注解语法：#[自定义宏名] 或 #[自定义宏名(参数)]，对标 Rust 原生的 #[inline]/#[allow(dead_code)]。

    核心场景：为任意 Rust 语法项（函数、结构体、枚举、模块、变量等）添加自定义注解，实现「装饰器」逻辑，比如：日志打印、性能统计、权限校验、代码注入等。

    核心特点：无限制标记位置，输入是被标记项的语法树，输出是修改后的语法树（可增删改查）。

    经典示例：实现一个日志属性宏 #[log]

    作用：为任意函数自动在执行前后打印「函数名 + 执行开始 / 结束」日志。

    #[proc_macro_attribute] // 固定注解：属性宏的入口标记
pub fn log(_attr: TokenStream, item: TokenStream) -> TokenStream {
    // _attr: 注解的参数（如#[log(debug)]中的debug）
    // item: 被标记的语法项（这里是函数）
    let input = parse_macro_input!(item as ItemFn);
    let name = input.sig.ident; // 获取函数名
    let block = input.block;    // 获取函数体

    // 生成代码：在函数执行前后注入日志打印
    let gen = quote! {
        fn #name() {
            println!("函数 {} 开始执行", stringify!(#name));
            #block
            println!("函数 {} 执行结束", stringify!(#name));
        }
    };
    gen.into()
}

// 调用
#[log]
fn say_hi() {
    println!("Hello World!");
}
// 调用say_hi()会输出：
// 函数 say_hi 开始执行
// Hello World!
// 函数 say_hi 执行结束

    tokio::main/tokio::test 就是属性宏，它的作用是将普通函数包装为异步运行时的入口函数。

    3.函数式宏 (Function-like Macros)¶

    调用语法：自定义宏名!(参数)，对标 Rust 原生的 vec!()/println!()。

    核心场景：替代声明式宏，实现「更复杂的宏调用逻辑」，比如需要类型检查、条件生成的宏。

    核心特点：调用方式和声明式宏一致，但内部是完整的 Rust 函数逻辑，能力远超声明式宏。

    注解标记：#[proc_macro]

    3.5.过程宏的优缺点（选型核心）¶

    ✅ 优点（绝对主流）：

    1. 能力无上限：可以实现任何你能想到的代码生成逻辑，是 Rust 元编程的核心；
    2. 强类型感知：能解析语法树，做类型检查、语法校验，生成的代码绝对安全；
    3. 生态核心：几乎所有 Rust 优秀库都依赖过程宏，是 Rust 生态的基石；
    4. 可维护性强：用 Rust 编写逻辑，比嵌套的声明式宏可读性高得多。

    ❌ 缺点：

    1. 有一定学习成本：需要掌握 syn/quote 的使用；
    2. 编译速度稍慢：过程宏是编译期执行的函数，复杂逻辑会增加编译时间；
    3. 必须独立 crate：增加了项目的工程复杂度。

    5.4.类型级编程¶

    1.核心定位与定义¶

    这是 Rust 元编程中最优雅、最极致、最容易被忽略的核心能力，也是 Rust 类型系统强大的终极体现。

    类型级编程：完全利用 Rust 的类型系统，在「编译阶段」完成计算、逻辑判断、数据存储和状态表达。

    简单说：把「类型」当作「值」来用，把「trait 约束 / 关联类型 / 泛型」当作「逻辑」来用，所有逻辑都在编译期被编译器验证和计算，运行时无任何痕迹、零开销、零错误。

    2 核心本质与核心载体¶

    2.1. 类型级编程的本质：编译期的类型计算¶

    Rust 的类型系统是「图灵完备」的 —— 理论上，你能在 Rust 的类型系统中实现任何可计算的逻辑。

    类型级编程的核心价值：将运行时的逻辑，提前到编译期完成验证和计算，比如：

    • 编译期计算数组的长度是否合法；
    • 编译期验证泛型参数的取值范围；
    • 编译期计算斐波那契数列；
    • 编译期保证枚举的变体数量。
    2.2. 类型级编程的核心载体（无外部依赖，原生支持）¶

    所有类型级编程都基于 Rust 原生的类型系统特性，无需任何宏或第三方库，核心载体包括：

    ✅ 关联类型 (Associated Types)

    ✅ 常量泛型 (Const Generics，Rust 1.59 稳定，核心核心！)

    ✅ Trait 约束与特化 (Trait Bounds & Specialization)

    ✅ 类型别名 (Type Aliases)

    ✅ 零大小类型 (ZST, Zero-Sized Types)

    3.示例¶

    3.1.编译期计算斐波那契数¶
    // 定义一个trait，关联类型存储编译期计算的斐波那契值
trait Fib {
    const VAL: u64;
}

// 基准情况：Fib<0> = 0
struct Fib0;
impl Fib for Fib0 { const VAL: u64 = 0; }

// 基准情况：Fib<1> = 1
struct Fib1;
impl Fib for Fib1 { const VAL: u64 = 1; }

// 递归情况：Fib<n> = Fib<n-1> + Fib<n-2>，编译期计算
struct FibN<N1, N2>(N1, N2);
impl<N1: Fib, N2: Fib> Fib for FibN<N1, N2> {
    const VAL: u64 = N1::VAL + N2::VAL;
}

// 编译期计算斐波那契第10项
type Fib10 = FibN<FibN<FibN<FibN<FibN<Fib1, Fib1>, Fib1>, Fib1>, Fib1>, Fib1>;

fn main() {
    // 编译期就确定了值，运行时直接使用常量，零开销！
    assert_eq!(Fib10::VAL, 55);
}
    3.2.编译期验证数组长度合法性¶
    // 常量泛型 + trait约束：编译期保证数组长度必须大于0
fn safe_array<T, const N: usize>(arr: [T; N]) -> [T; N]
where
    [(); N]: std::marker::ConstParamTy,
{
    // 编译期检查：N为0则直接报错，无运行时panic！
    assert!(N > 0, "数组长度不能为0");
    arr
}

fn main() {
    let arr1 = safe_array([1,2,3]); // ✅ 编译通过
    let arr2 = safe_array([]);      // ❌ 编译报错：assertion failed: N > 0
}

    4.类型级编程的优缺点¶

    ✅ 优点：

    1. 极致性能：零运行时开销，编译期完成所有计算，运行时直接用结果；
    2. 绝对安全：编译期报错，无运行时 panic，无逻辑错误；
    3. 原生支持：无需任何依赖，纯类型系统实现；
    4. 优雅简洁：用类型表达逻辑，代码可读性高，符合 Rust 哲学。

    ❌ 缺点：

    1. 学习成本高：需要对 Rust 类型系统有深入理解；
    2. 语法繁琐：复杂逻辑会导致类型嵌套过深；
    3. 编译速度：极复杂的类型计算会增加编译时间。
