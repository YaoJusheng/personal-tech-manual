    1.简介¶

    在 Python 项目中，setuptools 用于构建和打包代码，tox 用于测试代码在不同的环境下的行为，并确保一致性。下面将对 setuptools、tox 的使用进行详解，同时介绍如何生成文档和发布包到 PyPI 的流程。

    参考文档:

    1.Setuptools

    2.tox

    2. 使用 setuptools 构建和打包¶

    setuptools 是一个用于构建、打包和分发 Python 项目的工具。通过创建 setup.py 文件，你可以定义项目的元数据、依赖项和其他配置信息。

    2.0.项目结构¶

    my_package/
├── my_package/                # 主包目录
│   ├── __init__.py            # 包初始化文件
│   └── module1.py             # 其他模块文件
├── tests/                     # 单元测试文件夹
│   ├── __init__.py
│   ├── conftest.py
│   └── test_module1.py
├── dist
│   ├── xxx-0.1.0-py3-none-any.whl
│   └── xxx-0.1.0.tar.gz
├── README.md                  # 项目描述
├── LICENSE                    # 开源协议文件
├── setup.py                   # setuptools 的配置文件
├── pyproject.toml             # 项目配置文件（推荐）
├── MANIFEST.in
├── requirements.txt           # 项目依赖文件
├── docs/                      # Sphinx 文档目录
│   ├── source/
│   │   ├── my_package.module.rst  # my_package/module.py 的 API 文档
│   │   ├── my_package.rst         # my_package 包的 API 文档
│   │   ├── conf.py                # Sphinx 配置文件
│   │   ├── index.rst              # 文档主入口文件, 文档主页（用于Sphinx文档生成）
│   │   ├── usage.rst              # 其他文档内容
│   ├── build/                # 构建输出文件夹
│   │   ├── doctrees/
│   │   └── html/
│   ├── Makefile
│   ├── make.bat
│   └── ...
└── build/
└── tox.ini

    说明:

    • build 目录：Sphinx 在执行构建命令（例如 make html）时，会将生成的文档文件（例如 HTML 文件）输出到这个目录。这个目录通常包含多个子目录，如 html（用于生成的 HTML 文件）、latex（用于生成的 LaTeX 文件）等。

    • doctrees 目录：这是一个用于存储文档树信息的目录。Sphinx 会在第一次构建时生成一个或多个 .doctree 文件，这些文件包含文档结构的缓存，以加快后续构建的速度。该信息对 Sphinx 的增量构建非常重要，特别是在修改文档时，可以减少构建时间。

    2.1 创建 setup.py¶

    2.1.1.示例¶

    setup.py 文件是 Python 项目的构建脚本，setuptools 读取这个文件来构建、打包和安装项目。

    from setuptools import setup, find_packages

setup(
    name="myproject",  # 项目名称
    version="0.1.0",  # 版本
    packages=find_packages(),  # 自动查找项目中的所有包
    install_requires=[
        "requests>=2.25.1",  # 项目依赖的包
    ],
    extras_require={
        'dev': ['tox', 'pytest'],  # 可选依赖项，用于开发环境
    },
    entry_points={
        'console_scripts': [
            'mycommand=myproject.module:function',  # 可执行命令行脚本
        ],
    },
    author="Your Name",
    author_email="youremail@example.com",
    description="A short description of the project",
    url="https://github.com/yourusername/myproject",
    classifiers=[
        'Programming Language :: Python :: 3',
        'License :: OSI Approved :: MIT License',
        'Operating System :: OS Independent',
    ],
)
    • find_packages() 会自动查找项目中的所有包和模块。
    • install_requires 用于定义项目运行时的依赖。
    • extras_require 用于指定开发时的依赖（如 tox、pytest 等）。

    • entry_points 定义可执行脚本。

    • name: 包的名称。

    • version: 包的版本号。
    • author 和 author_email: 作者信息。
    • description 和 long_description: 简单描述和详细描述（可以通过读取 README.md 提供的内容）。
    • packages: 指定需要打包的模块或包。

    2.1.2.选项介绍¶

    1. extras_require

    2. 用途: 定义可选依赖项，使用户可以选择性地安装。

    3. 用法: 使用字典指定不同的可选功能包。
    4. 建议: 使用 extras_require 来管理开发、测试或其他可选功能的依赖项。
    from setuptools import setup

setup(
    name='your-package',
    version='0.1',
    packages=['your_package'],
    install_requires=[
        'numpy',
    ],
    extras_require={
        'dev': ['check-manifest'],
        'test': ['pytest', 'pytest-cov'],
    },
)

    1. setup_requires

    2. 用途: 指定构建和安装包本身所需的依赖项。

    3. 注意: 这些依赖项在安装包之前安装，仅在构建时使用。
    4. 建议: setup_requires 通常用于指定构建工具（如 wheel）。尽量减少使用，以避免复杂的构建依赖链。
    from setuptools import setup

setup(
    name='your-package',
    version='0.1',
    packages=['your_package'],
    setup_requires=[
        'wheel',
        'pytest-runner',
    ],
)
    • wheel: 这个包用于生成 .whl 文件，这是 Python 包的一种分发格式。指定它可以确保生成 wheel 格式的包。

    • pytest-runner: 这是一个用于运行 pytest 的工具。它集成了 pytest，使得可以通过 python setup.py test 命令来运行测试。

    • tests_require

    • 用途: 指定运行测试所需的依赖项。

    • 注意: tests_require 在现代 setuptools 中逐渐被弃用，建议使用 extras_require 结合 test。
    • 建议: 使用 extras_require 中的 test 子集来替代 tests_require，如 extras_require={'test': ['pytest']}。这样可以通过 pip install your-package[test] 来安装测试依赖。
    from setuptools import setup

setup(
    name='your-package',
    version='0.1',
    packages=['your_package'],
    tests_require=['pytest'],
)

    结构化管理: 使用 extras_require 来分离不同环境的依赖项（如开发、测试、文档）。

    减少复杂性: 避免在 setup_requires 中添加过多依赖，以简化构建过程。

    现代化替代: 使用 pyproject.toml 和 PEP 518 配置构建依赖，以支持更多现代化工具链。

    2.2 构建项目¶

    2.2.1.打包¶

    在项目根目录下运行以下命令，构建项目并生成 .whl 文件和 .tar.gz 文件：

    pip install wheel
python setup.py sdist bdist_wheel
python3 setup.py develop  # 开发测试

    构建后，dist/ 目录中会生成 .whl 文件和 .tar.gz 文件。

    • sdist 是用来生成源码分发包的（生成 .tar.gz 文件）。
    • bdist_wheel 是用来生成二进制分发包的（生成 .whl 文件）。

    如果 wheel 模块没有安装，bdist_wheel 命令将无法使用，因此需要安装它。

    2.2.2.包名参数与配置¶

    生成的 .whl 文件名遵循 Wheel 文件格式规范，通常结构为：{distribution}-{version}(-{build tag})?-{python tag}-{abi tag}-{platform tag}.whl。接下来我会详细解释每个参数。

    假设生成的文件名为 xxx-0.1.0-py3-none-any.whl，下面是对该名字中各参数的解析：

    1. xxx: Distribution Name

    2. 分发名：xxx 是包的分发名，通常与你的包的名称一致。这个名称在你的 setup.py 文件中通过 name 参数指定。

    配置方法： 在 setup.py 中通过 name="xxx" 来配置：

    setup(
    name="xxx",
    ...
)

    1. 0.1.0: Version

    2. 版本号：0.1.0 是包的版本号。这个版本号也是在 setup.py 文件中通过 version 参数指定的。它必须符合 PEP 440 版本号规范。

    配置方法： 在 setup.py 中通过 version="0.1.0" 来配置：

    setup(
    version="0.1.0",
    ...
)

    1. py3: Python Tag

    2. Python 版本标签：py3 表示该包支持 Python 3。该标签标识了包与哪些 Python 版本兼容。可以有多个不同的值，例如：

    3. py2：表示支持 Python 2

    4. py3：表示支持 Python 3
    5. cp37：表示仅支持 Python 3.7
    6. cp38.cp39：表示同时支持 Python 3.8 和 Python 3.9

    配置方法：

    • 这个标签由 setup.py 中的 python_requires 参数配置，可以指定支持的 Python 版本范围：
    setup(
    python_requires='>=3.6',
    ...
)

    1. none: ABI Tag (Application Binary Interface Tag)

    2. ABI 标签：none 表示该包不依赖特定的 Python ABI（应用二进制接口）。通常纯 Python 包不需要依赖 ABI，因此显示为 none。如果包包含 C 扩展模块，它可能会指定特定的 ABI，例如 cp37m（Python 3.7 的 ABI）。

    配置方法：

    • 纯 Python 包不需要特定配置。如果你的包包含编译的 C 扩展模块，Python 的 distutils 和 setuptools 会自动检测 ABI 并配置。

    • any: Platform Tag

    • 平台标签：any 表示该包可以在任何平台上运行，这通常用于纯 Python 包。如果你的包依赖于特定的操作系统或架构，它会指定为某个平台，例如 manylinux1_x86_64（Linux x86_64 平台）或 win_amd64（Windows x86_64 平台）。

    配置方法：

    • 纯 Python 包不需要特定配置。如果你的包是平台相关的，可以通过 setuptools 和 distutils 来自动处理平台标签。
    from setuptools import setup

setup(
    name="xxx",
    version="0.1.0",
    python_requires=">=3.6",  # 只支持 Python 3.6 及以上版本
    packages=["xxx"],
)

    这将会生成如下格式的 Wheel 文件：

    xxx-0.1.0-py3-none-any.whl

    2.2.3.Wheel 文件名中的可选字段¶

    1. Build Tag（可选）：

    2. 该部分是可选的，用于区分构建的不同版本（同一版本号下的不同构建）。如果指定了 build 参数，文件名中将会包含类似于 .post1、.dev0 等的后缀。

    配置方法： 在 setup.py 中通过 build 参数配置：

    setup(
    name="xxx",
    version="0.1.0",
    build="1",  # 设置构建号
    ...
)

    对应的文件名可能是 xxx-0.1.0.post1-py3-none-any.whl。

    1. ABI Tag 和 Platform Tag：

    2. 当你构建 C 扩展或依赖于操作系统时，这两个部分会包含更具体的 ABI 和平台标签，生成的文件名可能类似于

      xxx-0.1.0-cp37-cp37m-manylinux1_x86_64.whl

      ，其中：

      • cp37-cp37m 表示针对 Python 3.7 和 Python 3.7 的 ABI。
      • manylinux1_x86_64 表示它支持 Manylinux 1 规范的 x86_64 架构的 Linux 系统。

    2.2.4.Wheel 文件命名总结¶

    一个典型的 Wheel 文件名有以下五个部分：

    {distribution}-{version}(-{build tag})?-{python tag}-{abi tag}-{platform tag}.whl
    1. Distribution Name：包的分发名，例如 xxx
    2. Version：包的版本号，例如 0.1.0
    3. Build Tag（可选）：构建号，例如 .post1（表示第一次后续发布）
    4. Python Tag：表示 Python 版本，例如 py3
    5. ABI Tag：应用二进制接口标签，例如 none（纯 Python 包）
    6. Platform Tag：平台标签，例如 any

    2.3 发布到 PyPI¶

    2.3.1.发布上传¶

    要将构建的包发布到 PyPI，你需要：

    1. 安装 twine，它是发布包的工具：
    pip install twine
    1. 使用 twine 上传包：
    twine upload dist/*

    你需要在 PyPI 上注册并获得 API 令牌，然后在上传时提供凭据。

    2.3.2.配置私有化pypi¶

    2.3.2.1.配置 ~/.pypirc¶

    为了能够将包发布到私有 PyPI 服务器，需要配置 ~/.pypirc 文件。这是一个全局的配置文件，定义了不同的 PyPI 服务器。编辑（或者创建）~/.pypirc，例如：

    [distutils]
index-servers =
    pypi
    private

[pypi]
username = __token__
password = pypi-token

[private]
repository: http://localhost:8080
username: myuser
password: mypassword
    • pypi 部分是上传到官方 PyPI 使用的。
    • private 部分配置的是你的私有 PyPI 服务器地址和认证信息。
    2.3.2.2. setup.py 配置¶

    确保你的 setup.py 文件已经准备好，以下是一个最基本的 setup.py 文件示例：

    from setuptools import setup, find_packages

setup(
    name="mypackage",
    version="0.1.0",
    description="A sample Python package",
    packages=find_packages(),
)
    2.3.2.3. 构建包¶
    python setup.py sdist bdist_wheel

    这将在 dist/ 目录下生成 .tar.gz 和 .whl 文件。

    2.3.2.4. 上传到私有 PyPI¶

    使用 twine 上传包到私有 PyPI 服务器。

    pip install twine
twine upload -r private dist/*

    其中，-r private 表示上传到 ~/.pypirc 文件中定义的 private 仓库。

    2.3.2.5. 上传到 DevPI (如果使用的是 devpi)¶

    对于 devpi，可以直接使用 devpi 命令上传包：

    devpi upload

    2.4. 安装依赖包与版本管理¶

    2.4.1. 使用 pip 安装¶

    为了从私有 PyPI 安装包，使用 --extra-index-url 参数：

    pip install mypackage --extra-index-url=http://localhost:8080/simple

    这会告诉 pip 除了官方的 PyPI 之外，还从 http://localhost:8080/simple 获取包。

    如果你想 仅从私有 PyPI 安装，可以使用 --index-url 参数：

    pip install mypackage --index-url=http://localhost:8080/simple

    说明:

    • ==：锁定到一个精确版本。
    • ~=：允许小版本更新，但不跨越主版本。
    • >=：安装指定版本及其以上版本。
    • <=：安装指定版本及其以下版本。
    • >：安装高于指定版本的版本。
    • <：安装低于指定版本的版本。
    • !=：避免安装特定版本。

    2.4.2.uv¶

    uv: 一个rust写的pip包管理工具, 实现高效快速的安装体验.

    • 安装: pip install --no-cache-dir uv
    • 使用: uv pip install <mypackage>

    2.4.3.检查环境配置¶

    确保你在当前的虚拟环境或系统环境中可以正常导入 <mypackage> 模块。如果该模块尚未安装到当前环境，请在项目根目录下安装：

    pip install -e .

    此命令会通过 setup.py 将当前项目以 "editable" 模式安装到虚拟环境中，使 Sphinx 能够在文档构建时找到并导入 mypackage。

    2.4.4.依赖收集¶

    1.pip¶
    pip freeze > requirements.txt
    2.pipreqs¶

    这个命令会递归地分析项目代码中的导入语句，找出用到的第三方依赖，并将它们写入 requirements.txt 文件中。

    pip install pipreqs

pipreqs ./ > requirements.txt

# 查看包含 UTF-8 BOM 的文件
grep -r -l $'\xef\xbb\xbf' .
pipreqs ./ --ignore "path/to/directory,file_name" --force > requirements.txt

# 多项目依赖处理
ls | grep test | grep -Ev "db|log" | xargs -I {} sh -c "pipreqs ./{}  >> requirements.txt"
# 合并
ls | grep test | grep -Ev "db|log" | xargs -I {} sh -c "echo \"-r ./{}/requirements.txt\"  >> requirements.txt"
    3.pipdeptree¶

    这将展示项目的所有依赖及其依赖关系树。可以根据这个树来手动整理出第三方依赖并写入依赖文件。

    pip install pipdeptree

pipdeptree -p <your_project_name>
    4.poetry¶
    curl -sSL https://install.python-poetry.org | python3 -

poetry add <package_name>

poetry export -f requirements.txt --output requirements.txt

    3. 使用 tox 进行多环境测试¶

    tox 是一个自动化测试工具，它可以在多个 Python 版本和依赖环境中运行测试。

    3.1 创建 tox.ini¶

    tox 的配置文件通常是 tox.ini。你可以在其中定义要测试的 Python 版本、依赖项和测试命令。

    [tox]
envlist = py38, py39, py310  # 要测试的 Python 版本
requires =
  tox-pipenv
  wheel
  setuptools >= 65.0.0
toxworkdir = {toxinidir}/.tox
temp_dir = {toxworkdir}/.tmp

[pytest]
minversion = 6.0
addopts = -ra -q --maxfail=3 --disable-warnings --cov=noeliclight --cov-report=term-missing --cov-report=xml
testpaths = tests

[testenv]
setenv =
  PYTHONPATH = {toxinidir}
deps =
  pytest
  pytest-xdist
  pytest-cov
  -rrequirements.txt
changedir = tests
commands = pytest \
          --cov=noeliclight       \
          --basetemp={envtmpdir}  \
          -n 3                    \
          {posargs}

[testenv:dev]
basepython = python3.10
usedevelop = True

[testenv:.pkg]
basepython = python3.10
usedevelop = True
deps = -rrequirements.txt

[testenv:docs]
description = invoke sphinx-build to build the HTML docs
basepython = python3.10
deps = sphinx >= 7.3.0, <= 8.0.2
commands =
    sphinx-build -d "{toxworkdir}/docs_doctree" doc "{toxworkdir}/docs_out" --color -W -bhtml {posargs}
    python -c 'import pathlib; print("documentation available under file://\{0\}".format(pathlib.Path(r"{toxworkdir}") / "docs_out" / "index.html"))'

    • envlist 定义了要测试的 Python 版本。

    • deps 定义了测试时的依赖项。 deps 的行为:

    • 如果 [testenv] 中定义了 deps，而 [testenv:dev]、[testenv:.pkg] 等没有定义 deps，它们会继承 [testenv] 中的 deps。

    • 如果这些具体环境中定义了自己的 deps，它们的定义会覆盖 [testenv] 中的设置，而不是合并。

    • commands 是运行的测试命令。

    • allowlist_externals = sphinx-build：这是 tox 的配置选项，允许 tox 执行外部命令（即不是 Python 代码或包内的命令），例如 sphinx-build。这里指定了允许调用 sphinx-build 命令（外部工具）。这种做法有助于避免在执行环境中由于外部命令限制引发的错误。

    • sphinx-build -d "{toxworkdir}/docs_doctree" doc "{toxworkdir}/docs_out" --color -W -bhtml {posargs}

    • sphinx-build：Sphinx 文档生成工具，用于将 reStructuredText 或 Markdown 文件生成 HTML、LaTeX 或其他格式的文档。

    • -d "{toxworkdir}/docs_doctree"：指定 Sphinx 存储文档缓存（doctree 文件）的目录，{toxworkdir} 是 tox 的工作目录，docs_doctree 是缓存目录。
    • doc：表示文档源文件所在的目录，通常包含 .rst 或 .md 文件。
    • "{toxworkdir}/docs_out"：表示 Sphinx 生成的 HTML 文档将输出到的目录，这里指定输出到 {toxworkdir} 目录中的 docs_out 文件夹。
    • --color：启用彩色输出，以提高输出的可读性。
    • -W：将所有警告视为错误，这样如果在文档构建过程中产生了警告，构建将会失败。
    • -bhtml：指定构建 HTML 格式的文档（Sphinx 支持生成多种格式，如 LaTeX、man 等，这里指定 HTML 格式）。

    • {posargs}：占位符，用于允许命令行传递额外参数给 tox。运行 tox -e docs -- <extra_args> 时传递的 <extra_args> 会替换 {posargs}。

    • python -c 'import pathlib; print("documentation available under file://{0}".format(pathlib.Path(r"{toxworkdir}") / "docs_out" / "index.html"))'

    • python -c：执行一段 Python 代码，-c 参数表示从命令行直接执行 Python 代码。

    • import pathlib; print(...)：这段代码使用 pathlib 模块来处理文件路径。

      • pathlib.Path(r"{toxworkdir}") / "docs_out" / "index.html"：构建出生成的文档 index.html 文件的绝对路径。
      • file://...：使用 file:// 协议格式，表示文档文件在本地系统的路径。

    • 这段命令的输出将会打印一条消息，告知用户生成的文档文件在本地文件系统的访问路径，例如：

      documentation available under file:///path/to/.tox/docs_out/index.html

    • 继承机制

    • 继承: [testenv] 中定义的选项会被其他所有环境（如 [testenv:dev], [testenv:.pkg], [testenv:docs]）继承。
    • 覆盖: 如果某个选项在具体的环境中被再次定义，具体环境中的定义会覆盖 [testenv] 中的设置。

    3.2 运行 tox¶

    在项目根目录运行 tox：

    tox
tox -r  # 重虚拟环境
tox -rvv # debug模式

tox -e py310
tox -e docs

    tox 会自动创建不同版本的虚拟环境，并在其中运行测试。

    3.3 将 tox 添加到 setup.py¶

    为了确保开发时的依赖一致性，可以将 tox 添加到 setup.py 的 extras_require 中：

    extras_require={
    'dev': ['tox', 'pytest'],
},

    4. 文档生成¶

    Python 中有很多工具可以帮助生成项目的文档，最常用的是 Sphinx。

    4.1 安装 Sphinx¶

    pip install sphinx

    4.2 初始化 Sphinx¶

    在项目根目录下运行以下命令初始化 Sphinx 配置：

    sphinx-quickstart --ext-autodoc --ext-viewcode

    按照提示填写项目名称、作者等信息。该命令会生成一个 docs/ 目录，其中包含 conf.py 文件和初始的 index.rst 文件。

    4.3 配置 conf.py¶

    在 docs/conf.py 中，设置项目路径并配置自动文档生成插件 autodoc：

    import os
import sys
sys.path.insert(0, os.path.abspath('..'))

extensions = ['sphinx.ext.autodoc']  # 启用 autodoc 扩展

    4.4 生成文档¶

    为了生成 HTML 格式的文档，运行以下命令：

    $ sphinx-apidoc -o docs/ .
# or
$ sphinx-apidoc --ext-autodoc --ext-viewcode -o docs <package>

$ pip install -e .
$ make clean && make html

    文档将会生成在 docs/build/html/ 中，你可以打开 index.html 查看生成的文档。

    4.5.在 Read the Docs 上发布文档¶

    4.5.1 连接到 Read the Docs¶

    1. 登录 Read the Docs 并创建一个账户。
    2. 连接 GitHub、GitLab 或其他代码托管平台，并导入你的项目。

    4.5.2 配置 .readthedocs.yml¶

    为了确保 Read the Docs 正确构建项目文档，建议在项目根目录下添加 .readthedocs.yml 配置文件：

    # .readthedocs.yml
version: 2

sphinx:
  configuration: docs/conf.py

python:
  version: 3.8
  install:
    - requirements: requirements.txt
    • sphinx.configuration：指定 conf.py 的路径。
    • python.version：指定 Python 版本。
    • install.requirements：指定项目的依赖项。

    4.5.3 自动构建文档¶

    每次你更新 GitHub 仓库，Read the Docs 会自动构建并发布最新的文档。

    集成到 CI/CD 流程:

    • 一个 GitHub Actions 的示例:
    name: Documentation

on:
  push:
    branches:
      - main

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v2
      - name: Set up Python
        uses: actions/setup-python@v2
        with:
          python-version: '3.x'
      - name: Install dependencies
        run: |
          python -m pip install --upgrade pip
          pip install -r requirements.txt
          pip install sphinx sphinx-rtd-theme
      - name: Build documentation
        run: |
          cd docs
          make html
      - name: Upload documentation
        uses: actions/upload-artifact@v2
        with:
          name: documentation
          path: docs/_build/html/
    • .gitlab-ci.yml
    image: python:3.10

stages:
  - build
  - deploy

variables:
  # Specify the documentation source directory
  DOCS_DIR: "docs"
  BUILD_DIR: "${DOCS_DIR}/build/html"

build_docs:
  stage: build
  script:
    - pip install sphinx sphinx-rtd-theme  # 安装 Sphinx 和所需主题
    - cd $DOCS_DIR
    - make html  # 生成文档
  artifacts:
    paths:
      - $BUILD_DIR  # 保存生成的文档供后续阶段使用

deploy_docs:
  stage: deploy
  script:
    - echo "Deploying documentation to GitLab Pages"
    - mv $BUILD_DIR public  # 将构建的文档移动到 public 目录
  artifacts:
    paths:
      - public  # 将 public 目录中的文档发布为 GitLab Pages
  only:
    - main  # 仅在主分支上运行部署

    说明:

    • Docker 镜像：使用 python:3.10 作为 CI/CD 环境。
    • 阶段：定义两个阶段：build 和 deploy。
    • 变量：使用变量指定文档源目录和构建目录。
    • 构建文档：
    • 在 build_docs 阶段，安装 Sphinx 及其所需的主题（例如 sphinx-rtd-theme）。
    • 切换到文档目录并运行 make html 生成 HTML 文档。
    • 将生成的文档路径保存为构建工件（artifacts），以便后续使用。
    • 部署文档：
    • 在 deploy_docs 阶段，将生成的文档移动到 public 目录。
    • 将 public 目录中的内容发布为 GitLab Pages。
    • 条件：设置仅在 main 分支上执行部署。

    4.6.工具说明¶

    sphinx-apidoc 和 sphinx-quickstart 都是与 Sphinx 相关的命令行工具，但它们的功能不同，用于不同的场景。

    4.6.1. Sphinx-quickstart：快速初始化 Sphinx 项目¶

    sphinx-quickstart 是用来初始化 Sphinx 项目文档的工具，帮助你创建一个新的 Sphinx 项目框架，自动生成一些必要的配置文件、Makefile 和初始文档模板。

    功能:

    • 创建 Sphinx 项目的基本文件结构。
    • 自动生成初始的 conf.py 配置文件。
    • 自动生成 index.rst，作为文档的主入口。
    • 选择是否分离源文件和构建文件（source/ 和 build/ 目录）。
    • 提供选项来配置文档语言、扩展、主题等。

    使用场景:

    sphinx-quickstart 适用于从零开始创建一个 Sphinx 项目，它帮助你创建必要的 Sphinx 项目框架和基础文档模板。在构建项目的基础文档时，你可以使用这个工具。

    使用示例:

    要使用 sphinx-quickstart 创建一个新项目，可以执行以下命令：

    sphinx-quickstart

    你会被问到一些问题，诸如项目名称、作者、文档语言、是否分离 source 和 build 目录等。回答这些问题后，sphinx-quickstart 将为你生成一个基本的 Sphinx 项目结构。

    输出结果:

    Project name: MyProject

    • 简单快速创建一个新的 Sphinx 项目。
    • 自动生成 conf.py，你可以根据项目的需求进行修改。
    • 初始模板易于扩展和自定义。

    4.6.2. Sphinx-apidoc：自动生成 API 文档¶

    sphinx-apidoc 是一个用于根据 Python 代码的 docstring 自动生成 API 文档的工具。它可以扫描给定的 Python 包或模块目录，生成 .rst 文件。这些 .rst 文件会包含从代码的注释和 docstring 中提取的 API 文档。

    功能:

    • 根据代码中的 docstring 自动生成 .rst 文档。
    • 针对模块、类、方法和函数生成详细的 API 文档。
    • 为 Sphinx 项目生成文档结构的 .rst 文件。
    • 提供多个选项以控制生成文档的细节（例如排除特定模块、递归生成等）。

    使用场景:

    sphinx-apidoc 适用于你已经有一个 Python 项目，并且想要自动生成该项目的 API 文档时使用。你不需要手动编写 .rst 文件，而是根据代码中的注释和 docstring 自动生成。

    使用示例:

    假设你有一个 Python 包或模块，存储在 my_package 目录下，你可以运行以下命令生成 API 文档的 .rst 文件：

    sphinx-apidoc -o docs/source my_package/

    输出结果:

    此命令会在 docs/source/ 目录下生成多个 .rst 文件，每个文件对应 Python 项目中的不同模块和包。例如，如果你的 my_package 包有一个 module.py 模块，sphinx-apidoc 将生成如下 .rst 文件：

    docs/
├── source/
│   ├── my_package.module.rst  # my_package/module.py 的 API 文档
│   ├── my_package.rst         # my_package 包的 API 文档
│   └── ...
├── Makefile
└── build/

    在生成的 .rst 文件中，Sphinx 会通过 autodoc 扩展从代码中的 docstring 自动提取注释并生成文档。

    优点:

    • 自动生成 .rst 文件，避免手动编写 API 文档。
    • 自动生成类、方法、函数的详细文档，尤其适合大型项目。
    • 支持递归扫描包和模块。

    4.6.3. Sphinx-quickstart 和 Sphinx-apidoc 的区别¶

    • 功能不同：
    • sphinx-quickstart 用于初始化一个 Sphinx 项目，生成初始的文档和配置文件。它是创建文档项目的第一步。
    • sphinx-apidoc 则是生成 API 文档的工具，帮助你基于现有的 Python 代码生成 API 文档。
    • 使用场景不同：
    • sphinx-quickstart 适用于从零开始创建 Sphinx 项目，包括生成文档结构和配置。
    • sphinx-apidoc 适用于已经有代码的情况下，自动生成项目的 API 文档。
    • 输出结果不同：
    • sphinx-quickstart 输出整个 Sphinx 项目的结构，包括配置文件、Makefile 等。
    • sphinx-apidoc 只生成 .rst 文件，针对 Python 包和模块的 API 文档，依赖 autodoc 来解析 docstring。

    4.6.4. Sphinx-quickstart 和 Sphinx-apidoc 组合使用¶

    可以先使用 sphinx-quickstart 创建 Sphinx 项目的基础框架，然后使用 sphinx-apidoc 自动生成项目的 API 文档。

    示例流程：

    1. 使用 sphinx-quickstart 创建 Sphinx 项目：
    sphinx-quickstart
    1. 使用 sphinx-apidoc 生成 API 文档：
    sphinx-apidoc -o docs/source my_package/
    1. 将生成的 .rst 文件添加到 index.rst 中，确保 Sphinx 可以正确地包含 API 文档。例如，将以下行添加到 index.rst 中：
    .. toctree::
   :maxdepth: 2
   :caption: Contents:

   my_package
    1. 构建文档：
    make html

    最终，API 文档将被集成到生成的 HTML 文档中。

    5. 发布流程¶

    5.1 本地打包¶

    1. 首先，确保你已经更新了 setup.py 文件。
    2. 运行以下命令来生成分发包：
    python setup.py sdist bdist_wheel
    1. 验证包是否构建正确：
    twine check dist/*

    5.2 上传到 PyPI¶

    使用 twine 将包上传到 PyPI：

    twine upload dist/*

    5.3 预发布测试 (TestPyPI)¶

    在发布到正式的 PyPI 前，可以使用 TestPyPI 进行预发布测试：

    1. 上传包到 TestPyPI：
    twine upload --repository testpypi dist/*
    1. 安装 TestPyPI 包：
    pip install --index-url https://test.pypi.org/simple/ --extra-index-url https://pypi.org/simple your-package
