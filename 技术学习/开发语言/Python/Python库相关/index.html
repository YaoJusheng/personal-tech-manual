    1.常用内置库¶

    1.1.标准库整理¶

    • functools: 一系列评估高阶函数的接口
    • itertools: 一个创建快速、高效迭代器的模块。官方文档
    • operator: 侵犯相关
    • collections：实现了一些特定的数据类型，可以替代Python中常用的内置数据类型
    • threading: 线程库
    • multiprocessing：进程库
    • math：数学方法
    • asyncio： 异步操作
    • uvloop: 一个基于 libuv 的高性能异步事件循环（event loop）,可以提高异步应用的性能。
    • time: 时间模块
    • datetime: datetime模块是对时间模块的一个高级封装(时间包基于C语言的库函数)
    • random模块: Python 中的 random 模块用于生成随机数
    • re: 正则模块
    • sys模块: sys 模块提供了一系列有关Python运行环境的变量和函数
    • os模块：os模块包含通用的网络功能，如文件操作、目录等，与具体的平台无关
    • subprocess：子进程管理模块，常用call()、run()、Popen()等方法。官方文档
    • shutil: 对文件(夹)、压缩包进行处理的模块（移动/复制/压缩/解压/修改权限）。
    • hashlib/hmac/uuid: 消息摘要/加密/全局唯一标识符
    • pickle: 序列化处理模块，常用dump()、load()方法。
    • configparser:ini配置文件解析模块。
    • logging: 内置模块，日志相关。
    • optparse: 命令行解析模块。
    • docopt: 命令行解析模块。
    • json: JSON 文件处理模块。
    • csv: CSV 文件处理模块。
    • xml.etree.ElementTree: 内置模块，解析 XML 等文件。
    • tempfile: 内置模块，临时文件操作
    • timeit: 测量小代码片段的执行时间。官方文档
    • ...

    1.2.基础部分¶

    • 常用标准库（functools/itertools/opertor/collections）
    • 推导式/高阶函数： map/filter/reduce
    • map 是用于映射的，它将一个函数应用到序列的每个元素。
    • filter 用于过滤，它保留那些满足指定条件的元素。
    • reduce 用于归约，它将序列的元素进行累积计算，最终返回一个值。
    • nametuple/dataclass/pydantic
    • namedtuple 是 collections 模块中的一个工厂函数，用于创建不可变的类
      • 优点：轻量级、内存效率高、可以通过名字访问字段、默认不可变。
      • 适用场景：轻量的数据结构，只需要存储简单的属性数据且不需要对数据进行验证和更改
    • dataclass 引入于 Python 3.7，是一种简单的类装饰器，用于减少样板代码，可以自动生成常用的特殊方法（如 __init__、__repr__ 等）。
      • 优点：支持可变和不可变模式（通过 frozen=True），可以定义默认值、类型提示和数据验证（通过 __post_init__）。
      • 适用场景：需要简单的数据存储类，并且可能需要一些轻量的验证和变换。
    • pydantic是一个数据验证和设置管理的库，基于 Python 的类型注解。是一个非常强大的数据类。
      • 优点：提供强大的数据验证、解析和序列化功能，支持 JSON schema 生成和验证、并且能处理复杂的嵌套数据结构。
      • 适用场景：需要严格的数据验证、数据解析，以及 JSON API 交互时的数据验证。
    • metaclass/ 动态继承/type
    • metaclass 是用来定义类的类。也就是说，metaclass 控制类的创建行为。通常类是通过 type 来创建的，而 metaclass 可以让你在类创建的时候插入逻辑，比如修改类的属性、方法等。
    • 使用场景:
      • 需要在框架层面上动态控制类的行为。
      • 需要在 ORM 系统中根据不同的数据库模式动态创建不同的模型类。
      • 动态修改第三方库的行为，或者对类进行装饰，而不直接修改类的代码。
    • 生成器/迭代器的理解
    • 迭代器是一个对象，它实现了两个方法：__iter__() 和 __next__()。迭代器可以记住遍历的位置，每次调用 __next__() 方法时，迭代器返回容器的下一个元素。当没有更多元素时，迭代器会抛出 StopIteration 异常。
    • 生成器是简化迭代器创建的特殊函数，它们允许你使用 yield 关键字来产生值。生成器在每次生成一个值时会“暂停”，并在需要时恢复执行。这使得生成器非常高效，因为它们可以按需生成序列的元素，而不是一次性将所有元素生成。本质上就是一种特殊的迭代器,
    • 适用场景：
      • 生成器可以来处理大文件
      • 结合高阶函数与生成器实现流式数据处理
      • 使用生成器表达式代替列表推导式
    • 同步/异步（asyncio/yield）以及阻塞与非阻塞
    • 同步操作指的是任务按顺序执行，即当前任务必须等待前一个任务完成后才能开始。同步操作是阻塞的，执行线程会一直等待任务完成。
    • 异步操作指的是任务可以并发执行，即任务不会阻塞执行线程。当一个任务开始执行后，可以立即切换去执行其他任务，原任务会在未来的某个时间点完成。
    • 阻塞调用是指程序在执行某个操作时，必须等待操作完成后才能继续执行下一个指令。如果一个函数调用会使程序等待某个条件满足（如 I/O 完成或锁释放），这就是阻塞调用。
    • 非阻塞调用是指程序在执行某个操作时，如果条件不满足不会等待，而是立即返回。例如，如果一个函数试图读取数据，但数据尚未准备好，非阻塞调用会立即返回，而不是等待数据变得可用。
    • 场景:
      • 同步阻塞: 普通的 I/O 操作。
      • 同步非阻塞: 轮询检查的方式可以视作同步非阻塞。
      • 异步阻塞: 使用 asyncio.run 时，如果在事件循环中直接调用一个阻塞的操作。
      • 异步非阻塞: 使用 asyncio 的异步 I/O。异步非阻塞通常与回调机制或 async/await 语法结合使用。通常依赖于事件循环或回调机制，当一个异步操作被触发时，事件循环会继续处理其他任务，而不等待当前任务完成。
    • 装饰器的使用/如何编写一个类装饰器？它与函数装饰器有什么区别？/如何在装饰器中维护状态，确保每次调用函数时都能记住之前的状态？
    • 装饰器本质上是一个函数，它接受另一个函数作为参数，并返回一个新的函数或修改后的函数。装饰器通常用于扩展或修改被装饰函数的行为，而无需更改函数的代码。
    • 装饰器通过闭包（closures）实现
    • 当一个函数被装饰时，它的元数据（如函数名、文档字符串等）可能会被装饰器覆盖。为了保留被装饰函数的元数据，可以使用 functools 模块中的 @wraps 装饰器。
    • 类装饰器本质上是一个类，该类实现了 __call__ 方法，使其行为与函数装饰器类似。类装饰器与函数装饰器的区别在于，类装饰器可以利用类的属性和方法，使其在实现复杂功能时更具灵活性。
    • 可以在装饰器内部定义状态变量，或者使用类装饰器将状态存储为类的属性。以此记住之前的状态
    • 内存管理/进程-线程-协程/GIL：
    • 内存管理：
      • 引用计数：Python 使用引用计数机制来管理内存，每个对象都有一个计数器，用于记录有多少个引用指向它。当一个对象的引用计数为零时，该对象会被立即销毁并释放内存。
      • 垃圾回收：除了引用计数，Python 还使用垃圾回收机制来处理循环引用。Python 的垃圾回收器会定期检查对象之间是否存在循环引用，并在没有其他引用的情况下将其回收。
      • 内存分配：Python 的内存分配由内置的内存管理器管理，该管理器会根据需要分配内存，并优化小对象（小于 512 字节）的分配效率。
    • 内存泄漏：内存泄漏是指内存中某些不再使用的对象无法被回收，从而导致内存占用不断增加。避免方法：
      • 消除循环引用：尽量避免循环引用，或者使用 weakref 模块创建弱引用。
      • 手动删除对象：使用 del 关键字显式删除不再需要的对象。
      • 限制全局变量的使用：避免过多使用全局变量，因为全局变量的生命周期较长，可能会导致内存不能及时释放。
    • GIL 是 CPython 解释器中存在的一种机制，它保证同一时刻只有一个线程执行 Python 字节码，确保多线程操作的原子性。
      • 在 CPU 密集型任务中，多线程程序在每次切换线程时都会因为 GIL 的竞争导致频繁的上下文切换，反而降低了程序的执行效率。
      • 在 I/O 密集型任务中，由于线程大部分时间处于等待 I/O 操作的状态，GIL 的影响较小，性能影响不明显。

    1.3.异步理解¶

    trio、anyio 和 asyncio 是 Python 中用于编写异步并发程序的库。

    1.3.1. asyncio¶

    联系:

    • 核心库: asyncio 是 Python 标准库中的异步 I/O 库，从 Python 3.4 开始引入。它是异步编程的核心，提供了事件循环、任务、协程等基本机制。
    • 流行度: 由于其标准库的地位，asyncio 是使用最广泛的异步编程工具，几乎所有主流的异步框架（如 FastAPI、Tornado、aiohttp）都基于 asyncio。

    区别:

    • 复杂度: asyncio 的 API 被认为比较复杂，尤其是处理异常和取消任务的场景。它的设计有些低层次，允许用户对事件循环进行大量控制，但这也带来了使用上的复杂性。
    • 例外处理: 处理任务的取消和异常传播相对复杂，需要手动管理任务和 Future 对象。
    • 兼容性: 由于它是标准库的一部分，asyncio 与 Python 的兼容性和集成性是最好的，许多库和框架也都围绕 asyncio 构建。

    选择:

    • 如果你在使用依赖于 asyncio 的现有框架或库（如 FastAPI、aiohttp），或者你需要构建一个与 Python 生态系统广泛兼容的项目，asyncio 是默认的选择。

    1.3.2. trio¶

    联系:

    • 高层库: trio 是一个第三方库，专注于更高层次的异步编程接口，提供了更加简单和直观的 API。它主要面向异步任务管理、取消和异常处理等复杂场景的简化。
    • 实验性: trio 倡导“structured concurrency”即结构化并发的编程理念，这一理念在一定程度上影响了 Python 3.11 中引入的 TaskGroup。

    区别:

    • 简单性: trio 的 API 被设计得更为简单和直观，尤其是在任务取消和异常传播方面。它使用了结构化并发（structured concurrency），使得任务的生命周期更容易管理。
    • 无回调: trio 几乎完全避免了使用回调（callback），这使得代码更加线性和容易理解。
    • 兼容性: 因为 trio 是一个第三方库，它的生态系统相对 asyncio 更小。如果你依赖的库不支持 trio，你可能需要额外的工作来集成。

    选择:

    • 如果你正在开发一个新的异步项目，并且希望尽可能减少异步编程的复杂性，trio 是一个不错的选择。它的 API 更加友好，且适合初学者。
    • 如果你需要开发对错误处理、任务取消、以及任务调度有更严格控制的系统，trio 提供了更强大的工具。

    1.3.3. anyio¶

    联系:

    • 桥接库: anyio 是一个用于在 asyncio 和 trio 之间提供兼容性的库。它提供了一个统一的 API，可以在两者之间无缝切换。
    • 异步工具集: anyio 提供了一套高级的异步工具，旨在使代码更容易在 trio 和 asyncio 上运行。

    区别:

    • 多后端支持: anyio 的设计目标是成为 asyncio 和 trio 之间的桥梁。它允许你编写的代码可以在这两者之间切换，而无需进行大量修改。
    • 结构化并发: anyio 通过统一的 API 提供了结构化并发支持，类似于 trio，但也能在 asyncio 上使用。
    • 生态系统: anyio 的生态系统和社区支持比 asyncio 和 trio 稍小，因为它主要是一个兼容性层。

    选择:

    • 如果你希望编写同时兼容 trio 和 asyncio 的代码，或者你想在现有项目中引入结构化并发的概念，anyio 是一个非常好的选择。
    • 如果你正在开发一个库或工具，并且希望它能与不同的异步运行时兼容，那么 anyio 是最佳选择。

    1.3.4.uvloop¶

    基本概念:

    • 高性能事件循环: uvloop 是一个高性能的事件循环实现，它是完全兼容 asyncio 的替代品。uvloop 主要基于 libuv 库，这也是 Node.js 中使用的底层 I/O 引擎。
    • 完全兼容: uvloop 与 asyncio 完全兼容，因此你可以无缝地在现有的 asyncio 代码中使用 uvloop。使用 uvloop 只需将事件循环设置为 uvloop 提供的循环即可。

    优点:

    • 高性能: uvloop 显著提高了 I/O 操作的性能，尤其是在处理大量网络请求时。官方基准测试显示，与默认的 asyncio 事件循环相比，uvloop 的性能通常要高出 2 倍到 4 倍。
    • 高效的系统调用: 由于 uvloop 使用了 libuv 底层库，它在处理系统调用（如网络 I/O）方面非常高效，减少了上下文切换和内核态与用户态之间的开销。

    使用方式：要在 asyncio 程序中使用 uvloop，只需简单地将 uvloop 设置为默认事件循环：

    import asyncio
import uvloop

asyncio.set_event_loop_policy(uvloop.EventLoopPolicy())

# 或全局使用:
uvloop.install()
    1.3.4.1.设置uvloop¶

    1. asyncio.set_event_loop_policy(uvloop.EventLoopPolicy()):

    2. 这行代码的作用是设置一个新的事件循环策略（event loop policy），并用 uvloop 替换掉 Python 的默认事件循环。

    3. asyncio.set_event_loop_policy() 函数允许你为当前应用程序设置事件循环策略。这里传入的 uvloop.EventLoopPolicy() 会告诉 asyncio 使用 uvloop 来创建新的事件循环。

    4. 效果：该行代码启用后，FastAPI（或其他基于 asyncio 的框架）将在程序启动时使用 uvloop 作为默认的事件循环。

    5. uvloop.install():

    6. 这行代码是 uvloop 提供的一个简便的方法来安装和激活 uvloop 事件循环。调用 uvloop.install() 后，uvloop 会被设置为当前的全局默认事件循环，不再需要手动设置事件循环策略。

    7. 效果：启用后，uvloop 会替代 Python 默认的 asyncio 事件循环，直接生效。

    区别：

    • asyncio.set_event_loop_policy(uvloop.EventLoopPolicy())：通过显式设置 asyncio 的事件循环策略，告诉它在创建新的事件循环时，应该使用 uvloop。这可以为更复杂的场景提供灵活性，比如不同的线程使用不同的事件循环。
    • uvloop.install()：这是一个简单的“安装”操作，直接将 uvloop 设置为默认的事件循环，实现起来更简洁，并且全局生效。

    使用场景：

    • 你可以选择两者之一来替换默认的 asyncio 事件循环。通常uvloop.install() 更加简便，适用于大多数情况下希望全局替换事件循环的场景。

    • 你可能会启用这两行中的一行，以提高应用程序的性能。特别是像 FastAPI 这样的异步 Web 框架，它们可以通过 uvloop 受益于更快的网络 I/O 处理，从而加速接口响应。

    • 如果你希望全局启用 uvloop：只需使用 uvloop.install() 即可。这是最简单的方式，并且适用于大多数场景。

    import uvloop
uvloop.install()  # 简单地全局启用 uvloop
    • 如果你有复杂的事件循环管理需求（例如多线程处理）：使用 asyncio.set_event_loop_policy() 提供更灵活的控制。通过这种方式，你可以在不同线程中使用不同的事件循环策略。
    import uvloop
import asyncio
asyncio.set_event_loop_policy(uvloop.EventLoopPolicy())  # 显式设置策略

    性能影响：

    • 使用 uvloop 可以显著提升基于 asyncio 的异步应用程序的性能，特别是在处理大量并发 I/O 操作（如 HTTP 请求、数据库查询等）时。

    4.0.1.WSGI/ASGI¶

    1.WSGI (Web Server Gateway Interface)¶

    • 定义: WSGI 是一种用于将 Web 服务器与 Python Web 应用连接的标准接口。它为应用服务器和 Web 应用之间的通信提供了标准化的协议。

    • 特性:

    • 适用于同步 Web 应用。

    • 简单的请求-响应模型。

    • 是 Python Web 框架如 Flask、Django 的默认协议。

    • 示例:

    def simple_app(environ, start_response):
    status = '200 OK'
    headers = [('Content-type', 'text/plain')]
    start_response(status, headers)
    return [b"Hello, World!"]

# 此函数会被 WSGI 服务器调用

    • 缺点:

    • 不支持异步请求，因而不适用于需要高并发、实时通信或 WebSocket 的应用。

    2.ASGI (Asynchronous Server Gateway Interface)¶

    • 定义: ASGI 是 WSGI 的异步版本（异步服务器网关接口），它支持异步应用程序和协议，如 WebSocket、HTTP/2 等。

    • 特性:

    • 支持异步 Web 应用。

    • 可处理多种协议（HTTP、WebSocket）。

    • 更适合现代 Web 应用场景，如实时数据推送、长连接等。

    • 示例:

    async def app(scope, receive, send):
    assert scope['type'] == 'http'
    await send({
        'type': 'http.response.start',
        'status': 200,
        'headers': [
            (b'content-type', b'text/plain'),
        ]
    })
    await send({
        'type': 'http.response.body',
        'body': b'Hello, World!',
    })

# 此函数会被 ASGI 服务器调用

    • 优势:

    • 支持异步任务，有助于提高并发性和性能。

    • 更适合 WebSocket、实时聊天、推送通知等场景。

    4.0.2.常见 Web 服务器¶

    1.Gunicorn (Green Unicorn)¶

    • 定义: Gunicorn 是一个用于运行 Python WSGI 应用程序的 Web 服务器。

    • 特性:

    • 简单、高效的 WSGI 应用服务器。

    • 支持多进程模型，适合处理高并发请求。

    • 通过 pre-fork 模式处理请求，减少了创建新进程的开销。

    • 示例:

    gunicorn myapp:app -w 4 -b 0.0.0.0:8000

    这条命令将启动一个 WSGI 应用，使用 4 个工作进程（workers）。

    • 用途: Gunicorn 常用于部署 Flask、Django 等 WSGI 应用，是生产环境下的常见选择。
    2.uWSGI¶

    • 定义: uWSGI 是一个功能强大的应用服务器，支持多种协议（包括 WSGI、HTTP、FastCGI 等）。它不仅可以运行 WSGI 应用，还能处理管理任务（如进程管理、负载均衡等）。

    • 特性:

    • 支持 WSGI、uWSGI、FastCGI、HTTP、ASGI（通过插件）等多种协议。

    • 适用于生产环境，提供强大的进程管理和监控能力。

    • 通过配置文件或命令行参数进行高度定制化。

    • 示例: 启动一个 WSGI 应用：

    uwsgi --http :8080 --wsgi-file myapp.py
    • 用途: uWSGI 通常用于在生产环境中部署 WSGI 应用程序，但它也支持 ASGI，适用于高性能、高并发的 Web 应用。
    3.Uvicorn¶

    • 定义: Uvicorn 是一个用于运行 ASGI 应用程序的高性能 Web 服务器，基于 uvloop 和 httptools 构建。

    • 特性:

    • 支持异步 ASGI 应用，特别适合 FastAPI、Starlette 等异步框架。

    • 极高的性能表现，特别适合 I/O 密集型应用。
    • 可以通过命令行选项调整行为，如开启 HTTP/2、WebSocket 支持。

    • 单一进程：默认情况下，Uvicorn 运行在单个进程中，主要依赖于异步 I/O 来处理并发。

    • 示例:

    uvicorn myapp:app --host 0.0.0.0 --port 8000
    • 用途: Uvicorn 是运行 ASGI 应用（如 FastAPI）的理想选择，适合需要高并发、低延迟的 Web 应用。
    4.Hypercorn¶

    • 定义: Hypercorn 是一个支持 ASGI 和 WSGI 的 Web 服务器，提供对 HTTP/2 和 HTTP/3 的支持。

    • 特性:

    • 支持 ASGI 和 WSGI 应用，能够运行如 Flask、Django、FastAPI 等框架的应用。

    • 提供对 HTTP/2、HTTP/3、WebSocket 的支持。

    • 灵活的配置和扩展性。

    • 示例:

    hypercorn myapp:app --bind 0.0.0.0:8000
    • 用途: Hypercorn 是一个多功能的 Web 服务器，适合需要现代协议支持（如 HTTP/2、WebSocket）的 Web 应用。
    5.Daphne¶

    • 定义: Daphne 是 Django Channels 的官方 ASGI 服务器，实现了 HTTP、HTTP/2、WebSocket 和 HTTP 长轮询。

    • 特性:

    • 主要用于 Django ASGI 应用，如 Django Channels 项目。

    • 支持 WebSocket 和 HTTP/2，适合实时通信和异步任务。

    • 可以与 ASGI 中间件和协议栈配合使用。

    • 示例:

    daphne myapp:app -b 0.0.0.0 -p 8000
    • 用途: Daphne 是 Django Channels 项目的推荐 ASGI 服务器，特别适合需要 WebSocket 支持的 Django 应用。

    4.2.启动时运行¶

    启动时运行：一般来说，希望服务器程序（例如 Uvicorn）在服务器启动时自动启动，并且不需要任何人为干预，让进程始终与您的 API 一起运行（例如 Uvicorn 运行FastAPI 应用程序） 。

    自动重新启动：客户端将收到 500 内部服务器错误，但应用程序将继续处理下一个请求，而不是完全崩溃。尽管如此，在某些情况下，一些代码可能会导致整个应用程序崩溃，此时需要一个工具可以负责重新启动应用程序。

    可以完成这项工作的工具的一些示例是：

    • Docker
    • Kubernetes
    • Docker Compose
    • Docker in Swarm Mode
    • Systemd
    • Supervisor

    4.3.进程/内存¶

    必须有一个单个组件来处理公共IP中的端口。 然后它必须有一种方法将通信传输到复制的进程/worker。

    以下是一些可能的组合和策略：

    • Gunicorn管理Uvicorn workers
    • Gunicorn 将是监听 IP 和 端口 的 进程管理器，复制将通过 多个 Uvicorn 工作进程 进行
    • Uvicorn管理Uvicorn workers
    • 一个 Uvicorn 进程管理器 将监听 IP 和 端口，并且它将启动 多个 Uvicorn 工作进程
    • Kubernetes和其他分布式容器系统
    • Kubernetes 层中的某些东西将侦听 IP 和 端口。 复制将通过拥有多个容器，每个容器运行一个 Uvicorn 进程
