    1.py文件简介¶

    在 Python 中，常见的文件后缀包括 .py、.pyc、.pyo、和 .pyd。这些文件扩展名代表了不同类型的 Python 文件。 总结：

    • 在开发过程中，主要使用 .py文件，因为它们包含源代码，允许修改和调试。
    • .pyc和 .pyo 文件用于优化加载速度，特别是在生产环境中。.pyo 文件相对于 .pyc 文件提供更多的优化。
    • .pyd 文件是 Windows 平台上用于 Python 扩展的，不需要直接处理它们，通常是由第三方库提供的。
    • 最佳实践取决于使用环境和需求。在开发过程中，主要关注 .py 文件，而在部署或生产环境中，考虑使用 .pyc 或 .pyo 文件来提高加载速度。.pyd 文件一般是由第三方库提供的，用于与底层系统进行交互。

    1.1. .py文件¶

    1. 描述： .py 文件是标准的 Python 源代码文件，包含可执行的 Python 代码。这些文件包含 Python 代码，可以使用 Python 解释器直接运行。

    2. 使用场景：源码开发、修改和调试，用于编写和运行 Python 代码。

    3. 生成方式：手动编写 Python 代码。

    4. 最佳实践：在开发阶段使用，需要修改、开发新功能或调试时，使用 .py 文件。

    5. 使用流程：你可以在.py 文件中编写 Python 代码，并通过命令行或集成开发环境（IDE）来运行这些代码。在命令行中使用 python your_file.py 来执行其中的代码。

    1.2. .pyi文件¶

    1. 描述：.pyi 文件通常是 Python 的类型提示文件（Type Hinting），用于提供代码的静态类型信息，以帮助 IDE、静态分析工具和类型检查器更好地理解代码。

    2. 使用场景：

    3. 编写类型提示：在一个独立的 .pyi 文件中，对函数、类等进行类型声明，例如：

    # example.pyi
def add(a: int, b: int) -> int: ...
    • 类型检查：使用类型检查工具，如 Mypy，在代码中引入 .pyi 文件，并进行类型检查：
    mypy your_code.py --ignore-missing-imports
    • 提供类型信息：将 .pyi 文件与相应的 Python 文件一起提供，让 IDE 自动识别和使用。

    .pyi 文件通常用于对现有代码进行类型提示补充，使得代码更具可读性、可维护性，并且能够通过类型检查工具发现潜在的类型错误。

    1.3. .pyc文件¶

    1. 描述： .pyc 文件是 Python 编译文件（Python Compiled File）的一种形式。它是由 Python 解释器根据 .py 源文件编译生成的字节码文件。

    2. 使用场景： 需要 Python 源代码编译生成的字节码文件，提高程序的加载速度。

    3. 生成方式：自动生成，当你运行 .py 文件时，Python 解释器会在同一目录下生成相应的 .pyc 文件。

    4. 最佳实践： .pyc 文件加速 Python 程序的启动时间，特别是在多次运行相同代码时。这些文件可帮助减少代码加载时间，但如果源代码经常变动，它们可能会变得不稳定。

    5. 使用流程：当你运行一个 Python 脚本（.py 文件）时，Python 解释器会在运行时把源代码编译成字节码，然后在同一目录下生成 .pyc 文件。这个.pyc 文件包含 Python 源代码的编译版本，当下次运行相同的代码时，Python 解释器会先检查是否有相应的 .pyc 文件，如果有，则使用它来提高加载速度。

    python -m compileall your_directory   # 递归地编译指定目录中的 Python 文件并生成 .pyc 文件
    1. 运行方式：.pyc 文件通常由 Python 解释器自动加载。当你导入一个与之对应的 .py 文件时，如果已存在相应的 .pyc 文件并且是最新版本，Python 解释器会使用它来加速加载对应的代码。你不需要直接运行 .pyc 文件，而是运行相应的 .py 文件。当 Python 解释器发现相应的 .py 文件有更新，它会自动重新编译生成新的 .pyc 文件。

    1.4. .pyo文件¶

    1. 描述：.pyo 文件也是 Python 编译文件的一种形式，与 .pyc 类似，但通常由 Python 优化器生成。

    2. 使用场景： Python 源代码编译生成的字节码文件，但 .pyo 文件通常用于生产环境。通常更小、更快。

    3. 生成方式：使用 Python 自带的 -O 参数或使用 -OO 进行优化。例如：python -O -m py_compile your_file.py。

    4. 最佳实践：常用于生产环境，删除了一些调试信息，以提高运行速度和减小文件大小。

    5. 使用流程：生成 .pyo 文件的方式类似于生成 .pyc 文件。不过，Python 优化器会进行额外的优化，比如删除一些调试信息等。这使得 .pyo 文件的大小更小，运行速度更快。

    python -O -m py_compile your_file.py  # 生成优化的 .pyo 文件
python -OO -m py_compile your_file.py  # 强制优化，移除 docstrings 以生成更小的 .pyo 文件
# 上面的方法可能无效，可更换代码执行：compile_pyo.py
"""
import py_compile

py_compile.compile("xxx.py", cfile="xxx.pyo", optimize=2)
"""
# run
python3 compile_pyo.py
# or
python3 -c "import py_compile;py_compile.compile('main.py', cfile='main.pyo', optimize=2)"
    1. 运行方式：和 .pyc 文件类似，.pyo 文件也是由 Python 解释器自动加载。Python 解释器会在需要时自动创建和使用 .pyo 文件，以加速代码执行。同样，不需要直接运行 .pyo 文件，而是运行相应的 .py 文件。

    1.5. .pyd文件¶

    1. 描述：.pyd 文件是 Python 在 Windows 平台上使用的 DLL（Dynamic Link Library）文件。它通常包含 Python 的 C/C++ 扩展模块，提供了与底层操作系统或库的接口。

    2. 使用场景：.pyd 文件是 Windows 平台上的 Python 扩展模块文件，通常用于加载 Python 的 C/C++ 扩展模块。

    3. 生成方式：由第三方库提供的 Python 扩展模块，使用 Windows 平台的 C/C++ 编译器生成。

    4. 最佳实践：在 Windows 平台上使用，尤其是当需要与 C/C++ 代码进行交互时，这些文件提供了接口，例如访问操作系统特定的功能或库。

    5. 使用流程：这些文件用于 Python 的扩展，允许 Python 与 C/C++ 代码进行交互。你通常不需要直接处理 .pyd 文件，而是在 Python 中使用扩展模块时，Python 会自动加载它们。 总体而言，.py 文件包含源代码，.pyc 和 .pyo 文件是编译后的字节码文件用于加速加载，而 .pyd 文件是用于 Windows 平台的扩展模块文件。在开发和部署 Python 应用时，一般只需关注 .py 文件即可。.pyc 和 .pyo 文件是 Python 解释器自动生成的，而 .pyd 文件通常是由第三方库提供的。

    6. 编写 C/C++ 扩展

    7. 使用 Python 的 distutils 或 setuptools
    from distutils.core import setup, Extension

module = Extension('模块名称', sources=['你的C/C++文件.c'])

setup(name='模块名称',
      version='1.0',
      description='描述你的模块',
      ext_modules=[module])
    • 编译为 .pyd 文件: 将会编译你的 C/C++ 扩展文件，并生成一个 .pyd 文件
    python setup.py build
    1. 运行方式：.pyd 文件通常是由第三方库提供的，用于 Python 的扩展。它们会被 Python 解释器动态加载，并用于扩展 Python 的功能。你不会直接运行 .pyd 文件，而是在 Python 中导入相应的模块以使用其中的功能。

    1.6. .pyx文件¶

    1. 描述：.pyx 文件是与 Cython 相关的文件格式。Cython 是一个将 Python 代码转换为 C 或 C++ 代码的编译器，允许你在 Python 中使用 C 的静态类型声明以提高性能，并能够轻松地与现有的 C 库进行交互。

    2. 使用场景：

    3. 性能优化：Cython 可以帮助提高 Python 代码的性能，特别是涉及到计算密集型的任务时。

    4. 与 C/C++ 交互：Cython 允许 Python 代码与 C/C++ 库进行交互，使得 Python 能够调用 C/C++ 的函数或利用其功能。

    5. Python 扩展：Cython 可以用来编写 Python 的 C 扩展模块，以便在 Python 中使用更底层的 C 代码。

    6. 生成方式：由开发者编写Cython 代码，可以结合 Python 语法和 C 语言的特性，例如类型声明和内存管理。

    7. 最佳实践：

    # example.pyx
cdef int calculate_sum(int a, int b) nogil:
    return a + b

# 用 Cython 编译器将其编译成 C 或 C++ 代码，并生成对应的模块
cythonize -i -3 example.pyx

    2.Python 命令行标志¶

    Python命令行标志（Command Line Flags）是在运行Python解释器时使用的参数。这些标志可以用于控制Python解释器的行为和配置。 下面是一些常用的Python命令行标志及其说明：

    • -c <command>：在命令行直接执行Python代码。可以使用这个标志来执行简单的Python语句或一行代码。 示例：python -c "print('Hello, World!')"

    • -m <module>：执行指定的Python模块。这个标志用于执行模块级别的代码，类似于python -m module_name。 示例：python -m http.server

    • -i：进入交互模式。在执行完脚本后进入Python解释器的交互模式，可以与解释器进行交互。 示例：python -i script.py

    • -O和-OO：优化标志。-O标志会在编译时删除断言语句，-OO标志会删除断言语句和文档字符串。这些标志可用于在发布生产代码时减小程序的大小。 示例：python -O script.py

    • -B：禁用编译时的断言语句。在编译时会忽略所有的assert语句。 示例：python -B script.py

    • -v：显示详细的调试信息。在执行脚本时打印更多的调试信息，包括导入的模块和搜索路径等。 示例：python -v script.py

    • -V: 显示版本

    • -s <module>：禁用指定的模块。使用该标志可禁用某些模块，以防止其被导入和使用。 示例：python -s tkinter script.py

    • -W <action>：控制警告消息的处理方式。该标志用于设置警告消息的处理级别，如忽略、提示或抛出异常。 示例：python -W ignore script.py

    • -E：禁用环境变量的导入。使用该标志可防止Python解释器导入任何环境变量。 示例：python -E script.py

    • -X <option>：设置特定的解释器选项。该标志用于设置一些特定的解释器选项，例如启用调试模式、启用优化等。 示例：python -X dev script.py

    • -u：强制标准输出和标准错误流的刷新。该标志用于在脚本执行期间立即刷新输出，而不是等待缓冲区填满。 示例：python -u script.py

    • -h或--help：显示帮助信息。打印Python解释器的命令行选项和标志的帮助信息。 示例：python -h

    请注意，不同的Python版本可能会有不同的命令行标志和选项，因此在使用时请参考相应版本的文档或运行python --help命令来获取准确的信息。

    3.编译工具¶

    常用的几种工具和技术可用于将 Python 代码转换、优化或编译成可执行文件、字节码文件或扩展模块。 以下是其中几种：

    • compileall/py_compile：（递归）编译指定文件或目录，生成.pyc 或 .pyo 文件。
    • Cython： 可以将 Python 代码转换为 C 代码，并编译为 Python C 扩展模块（.pyd 文件），用于提高性能和与 C/C++ 交互。
    • PyInstaller: 将Python代码打包成独立的可执行文件。
    • Nuitka: 一个Python编译器，它可以将Python代码编译成C++代码，然后将其打包成可执行文件。
    • cx_Freeze: 将Python代码打包成独立的可执行文件。
    • py2exe: 打包为Windows可执行文件。
    • py2app: 打包为MacOS可执行文件。

    以下是一些考虑因素和使用建议：

    1. 考虑因素：

    2. 目标需求：确定你的目标是优化性能、生成独立的可执行文件，还是创建 C/C++ 扩展模块。

    3. 跨平台兼容性：某些工具可能更适合特定平台，而另一些工具可能提供跨平台支持。
    4. 易用性：一些工具可能对于新手更友好，提供更简单易懂的使用方法，而另一些则可能需要更多的配置和了解。

    5. 性能优化：如果目标是提高代码的执行速度和性能，一些工具可能比其他工具更有效。

    6. 使用建议：

    7. Cython：适用于与 C/C++ 交互或对 Python 代码进行性能优化。对于需要执行速度和与 C/C++ 交互的项目，Cython 是一个强大的选择。

    8. PyInstaller/cx_Freeze：适用于将 Python 代码打包成可执行文件，方便部署和分享。它们提供了将 Python 代码转换为独立的可执行文件的简单方法。
    9. Nuitka：适用于优化 Python 代码并生成更快的执行二进制文件。它可以加速你的 Python 代码，生成更快的可执行文件。
    10. py2exe/py2app：特别适用于在 Windows/MacOS 环境中生成可执行文件，使得用户无需安装 Python 就能运行你的应用。

    选择适合需求的工具非常重要。考虑到目标、平台兼容性以及工具的易用性和性能优化，从而选择最适合的编译工具。同时，使用工具时可以根据文档和示例来了解其用法，这将有助于更好地利用它们的功能和优势。

    3.1.Cython¶

    1. 描述：Cython 是用于编写 C 扩展的 Python 编译器，它允许你将 Python 代码转换成 C 代码，然后编译成 Python C 扩展模块（.pyd 文件）。

    2. 优势：用于性能优化和与 C/C++ 交互，提高性能并简化与现有 C 代码的集成。

    3. 使用方法：编写 Python 代码并使用 Cython 的特定语法（类似 Python 和 C 的混合）来标识需要优化的部分。之后使用 Cython 编译器将其编译成 C 代码。

    4. 安装 Cython: pip install cython

    5. 编写 Cython 文件: 创建一个 .pyx 文件，这是 Cython 文件的扩展名。在这个文件中，你可以使用 Python 风格的语法，同时利用 Cython 提供的 C 扩展特性。
    6. 编写 setup.py 文件:
    from distutils.core import setup
from Cython.Build import cythonize

setup(
    ext_modules = cythonize("你的文件名或文件列表")
)
    • 运行 setup.py 文件: python setup.py build_ext --inplace 这个命令会在当前目录下生成 .c 文件和共享对象文件（.so 或 .pyd）。生成的 .so 文件可以被 Python 直接导入并使用。

    3.2.Pyinstaller¶

    1. 描述：PyInstaller 是将 Python 应用程序打包成独立的可执行文件（针对特定平台）的工具。

    2. 优势：适用于发布 Python 应用，方便快速地将 Python 代码打包成可执行文件，不需要用户安装 Python 解释器。

    3. 使用方法：你可以使用 PyInstaller 在命令行中调用，指定入口文件并生成独立的可执行文件，会在一个名为 dist 的目录中生成一个或多个目录，其中包含可执行文件及其相关的依赖文件。

    4. 安装：pip install pyinstaller

    5. 运行 PyInstaller：pyinstaller your_script.py

    参考： 1.Pyinstaller文档 2.Pyinstaller实战指南

    3.3.Nuitka¶

    1. 描述：Nuitka 是用于将 Python 代码编译成 C 或 C++ 代码的编译器，以提高执行速度。

    2. 优势：用于优化和加速 Python 代码，生成更快速的二进制文件。

    3. 使用方法：通过 Nuitka 编译器来编译 Python 代码为二进制文件，它可以产生与系统架构相关的独立可执行文件。

    3.3.1.--onefile和--standalone¶

    --standalone 和 --onefile 的含义：

    1. --standalone：
    2. 作用：将 Python 解释器和所有必要的依赖库（包括 .py 文件、动态链接库 .so/.dll 文件等）打包到一个目录中。
    3. 结果：编译后，你将得到一个包含可执行文件和所有依赖的独立目录，所有需要的库文件和 Python 解释器会放在这个目录中。你可以将这个目录移植到其他系统，并在没有外部 Python 环境的情况下运行该程序。
    4. 依赖管理：所有依赖以原始文件形式存储在目录中，运行时直接加载，无需解压或临时文件。

    5. 运行时行为：可执行文件直接调用目录内的依赖，启动速度快（无解压过程）。

    6. 用法：--standalone 会创建一个独立的文件夹，文件夹中包括编译后的可执行文件、Python 解释器、依赖库等。

    7. 典型输出结构：

      your_script.dist/
├── your_script       # 主可执行文件
├── libpython3.10.so  # Python 解释器动态库（Linux）
├── websockets/       # 第三方库目录（含 .pyc、.so 文件）
├── cv2/              # 含 C扩展模块（如 cv2.cv2.so）
└── ...               # 其他依赖文件

    8. --onefile：

    9. 作用：将所有东西（包括解释器、库、源码）打包成一个单独的可执行文件。在程序启动时，Nuitka 会将所有依赖解压到临时目录中，并运行程序。

    10. 结果：你会得到一个单一的可执行文件，不再需要包含依赖的目录结构。一旦程序启动，Nuitka 会自动在后台解压所有依赖，并在解压后的临时目录中运行程序。
    11. 依赖管理：依赖以压缩形式打包到可执行文件中，运行时自动解压到临时目录（如 $TMPDIR/_nuitka_<随机字符串>），并在进程结束后清理。

    12. 运行时行为：启动时需解压依赖到临时目录（耗时取决于文件大小），运行中通过临时路径加载依赖。

    13. 用法：--onefile 会打包所有依赖和解释器成一个单一的二进制文件。

    14. 典型输出结构：

      （仅一个文件）
your_script  # 单文件可执行程序（内部封装所有依赖）

    区别：只用 --onefile vs --standalone + --onefile

    • 只用 --onefile：
    • Nuitka 仍然会将所有依赖、解释器和程序打包到一个单一文件中。
    • Nuitka 不会创建一个独立的临时目录，直接将整个运行环境打包为一个文件。

    • 当你运行这个可执行文件时，Nuitka 会将所有内容解压到目标系统的临时目录中（如 /tmp），并从那里运行程序。

    • --standalone + --onefile：

    • Nuitka 首先会创建一个独立的临时目录，将所有依赖库和解释器打包到这个目录中（--standalone 的行为）。

    • 然后，这个目录会进一步打包成一个单一文件（--onefile 的行为），从而生成一个最终的二进制文件。

    • 在程序运行时，依赖库和解释器同样会解压到目标系统的临时目录中。

    • 总结

    • 什么时候使用 --standalone：

      • 如果你有复杂的依赖，或者项目包含大量第三方库、动态链接库等时，--standalone 可以确保这些依赖都被完整打包。

    • 什么时候仅使用 --onefile：

      • 如果你的项目较为简单、依赖较少（特别是没有外部资源或动态库），--onefile 已经足够。

    3.3.2.--include-package 和 --include-module¶

    1. --include-package=<package>

    2. 作用：--include-package 会将指定的整个包及其子模块递归地全部包含在打包过程中。

    3. 场景：如果你有一个复杂的 Python 包，它包含多个子模块或子包，并且你希望这些模块在编译后的二进制文件中都包含进去，那么你可以使用这个选项。例如：

    nuitka --include-package=asyncio app.py

    这意味着 asyncio 包的所有模块（包括子模块、子包）都会被包含在编译后的二进制文件中。

    • 使用场景：

    • 如果你在使用某个 Python 包的多个子模块时，并且不希望遗漏任何依赖模块。

    • 当你不确定包中有多少子模块时，这是一个较为安全的选择，因为它会递归地包含整个包。

    • 例子：

    nuitka --standalone --onefile --include-package=asyncio app.py

    这将递归地将 asyncio 包的所有模块都包含在编译后的二进制文件中。使用 --standalone 时，它将确保在运行时不会丢失任何 asyncio 的依赖项。

    1. --include-module=<module>

    2. 作用：--include-module 会仅包含指定的单个模块，而不是整个包。

    3. 场景：当你只需要包含一个特定的模块，而不希望递归包含整个包时使用。比如，假设你只依赖于 anyio 中的某个模块，并不需要包含其他模块，那么你可以使用这个选项。例如：

    nuitka --include-module=anyio._backends app.py

    这将只包含 anyio._backends 这个模块，不会递归包含 anyio 包中的其他模块。

    • 使用场景：

    • 如果你只依赖某个特定模块而不是整个包。

    • 你希望控制生成的二进制文件的大小，并且只包含必要的模块。

    • 你需要避免打包不必要的模块或避免模块冲突。

    • 例子：

    nuitka --standalone --onefile --include-module=anyio._backends app.py

    在这个例子中，anyio._backends 模块将被包含在二进制文件中，而不会递归包含 anyio 包中的其他模块。

    1. 具体模块分析

    2. --include-package=asyncio：包含整个 asyncio 包，适用于涉及到协程、事件循环等复杂异步操作时，通常在 FastAPI、Uvicorn 等异步框架中都依赖于 asyncio，因此需要确保整个包的所有模块被包含。

    3. --include-package=anyio：anyio 是一个用于处理异步操作的库，它可以跨越多种异步事件循环（如 asyncio 和 trio），通过此选项，整个 anyio 包（包括 anyio._backends 和其他子模块）都会被包含。

    4. --include-module=anyio._backends：只包含 anyio 中的 _backends 模块，这个模块用于选择和管理不同的异步事件循环后端。anyio._backends 包含了与 asyncio、trio 事件循环的集成。因此，打包时如果只需要后端管理功能，可以只包含这个模块。

    5. --include-package vs --include-module 总结

    6. --include-package：

    7. 递归包含整个包和它的所有子模块。
    8. 适合用于你不确定具体依赖哪些子模块的场景，确保不会遗漏任何子模块。
    9. 生成的二进制文件会相对较大，因为所有子模块都会被包含。
    10. --include-module：
    11. 只包含指定的单个模块，不会递归包含其他子模块。
    12. 适合用于你明确知道只依赖某个模块的场景。
    13. 生成的二进制文件会较小，只有你显式包含的模块会被打包。

    • 场景 1：依赖外部 Python 解释器和库的二进制文件

    • 场景 2: 将所有源文件、依赖以及 Python 解释器打包成一个独立的二进制文件

    3.3.3.其他参数项¶

    1.--enable-plugin=anti-bloat¶

    anti-bloat插件的主要作用是减少编译后可执行文件的大小，通过以下方式实现：

    功能特点：

    • 移除无用代码：自动检测并移除不会被执行的代码分支
    • 优化导入：分析实际使用的模块和函数，排除未使用的部分
    • 减小文件体积：显著减少编译后二进制文件的大小
    • 提高性能：通过移除不必要的代码来提高运行效率

    工作原理：

    # 示例：Nuitka会分析这些代码并优化
import os
import sys

# 如果这个条件永远不会满足，Nuitka会移除整个分支
if False:
    print("这段代码会被移除")
    import pandas  # 这个导入也会被移除

# 只有实际使用的代码会被保留
def used_function():
    return "Hello"

def unused_function():
    return "World"  # 如果从未调用，会被移除

print(used_function())  # 只保留这个函数
    2. --assume-yes-for-downloads（自动下载依赖）¶

    这个参数的作用是：

    功能特点：

    • 自动下载缺失的依赖：当Nuitka检测到需要但未安装的依赖时，自动下载
    • 无需交互确认：避免在编译过程中需要用户手动确认下载操作
    • 简化构建流程：让构建过程更加自动化，无需人工干预

    使用场景：

    # 没有这个参数时，可能需要手动确认
python -m nuitka --enable-plugin=anti-bloat manage.py
# 可能会提示：需要下载xxx依赖，是否继续？[y/N]

# 有了这个参数后，自动下载无需确认
python -m nuitka --enable-plugin=anti-bloat --assume-yes-for-downloads manage.py
    3.--static-libpython=no¶

    禁用静态链接Python库。设置为"no"意味着不将Python解释器静态链接到编译后的二进制文件中，这样可以减小生成的二进制文件大小，但需要目标系统上安装兼容的Python运行时。

    4.--disable-ccache¶

    禁用ccache（编译器缓存）。ccache通常用于加速重复编译，但在某些情况下（如CI/CD环境）可能需要禁用以确保编译的一致性和避免缓存相关的问题。

    5.--python-flag=-S,-O,no_docstrings¶

    设置Python解释器标志：

    • -S: 禁用site.py模块的导入，这可以加快启动时间
    • -O: 启用基本优化，如移除assert语句
    • no_docstrings: 移除文档字符串以减小生成文件的大小

    参考： 1.Nuitka文档 2.Nuitka 用户手册 3.Python 打包工具 Nuitka 入门指南 4.Python打包神器—Nuitka

    3.4.cx_Freeze¶

    1. 描述：cx_Freeze 是用于将 Python 脚本打包成可执行文件的工具。

    2. 优势：用于将 Python 代码转换为可独立运行的可执行文件，不需要安装 Python 解释器。

    3. 使用方法：类似 PyInstaller，可以指定入口文件和输出目录，从而生成独立的可执行文件，cx_Freeze 会创建一个 build 目录，里面包含生成的可执行文件及其相关的依赖文件。

    4. 安装：pip install cx_Freeze

    5. 编写 setup.py 文件：
    from cx_Freeze import setup, Executable

# 定义你的 Executable
executables = [Executable('your_script.py')]  # 替换 'your_script.py' 为你的 Python 脚本文件名

# 配置 setup
setup(name='YourApp',
      version='1.0',
      description='Description of your app',
      executables=executables)
    • 运行 setup.py：python setup.py build

    3.5.py2exe¶

    1. 描述：py2exe 是 Windows 平台上将 Python 脚本转换为可执行文件的工具。

    2. 优势：用于在 Windows 环境中将 Python 代码打包成可执行文件，不需要用户安装 Python。

    3. 使用方法：使用 py2exe 提供的 API 来编译 Python 代码为 Windows 下的可执行文件，py2exe 会创建一个 dist 目录，里面包含生成的可执行文件及其相关的依赖文件。用户可以直接运行 dist 目录中的 .exe 文件。

    4. 安装：pip install py2exe

    5. 编写 setup.py 文件：
    from distutils.core import setup
import py2exe

setup(console=['your_script.py'])  # 将 'your_script.py' 替换为你的 Python 脚本文件名
    • 在命令行中运行 setup.py：python setup.py py2exe

    3.6.py2app¶

    1. 描述：py2app 可以将你的 Python 脚本或应用转换为 macOS 上的独立应用程序（.app 文件），使得 Python 应用在 macOS 系统上可以像其它应用一样运行。

    2. 优势：它能够处理应用程序所需的依赖项（库、模块等），确保转换后的应用包含所有必要的文件。py2app 可以将 Python 解释器和应用代码打包到一个应用中，而无需用户安装 Python。

    3. 使用方法: 使用工具提供的 API 来编译 Python 代码为相应平台上的可执行文件。

    4. 安装: pip install py2app

    5. 创建 setup.py 文件: 需要创建一个 setup.py 文件来定义应用的配置。示例：
    from setuptools import setup

APP = ['your_script.py']
DATA_FILES = []
OPTIONS = {
    'argv_emulation': True,
    'packages': ['your_packages'],
}

setup(
    app=APP,
    data_files=DATA_FILES,
    options={'py2app': OPTIONS},
    setup_requires=['py2app'],
)
    • 构建 .app 文件: python setup.py py2app 这将创建一个类似 dist/YourApp.app 的目录，其中 YourApp 是你在 setup.py 文件中定义的应用名称。
