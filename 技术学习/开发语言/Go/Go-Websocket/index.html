    一、简介¶

    WebSocket是一种基于TCP协议，在单个TCP连接上进行全双工通信的协议, 使得客户端和服务器之间的数据交换变得更加简单，允许服务端主动向客户端推送数据。与传统的HTTP请求-响应模式不同，WebSocket连接可以保持持久性，双方可以通过发送消息进行实时通信。在WebSocket API中，浏览器和服务器只需要完成一次握手，两者之间就直接可以创建持久性的连接，并进行双向数据传输。需要安装第三方包，如： go get -u -v github.com/gorilla/websocket

    WebSocket 流程：WebSocket 连接由浏览器请求，并由服务器响应，然后建立连接，此过程通常称为握手。

    • 发起握手
    • 从客户端接收数据帧
    • 发送数据帧给客户端
    • 关闭握手

    优点：

    • 轻量级报头减少了数据传输开销。
    • 单个 Web 客户端仅需要一个 TCP 连接。
    • WebSocket 服务器可以将数据推送到 Web 客户端。

    参考: golang中用到的的websocket库

    选型建议速查：

    • 普通业务（聊天室、后台通知）→ gorilla/websocket 或 nhooyr.io/websocket
    • 并发 >50 k、延迟 <5 ms → gobwas/ws 或 nbio+gws
    • 教学、想彻底理解协议 → 手写 net/http 握手
    • 需要 http/2 或 WASM → nhooyr.io/websocket

    一行命令开始：

    go get github.com/gorilla/websocket      # 最稳
go get github.com/gobwas/ws              # 性能极限
go get nhooyr.io/websocket               # 现代 API

    注意未来趋势

    • gorilla/websocket 已进入维护模式，官方推荐 逐步迁移到 nhooyr.io/websocket 。
    • 对于百万级长连接，社区压测显示 Go+gobwas/ws 内存占用比 Java-Netty 少 30-50 %

    2.1.gorilla/websocket¶

    gorilla/websocket 是 Go 语言中最流行的 WebSocket 库之一，提供了完整的 WebSocket 协议实现，支持 RFC 6455 标准，被广泛应用于各种实时通信场景。以下是其核心特性、最佳实践及示例代码：

    核心特性:

    1. 完整的协议支持：支持文本帧、二进制帧、Ping/Pong 心跳、连接关闭握手等。
    2. 高性能：基于标准库 net/http，优化了内存分配和并发处理。
    3. 简洁 API：提供简单的读写接口，易于集成到现有 HTTP 服务中。
    4. 扩展性：支持自定义子协议、压缩（permessage-deflate）和超时控制。
    5. 社区活跃：维护频繁，文档完善。

    2.1.1.基本 WebSocket 服务器¶

    包含连接管理、消息广播、错误处理：

    package main

import (
    "log"
    "net/http"

    "github.com/gorilla/websocket"
)

// 升级 HTTP 连接到 WebSocket
var upgrader = websocket.Upgrader{
    CheckOrigin: func(r *http.Request) bool { return true }, // 允许所有来源
    ReadBufferSize:  1024,
    WriteBufferSize: 1024,
}

// 客户端管理
type ClientManager struct {
    clients    map[*Client]bool  // 在线客户端
    register   chan *Client      // 注册请求
    unregister chan *Client      // 注销请求
    broadcast  chan []byte       // 广播消息
}

// 客户端
type Client struct {
    socket *websocket.Conn       // WebSocket 连接
    send   chan []byte           // 消息发送通道
}

// 消息处理器
func (c *Client) read(manager *ClientManager) {
    defer func() {
        manager.unregister <- c
        c.socket.Close()
    }()

    for {
        // 读取消息
        _, message, err := c.socket.ReadMessage()
        if err != nil {
            if websocket.IsUnexpectedCloseError(err, websocket.CloseGoingAway, websocket.CloseAbnormalClosure) {
                log.Printf("error: %v", err)
            }
            break
        }
        // 广播消息
        manager.broadcast <- message
    }
}

// 消息发送器
func (c *Client) write() {
    defer c.socket.Close()
    for message := range c.send {
        err := c.socket.WriteMessage(websocket.TextMessage, message)
        if err != nil {
            log.Printf("error: %v", err)
            break
        }
    }
}

// WebSocket 处理函数
func wsHandler(manager *ClientManager) http.HandlerFunc {
    return func(w http.ResponseWriter, r *http.Request) {
        // 升级 HTTP 连接到 WebSocket
        socket, err := upgrader.Upgrade(w, r, nil)
        if err != nil {
            log.Println(err)
            return
        }

        // 创建客户端
        client := &Client{socket: socket, send: make(chan []byte, 256)}
        manager.register <- client

        // 启动读写协程
        go client.read(manager)
        go client.write()
    }
}

func main() {
    // 初始化客户端管理器
    manager := &ClientManager{
        clients:    make(map[*Client]bool),
        register:   make(chan *Client),
        unregister: make(chan *Client),
        broadcast:  make(chan []byte),
    }

    // 启动管理器
    go func() {
        for {
            select {
            case client := <-manager.register:
                manager.clients[client] = true
                log.Println("新客户端连接")
            case client := <-manager.unregister:
                if _, ok := manager.clients[client]; ok {
                    delete(manager.clients, client)
                    close(client.send)
                    log.Println("客户端断开连接")
                }
            case message := <-manager.broadcast:
                for client := range manager.clients {
                    select {
                    case client.send <- message:
                    default:
                        close(client.send)
                        delete(manager.clients, client)
                    }
                }
            }
        }
    }()

    // 注册路由
    http.HandleFunc("/ws", wsHandler(manager))

    // 启动服务器
    log.Println("服务器启动，监听 :8080")
    log.Fatal(http.ListenAndServe(":8080", nil))
}

    2.1.2.客户端示例¶

    package main

import (
    "log"
    "os"
    "os/signal"
    "time"

    "github.com/gorilla/websocket"
)

func main() {
    // 连接服务器
    conn, _, err := websocket.DefaultDialer.Dial("ws://localhost:8080/ws", nil)
    if err != nil {
        log.Fatal("连接失败:", err)
    }
    defer conn.Close()

    // 启动读取协程
    go func() {
        defer conn.Close()
        for {
            _, message, err := conn.ReadMessage()
            if err != nil {
                log.Println("读取消息失败:", err)
                return
            }
            log.Printf("收到消息: %s", message)
        }
    }()

    // 发送心跳包
    go func() {
        ticker := time.NewTicker(5 * time.Second)
        defer ticker.Stop()

        for {
            select {
            case <-ticker.C:
                err := conn.WriteMessage(websocket.PingMessage, []byte{})
                if err != nil {
                    log.Println("发送心跳失败:", err)
                    return
                }
            }
        }
    }()

    // 等待中断信号
    interrupt := make(chan os.Signal, 1)
    signal.Notify(interrupt, os.Interrupt)
    <-interrupt
    log.Println("程序退出")
}

    defer conn.Close()

    }

    2.2.5. wsutil.ReadServerData 和 ws.ReadFrame 的区别¶

    它们分别对应了不同层级的读取操作：wsutil.ReadServerData 是一个更高级别的封装，旨在简化常见操作；而 ws.ReadFrame 则是一个低级别的接口，提供了对 WebSocket 帧的精细控制。

    • ws.ReadFrame (低级别API)

    • 示例：

      // 读取单个WebSocket帧
frame, err := ws.ReadFrame(conn)

    • 特点：

      • 帧级别控制：读取单个WebSocket帧，包含完整的帧头信息

      • 手动处理分片：需要手动处理WebSocket协议层的分片（FIN位、OpContinuation）

      • 完整帧信息：可以访问 frame.Header.Fin、 frame.Header.OpCode、frame.Payload

      • 精确控制：对每个帧的处理有完全控制权

    • wsutil.ReadServerData (高级别API)

    • 示例：

      // 读取完整消息（自动处理WebSocket分片）
data, opCode, err := wsutil.ReadServerData(conn)

    • 特点：

      • 消息级别控制：自动组装WebSocket协议层的分片，返回完整消息

      • 自动分片处理：内部处理WebSocket的FIN位和continuation帧

      • 简化接口：直接返回完整的消息数据

      • 隐藏细节：无法访问单个帧的详细信息

    1.核心差异：¶
    特性维度 wsutil.ReadServerData (高级API) ws.ReadFrame (低级API)
    抽象级别 高（消息级别） 低（帧级别）
    返回值 操作码 (OpCode), 有效负载 ([]byte), 错误 (error) 帧头 (Header), 有效负载读取器 (io.Reader), 错误 (error)
    数据帧处理 自动处理（合并分片消息、处理控制帧） 手动处理（需自行处理分片、控制帧、掩码等）
    使用便捷性 高（开箱即用，适合大多数常见场景） 低（需要更多代码和理解，但更灵活）
    控制粒度 低 高（可完全控制读写过程）
    典型应用场景 常规消息处理（如聊天应用、实时通知） 高性能、自定义协议、需要精细内存管理、教育或调试
    2.📊 深入理解 API 工作流程¶
    2.1.wsutil.ReadServerData 的工作流程¶

    wsutil.ReadServerData 是一个高级辅助函数，用于从服务器角度读取来自客户端的数据。它的工作流程可以概括为：

    1. 内部缓冲与帧处理：该函数会处理 WebSocket 协议底层可能出现的消息分片（ fragmentation ），自动将多个帧拼接成一个完整的应用层消息。
    2. 控制帧处理：它还会自动处理 Ping、Pong 和 Close 等控制帧（例如，收到 Ping 帧会自动发送 Pong 响应）。
    3. 返回应用数据：最终直接返回消息的操作码（如文本 OpText 或二进制 OpBinary ）和完整的有效负载数据（ []byte ）。
    // 示例：使用 wsutil.ReadServerData
for {
    data, opCode, err := wsutil.ReadServerData(conn) // conn 是 net.Conn
    if err != nil {
        // 处理错误（包括连接关闭）
        break
    }
    fmt.Printf("收到消息 (OpCode %d): %s\n", opCode, string(data))
    // 直接处理完整的应用消息 data
}

    优点：使用非常方便，无需关心协议细节，适合快速开发。缺点：隐藏了细节，灵活性较低，且由于其内部可能使用缓冲，在某些极端性能场景下可能非最优。

    2.2.ws.ReadFrame 的工作流程¶

    ws.ReadFrame 是一个低级原语，用于从连接中读取单个 WebSocket 帧。

    1. 读取帧头：它首先读取帧头（ws.Header），其中包含操作码（OpCode）、是否最终帧（Fin）、负载长度、掩码键等元信息。
    2. 返回负载读取器：它返回一个 io.Reader 来读取帧的有效负载。你需要自己根据帧头信息来读取和处理负载数据。
    // 示例：使用 ws.ReadFrame
for {
    header, reader, err := ws.ReadFrame(conn) // conn 是 net.Conn
    if err != nil {
        // 处理错误
        break
    }

    // 手动检查帧头信息
    fmt.Printf("帧操作码: %v, 是否是最终帧: %v\n", header.OpCode, header.Fin)

    // 根据操作码进行不同的处理
    switch header.OpCode {
    case ws.OpPing:
        // 手动处理 Ping 帧，发送 Pong 响应
        ws.WriteFrame(conn, ws.NewPongFrame(nil))
        continue // 继续读取下一帧
    case ws.OpClose:
        // 处理关闭帧
        return
    case ws.OpText, ws.OpBinary:
        // 读取应用数据帧的负载
        payload, err := io.ReadAll(reader)
        if err != nil {
            // 处理错误
            break
        }
        fmt.Printf("收到数据帧: %s\n", string(payload))
    case ws.OpContinuation:
        // 处理连续帧（分片消息的一部分）
        // 需要自己将多个连续帧和初始帧拼接起来
    // 其他操作码...
    }
}

    优点：提供了极高的灵活性和控制力。你可以根据帧头信息实现任何自定义逻辑，例如处理特定控制帧、实现自定义分片逻辑、精细管理内存（例如，避免不必要的字节切片分配）。缺点：使用复杂，需要深入了解 WebSocket 协议规范，代码量更大，容易出错。

    3.⚙️ 如何选择 API¶

    选择使用哪个函数，取决于你的具体需求：

    1. 追求开发效率与简单性：如果你的目标是快速构建一个标准的 WebSocket 应用（如聊天室、实时数据展示），wsutil.ReadServerData 是理想选择。它让你远离协议细节，专注于业务逻辑4。
    2. 需要极致性能与控制：如果你在构建高性能中间件、网关（如需要处理数十万并发连接25），或者需要实现非标准协议（可能在 WebSocket 帧上定义自定义格式），那么 ws.ReadFrame 的低级控制能力至关重要。它允许你进行零拷贝优化、精细的内存管理和自定义协议解析。
    3. 理解协议与调试：如果你想深入学习 WebSocket 协议或调试帧级别的问题，ws.ReadFrame 是最好的工具。
    4.💎 使用注意事项¶
    • 错误处理：无论使用哪个函数，都必须检查返回的 error。网络操作充满了不确定性。
    • 连接管理：确保在适当的时候关闭连接（conn.Close()），尤其是在发生错误或完成通信后。
    • 并发性：WebSocket 连接可能同时进行读写操作。在 Go 中，通常使用单独的 goroutine 来处理读和写循环1。
    • 性能考量：对于超大规模部署，结合使用 epoll 等 I/O 多路复用技术和 gobwas/ws 的低级 API，可以显著降低内存占用和提高性能25。

    5.1.ws.ReadFrame¶
    // existing code
// readLoop Message read loop
func (c *Client) readLoop() {
    defer func() {
        logger.Debug("WebSocket readLoop exiting, connection lost")
        atomic.StoreInt32(&c.connected, 0)

        if c.onDisconnect != nil {
            c.onDisconnect(nil)
        }

        // Try to reconnect
        if c.config.EnableReconnect && atomic.LoadInt32(&c.reconnecting) == 0 {
            logger.Warn("WebSocket attempting to reconnect...")
            c.tryReconnect()
        }
    }()

    // Buffers for fragmented message assembly
    var fragBuf bytes.Buffer
    var assembling bool
    var assemblingOp ws.OpCode

    for {
        select {
        case <-c.stopCh:
            return
        default:
        }

        // Get connection references to avoid long-term lock holding
        c.mu.RLock()
        conn := c.conn
        readTimeout := c.config.ReadTimeout
        c.mu.RUnlock()

        if conn == nil {
            return
        }

        // Set read timeout
        if readTimeout > 0 {
            if err := conn.SetReadDeadline(time.Now().Add(readTimeout)); err != nil {
                logger.Warn(fmt.Sprintf("failed to set read deadline: %v", err))
            }
        }

        // Read a single frame to support fragmentation handling
        frame, err := ws.ReadFrame(conn)

        // Clear deadline
        conn.SetReadDeadline(time.Time{})

        if err != nil {
            if netErr, ok := err.(net.Error); ok && netErr.Timeout() {
                logger.Debug(fmt.Sprintf("WebSocket read timeout: %v", err))
                continue
            }
            logger.Error(fmt.Sprintf("WebSocket read error: %v", err))
            c.handleError(fmt.Errorf("failed to read message: %w", err))
            atomic.AddInt64(&c.stats.errors, 1)
            return
        }

        // Update stats with payload size
        atomic.AddInt64(&c.stats.bytesTransferred, int64(len(frame.Payload)))

        switch frame.Header.OpCode {
        case ws.OpBinary, ws.OpText, ws.OpContinuation:
            fin := frame.Header.Fin
            op := frame.Header.OpCode

            // Start new fragmented message if OpBinary/OpText and not final
            if (op == ws.OpBinary || op == ws.OpText) && !fin {
                // Begin assembling
                fragBuf.Reset()
                assembling = true
                assemblingOp = op
                // Enforce max size as we append
                if _, err := fragBuf.Write(frame.Payload); err != nil {
                    logger.Error("failed to buffer fragmented payload")
                    assembling = false
                    continue
                }
                if c.config.MaxMessageSize > 0 && fragBuf.Len() > c.config.MaxMessageSize {
                    logger.Warn("fragmented message exceeded MaxMessageSize, discarding")
                    assembling = false
                    fragBuf.Reset()
                }
                continue
            }

            // Continuation frame for an ongoing fragmented message
            if op == ws.OpContinuation && assembling {
                if _, err := fragBuf.Write(frame.Payload); err != nil {
                    logger.Error("failed to append continuation payload")
                    assembling = false
                    fragBuf.Reset()
                    continue
                }
                if c.config.MaxMessageSize > 0 && fragBuf.Len() > c.config.MaxMessageSize {
                    logger.Warn("fragmented message exceeded MaxMessageSize during continuation, discarding")
                    assembling = false
                    fragBuf.Reset()
                    continue
                }
                if !fin {
                    // More continuations expected
                    continue
                }
                // Final continuation frame: dispatch assembled
                data := make([]byte, fragBuf.Len())
                copy(data, fragBuf.Bytes())
                opToDispatch := assemblingOp
                assembling = false
                fragBuf.Reset()

                atomic.AddInt64(&c.stats.messagesReceived, 1)
                c.handleEventMessage(data, opToDispatch)
                if c.onMessage != nil {
                    c.onMessage(data, opToDispatch)
                }
                continue
            }

            // Non-fragmented complete frame
            if (op == ws.OpBinary || op == ws.OpText) && fin {
                data := frame.Payload
                // Size guard
                if c.config.MaxMessageSize > 0 && len(data) > c.config.MaxMessageSize {
                    logger.Warn("message exceeded MaxMessageSize, discarding")
                    continue
                }
                atomic.AddInt64(&c.stats.messagesReceived, 1)
                c.handleEventMessage(data, op)
                if c.onMessage != nil {
                    c.onMessage(data, op)
                }
                continue
            }

            // Unexpected continuation without context
            if op == ws.OpContinuation && !assembling {
                // Forward to application-level handler; some servers may send multiple logical
                // frames using continuation even if the first was marked FIN=true.
                data := frame.Payload
                // TODO: (Jusheng Yao) Size guard. Just discard for now. But need to think about it in the future.
                if c.config.MaxMessageSize > 0 && len(data) > c.config.MaxMessageSize {
                    logger.Warn("continuation message exceeded MaxMessageSize, discarding")
                    continue
                }
                atomic.AddInt64(&c.stats.messagesReceived, 1)
                c.handleEventMessage(data, ws.OpBinary)
                if c.onMessage != nil {
                    c.onMessage(data, ws.OpBinary)
                }
                continue
            }

        case ws.OpPing:
            c.handlePing(frame.Payload)
        case ws.OpPong:
            atomic.StoreInt64(&c.lastPong, time.Now().Unix())
            logger.Debug("✅ WebSocket received pong frame (frame-level), connection is healthy")
        case ws.OpClose:
            logger.Debug("WebSocket received close frame")
            return
        default:
            // Ignore other opcodes
        }
    }
}

    6.📖 总结¶

    简单来说，在 gobwas/ws 中选择 wsutil.ReadServerData 还是 ws.ReadFrame，是在 开发效率 和 性能控制 之间做权衡。

    对于大多数应用场景，wsutil.ReadServerData 因其简单性而绰绰有余。而当你需要深入硬件、处理海量连接或实现高度自定义的协议时，ws.ReadFrame 提供的精细控制能力则变得不可或缺。

    2.3.lxzan/gws¶

    lxzan/gws是一个专注于高性能与低内存开销的 Go WebSocket 库，通过优化 I/O 模型、缓冲池和并发设计，在连接数超过 2000 时相比 gorilla/websocket 拥有显著更高的吞吐量（近 2 倍带宽利用率）。

    安装：

    go get github.com/lxzan/gws

    2.3.1.⚙️ 核心特性¶

    1. 高性能设计
    2. 缓冲池技术：通过分级缓冲池（128B~256KB）复用内存，减少 GC 压力，尤其在大数据包场景性能提升显著。
    3. 多路复用模型：基于任务队列实现串行读取、并行处理，避免 goroutine 爆炸。
    4. 压缩优化：支持 per-message deflate，但需权衡 CPU 开销（压缩时延增加约 60 倍）。
    5. 易用性设计
    6. 事件驱动 API：无需手动管理读写循环，通过 OnOpen, OnMessage, OnClose 等回调处理逻辑。
    7. 零错误返回：Conn.WriteMessage() 等方法的错误在内部处理，简化用户代码。
    8. 扩展支持
    9. 传输层灵活：支持 TCP、KCP、Unix Domain Socket 及任何实现 net.Conn 的协议。
    10. 代理集成：通过 NewDialer 配置支持 SOCKS5 等代理协议。
    11. Autobahn 兼容：通过全部 WebSocket 协议测试用例。

    2.3.2.🧩 服务端¶

    package main

import (
    "github.com/lxzan/gws"
    "net/http"
)

type Handler struct{}

// 连接建立时触发
func (h *Handler) OnOpen(conn *gws.Conn) {
    conn.WriteString("Welcome!")
}

// 收到消息时触发
func (h *Handler) OnMessage(conn *gws.Conn, msg *gws.Message) {
    defer msg.Close() // 重要！释放消息内存
    conn.WriteMessage(msg.Opcode, msg.Bytes()) // 原样回传
}

// 连接关闭时触发
func (h *Handler) OnClose(conn *gws.Conn, code uint16, reason []byte) {
    println("Client disconnected:", string(reason))
}

func main() {
    upgrader := gws.NewUpgrader(&Handler{}, &gws.ServerOption{
        CompressEnabled: true,     // 启用压缩
        ReadAsyncEnabled: true,    // 异步处理消息
        ReadAsyncGoLimit: 1024,    // 最大并发协程数
    })

    http.HandleFunc("/ws", func(w http.ResponseWriter, r *http.Request) {
        socket, _ := upgrader.Upgrade(w, r) // 升级为 WebSocket
        go socket.ReadLoop()                // 启动事件循环
    })
    http.ListenAndServe(":3000", nil)
}

    2.4.lesismal/nbio¶

    lesismal/nbio 是一个专为高性能网络 I/O 设计的 Go 框架，尤其在 WebSocket 高并发场景下表现优异。

    2.4.1.核心特性¶

    1. 异步非阻塞 I/O 模型
    2. 基于事件驱动（epoll/kqueue），单线程可处理数十万连接，避免传统每连接一协程的资源消耗。
    3. 支持逻辑协程池处理业务，默认大小为 runtime.NumCPU() * 1024，平衡并发与资源占用。
    4. 高效内存管理
    5. 内置分级内存池（128B~256KB），复用消息缓冲区，显著减少 GC 压力。
    6. 通过 ReleaseWebsocketPayload: true 自动回收消息内存，需注意跨协程传递时的手动释放时机。
    7. 百万连接支持
    8. 在 4 核 2GB 内存的服务器上实测支持 100 万连接 + 500 万次 Echo 交互，峰值内存仅 1.94GB。
    9. 对比 gorilla/websocket，相同负载下带宽利用率提升近 2 倍。

    2.4.2.服务端实现¶

    package main

import (
    "github.com/lesismal/nbio"
    "github.com/lesismal/nbio/websocket"
    "log"
)

// 定义WebSocket事件处理器
type Handler struct{}

func (h *Handler) OnOpen(c *websocket.Conn) {
    log.Printf("Client connected: %s", c.RemoteAddr())
}

func (h *Handler) OnMessage(c *websocket.Conn, msgType websocket.MessageType, data []byte) {
    // 原样回传消息（Echo模式）
    c.WriteMessage(msgType, data)
}

func (h *Handler) OnClose(c *websocket.Conn, err error) {
    log.Printf("Client disconnected: %v", err)
}

func main() {
    // 创建WebSocket Upgrader
    upgrader := &websocket.Upgrader{
        Handler:      &Handler{}, // 绑定处理器
        CheckOrigin:  func(r *http.Request) bool { return true },
        KeepaliveTime: 30 * time.Second, // 心跳超时设置:cite[5]
    }

    // 配置并启动nbio引擎
    engine := nbio.NewEngine(nbio.Config{
        Network: "tcp",
        Addrs:   []string{":8080"},
    })

    engine.OnOpen(upgrader.OnOpen) // 注册连接建立回调
    engine.OnData(upgrader.OnData) // 注册数据回调

    // 启动引擎
    if err := engine.Start(); err != nil {
        log.Fatalf("Engine start failed: %v", err)
    }
    defer engine.Stop()

    select {} // 阻塞主协程
}

    3. 生产者-消费者模式¶

    通过 Hub 集中管理连接与消息路由：

    type Hub struct {
    clients    map[*websocket.Conn]bool
    broadcast  chan []byte
    register   chan *websocket.Conn
    unregister chan *websocket.Conn
}

func (h *Hub) Run() {
    for {
        select {
        case client := <-h.register:
            h.clients[client] = true
        case client := <-h.unregister:
            delete(h.clients, client)
        case msg := <-h.broadcast:
            for client := range h.clients {
                client.WriteMessage(websocket.TextMessage, msg)
            }
        }
    }
}
