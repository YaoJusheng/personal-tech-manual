    1.4 术语与指标口径¶

    为避免不同章节口径不一致，本文统一使用以下定义：

    术语 定义
    Agent 闭环 感知 -> 决策 -> 执行 -> 反馈 -> 记忆
    任务成功率 在评估窗口内，完成目标且满足格式约束的任务占比
    工具调用成功率 工具请求中返回成功状态且结果可解析的占比
    幻觉率 输出中无法由证据支持或与事实冲突内容的占比
    上线门禁 上线前必须满足的最小质量、安全和稳定性条件
    验收标准 面向具体场景的可量化通过条件

    指标计算建议：

    • 任务成功率 = 成功任务数 / 总任务数
    • 工具调用成功率 = 成功调用数 / 总调用数
    • 幻觉率 = 幻觉样本数 / 总样本数

    三、Agent基础理论与核心概念¶

    3.1 Agent 最小闭环¶

    典型闭环：感知 -> 决策 -> 执行 -> 反馈 -> 记忆。

    可抽象为：

    $$ S_{t+1} = f(S_t, O_t, A_t, R_t) $$

    其中：

    • $S_t$：当前状态（上下文、记忆、任务阶段）
    • $O_t$：观测（用户输入、工具返回、环境信号）
    • $A_t$：动作（回答、调用工具、发起子任务）
    • $R_t$：反馈（成功/失败、评分、人工修正）

    3.2 Agent 核心组件¶

    • Planner：任务拆解、步骤规划、优先级与停止条件
    • Reasoner：基于上下文进行推理和决策
    • Tool Executor：调用外部工具、接口、数据库、代码执行环境
    • Memory：短期记忆、长期记忆、向量记忆、结构化状态
    • Guardrails：输入输出校验、安全策略、合规过滤
    • Evaluator：质量评估、回归验证、自动打分

    3.3 常见误区¶

    • 误区1：把 Prompt 增强当成 Agent 系统
    • 误区2：忽略状态管理，导致多轮任务漂移
    • 误区3：无评测直接上线，问题难复现
    • 误区4：工具权限过大，埋下安全风险
    • 误区5：只看准确率，不看成本和时延

    四、基础阶段学习路径（0-1）¶

    4.1 阶段目标¶

    • 能独立实现一个可用的单 Agent
    • 能接入 2-3 个工具（搜索、数据库、HTTP API）
    • 能通过简单评测证明可用性

    4.2 必学知识地图¶

    1. LLM 基础
    2. Prompt 与函数调用
    3. JSON Schema 与结构化输出
    4. 会话状态管理（Redis/数据库）
    5. API 服务化（HTTP/gRPC）

    4.3 最小项目建议¶

    项目：智能运维助手（单 Agent）

    • 输入：自然语言问题（如 CPU 异常）
    • 能力：调用监控 API、查询日志、生成处置建议
    • 输出：结构化工单建议（含风险等级）

    交付标准：

    • 成功调用工具 >= 95%
    • 核心任务正确率 >= 80%
    • 平均响应时延 <= 5s（非流式）

    五、Python技术路线（工程主线）¶

    5.1 推荐技术栈¶

    • Web 框架：FastAPI
    • 数据校验：Pydantic
    • Agent 编排：LangGraph 或 AutoGen
    • 检索增强：LlamaIndex 或 LangChain + 向量库
    • 任务队列：Celery / Arq / Dramatiq
    • 数据层：PostgreSQL + Redis
    • 观测：OpenTelemetry + Prometheus + Grafana

    5.2 目录结构建议¶

    agent-platform/
    app/
        api/                # HTTP API
        core/               # 配置、日志、中间件
        agents/             # Agent 定义和状态图
        tools/              # 工具实现
        memory/             # 记忆层（短期/长期）
        retrieval/          # RAG 模块
        eval/               # 评测脚本
        workers/            # 异步任务
    tests/
    scripts/
    docker/

    5.3 实现步骤¶

    1. 先定义任务状态机（待处理、执行中、等待工具、完成、失败）
    2. 为每个工具定义输入输出 Schema
    3. 在 Agent 节点之间显式传递状态对象
    4. 增加失败重试和兜底策略
    5. 接入 tracing，保证每次调用可追踪

    5.4 Python 常见工程问题¶

    • asyncio 与阻塞 I/O 混用导致吞吐下降
    • 缺少超时配置，工具调用挂死
    • 输出未做 Schema 校验导致链路脆弱
    • 多租户下未做 key 隔离导致数据串读

    六、Go技术路线（高性能主线）¶

    6.1 推荐技术栈¶

    • Web 框架：Gin / Chi / Fiber
    • 配置与依赖注入：Viper / Wire
    • 并发编排：Goroutine + Context + Worker Pool
    • 模型接入：OpenAI 兼容 SDK（含重试和熔断）
    • 数据层：PostgreSQL + Redis + 向量库 SDK
    • 消息系统：Kafka / NATS / RabbitMQ
    • 可观测性：OpenTelemetry + Prometheus

    6.2 Go 的优势场景¶

    • 高并发 API 网关与工具路由层
    • 低延迟流式返回（SSE/WebSocket）
    • Agent 任务调度中枢
    • 大规模多租户成本敏感场景

    6.3 Go 实现关键点¶

    1. 全链路使用 context 传递取消信号和超时
    2. 每次工具调用设置独立 timeout 和重试上限
    3. 使用熔断器隔离高失败率依赖
    4. 通过幂等键避免任务重复执行
    5. 将 Agent 状态序列化到持久层，支持故障恢复

    6.4 Go 常见问题¶

    • 忽略 context cancel 造成资源泄漏
    • 错误处理只打印不分级，无法治理
    • 无统一请求 ID，问题排查困难
    • 并发写 map 引发竞态

    七、进阶核心：RAG、记忆与工具生态¶

    7.1 RAG 设计要点¶

    • 索引前治理：去重、分段、元数据打标、版本号
    • 检索策略：向量检索 + 关键词检索 + 重排
    • 召回控制：TopK、阈值、领域过滤
    • 回答约束：仅基于证据回答并标注来源

    7.2 记忆分层模型¶

    层级 保存内容 存储建议 生命周期
    短期记忆 当前会话状态 Redis 分钟到小时
    中期记忆 用户偏好/任务上下文 PostgreSQL 天到周
    长期记忆 用户画像/历史行为摘要 向量库 + 文档库 月到年

    7.3 工具接入规范¶

    • 工具注册需包含：名称、用途、权限等级、输入输出 Schema
    • 所有工具必须具备：超时、重试、限流、审计日志
    • 高风险工具（写文件、执行命令、调用支付）必须二次确认

    7.4 MCP 与函数调用实践¶

    • 统一工具发现和协议，降低多工具集成复杂度
    • 在服务端实现工具白名单与权限映射
    • 对每次工具调用记录：发起人、参数摘要、结果摘要、耗时

    八、多Agent与工作流架构¶

    8.1 何时需要多 Agent¶

    • 单 Agent 的工具过多导致决策不稳定
    • 任务天然可拆分（研究、编码、测试、审计）
    • 需要角色隔离（规划者、执行者、审查者）

    8.2 常见架构模式¶

    1. Supervisor-Worker
    2. Planner-Executor-Reviewer
    3. 黑板系统（共享状态）
    4. 事件驱动（消息总线解耦）

    8.3 状态机与容错¶

    建议状态：

    • INIT
    • PLAN_READY
    • EXECUTING
    • WAITING_TOOL
    • NEED_HUMAN
    • DONE
    • FAILED

    每个状态都应定义：

    • 可进入条件
    • 可退出条件
    • 超时与补偿策略
    • 可观测字段

    8.4 人机协同（Human-in-the-loop）¶

    • 高风险动作必须人工确认
    • 支持“人工接管”与“人工修正回灌”
    • 将人工修正沉淀到评测集，形成持续改进闭环

    九、生产级工程化：评测、监控、安全、成本¶

    9.1 评测体系¶

    离线评测：

    • 构建标准任务集（事实问答、工具调用、长任务）
    • 维度：正确性、完整性、引用质量、格式合规

    在线评测：

    • A/B 实验
    • 用户反馈打分
    • 关键路径回放

    建议指标：

    • 任务成功率（Task Success Rate）
    • 工具调用成功率（Tool Success Rate）
    • 幻觉率（Hallucination Rate）
    • P95 延迟（P95 Latency）
    • 单任务成本（Cost per Task）

    9.2 可观测性¶

    • 日志：请求级、工具级、状态级
    • 指标：QPS、错误率、时延、Token 消耗
    • 链路追踪：跨服务 TraceID 贯通
    • 告警：模型超时、工具失败激增、预算异常

    9.3 安全治理¶

    • Prompt Injection 防护（系统提示隔离 + 输入净化）
    • 工具最小权限原则（RBAC + ABAC）
    • 输出安全审查（敏感信息、越权行为）
    • 秘钥管理（KMS、动态密钥轮换）

    9.4 成本治理¶

    • 模型分层路由（小模型优先，复杂任务升级）
    • 缓存策略（语义缓存、检索缓存、工具结果缓存）
    • 上下文压缩（摘要、窗口裁剪、历史衰减）
    • 预算策略（按租户/项目限额）

    十三、常见问题与排错手册¶

    13.1 工具调用成功率低¶

    排查顺序：

    1. Schema 是否严格一致
    2. 参数默认值和边界值是否定义
    3. 工具超时和重试配置是否合理
    4. 是否有熔断和降级逻辑

    13.2 回答质量波动大¶

    • 检查检索召回是否稳定
    • 检查上下文拼接顺序和长度截断
    • 检查系统提示是否被用户输入污染
    • 检查模型路由策略是否频繁切换

    13.3 成本失控¶

    • 检查长会话历史是否持续膨胀
    • 检查是否重复检索和重复工具调用
    • 检查是否缺少缓存和摘要机制

    15.1 中小团队实施路径（5-15人）¶

    目标：在 8-12 周内形成一个可上线、可观测、可迭代的 Agent 业务系统。

    阶段 周期 关键输出 关键风险 兜底策略
    需求澄清 1 周 任务边界、成功标准、数据范围 范围失控 明确“不做项”清单
    MVP 开发 2-4 周 单 Agent + 2 个核心工具 工具不稳定 超时重试 + 降级
    质量治理 2 周 离线评测集 + 监控看板 指标不可解释 增加样本分层标签
    试运行 2 周 灰度流量 + 反馈闭环 线上波动大 按租户分批放量
    正式上线 1-2 周 发布流程、应急预案、值班机制 事故响应慢 Runbook + 演练

    建议角色配置：

    • 1 名后端负责人（服务架构与接口治理）
    • 1 名 AI 工程师（Agent、RAG、评测）
    • 1 名前端工程师（交互与反馈闭环）
    • 1 名 QA/测试（回归与稳定性测试）

    15.2 中大型团队实施路径（15人以上）¶

    目标：建立平台化能力，支持多业务线复用与治理。

    架构分层建议：

    • 平台层：模型网关、工具注册中心、权限中心、评测平台
    • 业务层：场景 Agent（客服、运维、研发协作、知识管理）
    • 运营层：提示词版本治理、数据治理、成本中心

    组织分工建议：

    • 平台组负责通用能力和统一规范
    • 业务组负责场景策略与效果迭代
    • SRE 组负责发布、监控、容灾和稳定性
    • 安全组负责合规、审计、越权防护

    15.3 关键评审门禁（Go/No-Go）¶

    上线前至少满足：

    1. 任务成功率达到预设阈值（例如 >= 85%）
    2. 高风险工具具备人工确认和审计日志
    3. P95 延迟、错误率、成本均在预算内
    4. 具备降级策略（模型降级或人工接管）
    5. 有可执行的事故预案与值班响应流程

    15.4 SLO 与容量规划模板¶

    SLO 示例：

    • 可用性：月度 >= 99.9%
    • 响应延迟：P95 <= 3s（轻任务），<= 8s（复杂任务）
    • 工具调用成功率：>= 97%
    • 幻觉率：<= 5%（定义需固定）

    容量规划建议：

    • 按峰值 QPS 的 2-3 倍预留弹性容量
    • 模型调用与工具调用分开限流，避免级联雪崩
    • Worker 与 API 分离扩缩容策略
    • 对高耗时任务启用异步化和任务队列

    17.2 工具调用封装（超时 + 重试）¶

    import asyncio
from collections.abc import Awaitable, Callable


async def call_with_retry(
    func: Callable[[], Awaitable[dict]],
    timeout_s: float = 5,
    retries: int = 2,
) -> dict:
    last_error = None
    for _ in range(retries + 1):
        try:
            return await asyncio.wait_for(func(), timeout=timeout_s)
        except Exception as err:  # noqa: BLE001
            last_error = err
    raise RuntimeError(f"tool_call_failed: {last_error}")

    17.3 节点编排示例（Planner -> Executor -> Finalizer）¶

    from typing import TypedDict


class GraphState(TypedDict):
    query: str
    plan: list[str]
    result: dict
    answer: str


def planner_node(state: GraphState) -> GraphState:
    state["plan"] = ["search_docs", "summarize"]
    return state


def executor_node(state: GraphState) -> GraphState:
    # 这里可替换为真实工具调用
    state["result"] = {"documents": ["doc-a", "doc-b"]}
    return state


def finalizer_node(state: GraphState) -> GraphState:
    docs = ", ".join(state["result"].get("documents", []))
    state["answer"] = f"已完成处理，命中文档: {docs}"
    return state

    18.2 工具调用封装（context + timeout + retry）¶

    package tools

import (
    "context"
    "errors"
    "time"
)

func CallWithRetry(ctx context.Context, call func(context.Context) (map[string]any, error), timeout time.Duration, retries int) (map[string]any, error) {
    var lastErr error
    for i := 0; i <= retries; i++ {
        cctx, cancel := context.WithTimeout(ctx, timeout)
        result, err := call(cctx)
        cancel()
        if err == nil {
            return result, nil
        }
        lastErr = err
    }
    if lastErr == nil {
        lastErr = errors.New("unknown tool error")
    }
    return nil, lastErr
}

    二十、端到端实战案例¶

    20.1 场景定义：智能运维告警处置 Agent¶

    业务目标：

    • 将高频告警从“人工排查”升级为“自动诊断 + 人工确认执行”。
    • 将平均处置时间（MTTR）从 30 分钟降低到 10 分钟以内。

    功能边界：

    • 支持告警摘要、日志检索、根因建议、处置方案生成。
    • 不直接执行高风险变更（重启核心服务、扩容生产集群），仅生成执行建议并走审批。

    20.2 需求拆解与能力映射¶

    需求项 Agent 能力 技术实现
    告警理解 语义解析 LLM + 规则模板
    证据收集 工具调用 监控 API + 日志 API + CMDB
    根因判断 推理与检索增强 ReAct + RAG
    输出建议 结构化输出 JSON Schema + 审核状态
    风险控制 人机协同 高风险动作二次确认

    20.3 系统链路设计¶

    请求主链路：

    1. 告警平台推送事件到 Agent API。
    2. Planner 解析告警并拆解子任务。
    3. Executor 依次调用监控、日志、配置工具。
    4. Evaluator 校验证据充分性与建议可信度。
    5. 输出处置建议并进入人工确认流。
    6. 将结果写回工单系统，沉淀为评测样本。

    关键技术点：

    • 全链路 RequestID/TraceID 贯通。
    • 工具调用统一封装超时与重试。
    • 关键步骤记录审计日志（调用人、参数摘要、结果摘要）。
    • 对模型输出进行 Schema 校验和敏感字段过滤。

    20.4 API 合同示例¶

    接口 方法 用途 关键字段
    /v1/tasks POST 创建处置任务 alert_id, severity, summary
    /v1/tasks/{task_id} GET 查询任务状态 status, progress, error
    /v1/tasks/{task_id}/result GET 查询最终建议 diagnosis, actions, risk_level
    /v1/tasks/{task_id}/approve POST 人工确认执行 approver, decision, comment

    响应结构建议：

    • 统一包含 code、message、request_id、data。
    • 所有失败场景返回 machine-readable 错误码。
    • 对外不暴露内部堆栈，仅记录到日志系统。

    20.5 状态机设计与容错策略¶

    状态 进入条件 退出条件 超时策略
    INIT 创建任务成功 完成任务规划 30s 超时失败
    PLAN_READY 规划完成 开始工具执行 60s 内必须启动
    EXECUTING 至少 1 个工具启动 收集到足够证据 单工具 5s 超时+重试
    NEED_HUMAN 风险等级高 人工审批完成 24h 超时自动关闭
    DONE 建议生成并存档 归档完成 无
    FAILED 不可恢复错误 人工重试或终止 记录根因标签

    失败补偿建议：

    • 工具失败优先回退到备用数据源。
    • 模型调用失败切换低成本备用模型。
    • 连续失败超过阈值触发熔断并告警。

    20.6 评测集与验收标准¶

    离线数据集建议：

    • 100 条历史告警样本（按严重级别均衡分布）。
    • 每条样本包含标准答案：根因、证据、建议动作、风险等级。

    验收指标建议：

    基线门禁可复用第 15.3 节与第 15.4 节，这里给出该场景的实例阈值：

    • 任务成功率 >= 85%
    • 工具调用成功率 >= 97%
    • 证据引用完整率 >= 90%
    • P95 响应时延 <= 8s
    • 高风险误判率 <= 3%

    20.7 上线方案与回滚预案¶

    灰度步骤：

    1. 先放量 5% 低风险告警。
    2. 观察 24 小时后提升到 20%。
    3. 满足指标后逐步覆盖全量。

    回滚触发条件：

    • 连续 10 分钟错误率 > 5%
    • 高风险建议误判达到预警阈值
    • 上游依赖连续不可用

    回滚动作：

    • 切换到规则引擎基础策略。
    • 暂停自动建议，仅保留告警摘要。
    • 发出故障通告并进入应急流程。
