    1.简介¶

    MLflow 是一个开源平台，专门用于帮助机器学习从业者和团队处理机器学习过程的复杂性。 MLflow 专注于机器学习项目的整个生命周期，确保每个阶段都是可管理、可追溯和可再现的。

    1.1.Tracking¶

    MLflow 跟踪是 MLflow 的主要服务组件之一。在这些指南中，您将了解 MLflow 跟踪在构建 ML 模型时可以如何增强 MLOps 相关活动。

    1.2.环境变量¶

    MLflow 是一个开源的机器学习平台，支持多种环境变量以配置不同的参数。下面是一些常用的 MLflow 环境变量：

    • MLFLOW_SERVER_DEFAULT_ARTIFACT_ROOT: 指定默认的存储库位置，用于存储实验运行产生的文件和输出。例如，可以将该变量设置为 file:/mlflow/artifacts，表示将存储库保存在 /mlflow/artifacts 目录下。
    • MLFLOW_SERVER_DEFAULT_FILE_STORE_PATH: 指定文件存储库的默认路径。例如，可以将该变量设置为 /mlflow/tracking，表示将文件存储库保存在 /mlflow/tracking 目录下。
    • MLFLOW_SERVER_HOST: 指定 MLflow 服务器的主机地址。例如，可以将该变量设置为 0.0.0.0，表示服务器将在所有可用的网络接口上监听请求。
    • MLFLOW_SERVER_PORT: 指定 MLflow 服务器的监听端口。例如，可以将该变量设置为 5000，表示服务器将在本地 5000 端口监听请求。
    • MLFLOW_TRACKING_URI: 指定 MLflow 服务器的跟踪 URI。例如，可以将该变量设置为 http://localhost:5000，表示将使用本地的 MLflow 服务器进行跟踪。
    • MLFLOW_S3_ENDPOINT_URL: 指定 Amazon S3 存储的端点 URL。例如，可以将该变量设置为 https://s3.amazonaws.com，表示将使用 Amazon S3 存储进行实验运行产生的文件和输出的存储。
    • MLFLOW_S3_IGNORE_TLS: 如果使用不安全的 TLS 连接，则可以将该变量设置为 true，以跳过 TLS 证书验证。
    • MLFLOW_ARTIFACT_UPLOAD_RETRY_COUNT: 指定文件上传失败时的重试次数。
    • MLFLOW_TRACKING_USERNAME: 指定用户名，用于对 MLflow 服务器进行身份验证。
    • MLFLOW_TRACKING_PASSWORD: 指定密码，用于对 MLflow 服务器进行身份验证。
    • MLFLOW_BACKEND_STORE_URI: 指定用于保存 MLflow 的元数据和状态的后端存储 URI。例如，可以将该变量设置为 file:///mlflow/database.db，表示将元数据和状态存储在本地的 /mlflow/database.db 文件中。
    • MLFLOW_BACKEND_STORE_TYPE: 指定用于保存 MLflow 的元数据和状态的后端存储类型。例如，可以将该变量设置为 sqlalchemy，表示将使用 SQLAlchemy 数据库引擎进行元数据和状态的存储。
    • MLFLOW_DEFAULT_ARTIFACT_ROOT: 指定默认的存储库位置，用于存储实验运行产生的文件和输出。与 MLFLOW_SERVER_DEFAULT_ARTIFACT_ROOT 类似，但仅适用于本地运行的客户端。
    • MLFLOW_DEFAULT_BACKEND_STORE_URI: 指定用于保存 MLflow 的元数据和状态的后端存储 URI。与 MLFLOW_BACKEND_STORE_URI 类似，但仅适用于本地运行的客户端。
    • MLFLOW_EXPERIMENT_NAME: 指定要使用的实验名称。如果指定了该变量，则所有的实验运行将被归为该实验的一部分。
    • MLFLOW_RUN_NAME: 指定要使用的运行名称。如果指定了该变量，则该运行将使用指定的名称，否则将自动生成一个随机名称。
    • MLFLOW_RUN_ID: 指定要使用的运行 ID。如果指定了该变量，则该运行将使用指定的 ID，否则将自动生成一个随机 ID。

    see: https://mlflow.org/docs/latest/cli.html

    2.快速开始¶

    2.1.部署¶

    训练机器学习模型并确保其性能后，下一步是将其部署到生产环境。这个过程可能很复杂，但 MLflow 通过提供一套简单的工具集来简化它，用于将您的 ML 模型部署到各种目标，包括本地环境、云服务和 Kubernetes 集群。

    通过使用MLflow部署工具集，您可以享受以下好处：

    • 轻松部署：MLflow 提供了一个简单的界面，用于将模型部署到各种目标，无需编写样板代码。
    • 依赖项和环境管理：MLflow 确保部署环境与训练环境一致，捕获所有依赖项。这可确保模型无论部署在何处都能一致运行。
    • 打包模型和代码：使用 MLflow，不仅模型，而且任何补充代码和配置都与部署容器一起打包。这确保模型可以无缝执行而不会缺少任何组件。
    • 避免供应商锁定：MLflow 提供了打包模型的标准格式和用于部署的统一 API。您可以轻松地在部署目标之间切换，而无需重写代码。
    # deploy
mlflow server --backend-store-uri mysql://dev_cloud:flexic_cloud@localhost/mlflow --default-artifact-root ftp://ftp_example:ftp_example@180.167.218.82:201/pub/test_example_ai_cloud/ --host 0.0.0.0 --port 5001 --workers 4 --gunicorn-opts --config=/data/apps/mlflow_server/gunicorn_config.py

# 进程检查
sudo mkdir -p /var/run/mlflow_server
sudo touch /var/run/mlflow_server/gunicorn.pid
sudo chown -R web_dev:web_dev /var/run/mlflow_server
lsof -i:5001

    3.表关系¶

    MLflow 使用后端存储来管理运行和实验的元数据，可以将其配置为使用 PostgreSQL、MySQL 等数据库。这些表通常包括 runs, experiments, metrics, parameters, and tags，以及由外键定义的关系，例如作为 run_uuid 将运行链接到指标和参数

    +-----------------------+
| Tables_in_mlflow      |
+-----------------------+
| alembic_version       |
| experiment_tags       |
| experiments           |
| latest_metrics        |
| metrics               |
| model_version_tags    |
| model_versions        |
| params                |
| registered_model_tags |
| registered_models     |
| runs                  |
| tags                  |
+-----------------------+

    3.1.MLflow ER 图概述¶

    在 MLflow 的数据库结构中，主要包含以下几张表：

    1. experiments：存储实验的元数据。
    2. runs：存储每次模型训练（即“运行”）的元数据。
    3. metrics：存储每个运行的度量指标。
    4. params：存储每个运行的参数。
    5. tags：存储每个运行的标签信息。
    6. latest_metrics：存储最新的度量指标（用于优化查询性能）。
    7. registered_models：存储注册的模型信息。
    8. model_versions：存储模型的版本信息。
    9. model_version_tags：存储模型版本的标签信息。

    3.2.ER 图说明¶

    下面是各个表之间的 ER 关系描述：

    • experiments 与 runs：每个实验可以有多个运行（1 对多的关系）。
    • runs 与 metrics、params、tags：每个运行可以有多个度量指标、参数和标签（1 对多的关系）。
    • registered_models 与 model_versions：每个注册的模型可以有多个版本（1 对多的关系）。
    • model_versions 与 model_version_tags：每个模型版本可以有多个标签（1 对多的关系）。
    • runs 与 latest_metrics：latest_metrics 存储每个运行的最新度量指标，是一个 1 对 1 的关系，用于加快查询性能。

    3.4.各表字段描述¶

    1. experiments¶

    • experiment_id：实验的唯一标识符。
    • name：实验名称。
    • lifecycle_stage：生命周期状态，标记为 active 或 deleted。
    • artifact_location：实验的 artifact 存储路径。

    2. runs¶

    • run_uuid：每个运行的唯一标识符。
    • experiment_id：关联的实验 ID。
    • start_time：运行开始的时间戳。
    • end_time：运行结束的时间戳。
    • status：运行状态（如成功、失败等）。
    • artifact_uri：运行的 artifact 存储路径。

    3. metrics¶

    • key：度量指标的名称。
    • value：度量指标的值。
    • timestamp：度量指标的时间戳。
    • run_uuid：关联的运行 ID。

    4. params¶

    • key：参数的名称。
    • value：参数的值。
    • run_uuid：关联的运行 ID。

    5. tags¶

    • key：标签的名称。
    • value：标签的值。
    • run_uuid：关联的运行 ID。

    6. latest_metrics¶

    • key：最新度量指标的名称。
    • value：最新度量指标的值。
    • timestamp：最新度量指标的时间戳。
    • run_uuid：关联的运行 ID。

    7. registered_models¶

    • name：注册的模型名称。
    • creation_time：模型注册的时间戳。
    • last_updated_time：模型最后更新的时间戳。

    8. model_versions¶

    • name：注册的模型名称（对应 registered_models）。
    • version：模型版本号。
    • run_uuid：关联的运行 ID。
    • creation_time：模型版本创建的时间戳。

    9. model_version_tags¶

    • key：模型版本的标签名称。
    • value：模型版本的标签值。
    • model_version_id：关联的模型版本 ID。

    4.容器化部署¶

    1.示例¶

    1.1.示例-1-compose编排¶

    使用上述环境变量和 MLflow 镜像 ghcr.io/mlflow/mlflow 重新编排部署 MLflow：

    version: '3.8'
services:
  mlflow:
    image: ghcr.io/mlflow/mlflow:latest
    ports:
      - "5000:5000"
    environment:
      - MLFLOW_SERVER_HOST=0.0.0.0
      - MLFLOW_SERVER_PORT=5000
      - MLFLOW_TRACKING_URI=http://localhost:5000
      - MLFLOW_BACKEND_STORE_URI=sqlite:////mlflow/database.db
      - MLFLOW_SERVER_DEFAULT_ARTIFACT_ROOT=file:/mlflow/artifacts
      - MLFLOW_DEFAULT_BACKEND_STORE_URI=sqlite:////mlflow/database.db
      - MLFLOW_DEFAULT_ARTIFACT_ROOT=file:/mlflow/artifacts
    volumes:
      - ./mlflow:/mlflow

    1.2.示例-2-搭配MinIO¶

    将 MLflow 的数据存储到 MinIO 对象存储中，可以使用以下步骤来修改上述场景的部署示例：

    version: '3.8'
services:
  mlflow:
    image: ghcr.io/mlflow/mlflow:latest
    ports:
      - "5000:5000"
    environment:
      - MLFLOW_SERVER_HOST=0.0.0.0
      - MLFLOW_SERVER_PORT=5000
      - MLFLOW_TRACKING_URI=http://localhost:5000
      - MLFLOW_BACKEND_STORE_URI=s3://mlflow
      - MLFLOW_SERVER_DEFAULT_ARTIFACT_ROOT=s3://mlflow/artifacts
      - AWS_ACCESS_KEY_ID=minio
      - AWS_SECRET_ACCESS_KEY=minio123
      - MLFLOW_S3_ENDPOINT_URL=http://minio:9000
      - MLFLOW_TRACKING_USERNAME=“”
      - MLFLOW_TRACKING_PASSWORD=”“
      - MLFLOW_EXPERIMENT_NAME="test"
    depends_on:
      - minio
    volumes:
      - ./mlflow:/mlflow
  minio:
    image: minio/minio
    command: server /data
    ports:
      - "9000:9000"
    environment:
      - MINIO_ACCESS_KEY=minio
      - MINIO_SECRET_KEY=minio123
    volumes:
      - ./data:/data

    1.3.示例-3-搭配FTP¶

    将 MLflow artifacts 数据存储到 FTP，元数据信息存储到 PostgreSQL 的编排部署:

    version: '3.8'

services:
  mlflow:
    image: ghcr.io/mlflow/mlflow
    environment:
      - MLFLOW_BACKEND_STORE_URI=postgresql://postgres:your-password@postgres:5432/postgres
      - MLFLOW_TRACKING_URI="http://localhost:5000"
      - MLFLOW_ARTIFACT_ROOT=ftp://ftpuser:your-password@ftpd:21/
    ports:
      - "5000:5000"
    volumes:
      - ./mlflow:/mlflow

  postgres:
    image: postgres
    environment:
      - POSTGRES_PASSWORD=your-password
    volumes:
      - ./postgres-data:/var/lib/postgresql/data

  # 基于 Pure-FTPd 服务器的 FTP 服务镜像，支持多用户和虚拟用户。此外，它还支持 TLS/SSL 和 IPv6。
  # 优点：配置简单，镜像体积较小，支持虚拟用户，可通过环境变量或挂载配置文件进行配置；
  # 缺点：仅支持 FTP 协议，不支持 SFTP，可能会受到安全风险的威胁。
  ftpd:
    image: stilliard/pure-ftpd
    privileged: true
    restart: always
    environment:
      - PUBLICHOST=localhost
      - FTP_USER_NAME=ftpuser
      - FTP_USER_PASS=your-password
      - FTP_USER_HOME=/home/ftpuser
      - FTP_USER_GROUP=ftpusers
      - FTP_USER_UID=1001
      - FTP_USER_GID=1001
      - FTP_PASSIVE_PORTS=30000:30009
      - FTP_TLS=1
    ports:
      - "21:21"
      - "30000-30009:30000-30009"
    volumes:
      - ./ftp-data:/home/ftpuser

  # 基于 OpenSSH 的 SFTP 服务镜像，支持 SSH 密钥和密码身份验证。该镜像具有良好的安全性和可靠性，支持容器化和分布式环境。
  # 优点：支持 SFTP 协议，支持 SSH 密钥认证，可通过环境变量或挂载配置文件进行配置；
  # 缺点：镜像体积较大，仅支持 SFTP 协议，不支持 FTP。
  sftp:
    image: atmoz/sftp
    privileged: true
    restart: always
    ports:
      - "22:22"
    environment:
      SFTP_USERS: mlflow:mlflow:::/home/mlflow,user1:1234:::/home/user1,user2:abcd:::/home/user2
    command:   # 用户名:密码:uid:gid:文件夹1/子文件夹1,文件夹2,文件夹3
      - mlflow:mlflow:::upload,user1:1234:::upload,user2:abcd:::upload
    volumes:
      - ./ftp-data:/home/mlflow
      - ./ftp-data/user1:/home/user1:rw  # read & write
      - ./ftp-data/user2:/home/user2:ro  # read only

  # 基于 vsftpd 服务器的 FTP 服务镜像，支持虚拟用户、TLS/SSL 和被动模式数据传输。该镜像还支持禁用匿名用户访问和限制 FTP 用户对主机文件系统的访问。
  # 轻量级、快速、安全且易于配置的 FTP 服务器，它支持多种用户认证方式、IP 限制、上传/下载带宽限制等功能。
  # 优点：配置简单，支持 FTP 和 FTPS 协议，支持虚拟用户，可通过环境变量或挂载配置文件进行配置；
  # 缺点：镜像体积较大，不支持 SFTP 协议，可能会受到安全风险的威胁。
  vsftpd:
    image: fauria/vsftpd
    privileged: true
    restart: always
    environment:
      - FTP_USE=example
      - FTP_PASS=example
      - FTP_PASSIVE_PORTS=30000:30009
      - PASV_MAX_PORT=21110
      - PASV_MIN_PORT=21100
    volumes:
      - ./data:/home/vsftpd
    ports:
      - "21:21"
      - "21100-21110:21100-21110" # 其中 21100-21110 是被动模式数据传输所需的端口范围
