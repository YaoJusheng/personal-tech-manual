    设计模式（Design pattern）是经过总结、优化的，对我们经常会碰到的一些编程问题的可重用最佳解决方案。

    三种最基本的设计模式：

    • 创建型模式：提供实例化的方法，为适合的状况提供相应的对象创建方法。
    • 结构型模式：通常用来处理实体之间的关系，使得这些实体能够更好地协同工作。
    • 行为型模式：用于在不同的实体建进行通信，为实体之间的通信提供更容易，更灵活的通信方法。

    设计模式的六大原则：

    • 开闭原则（Open Close Principle）：对扩展开放，对修改关闭。
    • 里氏代换原则（Liskov Substitution Principle）：任何基类可以出现的地方，子类一定可以出现。
    • 依赖倒转原则（Dependence Inversion Principle）：针对接口编程，依赖于抽象而不依赖于具体。
    • 接口隔离原则（Interface Segregation Principle）：使用多个隔离的接口，比使用单个接口要好。它还有另外一个意思是：降低类之间的耦合度。
    • 迪米特法则，又称最少知道原则（Demeter Principle）：一个实体应当尽量少地与其他实体之间发生相互作用，使得系统功能模块相对独立。
    • 合成复用原则（Composite Reuse Principle）：尽量使用合成/聚合的方式，而不是使用继承。

    📚 使用建议¶

    1. 学习路径：建议按照创建型 → 结构型 → 行为型的顺序学习
    2. 实践应用：每个模式都提供了实际的使用场景，可以结合项目需求选择合适的模式
    3. 语言选择：根据项目技术栈选择对应的语言实现进行参考
    4. 扩展学习：可以基于提供的示例代码进行扩展和改进

    具体模式分类表¶

    创建型模式（Creational Patterns）¶

    序号 模式名称 英文名称 核心目的 主要应用场景
    1 单例模式 Singleton 保证一个类只有一个实例 数据库连接池、日志记录器、配置管理器
    2 工厂方法模式 Factory Method 创建对象的接口，由子类决定实例化类型 UI组件库、数据库驱动程序、日志系统
    3 抽象工厂模式 Abstract Factory 创建一系列相关或相互依赖的对象 跨平台UI组件库、数据库访问层、游戏主题
    4 建造者模式 Builder 分步骤创建复杂对象 SQL查询构建器、HTTP请求构建器、配置对象构建
    5 原型模式 Prototype 通过复制已有对象来创建新对象 游戏角色复制、文档模板、配置对象复制

    结构型模式（Structural Patterns）¶

    序号 模式名称 英文名称 核心目的 主要应用场景
    1 适配器模式 Adapter 让接口不兼容的对象能够相互合作 数据格式转换、API接口适配、遗留系统集成
    2 桥接模式 Bridge 将抽象部分与实现部分分离 跨平台应用、图形渲染系统、数据库驱动程序
    3 组合模式 Composite 将对象组合成树形结构表示"部分-整体"层次 文件系统、GUI组件树、公司组织架构、菜单系统
    4 装饰器模式 Decorator 动态地给对象添加新功能 咖啡店饮品定制、文本格式化、数据流处理、UI组件增强
    5 外观模式 Facade 为复杂子系统提供简单接口 家庭影院系统、计算机启动过程、API网关
    6 享元模式 Flyweight 通过共享技术有效支持大量细粒度对象 文本编辑器字符、游戏棋子、网页图标
    7 代理模式 Proxy 提供对象的替代品或占位符 延迟初始化、访问控制、远程服务、缓存代理

    行为型模式（Behavioral Patterns）¶

    序号 模式名称 英文名称 核心目的 主要应用场景
    1 解释器模式 Interpreter 定义语言文法表示并解释执行 数学表达式计算器、SQL解析、正则表达式、编程语言解释器
    2 模板方法模式 Template Method 在超类中定义算法框架，子类重写特定步骤 数据挖掘算法、Web框架、游戏AI
    3 责任链模式 Chain of Responsibility 将请求沿着处理者链进行发送 Web过滤器链、异常处理、审批流程、日志处理
    4 命令模式 Command 将请求封装成对象 GUI按钮和菜单项、宏录制、队列请求、日志和撤销操作
    5 迭代器模式 Iterator 提供访问聚合对象元素的方法 集合遍历、数据库结果集遍历、文件系统遍历
    6 中介者模式 Mediator 定义对象间的交互方式 对话框控件交互、聊天室、航空管制系统、MVC架构
    7 备忘录模式 Memento 在不破坏封装的前提下保存对象状态 文本编辑器撤销功能、游戏存档、数据库事务回滚、浏览器历史记录
    8 观察者模式 Observer 定义对象间一对多的依赖关系 事件系统、MVC架构、股票价格监控、新闻订阅
    9 访问者模式 Visitor 在不改变元素类的前提下定义新操作 编译器、AST遍历、文件系统操作、报表生成
    10 状态模式 State 让对象在内部状态变化时改变行为 游戏角色状态、订单状态、TCP连接状态、媒体播放器状态
    11 策略模式 Strategy 定义算法族并使它们可以互换 支付方式选择、排序算法选择、压缩算法选择、路径规划算法

    🔍 分析实现¶

    一、创建型¶

    1.1.单例模式（Singleton）¶

    单例模式是一种创建型设计模式，让你能够保证一个类只有一个实例，并提供一个访问该实例的全局节点。

    使用场景：

    • 数据库连接池
    • 日志记录器
    • 配置管理器
    • 线程池

    优点：

    • 保证一个类只有一个实例
    • 提供全局访问点
    • 延迟初始化，节约系统资源

    缺点：

    • 违反单一职责原则
    • 在多线程环境下需要特殊处理
    • 单元测试困难

    实现示例:

    • Python
    • Go
    • Rust

    1.2.工厂方法模式（Factory Method）¶

    工厂方法模式是一种创建型设计模式，它提供了一种创建对象的接口，但由子类决定要实例化的类是哪一个。工厂方法让类把实例化推迟到子类。

    使用场景：

    • 需要根据不同条件创建不同类型的对象
    • 不知道具体需要创建哪种类型的对象
    • 希望用户能够扩展库或框架的内部组件
    • UI组件库、数据库驱动程序、日志系统

    优点：

    • 避免创建者和具体产品之间的紧密耦合
    • 符合单一职责原则，将产品创建代码移入一个位置
    • 符合开闭原则，无需更改现有代码就可以在程序中引入新的产品类型

    缺点：

    • 代码可能会变得复杂，因为需要引入许多新的子类
    • 需要创建对应的工厂层次结构

    实现示例:

    • Python
    • Go
    • Rust

    1.3.抽象工厂模式（Abstract Factory）¶

    抽象工厂模式是一种创建型设计模式，它提供了一个接口，用于创建一系列相关或相互依赖的对象，而无需指定它们具体的类。抽象工厂模式也被称为Kit模式。

    使用场景：

    • 系统需要独立于其产品的创建、组合和表示
    • 系统需要由多个产品系列中的一个来配置
    • 需要强调一系列相关产品对象的设计以便进行联合使用
    • 跨平台UI组件库、数据库访问层、游戏中的不同主题

    优点：

    • 分离了具体的类
    • 使得易于交换产品系列
    • 有利于产品的一致性
    • 符合开闭原则

    缺点：

    • 难以支持新种类的产品
    • 增加了系统的抽象性和理解难度

    实现示例:

    • Python
    • Go
    • Rust

    1.4.建造者模式（Builder）¶

    建造者模式是一种创建型设计模式，它允许你分步骤创建复杂对象。该模式允许你使用相同的创建代码生成不同类型和形式的对象。

    使用场景：

    • 需要创建的对象有复杂的内部结构
    • 需要创建的对象的属性相互依赖，需要指定其生成顺序
    • 对象的创建过程独立于创建该对象的类
    • SQL查询构建器、HTTP请求构建器、配置对象构建

    优点：

    • 可以分步创建对象，暂缓创建步骤或递归运行创建步骤
    • 生成不同形式的产品时，可以复用相同的制造代码
    • 符合单一职责原则，可以将复杂构造代码从产品的业务逻辑中分离出来

    缺点：

    • 由于该模式需要新增多个类，因此代码整体复杂程度会有所增加

    实现示例:

    • Python
    • Go
    • Rust

    1.5.原型模式（Prototype）¶

    原型模式是一种创建型设计模式，它允许复制已有对象，而无需使代码依赖它们所属的类。该模式将克隆过程委派给被克隆的实际对象。

    使用场景：

    • 对象的创建成本较高（如需要大量计算或网络请求）
    • 需要创建大量相似对象
    • 对象的初始化需要消耗大量资源
    • 游戏中的角色复制、文档模板、配置对象复制

    优点：

    • 可以克隆对象，而无需与它们所属的具体类相耦合
    • 可以克隆预生成原型，避免反复运行初始化代码
    • 可以更方便地生成复杂对象
    • 可以用继承以外的方式来处理复杂对象的不同配置

    缺点：

    • 克隆包含循环引用的复杂对象可能会非常麻烦
    • 需要为每个类实现克隆方法

    实现示例:

    • Python
    • Go
    • Rust

    二、结构型¶

    2.1.适配器模式（Adapter Class/Object）¶

    适配器模式是一种结构型设计模式，它允许接口不兼容的对象能够相互合作。适配器模式通过创建一个包装类，将一个类的接口转换成客户端所期望的另一种接口。

    使用场景：

    • 需要使用现有的类，但其接口不符合你的需求
    • 想要创建一个可以复用的类，该类可以与其他不相关的类协同工作
    • 需要在不修改现有代码的情况下使用第三方库
    • 数据格式转换、API接口适配、遗留系统集成

    优点：

    • 可以让任何两个没有关联的类一起运行
    • 提高了类的复用性
    • 增加了类的透明性
    • 灵活性好

    缺点：

    • 过多地使用适配器，会让系统非常零乱
    • 由于在Java中类不支持多重继承，一次最多只能适配一个适配者类

    实现示例:

    • Python
    • Go
    • Rust

    2.2.桥接模式（Bridge）¶

    桥接模式是一种结构型设计模式，它可以将抽象部分与实现部分分离，使它们都可以独立地变化。桥接模式通过组合的方式建立两个类层次结构之间的联系。

    使用场景：

    • 不希望在抽象和实现之间有一个固定的绑定关系
    • 类的抽象以及它的实现都应该可以通过生成子类的方法加以扩充
    • 对一个抽象的实现部分的修改应对客户不产生影响
    • 跨平台应用、图形渲染系统、数据库驱动程序

    优点：

    • 分离抽象接口及其实现部分
    • 桥接模式提高了系统的可扩充性
    • 实现细节对客户透明
    • 符合开闭原则

    缺点：

    • 桥接模式的引入会增加系统的理解与设计难度
    • 桥接模式要求正确识别出系统中两个独立变化的维度

    实现示例:

    • Python
    • Go
    • Rust

    2.3.组合模式（Composite）¶

    组合模式是一种结构型设计模式，它可以将对象组合成树形结构来表示"部分-整体"的层次结构。组合模式使得用户对单个对象和组合对象的使用具有一致性。

    使用场景：

    • 需要表示对象的部分-整体层次结构
    • 希望用户忽略组合对象与单个对象的不同，统一地使用组合结构中的所有对象
    • 文件系统、GUI组件树、公司组织架构、菜单系统

    优点：

    • 高层模块调用简单，节点自由增加
    • 符合开闭原则，对扩展开放，对修改关闭
    • 叶节点和树枝节点实现相同的接口，所以它们之间没有差别

    缺点：

    • 树叶和树枝的声明都是实现类，而不是接口，违反了依赖倒置原则
    • 设计较复杂，客户端需要花更多时间理清类之间的层次关系

    实现示例:

    • Python
    • Go
    • Rust

    2.4.装饰器模式（Decorator）¶

    装饰器模式是一种结构型设计模式，它允许向一个现有的对象添加新的功能，同时又不改变其结构。这种类型的设计模式属于结构型模式，它是作为现有的类的一个包装。

    使用场景：

    • 需要扩展一个类的功能，或给一个类添加附加职责
    • 需要动态的给一个对象添加功能，这些功能可以再动态的撤销
    • 需要增加由一些基本功能的排列组合而产生的非常大量的功能
    • 咖啡店饮品定制、文本格式化、数据流处理、UI组件增强

    优点：

    • 装饰类和被装饰类可以独立发展，不会相互耦合
    • 装饰模式是继承的一个替代模式，装饰模式可以动态扩展一个实现类的功能
    • 装饰模式完全遵守开闭原则

    缺点：

    • 装饰模式会增加许多子类，过度使用会增加程序得复杂性
    • 动态装饰时，多层装饰时会更复杂

    实现示例:

    • Python
    • Go
    • Rust

    2.5.外观模式（Facade）¶

    外观模式是一种结构型设计模式，它能为程序库、框架或其他复杂类集合提供一个简单的接口。外观类为包含许多活动部件的复杂子系统提供一个简单的接口。

    使用场景：

    • 需要为一个复杂子系统提供一个简单接口
    • 客户程序与多个子系统之间存在很大依赖性
    • 需要构建一个层次结构的子系统
    • 家庭影院系统、计算机启动过程、API网关

    优点：

    • 减少系统相互依赖
    • 提高灵活性和安全性
    • 简化客户端调用

    缺点：

    • 不符合开闭原则，如果要改东西很麻烦，继承重写都不合适

    实现示例:

    • Python
    • Go
    • Rust

    2.6.享元模式（Flyweight）¶

    享元模式是一种结构型设计模式，它通过共享技术有效地支持大量细粒度对象的复用。享元模式以共享的方式高效地支持大量的细粒度对象，通过共享已经存在的对象来大幅度减少需要创建的对象数量、避免大量相似对象的开销，从而提高系统资源的利用率。

    使用场景：

    • 系统中存在大量相似对象
    • 需要缓冲池的场景
    • 对象的大部分状态都可以外部化
    • 文本编辑器中的字符、游戏中的棋子、网页中的图标

    优点：

    • 大大减少对象的创建，降低系统的内存，使效率提高
    • 外部状态相对独立，不会影响其内部状态

    缺点：

    • 提高了系统的复杂度，需要分离出内部状态和外部状态
    • 为了使对象可以共享，需要将一些状态外部化，这使得程序的逻辑复杂化

    实现示例:

    • Python
    • Go
    • Rust

    2.7.代理模式（Proxy）¶

    代理模式是一种结构型设计模式，它让你能够提供对象的替代品或其占位符。代理控制着对于原对象的访问，并允许在将请求提交给对象前后进行一些处理。

    使用场景：

    • 延迟初始化（虚拟代理）
    • 访问控制（保护代理）
    • 本地执行远程服务（远程代理）
    • 记录日志请求（日志记录代理）
    • 缓存请求结果（缓存代理）
    • 智能引用

    优点：

    • 可以在客户端毫无察觉的情况下控制服务对象
    • 如果客户端对服务对象的生命周期没有特殊要求，你可以对生命周期进行管理
    • 即使服务对象还未准备好或不存在，代理也可以正常工作
    • 符合开闭原则

    缺点：

    • 代码可能会变得复杂，因为需要新建许多类
    • 服务响应可能会延迟

    实现示例:

    • Python
    • Go
    • Rust

    三、行为型¶

    3.1.解释器模式（Interpreter）¶

    解释器模式是一种行为型设计模式，它定义了一种语言的文法表示，并定义一个解释器来处理这种语言中的句子。解释器模式为语言中的每一个文法规则定义一个类，并且用这些类来解释语言中的句子。

    使用场景：

    • 需要解释执行语言句子或表达式
    • 语法相对简单，对效率要求不高
    • 数学表达式计算器、SQL解析、正则表达式、编程语言解释器

    优点：

    • 易于改变和扩展文法
    • 实现文法较为容易
    • 增加新的解释表达式较为方便

    缺点：

    • 对于复杂文法难以维护
    • 执行效率较低
    • 会产生大量的类

    实现示例:

    • Python
    • Go
    • Rust

    3.2.模板方法模式（Template Method）¶

    模板方法模式是一种行为型设计模式，它在超类中定义了一个算法的框架，允许子类在不改变算法整体结构的情况下重写算法的特定步骤。模板方法模式通过把不变的行为搬移到超类，去除了子类中的重复代码。

    使用场景：

    • 多个类有相同的方法，逻辑相同，细节不同
    • 重要、复杂的算法，可以把核心算法设计为模板方法
    • 重构时，把相同的代码抽取到父类中
    • 数据挖掘算法、Web框架、游戏AI

    优点：

    • 提取了公共代码部分，易于维护
    • 行为由父类控制，子类实现
    • 符合开闭原则

    缺点：

    • 每一个不同的实现都需要一个子类来实现，导致类的个数增加
    • 类数目的增加，间接地增加了系统实现的复杂度

    实现示例:

    • Python
    • Go
    • Rust

    3.3.责任链模式（Chain of Responsibility）¶

    责任链模式是一种行为型设计模式，它让你能够将请求沿着处理者链进行发送。收到请求后，每个处理者均可对请求进行处理，或将其传递给链上的下个处理者。责任链模式避免了请求发送者与接收者之间的耦合关系。

    使用场景：

    • 有多个对象可以处理同一个请求，具体哪个对象处理该请求在运行时刻自动确定
    • 在不明确指定接收者的情况下，向多个对象中的一个提交一个请求
    • 可动态指定一组对象处理请求
    • Web过滤器链、异常处理、审批流程、日志处理

    优点：

    • 降低耦合度，发送者和接收者解耦
    • 简化了对象，对象不需要知道链的结构
    • 增强给对象指派职责的灵活性
    • 增加新的请求处理类很方便

    缺点：

    • 不能保证请求一定被接收
    • 系统性能将受到一定影响，而且在进行代码调试时不太方便
    • 可能会造成循环调用

    实现示例:

    • Python
    • Go
    • Rust

    3.4.命令模式（Command）¶

    命令模式是一种行为型设计模式，它将请求封装成对象，从而使你可用不同的请求对客户进行参数化；对请求排队或记录请求日志，以及支持可撤销的操作。命令模式把发出命令的责任和执行命令的责任分割开，委派给不同的对象。

    使用场景：

    • 需要将请求调用者和请求接收者解耦
    • 需要在不同的时间指定、排列和执行请求
    • 需要支持撤销操作
    • 需要支持日志记录、事务等功能
    • GUI按钮和菜单项、宏录制、队列请求、日志和撤销操作

    优点：

    • 降低了系统耦合度
    • 新的命令可以很容易添加到系统中去
    • 可以比较容易地设计一个命令队列和宏命令
    • 可以方便地实现对请求的撤销和重做

    缺点：

    • 使用命令模式可能会导致某些系统有过多的具体命令类

    实现示例:

    • Python
    • Go
    • Rust

    3.5.迭代器模式（Iterator）¶

    迭代器模式是一种行为型设计模式，它提供了一种方法来访问聚合对象中各个元素，而又不暴露该对象的内部表示。迭代器模式将遍历行为从聚合对象中分离出来，并将其封装到一个迭代器对象中。

    使用场景：

    • 需要访问一个聚合对象的内容而无需暴露它的内部表示
    • 需要支持对聚合对象的多种遍历
    • 需要为遍历不同的聚合结构提供一个统一的接口
    • 集合遍历、数据库结果集遍历、文件系统遍历

    优点：

    • 支持以不同的方式遍历一个聚合对象
    • 迭代器简化了聚合类
    • 在同一个聚合上可以有多个遍历
    • 增加新的聚合类和迭代器类都很方便

    缺点：

    • 由于迭代器模式将存储数据和遍历数据的职责分离，增加新的聚合类需要对应增加新的迭代器类

    实现示例:

    • Python
    • Go
    • Rust

    3.6.中介者模式（Mediator）¶

    中介者模式是一种行为型设计模式，它定义了一系列对象之间的交互方式，使原本直接交互的对象通过中介者对象来进行交互，从而使各对象不需要显式地相互引用，让其耦合松散，而且可以独立地改变它们之间的交互。

    使用场景：

    • 系统中对象之间存在比较复杂的引用关系，导致它们之间的依赖关系结构混乱而且难以复用该对象
    • 想通过一个中间类来封装多个类中的行为，而又不想生成太多的子类
    • 对话框中的控件交互、聊天室、航空管制系统、MVC架构

    优点：

    • 减少类间的依赖，把多对多的依赖关系变成一对多的依赖关系
    • 类间的各个对象之间的交互被封装在中介者当中，使得系统的交互更加清晰
    • 中介者模式符合迪米特法则

    缺点：

    • 中介者会庞大，变得复杂难以维护

    实现示例:

    • Python
    • Go
    • Rust

    3.7.备忘录模式（Memento）¶

    备忘录模式是一种行为型设计模式，它在不破坏封装性的前提下，捕获一个对象的内部状态，并在该对象之外保存这个状态，以便以后当需要时能将该对象恢复到原先保存的状态。备忘录模式又叫快照模式。

    使用场景：

    • 需要保存/恢复数据的相关状态场景
    • 提供一个可回滚的操作
    • 需要监控的副本场景中，但又不希望提供它的全部访问权限
    • 文本编辑器的撤销功能、游戏存档、数据库事务回滚、浏览器历史记录

    优点：

    • 给用户提供了一种可以恢复状态的机制，可以使用户能够比较方便地回到某个历史的状态
    • 实现了信息的封装，使得用户不需要关心状态的保存细节

    缺点：

    • 消耗资源，如果类的成员变量过多，势必会占用比较大的资源
    • 而且每一次保存都会消耗一定的内存

    实现示例:

    • Python
    • Go
    • Rust

    3.8.观察者模式（Observer）¶

    观察者模式是一种行为型设计模式，它定义了对象间的一种一对多的依赖关系，当一个对象的状态发生改变时，所有依赖于它的对象都得到通知并被自动更新。观察者模式又叫发布-订阅模式。

    使用场景：

    • 当一个抽象模型有两个方面，其中一个方面依赖于另一个方面
    • 当对一个对象的改变需要同时改变其他对象，而不知道具体有多少对象有待改变
    • 当一个对象必须通知其他对象，而它又不能假定其他对象是谁
    • 事件系统、MVC架构、股票价格监控、新闻订阅

    优点：

    • 观察者和被观察者是抽象耦合的
    • 建立一套触发机制

    缺点：

    • 如果一个被观察者对象有很多的直接和间接的观察者的话，将所有的观察者都通知到会花费很多时间
    • 如果在观察者和观察目标之间有循环依赖的话，观察目标会触发它们之间进行循环调用

    实现示例:

    • Python
    • Go
    • Rust

    3.9.访问者模式（Visitor）¶

    访问者模式是一种行为型设计模式，它能让你在不改变各元素的类的前提下定义作用于这些元素的新操作。访问者模式将数据结构与数据操作分离，使得操作集合可相对自由地演化。

    使用场景：

    • 对象结构中对象对应的类很少改变，但经常需要在此对象结构上定义新的操作
    • 需要对一个对象结构中的对象进行很多不同的并且不相关的操作
    • 对象结构不能改变但是需要拓展功能
    • 编译器、AST遍历、文件系统操作、报表生成

    优点：

    • 符合单一职责原则
    • 优秀的扩展性
    • 灵活性

    缺点：

    • 具体元素对访问者公布细节，违反了迪米特法则
    • 具体元素变更比较困难
    • 违反了依赖倒置原则，依赖了具体类，没有依赖抽象

    实现示例:

    • Python
    • Go
    • Rust

    3.10.状态模式（State）¶

    状态模式是一种行为型设计模式，它让你能在一个对象的内部状态变化时改变其行为，使该对象看上去就像是改变了自身所属的类一样。状态模式将与特定状态相关的行为局部化，并且将不同状态的行为分割开来。

    使用场景：

    • 对象的行为依赖于它的状态，并且可以根据它的状态改变而改变它的相关行为
    • 代码中包含大量与对象状态有关的条件语句
    • 状态转换逻辑复杂，且状态数量较多
    • 游戏角色状态、订单状态、TCP连接状态、媒体播放器状态

    优点：

    • 封装了转换规则
    • 枚举可能的状态，在枚举状态之前需要确定状态种类
    • 将所有与某个状态有关的行为放到一个类中，并且可以方便地增加新的状态
    • 允许状态转换逻辑与状态对象合成一体，而不是某一个巨大的条件语句块

    缺点：

    • 状态模式的使用必然会增加系统类和对象的个数
    • 状态模式的结构与实现都较为复杂，如果使用不当将导致程序结构和代码的混乱

    实现示例:

    • Python
    • Go
    • Rust

    3.11.策略模式（Strategy）¶

    策略模式是一种行为型设计模式，它定义了一系列算法，并将每个算法封装起来，使它们可以互换。策略模式让算法独立于使用它的客户端而变化，使得算法可以在运行时动态地选择和切换。

    使用场景：

    • 需要在运行时选择算法
    • 有多种实现同一功能的方式
    • 需要避免使用条件语句选择算法
    • 支付方式选择、排序算法选择、压缩算法选择、路径规划算法

    优点：

    • 算法可以自由切换
    • 避免使用多重条件判断
    • 扩展性良好，符合开闭原则

    缺点：

    • 策略类会增多
    • 所有策略类都需要对外暴露

    实现示例:

    • Python
    • Go
    • Rust
