## Core Principles
- Write production-ready, secure, and maintainable code

    示例2¶
    # PROJECT DEVELOPMENT RULES & PROMPTS (CURRENT PROJECT ONLY)
## Core Constraints (NON-NEGOTIABLE)
1. No breaking changes to existing business logic and project structure
2. Zero intermediate/generated/explain files are allowed in the project root
3. Minimal modification only, no full refactor of existing valid code
4. All native build/run commands are preserved, no custom wrapper scripts
5. All sensitive configs use environment variables, no hardcode plaintext info

## 1. Frontend Development Rules (TS/JS/Vue/React)
### 1.1 Code Standard
- Use ES6+ syntax for all JS/TS code, async/await for promise handling
- Vue3: Composition API only for new code, preserve existing Options API
- React: Follow Hooks Rules strictly, functional components priority
- TS strict mode: enabled in tsconfig.json, explicit type for all variables
### 1.2 Build & Run
- Native commands: npm run dev / npm run build (no custom build scripts)
- Dependencies: fixed version in package.json, no ^/~ fuzzy version
- Forbid unused npm packages, clean up redundant deps regularly

## 2. Backend Development Rules (Python/Go/Rust/Bash/Docker)
### 2.1 Python
- PEP8 code standard, venv for isolated environment
- Native run: python3 main.py / uvicorn main:app --reload
- Forbid __pycache__/.pyc files, add to .gitignore
### 2.2 Go
- Go fmt for code format, go mod for dependency management
- Native run: go run main.go / go build -o app main.go
- No unused imports/packages, explicit error return
### 2.3 Rust
- Stable Rust toolchain only, cargo for build/run
- Native run: cargo run / cargo build --release
- No unsafe Rust unless specified, use match instead of unwrap()
### 2.4 Bash/Shell/Docker
- POSIX-compliant shell scripts with set -euo pipefail
- Docker: official slim base images, fixed version tags, multi-stage build
- No executable permission for non-script files

## 3. Mobile Development Rules (React Native / Flutter)
- React Native: preserve metro bundler config, npm run android/ios
- Flutter: flutter run / flutter build apk/ios, no redundant plugins
- Forbid unused assets/resources, compress images to reduce app size

## 4. Project-Specific Prompts (IMPORTANT)
### 4.1 Build & Deploy
- [PROMPT] Run command before build: npm install / pip install -r requirements.txt / go mod tidy
- [PROMPT] Production build must disable source map for frontend
- [PROMPT] Rust production build: cargo build --release (no debug mode)
### 4.2 Common Issues
- [PROMPT] Python: activate venv first before running the project
- [PROMPT] Go: set GOPROXY if network issue occurs
- [PROMPT] Rust: rustup update to latest stable version
### 4.3 Team Collaboration
- [PROMPT] All code commits follow conventional commit standard
- [PROMPT] No large binary files in git, use .gitignore strictly

## Final Rule
All rules are additive only, no deletion of existing valid code/config/features
ZERO extra files generated in the project, all build cache are ignored

    二、Windsurf¶

    参考：如何为 Windsurf 配置全局和项目规则

    .windsurfrules 是为 Windsurf 提供的一个配置文件，允许开发者定义项目中 AI 的行为规则。

    有两组规则：

    • global_rules.md 为独立于工作区遵循的全局规则
    • .windsurfrules 为特定于工作区的本地规则。.windsurfrules 文件的大小限制为 6,000 字符，

    2.1.全局规则¶

    global_rules.md 文件保存全局规则。

    示例1¶

    # 🌍 Global Baseline Rules for AI-Assisted Development

## 1. 🚨 Primary Directives (MUST FOLLOW)
- **Security First**: NEVER generate code that hardcodes secrets, API keys, or credentials. Always assume input needs validation and sanitization to prevent injection attacks.
- **Legal & Licensing**: Do not generate code that infringes on copyrights or patents. Prefer permissive开源 licenses (MIT, Apache 2.0) when creating new code snippets.
- **No Placeholders**: NEVER leave `TODO`, `FIXME`, or placeholder values in code meant for production. If something is incomplete, state it explicitly in comments.

## 2. 📏 Core Code Quality
- **Clarity over Cleverness**: Write code that is straightforward and easily understandable by intermediate developers. Avoid overly complex one-liners.
- **Error Handling**: Always consider failure cases. Silent failures are unacceptable.
- **Consistency**: Within any single file or logical unit, maintain consistent style and patterns.

## 3. ⚡ Performance & Architecture
- **Resource Awareness**: Be mindful of potential performance impacts (e.g., unnecessary large loops, blocking calls, memory leaks).
- **Idiomatic Patterns**: Use the standard, community-accepted idioms for the given programming language and framework.

## 4. 🔄 Interaction Protocol
- **Assume Legacy Context**: Your suggestions might be applied to an existing codebase. Prefer incremental, compatible improvements over rewriting from scratch unless explicitly requested.
- **Flag Major Changes**: If your suggestion involves a significant refactor, dependency addition, or architectural shift, clearly state the trade-offs and reasoning upfront.

    示例2¶

    # WINDSURF - GLOBAL DEVELOPMENT RULES (ALL PLATFORMS & LANGUAGES)
## CORE CONSTRAINTS [HIGH PRIORITY, NO EXCEPTIONS]
1. Minimal modification only, no full refactor or rewrite of existing valid code/configuration/features
2. Never alter original core functions of any project/language/framework unless explicitly specified
3. Zero auto-generated files: Forbid all intermediate/cache/temp/explain/docs/log files in any project
4. Full native ecology compatibility, no non-standard syntax/custom forced specifications
5. Cross-platform & cross-language consistency, unified base rules with no conflicting configurations
6. All sensitive information uses environment variables only, no plaintext hardcoding

## 1. GLOBAL SHARED BASE RULES [APPLY TO ALL DEVELOPMENT DIRECTIONS]
### 1.1 File & Code Standard
- File encoding: UTF-8 only, no BOM header for all text/code files
- Line endings: LF (Unix) for cross-platform compatibility, disable CRLF auto-convert
- Indentation: 4 spaces only, no tab characters, unified for all languages
- File naming: kebab-case for directories/static files; camelCase for JS/TS vars/funcs; PascalCase for class/interface/Component; snake_case for Python/Go/Rust/Shell vars/funcs; UPPER_SNAKE_CASE for all constants
- Code cleanup: Remove commented dead code/unused imports/unnecessary comments permanently
- Path rule: Relative paths only for internal project references, no hardcoded absolute paths

### 1.2 Dependency & Build Rule
- All dependencies use fixed specific versions, no fuzzy version symbols (^/~) to avoid compatibility issues
- Only use official stable package sources, forbid untrusted third-party mirrors
- Clean up unused dependencies/packages regularly, no redundant dependencies in project
- Preserve native build/run commands for all languages, no custom wrapper scripts or pipelines
- Build cache files are allowed as language-native necessity only, auto-ignore and not commit to version control

### 1.3 Security & Performance Rule
- All network requests add timeout configuration, avoid infinite waiting
- Minimal resource consumption: Forbid memory leaks, infinite loops and deep nested logic
- No executable permissions for non-script files; limit sudo usage for shell/docker operations
- All IO/network/database operations add exception capture, no unhandled errors

## 2. FRONTEND DEVELOPMENT RULES [TS / JAVASCRIPT / VUE 2&3 / REACT / REACT-TS]
### 2.1 TypeScript & JavaScript Core Rules
- Mandatory ES6+ syntax: Arrow function, ES Module import/export, Promise/async-await instead of ES5 polyfill
- TypeScript: Enable strict mode in tsconfig.json for new code, preserve non-strict config for existing TS projects
- No implicit any type for TS, explicit type definitions for all variables/functions/components
- Forbid global variable pollution, use ES Module/namespace to isolate code scope
- Prohibit eval()/with() syntax to avoid security risks; complete Promise error catch for all async logic
- No production source map files, disable .map generation to reduce bundle size

### 2.2 Vue (2.x / 3.x) Exclusive Rules
- Preserve original Vue syntax: Support both Options API and Composition API, no forced migration
- SFC standard: Keep <template> + <script> + <style> structure, use scoped attribute for component style isolation
- All props add type/required/default validation; no direct props modification in child components
- Use emit for parent-child communication, Pinia/Vuex for state management with standard mutations/actions
- Preserve native Vue build commands: npm run dev/serve/build, no custom vite/webpack config override

### 2.3 React / React-TS Exclusive Rules
- Follow official React Hooks Rules strictly: No hooks in loops/conditions/nested functions
- Keep existing Class Components, use Functional Components + Hooks only for new code
- State immutability: No direct mutation of state/props, return new objects/arrays for updates
- Minimal direct DOM manipulation, use React ref only for necessary native DOM operations
- Preserve native React build commands: npm run start/build, no custom config rewrite

## 3. BACKEND DEVELOPMENT RULES [PYTHON / GO / RUST / BASH/SHELL / DOCKER]
### 3.1 Python Backend Rules
- Follow PEP 8 code specification, minimal compliance only (no forced format rewrite for existing code)
- Use virtual environment (venv/poetry/pipenv) for isolated dependencies, no global package installation
- Forbid __pycache__/.pyc compiled cache files, add to ignore list automatically
- All IO/db/network operations add try-except exception capture, no bare except clauses
- Preserve native run/build commands: python3 main.py / uvicorn/fastapi run / python manage.py runserver
- No redundant framework modification, keep original Django/Flask/FastAPI configurations

### 3.2 Go Backend Rules
- Follow official Go code style, use gofmt for auto-formatting, no third-party linter override
- Mandatory go mod for dependency management, fixed versions in go.mod/go.sum
- Explicit error return for all functions, no _ ignore of critical errors
- Use goroutine + channel only for necessary concurrent logic, avoid race conditions
- Preserve native commands: go run / go build / go test, no custom build flags
- Forbid unused imports/packages/variables, clean up redundant code automatically

### 3.3 Rust Backend Rules
- Stable Rust toolchain only for production, managed by rustup, no nightly/beta version
- Follow native rustfmt/cargo clippy rules, fix error-level warnings only
- Cargo.toml uses fixed dependency versions, no redundant features enabled
- No unsafe Rust code unless explicitly specified for critical performance scenarios
- Use match/if let for error handling, avoid unwrap() in production code
- Preserve native cargo commands: cargo run / cargo build --release / cargo test
- Ignore native target build cache, no extra generated files in project root

### 3.4 Bash / Shell Script Rules (POSIX Compatible)
- All scripts start with standard shebang (#!/bin/bash / #!/bin/sh), no missing shebang
- Add set -euo pipefail at script top: exit on error, check unset variables, capture pipe failures
- POSIX-compliant syntax only, no shell-specific features for max cross-shell compatibility
- snake_case for variables/functions, UPPER_SNAKE_CASE for constants, .sh suffix for all shell files
- No hardcoded paths/sensitive info, use env vars for dynamic configuration
- Execution rule: chmod +x script.sh && ./script.sh only, no unnecessary sudo

### 3.5 Docker Rules (Dockerfile / Docker Compose)
- Use official minimal base images (alpine/slim), fixed version tags only (no latest tag)
- Multi-stage build for minimal final image size, separate builder and runtime stages
- Create non-root user for runtime containers, no root user by default
- Forbid large unnecessary dependencies, clean up build cache in Dockerfile layers
- Preserve native docker commands: docker build / docker run / docker compose up
- No auto-generated docker-compose override files, keep original compose config

## 4. MOBILE DEVELOPMENT RULES [CROSS-PLATFORM & NATIVE]
### 4.1 Cross-Platform Mobile (React Native / Flutter)
- React Native: Follow frontend React rules + RN native specs, preserve metro bundler config
- Flutter: Use dartfmt for auto-format, fixed pubspec.yaml dependency versions
- Preserve native run/build commands: npm run android/ios (RN) | flutter run/build (Flutter)
- Minimal native bridge code for RN/Flutter, no redundant Kotlin/Swift/Java logic
- Forbid unused assets/resources, compress images/videos to reduce app package size

### 4.2 Native Mobile (iOS / Android)
- iOS (Swift/Objective-C): Follow Apple official syntax, Xcode native build commands only
- Android (Kotlin/Java): Follow Google Android style, Gradle fixed dependency versions
- Request only necessary permissions (camera/location/storage), add manifest descriptions
- No hardcoded app config, use environment variables for API base URL and sensitive info
- Generate final APK/IPA install packages only, no intermediate build cache files

## 5. FORBIDDEN OPERATIONS [ABSOLUTE RULE, NO EXCEPTIONS]
1. Generate any intermediate/cache/temp/explain/docs/log files in project root or source directory
2. Modify original core functions/business logic/configuration of existing projects
3. Use untrusted third-party dependencies/mirrors/packages with security risks
4. Hardcode plaintext sensitive information (password/token/key/secret) in any code file
5. Add redundant custom syntax or non-standard specifications that break native compatibility
6. Leave commented dead code/unused imports/unnecessary files in project
7. Use global installation of dependencies for project-specific development

## FINAL VALIDATION RULE [BOTTOM LINE]
All rules are additive only, no destructive changes to any existing project.
All configurations follow native language/framework rules, no over-engineering.
Zero generated files in all scenarios, full compliance with minimal modification principle.

  - Single empty line at file end

- Headers
  - ATX style with space after #
  - No emoji, proper nesting, blank lines
- Lists/Code
  - 2 space indent, proper markers
  - Language-specified fenced blocks
  - Proper link syntax [text](url)

  - Tables: headers, alignment, consistent width

    2.2.8. 安全性¶

    定义安全性要求：

    ### Security

- Input Validation: Validate and sanitize ALL user inputs on the backend(sanitize data, validate types, escape properly, secure uploads).
- Authentication/Authorization: Implement robust auth (e.g., JWT, OAuth 2.0) and role-based access control (RBAC).
- Secrets Management: Never hardcode secrets. Use environment variables or dedicated secret managers (Vault, K8s Secrets).
- Dependencies: Regularly scan for and update vulnerable dependencies (using Snyk, Dependabot).
- OWASP Top 10: Actively guard against common vulnerabilities like injection, XSS, and CSRF.
