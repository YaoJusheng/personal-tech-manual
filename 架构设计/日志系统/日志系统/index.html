    1.日志工具¶

    1.1. python-json-logger¶

    简介¶

    • python-json-logger 是 logging 模块的扩展，支持将日志格式化为 JSON。
    • 适用于需要结构化日志记录（如 JSON）且想与 Python 标准日志系统兼容的场景。

    优点¶

    1. 轻量级，基于 Python 原生 logging。
    2. 易于集成到已有的 logging 配置中。
    3. 输出 JSON 格式，适合与日志分析工具（如 ELK 或 Loki）配合使用。

    缺点¶

    1. 功能单一，仅在 logging 上扩展 JSON 格式。
    2. 配置较复杂，无法提供额外的日志增强功能（如异步写入、性能优化等）。

    示例代码¶

    import logging
from pythonjsonlogger import jsonlogger

# 配置日志
logger = logging.getLogger("my_logger")
logger.setLevel(logging.DEBUG)

# 设置处理器和 JSON 格式
log_handler = logging.StreamHandler()
formatter = jsonlogger.JsonFormatter(
    "%(asctime)s %(levelname)s %(message)s %(name)s"
)
log_handler.setFormatter(formatter)
logger.addHandler(log_handler)

# 使用日志
logger.info("This is an info log", extra={"user": "alice", "action": "login"})
logger.error("This is an error log", extra={"user": "bob", "action": "logout"})

    输出示例（JSON 格式）：

    {"asctime": "2024-11-26 10:00:00", "levelname": "INFO", "message": "This is an info log", "name": "my_logger", "user": "alice", "action": "login"}

    1.2. loguru¶

    简介¶

    • loguru 是一个现代化的日志库，提供了开箱即用的强大日志功能。
    • 适合想要快速实现高级日志功能但不想配置 logging 的场景。

    优点¶

    1. 易用，开箱即用，无需复杂配置。
    2. 提供丰富的功能：异步写入、动态日志级别、文件轮转、丰富的格式化选项。
    3. 支持漂亮的彩色控制台日志输出。

    缺点¶

    1. 与标准 logging 不完全兼容（例如标准库中的一些扩展可能需要手动适配）。
    2. 过于强大可能显得臃肿，对一些轻量需求不适合。

    示例代码¶

    from loguru import logger

# 配置日志
logger.add("app.log", rotation="10 MB", retention="10 days", compression="zip", format="{time} {level} {message}")

# 使用日志
logger.info("This is an info log with user: {user}", user="alice")
logger.error("This is an error log with action: {action}", action="logout")

    输出示例（控制台带颜色，文件是纯文本或 JSON 格式）：

    2024-11-26 10:00:00.123 | INFO     | This is an info log with user: alice
2024-11-26 10:00:01.456 | ERROR    | This is an error log with action: logout

    1.3. structlog¶

    简介¶

    • structlog 专注于结构化日志，与标准 logging 和 JSON 格式化完美结合。
    • 非常适合复杂应用（微服务、大型项目）中需要高度结构化日志的场景。

    优点¶

    1. 高度可定制，可与多种日志处理器（如 JSON 或普通文本格式）结合。
    2. 支持异步日志记录。
    3. 提供日志增强功能（上下文变量绑定、链式日志构造）。

    缺点¶

    1. 起步较难，配置复杂。
    2. 对于简单的日志需求可能显得过于繁琐。

    示例代码¶

    import structlog
import logging
from pythonjsonlogger import jsonlogger

# 配置基础日志
logging.basicConfig(level=logging.INFO)
handler = logging.StreamHandler()
handler.setFormatter(jsonlogger.JsonFormatter())
logging.getLogger().addHandler(handler)

# 配置 structlog
structlog.configure(
    processors=[
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.JSONRenderer()
    ],
    context_class=dict,
    wrapper_class=structlog.make_filtering_bound_logger(logging.INFO),
    logger_factory=structlog.stdlib.LoggerFactory(),
    cache_logger_on_first_use=True,
)

# 使用 structlog
logger = structlog.get_logger()

logger.info("User login", user="alice", action="login")
logger.error("User logout failed", user="bob", action="logout", error="timeout")

    输出示例（JSON 格式）：

    {"timestamp": "2024-11-26T10:00:00", "event": "User login", "user": "alice", "action": "login"}
{"timestamp": "2024-11-26T10:00:01", "event": "User logout failed", "user": "bob", "action": "logout", "error": "timeout"}

    1.4.1.第三方工具对比¶

    1.4.2.一些python日志配置的解析¶

    1.disable_existing_loggers¶
    1. 禁用或忽略现有日志记录器：当使用 logging.config.dictConfig() 或 logging.config.fileConfig() 等函数加载新的日志配置时，disable_existing_loggers 选项决定了是否禁用在配置文件之外已经创建的日志记录器。如果设置为 True，则新配置会禁用所有未在配置中明确定义的现有日志记录器；如果设置为 False，则这些日志记录器不会被禁用，将继续按照它们原来的配置工作。
    2. 控制日志配置的覆盖行为：这个选项允许开发者控制新的日志配置是否覆盖旧的配置。如果设置为 True，则新配置中的设置会覆盖所有未明确列出的日志记录器的配置；如果设置为 False，则新配置不会影响那些已经存在的、未在新配置中定义的日志记录器。
    3. 避免意外的日志记录器禁用：在某些情况下，开发者可能希望保留应用程序中某些部分的默认日志记录器行为。在这种情况下，将 disable_existing_loggers 设置为 False 可以确保新的配置不会意外地禁用这些日志记录器。
    4. 灵活的日志配置管理：在复杂的应用程序中，可能需要对不同的日志记录器应用不同的配置策略。disable_existing_loggers 提供了一种机制，允许开发者在全局配置和局部配置之间进行平衡，从而实现更灵活的日志管理。

    建议和使用示例：

    • 如果您希望在新的日志配置中重新定义所有记录器的配置，并且不希望旧的日志记录器影响新的配置，可以将 disable_existing_loggers 设置为 True。
    • 如果您希望保留现有记录器的配置，并且不希望新的配置覆盖它们，通常会将 disable_existing_loggers 设置为 False。这样，您就可以只重新定义需要修改的日志记录器，而保留其他记录器的现有配置。
    2.propagate¶
    1. 控制日志消息的传递：propagate 属性允许你控制日志消息是否应该传递给父级 Logger。如果设置为 True，则日志消息会向上传播；如果设置为 False，则日志消息不会传递给父级 Logger，只会在当前 Logger 上处理。
    2. 避免重复日志记录：在某些情况下，你可能不希望相同的日志消息在多个 Logger 上被记录。通过将 propagate 设置为 False，你可以防止日志消息在父级 Logger 上重复记录。
    3. 灵活的日志管理：propagate 提供了一种机制，允许开发者在不同的 Logger 级别上进行更细粒度的日志管理。这对于大型应用程序尤其有用，因为它们可能需要在不同的组件或模块级别上有不同的日志记录策略

    1.5.推荐与建议¶

    选择工具的关键点：

    1. 简单需求（标准日志 + JSON 输出）:
    2. 选择 python-json-logger。
    3. 优势：与 logging 兼容，轻量级，简单配置。
    4. 开发便捷性和多功能（现代化工具）:
    5. 选择 loguru。
    6. 优势：无需复杂配置，支持文件轮转和异步，适合中小型项目。
    7. 复杂场景和微服务架构（高度结构化日志）:
    8. 选择 structlog。
    9. 优势：支持上下文绑定、JSON 原生支持、灵活性强。

    1.6.最佳实践¶

    1. 对于中型项目（需要 JSON 日志和开发效率）:

    推荐使用 loguru + 自定义 JSON 格式化。

    from loguru import logger
import json

def json_formatter(message):
    """自定义 JSON 格式"""

    2.3.日志监控分析¶

    2.3.1.ELK¶

    ELK是三个开源软件的缩写，分别为：Elasticsearch 、 Logstash以及Kibana , 它们都是开源软件。不过现在还新增了一个Beats，它是一个轻量级的日志收集处理工具(Agent)，Beats占用资源少，适合于在各个服务器上搜集日志后传输给Logstash，官方也推荐此工具，目前由于原本的ELK Stack成员中加入了 Beats 工具所以已改名为Elastic Stack。

    Elastic Stack 组件包含：

    • Elasticsearch 是一个实时的分布式搜索和分析引擎，它可以用于全文搜索，结构化搜索以及分析。它是一个建立在全文搜索引擎 Apache Lucene 基础上的搜索引擎，使用 Java 语言编写。细可参考Elasticsearch权威指南

    • Logstash 是一个具有实时渠道能力的数据收集引擎，主要用于日志的收集与解析，并将其存入 ElasticSearch 中。

    • Kibana 是一款基于 Apache 开源协议，使用 JavaScript 语言编写，为 Elasticsearch 提供分析和可视化的 Web 平台。它可以在 Elasticsearch 的索引中查找，交互数据，并生成各种维度的表图。

    • Beats 在这里是一个轻量级日志采集器，其实Beats家族有6个成员，早期的ELK架构中使用Logstash收集、解析日志，但是Logstash对内存、cpu、io等资源消耗比较高。相比 Logstash，Beats所占系统的CPU和内存几乎可以忽略不计.

    ELK Stack （5.0版本之后）--> Elastic Stack == （ELK Stack + Beats）。目前Beats包含六种工具：

    • Packetbeat： 网络数据（收集网络流量数据）

    • Metricbeat： 指标 （收集系统、进程和文件系统级别的 CPU 和内存使用情况等数据）

    • Filebeat： 日志文件（收集文件数据）

    • Winlogbeat： windows事件日志（收集 Windows 事件日志数据）

    • Auditbeat：审计数据 （收集审计日志）

    • Heartbeat：运行时间监控 （收集系统运行时的数据）

    上述 Filebeat 是一个轻量级的数据传送工具，主要用于转发和集中日志数据。它是 ELK（Elasticsearch、Logstash、Kibana）日志系统中的一个组件，尤其擅长于收集和转发日志事件。Filebeat 监视您指定的日志文件或位置，收集日志事件，并将它们转发到 Elasticsearch 或 Logstash 进行索引。引入 Filebeat 作为日志搜集器，主要是为了解决 Logstash 开销大的问题。相比 Logstash，Filebeat 所占系统的 CPU 和内存几乎可以忽略不计。

    ELK Stack 的特性和优势：

    1. 插件生态系统：

    2. 优势： ELK Stack 拥有庞大的插件生态系统，允许用户通过 Logstash 插件、Elasticsearch 插件和 Kibana 插件扩展功能。

    3. 文档导向存储：

    4. 优势： Elasticsearch 是一个文档导向的存储系统，支持更灵活的数据模型和复杂的查询。

    5. Beats 生态系统：

    6. 优势： ELK Stack 配套的 Beats 生态系统包括 Filebeat、Metricbeat、Packetbeat 等，提供更轻量级的数据采集器。

    ELK解决⽅案的问题：

    1. 不能处理多⾏⽇志，⽐如Mysql慢查询，Tomcat/Jetty应⽤的Java异常打印
    2. 不能保留原始⽇志，只能把原始⽇志分字段保存，这样搜索⽇志结果是⼀堆JSON格式⽂本，⽆法阅 读。
    3. 不符合正则表达式匹配的⽇志⾏，被全部丢弃。

    2.3.2.Graylog¶

    Graylog 是一个简单易用、功能较全面的日志管理工具，graylog也采用Elasticsearch作为存储和索引以保障性能，MongoDB用来存储少量的自身配置信息，master-node模式具有很好的扩展性，UI上自带的基础查询与分析功能比较实用且高效，支持LDAP、权限控制并有丰富的日志类型和标准（如syslog，GELF）并支持基于日志的报警。

    在日志接收方面通常是网络传输，可以是TCP也可以是UDP，在实际生产环境量级较大多数采用UDP，也可以通过MQ来消费日志。

    官方文档：graylog

    架构：

    1. Graylog Server:

    2. Graylog Server 是 Graylog 平台的核心组件，负责接收、处理和存储日志数据。

    3. 它基于 Elasticsearch 存储日志数据，并使用 MongoDB 存储配置信息和元数据。

    4. Graylog Server 还处理用户界面的请求，并与其他组件协同工作，如消息传递、数据索引等。

    5. Elasticsearch:

    6. Elasticsearch 用作 Graylog 的数据存储后端，提供了强大的全文搜索和分析功能。

    7. Graylog Server将日志数据索引到 Elasticsearch 中，使用户能够通过高效的搜索和查询功能检索和分析数据。

    8. MongoDB:

    9. MongoDB 用于存储 Graylog 的配置信息和元数据。

    10. Graylog Server 和其他组件使用 MongoDB 来管理用户和系统配置、仪表板配置等。

    11. Graylog Web Interface:

    12. Graylog Web Interface 是用户与 Graylog 交互的主要界面。

    13. 用户可以通过 Web Interface 查询、过滤和可视化日志数据，创建仪表板，管理警报规则等。

    14. Graylog Collector Sidecar:

    15. Graylog Collector Sidecar 是一个轻量级的代理，负责收集日志数据并将其发送到 Graylog Server。

    16. 它支持多种日志收集器，如 Beats、NXLog、Filebeat 等。

    Graylog 的特性和优势：

    1. 用户界面和易用性：

    2. 优势： Graylog 的用户界面被认为更直观和易于使用，适用于初学者和有经验的用户。

    3. 流和管道：

    4. 优势： Graylog 引入了“流”和“管道”的概念，允许用户更灵活地组织和处理日志数据。

    5. 企业支持：

    6. 优势： Graylog 提供企业版，提供额外的功能和支持选项，适用于企业级需求。

    7. 内置警报：

    8. 优势： Graylog 具有内置的警报系统，允许用户基于查询条件设置警报规则。

    9. 日志归档：

    10. 优势： Graylog 支持日志归档，使用户能够将旧的日志数据归档到长期存储。

    总体而言，ELK Stack 和 Graylog 在日志管理和分析领域都有着强大的功能。选择其中一个可能取决于你的特定需求、团队的技术偏好以及对特定功能的偏好。 ELK Stack 的插件生态系统和 Beats 生态系统提供了更多的可扩展性，而 Graylog 的用户界面和企业支持使其在易用性和企业级支持方面更具优势。

import grpc
from concurrent import futures
import logging
from pythonjsonlogger import jsonlogger
from graypy import GELFHandler
import uuid

# 配置日志
logHandler = logging.StreamHandler()
formatter = jsonlogger.JsonFormatter('%(asctime)s %(levelname)s %(name)s %(message)s %(trace_id)s')
logHandler.setFormatter(formatter)

# 添加Graylog处理器
graylog_handler = GELFHandler(host='your_graylog_server', port=12201)
graylog_handler.setFormatter(formatter)

logger = logging.getLogger()
logger.addHandler(logHandler)
logger.addHandler(graylog_handler)
logger.setLevel(logging.INFO)

# gRPC服务定义
class Greeter(helloworld_pb2_grpc.GreeterServicer):
    def SayHello(self, request, context):
        trace_id = context.invocation_metadata().get('trace_id', str(uuid.uuid4()))
        logger.info(f"Received request from {request.name}", extra={'trace_id': trace_id})
        return helloworld_pb2.HelloReply(message=f'Hello, {request.name}!')

def serve():
    server = grpc.server(futures.ThreadPoolExecutor(max_workers=10))
    helloworld_pb2_grpc.add_GreeterServicer_to_server(Greeter(), server)
