    1. 背景与目标¶

    • 场景：同一 SSO 能力覆盖 Web (SPA/SSR)、移动/桌面客户端、CLI、Server-to-Server (AK/SK)。
    • 目标：统一认证、最小侵入、可观测、可灰度、安全默认开启。

    2. 技术原理与流程设计¶

    2.1 认证技术原理¶

    • OIDC 授权码 + PKCE：交互式登录首选，防拦截/重放。
    • Device Authorization Flow：无浏览器/CLI 体验友好。
    • AK/SK HMAC 签名：机器到机器、批处理、CI/CD。
    • Token 策略：短效 Access Token + 旋转/一次性 Refresh Token；ID Token 只用于身份声明；必要时 DPoP/Token 绑定。

    2.2 Token的设计策略¶

    2.2.1 一次认证，多处通用（SSO）¶

    在微服务架构中，由统一的IAM（身份与访问管理）服务负责认证和授权，而业务服务A、B、C仅依赖IAM颁发的凭证来保障自身安全，这是一种典型的集中式认证模式。

    IAM服务承担OAuth2.0中的授权服务器角色，负责：

    • 用户登录认证（用户名密码、短信验证码、社交账号等）
    • 颁发访问令牌（Access Token）和刷新令牌（Refresh Token）
    • 可选：提供令牌校验接口（Introspection Endpoint）或公钥分发

    服务A、B、C作为资源服务器，职责：

    • 验证请求中携带的Access Token的有效性
    • 从Token中解析用户身份和权限信息
    • 根据业务逻辑执行授权（如检查用户角色是否允许访问某接口）

    通信协议：推荐使用JWT格式的Access Token，配合签名（RS256/HS256）或加密，使得资源服务器可以本地验证令牌，避免每次请求都回调IAM，降低延迟和耦合。

    1 为什么要通用？¶
    • 用户体验：用户登录一次IAM后，访问A、B、C无需重复认证。
    • 开发效率：各服务采用相同的令牌验证逻辑，复用IAM提供的公钥或校验库。
    • 维护简单：令牌撤销、刷新等策略集中管理。
    2 Token是否要完全一样？¶

    可以一样，但必须包含足够的约束信息，防止Token被滥用。推荐在JWT的payload中携带以下声明：

    声明名 含义 示例
    sub 用户唯一标识 "user_123"
    aud 受众，即允许访问的服务标识列表 ["A", "B"]
    exp 过期时间戳 1712345678
    iat 签发时间 1712342078
    scope 或 authorities 用户拥有的权限范围 "read:profile write:order"

    关键点：aud（受众）字段可以限定Token只能在哪些服务上使用。例如，IAM可以给某个客户端颁发的Token只允许访问A和B，不允许访问C。服务A在验证Token时，必须检查aud是否包含自身服务名（或ID），若不包含则拒绝请求。这样既实现了通用，又做到了权限隔离。

    3 为什么不建议每个服务颁发独立Token？¶
    • 增加复杂性：用户需要为每个服务单独获取Token，客户端需管理多个Token。
    • 违背SSO初衷：用户需多次认证或使用复杂的机制（如Token交换）。
    • 性能开销：每次服务调用可能都需要与IAM交互换取新Token。

    结论：采用相同Token，但通过aud字段限制访问范围是最佳实践。

    2.2.2 多端登录会话独立¶

    IAM需要能够区分不同客户端会话，并为每个会话颁发独立的Token对（Access Token + Refresh Token）。当用户在一个设备上登出时，仅撤销该设备的Token，不影响其他设备。

    1 关键设计¶
    • 客户端标识：登录请求中应携带客户端唯一标识（如设备ID），IAM记录该标识与Refresh Token的绑定关系。
    • 会话管理：IAM维护一个会话表，包含用户ID、客户端ID、Refresh Token（哈希存储）、登录时间、过期时间等。
    • Token颁发：每个会话独立颁发Access Token和Refresh Token，不同会话的Access Token互不干扰（即使内容相似，但jti（JWT ID）不同）。
    • 并发限制：可按需配置最大并发会话数，超出时踢掉最早或最旧的会话。
    2 多端登录下的Token通用性¶

    尽管不同设备的Token不同，但这些Token依然可以同时访问A、B、C（只要aud允许）。也就是说，设备1的Token和设备2的Token在资源服务器眼中都是有效的，资源服务器无需关心Token来自哪个设备，只需验证签名和aud即可。

    3 交互流程¶
    +---------+          +---------+          +-------+        +-------+
| 客户端   |          |   IAM   |          | 服务A  |        | 服务B |
+---------+          +---------+          +-------+        +-------+
     |                    |                    |               |
     |--- 登录 ----------> |                    |               |
     | (账号密码+设备ID)    |                    |               |
     |                    | -- 验证凭证 ------->|               |
     |                    | <-- 成功 ---------- |               |
     |                    |                    |               |
     |<-- 返回Token对 ---- |                    |               |
     | (AT1, RT1)         |                    |               |
     |                    |                    |               |
     |--- 请求服务A + AT1 ->|                    |              |
     |                    |                    |-- 验证AT1     |
     |                    |                    | (检查签名+aud) |
     |                    |                    |     |         |
     |<-- 业务响应 ------- |                    |               |
     |                    |                    |               |
     |--- 请求服务B + AT1 --------------------->|               |
     |                    |                    |               |-- 验证AT1
     |                    |                    |               | (检查签名+aud)
     |<-- 业务响应 ------- |                    |               |
    • 登录阶段：客户端携带设备ID调用IAM登录接口，IAM认证成功后，生成包含aud的Access Token和对应的Refresh Token，记录会话。
    • 业务请求：客户端将Access Token放在请求头（如Authorization: Bearer <AT>）中访问服务A/B。服务使用IAM的公钥本地验证JWT，检查签名、过期时间、aud是否包含本服务。
    • 令牌刷新：Access Token过期后，客户端使用Refresh Token向IAM换取新的Access Token。IAM需验证Refresh Token的有效性，并可选择延长会话或要求重新认证。
    4 实现时如何确保两者不冲突？¶
    • Token 生成：IAM 在生成 Access Token 时，根据当前会话的上下文（用户 ID、设备 ID、设备类型、用户角色等）计算 aud 列表，然后将 aud 和其他声明（如 sub、exp、jti）一起签名。
    • Token 验证：各资源服务器仅验证签名、过期时间和 aud 是否包含自身，不关心 Token 来自哪个设备。
    • 会话管理：IAM 维护会话表，以 (用户ID, 设备ID) 为键，记录 Refresh Token 等信息。刷新 Token 或登出时，根据设备 ID 操作对应会话。
    5 如果需要防止令牌跨设备使用，怎么办？¶

    如果业务场景对安全性要求极高（如金融应用），必须确保令牌只能用于颁发它的设备，那么有以下几种方案：

    方案一：在令牌中绑定设备标识，并由资源服务器验证¶
    • 在 JWT 中添加 device_id 声明。
    • 客户端在每次请求时，必须在 HTTP 头中携带设备 ID（例如 X-Device-ID）。
    • 资源服务器验证令牌中的 device_id 与请求头中的设备 ID 是否一致。

    缺点：

    • 要求客户端诚实上报设备 ID（可能被伪造）。
    • 增加客户端实现复杂度。
    • 如果设备 ID 是固定的（如 UUID），攻击者可以同时窃取令牌和设备 ID。
    方案二：采用 Proof of Possession (PoP) 令牌¶
    • IAM 颁发令牌时，将令牌绑定到客户端持有的某个密钥（如非对称密钥对的公钥）。
    • 客户端在请求时必须用私钥对请求的一部分进行签名（例如使用 OAuth 2.0 DPoP 或 Mutual TLS）。
    • 资源服务器验证签名，确保请求者拥有与令牌绑定的私钥。

    优点：即使令牌被窃取，没有私钥也无法使用，实现了真正的设备绑定。 缺点：实现复杂，需要客户端支持密钥管理。

    方案三：IP 或网络环境绑定（谨慎使用）¶
    • IAM 在颁发令牌时记录客户端的 IP 地址或网络特征。
    • 资源服务器可要求 IAM 在线校验（令牌 introspection），并检查 IP 是否匹配。
    • 但 IP 可能变化（如移动网络切换），容易导致误伤。

    2.2.3 增强安全性的建议¶

    • 短命Access Token：建议有效期15分钟~2小时，降低泄露风险。
    • Refresh Token轮换：每次刷新时颁发新的Refresh Token，旧的立即失效，防止重放攻击。
    • 绑定客户端：Refresh Token与设备ID绑定，若设备ID不匹配则拒绝刷新。
    • 签名算法：生产环境使用RS256（非对称）而非HS256，便于各服务独立验证且无需共享密钥。
    • 审计日志：IAM记录所有登录、刷新、登出操作，服务A/B/C记录关键业务操作，便于追踪。

    3. Web (SPA/SSR) 生产方案¶

    3.1 流程步骤¶

    1) 前端生成 state、nonce、code_verifier/challenge 并缓存 state/nonce。 2) 重定向 IdP /authorize 携带 client_id、redirect_uri、scope、state、nonce、code_challenge。 3) IdP 检查自身 Cookie：未登录则认证；已登录直发 code。 4) 回调带 code 命中后端；后端携 code + code_verifier 调用 /token。 5) 后端校验 ID Token (签名、iss/aud/exp/nonce)，销毁原始 Token，只写业务 Session Cookie (HttpOnly, Secure, SameSite=Lax/None)。 6) 前端内存持有 Access/Refresh，用于刷新；401/419 触发刷新一次，失败重登。 7) 高敏操作可用 prompt=login 或 max_age 强制重新认证/MFA。

    3.3 Go 实现示例 (Gin + coreos/oidc)¶

    // 登录回调: 用 code + verifier 换 token, 校验 ID Token, 写 HttpOnly 会话 Cookie
func OIDCCallback(c *gin.Context, provider *oidc.Provider, cfg OAuthCfg, store SessionStore) {
    code := c.Query("code")
    if code == "" { c.AbortWithStatus(http.StatusBadRequest); return }

    verifier := oauth2.GenerateVerifier() // 实际应从用户态存取
    oauthCfg := oauth2.Config{
        ClientID:     cfg.ClientID,
        ClientSecret: cfg.ClientSecret,
        Endpoint:     provider.Endpoint(),
        RedirectURL:  cfg.RedirectURL,
        Scopes:       []string{oidc.ScopeOpenID, "profile", "email"},
    }
    tok, err := oauthCfg.Exchange(c, code, oauth2.SetAuthURLParam("code_verifier", verifier))
    if err != nil { c.AbortWithStatus(http.StatusUnauthorized); return }

    idToken, _ := tok.Extra("id_token").(string)
    v := provider.Verifier(&oidc.Config{ClientID: cfg.ClientID})
    _, err = v.Verify(c, idToken)
    if err != nil { c.AbortWithStatus(http.StatusUnauthorized); return }

    sid, _ := store.CreateSession(tok.AccessToken, tok.RefreshToken)
    http.SetCookie(c.Writer, &http.Cookie{
        Name:     "app.sid",
        Value:    sid,
        Path:     "/",
        HttpOnly: true,
        Secure:   true,
        SameSite: http.SameSiteLaxMode,
    })
    c.Redirect(http.StatusFound, "/")
}

    6. AK/SK HMAC 签名方案¶

    6.1 流程步骤¶

    1) 客户端生成 timestamp、nonce，构造 Canonical Request（方法、URI、排序的 query、签名头、hashed body）。 2) 待签名串：ALG\nTIMESTAMP\nNONCE\nHASHED_REQUEST，用 SK 做 HMAC 得签名。 3) 组装 Authorization：ALG Credential=AK, SignedHeaders=..., Signature=...，附 x-date/x-nonce。 4) 服务端查 AK 得 SK，校验时间窗口与 nonce 去重，重算签名比对；通过后映射为服务账号身份。

    6.2 时序图¶

    sequenceDiagram
    participant C as Client
    participant RS as Resource
    participant K as KeyStore
    participant R as NonceCache

    C->>C: Build canonical, HMAC(SK)
    C->>RS: Request + Authorization
    RS->>K: Get SK by AK (cacheable)
    RS->>R: Nonce check/write
    RS->>RS: Recompute signature, check timestamp window
    RS-->>C: 200 OK or 401/403

    6.3 Go 校验片段¶

    func Verify(req *http.Request, lookup func(string) (string, error)) error {
    ak, sig, signedHeaders := parseAuth(req.Header.Get("Authorization"))
    sk, err := lookup(ak)
    if err != nil { return err }
    if skewTooLarge(req.Header.Get("x-date")) { return errors.New("expired") }
    if replayed(req.Header.Get("x-nonce")) { return errors.New("replay") }

    canon := buildCanonical(req, signedHeaders)
    expect := hmacSign(canon, sk)
    if !hmac.Equal([]byte(sig), []byte(expect)) {
        return errors.New("sig mismatch")
    }
    return nil
}

    def verify(req, sk_lookup):
    ak, sig, signed_headers = parse_auth(req.headers["Authorization"])
    sk = sk_lookup(ak)
    if skew_too_large(req.headers["x-date"]):
        raise ValueError("expired")
    if replayed(req.headers["x-nonce"]):
        raise ValueError("replay")

    canon = build_canonical(req, signed_headers)
    expect = hmac_sha256(canon, sk)
    if not hmac.compare_digest(sig, expect):
        raise ValueError("mismatch")

    func ConsumeStateNonce(ctx context.Context, redis *redis.Client, state, nonce string) error {
        key := "auth:state:" + state
        val, err := redis.GetDel(ctx, key).Result()
        if err != nil {
                return fmt.Errorf("invalid state")
        }
        var st struct{ Nonce string `json:"nonce"` }
        if err := json.Unmarshal([]byte(val), &st); err != nil {
                return err
        }
        if st.Nonce != nonce {
                return fmt.Errorf("nonce mismatch")
        }
        return nil
}

    def rotate_refresh_token(session_repo, sid: str, presented_rt: str):
        sess = session_repo.get(sid)
        if not sess or sess.revoked:
                raise AuthError("invalid_session")
        if not verify_hash(presented_rt, sess.refresh_token_hash):
                session_repo.revoke(sid)
                raise AuthError("token_reuse_detected")

        new_rt = issue_refresh_token()
        session_repo.update_refresh_hash(sid, hash_token(new_rt))
        new_at = issue_access_token(sess.user_id, sess.client_id)
        return {"access_token": new_at, "refresh_token": new_rt}

    11.3 Go：AK/SK 签名串规范化（核心）¶

    func BuildCanonical(req *http.Request, signedHeaders []string, bodyHash string) string {
        sort.Strings(signedHeaders)
        var hb strings.Builder
        for _, h := range signedHeaders {
                hb.WriteString(strings.ToLower(h))
                hb.WriteByte(':')
                hb.WriteString(strings.TrimSpace(req.Header.Get(h)))
                hb.WriteByte('\n')
        }
        canonical := strings.Join([]string{
                req.Method,
                req.URL.EscapedPath(),
                req.URL.RawQuery,
                hb.String(),
                strings.Join(signedHeaders, ";"),
                bodyHash,
        }, "\n")
        return canonical
}
