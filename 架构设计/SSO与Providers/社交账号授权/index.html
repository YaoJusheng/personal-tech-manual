    1. 简介¶

    • 场景：支持用户用第三方社交账号（GitHub/Google/Apple/微信/支付宝等）登录并与本地账号绑定。
    • 目标：统一授权入口、可审计、可解绑/切换、最小化对业务改造。

    2. 技术原理¶

    • 基于 OAuth 2.0 / OIDC：第三方作为外部 IdP，返回 Authorization Code -> 交换 Access Token/ID Token。
    • 我方应用不长期保存第三方 Access Token（除非需要代用户调用 API 时按最小权限和最短过期存储）。
    • 本地账号体系需做“绑定映射”：第三方主体 ID -> 本地用户 ID；支持多社交源、可解绑、冲突处理。

    3. 流程设计¶

    3.1 登录 / 绑定流程¶

    1) 用户点击“社交登录/绑定”，浏览器重定向至第三方授权页（带 state/nonce，scope 精简）。 2) 用户在第三方完成授权后，第三方回调携带 code (+id_token for OIDC)。 3) 我方后端用 code 向第三方 /token 换取 Access Token/ID Token。 4) 校验 ID Token (签名、iss/aud/exp、nonce)；若无 ID Token，则用 Access Token 调 /userinfo 获取唯一标识。 5) 根据第三方 subject 查找绑定关系： - 已绑定 -> 直接登录（发本地会话 Cookie/Token）。 - 未绑定 -> 若用户已登录本地，则创建绑定；未登录则引导注册/绑定（需二次确认）。 6) 可选：若需代用户访问第三方 API，安全持久化第三方 Access/Refresh Token（加密存储 + 最小权限）。

    3.2 解绑流程¶

    1) 用户在账户设置发起解绑。 2) 校验用户登录态，要求二次确认。 3) 删除绑定映射；可选调用第三方 revoke 接口吊销 Token。 4) 记录审计日志。

    3.3 时序图（以 OIDC 为例）¶

    sequenceDiagram
    participant U as User
    participant B as Browser
    participant TP as Third-Party IdP
    participant APP as Our Backend

    U->>B: Click social login
    B->>TP: /authorize (state, nonce, scope)
    TP-->>B: 302 with code (+id_token)
    B->>APP: Callback with code
    APP->>TP: /token (code)
    TP-->>APP: ID Token + Access Token
    APP->>APP: Verify ID Token / fetch userinfo
    APP->>APP: Lookup or create binding
    APP-->>B: Set local session, redirect home

    4. 接入要点（各主流平台）¶

    • GitHub：OAuth2，scope 建议 read:user user:email；回调需在开发者中心登记；无 ID Token，需 /user API 取 id/login/email。
    • Google：OIDC，默认有 ID Token；校验 iss/aud/exp；邮件唯一；回调必须 https。
    • Apple：OIDC，需使用 client_secret (JWT)；邮箱可能匿名；返回 id_token。
    • 微信开放平台 (扫码 Web)：OAuth2，code 换取 access_token + openid；再调 sns/userinfo 获取 unionid；回调域名白名单。
    • 支付宝：OAuth2，code 换 token 再调 alipay.user.info.share；需签名验签；回调域名白名单。

    5. 账号绑定模型¶

    表 social_binding 示例：

    • provider (GitHub/google/apple/wechat/alipay)
    • subject (第三方唯一 ID，如 sub/openid/unionid)
    • user_id (本地用户 ID)
    • access_token (可选，加密存储)
    • refresh_token (可选，加密存储)
    • expires_at
    • created_at / updated_at

    约束：(provider, subject) 唯一；解除绑定时删除记录。

    6. Go 实现示例（Gin + goth 思路 / 原生调用）¶

    // 伪代码：GitHub OAuth 回调
func GithubCallback(c *gin.Context) {
    code := c.Query("code")
    if code == "" { c.AbortWithStatus(http.StatusBadRequest); return }

    tok, err := exchangeCode(code) // POST https://github.com/login/oauth/access_token
    if err != nil { c.AbortWithStatus(http.StatusUnauthorized); return }

    ghUser, err := fetchGHUser(tok.AccessToken) // GET https://api.github.com/user
    if err != nil { c.AbortWithStatus(http.StatusUnauthorized); return }

    uid, err := upsertBinding("github", ghUser.ID, tok)
    if err != nil { c.AbortWithStatus(http.StatusInternalServerError); return }

    sid, _ := createSession(uid)
    http.SetCookie(c.Writer, &http.Cookie{Name: "app.sid", Value: sid, Path: "/", HttpOnly: true, Secure: true, SameSite: http.SameSiteLaxMode})
    c.Redirect(http.StatusFound, "/")
}

    7. Python 实现示例（FastAPI + authlib）¶

    import os
from authlib.integrations.starlette_client import OAuth
from fastapi import APIRouter, Request, Response

oauth = OAuth()

# 注册提供商
oauth.register(
    name="google",
    server_metadata_url="https://accounts.google.com/.well-known/openid-configuration",
    client_id=os.getenv('GOOGLE_CLIENT_ID'),
    client_secret=os.getenv('GOOGLE_CLIENT_ID'),
    client_kwargs={"scope": "openid email profile"},
)
oauth.register(
    name='github',
    client_id=os.getenv('GITHUB_CLIENT_ID'),
    client_secret=os.getenv('GITHUB_CLIENT_SECRET'),
    access_token_url='https://github.com/login/oauth/access_token',
    authorize_url='https://github.com/login/oauth/authorize',
    userinfo_endpoint='https://api.github.com/user',
    client_kwargs={'scope': 'user:email'},
)

router = APIRouter()

# 登录入口路由
@router.get("/login/google")
async def login_google(request: Request):
    redirect_uri = request.url_for("google_callback")
    return await oauth.google.authorize_redirect(request, redirect_uri)

# 回调处理
@router.get("/auth/google/callback")
async def google_callback(request: Request):
    token = await oauth.google.authorize_access_token(request)
    userinfo = token.get("userinfo") or await oauth.google.parse_id_token(request, token)
    uid = upsert_binding("google", userinfo["sub"], token)
    resp = Response(status_code=302)
    resp.headers["Location"] = "/"
    resp.set_cookie("app.sid", create_session(uid), httponly=True, secure=True, samesite="lax")
    return resp

    8. 安全与合规¶

    • state/nonce 必须校验，防止 CSRF/重放。
    • 回调域名严格白名单；仅 HTTPS。
    • scope 最小化；不需要的第三方权限不要申请。
    • Token 加密存储，必要时定期刷新/吊销；解绑时可调用 revoke。
    • 审计：记录授权、绑定、解绑、失败原因；对异常高频授权告警。

    9. 产品与体验¶

    • UI：区分“登录”与“绑定”；已登录用户点击社交按钮走绑定，而非直接登录覆盖。
    • 账号合并：若社交邮箱与本地邮箱相同，可提示合并；需二次确认。
    • 失败兜底：第三方不可用时提示改用账号密码/MFA；错误码映射友好提示。

    10. 上线 Checklist¶

    • 第三方应用配置：client_id/secret、回调 URL 白名单、scope、签名证书（支付宝/Apple）。
    • 内部配置：social_binding 表迁移、密钥加密、审计日志、告警。
    • 联调：各平台沙盒/测试号，覆盖授权、拒绝授权、撤销、解绑、异常网络。
    • 安全：state/nonce 校验单测；回调仅 https；敏感日志脱敏。

    11. 实现细节补充（生产必备）¶

    11.1 数据库 DDL（建议）¶

    CREATE TABLE social_binding (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    provider VARCHAR(32) NOT NULL,
    subject VARCHAR(128) NOT NULL,
    union_id VARCHAR(128) NULL,

    user_id BIGINT NULL,

    result VARCHAR(16) NOT NULL,
    reason VARCHAR(255) NULL,
    trace_id VARCHAR(64) NULL,
    created_at DATETIME NOT NULL

    11.2 绑定状态机（建议）¶

    • pending：收到回调但未完成本地绑定确认。
    • active：绑定生效，可用于登录。
    • revoked：用户解绑或风控冻结，不可登录。

    状态迁移：

    • 首次授权 -> pending -> 用户确认/合并完成 -> active
    • 用户解绑/风控命中 -> revoked
    • 再次授权成功 -> active（可恢复）

    12. API 设计（建议）¶

    12.1 发起授权¶

    • GET /social/{provider}/login
    • 生成 state、nonce 写 Redis：social:state:{state} TTL 10m。
    • 302 到第三方 authorize。

    12.2 授权回调¶

    • GET /social/{provider}/callback?code=&state=
    • 校验 state 一次性消费。
    • 换 token、拉取 userinfo。
    • 查绑定：已绑定直接登录；未绑定则跳绑定确认页。

    12.3 绑定确认¶

    • POST /social/{provider}/bind/confirm
    • 入参：ticket（回调后短时票据）、bind_mode（create/link）。
    • 要求：用户已登录本地账号或完成注册。
    • 结果：创建/更新绑定关系。

    12.4 解绑¶

    • POST /social/{provider}/unbind
    • 校验密码或 MFA（高风险操作）。
    • 删除或置 revoked；可选调用 revoke。

    13. 第三方差异化实现要点¶

    13.1 GitHub¶

    • 无 ID Token，必须调用 /user 与 /user/emails。
    • 优先使用 id 作为 subject，不使用可变 login。

    13.2 Google¶

    • 以 sub 作为 subject；邮箱可用于展示，不作为唯一主键。
    • 必须校验 iss/aud/exp/nonce。

    13.3 Apple¶

    • client_secret 为开发者签发 JWT，需定期轮换。
    • 邮箱可能隐藏（Private Relay），不可作为唯一绑定条件。

    13.4 微信/支付宝¶

    • 微信优先用 unionid 作为跨应用主体（有则用，无则退回 openid）。
    • 支付宝需要签名验签与网关证书管理。

    14. Go/Python 关键实现¶

    14.0 第三方库¶

    1 Python 生态¶

    • Authlib：功能强大，支持OAuth 1.0/2.0、OIDC，提供客户端和服务器端实现。与Flask、Django、Starlette等集成良好。
    • python-social-auth：老牌库，支持数十个提供商，易于集成到Django、Flask。
    • django-allauth：Django专用，不仅支持社交账号，还处理本地注册、邮箱验证等。
    • 轻量级选择：若不想引入大库，可用requests-oauthlib自行封装。

    2 Go 生态¶

    14.2 Go：绑定冲突处理（事务）¶

    func BindOrLogin(tx *gorm.DB, provider, subject string, userID uint64) (uint64, error) {
        var b SocialBinding
        err := tx.Clauses(clause.Locking{Strength: "UPDATE"}).
                Where("provider=? AND subject=?", provider, subject).
                First(&b).Error
        if err == nil {
                return b.UserID, nil // 已绑定直接登录
        }
        b = SocialBinding{Provider: provider, Subject: subject, UserID: userID, Status: "active"}
        if err := tx.Create(&b).Error; err != nil {
                return 0, err
        }
        return userID, nil
}

    15. 风控与安全增强¶

    • 风险信号：IP 变化、UA 变化、地理异常、短时失败过高。
    • 策略：命中风险时仅允许“绑定”不允许“直接登录”，并触发本地二次验证。
    • 回调防护：
    • 严格匹配 redirect_uri（不允许通配符）。
    • 校验 Referer/Origin（仅作辅助，不替代 state）。
    • 失败回调不回显底层错误，统一映射业务错误码。

    16. 异常码与重试策略¶

    错误码 场景 客户端处理
    invalid_state state 缺失/过期/重放 清理会话并重走授权
    provider_token_exchange_failed code 换 token 失败 提示重试，记录 provider 响应
    provider_userinfo_failed 拉取用户信息失败 短退避重试一次
    binding_conflict 同一社交主体绑定到其他账号 引导账号申诉或合并流程
    risk_challenge_required 风控触发 要求二次验证

    17. 运维与任务¶

    • 定时任务：
    • 刷新长期 third-party token（如需要代调用 API）。
    • 清理过期 state/ticket。
    • 清理 revoked 且超过保留期的敏感凭据。
    • 指标：
    • 社交登录成功率、绑定成功率、解绑成功率。
    • 各 provider 的 token 交换延迟与失败率。
    • 风控命中率与误杀率。

    • 并发：同一 provider/subject 并发回调仅创建一条绑定。
