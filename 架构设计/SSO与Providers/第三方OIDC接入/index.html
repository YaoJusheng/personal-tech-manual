    1. 概述¶

    本网关（Gateway）基于 OpenID Connect (OIDC) Authorization Code Flow 实现了与远程 IAM 系统的单点登录（SSO）集成，允许第三方应用通过标准 OIDC 协议完成用户认证。

    1.1 架构角色¶

    角色 说明
    IAM (OpenID Provider) 远程身份认证中心，提供标准 OIDC 端点（授权、令牌、用户信息等）
    Thirdparty (Relying Party) 本服务（第三方应用），作为 OIDC 依赖方代理所有 OIDC 交互
    User Client 客户端设备

    相关术语：

    角色 缩写 核心职责
    身份提供商 IDP (Identity Provider) 管理用户认证状态（如 Auth0、Keycloak、自研 IAM），提供登入/出端点
    OpenID 提供方 OP (OpenID Provider) 颁发 ID Token;
    提供 UserInfo Endpoint 获取详细用户信息;
    支持授权码模式、隐式模式、设备码模式等 OAuth 2.0 流程
    依赖方（第三方应用） RP (Relying Party) 接入 IDP 的应用，需触发 / 响应登出，清理本地会话
    服务提供方 SP (Service Provider) 不直接验证用户密码，而是信任 IDP 的验证结果;
    接收并验证来自 IDP 的令牌;
    根据令牌中的用户信息创建本地会话;
    提供具体业务功能
    终端用户 End-User 发起登出操作的主体

    1.2 认证方式¶

    • Token Endpoint 认证方法: client_secret_post（client_id 和 client_secret 通过表单提交，非 Basic Auth）
    • ID Token 签名算法: EdDSA（Ed25519 / OKP）
    • 支持的授权类型: authorization_code

    1.3 企业级标准流程¶

    1. 用户访问第三方应用
    2. 跳转 OIDC 登录 (IAM)
    3. 登录成功 → 第三方应用后端拿到：
    4. id_token
    5. access_token
    6. refresh_token
    7. 后端做：
    8. 验证 id_token
    9. 验证通过 → 拿到用户 ID（sub）
    10. id_token 用完即丢
    11. refresh_token 存入后端数据库 / Redis
    12. 只把 access_token 返回给前端
    13. 前端后续：
    14. 只存 access_token
    15. 调用接口时带上
    16. 过期了 → 向后端刷新接口请求新的 access_token
    17. 后端刷新：
    18. 拿自己保管的 refresh_token
    19. 去 OIDC 服务商刷新
    20. 返回新的 access_token 给前端
    21. refresh_token 继续存在后端

    2. 端点列表¶

    2.1 OIDC 标准端点（无需认证）¶

    方法 路径 说明
    GET /.well-known/openid-configuration OIDC 发现文档
    GET /.well-known/jwks.json JSON Web Key Set
    GET /oauth/authorize 授权端点
    POST /oauth/token 令牌端点
    GET /oauth/userinfo 用户信息端点

    2.1.1./.well-known/openid-configuration¶

    这个接口是 OIDC 认证中心（IAM）对外暴露的 “标准化配置清单”，也叫「OIDC 发现端点」。它的核心目的是让第三方应用（RP）无需手动配置任何 OIDC 端点地址、算法、规则，只需访问这一个接口，就能自动获取对接 IAM 所需的全部关键信息。

    简单类比：

    • 没有这个接口：第三方应用要手动写死 https://iam.example.com/oauth/authorize、https://iam.example.com/oauth/token 等所有地址，还要手动配置签名算法、令牌有效期等规则；
    • 有这个接口：第三方应用只需要知道 IAM 的根域名（如 https://iam.example.com），访问 /.well-known/openid-configuration 就能自动拿到所有配置，实现 “无感对接”。

    核心作用:

    1. 标准化配置发现（最核心）

    接口返回 IAM 所有 OIDC 核心端点、算法、规则的标准化字段，第三方应用无需硬编码任何配置，彻底解决 “配置不一致、对接需反复沟通” 的问题。

    1. 降低对接成本（企业级核心价值）

    2. 对 IAM 侧：修改端点地址 / 算法时，只需更新这个接口的返回值，所有第三方应用自动适配，无需逐个通知；

    3. 对第三方应用侧：对接任意符合 OIDC 标准的 IAM 系统时，只需传入 IAM 根域名，自动完成配置加载，无需针对不同 IAM 写不同的对接代码。

    4. 合规性与生态兼容（企业级必备）

    所有主流 OIDC 客户端库（如 oidc-client-ts、Spring Security OAuth2、Keycloak 客户端）都默认依赖这个接口实现自动化对接。如果没有这个接口，第三方应用无法使用标准化客户端库，只能手动硬编码配置，既不符合 OIDC 规范，也无法对接外部 IAM（如微信企业互联、Azure AD、Okta 等）。

    使用场景：

    1. 场景 1：第三方应用初始化 OIDC 客户端（最常用）

    第三方应用启动时，首先访问该接口，自动加载 IAM 的所有配置，示例流程：

    1. 第三方应用配置文件中仅配置 IAM 根域名：issuer: https://iam.example.com；
    2. 应用启动时，自动请求 https://iam.example.com/.well-known/openid-configuration；
    3. 解析返回结果，自动填充授权端点、令牌端点、公钥地址等所有配置；
    4. 后续 OIDC 流程（跳转登录、换 token、验签）均使用自动获取的配置。

    核心价值：如果 IAM 后期调整了端点路径（如从 /oauth/authorize 改为 /iam/oauth/authorize），第三方应用无需修改任何代码，重启后自动获取新配置。

    1. 场景 2：ID Token 验签时的算法 / 公钥匹配

    第三方应用验证 ID Token 签名时，需要知道 IAM 用了什么算法、去哪里拿公钥：

    1. 解析 ID Token 的 Header，拿到签名算法（如 alg: RS256）和密钥 ID（kid）；
    2. 从 openid-configuration 返回的 jwks_uri 拿到公钥列表；
    3. 根据 kid 匹配对应的公钥，用 id_token_signing_alg_values_supported 确认算法合法；
    4. 完成验签。

    核心价值：IAM 轮换密钥 / 算法时，第三方应用自动适配，无需手动更新公钥 / 算法配置。

    1. 场景 3：多环境适配（开发 / 测试 / 生产）

    企业通常有多个 IAM 环境：

    • 开发环境：https://iam-dev.example.com
    • 测试环境：https://iam-test.example.com
    • 生产环境：https://iam.example.com

    第三方应用只需切换 IAM 根域名，访问对应环境的 /.well-known/openid-configuration，即可自动适配不同环境的配置，无需为每个环境写死一套端点地址。

    1. 场景 4：对接外部 OIDC 身份提供商

    如果企业需要对接外部 IAM（如用微信企业互联作为 OIDC 身份源），微信企业互联会暴露标准的 /.well-known/openid-configuration 接口，企业内部应用只需访问该接口，即可自动完成对接，无需手动配置微信的授权 / 令牌端点。

    2.1.2./.well-known/jwks.json¶

    /.well-known/jwks.json 是 OIDC 认证中心（IAM）对外暴露的JSON Web Key Set（JWKS）接口，核心作用是公开提供用于验证 ID Token/Access Token 签名的公钥集合，是第三方应用验证令牌合法性的 “唯一可信来源”。

    简单类比：

    • IAM 用私钥给 ID Token 签名（相当于 “盖章”）；
    • 第三方应用用 IAM 公开的公钥验证这个 “章” 的真伪；
    • 这个接口就是 IAM 把 “验章工具（公钥）” 公之于众的唯一渠道。

    企业级使用场景：

    1. 场景 1：第三方应用验证 ID Token 签名（最常用）

    这是该接口最核心的使用场景，完整流程：

    1. 第三方应用从 IAM 获取 ID Token（如 eyJhbGciOiJSUzI1NiIsImtpZCI6IjE2Yz...）；
    2. 解析 ID Token 的 Header 部分，提取两个关键信息：
      • alg：签名算法（如 RS256）；
      • kid：密钥 ID（如 oidc-rsa-key-2026，标识用哪把密钥签名）；
    3. 第三方应用请求 https://iam.example.com/.well-known/jwks.json，获取公钥列表；
    4. 根据 kid 从公钥列表中匹配对应的公钥；
    5. 用该公钥验证 ID Token 的签名，确认令牌合法。

    企业级价值：即使 IAM 轮换密钥，第三方应用只需重新请求该接口，自动匹配新 kid 对应的公钥，无需修改任何代码。

    1. 场景 2：多第三方应用统一验签（规模化对接）

    企业内有多个业务系统（如电商、CRM、OA）对接同一个 IAM 时：

    • 所有第三方应用都从同一个 jwks.json 接口获取公钥；
    • IAM 只需维护一套公钥，所有应用自动同步；

    • 避免 “每个应用手动配置公钥” 带来的不一致和维护成本。

    • 场景 3：跨环境验签（开发 / 测试 / 生产）

    企业通常有多个 IAM 环境：

    • 开发环境：https://iam-dev.example.com/.well-known/jwks.json；
    • 测试环境：https://iam-test.example.com/.well-known/jwks.json；
    • 生产环境：https://iam.example.com/.well-known/jwks.json。

    第三方应用只需切换 IAM 域名，即可自动获取对应环境的公钥，无需为每个环境手动配置不同的公钥文件，大幅降低多环境对接成本。

    1. 场景 4：对接外部 OIDC 身份提供商

      如果企业需要对接外部 IAM（如 Okta、Azure AD、微信企业互联），这些外部 IAM 都会暴露标准的 jwks.json 接口：

      • 企业内部应用无需手动索要 / 配置公钥；
      • 只需访问该接口，即可自动获取公钥验证外部 IAM 签发的令牌；
      • 符合 OIDC 生态规范，无需自定义验签逻辑。

    2.1.3./oauth/authorize（授权端点）¶

    接口定位：OIDC 登录流程的唯一入口

    • 请求方式：GET（必须，OIDC 标准）
    • 调用方：第三方应用引导用户浏览器跳转（前端）
    • 核心作用：
    • 引导未登录用户到 IAM 统一登录页；
    • 传递第三方应用的身份（client_id）、回调地址（redirect_uri）等关键参数；
    • 用户登录成功后，返回一次性授权码（code） 给第三方应用。

    2.1.4./oauth/token（令牌端点）¶

    接口定位：OIDC 流程的核心枢纽

    • 请求方式：POST（必须，OIDC 标准）
    • 调用方：第三方应用后端（绝对禁止前端调用）
    • 请求格式：application/x-www-form-urlencoded（表单）
    • 核心作用：
    • 用 /oauth/authorize 返回的 code 换取真正的令牌；
    • 验证第三方应用的合法性（client_id + client_secret）；
    • 返回 id_token（身份凭证）、access_token（接口凭证）、refresh_token（续期凭证）。

    2.1.5./oauth/userinfo（用户信息端点）¶

    接口定位：补充用户详细身份信息

    • 请求方式：GET/POST（推荐 GET）
    • 调用方：第三方应用前端（常用）或后端
    • 核心作用：
    • 基于 access_token 获取用户的详细信息（id_token 仅包含基础信息）；
    • 补充 id_token 中未包含的字段（如用户姓名、手机号、部门、角色等）；
    • 验证 access_token 的有效性（未过期、未吊销）。

    2.2 客户端管理端点（需要认证）¶

    3.2 流程说明¶

    1. 发起登录 - 第三方应用将用户重定向到 GET /api/v1/oidc/login
    2. 生成安全参数 - Gateway 生成加密随机 state（CSRF 防护）和 nonce（ID Token 重放防护），存储到内存 StateManager
    3. 重定向到 IAM - Gateway 返回 302 重定向到 IAM 的 /oauth/authorize，携带 response_type=code、client_id、redirect_uri、scope、state、nonce
    4. 用户认证 - 用户在 IAM 登录页完成身份验证
    5. IAM 回调 - IAM 将授权码通过 302 重定向回 Gateway 的 /api/v1/oidc/callback
    6. 验证 State - Gateway 验证回调中的 state 参数与存储值一致（防 CSRF）
    7. 授权码换令牌 - Gateway 使用 client_secret_post 方式向 IAM /oauth/token 提交授权码，获取 access_token、id_token、refresh_token
    8. 获取用户信息 - Gateway 使用 access_token 调用 IAM /oauth/userinfo 获取用户详情
    9. 返回结果 - Gateway 将令牌和用户信息返回给第三方应用

    3.3 单客户端示例流程¶

    要实现基于 OIDC 协议的单点登录，需要让第三方应用（RP，Relying Party）作为客户端接入 IAM（OP，OpenID Provider）。第三方应用包含前端 A（浏览器端）和后端 A（服务器端），典型的部署是后端 A 作为 OAuth 2.0 的机密客户端，能够安全保存 client_secret。推荐使用 Authorization Code Flow 进行认证，该流程安全性高，适合有后端服务的场景。

    3.3.1.应用注册与配置¶

    在开始集成之前，第三方应用需要在 IAM 系统中完成注册，并获取以下信息：

    • client_id：应用的唯一标识符。
    • client_secret：应用的密钥，用于后端与 IAM 的通信，必须保存在后端 A，不能暴露给前端。
    • redirect_uri：授权码回调地址，必须是后端 A 提供的 HTTPS 端点，例如 https://backend.example.com/oidc/callback。
    • scope：至少包含 openid，还可包含 profile、email 等，用于获取用户信息。
    • IAM 的配置信息：通过 IAM 的 OIDC 发现端点（/.well-known/openid-configuration）获取授权端点、令牌端点、用户信息端点等 URL。

    3.3.3.步骤详解¶

    步骤 1：用户访问前端 A 用户在浏览器中打开第三方应用的首页（例如 https://app.example.com）。前端 A 加载页面，并向后端 B 发起请求以获取用户状态（例如调用 /api/user/profile）。

    步骤 2：后端 B 检查用户会话 后端 B 检测到当前请求未携带有效的会话凭证（如 session cookie 或 JWT），判定用户未登录，返回 401 或重定向指令。前端 A 根据业务逻辑决定触发登录流程。

    步骤 3：前端 A 发起登录请求 前端 A 向后端 B 的登录接口（例如 /auth/login）发起请求，告知需要登录。

    步骤 4：后端 B 生成并存储 state 和 nonce 后端 B 接收到登录请求后，生成一个随机的 state 参数（用于防止 CSRF 攻击）和一个 nonce 参数（用于防止 replay attack）。将 state 和 nonce 存储在用户会话（如服务器 session 或加密 cookie）中。然后构建 OIDC 认证请求 URL，重定向到 IAM 的授权端点。构造的 URL 包含以下参数：

    • client_id：应用的 client_id。

    • response_type：固定为 code。

    • scope：至少包含 openid，可添加 profile 等。

    • 假设scope=openid%20profile%20email：请求了三个标准 scope。

    • 用户授权后，IAM 返回的 id_token 将包含 sub（来自 openid）、以及来自 profile 和 email 的 Claims。如：

      {
  "iss": "https://iam.example.com",
  "sub": "248289761001",
  "aud": "service-a",
  "exp": 1516239022,
  "iat": 1516239022,
  "nonce": "xyz456",
  "name": "张三",
  "given_name": "三",
  "family_name": "张",
  "preferred_username": "zhangsan",
  "picture": "https://iam.example.com/avatar/248289761001.jpg",
  "email": "zhangsan@example.com",
  "email_verified": true,
  "updated_at": 1614556800
}
      • sub：用户的唯一标识，由 IAM 保证在不同客户端间相同（隐私考虑，可能不同客户端返回不同的 sub，但同一客户端始终一致）。
      • name、given_name、family_name、preferred_username、picture：来自 profile scope。
      • email、email_verified：来自 email scope。

    • redirect_uri：后端 B 的回调地址（如 https://backend.example.com/oidc/callback）。

    • state：刚刚生成的 state。

    • nonce：生成的 nonce（可选，但在 OIDC 中强烈推荐）。

    • prompt：可选，根据需要决定是否强制重新认证。

    后端 B 通过 HTTP 302 重定向响应，让浏览器跳转到该 URL。

    步骤 5：浏览器重定向到 IAM 授权端点 浏览器按照重定向地址访问 IAM。IAM 检查用户是否已登录（通过 IAM 自身的会话 cookie）。如果未登录，IAM 展示登录页面。

    步骤 6：用户认证 用户在 IAM 的登录页面输入凭证（用户名、密码，可能包含 MFA）。IAM 验证身份。

    步骤 7：用户授权 认证成功后，IAM 检查用户是否已授权该应用（通常首次会询问是否允许应用获取信息）。若已授权或配置为自动同意，则进入下一步。IAM 生成一个授权码（code），该码有效期短（通常几分钟），且只能使用一次。

    步骤 8：IAM 重定向回后端 B 的回调地址 IAM 将浏览器重定向到 redirect_uri，并在 URL 中添加参数 code（授权码）和 state（原样返回步骤 4 的 state）。例如：

    https://backend.example.com/oidc/callback?code=AUTH_CODE&state=xyz123

    步骤 9：后端 B 处理回调，验证 state，并用 code 交换 token 后端 B 接收到回调请求后：

    • 提取 state，与步骤 4 存储在会话中的 state 进行比较。如果不匹配，则终止流程，返回错误（CSRF 攻击可能）。
    • 从会话中取出 nonce（后续验证 id_token 使用）。
    • 使用 client_id 和 client_secret，向 IAM 的令牌端点（token endpoint）发送 POST 请求，换取 token。请求参数如下（按 OAuth 2.0 标准）：
    POST /token HTTP/1.1
Host: iam.example.com
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=AUTH_CODE
&redirect_uri=https://backend.example.com/oidc/callback
&client_id=YOUR_CLIENT_ID
&client_secret=YOUR_CLIENT_SECRET

    IAM 验证 code、client_id、client_secret 和 redirect_uri 是否匹配。成功后，返回 JSON 响应，包含：

    {
  "access_token": "ACCESS_TOKEN",
  "token_type": "Bearer",
  "expires_in": 3600,
  "refresh_token": "REFRESH_TOKEN",
  "id_token": "ID_TOKEN"
}

    步骤 10：后端 B 验证 id_token 后端 B 收到令牌后，必须验证 id_token 的有效性。验证内容：

    • 签名是否正确（使用 IAM 的公钥，可从发现端点获取 jwks_uri）。
    • iss（issuer）是否与预期的 IAM 地址一致。
    • aud（audience）是否包含本应用的 client_id。
    • exp 是否未过期。
    • nonce 是否与会话中存储的 nonce 一致（防止重放攻击）。
    • azp（authorized party）可选，若存在应检查。

    验证通过后，可以从 id_token 中提取用户标识（如 sub）以及其他声明的用户信息（如姓名、邮箱）。

    步骤 11：后端 B 创建本地会话 后端 B 根据 id_token 中的用户标识（sub）创建或关联本地用户记录（可根据业务需求决定是否自动注册）。然后为浏览器创建一个会话（如设置 httpOnly Session Cookie 或签发自己的 JWT），并将会话与用户绑定。此时，用户在后端 B 的视角已登录。

    步骤 12：后端 B 重定向回前端 A 后端 B 完成会话创建后，将浏览器重定向回前端 A 的某个页面（例如应用首页或用户原访问页面），可在重定向 URL 中附加一些状态信息（如登录成功标志）。浏览器会携带后端 B 设置的会话 cookie。

    步骤 13：前端 A 再次请求资源 用户浏览器现在带着后端 B 的会话 cookie 访问前端 A，前端 A 再次向后端 B 发起 API 请求（例如 /api/user/profile）。由于会话 cookie 自动附带，后端 B 可以识别用户身份，返回对应用户数据。

    步骤 14：后续请求 此后，所有 API 请求都通过会话 cookie 进行身份认证。若需要调用 IAM 的资源（如获取更多用户信息），后端 B 可以使用之前保存的 access_token 调用 IAM 的 UserInfo 端点：

    GET /userinfo HTTP/1.1
Host: iam.example.com
Authorization: Bearer ACCESS_TOKEN

    UserInfo 端点返回用户的标准声明（如 sub、name、email 等）。

    IAM-->>Backend: 10. 返回id_token, access_token, refresh_token
    Backend->>Backend: 11. 验证id_token（签名：从配置的jwks_uri获取公钥，验证iss、aud、exp、nonce）
    Backend->>Backend: 12. 创建本地会话，设置Cookie
    Backend-->>Browser: 13. 302重定向到前端首页
    Browser->>Backend: 14. 请求API（携带Cookie）
    Backend->>Backend: 15. 验证会话，返回资源
    Backend-->>Browser: 16. 响应数据
    a. GET /auth/login —— 发起认证请求¶
    • 生成随机 state 和 nonce，存储在服务器端会话（如 req.session）中。
    • 使用 OIDC 库构造授权 URL，参数包括：
    • client_id
    • redirect_uri
    • response_type=code
    • scope=openid profile email（根据需求）
    • state（上一步生成的值）
    • nonce（上一步生成的值）
    • 返回 302 重定向到该 URL。
    b. GET /auth/callback —— 处理 IAM 回调¶
    • 获取请求参数中的 code 和 state。
    • 验证 state：与 req.session 中存储的 state 比对，不一致则返回错误（可能 CSRF 攻击）。
    • 使用 code、client_id、client_secret、redirect_uri 向 IAM 令牌端点发起 POST 请求，换取令牌。
    • 从响应中获取 id_token，验证其签名、iss、aud、exp、nonce（与会话中的 nonce 比对）。
    • 验证通过后，从 id_token 中提取用户唯一标识 sub 及其他需要的 claims。
    • 创建本地用户会话：
    • 在服务器端创建会话（如生成一个随机 session ID，存入 Redis），关联用户 sub。
    • 向前端设置 HttpOnly、Secure、SameSite=Lax 的 Cookie（如 session_id=...）。
    • 可选：使用 access_token 调用 UserInfo 端点获取更多用户信息，补充到本地会话中。
    • 最后将浏览器重定向到前端首页（如 https://a.example.com），或原请求页面（可从 state 中解码）。
    c. GET /api/user/profile —— 示例受保护 API¶
    • 从请求中读取会话 Cookie，解析出 session ID，从存储中获取用户信息。
    • 若会话无效或过期，返回 401 Unauthorized。
    • 若有效，返回用户基本信息（如 ID、姓名、邮箱）。
    d. POST /auth/logout —— 登出¶
    • 清除本地会话（删除 Redis 中的 session 数据，并清除 Cookie）。
    • 构造 IAM 的登出 URL（从发现端点获取 end_session_endpoint），参数包括：
    • id_token_hint：之前登录时获得的 id_token（如果本地存储了的话）。
    • post_logout_redirect_uri：登出后重定向回本服务的 URL（如 https://a.example.com/logout-success）。
    • 返回 302 重定向到该 URL，让浏览器跳转到 IAM 注销全局会话。

    3.3.5.注意事项¶

    • state 参数：必须使用，并验证其一致性，防止 CSRF 攻击。state 应在后端生成，并与用户会话绑定。
    • nonce 参数：在认证请求中加入，并在 id_token 中验证，确保令牌是响应本次请求而非重放的。
    • redirect_uri 精确匹配：IAM 必须验证回调地址与注册时的一致，防止开放重定向。
    • client_secret 保护：绝对不可暴露在浏览器端，仅用于后端与 IAM 的通信。
    • 授权码一次性使用：确保 code 只能被交换一次，IAM 应做好失效处理。
    • 使用 HTTPS：所有通信必须通过 HTTPS 加密，防止中间人攻击。
    • id_token 验证：必须验证签名、发行者、受众和 nonce。
    • 会话管理：后端 A 的会话应安全设置（如 HttpOnly、Secure、SameSite 标志）。

    3.4.2.用户访问服务B（已在服务A登录，但未访问过服务B）¶

    此时，IAM 域下已有用户的全局会话（由服务A登录时建立）。

    步骤 1：用户请求服务B: 浏览器访问 https://b.example.com，服务B后端检测无本地会话，同样生成 state 和 nonce，构造授权请求重定向到 IAM。

    步骤 2：IAM 处理请求:

    • IAM 收到请求后，检查自身域名下的 Cookie，发现存在有效的用户全局会话。
    • 由于用户已认证，IAM 不再显示登录页，而是直接生成授权码 code，并立即重定向回服务B的回调地址（附带 state）。

    步骤 3：服务B后端处理回调:

    • 验证 state，用 code 交换令牌，验证 id_token（包括 nonce），建立本地会话。
    • 整个过程对用户透明，浏览器快速重定向后即完成服务B的登录，实现单点登录。

    3.4.3.后续跨服务访问¶

    • 用户访问服务C、服务D…时，重复上述过程，每个服务独立建立本地会话，但均依赖 IAM 的全局会话实现静默授权。
    • 各服务的本地会话相互独立，过期时间可不同；IAM 全局会话的过期时间决定了用户是否需要重新输入密码。

    本地会话管理：

    • 每个服务应使用安全的会话 Cookie（HttpOnly、Secure、SameSite=Lax），并设置合理的过期时间。
    • 本地会话应与 IAM 全局会话生命周期解耦，但通常建议在本地会话过期后，利用 IAM 的静默认证重新建立（即再次重定向到 IAM，由 IAM 根据全局会话直接返回 code）。

    IAM 全局会话域：

    • IAM 的 Cookie 通常设置在 IAM 的域名下（如 iam.example.com），浏览器在访问 IAM 授权端点时会自动携带该 Cookie，从而识别用户。
    • 各服务域不同，无法直接读取 IAM 的 Cookie，因此必须通过重定向到 IAM 来利用该 Cookie。

    令牌验证与公钥管理：

    • 所有服务需从 IAM 的 jwks_uri 获取并缓存公钥，用于验证 id_token 的签名。
    • 推荐使用成熟的 OIDC 客户端库（如 node-openid-client、Spring Security OAuth2 Client）简化实现，避免自行处理细节。

    3.5 标准 OIDC scope 及其对应的 Claims¶

    OIDC 规范定义了一组标准 scope，每个 scope 对应一组预定义的 Claims。当 scope 被请求且用户授权后，这些 Claims 会出现在 id_token 中（或可通过 UserInfo 端点获取）。

    Scope 包含的 Claims（来自 id_token 或 UserInfo） 说明
    openid iss、sub、aud、exp、iat、auth_time、nonce、acr、amr、azp 必须存在，标识这是一个 OIDC 请求，返回核心身份标识。
    profile name、family_name、given_name、middle_name、nickname、preferred_username、profile、picture、website、gender、birthdate、zoneinfo、locale、updated_at 请求用户的个人资料信息。
    email email、email_verified 请求用户的电子邮件地址及验证状态。
    address address（一个 JSON 对象，包含 formatted、street_address、locality、region、postal_code、country） 请求用户的邮政地址。
    phone phone_number、phone_number_verified 请求用户的电话号码及验证状态。
    offline_access 无 Claims，但请求 IAM 颁发 refresh_token，用于离线获取新令牌。 需要时包含，通常需用户额外授权。

    注意：以上 Claims 仅是 OIDC 核心规范定义，实际 IAM 实现可能提供额外的自定义 Claims，或对某些 Claims 进行调整。

    scope 与隐私考虑：

    • 最小权限原则：客户端应只请求必要的 scope，避免过度收集用户信息。
    • 用户授权：IAM 通常会根据 scope 向用户展示授权页面，让用户决定是否允许客户端访问这些信息。
    • Claims 筛选：IAM 可以根据 scope 对返回的 Claims 进行过滤，确保客户端只能获取已授权的数据。

    3.6 单点登出（Logout）¶

    3.6.1.RP-Initiated Logout 的登出流程¶

    当用户从第三方应用退出时，可能需要同时注销 IAM 的会话（全局/单端 登出），以实现单点登出。OIDC 提供了 RP-Initiated Logout 机制：

    1. 前端 A 调用后端 A 的登出接口。
    2. 后端 A 清除本地会话，并构建 IAM 的注销请求 URL（从发现端点获取 end_session_endpoint），参数通常包括：
    3. id_token_hint：之前获得的 id_token（用于识别用户会话）。
    4. post_logout_redirect_uri：登出后重定向回的应用地址。
    5. state：可选，用于 CSRF 防护。
    6. 后端 A 返回重定向，浏览器跳转到 IAM 的注销端点。
    7. IAM 注销用户全局会话 (或单端会话)，并根据 post_logout_redirect_uri 重定向回应用。
    8. 前端 A 显示登出成功页面。

    IAM->>User: 6. 重定向回RP的post_logout_redirect_uri

    3.6.2.OP-Initiated Logout 的登出流程¶

    1 全局登出¶

    当用户在 OP 直接登出时，需要通知所有已登录的 RP 清理各自的本地会话。OIDC 规范定义了两种实现方式：

    • Front-Channel Logout (前端通道登出)：依赖于用户浏览器。OP 返回一个包含多个 iframe 的页面，每个 iframe 的 src 指向一个已登录 RP 的登出端点，通过浏览器并行加载这些 iframe 来触发各 RP 的登出。
    • Back-Channel Logout (后端通道登出)：不依赖用户浏览器。OP 直接在后端向每个 RP 预先注册的 backchannel_logout_uri 发送 HTTP 请求（通常是 POST），携带一个 logout_token 来通知 RP 登出。这种方式更可靠，但 RP 需要提供一个可供 OP 直接访问的接口。
    特性 Front-Channel Logout (前端通道) Back-Channel Logout (后端通道)
    通信方式 通过用户浏览器（iframe） 服务器对服务器（Server-to-Server）
    可靠性 依赖用户浏览器保持打开状态 高，不依赖用户端
    实现复杂度 较低，只需在 RP 暴露一个 GET 端点 较高，RP 需验证 logout_token 签名
    适用场景 所有 RP 都是 Web 应用 包含原生应用、后端服务或对可靠性要求高的场景
    2 单端登出¶

    当用户在 OP 的账户管理页面（或直接在 OP 的某个设备上）选择“登出此设备”时，需要通知所有与该设备关联的 RP 销毁其本地会话。OP 必须通过某种机制将 sid 传递给各 RP。

    2.1.Front-Channel Logout（前端通道）¶

    依赖于浏览器，OP 返回一个包含多个 iframe 的页面，每个 iframe 的 src 指向一个 RP 的登出端点，并在 URL 中携带 sid。

    • 每个 RP 必须暴露一个 frontchannel_logout_uri，该 URL 接受 sid 参数。

    • 此方式依赖于用户浏览器保持打开状态，且 RP 的端点必须是 GET 请求（容易被滥用，需配合 CSRF 防护，例如只允许通过 sid 查找，且不产生副作用时返回空像素）。
    2.2.Back-Channel Logout（后端通道）¶

    不依赖浏览器，OP 直接向后端向每个 RP 的 backchannel_logout_uri 发送 HTTP POST 请求，请求体中包含一个 Logout Token，其中同样携带 sid 声明。

    graph TD
    A[前端触发登录] --> B[第三方后端获取 Token<br/>（id_token/access_token/refresh_token）]
    B --> C[后端存储 Token 到分布式缓存（Redis）]
    C --> D[前端携带 access_token 调用业务接口]
    D --> E[后端校验 access_token 有效性]
    E -->|有效| F[处理业务请求]
    E -->|失效| G[后端自动用 refresh_token 刷新 Token]
    G -->|刷新成功| H[更新 Redis 中的 Token，<br/>返回新 access_token 给前端]
    G -->|刷新失败| I[返回 401，前端引导用户重新登录]

    核心原则：

    • access_token：前端存储 + 后端缓存，用于接口访问；
    • refresh_token：仅后端存储（绝对不返回前端），用于刷新 Token；
    • 所有 Token 校验、刷新逻辑统一在后端拦截器 / 中间件中实现，业务接口无需感知。

    3.7.1.Token 存储方案（企业级）¶

    1. 存储介质选择¶

    优先使用 Redis（分布式缓存，支持过期时间、原子操作），替代本地内存 / 数据库：

    • 本地内存：分布式部署时 Token 无法共享；
    • 数据库：读写性能低，无法自动过期；
    • Redis：支持 EXPIRE 过期时间、SETNX 原子锁，适配分布式部署。
    2. 存储结构设计¶
    Redis Key 格式 Value 格式（JSON） 过期时间 说明
    oauth:token:user:{userId} {"access_token":"xxx","refresh_token":"xxx","expires_at":1719874400,"client_id":"erp-123"} 与 refresh_token 有效期一致（如 7 天） 按用户 ID 存储，关联用户唯一标识
    oauth:token:access:{accessToken} {"userId":"10001","client_id":"erp-123","expires_at":1719874400} 与 access_token 有效期一致（如 1 小时） 快速通过 access_token 定位用户

    说明：

    • expires_at：Token 过期时间戳（秒），提前 30 秒触发刷新；
    • 双 Key 设计：既支持按用户 ID 查 Token（刷新用），也支持按 access_token 查用户（校验用）。

    3.7.2.自动刷新 Token 方案¶

    1. 刷新时机¶
    • 主动刷新：后端拦截器检测到 access_token 即将过期（如剩余时间 < 30 秒），自动触发刷新；
    • 被动刷新：前端调用接口时，access_token 已失效，后端先刷新 Token 再处理业务。
    2. 核心规则（企业级）¶
    • 必校验项：
    • Token 是否存在于 Redis（防止伪造）；
    • Token 是否过期（Redis 自动过期，无需手动判断）；
    • Token 所属的 client_id 是否匹配（防止跨应用使用 Token）；
    • 可选校验项：
    • 用户状态（是否禁用 / 注销）；
    • IP 白名单（仅允许指定 IP 访问）；
    • Token 签发时间（防止重放攻击）。
    3. 关键安全规则（企业级避坑）¶
    1. refresh_token 绝对不返回前端：仅存储在后端 Redis，前端永远无法接触；
    2. Token 传输加密：所有接口必须走 HTTPS，防止 Token 被劫持；
    3. Token 过期时间合理：
    4. access_token：1 小时（短有效期，降低泄露风险）；
    5. refresh_token：7~30 天（长有效期，提升用户体验）；
    6. Token 吊销机制：用户登出 / 改密码时，删除 Redis 中的 Token 记录，立即失效；
    7. 防重放攻击：可在 access_token 中加入 jti（唯一标识），Redis 记录已使用的 jti（适用于敏感接口）。
    4.校验过程¶

    1. 获取用户标识：从请求中携带的旧访问令牌（即使过期）中解析出 sub（如果 JWT 过期但仍可解析），或者通过请求的会话 Cookie 获取 user_id。如果无法获取，则返回 401 Unauthorized。

    2. 从数据库读取刷新令牌：根据用户标识查询 user_tokens 表，获取对应的刷新令牌（加密存储，需解密）。

    3. 向 IAM 发起刷新请求：

    POST /token HTTP/1.1
Host: iam.example.com
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token
&refresh_token=<decrypted_refresh_token>
&client_id=<client_id>
&client_secret=<client_secret>

    （注意：对于公客户端，应省略 client_secret，但这里是机密客户端）

    3.7.3.总结¶

    核心环节 关键方案
    Token 存储 Redis 双 Key 设计（按用户 ID/access_token 索引），支持自动过期
    自动刷新 拦截器检测 Token 过期，后端调用 IAM /oauth/token 刷新，加分布式锁防重复
    Token 校验 全局拦截器统一校验，校验存在性 / 有效性 /client_id，失败则引导重新登录
    安全保障 refresh_token 仅后端存储、HTTPS 传输、合理过期时间、支持吊销

    4.2 JWKS 端点¶

    GET /.well-known/jwks.json

    返回用于验证 ID Token 签名的公钥集合。

    响应示例:

    {
  "keys": [
    {
      "kty": "OKP",
      "use": "sig",
      "kid": "1",
      "alg": "EdDSA",
      "crv": "Ed25519",
      "x": "..."

    完整示例：

    package main

import (
    "crypto/ed25519"
    "encoding/base64"
    "encoding/json"
    "fmt"
    "io/ioutil"
    "net/http"
    "strings"
    "time"

    "github.com/golang-jwt/jwt/v5"

// 定义 JWKS 结构（适配 Ed25519 公钥）
type JWKS struct {
    Keys []JWK `json:"keys"`
}
type JWK struct {
    Kty string `json:"kty"` // 密钥类型，OKP=Octet Key Pair（Ed25519 属于此类）
    Use string `json:"use"` // 用途，sig=签名
    Kid string `json:"kid"` // 密钥ID
    Alg string `json:"alg"` // 算法，EdDSA
    Crv string `json:"crv"` // 曲线类型，Ed25519
    X   string `json:"x"`   // Ed25519 公钥的 x 分量（Base64 URL 编码）
}

// 从 JWKS 端点获取 Ed25519 公钥
func getEd25519PublicKey(jwksURL, kid string) (ed25519.PublicKey, error) {
    // 1. 请求 JWKS 接口
    resp, err := http.Get(jwksURL)
    if err != nil {
        return nil, fmt.Errorf("请求 JWKS 失败: %v", err)
    }
    defer resp.Body.Close()

    // 2. 解析 JWKS 响应
    body, err := ioutil.ReadAll(resp.Body)
    if err != nil {
        return nil, fmt.Errorf("读取 JWKS 响应失败: %v", err)
    }

    var jwks JWKS
    if err := json.Unmarshal(body, &jwks); err != nil {
        return nil, fmt.Errorf("解析 JWKS JSON 失败: %v", err)
    }

    // 3. 根据 kid 匹配对应的公钥
    var targetJWK JWK
    found := false
    for _, key := range jwks.Keys {
        if key.Kid == kid && key.Alg == "EdDSA" && key.Crv == "Ed25519" {
            targetJWK = key
            found = true
            break
        }
    }
    if !found {
        return nil, fmt.Errorf("未找到 kid=%s 的 Ed25519 公钥", kid)
    }

    // 4. 解码 x 分量为 Ed25519 公钥（Base64 URL 解码）
    xBytes, err := base64.RawURLEncoding.DecodeString(targetJWK.X)
    if err != nil {
        return nil, fmt.Errorf("解码公钥 x 分量失败: %v", err)
    }
    if len(xBytes) != ed25519.PublicKeySize { // Ed25519 公钥固定 32 字节
        return nil, fmt.Errorf("公钥 x 分量长度非法，需 %d 字节，实际 %d 字节", ed25519.PublicKeySize, len(xBytes))
    }

    return ed25519.PublicKey(xBytes), nil
}

// 验证 ID Token 签名及核心字段
func verifyIDToken(idToken, jwksURL, expectedIssuer, expectedAudience string) (bool, jwt.MapClaims, error) {
    // 步骤1：解析 ID Token Header，获取 kid（无需验证，仅用于匹配公钥）
    parts := strings.Split(idToken, ".")
    if len(parts) != 3 {
        return false, nil, fmt.Errorf("ID Token 格式非法，非标准 JWT")
    }
    headerBytes, err := base64.RawURLEncoding.DecodeString(parts[0])
    if err != nil {
        return false, nil, fmt.Errorf("解析 ID Token Header 失败: %v", err)
    }
    var header map[string]string
    if err := json.Unmarshal(headerBytes, &header); err != nil {
        return false, nil, fmt.Errorf("反序列化 Header 失败: %v", err)
    }
    kid, ok := header["kid"]
    if !ok {
        return false, nil, fmt.Errorf("ID Token Header 中无 kid 字段")
    }

    // 步骤2：从 JWKS 获取对应公钥
    publicKey, err := getEd25519PublicKey(jwksURL, kid)
    if err != nil {
        return false, nil, fmt.Errorf("获取公钥失败: %v", err)
    }

    // 步骤3：验证签名 + 核心字段
    token, err := jwt.Parse(idToken, func(token *jwt.Token) (interface{}, error) {
        // 校验算法是否匹配（防止算法混淆攻击）
        if token.Method.Alg() != "EdDSA" {
            return nil, fmt.Errorf("不支持的签名算法: %s，预期 EdDSA", token.Method.Alg())
        }
        return publicKey, nil
    },
        // 验证核心字段（OIDC 强制要求）
        jwt.WithIssuer(expectedIssuer),       // 验证签发方（iss）
        jwt.WithAudience(expectedAudience),   // 验证受众（aud）
        jwt.WithExpirationRequired(),         // 要求包含过期时间（exp）
        jwt.WithIssuedAtRequired(),           // 要求包含签发时间（iat）
        jwt.WithLeeway(10*time.Second),       // 允许 10 秒时间误差（解决时钟偏移）
    )

    // 步骤4：处理验证结果
    if err != nil {
        return false, nil, fmt.Errorf("ID Token 验证失败: %v", err)
    }
    if !token.Valid {
        return false, nil, fmt.Errorf("ID Token 无效（签名/字段验证失败）")
    }

    // 提取 Claims（用户信息）
    claims, ok := token.Claims.(jwt.MapClaims)
    if !ok {
        return false, nil, fmt.Errorf("解析 ID Token Claims 失败")
    }

    return true, claims, nil
}

func main() {
    // 配置参数（替换为实际值）
    const (
        idToken          = "eyJhbGciOiJFZERTQSIsImtpZCI6IjEifQ.eyJpc3MiOiJodHRwczovL2lhbS5leGFtcGxlLmNvbSIsImF1ZCI6ImVycC1zeXN0ZW0tMTIzIiwic3ViIjoiuserLTEwMDAxIiwiZXhwIjoxNzE5ODc0NDAwLCJpYXQiOjE3MTk4NzA4MDB9.xxxx" // 待验证的 ID Token

    5. SSO 便捷端点¶

    以下三个端点封装了完整的 OIDC 登录/登出流程，简化第三方应用的集成工作。

    5.1 SSO 登录¶

    GET /api/v1/oidc/login

    请求参数 (Query String):

    参数 类型 必填 说明
    redirect_uri string 否 登录成功后返回的应用地址（覆盖默认配置）
    prompt string 否 login / consent / none
    login_hint string 否 预填用户名/邮箱提示

    响应: HTTP 302 重定向到 IAM 授权页面。

    行为说明:

    1. Gateway 自动生成加密随机 state 和 nonce
    2. 将 {state, nonce, redirect_uri, created_at} 存入 StateManager（有效期由 StateTTL 配置控制，默认 300 秒）
    3. 构建 IAM 授权 URL 并返回 302 重定向

    5.2 SSO 回调¶

    GET /api/v1/oidc/callback?code=xxx&state=xxx

    IAM 授权完成后回调此端点。Gateway 自动完成授权码交换和用户信息获取。

    请求参数 (Query String):

    参数 类型 必填 说明
    code string 是 IAM 返回的授权码
    state string 是 用于 CSRF 验证的 state 值
    error string 否 错误代码（授权失败时）
    error_description string 否 错误描述

    成功响应 (200 OK):

    {
  "access_token": "eyJhbGciOiJFZERTQSIsInR5cCI6IkpXVCJ9...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "refresh_token": "dhl1kJg1oE...",
  "id_token": "eyJhbGciOiJFZERTQSIsInR5cCI6IkpXVCJ9...",
  "user": {
    "sub": "user-uuid-001",
    "name": "张三",
    "preferred_username": "zhangsan",
    "email": "zhangsan@example.com",
    "email_verified": true,
    "org_id": "org-001"
  }
}

    内部处理流程:

    1. 校验 state 参数（与 StateManager 存储值比对，消费后失效）
    2. 使用 authorization_code + client_secret_post 向 IAM 请求令牌
    3. 使用 access_token 调用 IAM /oauth/userinfo 获取用户信息
    4. 返回令牌和用户信息

    6. OIDC相关的库¶

    6.1.Go 语言库分析¶

    库 适用角色 说明
    coreos/go-oidc RP（客户端） 这是一个专门为 OIDC 客户端（RP） 设计的库。它提供了发现、ID Token 验证、UserInfo 请求等功能，通常与 golang.org/x/oauth2 配合实现完整的授权码流程。它不包含构建 OP 的能力，因此适合第三方服务集成。

    ory/fosite OP（服务端框架） 这是一个 OAuth2 和 OIDC 的服务端框架，主要用于构建授权服务器（即 OP）。它高度模块化，允许你自定义存储、加密等组件。虽然它也包含一些客户端功能，但主要定位是服务端。因此，它更适合用于实现 IAM 系统本身，而不是第三方应用集成。

    库 适用角色 说明
    Authlib OP + RP 这是一个通用的 OAuth 和 OIDC 框架，同时支持服务端和客户端。你可以用 Authlib 来构建一个完整的 OIDC 服务端（OP），也可以用它的客户端组件来接入现有的 IAM（RP）。由于其通用性，在第三方应用中作为 RP 使用是非常流行的选择。
    django-oidc-drf RP（客户端） 这是专门为 Django + DRF 项目开发的 OIDC 客户端库。它简化了 RP 的实现，提供了登录、回调、用户映射等封装。它不用于构建 OP，只用于第三方 Django 应用集成 IAM。
    fastapi-oidc / fastapi-oidc-simple RP（客户端） 这些是 FastAPI 的扩展，专为 RP 角色设计，用于快速为 FastAPI 应用添加 OIDC 认证。它们封装了登录流程和依赖注入保护，不涉及 OP 功能。
    python-keycloak RP + 管理客户端 这是一个 Keycloak 的客户端库，主要用于与 Keycloak（一种常见的 IAM 系统）交互。它可以用于 OIDC 登录（RP），也可以调用 Keycloak 的管理 API（例如创建用户、管理角色）。因此，它既可以作为第三方应用的 OIDC 客户端，也可以用于管理 IAM 本身，但不是用于构建一个新的 OP，而是与已有的 OP（Keycloak）交互。

    库 适用角色 说明

    oidcrs RP（客户端） 另一个旨在实现完整规范的 OIDC 客户端库。但从其 crates.io 页面标注“Work in Progress. Please, don‘t use it in production.”来看，目前不适合生产环境，建议保持关注但暂不采用。
    tiny-oidc-rp RP（客户端） 正如其名，这是一个“微小”的 OIDC RP 实现，仅支持授权码流程。它的定位非常明确：为小型服务或内部工具提供“通过 Google/Microsoft 登录”的能力。如果你的需求简单且明确，它可以是一个轻量级的选择。
    oidc-rp RP（客户端） 另一个较新的 RP 客户端库，强调支持 OIDC Discovery、自动刷新 JWKS 以及自动刷新 Access Token。它的设计理念是“通过统一简单的接口，将多个底层库（如 openidconnect）的功能整合起来”。可以作为 openidconnect 的一个更上层的简化替代来评估。
    axum-oidc RP（客户端） 这是一个针对 Axum Web 框架的专用 OIDC 客户端中间件。如果你的第三方服务是使用 Axum 构建的，这个库可以提供与框架深度集成的体验，例如通过中间件自动保护路由。
    riley-auth-core OP（服务端框架） 这是一个完整的、可自托管的 OIDC 身份提供商（OP） 实现。它使用 Rust 编写，将身份认证委托给上游（如 Google、GitHub），然后自己作为中心 IAM 向下游应用颁发令牌。如果你的目标是构建一个轻量、高性能的内部 IAM 系统（取代 Keycloak 等），这是一个值得研究的项目。
    rauthy OP（服务端框架） 这是另一个用 Rust 编写的、生产就绪的 OIDC 和 OAuth 2.0 身份提供商。它比 riley-auth 更成熟，支持 WebAuthn/Passkeys、MFA、DPoP 等高阶安全特性，并内置了基于 Hiqlite 的高可用部署能力。同样，这是用于构建 IAM 系统本身的，而不是第三方应用集成。
    oxide-auth OP（服务端框架） 一个符合 OAuth 2.0 标准的服务端库，可用于构建授权服务器。虽然它主要聚焦于 OAuth2，但可以作为构建 OIDC 服务端的基础。
    相关生态库 工具 jsonwebtoken 用于处理 JWT（签名、验证），casbin-rs 用于实现 RBAC/ABAC 等精细权限控制，它们通常与上述库配合使用。

    6.4.第三方服务接入 IAM¶

    在之前的讨论中，有多个独立第三方服务（A、B、C），它们需要作为 RP 接入一个统一的 IAM 系统（OP）。因此，第三方服务端需要的是 OIDC 客户端库，即上面列出的 RP 角色 的库。

    • Go 语言：推荐使用 coreos/go-oidc + golang.org/x/oauth2，它专注于 RP 需求，轻量且稳定。
    • Python 语言：推荐使用 Authlib（通用框架，可适配任何 Web 框架），或根据具体框架选择 django-oidc-drf（Django）或 fastapi-oidc（FastAPI），它们都能高效实现 RP 功能。

    6. Go：ory/fosite 提供了最大的灵活性和可定制性。

    7. 安全说明¶

    7.1.防护与安全¶

    7.1.1 防护与存储¶

    1. CSRF 防护: SSO 登录流程通过 state 参数实现 CSRF 防护，state 一次性使用，过期自动失效
    2. 重放防护: 通过 nonce 将客户端会话与 ID Token 绑定，防止 ID Token 重放
    3. Token 传输: client_secret 通过 client_secret_post 方式在表单体中传输（非 URL 参数），避免日志泄露
    4. HTTPS: 生产环境必须全链路使用 HTTPS
    5. Discovery 缓存: OIDC 发现文档缓存 5 分钟，平衡性能与配置更新及时性
    6. client_secret存储：使用环境变量或密钥管理服务（KMS），禁止硬编码。
    7. state和nonce的存储：应使用服务器端会话（如Redis），而不是客户端可读的Cookie。
    8. 会话Cookie：设置HttpOnly、Secure、SameSite=Lax，且合理设置MaxAge。
    9. 密钥轮转: 通过 JWKS 端点支持公钥轮转，无需重启服务

    7.1.2 PKCE理解¶

    PKCE（Proof Key for Code Exchange，发音 “pixie”）是 OAuth 2.0 的一个扩展，用于增强授权码流程的安全性，特别是在公共客户端（如原生移动应用、单页应用）无法安全保存 client_secret 的场景下。它通过引入动态的 code_verifier 和 code_challenge，将授权码与发起请求的客户端绑定，防止授权码被拦截后滥用。

    1. 为什么需要 PKCE？¶

    在传统的 OAuth 2.0 授权码流程（Authorization Code Flow）中，客户端使用 client_secret 在令牌端点换取访问令牌。对于机密客户端（如 Web 服务器），client_secret 可以安全存储，授权码即使被拦截，攻击者也缺少 client_secret 无法完成令牌交换。

    但对于公共客户端（如手机 App、单页应用），无法安全存储 client_secret。如果攻击者截获了授权码，就可以直接使用该授权码（无需 client_secret，因为公共客户端通常不使用）向令牌端点换取令牌，从而窃取用户授权。

    PKCE 解决了这个问题：它让客户端在发起授权请求时生成一个随机字符串 code_verifier，并发送其变换值 code_challenge。在令牌交换时，客户端再提供原始的 code_verifier，授权服务器验证两者是否匹配，确保只有最初发起请求的客户端才能使用该授权码。

    2. PKCE 核心概念¶
    • code_verifier：一个高熵的随机字符串，由客户端生成并保存。
    • code_challenge：通过某种变换（如 SHA-256 哈希）对 code_verifier 进行转换得到的结果。
    • code_challenge_method：指定变换方法，有两种：
    • plain：code_challenge 直接等于 code_verifier（不安全，已不推荐）
    • S256：code_challenge = BASE64URL-ENCODE(SHA-256(ASCII(code_verifier)))（推荐）
    3. PKCE 完整流程¶

    以支持 PKCE 的 OIDC 授权码流程为例：

    sequenceDiagram
    participant Client as 客户端（App/SPA）
    participant AuthServer as 授权服务器
    participant ResourceServer as 资源服务器

    Note over Client: 1. 生成 code_verifier
    Client->>Client: code_verifier = random(43-128 chars)
    Client->>Client: code_challenge = BASE64URL(SHA256(code_verifier))

    Client->>AuthServer: 2. 授权请求<br>GET /authorize?<br>response_type=code<br>&client_id=xxx<br>&redirect_uri=...<br>&scope=openid<br>&code_challenge={code_challenge}<br>&code_challenge_method=S256<br>&state=...
    AuthServer->>AuthServer: 保存 code_challenge 和 state

    Client->>User: 3. 用户登录并授权
    AuthServer->>Client: 4. 授权码回调<br>GET {redirect_uri}?code=xxx&state=...

    Client->>AuthServer: 5. 令牌请求<br>POST /token<br>grant_type=authorization_code<br>&code=xxx<br>&client_id=xxx<br>&redirect_uri=...<br>&code_verifier={code_verifier}

    AuthServer->>AuthServer: 6. 验证<br>计算 code_challenge<br>比较与存储的值是否一致
    AuthServer-->>Client: 7. 返回 id_token + access_token

    步骤详解:

    1. 生成凭证 客户端生成一个随机的 code_verifier（长度至少 43 个字符，最高 128 个字符），使用 S256 方法计算 code_challenge。
    // 示例（JavaScript）
function generateCodeVerifier() {
    const array = new Uint8Array(32);
    crypto.getRandomValues(array);
    return base64urlEncode(array);
}

async function generateCodeChallenge(verifier) {
    const encoder = new TextEncoder();
    const data = encoder.encode(verifier);
    const digest = await crypto.subtle.digest('SHA-256', data);
    return base64urlEncode(new Uint8Array(digest));
}

    1. 发起授权请求 客户端将 code_challenge 和 code_challenge_method 添加到授权请求的查询参数中（同时仍需携带 state 防 CSRF）。

    2. 用户授权 用户登录并同意授权，授权服务器保存收到的 code_challenge 和 code_challenge_method，与授权码关联。

    3. 返回授权码 授权服务器将授权码通过回调 URL 返回给客户端（与常规流程相同）。

    4. 交换令牌 客户端向令牌端点发送请求，除常规参数外，还需提供原始的 code_verifier。

    5. 验证 授权服务器根据之前保存的 code_challenge_method 和 code_verifier，重新计算 code_challenge，并与存储的值比对。若匹配，则颁发令牌；否则拒绝请求。

    4. PKCE 的安全性分析¶
    • 防止授权码拦截攻击：即使攻击者截获了授权码，由于不知道 code_verifier，无法在令牌交换环节通过验证。
    • 与 state 配合：state 防止 CSRF，PKCE 防止授权码滥用，两者互补。
    • 代码验证器的随机性：要求 code_verifier 具有高熵，使攻击者无法猜测或枚举。
    • 变换方法 S256：plain 方法已不推荐使用，因为它等同于将 code_verifier 明文传输，失去了保护意义。
    5. PKCE 在 OIDC 中的角色¶
    • OIDC 基于 OAuth 2.0，因此 PKCE 同样适用于 OIDC 授权码流程。
    • 在 OIDC 规范（OpenID Connect Core 1.0）中，虽然未强制要求，但推荐使用 PKCE 来增强公共客户端的安全性。
    • 在授权请求中，code_challenge 和 code_challenge_method 作为扩展参数传递，不影响 OIDC 的核心语义。
    • 最终获得的 id_token 和 access_token 与常规 OIDC 无异。

    7.2.令牌¶

    令牌类型 用途 典型有效期 敏感性
    访问令牌 用于调用资源服务器（API）的凭证，通常放在请求的 Authorization: Bearer 头中。 短（几分钟~几小时） 泄露后可在有效期内访问资源，但影响窗口有限。
    刷新令牌 用于在访问令牌过期时获取新的访问令牌，避免用户重新登录。 长（几天~几个月） 泄露后相当于长期访问权限，需要极高保护。

    两种令牌在颁发时由 IAM 绑定在一起，但客户端必须安全存储并管理它们之间的对应关系。

    7.2.1 不同客户端类型下的令牌存储管理¶

    1. 基于后端的 Web 应用（机密客户端）¶
    • 特点：服务端拥有 client_secret，可以安全保存令牌，前端（浏览器）仅通过会话 Cookie 与服务端交互。
    • 存储位置：
    • 访问令牌：存储在后端（如服务器内存、Redis、数据库），绝不发送到前端。前端发起 API 请求时，携带的是后端生成的会话 Cookie，后端再将请求转发给资源服务器，并在转发时附加访问令牌。
    • 刷新令牌：同样存储在后端（加密存储），与用户会话绑定。
    • 对应关系管理：
    • 后端在用户登录后，在服务器端会话（如 session）中记录用户标识（sub），并关联存储的刷新令牌（和访问令牌）。通常用 session_id 作为键，值包含用户 ID、刷新令牌、访问令牌（可选）等信息。
    • 刷新时，后端从当前会话中取出刷新令牌，向 IAM 发起刷新请求，获得新令牌后更新存储。
    • 优点：令牌完全由后端控制，避免 XSS 攻击；Cookie 的 HttpOnly 属性可防止 JavaScript 访问会话标识。
    2. 单页应用（SPA，公客户端）¶
    • 特点：无后端或仅有轻量后端，令牌必须存储在前端以便直接调用 API。无法安全保存 client_secret，必须使用 PKCE 增强。
    • 存储位置：
    • 访问令牌：存储在前端内存（变量）中是最安全的方式（页面刷新会丢失，可结合 silent authentication 恢复）。也可以存储在 sessionStorage（仅当前标签页）或 localStorage（持久），但 localStorage 容易受 XSS 攻击，需权衡。
    • 刷新令牌：强烈不建议存储在前端，因为刷新令牌是长期凭证。如果必须使用（如无后端纯前端应用），可以存储在内存中，或使用更安全的机制如 iframe 代理刷新（通过父窗口存储）。现代实践倾向于使用 Backend for Frontend (BFF) 模式，将刷新令牌交给 BFF 管理，前端仅持有短期访问令牌或会话 Cookie。
    • 对应关系管理：
    • 如果前端直接持有刷新令牌（不推荐），则访问令牌过期时，前端用内存中的刷新令牌通过 AJAX 向 IAM 的令牌端点发起刷新，新令牌返回后替换内存中的访问令牌。
    • 更安全的做法是：前端访问自身后端 BFF，BFF 持有刷新令牌，并负责刷新访问令牌，然后将新的访问令牌返回给前端（或 BFF 作为代理转发请求）。
    3. 原生应用（移动端/桌面端）¶
    • 特点：应用运行在用户设备上，可以安全存储少量数据，但不如后端可控。
    • 存储位置：
    • 访问令牌：存储在应用的安全存储区（如 iOS 的 Keychain、Android 的 Keystore）。
    • 刷新令牌：同样存储在安全存储区，且应使用设备提供的加密机制。
    • 对应关系管理：
    • 应用本地存储中，通常以用户标识为键，存储令牌对。刷新时直接从本地存储读取刷新令牌，发起请求后更新存储。
    • 风险：设备可能被 Root/越狱，攻击者可读取应用存储。因此建议访问令牌有效期尽量短，且支持远程撤销刷新令牌。

    7.2.2 刷新令牌与访问令牌的对应关系管理¶

    在 OAuth 2.0 中，刷新令牌和访问令牌的对应关系由 IAM 负责维护，但对客户端而言，需要在本地管理“哪个刷新令牌对应哪个用户的哪个访问令牌”。

    1. 后端应用的对应关系管理¶
    • 存储结构示例（以关系型数据库为例）：
    CREATE TABLE user_tokens (
    user_id VARCHAR(255) PRIMARY KEY,
    refresh_token TEXT NOT NULL,   -- 加密存储
    access_token TEXT,             -- 可选，如果存储当前访问令牌
    access_token_expires_at TIMESTAMP,
    updated_at TIMESTAMP
);

    • 使用：

    • 用户登录成功后，将 IAM 返回的刷新令牌和访问令牌存入该表，关联到本地用户 ID。

    • 当需要调用资源 API 时，先从数据库取出当前用户的访问令牌，如果未过期则直接使用；如果过期，则用该用户的刷新令牌向 IAM 换取新令牌，并更新数据库。

    • 与用户会话的关系：

    • 用户会话（如 session）中只需存储 user_id，后端处理请求时通过 user_id 查找对应的令牌记录。

    2. 前端/移动端的对应关系管理¶
    • 存储结构：通常以用户标识为键（如 sub），在本地安全存储中保存一个对象：{ refresh_token: "...", access_token: "...", expires_in: ... }。
    • 刷新流程：
    • 拦截 API 请求，发现 401 或预先判断访问令牌即将过期。
    • 从本地存储读取该用户的刷新令牌。
    • 向 IAM 的令牌端点发起刷新请求（携带 refresh_token、client_id，无需 client_secret，公客户端需确保 PKCE 绑定）。
    • 获得新令牌后，更新本地存储中的 access_token 和 expires_in，并重试原请求。
    • 多用户管理：如果应用支持多账户，需要按用户 ID 区分存储不同的令牌。

    7.2.3 刷新流程的细节与安全¶

    1. 刷新令牌的使用限制¶
    • 刷新令牌通常与特定的 client_id 和可选的 scope 绑定。刷新时，IAM 可能会限制新令牌的 scope 不超过原刷新令牌的 scope。
    • 刷新令牌应支持令牌轮换：每次刷新时，IAM 颁发一个新的刷新令牌，并使旧的刷新令牌失效。这可以减少刷新令牌泄露的长期风险。客户端需要存储新的刷新令牌。
    2. 并发刷新问题¶
    • 当多个请求同时发现访问令牌过期并触发刷新时，可能导致多次使用同一个刷新令牌。如果 IAM 支持令牌轮换，只有第一次刷新会成功，后续尝试会因旧令牌失效而失败。
    • 解决方案：客户端应实现刷新锁（如使用互斥锁），确保同一时刻只有一个刷新请求进行。其他请求等待刷新完成后，共享新的访问令牌。
    3. 刷新令牌的撤销¶
    • 客户端应能够主动撤销刷新令牌（例如用户登出、修改密码）。通常调用 IAM 的撤销端点（revocation_endpoint）并传入刷新令牌。
    • 后端应用在删除数据库中的刷新令牌记录前，应调用撤销端点通知 IAM 使该令牌失效。

    7.2.4 安全建议总结¶

    1. 按客户端类型选择存储位置：
    2. 后端应用：令牌保存在后端，前端仅使用会话 Cookie。
    3. SPA：访问令牌尽量保存在内存，避免 localStorage；刷新令牌交由 BFF 管理或使用 iframe 代理。
    4. 原生应用：使用系统安全存储（Keychain/Keystore）加密保存。
    5. 加密存储：无论存储在哪里，令牌（尤其是刷新令牌）应加密存储，密钥由环境变量或 KMS 管理。
    6. 最小化访问令牌生命周期：缩短访问令牌有效期（如 15 分钟），减少泄露影响。
    7. 刷新令牌轮换：支持轮换机制，每次刷新颁发新刷新令牌，使旧令牌失效。
    8. 绑定客户端上下文：对于公客户端，使用 PKCE 确保授权码和刷新令牌只能被原客户端使用。
    9. 监控与审计：记录令牌的颁发、刷新、撤销事件，发现异常行为及时响应。

    8. 错误处理¶

    所有端点在遇到错误时返回统一的 JSON 错误格式：

    {
  "code": 401,
  "message": "authorization failed: access_denied: user denied consent"
}

    常见错误码：

    HTTP 状态码 场景
    400 参数缺失或无效、state 过期/不匹配
    401 令牌无效、授权码交换失败、Bearer Token 缺失
    500 OIDC 未配置、IAM 服务不可用、内部错误
