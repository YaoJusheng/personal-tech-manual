    1. 背景与介绍¶

    1.1 什么是 SSO (Single Sign-On)¶

    单点登录（Single Sign-On，简称 SSO）是一种身份验证机制，允许用户使用一组凭据（如用户名和密码）登录后，即可访问多个相互信任的应用系统，而无需再次进行身份验证。

    1.2 解决的问题¶

    在微服务架构或企业级应用群中，通常存在多个独立的子系统（如 OA、CRM、ERP、Wiki 等）。如果没有 SSO：

    • 用户体验差：用户需要记忆多套账号密码，频繁登录。
    • 管理成本高：管理员需要在每个系统中维护用户信息。
    • 安全隐患：多套密码容易导致用户设置简单密码或复用密码，且难以统一实施安全策略（如 MFA、强制改密）。

    1.3 核心目标¶

    • 一次登录，全网通行。
    • 一次登出，全网清除（Single Logout, SLO）。
    • 统一的身份认证与授权管理。

    2. 主流 SSO 实现方案与解析¶

    方案 核心特点 适用场景 优缺点
    OIDC 1.0 基于 OAuth 2.0，轻量、RESTful、JWT 互联网应用、前后端分离、移动端、微服务架构 优点：标准化、易集成、轻量；
    缺点：无成熟联邦能力
    SAML 2.0 XML 格式、重量级、企业级 企业内网、跨组织的大型系统（如政企对接） 优点：安全、成熟、联邦能力强；
    缺点：复杂、重
    CAS (Central Authentication Service) 轻量、开源、单点登出支持好 高校、中小型企业内网系统 优点：简单易部署；
    缺点：非国际标准、集成性弱

    最佳实现方案建议:

    • 互联网 / 微服务场景：优先选择 OIDC 1.0（目前行业主流，如微信开放平台、支付宝开放平台均基于 OIDC 思想实现）；
    • 企业内网 / 跨组织场景：选择 SAML 2.0（符合企业级安全规范，如微软 Azure AD、Okta 均支持）；
    • 小型内网 / 高校场景：可选择 CAS（部署成本低，开源社区成熟）。

    2.1 CAS (Central Authentication Service)¶

    • 机制：CAS（中央认证服务） 基于 Ticket（票据）的验证机制。

    • 架构：

    • CAS Server (服务端)：独立部署的中央认证服务，负责用户的登录认证和发放票据。

    • CAS Client (客户端)：集成在各个应用中（如教务系统、考勤系统）的组件，负责保护应用资源并与 CAS Server 通信。

    • 关键票据：

    • TGT (Ticket Granting Ticket 票据授予票据)：相当于你的“会员卡”。用户登录成功后，CAS Server 会在浏览器中设置一个名为 TGC 的 Cookie，这个 Cookie 就代表了用户的全局会话。只要 TGT 有效，用户就无需再次输入密码。

    • ST (Service Ticket 服务票据)：相当于进入某个具体影厅的“电影票”。当你要访问某个具体应用时，CAS Server 会生成一个一次性的、短期的 ST，通过浏览器 URL 传递给应用。应用拿着这张 ST 去后端验证真伪，验证通过后，才允许你进入。

    • PT（Proxy Ticket）：用于代理访问（如后台服务间调用）。

    • 流程：

    • 用户访问应用 A，重定向到 CAS Server。

    • CAS Server 认证用户，生成 TGT (Ticket Granting Ticket) 存入 Cookie，并颁发 ST (Service Ticket)。
    • 重定向回应用 A，应用 A 后端向 CAS Server 验证 ST。
    • 验证通过，应用 A 创建自身 Session。
    sequenceDiagram
    participant User as 用户 (浏览器)
    participant App as 应用A (CAS Client)
    participant CAS as CAS Server

    User->>App: 1. 访问应用A
    App->>User: 2. 重定向至CAS Server登录页 (携带service参数)
    User->>CAS: 3. 携带service参数，访问CAS Server
    CAS->>User: 4. 检查无全局会话(TGT)，显示登录页
    User->>CAS: 5. 提交凭证
    CAS->>CAS: 6. 认证成功，创建TGT (通过TGC Cookie记录)
    CAS->>User: 7. 生成一次性ST，重定向回应用A (携带ST)
    User->>App: 8. 携带ST，访问应用A的回调地址
    App->>CAS: 9. 后端携带ST和service，向CAS Server验证票据
    CAS->>App: 10. 验证成功，返回用户信息 (XML格式)
    App->>App: 11. 解析用户信息，创建本地会话 (如设置会话Cookie)
    App->>User: 12. 重定向至应用首页，携带本地会话Cookie

    流程关键点：

    • service 参数：贯穿始终，用于标识是哪个应用在请求认证。
    • 票据验证：第9步的票据验证是后端到后端的通信，保证了 ST 的安全性。
    • 用户信息获取：CAS 2.0 及之后版本，应用可以通过验证接口获取到更详细的用户属性（如姓名、学号、部门）
    • 优点：成熟、开源、简单。
    • 缺点：协议私有（虽有标准但不如 OIDC 通用），主要用于 Web 场景，对移动端/API 支持较弱。

    2.2 OAuth 2.0 + OIDC (OpenID Connect) [推荐]¶

    • 机制与关联：OpenID Connect （OIDC） 是实现SSO目标的一种具体的、现代的技术协议。OAuth 2.0 是授权协议，OIDC 是建立在 OAuth 2.0 之上的 身份认证层。

    • OAuth 2.0 解决的是“授权”问题，例如“允许应用访问我的GitHub仓库”。

    • OIDC 解决的是“认证”问题，例如“证明我是谁”。它通过一个叫 ID Token 的JSON Web Token （JWT）来安全地传递用户身份信息，用于认证用户身份（证明 “你是你”）。
    • 保留了 OAuth 2.0 的 Access Token，用于授权访问资源。

    • 新增 UserInfo Endpoint，用于获取用户的详细信息。

    • 核心概念：

    • Access Token：用于访问 API 资源。

    • ID Token (JWT)：包含用户身份信息。

    • Refresh Token：用于刷新 Access Token。

    • 优点：

    • 标准化：业界标准，生态丰富。

    • 多端支持：完美支持 Web、App、CLI、IoT 等多种客户端。

    • 无状态：基于 JWT，服务端可无状态扩展。

    • 缺点：协议相对复杂，有一定的学习成本。

    • 流程：

    sequenceDiagram
    participant User as 用户 (浏览器)
    participant App as 应用 (SP)
    participant IdP as 身份提供商 (IdP/OIDC)

    User->>App: 1. 访问应用资源
    App->>User: 2. 重定向至IdP (携带client_id, redirect_uri, state)
    User->>IdP: 3. 携带重定向请求，访问IdP
    IdP->>User: 4. 用户未登录，显示登录页
    User->>IdP: 5. 提交凭证 (如用户名/密码+MFA)
    IdP->>User: 6. 认证通过，重定向回应用 (携带授权码code和state)
    User->>App: 7. 携带授权码，访问应用回调接口
    App->>IdP: 8. 后端发送授权码 + client_secret 换取令牌
    IdP->>App: 9. 返回ID Token和Access Token
    App->>App: 10. 验证ID Token (签名, iss, aud等)，<br>解析用户身份，创建本地会话 (如设置HttpOnly Cookie)
    App->>User: 11. 重定向至应用首页，携带本地会话Cookie
    User->>App: 12. 携带Cookie访问资源，应用验证会话，授权访问
    1. 访问资源：用户在浏览器中访问应用A。
    2. 发现未认证，重定向至IdP：应用A后端检查发现用户没有有效会话，将其重定向到OIDC身份提供商（IdP）的认证端点。重定向URL中携带了应用A的client_id、回调地址redirect_uri和一个用于防止跨站请求伪造（CSRF）的随机数state。
    3. 浏览器前往IdP：用户浏览器跟随重定向，访问IdP。
    4. IdP要求登录：IdP发现用户未在此登录，则显示登录页面。
    5. 用户提交凭证：用户输入凭证，可能包括密码和MFA二次验证。
    6. IdP颁发授权码：IdP验证凭证通过，生成一个极短期的授权码（Authorization Code），并将浏览器重定向回步骤2中指定的redirect_uri。重定向URL中携带了这个授权码和最初的state参数。
    7. 应用回调接收授权码：浏览器访问应用A的回调接口（/callback），携带了授权码。
    8. 后端换取令牌：应用A的后端服务，使用授权码，连同自己的client_id和client_secret，通过后端渠道（安全，不经过浏览器）向IdP的令牌端点（Token Endpoint）发起请求，换取令牌。
    9. IdP返回令牌：IdP验证授权码和客户端密钥无误后，返回ID Token（身份令牌，JWT格式）和Access Token（访问令牌）。
    10. 后端验证令牌，创建本地会话：应用A后端验证ID Token的签名、颁发者（iss）、接收方（aud）和有效期（exp）。验证通过后，从中解析出用户信息（如用户ID、邮箱）。随后，应用A销毁这些令牌，并为该用户创建自己的“本地会话”（例如，在服务器端Redis中存入会话数据，并向浏览器写入一个名为app.sid的HttpOnly Cookie）。这是“令牌与会话分离”原则的核心体现。
    11. 重定向至应用首页：应用后端完成本地会话创建后，将浏览器重定向到应用的首页。
    12. 携带会话访问资源：浏览器访问首页时，会自动携带app.sid这个Cookie。应用后端根据此Cookie找到对应的本地会话，确认用户已登录，最终返回请求的资源页面。后续的所有请求都将依赖这个本地会话，而不再涉及SSO流程。

    2.3 SAML 2.0 (Security Assertion Markup Language)¶

    • 机制：SAML 2.0 是企业级的 XML 协议，核心是「断言（Assertion）」传递，用于跨组织的身份认证。基于 XML 的安全断言交换协议。

    • 核心角色:

    • 主体 (Principal)：用户。

    • 身份提供商 (IdP，Identity Provider)：负责认证用户并颁发身份“断言”，比如公司的统一认证系统。

    • 服务提供商 (SP，Service Provider)：负责提供服务并信任 IdP 的断言，比如 Salesforce、Workiva 等应用。

    • 关键概念：元数据与断言

    • 元数据 (Metadata)：SAML 集成最关键的一步是交换元数据。IdP 和 SP 会互相交换一个 XML 文件，这个文件里包含了对方的公钥证书、支持的绑定方式、端点地址等。这相当于双方在正式通信前，先互相出示了“身份证”并交换了加密方式。

    • 断言 (Assertion)：IdP 认证用户后，生成的一个 XML 格式的“身份证明文件”。里面包含了用户的身份信息（如邮箱）、属性（如部门）和授权信息。这个文件通常是经过数字签名的，确保它来自可信的 IdP 且未被篡改。

    • 流程：

    sequenceDiagram
    participant User as 用户 (浏览器)
    participant SP as 服务提供商 (应用)
    participant IdP as 身份提供商

    User->>SP: 1. 访问应用 (SP)
    SP->>User: 2. 生成SAML请求，重定向至IdP
    User->>IdP: 3. 携带SAML请求，访问IdP
    IdP->>IdP: 4. 用户可能未登录，要求认证
    User->>IdP: 5. 提交凭证
    IdP->>IdP: 6. 生成带签名的SAML断言
    IdP->>User: 7. 将SAML断言通过浏览器POST到SP的ACS地址
    User->>SP: 8. 浏览器将断言POST到SP
    SP->>SP: 9. 验证断言签名，解析用户信息，创建本地会话
    SP->>User: 10. 登录成功，访问资源

    流程关键点：

    • ACS URL (断言消费地址)：SP 上的一个特定端点，用于接收 IdP 发来的 SAML 断言，是流程中的关键一环。
    • 数字签名：SAML 响应通常带有 IdP 的签名，SP 必须验证此签名，这是信任的基础。

    • 安全建议：为了保证安全，SAML 的实施应始终启用 TLS，并建议对请求和响应进行签名，甚至对断言本身进行加密。

    • 场景：主要用于企业级 B2B 对接（如企业内部系统对接 SaaS 服务）。

    • 优点：安全性高，支持复杂的企业级属性传递。
    • 缺点：XML 报文冗长，传输效率低，浏览器前端解析困难，不适合移动端。

    2.4 方案选型建议¶

    对于现代微服务架构、包含多端（Web/App/CLI）的系统，首选 OIDC (OAuth 2.0 + JWT) 方案。

    特性 OIDC (现代派) CAS (校园派) SAML (企业派)
    数据格式 JSON (基于JWT) 自定义 / XML XML
    通信方式 RESTful / HTTP HTTP 重定向 + 后端验证 HTTP 重定向 + 前端POST
    复杂度 低 低 高
    适用场景 现代应用、移动APP、单页应用、微服务 教育机构、企业内部老系统 大型企业、云应用 (Salesforce, Office 365)
    客户端支持 极好，几乎所有语言都有库 较好，支持主流语言 支持广泛，但实现较重
    安全模型 基于 Token (JWT) 基于票据 (Ticket) 基于签名/加密的 XML 断言

    如何选择？

    • 如果你正在开发一个新系统，尤其是包含移动端或前后端分离的应用：请优先选择 OIDC。它更现代、更灵活、更安全。
    • 如果你身处高校或需要快速为一批内部老旧的 Web 系统实现 SSO：CAS 是一个非常成熟且轻量的选择，它的“门票”模型非常直观，对接成本低。
    • 如果你的公司需要采购或对接一个外部的大型企业级 SaaS 应用（如 Workday、Salesforce）：SAML 几乎是不二之选。这是企业间建立信任关系的“标准语言”。

    3. 架构设计与落地最佳实践¶

    3.1 核心架构¶

    架构包含三个核心角色：

    1. Identity Provider (IdP): 认证中心（如 Keycloak, Auth0, 自研服务）。负责用户认证、颁发 Token。
      • IdP (Identity Provider) 是 IAM (Identity and Access Management) 系统中的一个核心角色。简单来说，IAM 是一个完整的管控体系，而 IdP 是其中负责“发身份证”的模块。
    2. Client (SP): 客户端（Web, CLI, SDK）。负责引导登录、存储 Token。
    3. Resource Server (API): 资源服务器（后端微服务）。负责校验 Token、提供数据。

    3.2 多客户端落地方案¶

    3.2.1 Web 端 (SPA/SSR)¶

    • 模式：Authorization Code + PKCE。
    • 会话分层：
    • IdP 会话：在 IdP 顶级域写全局 Cookie (SameSite=None; Secure; HttpOnly; Domain=.iam.com)，用于判断是否已登录（SSO）。
    • 业务会话：应用后端收到 code 换 Token 后，验证 ID Token，销毁前端 token，仅下发本地短期 Session Cookie（HttpOnly，短过期）；前端只在内存保存 Access/Refresh Token 以便续期，避免落盘。
    • 流程细化： 1) 浏览器生成 state、nonce、code_verifier/challenge；保存 state/nonce 以校验重放。 2) 浏览器重定向 IdP /authorize，携带 client_id、redirect_uri、scope、state、nonce、code_challenge。 3) IdP 查自身 Cookie：未登录则展示登录页并完成认证；已登录直接跳过登录。 4) IdP 颁发一次性 code，重定向回 redirect_uri。 5) 浏览器命中回调，将 code 交给后端（或前端调用后端）连同 code_verifier 向 /token 换取 Token（后台通道）。 6) 后端校验 ID Token 签名与 iss/aud/exp/nonce，销毁原始 Token，仅写业务 Session Cookie；在内存保存当前 Access/Refresh Token 以便刷新。 7) 后续请求带业务 Session Cookie；401/419 触发一次刷新，失败则重定向重新登录。
    • Token 管理：
    • Access Token 超短效（5~15 分钟）；Refresh Token 旋转且一次性使用，被盗后下次刷新失败。
    • Silent refresh：隐藏 iframe 或后台 /refresh 请求刷新，401/419 触发刷新一次，失败则重定向重新登录，避免刷新风暴。
    • 高敏操作使用 prompt=login 或 max_age 强制二次登录/MFA。
    • 安全措施：
    • CSRF：业务 Session Cookie 配 SameSite=Lax/Strict + CSRF Token；仅 HTTPS。
    • XSS：不把 Token 放 localStorage；开启 CSP；回调页仅做跳转，不渲染业务。
    • 跨子域：Cookie Domain=.example.com，若跨主域则用前端内存 Token + 后端会话隔离。
    • 错误与重试：
    • 处理 invalid_grant、invalid_client、interaction_required；授权请求绑定 state/nonce，重放直接拒绝。

    3.2.2 CLI (命令行工具)¶

    • 模式优先级：Device Authorization Flow (RFC 8628) 优先；无浏览器或需全自动脚本时再考虑本地回环 Auth Code (监听 127.0.0.1)。
    • Device Flow 落地： 1) CLI 请求设备授权端点（建议统一为 /oauth/device_authorization），拿 device_code、user_code、verification_uri、interval、expires_in。 2) CLI 打印指引：“请访问 URL 输入代码 USER_CODE”；同时启动轮询，遵守 interval 节流。 3) 轮询 /token 时处理 authorization_pending、slow_down（回退轮询间隔）、expired_token（提示重试登录）。 4) 成功后拿到 Access/Refresh Token，落盘前加密或最少 chmod 600 存于 ~/.config/<app>/credentials；有条件则优先系统钥匙串/秘钥环。 5) 支持多 Profile/租户：~/.config/<app>/profiles/<name>.json，互不覆盖。
    • Token 生命周期：
    • 开机自动刷新：调用 /token with refresh_token，失败（invalid_grant）则提示重新授权。
    • 重试上限：连续 3 次失败停止，避免无限刷屏。
    • 离线/隔离环境兜底：提供一次性 device_code+二维码打印，或回退到本机回环 Auth Code（浏览器可访问 127.0.0.1）。

    3.2.3 SDK (Mobile/Desktop)¶

    • 模式：Authorization Code + PKCE，强制使用系统浏览器（iOS ASWebAuthenticationSession + Universal Link，Android Custom Tabs + App Link，桌面系统默认浏览器）。
    • 回调与深链：
    • 定义安全回调：移动端用 HTTPS Universal/App Link；桌面可用 http://127.0.0.1:<port>/callback 或自定义 URI Scheme（需校验来源）。
    • 应用被系统回收后恢复：在回调中携带状态，重建内存态并落安全存储。
    • Token 存储：
    • 仅存于 Keychain/Keystore/DPAPI 等安全存储；不落明文文件；内存中仅保留当前 Access Token。
    • Access Token 超短效，Refresh Token 旋转；设备时间漂移需加容差校验。
    • 跨应用 SSO：
    • 借助系统浏览器共享 IdP 会话实现同设备多应用 SSO；提供“强制重新认证”开关以隔离高敏场景。
    • 无法用 WebView，否则无法复用会话且安全性低。
    • 网络切换/离线：
    • 断网时优雅降级：提示离线，待网络恢复后自动刷新；刷新失败再提示重新登录。

    3.2.4 跨端一致性与安全基线¶

    • 统一策略：Access Token 短效 + Refresh Token 旋转/一次性；高敏接口用 prompt=login 或 max_age；必要时启用 DPoP/Token 绑定 (cnf/jkt) 防盗用。
    • SLO：Back-channel 优先（失败重试+审计），Front-channel 兜底；吊销 Refresh Token，同时将 Access Token 加入短期 denylist。
    • 旧系统接入：通过反向代理/sidecar (oauth2-proxy) 统一校验 Token，再用 Header 透传用户身份，减少改造量。
    • 观测与灰度：登录/刷新/SLO 指标与错误码告警；分端灰度发布，提供客户端错误码对照与重试/回退矩阵，降低接入摩擦。

    3.2.5 AK/SK 签名登录（云平台场景）¶

    • 适用场景：服务到服务（Server-to-Server）、离线作业、CI/CD、无浏览器环境；用于鉴权而非交互式登录。可与 OIDC 并存。
    • 核心思路：客户端持有 AccessKey (AK) + SecretKey (SK)，对请求做 HMAC 签名，服务端用存档的 SK 校验。可选加入临时会话 Token 以支持短期凭证。
    • 请求签名流程： 1) 生成 timestamp (ISO8601) 与 nonce（一次性随机串）。 2) 构造 Canonical Request：HTTP 方法、URI、规范化查询串、规范化头（如 host, x-date, x-nonce）、Hashed Body。 3) 计算待签名串：ALG\nTIMESTAMP\nNONCE\nHASHED_CANONICAL_REQUEST。 4) 用 SK 做 HMAC (如 HMAC-SHA256) 得到签名；构造 Authorization 头：ALG Credential=AK, SignedHeaders=..., Signature=...。 5) 附加头部：x-date/x-timestamp、x-nonce、Authorization，可选 x-session-token（临时凭证）。
    • 服务端校验： 1) 从 Authorization 中取 AK，查库取 SK（可缓存）。 2) 校验时间漂移（如 ±5 分钟）与 nonce 去重（Redis set/TTL）。 3) 重新构造 Canonical Request 并用 SK 计算签名比对。 4) 校验通过后，将主体身份映射为内部用户/服务账号，写入请求上下文。
    • 密钥管理：
    • AK/SK 存密钥柜；支持轮换与失效时间；暴露查询接口供资源端实时获取或定时拉取缓存。
    • 支持临时 AK/SK（STS），由上游通过 OIDC 角色换取，时效短、可撤销。
    • 重放防护：
    • nonce + 窗口内去重；timestamp 限定有效期；必要时绑定来源 IP/子网。
    • 对幂等写操作可要求 x-idempotency-key，服务端做去重。
    • 错误处理：建议统一错误码风格为 snake_case（如 signature_mismatch、request_expired、access_key_not_found、invalid_signature），便于跨语言客户端一致处理。

    3.2.6 统一规范（术语 / 参数 / 端点 / 错误码）¶

    为避免多端实现口径不一致，建议按以下规范统一：

    • 术语统一：
    • 认证中心统一称 IdP；业务后端统一称 App Backend；资源服务统一称 Resource Server。
    • 交互式登录统一使用 Authorization Code + PKCE；CLI 统一使用 Device Authorization Flow。
    • 参数命名统一：
    • OIDC/OAuth 参数保持 RFC 原名：client_id、redirect_uri、scope、state、nonce、code_challenge、code_verifier。
    • Device Flow 的 grant_type 固定为 urn:ietf:params:oauth:grant-type:device_code。
    • AK/SK 请求头统一：Authorization、x-date、x-nonce、x-session-token（可选）。
    • 端点命名统一：
    • 授权端点：/oauth/authorize
    • 令牌端点：/oauth/token
    • 设备授权端点：/oauth/device_authorization
    • JWKS 端点：/.well-known/jwks.json
    • 元数据端点：/.well-known/openid-configuration
    • 错误码规范：
    • 使用小写 snake_case，示例：invalid_grant、authorization_pending、slow_down、expired_token、signature_mismatch。
    • 返回结构建议：{"error":"<code>","error_description":"<human_readable_message>","trace_id":"<id>"}。
    • 安全基线统一：
    • 所有回调地址使用 HTTPS（本机回环 127.0.0.1 例外）。
    • state/nonce 一次性消费并设置短 TTL。
    • Refresh Token 开启旋转与重用检测，命中重用立即吊销会话。

  SB->>IdP: 授权请求(state/nonce/challenge)

    sequenceDiagram
  autonumber

    3.5 TokenNum机制¶

    3.5.1 方案¶

    方案：Per-Client Session + 全局失效令牌号

    最小化改动的两层 token 验证模型：

    Layer 1 (全局失效): db.TokenNum > claims.TokenNum → 拒绝
                        触发时机：改密码、强制下线全部端
Layer 2 (单端 Session): cache.sess:{uid}:{client} != claims.SessionID → 拒绝
                        触发时机：该端正常登出、同端重新登录
    场景 效果
    Web 重新登录 仅刷新 sess:uid:web，CLI/SDK session 不受影响
    Web 登出 仅删 sess:uid:web，其他端保持
    改密码/强制下线 全局 [TokenNum++]，所有端全失效
    跨服务 SSO JWT Ed25519 签名 + [/auth/jwks]

    SSO 原理：Ed25519 签名的 JWT 本身就是 SSO token，外部服务通过 [/auth/jwks]验证签名即可信任，claims 中补充 ClientType + SessionID 不影响跨服务验证。

    改动：

    1. [jwt.go] — Claims 加 ClientType/SessionID，更新 GenerateTokenPair 签名
    2. [auth.go] — LoginRequest 加可选 client_type
    3. [auth_service.go] — Login 不再 bump TokenNum，写 session 到 cache；Logout 删单端 session；新增 LogoutAll
    4. [auth_middleware.go] — 双层验证
    5. [auth.go] — 新增 LogoutAll controller
    6. [auth.go] — 注册新路由(POST /auth/protected/logout-all)

    3.6 OIDC支持¶

    3.6.1 OIDC 设计方案¶

    该 IAM 服务器的定位本身就是身份提供者（IdP）。没有 OIDC，第三方服务只能通过 [/auth/jwks] 手动验证签名——完整的 OIDC 标准让任何支持 OAuth2/OIDC 的框架都能直接接入（Keycloak 客户端、Passport.js、go-oidc 等）。

    核心设计原则：OIDC 作为叠加层，不触碰现有业务逻辑

    现有内部 JWT ──────────────────────────────────────► 内部 API 鉴权（不变）
                                                      ▲
OIDC Layer ─► 共用同一对 Ed25519 密钥 ─────────────────┘
              ─► 独立 IDTokenClaims（OIDC 标准字段）
              ─► 独立 oidc_clients 表
              ─► authorization_code 存 cache（10min TTL）

    标准 Authorization Code Flow:

    Browser/SDK                IAM                     Third-party App
    │                       │                            │
    │── GET /oauth/         │                            │
    │   authorize?          │                            │

    接口设计：

    • GET /.well-known/openid-configuration：OIDC 标准的配置发现接口，让业务系统（RP）自动获取认证中心（IdP）的核心配置（无需手动配置端点地址、签名算法等），是 OIDC 标准化的核心体现。

    • GET /.well-known/jwks.json：提供认证中心用于签名 ID Token 的 公钥，业务系统通过该接口获取公钥，验证 ID Token 的签名合法性（核心安全接口）。

    • GET /oauth/authorize：OIDC 的授权端点，业务系统引导用户跳转到该接口完成登录认证，最终获取授权码（code），是 OIDC 登录流程的入口。
    • POST /oauth/token：业务系统后端用授权码（code）换取 ID Token/Access Token/Refresh Token 的核心接口，必须后端对后端调用（前端绝对不能调用）。
    • GET /oauth/userinfo：业务系统通过 Access Token 获取用户详细身份信息的接口（ID Token 仅包含基础信息，该接口可获取更多信息）。
    • 请求方式
      • 请求头必须携带 Access Token：Authorization: Bearer {access_token}
      • 支持 GET/POST，推荐 GET

    1.核心对比总表¶
    对比维度 仅用 JWT + /auth/jwks 手动验证（无 OIDC） 标准 OIDC 单点登录
    本质定位 自定义登录方案，JWT 只是凭证 标准化身份协议（OAuth 2.0 + 身份层）
    是否有标准 无统一标准，全靠自己定义 严格遵循 OIDC 1.0 国际标准
    核心流程 登录 → 直接返回 JWT → 手动拿公钥验签 授权码模式：code → 后端换 token → 标准验签
    端点规范 自定义：/login、/login/verify、/auth/jwks 固定标准：
    • .well-known/openid-configuration
    • /oauth/authorize
    • /oauth/token
    • /userinfo
    • /.well-known/jwks.JSON
    客户端标识 自定义（可能无 client_id /client_secret） 标准：client_id（公开）+ client_secret（保密）
    授权中间态 无授权码，JWT 可能直接暴露在 URL / 前端 有授权码 code，一次性、短时、后端兑换
    JWT 用途 既当身份凭证，又当权限凭证 id_token = 身份；access_token = 权限；refresh_token = 续期
    安全机制 全靠人工实现，容易漏防 CSRF、重放、劫持 内置：state/nonce/PKCE/ 授权码隔离 / 令牌轮换

    对接成本 每个第三方都要重新沟通、适配 一套标准对接所有接入方
    扩展性 差，无法对接外部 IdP 强，可对接微信 / 企业微信 / 飞书 / 钉钉 / Azure AD/Okta

    开发难度 低，会 JWT 即可 中高，需理解协议，但有成熟框架
    生产安全性 中等偏下（容易裸奔、前端持 JWT） 高（企业级 / 金融 / 政务标配）

    4.2 Go (Gin / Standard Lib)¶

    使用 github.com/coreos/go-oidc/v3。

    package main

import (
    "context"
    "net/http"
    "strings"

    "github.com/coreos/go-oidc/v3/oidc"
    "github.com/gin-gonic/gin"
)

func AuthMiddleware(provider *oidc.Provider, clientID string) gin.HandlerFunc {
    verifier := provider.Verifier(&oidc.Config{ClientID: clientID})

    return func(c *gin.Context) {
        authHeader := c.GetHeader("Authorization")
        if authHeader == "" {
            c.AbortWithStatus(http.StatusUnauthorized)
            return
        }
        tokenStr := strings.TrimPrefix(authHeader, "Bearer ")

        // 验证 Token
        idToken, err := verifier.Verify(context.Background(), tokenStr)
        if err != nil {
            c.AbortWithStatusJSON(http.StatusUnauthorized, gin.H{"error": "Invalid token"})
            return
        }

        // 解析 Claims
        var claims struct {
            Sub   string `json:"sub"`
            Email string `json:"email"`
        }
        if err := idToken.Claims(&claims); err != nil {
            c.AbortWithStatus(http.StatusInternalServerError)
            return
        }

        c.Set("user_id", claims.Sub)
        c.Next()
    }
}

func main() {
    ctx := context.Background()
    provider, _ := oidc.NewProvider(ctx, "https://idp.example.com")

    r := gin.Default()
    r.Use(AuthMiddleware(provider, "my-api-resource"))

    r.GET("/api/data", func(c *gin.Context) {
        userID, _ := c.Get("user_id")
        c.JSON(200, gin.H{"user_id": userID, "data": "secure data"})
    })
    r.Run()
}

    let public_key = b"-----BEGIN PUBLIC KEY...-----";
    let key = DecodingKey::from_rsa_pem(public_key).map_err(|_| StatusCode::INTERNAL_SERVER_ERROR)?;

    let mut validation = Validation::new(Algorithm::RS256);
    validation.set_audience(&["my-api-resource"]);

    match decode::<Claims>(token, &key, &validation) {
        Ok(_) => Ok(next.run(req).await),
        Err(_) => Err(StatusCode::UNAUTHORIZED),
    }
}

#[tokio::main]
async fn main() {
    let app = Router::new()
        .route("/api/data", get(|| async { "secure data" }))
        .layer(middleware::from_fn(auth_middleware));

    axum::Server::bind(&"0.0.0.0:3000".parse().unwrap())
        .serve(app.into_make_service())
        .await
        .unwrap();
}

    5. 要求与技术挑战分析¶

    5.1 安全性挑战¶

    • Token 泄露：Access Token 一旦泄露，攻击者可短期内冒充用户。
    • 对策：Access Token 设置短有效期（如 15 分钟），配合 Refresh Token 使用；Web 端尽量使用 HttpOnly Cookie 存储；关键操作二次验证。
    • CSRF 攻击：如果 Token 存放在 Cookie 中，需防范 CSRF。
    • 对策：使用 SameSite=Strict/Lax 属性；配合 CSRF Token。
    • XSS 攻击：如果 Token 存放在 LocalStorage，易被 XSS 窃取。
    • 对策：首选 HttpOnly Cookie；严格的内容安全策略 (CSP)。

    5.2 单点登出 (SLO)¶

    • 难点：用户在一个应用登出，如何通知其他所有已登录的应用也登出？
    • 方案：
    • Front-Channel SLO：IdP 通过浏览器重定向或 iframe 依次调用各应用的登出端点。依赖浏览器环境，不可靠。
    • Back-Channel SLO：IdP 通过后端 HTTP 请求直接通知各应用后端销毁 Session。更可靠，但要求应用后端可被 IdP 访问。
    • 短效 Token：不强制 SLO，依赖 Access Token 短有效期自动过期。最简单，但有延迟。

    5.3 高可用与性能¶

    • IdP 是单点故障：IdP 挂了，所有系统无法登录。
    • 对策：IdP 必须集群部署，数据库高可用。
    • Token 校验性能：每个请求都去 IdP 校验 Token 会造成巨大延迟。
    • 对策：使用 JWT (Stateless)，资源服务器本地校验签名（公钥验证），无需请求 IdP。仅在 Token 刷新或注销时交互。

    5.4 遗留系统改造¶

    • 难点：老旧系统不支持 OIDC。
    • 方案：使用 Sidecar 代理模式（如 OAuth2-Proxy），在网关层统一处理登录，验证通过后将用户信息通过 HTTP Header 传给后端应用。

    6. IAM 与 IdP 的关系及落地改造¶

    6.1 概念辨析：IAM vs IdP¶

    概念 全称 定义 形象比喻
    IdP Identity Provider (身份提供商) 角色/功能。专注于“身份认证” (Authentication)，负责验证用户是谁，并颁发凭证 (Token/Assertion)。 公安局办证大厅。负责核实身份，给你发身份证（Token）。
    IAM Identity and Access Management (身份与访问管理) 系统/体系。包含 IdP，还包含权限控制 (Authorization)、用户生命周期管理、审计等。 整个城市的安保体系。包含办证大厅 (IdP)、门禁系统 (权限控制)、户籍管理 (用户管理)。

    结论：

    • 在 SSO 架构中，IAM 系统扮演了 IdP 的角色。
    • 现有的 go-iam-server 如果仅提供“用户管理”和“登录接口”，它是一个 IAM 系统的雏形；当它实现了 OIDC/SAML 协议并能为其他应用提供统一登录服务时，它就成为了一个成熟的 IdP。

    6.2 如何利用现有 IAM (Go IAM Server) 实现 SSO¶

    目前的 go-iam-server (基于 Go Fiber + JWT) 通常是 API 模式（App 直接 POST 账号密码，IAM 返回 JWT）。要升级为 SSO IdP 模式，需要进行以下架构改造：

    6.2.1 核心改造点¶

    1. 托管统一登录页面 (Hosted Login Page)

      • 现状：各业务 App 自己画登录框，调用 IAM 的 /login 接口。这不是 SSO。
      • 改造：IAM 必须提供一个独立的 Web 页面（如 https://auth.iam.com/login）。
      • 原因：只有在 IAM 自己的域名下输入密码，才能在 IAM 域名下建立全局 Cookie (SSO Session)，实现“一次登录，全网通行”。

    2. 实现 OAuth2/OIDC 协议端点

      • 需要增加以下标准接口：
        • GET /authorize: 接收 client_id, redirect_uri。检查 IAM 自身 Cookie。若未登录 -> 跳转 IAM 登录页；若已登录 -> 生成 Code 跳转回 App。
        • POST /token: 接收 code，验证后颁发 access_token 和 id_token。
        • GET /jwks: 公开公钥，供业务 App 本地校验 Token 签名。

    3. Client 注册管理

      • 在 IAM 数据库中增加 oauth_clients 表，管理合法的接入方：
        • client_id: 应用唯一标识。
        • client_secret: 应用密钥（仅后端 App 需要）。
        • redirect_uris: 允许回调的白名单地址（防止重定向攻击）。

    6.2.2 改造后的交互流程 (OIDC)¶

    假设用户访问 Go Dataset (App A)：

    1. App A 发现未登录，重定向浏览器至 IAM (auth.iam.com/authorize?...)。
    2. IAM 检查浏览器在 auth.iam.com 下是否有 Cookie。
      • 情况一（无 Cookie）：IAM 渲染登录页面。用户输入账号密码 -> IAM 验证通过 -> 写入 IAM 域名的 Cookie -> 生成 Code 跳转回 App A。
      • 情况二（有 Cookie）：IAM 识别出用户已登录（SSO 生效），直接生成 Code 跳转回 App A。
    3. App A 后端用 Code 换取 Token，建立自己的会话。

    6.2.3 技术栈建议 (Go)¶

    在 go-iam-server 中，不要从零手写协议细节，建议集成成熟库：

    • 服务端 (IdP 实现): 推荐使用 github.com/go-oauth2/oauth2 或 github.com/ory/fosite (更底层、更灵活)。
    • 会话管理: 继续使用 Fiber 的 Session 或 JWT，并设置 Cookie 的 Domain=.iam.com、Secure=true、HttpOnly=true、SameSite=Lax（或根据场景调整），保证跨子域 SSO。
    • 密钥轮换: 使用 JWKS 暴露公钥；私钥定期轮换，保留 Overlap 窗口以兼容旧 Token。
    • 多租户: 若需多租户，Client 注册表中新增 tenant_id，在 Token 中携带 tid，资源端按租户校验。

    6.2.4 单点登出 (SLO) 改造¶

    • Back-Channel 优先：IAM 持有各客户端的登出回调地址，后端逐一调用，确保可靠；失败重试并记录审计日志。
    • Front-Channel 兜底：在浏览器侧 iframe/redirect 通知各前端清 Cookie，用于无法后端互通的场景。
    • 短效 Token 策略：Access Token 15 分钟，Refresh Token 可撤销；SLO 时在 IAM 侧吊销 Refresh Token，并将 Access Token 加入 denylist（短时内拦截未过期 Token）。

    6.2.5 渐进式上线步骤（Checklist）¶

    1) 开启 IdP 登录页：部署 auth.iam.com/login，统一密码输入域。 2) 接入 OIDC 端点：/authorize、/token、/jwks、/.well-known/openid-configuration。 3) 注册客户端：为 Dataset、edge GUI、CLI 等分配 client_id/secret 与 redirect_uris。 4) Web 迁移：前端改为 Auth Code + PKCE；后端用 Code 换 Token，校验 ID Token。 5) CLI 迁移：实现 Device Flow；存储 Token 到 ~/.config/coolker/credentials，定期刷新。 6) SDK/移动端：同样使用 PKCE，通过系统浏览器完成授权。 7) SLO 配置：登记各应用的 back-channel logout 端点，启用短效 Token 与 Refresh 吊销。 8) 灰度发布：按应用/用户分组灰度，观察登录成功率与报错率。 9) 监控审计：接入日志/指标/告警，记录登录、令牌颁发、刷新、吊销、SLO 事件。

    6.3 验收与测试要点¶

    • 协议合规：通过 OIDC Conformance Test（或最小集：nonce 校验、aud/iss/sub 校验、过期校验、错误码校验）。
    • 安全基线：CSRF（Cookie 场景）、CSP、防暴力破解（登录限流）、MFA 入口可扩展。
    • 回调安全：严格校验 redirect_uri、origin；拒绝 * 通配符；禁止非 HTTPS（除开发环境）。
    • Token 校验：本地校验签名、aud/iss、exp/nbf、nonce；刷新流正确处理错误码（invalid_grant/slow_down）。
    • 跨域/跨子域：Cookie 域与 SameSite 策略验证；前后端不同二级域名的 SSO 验证。

    6.4 监控与运维¶

    • 核心指标：登录成功率、Token 颁发/刷新/吊销 QPS，Authorize/Token 延迟，SLO 成功率，错误分布（invalid_client、invalid_grant、invalid_scope）。
    • 日志审计：记录登录/登出、Token 发放/刷新/吊销、客户端注册变更；敏感操作加脱敏处理。
    • 容量规划：授权端点与 Token 端点分层水平扩展；JWKS 缓存；数据库/Redis 高可用；限流与熔断保护 IdP。
    • 密钥管理：KMS/密钥柜托管私钥，定期轮换；JWKS 提供多把公钥，旧钥保留一段时间以平滑切换。
