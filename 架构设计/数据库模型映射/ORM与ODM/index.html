    1.ORM（对象关系映射）- 关系型数据库¶

    1.1.理解：¶

    • ORM提供了一种方式，将对象模型映射到关系数据库中的表结构。通过ORM，开发者可以用面向对象的方式操作数据库，而不需要直接编写SQL语句。
    • ORM框架通常会生成SQL查询，处理事务、连接、缓存等数据库操作。

    1.2.使用场景：¶

    1. 关系数据库：适用于MySQL、PostgreSQL、Oracle等关系数据库。
    2. 复杂查询：当需要进行复杂的SQL查询时，ORM可以简化查询逻辑，特别是当涉及到多表关联、复杂的WHERE条件等。
    3. 数据一致性：ORM有助于保持数据的完整性和一致性，因为它会自动处理外键关系、事务等。
    4. 开发效率：减少了手写SQL的需求，提高了开发效率，特别是对于不熟悉SQL的开发者。
    5. 跨平台：许多ORM框架支持多种数据库，方便在不同数据库之间切换。

    示例框架：SQLAlchemy（Python）、Entity Framework（.NET）、Hibernate（Java）。

    1.3.示例¶

    • SQLAlchemy (Python):
    • 链接：SQLAlchemy
    from sqlalchemy import create_engine, Column, Integer, String
from sqlalchemy.ext.declarative import declarative_base
from sqlalchemy.orm import sessionmaker

Base = declarative_base()
engine = create_engine('sqlite:///example.db')

class User(Base):
    __tablename__ = 'users'
    id = Column(Integer, primary_key=True)
    name = Column(String)

Base.metadata.create_all(engine)
Session = sessionmaker(bind=engine)
session = Session()
new_user = User(name='Alice')
session.add(new_user)
session.commit()

    过滤器 说明
    with_entities(...) 只查询相应字段信息
    filter() 把过滤器添加到原查询上，返回一个新查询
    filter_by() 把等值过滤器添加到原查询上，返回一个新查询
    limit() 使用指定的值限定原查询返回的结果数量
    offset() 设置结果范围的开始位置，偏移原查询返回的结果，返回一个新查询
    order_by() 根据指定条件对原查询结果进行排序，返回一个新查询
    group_by() 根据指定条件对原查询结果进行分组，返回一个新查询

    distinct() 返回去重后的查询结果

    having() 返回结果中符合条件的数据，必须跟在group by后面，其他地方无法使用。

    1.4.8.字段类型¶

    方法 说明
    Integer 整数类型，对应数据库中的整型字段（32位整数）。
    BigInteger 也表示整数，但它可以存储更大范围的整数值（64位整数）。
    SmallInteger 也表示整数，但它通常映射到较小的整数范围（16位整数）。
    Numeric 表示精确的十进制数值，通常用于处理需要精确计算的货币、金融或其他需要精确度的数据。
    TINYINT 较小的整数类型，通常映射到8位整数（0到255，或-128到127）。
    Float 浮点数类型，对应数据库中的浮点型字段。
    Double 双精度浮点类型
    String 字符串类型，对应数据库中可变长度的字符型字段（VARCHAR或NVARCHAR）。
    CHAR 表示固定长度的字符串，它对应数据库中的CHAR类型字段。
    VARCHAR 可变长度的字符型字段类型
    Text 表示较长的文本字符串，对应数据库中的TEXT类型字段。
    JSON JSON类型，对应数据库中的JSON字段
    Unicode 表示可变长度的Unicode字符串。
    UnicodeText 长度不限的 Unicode 字符串类型。
    UUID 表示通用唯一标识符，对应数据库中的UUID类型字段。
    Boolean 布尔类型
    Binary 二进制类型
    PickleType 建立在 Binary 类型之上，使用 pickle 序列化的 Python 对象。
    BLOB 表示二进制大对象，例如存储图像、音频、视频或其他二进制数据。对应数据库中的BLOB或BYTEA类型。
    Date 日期类型
    Time 时间类型
    DateTime 日期时间类型
    TIMESTAMP 时间戳类型
    Interval 时间间隔类型
    Enum 表示枚举类型的字符串，对应数据库中的ENUM类型字段。
    TupleType 表示元组的复合类型
    ARRAY 表示 SQL 数组类型。

    1.4.9.字段约束¶

    • 单字段约束：
    • primary_key=True：指定该列为主键，用于唯一标识每条记录。
    • unique=True：指定该列的值必须是唯一的，不能重复。
    • nullable=False：指定该列的值不能为空，必须有值。
    • default=value：指定该列的默认值为value。
    • index=True：为该列创建索引，以提高查询性能。
    • autoincrement=True：指定该列的值自动递增。
    • 组合约束：
    • db.UniqueConstraint：组合唯一索引，例如：db.UniqueConstraint("org_id", "email", name="unique_org_user_key")
    • db.Index：组合索引，例如：db.Index("org_project", "org_id", "project_id")
    • ...
    • 声明性约束：
    • db.CheckConstraint：约束检查，例如：db.CheckConstraint("status IN (0, 1, 4)")

    中文文档: sqlalchemy声明性配置

from sqlalchemy.dialects.mysql import TINYINT
from sqlalchemy import func, text

@declarative_mixin
class FixtureMixin:
    """A basis frame for common model class.

    Mainly define the id, timestamp is_del and other related frame info
    """

    __table_args__ = {
        "mysql_engine": "InnoDB",
        "mysql_charset": "utf8mb4",
        "mysql_collate": "utf8mb4_general_ci",
        "extend_existing": True,
    }

    id = db.Column(db.BigInteger, primary_key=True, autoincrement=True, nullable=False)
    create_time = db.Column(
        db.TIMESTAMP,

    is_del = db.Column(TINYINT, nullable=False, server_default=text("0"))

    1.4.11.Relationship¶

    Relationship 是 SQLAlchemy ORM（对象关系映射）中的一个类，用于定义两个数据库模型之间的关系。在 SQLAlchemy 中，关系可以是“一对多”（one-to-many）、“多对一”（many-to-one）、“多对多”（many-to-many）或“一对一”（one-to-one）。

    1.常用参数详解¶

    1. back_populates:

    2. 用于定义双向关系。

    3. 另一侧的 relationship 参数应该设置相同的 back_populates 值。
    class Parent(Base):
    __tablename__ = "parent"
    id = Column(Integer, primary_key=True)
    children = relationship("Child", back_populates="parent")

class Child(Base):
    __tablename__ = "child"
    id = Column(Integer, primary_key=True)
    parent_id = Column(Integer, ForeignKey("parent.id"))
    parent = relationship("Parent", back_populates="children")

    1. backref:

    2. 类型: Union[bool, str]

    3. 允许设置双向关系，类似于 back_populates，但更简单。自动在另一侧添加关系。
    4. 说明: 定义反向引用。例如，如果 A 有一个到 B 的关系，并且 backref 设置为 "bs"，那么 B 将自动获得一个到 A 的反向关系。
    class Parent(Base):
    __tablename__ = "parent"
    id = Column(Integer, primary_key=True)
    children = relationship("Child", backref="parent")

class Child(Base):
    __tablename__ = "child"
    id = Column(Integer, primary_key=True)
    parent_id = Column(Integer, ForeignKey("parent.id"))

    1. primaryjoin:

    2. 类型: Union[bool, ColumnElement]

    3. 说明: 用于定义关系中的主连接条件。通常是一个 SQL 表达式，指定如何将两个表连接起来。

    4. secondary:

    5. 类型: Union[bool, DeclarativeMeta, Table]
    6. 说明: 用于多对多关系，指定中间表。如果未设置，SQLAlchemy 会自动生成一个中间表。
    association_table = Table(
    "association",
    Base.metadata,
    Column("parent_id", ForeignKey("parent.id")),
    Column("child_id", ForeignKey("child.id"))
)

class Parent(Base):
    __tablename__ = "parent"
    id = Column(Integer, primary_key=True)
    children = relationship("Child", secondary=association_table, back_populates="parents")

class Child(Base):
    __tablename__ = "child"
    id = Column(Integer, primary_key=True)
    parents = relationship("Parent", secondary=association_table, back_populates="children")

    1. secondaryjoin:

    2. 类型: Union[bool, ColumnElement]

    3. 说明: 用于多对多关系，定义从主表到中间表的连接条件。

    4. foreign_keys:

    5. 类型: Iterable[Column]

    6. 说明: 指定外键列，用于定义关系中的外键。

    7. remote_side:

    8. 类型: Union[bool, ColumnElement, DeclarativeMeta]

    9. 说明: 指定关系中的远程侧（即另一侧的表或列）。

    10. lazy:

    11. 类型: str

    12. 说明: 指定加载策略。常用的值有：

      • "select": 默认值，延迟加载，当首次访问关系属性时，会发出一个单独的SELECT语句来获取相关对象。
      • "selectin"：使用SELECT IN批量加载相关对象，适用于一对多关系。
      • "joined": 联接加载, 使用JOIN语句立即加载相关对象。
      • "subquery": 类似 joined，但使用子查询加载。
      • "immediate": 立即加载。
      • "dynamic": 动态加载。返回查询对象，可用于筛选。

    13. uselist:

    14. 类型: bool

    15. 说明: 指定是否将关系作为列表处理。默认为 True，表示关系是多对一或一对多。对于一对一关系，应设置为 False。

    16. cascade:

    17. 类型: str

    18. 说明: 指定级联操作。常用的值有：
    19. "all": 级联所有操作（"save-update", "merge", "refresh-expire", "delete", "expunge"）。
    20. "all, delete-orphan": 级联所有操作，并删除孤立的子对象。
    21. "save-update": 级联保存和更新操作。
    22. "merge": 级联合并操作。
    23. "refresh-expire": 级联刷新和过期操作。
    24. "delete": 级联删除操作。
    25. "expunge": 级联清除操作。
    class Parent(Base):
    __tablename__ = "parent"
    id = Column(Integer, primary_key=True)
    children = relationship("Child", cascade="all, delete-orphan")

class Child(Base):
    __tablename__ = "child"
    id = Column(Integer, primary_key=True)
    parent_id = Column(Integer, ForeignKey("parent.id"))

    1. passive_deletes:

      • 类型: bool
      • 说明: 指定是否在删除父对象时自动删除子对象。默认为 False。

    2. passive_updates:

      • 类型: bool
      • 说明: 指定是否在更新父对象的外键时自动更新子对象。默认为 False。

    3. overlaps

      • overlaps 参数的作用是避免关系表中由于多次关联而导致的歧义或重复加载。
      • 通常用于以下场景：
      • 同一表的多个关系。
      • 一个表与另一个表通过不同路径建立关系。
      • 使用 secondary 参数定义多对多关系时。
      • 示例: 假设有一个 Team 和 Member 表，二者通过中间表 team_member 关联，同时 Team 和 Member 各自又通过其他路径与 Project 相关联，这种情况下可能会出现多重 JOIN 的问题。
      from sqlmodel import SQLModel, Field, Relationship
from typing import List

class Project(SQLModel, table=True):
    id: int = Field(primary_key=True)
    name: str

class Member(SQLModel, table=True):
    id: int = Field(primary_key=True)
    name: str
    projects: List["Project"] = Relationship(back_populates="members", overlaps="team")

class Team(SQLModel, table=True):
    id: int = Field(primary_key=True)
    name: str
    members: List["Member"] = Relationship(
        secondary="team_member", back_populates="teams", overlaps="projects"
    )
    projects: List["Project"] = Relationship(back_populates="teams", overlaps="members")

class TeamMember(SQLModel, table=True):
    team_id: int = Field(foreign_key="team.id", primary_key=True)
    member_id: int = Field(foreign_key="member.id", primary_key=True)

    2.ODM（对象文档映射）- 文档数据库 MonogoDB¶

    2.1.理解：¶

    • ODM用于将对象模型映射到文档存储（如MongoDB、CouchDB等NoSQL数据库）。它允许开发者以面向对象的方式操作文档数据库。
    • ODM处理文档的CRUD操作，通常会提供查询语言来简化对文档的查询。

    2.2.使用场景：¶

    1. 文档数据库：适用于MongoDB、CouchDB等文档型NoSQL数据库。
    2. 灵活性：当数据结构不固定或需要频繁变更时，ODM提供更大的灵活性。
    3. 大数据：处理大规模数据时，文档数据库的性能优势可以被ODM很好地利用。
    4. 嵌套数据：当数据有复杂的嵌套结构时，ODM可以更自然地处理这种结构。
    5. 快速开发：对于需要快速迭代和原型开发的项目，ODM可以简化数据模型的定义和操作。

    示例框架：Mongoose（Node.js）、Doctrine ODM（PHP）、PyMODM（Python）。

    2.3.示例¶

    • MongoEngine (Python):

    • 链接：MongoEngine

    • 示例：

      from mongoengine import *

connect('mydb')

class User(Document):
    name = StringField(required=True)
    email = EmailField(required=True)

user = User(name='Alice', email='alice@example.com').save()

    2.4.比较与选择¶

    • 数据结构：如果你的数据模型是严格结构化的，关系数据库和ORM可能更适合。如果数据结构是灵活的或嵌套的，ODM和文档数据库可能是更好的选择。
    • 查询复杂度：ORM在处理复杂SQL查询时可能更有优势，而ODM在处理文档查询时更直观。
    • 性能：对于高并发或大数据量场景，NoSQL数据库可能提供更好的性能，但这也取决于具体的使用情况。
    • 开发者熟悉度：选择你团队更熟悉的技术栈。
    • 扩展性：考虑未来的扩展需求，NoSQL数据库通常在横向扩展上更有优势。

    在实际项目中，选择ORM还是ODM应基于项目需求、数据模型、团队技术栈以及数据库的选择。两者都有其适用场景，关键在于理解它们的优势和限制，并根据具体情况做出最佳选择。

    3.ROM (对象内存映射) -NoSQL 之 Redis¶

    3.1.理解¶

    ROM 是指将对象模型映射到Redis数据库中的键值对上。Redis作为一个内存数据库，提供了极快的读写速度和丰富的数据结构（如字符串、列表、集合、哈希表等）

    3.2.使用场景¶

    1. 高速缓存：
    2. 当需要快速访问频繁读取的数据时，ROM可以将对象直接映射到Redis中，提供比传统数据库更快的访问速度。
    3. 计数器和排行榜：
    4. Redis的原子操作和数据结构（如Sorted Sets）非常适合实现计数器、排行榜等需要实时更新和查询的功能。
    5. 会话存储：
    6. 由于Redis的内存存储特性，非常适合存储用户会话信息，提供快速的读写操作。
    7. 实时分析和监控：
    8. 可以利用Redis的Pub/Sub功能进行实时数据推送和监控。
    9. 分布式锁：
    10. Redis可以用于实现分布式锁，确保在分布式环境中对共享资源的互斥访问。
    11. 任务队列：
    12. 利用Redis的列表结构，可以实现简单的任务队列，处理异步任务。

    3.3.示例¶

    • redis-om (Python): 提供了高级抽象，使得使用现代 Python 应用程序可以轻松地在 Redis 中建模和查询数据。

    3.4.对比分析¶

    1.基本概念和特性¶

    特性 Pydantic-Redis Redis-OM
    核心功能 基于 Pydantic 模型，将数据存储在 Redis 中，支持数据验证和类型提示。 提供类似 ORM 的接口，用于在 Redis 中轻松存储、查询和操作数据。
    依赖的核心库 Pydantic redis-py
    数据模型支持 基于 Pydantic 模型，支持复杂嵌套、字段验证和类型约束。 提供专用的 Redis 模型，支持索引、查询等高级功能。
    Redis 数据结构支持 主要支持 Redis 的 Hash 数据结构。 支持 Redis 的多种数据结构（如 Hash、JSON 等），并且支持索引和全文搜索功能。
    官方支持/维护 第三方社区支持，更新较活跃。 Redis 官方提供支持，更新频繁，文档详细。

    2.查询能力¶

    特性 Pydantic-Redis Redis-OM
    主键查询 支持基于主键的快速查询。 支持主键查询，使用自动生成的主键 pk。
    索引查询 不支持。 支持基于索引的复杂查询（如范围查询、相等查询等）。
    全文搜索 不支持。 支持 Redisearch 提供的全文搜索功能。
    查询语法 主键查找：Model.get(primary_key) 提供类似 SQL 的查询接口：Model.find().where(...) 或全文搜索语法。

    3.数据结构支持¶

    特性 Pydantic-Redis Redis-OM
    支持的 Redis 数据结构 主要是 Hash。 支持 Hash、JSON 等，提供丰富的数据操作功能。
    扩展功能 无特殊扩展。 原生支持 Redisearch 索引和全文搜索，适合复杂查询。

    4.性能¶

    特性 Pydantic-Redis Redis-OM
    性能优化 轻量级，适合简单的存储和读取操作。 更加复杂，适合大规模数据存储和查询操作。
    Redisearch 支持 不支持 Redisearch。 支持 Redisearch，提供高效的全文搜索和索引功能。

    5.生态和社区支持¶

    特性 Pydantic-Redis Redis-OM
    生态支持 依赖 Pydantic，适合小型到中型项目。 Redis 官方支持，适合中型到大型项目，功能更强大，文档更完善。
    更新频率 社区维护，更新活跃，但可能不够稳定。 Redis 官方支持，更新频率高，功能稳定。

    6.使用场景建议¶

    6.1.Pydantic-Redis¶

    适合以下场景：

    • 项目已经使用 Pydantic，且希望与 Redis 集成。
    • 数据模型较为简单，只需要主键查询和基本存储功能。
    • 不需要复杂的查询和全文搜索。
    • 更注重轻量级和简单性。
    6.2.Redis-OM¶

    适合以下场景：

    • 需要复杂查询功能（索引、范围查询、全文搜索）。
    • 项目需要支持 Redisearch 和其他高级功能。
    • 需要 ORM 式开发体验，简化 Redis 操作。
    • 项目规模较大，存储和操作的数据量较多。

    7.总结对比表¶

    特性 Pydantic-Redis Redis-OM
    核心功能 基于 Pydantic 的数据验证和存储。 提供 ORM 式接口，支持高级 Redis 功能。
    查询能力 主键查询。 主键查询、索引查询、全文搜索。
    数据结构支持 主要是 Hash。 支持 Hash、JSON，并支持索引和 Redisearch。
    适合场景 轻量级项目，简单数据存储和读取。 中大型项目，复杂查询和高级功能需求。
    维护与支持 社区维护，依赖 Pydantic。 Redis 官方支持，功能更强大，更新频繁。
