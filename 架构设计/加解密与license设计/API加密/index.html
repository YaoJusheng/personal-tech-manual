    API 加密方案设计¶

    本文基于主流技术栈与成熟落地经验，系统化给出“API 加密/防篡改/防重放/鉴权”的可落地方案，并提供 Python/Go 示例。

    重要说明：

    • TLS/HTTPS 是必须的底座，它解决“链路机密性与完整性”。
    • 业务侧所谓“API 加密”通常是指：在 HTTPS 之上进一步做 请求签名/防重放/应用层加密/端到端加密，以抵御更强的威胁模型（代理层不可完全信任、终端环境复杂等）。

    1. 背景与目标¶

    1.1 常见诉求¶

    • 防止 API 请求/响应在传输途中被窃听（机密性）。
    • 防止请求被篡改（完整性）。
    • 防止请求被伪造（身份鉴别/消息认证）。
    • 防止重放攻击（Replay）。
    • 保护敏感字段（例如 License、密钥、设备指纹、内部配置）。
    • 兼顾 Web/SDK/CLI 多客户端接入的可运维性（密钥分发、轮换、追踪、审计）。

    1.2 威胁模型（先定义再选型）¶

    建议按强度从低到高明确你要防什么：

    1) 网络窃听/中间人（MITM）：HTTPS 基本可覆盖。 2) 请求重放：攻击者复制一份合法请求重复发送。 3) 请求参数篡改：攻击者修改 body/query/header 造成越权或数据污染。 4) 客户端密钥泄露：SDK/CLI 里硬编码 key 被逆向。 5) 中间层不可信：网关/代理/日志系统可能看到敏感字段。 6) 服务端被动攻击面：签名算法/序列化不一致导致验签绕过、时间窗口与 nonce 管理不当。

    2. 主流方案总览与适用性¶

    2.1 只用 HTTPS（TLS）¶

    • 提供：链路加密 + 完整性 + 服务器身份校验（可选客户端证书）。
    • 不提供：请求级防重放、应用级消息认证、端到端字段加密。
    • 适用：绝大多数 Web/API 场景的基础安全要求。

    2.2 mTLS（双向 TLS）¶

    • 提供：客户端身份鉴别（证书级别）、链路加密。
    • 适用：服务间调用、内网/专线、设备到云（可控设备证书）的高安全场景。
    • 难点：证书生命周期管理（签发/吊销/更新）、移动端/浏览器限制（浏览器端 mTLS 部署成本高）。

    2.3 请求签名（HMAC-SHA256 / RSA/ECDSA/Ed25519）¶

    • 提供：消息完整性、防伪造（对称密钥 HMAC 或非对称签名）。
    • 常见：AWS SigV4、Stripe 事件签名、很多 IoT/支付接口。
    • 适用：SDK/CLI/设备端调用、对“参数不可篡改”要求高的接口。
    • 备注：签名本身不等同于“加密”，签名只证明“没被改且来自持钥方”。

    2.4 防重放（timestamp + nonce + window + idempotency）¶

    • 提供：阻止捕获流量后重复提交。
    • 适用：所有签名方案的标配；特别是转账/下发指令/License 激活等接口。
    • 实现要点：服务端要保存 nonce（短期），并限制时间窗。

    2.5 应用层加密（JWE / AES-GCM Envelope Encryption）¶

    • 提供：即使在 HTTPS 终止后（代理层/日志层）仍能保护敏感字段，实现“端到端字段加密”。
    • 适用：中间层不可信、需要保护敏感字段不落日志/不被网关看到的场景。
    • 典型：JWE（JSON Web Encryption）、自定义 envelope（随机数据密钥 + 公钥加密/密钥协商）。
    • 难点：密钥管理、调试复杂度、性能与报文膨胀。

    2.6 OAuth2/OIDC + DPoP（绑定 Token 到客户端密钥）¶

    • 提供：标准化鉴权（OAuth2/OIDC）+ 防 Token 被盗用（DPoP/持有证明）。
    • 适用：多客户端（Web/SDK/CLI）统一身份体系，Token 风险较高的场景。

    3. 选型建议（推荐组合）¶

    3.1 互联网通用“默认推荐”¶

    HTTPS + 标准鉴权（OIDC/JWT） + 请求级防重放（必要接口）:

    • Web：OIDC (Auth Code + PKCE)；Access Token 短效。
    • SDK/CLI：同样 OIDC（Device Flow / PKCE），关键接口加签名（可选）。

    3.2 需要“请求不可篡改 + 防重放”的推荐¶

    HTTPS + HMAC-SHA256 签名 + timestamp+nonce:

    • 对称密钥适合服务到服务/受控设备/SDK。
    • 如果担心对称密钥泄露与分发，使用非对称签名（客户端私钥签名，服务端公钥验签）。

    3.3 需要“端到端敏感字段加密（代理层不可见）”的推荐¶

    HTTPS + 字段级 AES-GCM +（可选）签名/DPoP:

    • 仅把敏感字段加密，而不是整包加密（减少调试成本与膨胀）。
    • 数据密钥（DEK）每次请求随机生成，使用服务端公钥/密钥协商加密 DEK（Envelope）。

    4. 方案落地：标准化请求签名协议（建议）¶

    为减少多语言/多客户端差异，建议统一“签名串规范”。

    4.1 请求头约定¶

    • X-Api-Key: 客户端标识（或 client_id）。
    • X-Timestamp: 秒级或毫秒级时间戳（建议秒）。
    • X-Nonce: 随机串（最少 96bit 随机）。
    • X-Body-SHA256: 请求体 SHA256（无 body 用空串的 SHA）。
    • X-Signature: 签名值（HEX/Base64）。
    • X-Signature-Alg: HMAC-SHA256 / ED25519 等。
    • （可选）Idempotency-Key: 幂等键（支付/下发命令等强烈建议）。

    4.2 Canonical String（签名原文）¶

    建议格式（换行分隔，避免歧义）：

    {METHOD}\n
{PATH_WITH_QUERY}\n
{X-Body-SHA256}\n
{X-Timestamp}\n
{X-Nonce}

    注意：

    • PATH_WITH_QUERY 的 query 需要“排序 + URL 编码一致”。
    • body hash 对 JSON 要求：客户端序列化必须稳定（建议用原始 body 字节做 hash，不要对对象重新序列化）。

    4.3 服务端验签流程¶

    1) 校验 X-Timestamp 在允许窗口内（例如 ±300 秒）。 2) 校验 nonce 未使用过（Redis setnx：key=nonce，ttl=窗口+冗余）。 3) 计算 body_sha256 与 header 一致。 4) 生成 canonical string，按指定算法验签。 5) 通过后再执行业务逻辑；失败返回明确错误码（不要泄露细节）。

    5. 方案落地：字段级加密（AES-GCM Envelope）¶

    5.1 推荐做法¶

    • 仅加密敏感字段（如 license_key、device_fingerprint、secret_payload），其余字段明文便于路由/限流/审计。
    • 使用 AES-256-GCM（同时提供机密性+完整性）。
    • 每次请求生成随机 DEK（数据密钥），并生成随机 IV。
    • DEK 用服务端公钥加密（RSA-OAEP）或通过 ECDH 派生共享密钥（更优，但复杂）。

    5.2 报文建议结构¶

    {
  "kid": "server-key-2026-01",
  "enc": "A256GCM",
  "dek_wrapped": "base64(...) ",
  "iv": "base64(...) ",
  "ciphertext": "base64(...) ",
  "aad": "base64(...)"
}

    aad（附加认证数据）建议包含：method、path、timestamp、nonce 等，确保上下文绑定，防止跨接口重放。

    6. Python/Go 落地示例¶

    6.1 HMAC-SHA256 请求签名（Python）¶

    import hashlib
import hmac
import time
import secrets

def sha256_hex(data: bytes) -> str:
    return hashlib.sha256(data).hexdigest()

def canonical_string(method: str, path_with_query: str, body_sha256: str, ts: str, nonce: str) -> bytes:
    return f"{method}\n{path_with_query}\n{body_sha256}\n{ts}\n{nonce}".encode("utf-8")

def sign_hmac_sha256(secret: bytes, msg: bytes) -> str:
    return hmac.new(secret, msg, hashlib.sha256).hexdigest()

def build_headers(method: str, path_with_query: str, body: bytes, api_key: str, secret: bytes) -> dict:
    ts = str(int(time.time()))
    nonce = secrets.token_hex(16)  # 128-bit
    body_hash = sha256_hex(body)
    msg = canonical_string(method, path_with_query, body_hash, ts, nonce)
    sig = sign_hmac_sha256(secret, msg)
    return {
        "X-Api-Key": api_key,
        "X-Timestamp": ts,
        "X-Nonce": nonce,
        "X-Body-SHA256": body_hash,
        "X-Signature-Alg": "HMAC-SHA256",
        "X-Signature": sig,
    }

    6.2 HMAC-SHA256 请求签名（Go）¶

    package apisign

import (
    "crypto/hmac"
    "crypto/rand"
    "crypto/sha256"
    "encoding/hex"
    "fmt"
    "time"
)

func sha256Hex(b []byte) string {
    sum := sha256.Sum256(b)
    return hex.EncodeToString(sum[:])
}

func canonicalString(method, pathWithQuery, bodySHA256, ts, nonce string) []byte {
    return []byte(fmt.Sprintf("%s\n%s\n%s\n%s\n%s", method, pathWithQuery, bodySHA256, ts, nonce))
}

func randomHex(n int) (string, error) {
    b := make([]byte, n)
    if _, err := rand.Read(b); err != nil {
        return "", err
    }
    return hex.EncodeToString(b), nil
}

func SignHMACSHA256(secret, msg []byte) string {
    mac := hmac.New(sha256.New, secret)
    mac.Write(msg)
    return hex.EncodeToString(mac.Sum(nil))
}

func BuildHeaders(method, pathWithQuery string, body []byte, apiKey string, secret []byte) (map[string]string, error) {
    ts := fmt.Sprintf("%d", time.Now().Unix())
    nonce, err := randomHex(16) // 128-bit
    if err != nil {
        return nil, err
    }
    bodyHash := sha256Hex(body)
    msg := canonicalString(method, pathWithQuery, bodyHash, ts, nonce)
    sig := SignHMACSHA256(secret, msg)

    return map[string]string{
        "X-Api-Key":        apiKey,
        "X-Timestamp":      ts,
        "X-Nonce":          nonce,
        "X-Body-SHA256":    bodyHash,
        "X-Signature-Alg":  "HMAC-SHA256",
        "X-Signature":      sig,
    }, nil
}

    6.3 AES-GCM 字段级加密（Python，示意）¶

    说明：示例强调流程。生产环境需要完善 kid、密钥轮换、错误码、以及对 aad 的一致性约束。

    import os
import base64
from cryptography.hazmat.primitives.ciphers.aead import AESGCM

def aesgcm_encrypt(plaintext: bytes, dek: bytes, aad: bytes) -> dict:

    }

    6.4 AES-GCM 字段级加密（Go，示意）¶

    package apienc

import (
    "crypto/aes"
    "crypto/cipher"
    "crypto/rand"
    "encoding/base64"
    "io"
)

type EncPayload struct {
    Enc        string `json:"enc"`
    IV         string `json:"iv"`
    Ciphertext string `json:"ciphertext"`
    AAD        string `json:"aad"`
}

func AESGCMEncrypt(plaintext, dek, aad []byte) (*EncPayload, error) {
    block, err := aes.NewCipher(dek)
    if err != nil {
        return nil, err
    }
    gcm, err := cipher.NewGCM(block)
    if err != nil {
        return nil, err
    }
    iv := make([]byte, 12)
    if _, err := io.ReadFull(rand.Reader, iv); err != nil {
        return nil, err
    }
    ct := gcm.Seal(nil, iv, plaintext, aad)
    return &EncPayload{
        Enc:        "A256GCM",
        IV:         base64.StdEncoding.EncodeToString(iv),
        Ciphertext: base64.StdEncoding.EncodeToString(ct),
        AAD:        base64.StdEncoding.EncodeToString(aad),
    }, nil
}

    7. 运维与安全要求（必须写清）¶

    7.1 密钥管理¶

    • HMAC 密钥：不得硬编码在公开环境；按 client_id 分配；支持轮换（双密钥验签窗口）。
    • 签名私钥：建议托管 KMS/HSM；客户端侧私钥尽量使用系统钥匙串/安全模块。
    • 加密密钥：kid 标识版本；服务端支持多版本解密；定期轮换。

    7.2 防重放策略¶

    • 时间窗：建议 300 秒（可按业务收紧）。
    • nonce：Redis SETNX + TTL，TTL >= 时间窗 + 网络抖动。
    • 幂等：对关键写接口强制 Idempotency-Key，服务端保存结果（或保存请求哈希）。

    7.3 兼容性与规范¶

    • 统一编码（UTF-8）、统一 Base64（标准或 URL-safe 需明确）。
    • 明确签名输入是“原始 body 字节”还是“规范化 JSON”。建议用原始 bytes。
    • 错误码规范：区分 invalid_signature、replay_detected、timestamp_out_of_range。

    8. 推荐落地路线（最小可用 → 增强）¶

    1) 先保证 HTTPS 全覆盖（含 HSTS、证书自动续期）。 2) 先做鉴权（OIDC/JWT 或 API Key），再做签名。 3) 对关键接口加签 + 防重放（HMAC + timestamp + nonce + 幂等键）。 4) 确有必要时再做字段级加密（仅敏感字段 AES-GCM）。 5) 统一密钥管理与轮换（KMS + kid + 双写窗口）。

    9. 托底方案¶

    本节将两套“已经在文档/工程中出现过的实现”沉淀为可复用的托底方案（A/B）。它们都适用于“HTTPS 之外还要再加一层”的场景，但侧重点与成本不同。

    9.1 托底方案A¶

    严格按来源章节：混合加密（RSA 包对称密钥 + AES 加密数据）+ 签名/哈希 + 时间戳防重放。来源描述支持“客户端私钥签名”或“哈希/HMAC”，并要求 query/body 都可加密，响应也可用同密钥加密返回。 来源只描述 API 加密的通用流程：HTTPS 底座 + RSA 公钥包裹对称密钥 + 对称加密数据 + HMAC/SHA256 做完整性/认证 + 时间戳防重放。

    密钥交换：客户端生成随机对称密钥（AES），用服务端公钥（RSA-OAEP）加密，随请求头发送。 数据加密：客户端用 AES 加密 query/body（含表单或文件转 base64）；可把加密后的 query 拼回 URL。 签名/哈希：将（加密后的）queryString + timestamp + 明文对称密钥 + body参数 顺序拼接，做 SHA256（或客户端私钥 RSA-SHA256，或 HMAC-SHA256）；放入签名头。 时间有效性：ts 必填；服务端比对时间窗防重放；nonce 去重可选（来源未硬性规定）。

    • 时间有效性：携带时间戳，服务端比较当前时间防重放；可选 nonce 去重。

    • 头部：

    • ek (X-Enc-Key): Base64(RSA-OAEP(pub_server, aes_key))
    • ts: 时间戳
    • sign: Base64(SHA256 / HMAC-SHA256 / RSA-SHA256 over 拼接串)
    • nonce（可选）：用于重放去重 体：ciphertext/iv/tag（建议 AES-GCM；来源未限定模式）；GET 可将加密 query 作为 ciphertext=... 拼接在 URL。

    • canonical 示例：{timestamp}\nMETHOD\n/path\nquery\nbase64(ciphertext)\nnonce。

    • 客户端：

    1) 获取服务端公钥；生成 aes_key、iv。 2) AES 加密 query/body（文件转 base64 再加密；表单同理）。 3) 拼接待签名串：query_enc + ts + plaintext_aes_key + body_enc（来源顺序），做 SHA256/HMAC 或 RSA 签名。 4) RSA-OAEP(pub_server) 包裹 aes_key -> ek；发送头 ts/sign/ek[/nonce] + 密文体。

    • 服务端：

    1) 校验时间窗（可选 nonce 去重）。 2) RSA-OAEP 私钥解 ek 得 aes_key。 3) 按相同顺序重算 SHA256/HMAC 或验 RSA 签名，比对 sign。 4) AES 解密 query/body，进入业务；需要回包机密性则用同 key 加密响应。 4) AES-GCM 解密明文，进入业务逻辑。

    • json { "headers": { "ek": "base64(rsa_oaep(aes_key))", "ts": "1736160000", "sign": "base64(sha256_or_hmac_or_rsa(signature_input))", "nonce": "optional" }, "body": { "ciphertext": "base64(enc_body)", "iv": "base64(12B)", "tag": "base64(16B)" }, "query": "ciphertext=..." // 若将加密 query 追加到 URL }

    • 优点：

    • 覆盖“密钥交换 + 数据机密性 + 完整性/来源校验 + 时间窗防重放”，并支持响应加密闭环。
    • 可选择客户端私钥签名（不可抵赖）或 HMAC/哈希（实现更简单）。
    • 缺点：
    • 签名串含明文对称密钥，需确保仅走 HTTPS 且避免日志泄露。
    • 签名拼接规则涉及“加密后再拼”，需严格一致，否则易验签失败。
    • 未定义 kid/轮换/错误码，nonce 去重也需自建。
    • 示例基于对称 HMAC，未覆盖非对称签名或多密钥版本管理。

    9.2 托底方案B：“RSA 包裹 AES key + AES-GCM + HMAC”¶

    9.2.1 已实现的协议要点¶

    服务端公开端点（获取 RSA 公钥）：

    • GET /api/v1/crypto/key
    • 返回：public_key_pem + fingerprint（公钥 DER 的 SHA-256 Base64）

    客户端每次请求生成临时对称密钥，采用混合加密：

    • AES-256-GCM 加密完整 JSON body，形成：
    • body JSON：{"ciphertext": b64, "nonce": b64, "tag": b64, "aad"?: b64}
    • RSA-OAEP-SHA256 加密（包裹）AES key：
    • header：X-Enc-Key = Base64(RSA_OAEP(server_pub, aes_key))
    • HMAC-SHA256 做请求签名：
    • header：X-Signature = Base64(HMAC_SHA256(aes_key, canonical))
    • canonical string（换行分隔）：
      • timestamp\nMETHOD\n/path\nquery\nbase64(ciphertext)
    • header：X-Timestamp（服务端校验时间窗，默认 ±300s）
    • header：X-Nonce（存在但当前服务端不做“去重存储”）
    • header：X-Sign-Alg（默认 HMAC-SHA256）

    服务端中间件处理流程（请求侧）：

    1) 跳过：HEAD/OPTIONS，以及配置的白名单路径；/crypto/key 允许不签名访问。 2) 校验必需头：X-Enc-Key、X-Signature、X-Timestamp。 3) 校验时间窗（取 X-Timestamp 前 10 位，支持秒/毫秒）。 4) RSA 解密 X-Enc-Key 得到 aes_key。 5) 按 canonical string 验证 HMAC。 6) AES-GCM 解密 body，替换 wsgi.input 为明文 JSON 供业务层继续处理。

    GET 请求的特殊处理：

    • 不允许 GET body（兼容性考虑）；加密载荷通过 query 参数 enc 传递：
    • enc=base64(json(AesPayload))
    • 验签时，服务端只取原始 query string 中的 enc=... 片段以避免被代理追加参数影响。

    9.2.2 与“传统 HMAC 签名”的关键差异¶

    • 传统 HMAC：客户端与服务端共享一个长期 secret（或 per-client secret），签名用于“身份认证 + 完整性”。
    • 本方案 HMAC：HMAC key = 每次请求随机生成的 AES key，并用 RSA 公钥加密后送到服务端。
    • 这保证了中间人无法篡改（因为拿不到 AES key，无法重算 HMAC），同时 payload 也被 AES-GCM 加密。
    • 但它不直接表达客户端身份：任何人都可以拿到服务端公钥并构造“自洽的加密请求”（是否允许仍需依赖业务鉴权，如 Token/API Key）。

    9.2.3 优点 / 缺点¶

    • 优点：
    • 无需客户端长期密钥（不需要预置 HMAC secret，也不要求客户端持有私钥）。
    • 实现简单且性能可控：AES-GCM 处理大 payload，RSA 只加密 32 字节 AES key。
    • 支持代理层不可见：应用层密文可避免敏感数据落入网关/日志（前提是不在中间层解密）。
    • 缺点：
    • 抗重放不足：当前只做时间窗校验，X-Nonce 没有做服务端去重存储；在时间窗内存在“原样重放”风险。
    • 身份属性弱：需要叠加 OIDC/JWT/API Key 等，否则仅靠该协议无法区分调用方。
    • 响应未加密（以当前实现为准）：中间件只处理请求解密；响应仍依赖 HTTPS。

    9.2.5 客户端/服务端落地步骤¶

    • 客户端：

    1) GET /api/v1/crypto/key 获取 public_key_pem（并可校验 fingerprint）。 2) 生成随机 aes_key(32B) + nonce(12B)，AES-GCM 加密明文得到 ciphertext/tag。 3) 计算 canonical string：ts\nMETHOD\n/path\nquery\nbase64(ciphertext)，用 HMAC-SHA256(aes_key, canonical) 得到签名。 4) 用 RSA-OAEP-SHA256(public_key) 包裹 aes_key，放入 X-Enc-Key；将签名放入 X-Signature，同时带 X-Timestamp、X-Nonce、X-Sign-Alg。 5) 发送 body：{"ciphertext":...,"nonce":...,"tag":...,"aad"?:...}；GET 则将 body base64 作为 enc=... query。

    • 服务端：

    1) 拦截除白名单外的请求；校验时间窗。 2) RSA 解包 aes_key，重算 canonical，验 HMAC。 3) AES-GCM 解密 body；将明文写回 wsgi.input 给业务；可把解密后的 JSON 合并回 query 供 GET 场景使用。 4) 记录 crypto.context（ts/nonce）用于审计；建议在业务侧或中间件层补充 nonce 去重。

    9.2.6 报文示例（POST，头+体）¶

    • Headers：
    • X-Enc-Key: Base64(rsa_oaep(aes_key))
    • X-Signature: Base64(hmac_sha256(aes_key, canonical))
    • X-Timestamp: 1736160000
    • X-Nonce: f3c5e9b1d2044d2f
    • X-Sign-Alg: HMAC-SHA256
    • Body：
    {
    "ciphertext": "base64(...)",
    "nonce": "base64(12B)",
    "tag": "base64(16B)",
    "aad": "base64(optional)"
}

    9.2.4 建议补强点（不改变主思路）¶

    • 强制 X-Nonce，服务端用 Redis SETNX(nonce, 1, TTL=window+delta) 做去重。
    • 将 nonce、path、method、content-type 等加入 AAD/canonical，进一步绑定上下文。
    • 由 AES key 派生两把子密钥（例如 HKDF）：一把用于 AES-GCM，一把用于 HMAC（更规范）。
    • 公钥下发增加 kid/版本号，客户端可做 fingerprint pinning，支持平滑轮换。

    10. 两种托底方案与其它方案对比与建议¶

    10.1 快速对比¶

    • 如果你需要“强客户端身份（设备级）+ 不可抵赖”：优先考虑 托底方案A（非对称签名是关键）。
    • 如果你需要“快速落地的端到端机密性（全包加密），且身份鉴权由 OIDC/JWT/API Key 解决”：优先考虑 托底方案B（混合加密 + 中间件解密）。
    • 如果你只需要“参数不可篡改 + 防重放”，且可安全分发 secret：用本文第 4 章的 标准 HMAC 签名协议即可（成本最低、审计最好做）。
    • 如果你只担心“网关/日志不可信但又不想整包加密”：优先做 字段级 AES-GCM envelope（本文第 5 章），调试/运维成本更低。

    10.2 推荐落地建议（按最小风险）¶

    1) 先统一身份鉴权（OIDC/JWT 或 API Key），避免把“身份”寄托在加密层。 2) 再决定是否需要端到端加密：

    • 只保护敏感字段：字段级 AES-GCM（最推荐）
    • 需要整包密文：托底方案B（最快落地）

    3) 最后再加“强设备身份/不可抵赖”：托底方案A（成本最高，但能力最强）

    10.4 选择提示（针对托底方案 A/B）¶

    • 想快速复刻现有代码、兼容 GET、可接受中间件预置逻辑：选 方案B，并补 nonce 去重、kid 轮换、可选响应加密。
    • 想按来源文档自定义 canonical/路径策略或需要极简模板：选 方案A，但需自行确定 canonical、错误码、轮换与去重策略。

    11. 其它方案的细化示例¶

    11.1 “标准 HMAC 签名 + 防重放”快速模板¶

    • 适用：客户端可安全存储对称密钥（服务-服务、受控设备、CLI）。
    • 实现步骤：

    1) 服务器为每个 client_id 分配 secret（KMS 存储），支持 kid 双密钥验签窗口。 2) 客户端：构造 canonical：METHOD\nPATH_WITH_QUERY\nbody_sha256\nts\nnonce，用 HMAC-SHA256(secret) 计算签名；发送必要 headers（见 4.1）。 3) 服务端：

    • 校验时间窗；nonce 用 Redis SETNX 去重；
    • 对原始 body 计算 SHA256，与 header 比对；
    • 重算签名，支持 kid 轮换（旧/新密钥各验一次）。

    4) 错误码：invalid_signature、replay_detected、timestamp_out_of_range、invalid_body_hash。

    11.2 “字段级 AES-GCM envelope”请求/响应示例¶

    • 适用：只加密敏感字段，便于调试与路由。
    • 请求示例（仅加密 license_key 字段）：
    {
    "kid": "server-key-2026-01",
    "enc": "A256GCM",
    "dek_wrapped": "base64(rsa_oaep(dek))",
    "iv": "base64(12B)",
    "ciphertext": "base64(enc({\"license_key\":\"...\"}))",
    "aad": "base64(method|path|ts|nonce)",
    "plain": {"username": "alice", "plan": "pro"}
}
    • 响应示例（可选）：同样用新的 dek 加密敏感字段返回；未加密字段便于客户端快速解析。
    • 服务端要点：依据 kid 解包 dek；仅将 ciphertext 部分解密并合入业务对象；明确 aad 组成（建议 method/path/ts/nonce）。

    11.3 “OIDC + DPoP”最小落地要点¶

    • 适用：Web/SDK/CLI 统一身份体系，希望防 Token 被盗用的重放。
    • 流程摘要：

    1) 客户端生成一对 DPoP 密钥（可存于安全存储/Keychain）； 2) 获取 Access Token 时带 DPoP header（含公钥 JWK + 对 token endpoint 的签名证明）； 3) 调用资源服务时在 header 带 DPoP 证明（对 ts|htm|htu|jti 的签名）； 4) 资源服务验证：Token（签名、有效期、scope）+ DPoP 证明（签名、时间窗、jti 去重）+ Token 绑定的 cnf 公钥与请求的 DPoP 公钥一致。

    • 注意：DPoP 只防“被窃取后滥用”，仍需 HTTPS；浏览器限制下可结合 Service Worker/ WebCrypto 保存私钥。
