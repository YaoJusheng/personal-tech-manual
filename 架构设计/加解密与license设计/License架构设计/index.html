    一、License 机制分析¶

    1.License 简介¶

    软件许可证（Software License）是一种用于控制和管理软件使用权限的机制。许可证的主要目的是防止未经授权的使用和复制，确保软件的合法性和合规性。它通常包含许可的使用条款、授权用户的信息、使用期限和功能限制等。

    一个License许可应该包含以下基本内容：

    • 授权用户（用户编号、MAC地址、系统 UUID、CPU序列号、主板序列号等）
    • 授权期限（开始结束时间、无限期等）
    • 授权功能（应用版本、应用功能、License等级）
    • 版权保护
    • 技术支持和升级服务

    如何实现License授权？

    要想生成一个安全性高的License，必须让其满足以下几个特征：

    • 保密性: License里携带的data信息具有一定的隐蔽性，这样可以防止想要破解License的人寻找到生成License的规律。
    • 防篡改: 防止License里携带的重要信息被篡改，例如License有效时间如果被篡改。
    • 时效性: License会记录软件可以使用的有效期，并在验证License的时候判断其是否过期。
    • 可找回: 用户申请的License一旦丢失或者要续期，基于第一次申请License时创建的源文件，再一次生成新的License，新的License会携带用户当初申请时的信息。

    2.软件License授权原理

    2.1.授权机制¶

    （1）使用 非对称加密 生成密钥对，包含私钥和公钥。

    （2）授权者（服务器）保留私钥，使用非对称加密 的 私钥对授权信息诸如使用截止日期，mac 地址等内容进行数字签名，生成 license 签名证书。

    （3）公钥给使用者（客户端），放在代码中使用，用于对许可进行数字签名校验，校验通过再校验许可的内容，验证 license 签名证书是否符合使用条件。

    2.2.非对称加密¶

    1.对称加密: 对称加密就是我们一般意义上的加密算法，这种算法在加密和解密时都使用同一个密钥，所以对称加密算法的密钥又叫做共享密钥。对称加密算法一般使用AES（Advanced Encryption Standard）加密算法。

    2.非对称加密: 非对称加密有两个密钥，一个公钥一个私钥。公钥是公开的，供多个人使用；私钥是非公开的，仅一个人或者少数群体使用。当非对称加密算法用作加解密时，公钥用来对明文加密，私钥用来给密文解密，这个顺序不能颠倒。可以这样理解，密文是私密的东西，只有少数人才能解密，所以少数人手里的私钥用来解密，多数人手里的公钥不能解密只能加密。同样还有其他的非对称加密算法，如：椭圆曲线密码学(ECC)方式，相比RSA，应用不够广泛，但密钥长度更短,但安全性更高

    2.3.数字签名¶

    一种防伪造技术，能够通过公钥验证的数据一定是私钥拥有者所签发的。因此除非私钥泄露（一般放在服务器），否则没有人能够伪造。

    为提高破解难度，对应用添加带上发布时间戳， 校验代码编译进二进制执行文件。

    上面的非对称加密可以解决“密钥分配问题”，但是它不能防止伪造消息的问题，因此需要授权者（服务端）以私钥对授权信息进行加密（签名）。

    注：签名时是私钥用来加密，公钥用来解密。

    为便于理解，可参考：数字签名

    1.先签名再加密

    2.先加密后签名

    该方式不安全的前提是：

    • 接收方Bob根据签名内容中的证书ID找到对应的Mallory公钥来验证签名。

    • 接收方Bob仅使用公钥验签来识别发送方的身份。

    2.4.其他¶

    出于性能方面的考虑，大多数情况下给消息加密还是使用的对称加密算法，为了解决“密钥分配问题”，只会在第一次发送共享密钥的时候才会使用非对称加密，一旦消息接收者得到了共享密钥，通信双方就能够通过共享密钥进行通信了。

    此外，使用非对称密钥对消息签名也可以防止消息被人篡改，由于性能原因一般不会对消息原文进行签名，而是先通过哈希算法形成消息摘要，再对消息摘要签名。消息接收者验签时会将消息的明文进行哈希，再将消息签名解密，两者比对如果一致则证明消息没有被篡改过。

    3.3.某license控制专利¶

    参考：一种License控制方法及装置

    License控制步骤：

    • StepA: 应用服务器的业务模块向应用服务器的License客户端申请License许可；

    • StepB: License客户端根据从License服务器获得的License数据对业务模块进行License许可；

    • License客户端定时向License服务器发送License增量数据 ；
    • License服务器根据所述License增量数据更新License全局数据 ，并向所有的License客户端发送更新后的License数据；

    • License客户端更新本地的License数据；

    • StepC: 业务模块获得License客户端的许可后执行业务操作。

    利用各应用服务器的License客户端提供License许可，从而提高了License许可的实时性，降低了系统的负荷，增强了License许可的可靠性。

    实施例中，将License的许可模块直接布署在应用服务器上，每个应用服务器均有单独的License客户端负责License许可，这样原本由License服务器独自承担的许可工作分解给众多的应用服务器，License服务器的并发处理压力将得到很大缓解。同时，当应用服务器上的License客户端运行异常时，业务模块也可以直接向License服务器进行许可，这样的机制提高了系统的运行效率，且有效的解决了License服务器的性能瓶颈。

    上述：

    1. License增量数据 指单个License客户端在一个更新周期内新占用以及新释放的License数量。
    2. License全局数据 包括License服务器所有License数量、正在使用的License数量和剩余可用的License数量。
    3. License许可包括：判断是否可以申请新License；或者判断是否可以释放License；或者判断是否可以执行业务操作。

    问题：Mac地址的改变导致license校验失败

    ~~原方案： Mac地址+salt~~

    解决思路：

    • 授权用户（用户编号、CPU序列号、主板序列号等）
    • 授权期限（开始结束时间、无限期等）
    • 授权功能（应用版本、应用类型、License类型）
    • 版权保护
    • 技术支持和升级、续签服务

    方案设计概述

    1. ==多因素结合生成UID==：
    2. 通过组合多个硬件和软件信息，如CPU序列号、硬盘序列号、主板序列号、BIOS信息、操作系统信息等，来生成一个更加稳定的UID。
    3. 使用加密哈希算法（如SHA-256）对这些信息进行哈希处理，生成一个唯一且不可逆的识别码。
    4. ~~设备 指纹技术：~~
    5. 通过收集客户端设备的各种信息，包括硬件配置、软件环境、网络信息等，生成设备指纹。设备指纹可以在设备配置变化不大的情况下保持一致。
    6. 定期更新和验证：
    7. 定期对UID进行验证，以检测潜在的伪造或设备变更。如果检测到变化，可以要求用户重新验证或重新申请License。

    ４.3.1.准备¶

    在 License 授权签发流程中，通常需要使用三对密钥对。

    1. License 签发密钥对

    用途

    • 用于对 License 数据进行签名和验证，以确保 License 的真实性和完整性。

    密钥对

    • 私钥1（Private Key）：用于签名 License 数据，通常由 License 签发方（如软件提供商）持有。
    • 公钥1（Public Key）：用于验证 License 数据的签名，通常嵌入到客户端软件中。

    • 验证 License

    }

    1. 通信加密密钥对 (服务端密钥对)

    用途

    • 用于在客户端和服务端之间的安全通信，防止中间人攻击和数据泄露。

    密钥对

    • 服务端私钥2（Server Private Key）：用于解密客户端发送的数据。
    • 服务端公钥2（Server Public Key）：用于加密客户端发送的数据，通常公开给所有客户端。

    1. 密钥对传递

    在安全的License授权和验证流程中，

    客户端的公钥需要传递给服务端以便于验证客户端签名的数据。这种传递可以通过几种常见的方法来实现：

    • 初次注册时传递: 客户端在首次注册或激活时，将公钥发送给服务端。
    • 通过文件传递: 将公钥文件上传到服务端，例如通过HTTP文件上传接口。
    • 使用密钥管理服务（KMS）
    • 嵌入到软件包中: 发行软件时，预先嵌入密钥对。

    而服务端的公钥需要传递给客户端，以便客户端能够加密发送的数据。为确保其安全性和可靠性，可以通过几种常见的方法来实现：

    • 通过HTTPS安全下载: 服务端生成一对公私钥，并将公钥保存到一个文件中。
    • 预先嵌入到客户端: 在发布客户端应用时，将服务端的公钥嵌入到应用代码中。
    • 使用证书管理服务（KMS）
    • 通过配置文件传递: 将公钥文件包含在客户端的配置文件中，并通过安全渠道分发。

    二、API加密¶

    1.必要性分析¶

    1. 数据保护： API 在传输过程中，可能包含敏感数据，如用户身份信息、支付信息等。未加密的通信容易遭受中间人攻击，导致敏感信息泄露。加密能有效保护数据免受窃取。
    2. 防止篡改： 未加密的 API 请求和响应容易被篡改，攻击者可能修改数据、伪造请求，导致业务逻辑被破坏。加密通过完整性校验防止数据被篡改。
    3. 身份认证： 在开放网络环境下，未加密的通信可能会导致身份被冒充。加密配合身份认证机制可以有效确认请求方的合法性。
    4. 合规需求： 一些行业和法规要求通信加密，如 GDPR、HIPAA。API 加密可以帮助企业满足合规要求，避免法律风险。

    2. 可行性分析¶

    1. 性能开销： 加密操作会带来一定的性能损耗，尤其在高并发场景下，密集的加密解密可能导致性能瓶颈。需要选择合适的加密算法和优化服务器资源，以确保性能不受严重影响。
    2. 加密算法选择： 对称加密和非对称加密都有各自的适用场景。对于性能要求较高的场景，对称加密（如 AES）速度较快，适合加密大数据。非对称加密（如 RSA）适合用来加密小数据，如 API 密钥或签名。
    3. 密钥管理： 加密机制的核心是密钥管理。密钥泄露会带来极大的安全风险，因此需要合理设计密钥生成、存储和轮换机制。使用硬件安全模块（HSM）或云密钥管理服务可以增强密钥管理的安全性。

    2. 必要性：确保请求方的身份合法性，并防止请求在传输过程中被篡改。通常配合 API 密钥使用。

    3. 建议：使用 HMAC（基于哈希的消息认证码）结合 API 密钥，对每个请求的关键参数（如时间戳、请求方法等）进行签名，服务器验证签名是否匹配。

import hashlib
import time

secret_key = b'secret_key'

def create_signature(data):
    timestamp = str(int(time.time()))
    message = f"{data}|{timestamp}".encode()
    return hmac.new(secret_key, message, hashlib.sha256).hexdigest()

data = "example_api_request"
signature = create_signature(data)
print(f"Signature: {signature}")

    from Crypto.Cipher import AES

def pad(s):
    return s + (16 - len(s) % 16) * ' '

def encrypt(message, key):
    cipher = AES.new(key.encode('utf-8'), AES.MODE_ECB)
    return base64.b64encode(cipher.encrypt(pad(message).encode('utf-8'))).decode()

key = 'thisisasecretkey'

    3. 对称加密： 采用相同的密钥进行加密和解密。具有加密速度快、计算量小的优点，但密钥的安全传输是问题。在本方案中，对称加密主要用于数据的实际加密传输。
    4. 非对称加密： 使用一对密钥（公钥和私钥）进行加密和解密。公钥用于加密数据，私钥用于解密数据。非对称加密可以确保密钥的安全传输，但加密速度较慢，不适合长文本加密。在本方案中，非对称加密主要用于对称密钥的加密。
    5. 哈希算法： 无论用户输入什么长度的原始数据，经过计算后输出的密文都是固定长度的，只要原数据稍有改变，输出的“摘要”便完全不同
    6. 签名算法： 一般指的是，通过私钥对数据进行签名，然后通过公钥对数据进行验签

    11. 数据加密： 客户端使用对称加密的明文密钥对接口数据进行加密，然后发送给服务端。服务端使用相同的对称加密密钥对数据进行解密。

    13. 数据有效性校验： 在数据发送前，客户端将当前时间作为数据的一部分发送给服务端。服务端收到数据后，解密得到参数中的时间，并与服务端当前时间进行比较，以验证请求是否过期，防止请求重放。

    4.实现¶

    • ==方案 1==：非对称加密 + 对称加密 + 数字签名, 设计细节:
    • 客户端生成随机对称密钥：客户端生成一个随机的对称密钥（如 AES），用于加密需要发送的数据。
    • 使用服务端公钥加密对称密钥：为了安全传输该对称密钥，客户端使用服务端的公钥对对称密钥进行加密。
    • 数据加密：使用对称密钥（AES）加密要发送的数据，生成加密的消息体。
    • 数据签名：客户端使用自己的私钥对加密数据进行签名，确保数据的完整性和来源的真实性。
    • 服务端解密和验证：服务端使用私钥解密对称密钥，再使用该对称密钥解密数据，同时使用客户端的公钥验证数据签名。
    • ~~方案 2~~：基于 HTTPS（TLS）的传输加密 + HMAC 签名, 设计细节:
    • HTTPS 加密通道：客户端和服务端通过 HTTPS 建立加密连接，所有传输的数据都经过 TLS 加密。
    • 数据签名：客户端发送数据前，使用预共享的密钥生成 HMAC 签名，将数据和签名一同发送给服务端。
    • 服务端验证：服务端接收数据后，使用预共享的密钥验证 HMAC 签名，确保数据的完整性和来源的真实性。

    4.1.技术步骤¶

    1、密钥生成与管理： 通过工具生成非对称密钥对，服务端负责保管私钥。（客户端也可以定期从服务端获取公钥（传输过程中有泄露的风险），并保存到LocalStorage。）

    2、加密算法选择： 对称加密算法可选用常用的AES，非对称加密算法可选用常用的RSA。哈希算法可选用SHA-256、MD5，签名算法可选用RSA-SHA256等。

    3、签名规则： 等queryString和body参数加密后，将queryString、时间戳、明文对称密钥、body参数按顺序进行拼接，然后通过SHA256算法进行哈希得到签名sign，然后将sign放在header中进行传递

    4、参数传递：

    • ek(encrypt-key)： xxxxxxx（非对称加密后的对称密钥）
    • ts： xxxxxxx（时间戳）

    • sign： xxxxxxx

    • [ ] 将请求参数queryString拼成"param1=value1&param2=value2"格式

    • [ ] 将queryString进行对称加密，得到"ciphertext=xxxxx"，并重新拼接到url后面

    query参数 -> 参考GET请求

    body参数: 通过对称加密 封装后的Form-data得到base64密文，直接用body进行传输

    对于文件: 读取文件内容封装进Form-data并通过对称加密得到base64密文保存到临时文件传输

    5、后端处理：

    • 请求有效验证： 获取header中ts参数，得到时间戳并判断时间戳是否超过一定时间；
    • 解密对称密钥： 通过私钥解密header中的ek参数，得到明文的对称密钥key；
    • 验签： 将queryString、时间戳、明文对称密钥、body参数按顺序进行拼接，然后通过SHA256算法进行哈希得到签名sign，与header中的sign做比较；
    • 解密参数： 通过明文对称密钥，分别解密queryString、body参数；
    • 加密响应结果： 请求完成后，将响应结果通过对称加密得到base64密文，并返回给客户端

    SDK (Software development kit) 软件开发包，是软件或平台面向开发者的一种快速开发工具或套件，使开发者可以更方便、灵活的使用软件或平台功能。 一般（比较注重生态的）软件或平台会提供不同语言或平台或框架的包，来适用不同的开发者。例如Selenium就提供了Python、Nodejs、Java等各种软件的SDK，这也是其比较流行的原因之一。

    1.SDK分析¶

    在开发 SDK 过程中，通常需要考虑以下几点：

    1. 模块化设计：将 SDK 的功能进行模块化拆分，每个模块各自负责一部分功能，如认证模块、加密模块、通信模块等。
    2. 安全性保障：结合加密、认证、签名等方式，确保数据在传输和存储过程中的安全。
    3. 接口设计：设计清晰易用的 API 接口，方便开发者集成和使用。
    4. 错误处理与日志：需要提供健壮的错误处理机制，并有良好的日志支持，方便调试与问题定位。
    5. 文档与测试：保证 SDK 提供完善的使用文档和测试用例，便于维护和集成。

    SDK 的实现流程可以分为以下几步：

    1. 初始化：SDK 的初始化过程，通常包括加载配置、设置 API 密钥等步骤。
    2. 用户认证：用户通过提供凭证（如 API 密钥或 OAuth 认证）进行认证。
    3. 数据加密：对发送的数据进行 AES 或 RSA 加密，保证数据在传输过程中的安全性。
    4. 数据签名：对请求进行 HMAC 或数字签名，确保数据的完整性。
    5. 通信模块：通过 HTTPS 进行安全的通信，SDK 调用 API 接口，发送数据给服务端。
    6. 错误处理与日志：在通信或处理数据过程中，对异常情况进行捕获，并记录日志。
    7. 返回处理：处理服务端返回的响应，进行数据解密、验签等操作。

    1. 轻量化：不需要复杂的用户管理或大规模的依赖，适合仅用于离线校验的场景。
    2. 模块化与封装：将 License 校验逻辑模块化，易于集成到其他项目中。
    3. 高安全性：通过数字签名、有效期检查等手段，确保 License 的安全性和唯一性。
    4. 快速校验：离线校验无需网络请求，减少了外部依赖，保证了校验速度。

    3.4.示例分析¶

    假设 License 文件中包含以下内容：

    {
    "license_id": "12345",
    "valid_until": "2025-12-31",
    "signature": "base64-encoded-signature",
    "permissions": ["use_feature_x"]
}
    1. 签名校验：使用 RSA 或 ECDSA 公钥对 license_id 和 valid_until 进行签名验证，确保 License 文件没有被篡改。
    2. 有效期检查：对比当前日期与 valid_until 确保 License 未过期。
    3. 权限校验：检查 permissions 数组，确保 License 授权用户使用特定功能。

    代码示例 (Python 使用 cryptography 库)：

    from cryptography.hazmat.primitives.asymmetric import rsa, padding
from cryptography.hazmat.primitives import hashes
import json
import datetime

def load_license(file_path):
    with open(file_path, 'r') as f:
        return json.load(f)

def verify_signature(public_key, license_data, signature):
    message = f"{license_data['license_id']}{license_data['valid_until']}".encode()
    public_key.verify(
        signature,
        message,
        padding.PKCS1v15(),
        hashes.SHA256()
    )

def check_validity(license_data):
    valid_until = datetime.datetime.strptime(license_data['valid_until'], '%Y-%m-%d')
    if datetime.datetime.now() > valid_until:
        raise ValueError("License expired")

def check_permissions(license_data, required_permission):
    if required_permission not in license_data['permissions']:
        raise PermissionError("Permission denied")

# 主函数
def validate_license(file_path, public_key, required_permission):
    license_data = load_license(file_path)
    signature = base64.b64decode(license_data['signature'])

    # 校验签名、有效期和权限
    verify_signature(public_key, license_data, signature)
    check_validity(license_data)
    check_permissions(license_data, required_permission)

    return "License is valid"

    这种设计更加轻量化，并且直接针对 License 校验，适用于无需用户管理的场景。
