    一、简介¶

    UniFFI 是 Mozilla 开发的跨语言绑定生成器，核心思路是 “一次编写 Rust 逻辑，自动生成多语言调用接口，多平台调用”，解决了为各语言手写 FFI 绑定极易出错的痛点，自动生成安全、符合目标语言风格的绑定代码，彻底免除手动编写 FFI 胶水代码的繁琐工作。- link。

    核心定位与设计目标¶

    • 安全优先：暴露的接口严格遵循 Rust 安全规范，外部调用不会触发未定义行为，对象必须满足 Send + Sync。
    • 多语言支持：官方支持 Kotlin（Android）、Swift（iOS）、Python、Ruby；第三方支持 C#、Go、JavaScript 等。
    • 极简开发：无需手动处理 FFI、类型转换、内存管理，只需定义接口，工具自动完成所有适配。
    • 生产级可靠：广泛用于 Firefox 移动端与桌面端，稳定性经过大规模验证。

    🔬 UniFFI 技术内幕：自动生成多语言绑定的实现方式¶

    UniFFI 通过两种方式定义接口并自动生成绑定，最终都是基于 C ABI 来保证二进制兼容性。

    • 基于 UDL 文件（声明式）：在 .udl 文件（类似 WebIDL）中集中定义接口，然后由 uniffi-bindgen 工具解析并生成各语言代码link-1。
    • 基于过程宏（代码注解式）：直接在 Rust 代码里用 #[uniffi::export] 等宏注解函数、结构体和 Trait-，比 UDL 更贴近 Rust 开发习惯，但复杂的 Trait 或自定义类型映射可能仍需 UDL 补充-2。
    // 过程宏示例：直接用宏导出接口
#[uniffi::export]
pub fn add(a: u32, b: u32) -> u32 {
    a + b
}
```text

### ✨ UDL 进阶：文件分类与代码生成

对于“UDL 文件是否支持分类，生成多个文件”的问题，答案是**部分支持，但需谨慎规划**。

- **单一文件 vs 模块化**：传统上通常是一个 `.udl` 文件对应一个 Rust 库，但 UniFFI 也支持用 `[External="crate_name"]` 语法导入其他 UDL 定义的类型，实现类型定义层面的拆分[-3](https://webkit.crisal.io/mozilla-release/source/third_party/rust/uniffi_bindgen/src/library_mode.rs)。
- **生成多个文件**：一个 UDL 文件会为每个目标语言**生成一个或多个绑定文件**。UniFFI 的 `library_mode` 功能支持处理 Cargo workspace 中多个 crate 的绑定生成，但通常建议**保持单个库的 UDL 定义集中管理**，以避免复杂的依赖关系。

### 🌐 UniFFI 语言生态概览

下表整理了 UniFFI 目前的支持情况，供你参考。

| 支持级别       | 语言                             | 说明                                                         |
| :------------- | :------------------------------- | :----------------------------------------------------------- |
| **官方支持**   | Kotlin, Swift, Python, Ruby      | UniFFI 团队维护，质量最高[-4](https://docs.rs/crate/uniffi_core/0.27.0) |
| **第三方支持** | C#, Golang, Dart                 | 由社区维护，质量可能有差异[-5](https://docs.rs/crate/uniffi_core/0.27.0) |
| **特殊支持**   | JavaScript (Gecko), React Native | 特定平台（Firefox）或通过第三方库支持[-6](https://hacks.mozilla.ac.cn/2023/08/autogenerating-rust-js-bindings-with-uniffi/) |
| **试验性支持** | C++                              | 通过 `uniffi-bindgen-cpp` 命令生成                           |

在实际使用中，如果觉得手写 UDL 繁琐，可以优先尝试过程宏。如果需要将一个大的组件接口拆分到多个文件，建议只在类型定义复用场景下使用 `[External]` 导入，并确保对被导入的 crate 有足够的控制权。

    标准流程¶

    集成 UniFFI 通常遵循以下标准三步走流程-7：

    1. 定义接口：通过 UDL 文件 或 Rust 过程宏 描述对外 API（函数、结构体、枚举、接口、错误等）。

    2. 实现逻辑：在 Rust 中实现定义好的接口，并在 build.rs 或 lib.rs 中引入脚手架代码。

    3. 编译产物：编译生成一个包含 C ABI 导出符号的动态/静态库，如Rust 编译为 cdylib 动态库，目标语言直接调用生成的绑定，像本地代码一样使用。

    4. 生成与使用：用 uniffi-bindgen 工具解析接口，生成目标语言的绑定文件。

    5. Rust 脚手架代码（暴露 C 兼容 FFI 层）

    6. 目标语言绑定代码（Kotlin/Swift/Python 等）

    以 Python 为例：

    uniffi-bindgen generate src/my_component.udl --language python --out-dir ./bindings

    多语言绑定最小示例¶

    方式 1：UDL 文件定义¶

    • 原理：使用 UniFFI Definition Language（类 WebIDL） 在独立 .udl 文件中完整描述接口。
    • 语法示例（math.udl）
    namespace Math {
    u64 add(u64 a, u64 b);
    u64 multiply(u64 a, u64 b);
};

dictionary CalcResult {
    u64 value;
    string message;
};

interface Calculator {
    CalcResult compute(string expr);
};
    • 优势：接口与 Rust 实现解耦、清晰直观、支持复杂类型、多语言一致性强、便于团队协作。
    • 生成命令
    uniffi-bindgen generate src/math.udl -l kotlin -o bindings/kotlin
uniffi-bindgen generate src/math.udl -l swift -o bindings/swift

    方式 2：Rust 过程宏（Proc-Macro）定义¶

    • 原理：直接在 Rust 代码中用 #[uniffi::export] 等宏标记对外接口，无需独立 UDL 文件。
    • 代码示例
    use uniffi::prelude::*;

#[uniffi::export]
pub fn add(a: u64, b: u64) -> u64 { a + b }

#[derive(Debug, uniffi::Record)]
pub struct CalcResult {
    pub value: u64,
    pub message: String,
}

#[uniffi::export]
pub struct Calculator;

#[uniffi::export]
impl Calculator {
    pub fn compute(&self, expr: String) -> CalcResult { /* ... */ }
}
    • 优势：接口与实现同文件、开发高效、适合小型库、无需学习 UDL 语法。
    • 生成方式：编译时自动生成 Rust 脚手架；绑定代码仍通过 uniffi-bindgen 从宏元数据生成。

    方式 3：混合模式（UDL + 宏）¶

    • 原理：核心接口用 UDL 统一管理，部分简单接口用宏快速补充，两者互补。
    • 适用场景：大型项目、模块划分复杂、需要灵活扩展接口时。
    • 注意：UDL 与宏定义的接口 不能重名，生成时会合并为统一组件接口。

    📦工程化实现¶

    🎯 多语言一键生成示例¶

    1. 目录结构¶
    src/
 ├── api.udl        # 仅 1 个接口定义文件
 └── lib.rs
bindings/
 ├── kotlin/        # 自动生成 Kotlin 绑定
 ├── swift/         # 自动生成 Swift 绑定
 └── python/        # 自动生成 Python 绑定
uniffi.toml         # 统一配置（关键！）
    2. 统一配置文件：uniffi.toml（核心）¶
    [bindings]
# 一次配置，所有语言共用
library_name = "my_core"

# Kotlin
[bindings.kotlin]
package_name = "com.example.core"
out_dir = "bindings/kotlin"

# Swift
[bindings.swift]
module_name = "MyCore"
out_dir = "bindings/swift"

# Python
[bindings.python]
out_dir = "bindings/python"
    3. 单条命令生成所有绑定¶
    uniffi-bindgen generate src/api.udl --config uniffi.toml

    执行后自动生成：

    bindings/
 ├── kotlin/MyCore.kt
 ├── swift/MyCore.swift
 └── python/my_core.py

    4. Rust 侧集成（极简）¶

    // lib.rs
uniffi::include_scaffolding!("api"); // 对应 api.udl

    📁 标准方法：多文件架构¶

    这是最直接、也是官方文档推荐的做法，即“一个Crate一个UDL”。

    • 架构：为每个逻辑模块（或每个namespace）分别创建一个Rust crate。每个crate下包含一个独立的UDL文件，并在各自的Cargo.toml中管理依赖关系。
    • 绑定生成：你需要为每个UDL文件分别运行uniffi-bindgen generate命令，生成对应的绑定文件。
    • 优点：结构清晰，每个模块独立编译、测试和版本管理，是UniFFI的标准实践，社区支持最好。

    #### 🚀 进阶方案：库模式 (Library Mode)\

    1.简介¶

    从UniFFI v0.25版本开始，引入了一种更先进的库模式 (Library Mode)，可以完美解决你的需求。

    • 架构：不需要显式的UDL文件！ 直接在Rust代码中使用#[uniffi::export]等过程宏来标注需要导出的接口-link。所有“导出”信息在编译时会被嵌入到生成的cdylib库中。
    • 绑定生成：不再需要传入UDL文件，而是将最终编译好的cdylib库文件（例如.so, .dylib, .dll）直接传给uniffi-bindgen命令-link。UniFFI会通过cargo_metadata分析库文件内部嵌入的元数据，自动识别出所有通过过程宏导出的模块及其复杂的依赖关系，然后一次性为所有模块生成对应的多语言绑定文件。
    • 优点：完全避免维护UDL文件，简化了构建流程，尤其适合管理多个相关模块之间复杂的依赖关系。
    2.对比¶
    特性 多文件架构 (传统方法) 库模式 (先进方法)
    核心原理 多个crate，多个UDL 单个crate，过程宏，分析最终库
    模块组织方式 每个逻辑模块一个独立crate 单个项目，用Rust模块 (mod) 组织
    绑定生成命令 对每个UDL文件运行一次uniffi-bindgen 对最终生成的 cdylib 库运行一次
    学习/使用曲线 较低，概念清晰，易于上手 中等，需要理解库模式和过程宏
    适用场景 各模块功能独立，耦合度低，需要独立版本 模块间有复杂依赖，希望简化构建流程
    项目复杂性 管理多个crate的配置和依赖 管理单一crate，但需熟悉过程宏用法

    建议：

    • 如果你是初次使用UniFFI，或者项目模块间耦合度低，推荐先从多文件架构开始。
    • 如果你的项目模块众多且相互依赖，希望简化构建流程，那么库模式是更现代、更强大的选择。
    3.示例¶

    库模式无需UDL文件，通过 Rust 的过程宏在代码中直接标注导出项，最终通过分析编译好的 cdylib 库一次性生成所有绑定。

    关键实现：以下是单个crate生成绑定文件的示例，多个文件的生成考虑使用多个crate来实现。

    • Cargo.toml
    [package]
name = "uniffi_lib_demo"
version = "0.1.0"
edition = "2021"

[lib]
crate-type = ["cdylib"]

[dependencies]
uniffi = { version = "0.28", features = ["build"] }
    • src/lib.rs (使用过程宏)
    #[uniffi::export]
pub fn greet(name: String) -> String {
    format!("Hello, {}!", name)
}

#[uniffi::export]
pub fn calculate_sum(a: i32, b: i32) -> i32 {
    a + b
}

uniffi::setup_scaffolding!(); // 关键步骤

    此方法使用 #[uniffi::export] 宏标记公开函数，并通过 uniffi::setup_scaffolding!() 生成所有必需的桥接代码。

    生成绑定文件¶

    库模式的核心是直接对已编译的共享库文件（如 .so, .dylib, .dll）进行操作，一次性生成所有绑定-。

    1. 构建项目生成动态库
    cargo build --release
    1. 基于动态库生成绑定
    uniffi-bindgen generate --library ./target/release/libuniffi_lib_demo.so \
                        --language kotlin \
                        --out-dir ./bindings

    路径说明：

    • libuniffi_lib_demo.so 是 Linux 下的文件名。
    • 在 macOS 上是 libuniffi_lib_demo.dylib，
    • 在 Windows 上是 uniffi_lib_demo.dll。

    多语言绑定设计¶

    核心思路¶

    基于 UniFFI (Universal FFI) 框架，将 Rust 核心库通过 FFI 暴露给其他语言。架构设计采用按服务拆分 cdylib 的方式，将核心业务服务按业务领域分为多个独立的动态库 crate（以下为示例）：

    Crate 导出内容
    auth 认证相关 API
    dataset 数据集相关 API
    types 共享类型定义

    实现步骤¶

    1. 定义 Rust 业务接口¶

    在 lib.rs 中实现核心业务逻辑，不直接导出 UniFFI 接口。各 FFI 服务 crate 在 src/ffi/{auth,dataset,types}/src/lib.rs 中通过 uniffi::setup_scaffolding!() 宏暴露 API。

    2. 构建动态库¶
    cargo build --release -p auth -p dataset -p types

    生成三个 .so/.dylib 文件：

    • libauth.{so|dylib}
    • libdataset.{so|dylib}
    • libtypes.{so|dylib}
    3. 生成语言绑定¶
    语言 工具 生成命令
    Python cargo run -p auth --features bindgen --bin uniffi-bindgen -- generate --language python 官方 UniFFI 内置
    Swift 同上, cargo run --features bindgen --bin uniffi-bindgen 官方 UniFFI 内置
    Go uniffi-bindgen-go (社区工具) make gen-go
    Node.js uniffi-bindgen-node-js (社区工具) make gen-node
    4. 安装绑定生成工具（Go/Node 专用）¶

    将 uniffi-bindgen-go 和 uniffi-bindgen-node-js 安装到本地 bin 目录（不入系统）。

    5. 验证绑定输出¶
    • Python → bindings/python/sdk/{auth,dataset,types}.py
    • Swift → bindings/swift/sdk/{auth,dataset,types}.swift + FFI 头文件
    • Go → bindings/go/sdk/{auth,dataset,types}/*.go
    • Node.js → bindings/node/sdk/{auth,dataset,types}/ (ESM 包格式)

    架构设计要点¶

    1. 服务分离 — 三个 crate 完全独立，各语言绑定可按需引入，避免捆绑无用代码
    2. 统一构建入口 — uniffi-bindgen.rs作为共享的 bindgen 二进制入口
    3. 条件编译 — bindgen feature 控制是否启用绑定生成，避免生产构建膨胀
    4. 本地工具链 — Go/Node 的社区工具版本固定在 .tools 中，与项目一起版本化管理
