    2.多应用架构设计¶

    2.0.简介¶

    1.架构组件¶

    在为开发团队提供一整套应用服务（一系列APP）时，共用一套用户管理和认证以及权限管理系统的最佳实践方案通常涉及以下几个关键组件：

    • 用户与设备：用户通过浏览器或移动设备访问系统。

    • API 网关：所有用户请求通过 API 网关进行负载均衡、路由、权限检查和速率限制。

    • 单点登录 (SSO)：提供统一的用户登录入口，支持多应用统一身份验证。SSO 将身份验证令牌分发给已登录的用户。

    • IAM (身份与访问管理)：

    • 用于用户角色、权限管理

    • 服务与用户之间的授权 (服务与资源的访问控制)。

    • 服务注册与发现：

    • 注册中心负责服务注册、发现和更新。

    • 实现动态的服务间通信。

    • 多应用服务：

    • 每个应用服务独立运行，关注其业务逻辑。

    • 服务间通过 API 互相通信。

    • 数据库与日志：

    • 每个服务拥有独立的数据库（支持分布式）。

    • 集中化的日志记录与监控用于系统诊断和分析。

    2.整体架构¶

    整体架构图:

    APIGateway -.-> |Check| ServiceRegistry[服务注册与发现]

    4.单点登录设计¶

    参考:

    1. 单点登录及常见解决方案原理
    2. 如何实现SSO

    单点登录（Single Sign On），简称为 SSO，是目前比较流行的企业业务整合的解决方案之一

    SSO的定义是在多个应用系统中，用户只需要登录一次就可以访问所有相互信任的应用系统

    SSO 一般都需要一个独立的认证中心（passport），子系统的登录均得通过passport，子系统本身将不参与登录操作

    当一个系统成功登录以后，passport将会颁发一个令牌给各个子系统，子系统可以拿着令牌会获取各自的受保护资源，为了减少频繁认证，各个子系统在被passport授权以后，会建立一个局部会话，在一定时间内可以无需再次向passport发起认证.

    特性 Cookie SSO Token SSO OAuth2 SSO CAS SSO
    协议标准化 无 无 是 是
    跨域支持 较差（需借助跨域方案） 好 好 好
    适用场景 同域或子域环境 分布式、多域环境 第三方授权、多系统整合 企业内部多系统整合
    易用性 高 高 中等 中等
    安全性 依赖 Cookie 的安全机制 较高 较高 较高
    复杂性 低 中等 高 中等

    适用场景建议:

    • 小型系统：选择 Cookie SSO，简单易用。
    • 分布式多域架构：选择 Token SSO，性能和跨域支持良好。
    • 多方整合：选择 OAuth2，支持第三方登录。
    • 企业内多系统整合：选择 CAS SSO，协议成熟稳定，支持扩展。
    1.CAS方案¶

    CAS（Central Authentication Service）单点登录方案 是一个开源的、用于企业级的单点登录解决方案。它提供了一套服务端和客户端的组件，使得在多个应用之间实现单点登录变得简单。

    sequenceDiagram
    title CSA单点登录方案
    participant 浏览器
    participant APP as APP (CAS Client)
    participant CASServer as CAS Server
    participant APP2 as APP2 (CAS Client)

    浏览器 ->> APP: 1. 访问APP
    APP -->> CASServer: 2. 未登录跳转到CAS Server
    CASServer -->> 浏览器: 3. 写入Cookie
    CASServer -->> APP: 4. 返回Server Ticket
    APP -->> CASServer: 5. 验证Server Ticket
    CASServer -->> APP: 6. 验证通过返回验证状态
    APP -->> 浏览器: 7. 登录状态写入session并设置app域下的Cookie

    浏览器 ->> APP2: 8. 访问APP2
    APP2 -->> CASServer: 9. 未登录跳转到CAS Server
    CASServer -->> APP2: 10. 返回Server Ticket
    APP2 -->> CASServer: 11. 验证Server Ticket
    CASServer -->> APP2: 12. 验证通过返回验证状态
    APP2 -->> 浏览器: 13. 登录状态写入session，并在app2域下写入Cookie

    具体流程如下：

    1. 用户访问app系统，app系统是需要登录的，但用户现在没有登录。
    2. 跳转到CAS server，即SSO登录系统，以后图中的CAS Server我们统一叫做SSO系统。 SSO系统也没有登录，弹出用户登录页。
    3. 用户填写用户名、密码，SSO系统进行认证后，将登录状态写入SSO的session，浏览器（Browser）中写入SSO域下的Cookie。
    4. SSO系统登录完成后会生成一个ST（Service Ticket），然后跳转到app系统，同时将ST作为参数传递给app系统。
    5. app系统拿到ST后，从后台向SSO发送请求，验证ST是否有效。
    6. SSO系统返回验证结果
    7. 验证通过后，app系统将登录状态写入session并设置app域下的Cookie。

    至此，跨域单点登录就完成了。以后我们再访问app系统时，app就是登录的。接下来，我们再看看访问app2系统时的流程。

    1. 用户访问app2系统，app2系统没有登录，跳转到SSO。
    2. 由于SSO已经登录了，不需要重新登录认证。
    3. SSO生成ST，浏览器跳转到app2系统，并将ST作为参数传递给app2。
    4. app2拿到ST，后台访问SSO，验证ST是否有效。
    5. SSO系统返回验证结果
    6. 验证成功后，app2将登录状态写入session，并在app2域下写入Cookie。 这样，app2系统不需要走登录流程，就已经是登录了。SSO，app和app2在不同的域，它们之间的session不共享也是没问题的。

    2. 基于 Python 的 LDAP 登录验证示例:

    使用 ldap3 库实现与 LDAP 的交互：

    from ldap3 import Server, Connection, ALL, SUBTREE

LDAP_SERVER = 'ldap://ldap.example.com'
BASE_DN = 'dc=example,dc=com'
USER_DN = 'ou=users,' + BASE_DN

def authenticate(username, password):
    try:
        # 初始化 LDAP 服务器
        server = Server(LDAP_SERVER, get_info=ALL)
        # 构造用户的 DN
        user_dn = f"uid={username},{USER_DN}"
        # 绑定用户（验证用户名和密码）
        conn = Connection(server, user=user_dn, password=password, auto_bind=True)

        # 验证成功，查询用户详细信息
        conn.search(
            search_base=BASE_DN,
            search_filter=f"(uid={username})",
            search_scope=SUBTREE,
            attributes=['cn', 'mail', 'memberOf']
        )
        if conn.entries:
            user_info = conn.entries[0]
            return {
                "status": "success",
                "username": username,
                "details": user_info.entry_attributes_as_dict
            }
        else:
            return {"status": "failure", "message": "User not found"}
    except Exception as e:
        return {"status": "failure", "message": str(e)}

    4.JWT方案¶

    JSON Web Token（JWT）是一个开放标准（RFC 7519），它定义了一种紧凑且独立的方式，可以在各方之间作为JSON对象安全地传输信息。此信息可以通过数字签名进行验证和信任。JWT可以使用秘密（使用HMAC算法）或使用RSA或ECDSA的公钥/私钥对进行签名。

    JSON WEB令牌结构由三部分组成：

    • Header（头部）：包括令牌的类型及正在使用的散列算法。
    • token类型：JWT
    • 加密方式：base64（HS256）
    • Payload（负载）：声明是关于实体（通常是用户）和其他数据的声明。索赔有三种类型：标准注册声明，公共的声明和私有的声明。
    • 用户身份信息（注意，这里因为采用base64编码，可解码，因此不要存放敏感信息）
    • 注册声明：如token的签发时间，过期时间，签发人等
    • Signature（签名）：必须采用编码标头，编码的有效负载，秘密，标头中指定的算法，并对其进行签名。
    1.负载说明¶

    1. 负载－标准的声明：

    2. iss：JWT签发者

    3. sub：JWT所面向的用户
    4. aud：接收JWT的一方
    5. exp：JWT的过期时间，这个过期时间必须要大于签发时间，这是一个秒数
    6. nbf：定义在什么时间之前，该JWT都是不可用的

    7. iat：JWT的签发时间

    8. 负载－公共的声明：可以添加任何信息，一般添加用户的相关信息或其他业务需要的必要信息，但不建议添加敏感信息，因为该部分在客户端可解密。

    9. 负载－私有声明：提供者和消费者所共同定义的声明，一般不建议存放敏感信息，因为base64是对称解密的，意味着该部分信息可以归类为明文信息

    3.优化设计¶

    1.Token 的刷新机制：

    • Access Token：短期有效（如 15 分钟），用于访问资源。
    • Refresh Token：长期有效（如 7 天），仅用于刷新 Access Token，不直接用于访问资源。
    • Access Token 放入 Authorization Header，Refresh Token 放入 HttpOnly Secure Cookie。
    • Token 轮换机制：
    • Access Token 快到期时，前端通过 Refresh Token 向 SSO 服务请求新的 Access Token，保证用户体验不被中断。
    • 旧的 Refresh Token 被废弃，返回新的 Refresh Token，同时更新到 Cookie。

    2.集中式权限管理：

    • 将权限控制逻辑集中在 SSO 服务中。
    • 子系统根据 Token 中的角色或权限字段，执行业务逻辑。

    3.单点退出（SSO Logout）：

    1. 添加 Token 黑名单：
    2. 在服务端维护一个全局黑名单，存储已被注销或失效的 Access Token。
    3. 子系统验证 Access Token 时，需检查 Token 是否在黑名单中。
    4. 单点注销：
    5. 用户在任意子系统发起注销请求。
    6. SSO 服务接收请求，删除用户的 Refresh Token 并将 Access Token 加入黑名单。
    7. SSO 通知其他子系统清理本地会话（如 WebSocket 或 HTTP 通知机制）。
    8. 黑名单同步机制：
    9. 集中存储：将黑名单存储在 Redis 中，所有子系统共享访问。
    10. 分布式事件通知：通过消息队列（如 RabbitMQ、Kafka）通知子系统更新黑名单。
    11. 前端同步退出：
    12. 前端监听用户的登录状态（如轮询、长连接），当发现 Token 已失效时，自动重定向到登录页。

    2. 服务发现:

    API 网关通过 Consul 动态获取服务实例地址：

    • FastAPI 网关示例：
    import requests
from fastapi import FastAPI, HTTPException

app = FastAPI()

CONSUL_URL = "http://localhost:8500"

def discover_service(service_name: str):
    """从 Consul 获取服务地址"""
    response = requests.get(f"{CONSUL_URL}/v1/catalog/service/{service_name}")
    if response.status_code == 200:
        services = response.json()
        if not services:
            raise HTTPException(status_code=404, detail="Service not found")
        # 随机返回一个服务地址
        service = services[0]
        return f"http://{service['ServiceAddress']}:{service['ServicePort']}"
    else:
        raise HTTPException(status_code=500, detail="Failed to discover service")

@app.get("/{service_name}/{path:path}")
async def proxy_request(service_name: str, path: str):
    """网关转发请求"""
    service_url = discover_service(service_name)
    response = requests.get(f"{service_url}/{path}")
    return response.json()

    2.3.1.方案分析¶

    基于Python开发多应用平台的技术栈落地方案和选型，我们需要考虑微服务架构、认证授权、监控系统、日志系统、可视化仪表盘和告警系统等多个方面。以下是详细的技术选型和使用建议：

    • Flask：轻量级Web框架，适合快速开发微服务。

    • FastAPI：高性能的异步Web框架，支持自动生成API文档。

    2.数据库连接和操作¶

    • SQLAlchemy：ORM工具，支持多种数据库。

    • psycopg2：PostgreSQL数据库的适配器。

    • OAuth2：使用OAuth2协议实现认证授权。

    • Authlib：Python库，支持OAuth2和JWT。

    • Flask-OAuthlib：Flask的OAuth库。

    4.监控系统¶

    • Prometheus：开源监控报警系统和时序列数据库。

    5.日志系统¶

    • logging：Python标准库，用于日志记录。

    功能模块：

    • 协议转换：支持 REST、GraphQL 等协议。

    实现流程：

    1. 解析用户请求，检查认证信息（JWT）。
    2. 查询服务地址（通过 Consul 服务发现）。
    3. 转发请求到目标服务，并返回响应。

    示例（基于 FastAPI 和 Consul）：

    from fastapi import FastAPI, Depends, Request
import requests

app = FastAPI()

CONSUL_URL = "http://localhost:8500"

def discover_service(service_name: str):
    """通过 Consul 查询服务地址"""
    response = requests.get(f"{CONSUL_URL}/v1/catalog/service/{service_name}")
    services = response.json()
    return f"http://{services[0]['ServiceAddress']}:{services[0]['ServicePort']}"

@app.get("/gateway/{service_name}/{path:path}")
async def gateway(service_name: str, path: str, request: Request):
    """转发请求到目标服务"""
    service_url = discover_service(service_name)
    forward_url = f"{service_url}/{path}"
    response = requests.request(
        method=request.method,
        url=forward_url,
        headers=request.headers,
        data=await request.body()
    )
    return response.json()

    2.GraphQL 服务代码

    • 基础服务初始化

    使用 Ariadne 实现：

    from ariadne import QueryType, make_executable_schema, graphql_sync
from ariadne.asgi import GraphQL
from fastapi import FastAPI, Request
import requests

# Define GraphQL schema
type_defs = """
type Query {
    userInfo(token: String!): User!
    applicationData(appId: String!): Application!
}

type User {
    id: ID!
    username: String!
    role: String!
}

type Application {
    id: ID!
    name: String!
    description: String
    owner: User
}
"""

query = QueryType()

# Mock SSO/IAM Service URL
SSO_SERVICE_URL = "http://localhost:8000"
APP_SERVICE_URL = "http://localhost:8001"

# Resolve userInfo query
@query.field("userInfo")
def resolve_user_info(_, info, token):
    response = requests.get(f"{SSO_SERVICE_URL}/user", headers={"Authorization": f"Bearer {token}"})
    if response.status_code != 200:
        raise Exception("User not found")
    return response.json()

# Resolve applicationData query
@query.field("applicationData")
def resolve_application_data(_, info, appId):
    response = requests.get(f"{APP_SERVICE_URL}/application/{appId}")
    if response.status_code != 200:
        raise Exception("Application not found")
    app_data = response.json()

    # Fetch owner info
    owner_response = requests.get(f"{SSO_SERVICE_URL}/user/{app_data['owner_id']}")
    if owner_response.status_code != 200:
        raise Exception("Owner not found")
    app_data["owner"] = owner_response.json()

    return app_data

# Create executable schema
schema = make_executable_schema(type_defs, query)

# FastAPI app
app = FastAPI()

# Mount GraphQL
graphql_app = GraphQL(schema, debug=True)
app.mount("/graphql", graphql_app)
    • userInfo：

    1.方案比较¶
    功能点 SSO（单点登录） IAM（身份和访问管理）
    主要目标 在多个系统中实现单一登录认证 管理用户身份及其对资源的权限
    核心功能 用户身份认证，生成令牌 用户管理、角色管理、权限细粒度控制

    from fastapi import FastAPI, Depends, HTTPException
from jose import JWTError, jwt

SECRET_KEY = "your_secret_key"
ALGORITHM = "HS256"

app = FastAPI()

def verify_token(token: str):
    try:
        payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
        return payload
    except JWTError:
        raise HTTPException(status_code=401, detail="Invalid token")

@app.get("/protected")
def protected_route(token: str = Depends(verify_token)):
    return {"message": "Access granted", "user": token}

    1.注册 gRPC 服务实现示例：¶
    import grpc
from concurrent import futures
import requests
from consul import Consul

# gRPC 服务定义
class MyService(grpc.MyServiceServicer):
    def GetData(self, request, context):
        return grpc.DataResponse(data="Hello from Service 1!")

# 注册服务到 Consul
def register_service():
    consul = Consul(host="localhost", port=8500)
    consul.agent.service.register(
        "iam-service-1",
        service_id="iam-service-1-id",
        address="127.0.0.1",
        port=50051,
        tags=["grpc", "example"],
        check={
            "grpc": "127.0.0.1:50051",
            "interval": "10s",
            "timeout": "1s"
        }
    )


# gRPC client for APP2
def call_app2_grpc():
    with grpc.insecure_channel("app2.service.consul:50052") as channel:
        stub = app2_pb2_grpc.App2ServiceStub(channel)
        response = stub.GetData(app2_pb2.DataRequest(name="From APP1"))
        return {"response_from_app2": response.data}


# 启动 gRPC 服务
def start_grpc_server():
    server = grpc.server(futures.ThreadPoolExecutor(max_workers=10))
    grpc.add_MyServiceServicer_to_server(MyService(), server)
    server.add_insecure_port("[::]:50051")
    register_service()  # 注册服务
    print("Service 1 running at 50051")
    server.start()
    server.wait_for_termination()

if __name__ == "__main__":
    start_grpc_server()

    5.GraphQL聚合¶

    在 API 网关上实现 GraphQL 服务，将 IAM/SSO、APP1 和 APP2 聚合为一个 GraphQL 接口。

    from fastapi import FastAPI
from strawberry.fastapi import GraphQLRouter
import strawberry

@strawberry.type
class Query:
    @strawberry.field
    def iam_message(self) -> str:
        # Mock call to IAM service
        return "IAM Service is online"

    @strawberry.field
    def app1_message(self) -> str:
        # Mock call to APP1 service
        return "APP1 Service is online"

    @strawberry.field
    def app2_message(self) -> str:
        # Mock call to APP2 service (via gRPC)
        return "APP2 Service is online"

schema = strawberry.Schema(query=Query)

app = FastAPI()
graphql_app = GraphQLRouter(schema)
app.include_router(graphql_app, prefix="/graphql")

if __name__ == "__main__":
    import uvicorn
    uvicorn.run(app, host="0.0.0.0", port=8080)
