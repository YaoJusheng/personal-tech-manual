    1. 适用范围与目标¶

    本规范用于指导团队在系统建设过程中进行统一、可落地、可持续演进的架构设计。

    适用对象：

    • 新项目立项与方案设计
    • 现有系统重构、拆分、扩容、上云
    • 核心模块（鉴权、任务调度、数据链路、模型服务等）设计与变更

    总体目标：

    • 可支撑业务增长：架构能力与业务复杂度同步演进
    • 可稳定运行：在高并发、异常场景下保持可用
    • 可快速交付：研发效率与工程治理并重
    • 可持续维护：降低系统复杂度与长期维护成本

    2. 架构设计目标与原则¶

    2.1 核心设计目标¶

    目标 核心要求 关键手段
    高可用 关键链路避免单点故障，具备降级、熔断、限流能力 冗余部署、健康检查、自动故障转移
    高性能 满足延迟、吞吐、资源利用率目标，保障关键路径性能 缓存、异步处理、负载均衡、数据库优化
    可扩展 支持横向扩展与模块化扩展，降低功能新增成本 微服务/模块化、无状态设计、事件驱动
    可维护 边界清晰、职责明确、文档完整，便于排障和交接 代码分层、自动化测试、规范化文档
    安全合规 数据安全、访问控制、审计、合规要求前置 身份认证、数据加密、零信任架构
    成本可控 在性能与可靠性约束下持续优化资源成本 弹性伸缩、Serverless、资源利用率监控

    2.2 设计原则¶

    核心理念：架构不是“技术堆叠”，而是围绕业务目标做取舍。所有技术选型需回答：解决了什么业务问题、代价是什么。

    1. 简单优先 (KISS)：优先选择最小可行方案（MVP 架构），仅在明确需求或可预见瓶颈下引入复杂机制。避免过度设计。
    2. 高内聚低耦合：领域边界清晰，模块职责单一。通过稳定接口解耦，避免跨层直连与循环依赖。
    3. 一致性与标准化：统一日志、监控、错误码、鉴权、配置管理规范。统一通信协议和数据契约，减少系统协同成本。
    4. 可观测性内建：监控、日志、Tracing、告警在设计阶段纳入，不在上线后补丁式添加。关键链路必须具备端到端可追踪能力。
    5. 失败可恢复 (Design for Failure)：假设故障必然发生。超时、重试、幂等、补偿、隔离必须明确。每个关键流程都应有异常处理策略与回滚预案。
    6. 安全前置 (Security by Design)：默认最小权限、默认拒绝、敏感数据最小暴露。在设计阶段明确认证、鉴权、审计、加密和密钥管理策略。

    3. 架构设计要素¶

    架构设计应覆盖“业务、应用、数据、技术、运维、安全”六个层面，避免只画组件图不落地。

    3.1 业务与领域建模 (DDD)¶

    • 业务梳理：明确业务场景、用户角色、核心流程、关键指标（如转化率、处理时延）。
    • 边界划分：划分领域边界（Bounded Context），定义上下游关系与防腐层（ACL）。
    • 模型设计：给出核心实体、聚合根（Aggregate Root）、值对象、状态流转与关键约束。
    • 事件驱动：识别领域事件（Domain Events），为事件驱动架构提供基础。

    输出建议：业务流程图（主流程 + 异常流程）、领域模型图（核心实体与关系）、事件风暴图（Event Storming）。

    3.2 模块划分与系统边界¶

    • 职责定义：定义模块职责、输入输出、依赖关系。
    • 通信方式：标注同步调用/异步消息、强依赖/弱依赖。
    • 链路识别：识别关键链路、非关键链路，区分实时路径与离线路径。

    设计检查点：是否存在“万能服务”或“共享大表”导致耦合？是否存在跨模块绕过接口直接访问内部资源？

    3.3 数据架构设计¶

    • 数据模型：表结构、索引、分区、冷热分层。
    • 生命周期：采集、存储、加工、归档、删除。
    • 一致性策略：强一致/最终一致，冲突处理与补偿机制（如 Saga、TCC）。
    • 数据质量：校验规则、幂等键、去重、回溯能力。
    • 现代架构：评估是否需要流批一体（如 Flink）、数据湖仓（Data Lakehouse）以支持实时分析与 AI 训练。

    必须明确：关键数据的主键策略与唯一性约束（如雪花算法、UUID）；读写路径与容量估算（QPS、数据增长率、峰值系数）；备份恢复目标（RPO/RTO）与异地灾备策略。

    3.4 接口与通信设计¶

    • 接口协议：REST / gRPC / 消息队列 / 事件流选型依据。
    • API 契约：版本、兼容策略、错误码、超时与重试策略。
    • 幂等设计：写操作幂等键、去重窗口、重放处理。

    推荐实践：对外 API 保持向后兼容；通过网关统一鉴权、限流、审计；异步链路必须定义“至少一次/至多一次/恰好一次”语义。

    3.5 性能与容量设计¶

    • 容量模型：建立平均流量、峰值流量、增长预测模型。
    • 关键指标：定义 P95/P99 延迟、吞吐、错误率、资源利用率。
    • 缓存策略：本地缓存/分布式缓存、失效策略、热点保护。

    必须输出：性能基线与压测目标；扩容策略（自动扩容阈值、扩容粒度、上限）。

    3.6 高可用与容灾设计¶

    • 单区高可用：无单点、健康检查、故障转移。
    • 跨区容灾：主备、多活、流量切换策略。
    • 降级熔断：故障隔离、优先级保障、降级白名单。

    必须明确：关键组件故障影响范围；故障恢复流程（自动/人工）；演练机制与频率（如季度故障演练）。

    3.7 安全与合规设计 (零信任架构)¶

    • 身份与权限：身份认证（AuthN）与权限控制（AuthZ）模型（如 RBAC、ABAC、PBAC）。
    • 零信任原则：不信任任何内部网络，所有服务间通信必须经过认证与授权（如 mTLS）。
    • 数据安全：敏感数据分类分级、传输加密、存储加密（KMS）、脱敏策略。
    • 审计与合规：审计日志留存策略与可追溯性要求；满足 GDPR、CCPA 等数据出境与隐私保护要求。
    • 供应链安全：依赖与镜像漏洞扫描（如 Trivy）、签名与来源可信（如 Sigstore）。

    3.8 可观测与运维设计¶

    • 监控体系：系统层、应用层、业务层三级指标。
    • 告警策略：告警分级、收敛去重、值班响应机制。
    • 链路追踪：Trace 贯通，覆盖核心调用链。
    • 发布策略：灰度、金丝雀、回滚策略与验证门禁。

    3.9 架构模式选型¶

    根据业务阶段、团队规模、交付节奏和运维能力选择模式，避免“为先进而先进”。

    模式 适用场景 优点 风险/代价
    单体架构 0~1 产品、小团队、需求频繁变化 开发部署简单、调试效率高 后期扩展困难、发布耦合
    模块化单体 中小规模业务、希望先治理边界 保持部署简单，边界逐步清晰 边界若不强制易退化成大泥球
    微服务架构 多团队并行、大规模复杂业务 独立扩展、独立发布、技术自治 运维复杂、治理成本高
    事件驱动架构 高并发异步处理、跨系统解耦 松耦合、吞吐高、削峰填谷 一致性与排障复杂
    Serverless / FaaS 流量波动大、事件触发型任务、快速试错 零运维、按需计费、自动弹性 冷启动延迟、厂商锁定、调试困难
    云原生架构 (Cloud Native) 容器化、微服务、DevOps 结合 极致弹性、高可用、标准化交付 基础设施复杂度高、学习曲线陡峭

    选型决策建议回答 5 个问题：

    1. 未来 12 个月业务复杂度会如何增长？
    2. 团队是否有能力承担分布式治理成本？
    3. 关键链路对一致性和时延的优先级如何？
    4. 发布频率与独立交付需求是否足够高？
    5. 能否提供可观测性与故障治理配套？

    3.10 技术栈选型方法¶

    技术栈选型应基于约束和目标，建议使用“评分矩阵”而非经验拍板。

    评分维度建议：

    • 性能与扩展性：吞吐量、延迟、水平扩展能力。
    • 稳定性与成熟度：社区活跃度、大厂背书、历史故障率。
    • 团队匹配度：团队现有技能栈、学习曲线。
    • 生态与维护性：周边工具链、文档质量、排障难度。
    • 成本：研发成本、运维成本、迁移成本、授权费用。
    • 安全与合规：漏洞修复速度、数据合规能力。

    示例评分模板（1~5 分）：

    维度 权重 方案 A 方案 B 方案 C
    性能与扩展性 25%
    稳定性与成熟度 20%
    团队匹配度 20%
    生态与维护性 15%
    成本 10%
    安全与合规 10%
    加权总分 100%

    常见选型对象：

    • 通信：REST / gRPC / MQ
    • 存储：MySQL / PostgreSQL / Redis / Elasticsearch / 对象存储
    • 任务处理：队列消费 / 定时任务 / 工作流引擎
    • 部署：VM / K8s / Serverless

    4. 架构文档撰写要求¶

    架构文档应“可评审、可执行、可追溯”，避免空泛描述。

    4.1 文档结构模板（建议）¶

    1. 背景与目标：业务背景、问题定义、目标边界。
    2. 约束与前提：时间、人力、成本、合规、上下游约束。
    3. 候选方案对比：至少 2 个方案，列优缺点与成本。
    4. 选型结论：明确为什么选、为什么不选。
    5. 详细设计：模块图、数据流、接口契约、部署拓扑。
    6. 非功能设计：性能、可用性、安全、可观测。
    7. 风险与回滚：风险列表、触发条件、处置预案。
    8. 实施计划：里程碑、资源、验收标准。

    4.2 写作要求¶

    • 结论先行：先给结论，再展开论证。
    • 量化表达：尽量给出数字目标与阈值（如 P99 < 200ms）。
    • 边界清晰：明确“做什么/不做什么”。
    • 图文一致：图中的模块命名与文中保持一致。
    • 术语统一：同一概念在全篇使用统一名称。

    4.3 图示与表达规范¶

    • 图类型建议：
    • 上下文图（C4 Level 1）
    • 容器图（C4 Level 2）
    • 组件图（C4 Level 3）
    • 时序图（关键流程）
    • 部署图（环境与网络）
    • 图示要素：每张图需包含标题、版本、更新时间、作者。
    • 关键标注：关键链路应标注协议、超时、重试、限流、鉴权。

    4.4 可追溯性要求¶

    • 文档需要关联需求、任务、变更记录、评审结论。
    • 重要设计决策建议采用 ADR（Architecture Decision Record） 记录。

    ADR 最小模板：

    • Context（背景）：决策的上下文和面临的问题。
    • Decision（决策）：最终选择的方案。
    • Consequences（影响）：决策带来的正面和负面影响。
    • Alternatives（备选方案）：考虑过但未被采纳的方案及原因。

    4.5 非功能需求（NFR）撰写模板¶

    非功能需求建议采用“指标 + 阈值 + 观测方式 + 验证手段”四元组描述。

    维度 指标示例 目标阈值示例 观测方式 验证方式
    性能 API P99 延迟 < 200ms APM + 指标看板 压测报告
    吞吐 峰值 QPS >= 20k 网关指标 压测与回放
    可用性 SLA >= 99.95% 可用性看板 月度复盘
    稳定性 变更失败率 < 10% 发布平台数据 迭代统计
    恢复能力 MTTR < 30min 事故系统 演练记录
    安全 高危漏洞存量 0 扫描平台 安全门禁
    成本 单请求成本 环比下降 10% 成本平台 月度审计

    落地要求：

    • 每个关键 NFR 都要有 Owner。
    • 每个 NFR 都要有告警阈值与升级策略。
    • NFR 未达标时需要给出临时策略和整改计划。

    4.6 文档版本管理要求¶

    • 文档头部：建议包含版本号、作者、评审人、最后更新时间、状态（草稿/评审中/已发布）。
    • 变更记录：每次重大变更必须记录变更摘要（What/Why/Impact）。
    • 版本映射：与代码版本、发布版本建立映射关系，便于问题追溯。

    5. 持续改进与迭代机制¶

    架构不是一次性产物，而是持续演进过程。

    5.1 迭代节奏¶

    • 里程碑评审：立项、上线前、重大变更前必须评审。
    • 周期复盘：建议双周/每月复盘架构健康度。
    • 季度治理：聚焦历史技术债与关键瓶颈治理。

    5.2 架构健康度指标¶

    建议建立可量化指标面板：

    • 可用性：SLA/SLO 达成率。
    • 性能：P95/P99 延迟、错误率、超时率。
    • 稳定性：故障次数、MTTR、变更失败率。
    • 工程效率：构建时长、发布频率、回滚次数。
    • 成本：单位请求成本、资源利用率。

    5.3 技术债管理¶

    • 台账管理：建立技术债台账，按业务影响和风险排序。
    • 容量预留：每个迭代预留治理容量（建议 10%~20%）。
    • 闭环验收：明确债务清理的验收标准与截止时间。

    5.4 变更管理与风险控制¶

    • 流程规范：重大架构变更需走 RFC/评审流程。
    • 变更前要求：影响评估、压测报告、回滚预案、灰度计划。
    • 变更后要求：指标对比、问题复盘、知识沉淀。

    5.5 复盘与知识沉淀¶

    • 事故复盘：线上事故必须形成 Postmortem，关注系统性改进。
    • 知识库：设计决策、故障案例、最佳实践沉淀到团队知识库。
    • 避坑指南：定期更新“反模式清单”，避免重复踩坑。

    5.6 触发架构迭代的信号¶

    出现以下情况时，应启动专项架构评估（可走轻量 RFC）：

    • 连续两个迭代出现同类稳定性故障。
    • 关键链路 P99 时延持续超阈值。
    • 峰值资源利用率长期 > 70% 且增长趋势明显。
    • 发布频率显著下降、回滚率上升。
    • 成本明显上升但业务收益未同步增长。
    • 新合规要求（数据跨境、审计、隐私）导致现有方案不满足。

    5.7 迭代治理闭环（建议流程）¶

    1. 发现问题：通过指标、告警、事故、反馈识别瓶颈。
    2. 根因分析：区分症状与根因，输出问题分类。
    3. 方案设计：候选方案对比、成本收益评估。
    4. 灰度验证：小流量试点，关注回归风险。
    5. 全面发布：逐步放量，执行发布门禁。
    6. 效果复盘：与改造前基线对比，沉淀经验。

    5.8 技术债治理优先级模型¶

    建议采用“影响 × 紧急度 × 复杂度”评分：

    • 影响（1~5）：对业务连续性/用户体验影响。
    • 紧急度（1~5）：问题发生频率与恶化速度。
    • 复杂度（1~5）：改造难度（复杂度高应分期）。

    优先级参考：

    • P0：影响高且紧急，立即治理。
    • P1：影响中高，纳入近期迭代。
    • P2：可规划进入季度治理。

    6. 评审与落地清单¶

    上线前建议逐项检查：

    • [ ] 是否有明确业务目标与非目标范围
    • [ ] 是否完成候选方案对比与选型依据
    • [ ] 模块边界、接口契约、数据模型是否完整
    • [ ] 性能容量目标是否量化并完成压测
    • [ ] 高可用、容灾、降级、回滚方案是否可执行
    • [ ] 安全、审计、合规要求是否覆盖
    • [ ] 监控、告警、Tracing 是否具备
    • [ ] 文档与图是否保持一致且可追溯
    • [ ] 迭代计划与技术债治理机制是否明确

    发布门禁建议（可自动化）：

    • [ ] 架构评审结论为通过（含条件通过项已关闭）
    • [ ] 压测、容灾演练、安全扫描报告齐全
    • [ ] 关键告警策略与值班机制已配置
    • [ ] 回滚预案已演练且负责人明确
    • [ ] ADR 与实施文档已更新并归档

    提示：建议将该清单纳入 PR 审查或发布门禁，确保架构方案真正落地。
