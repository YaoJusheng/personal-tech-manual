    一、简介¶

    JWT （JSON Web Token） 是目前最流行的跨域认证解决方案，是一种基于 Token 的认证授权机制。 从 JWT 的全称可以看出，JWT 本身也是 Token，一种规范化之后的 JSON 结构的 Token。

    JWT 自身包含了身份验证所需要的所有信息，因此，服务器不需要存储 Session 信息，大大减轻了服务端的压力。

    JWT 更符合 RESTful API 的 Stateless（无状态）原则。

    并且， 使用 JWT 认证可以有效避免 CSRF 攻击，因为 JWT 一般是存在在 localStorage 中，使用 JWT 进行身份验证的过程中是不会涉及到 Cookie 的。

    PyJWT 是一个用于在 Python 中编码和解码 JSON Web Tokens（JWT）的库。JWT 是一种开放的行业标准（RFC 7519），用于在两方之间安全地传递声明。

    二、主要组成与功能¶

    JSON Web Token 结构由三部分组成：

    • Header（头部）：包括令牌的类型及正在使用的散列算法。
    • typ（Type）： token类型, JWT
    • alg（Algorithm）：签名算法，Base64（HS256）
    • Payload（负载）：声明是关于实体（sub通常是用户）、其他数据的声明(Claims)和jti（JWT ID）。
    • Claims有三种类型：标准注册声明，公共的声明和私有的声明。
    • 用户身份信息（注意，这里因为采用base64编码，可解码，因此不要存放敏感信息）

    • 注册声明：如token的签发时间，过期时间，签发人等

    • Signature（签名）：必须采用编码标头，编码的有效负载，秘密，标头中指定的算法 (默认是 HMAC SHA256)，并对其进行签名。

    1.负载说明¶

    1. 负载－标准的声明：

    2. iss (issuer)：JWT签发者

    3. sub (subject)：JWT所面向的用户
    4. aud (audience)：接收JWT的一方
    5. exp (expiration time)：JWT的过期时间，这个过期时间必须要大于签发时间，这是一个秒数
    6. nbf (not before time)：定义在什么时间之前，该JWT都是不可用的.(生效时间)
    7. iat (issued at time)：JWT的签发时间

    8. jti (JWT ID)：JWT 唯一标识。

    9. 负载－公共的声明：可以添加任何信息，一般添加用户的相关信息或其他业务需要的必要信息，但不建议添加敏感信息，因为该部分在客户端可解密。

    10. 负载－私有声明：提供者和消费者所共同定义的声明，一般不建议存放敏感信息，因为base64是对称解密的，意味着该部分信息可以归类为明文信息

    1.JWT 内容设计¶

    • Header：
    {
  "alg": "HS256",
  "typ": "JWT"
}
    • Payload： 包含必要的用户信息和 Token 限制：
    {
  "sub": "1234567890",        // 用户 ID
  "name": "John Doe",         // 用户名
  "role": "admin",            // 用户角色
  "exp": 1712294400           // 过期时间 (Unix 时间戳)
}
    • Signature： 使用私钥签名，用于验证 Token 是否被篡改。

    2.Token 存储与传递¶

    • 推荐存储在浏览器的 HttpOnly Cookie 中，避免受到 XSS 攻击。

    • 每次请求时通过 Authorization 头携带 Token：

    Authorization: Bearer <JWT>

    3.安全性考量¶

    • 签名算法：推荐使用非对称加密算法（如 RS256）。
    • Token 过期时间：设置合理的过期时间，避免 Token 长时间暴露。
    • HTTPS：强制所有通信使用 HTTPS。

    4.JWT 签名与验证¶

    • 采用非对称加密（如 RS256），私钥由 SSO 服务管理，公钥分发给子系统用于验证。
    • 定期轮换签名密钥，提升安全性。
    • 子系统使用共享的公钥验证 JWT 签名的合法性。
    • 检查 Token 的过期时间（exp）和颁发时间（iat）。
    • 可扩展性验证，如检查用户状态是否被冻结。

    5.跨域支持¶

    • 配置 CORS，允许子系统跨域访问 SSO 服务。
    • 如果使用 Cookie，需启用跨域 Cookie 并设置 SameSite 属性。

    3.优化设计¶

    1.Token 的刷新机制¶

    • Access Token：短期有效（如 15 分钟），用于访问资源。
    • Refresh Token：长期有效（如 7 天），仅用于刷新 Access Token，不直接用于访问资源。
    • Access Token 放入 Authorization Header，Refresh Token 放入 HttpOnly Secure Cookie。
    • Token 轮换机制：
    • Access Token 快到期时，前端通过 Refresh Token 向 SSO 服务请求新的 Access Token，保证用户体验不被中断。
    • 旧的 Refresh Token 被废弃，返回新的 Refresh Token，同时更新到 Cookie。

    2.集中式权限管理¶

    • 将权限控制逻辑集中在 SSO 服务中。
    • 子系统根据 Token 中的角色或权限字段，执行业务逻辑。

    3.单点退出（SSO Logout）¶

    1. 添加 Token 黑名单：
    2. 在服务端维护一个全局黑名单，存储已被注销或失效的 Access Token。
    3. 子系统验证 Access Token 时，需检查 Token 是否在黑名单中。
    4. 单点注销：
    5. 用户在任意子系统发起注销请求。
    6. SSO 服务接收请求，删除用户的 Refresh Token 并将 Access Token 加入黑名单。
    7. SSO 通知其他子系统清理本地会话（如 WebSocket 或 HTTP 通知机制）。
    8. 黑名单同步机制：
    9. 集中存储：将黑名单存储在 Redis 中，所有子系统共享访问。
    10. 分布式事件通知：通过消息队列（如 RabbitMQ、Kafka）通知子系统更新黑名单。
    11. 前端同步退出：
    12. 前端监听用户的登录状态（如轮询、长连接），当发现 Token 已失效时，自动重定向到登录页。

    三、核心参数¶

    PyJWT 提供了创建和验证 JWT 的功能。JWT 通常由三部分组成：头部（Header）、载荷（Payload）和签名（Signature）。头部通常包含令牌的类型和所使用的签名算法；载荷包含声明（claims），即需要传递的数据；签名用于验证消息的完整性和真实性。

    1.encode/decode关键参数¶

    PyJWT 的 encode() 和 decode() 方法接受以下关键参数：

    参数名 类型 作用 示例
    key str 或 bytes 用于签名（加密）和验证（解密）的密钥 secret-key 或 RSA 私钥
    algorithm str 指定 JWT 签名算法（如 HS256, RS256） "HS256"
    headers dict 自定义 JWT 头部信息（如 kid） {"kid": "my-key-id"}
    audience (aud) str 验证 JWT 的目标接收方 "api.example.com"
    issuer (iss) str 验证 JWT 的签发者 "auth-server"
    expires_in int 设置 Token 过期时间（秒） 3600 (1小时)
    options dict 控制解码时的验证行为 {"verify_exp": False}

    2.realm 参数分析¶

    realm 并非 PyJWT 的直接参数，而是 HTTP 认证协议 中的概念，常见于 WWW-Authenticate 响应头，用于标识受保护资源的逻辑区域。例如：

    HTTP/1.1 401 Unauthorized
WWW-Authenticate: Bearer realm="api/protected", error="invalid_token"

    2.1.何时使用 realm？¶

    • 场景：当客户端请求需要认证的资源时，服务器通过 realm 提示客户端应在哪个“安全域”提供凭证。
    • 作用：帮助客户端区分不同认证区域（如不同 API 端点或服务）。

    四、签名算法¶

    常用的几种签名算法如下：

    1. HMAC (HS256): 对称加密，使用同一个密钥进行签名和验证。
    2. RSA (RS256): 非对称加密，使用私钥签名，公钥验证。
    3. ECDSA (ES256): 非对称加密，基于椭圆曲线，使用私钥签名，公钥验证。
    4. EdDSA (Ed25519): 非对称加密，基于椭圆曲线，性能好，安全性高。

    1. 算法概览表¶

    算法 类型 密钥长度 性能 安全性 适用场景
    HMAC SHA256 对称 256位 ⭐⭐⭐⭐⭐ ⭐⭐⭐⭐ 内部API、短期令牌
    RSA RS256 非对称 2048+位 ⭐⭐ ⭐⭐⭐⭐ 分布式系统、SSO
    ECDSA ES256 非对称 256位 ⭐⭐⭐ ⭐⭐⭐⭐⭐ 移动端、IoT设备
    EdDSA Ed25519 非对称 256位 ⭐⭐⭐⭐ ⭐⭐⭐⭐⭐ 现代应用、新项目
    PBKDF2 对称 可变 ⭐⭐ ⭐⭐⭐⭐ 密码衍生、密钥派生

    import jwt
import os
import hashlib
import hmac
from datetime import datetime, timedelta
from typing import Dict, Any

class HS256TokenManager:
    """HMAC HS256 对称签名"""

    def __init__(self, secret_key: str = None):
        # 建议：从环境变量或KMS获取，不要硬编码
        self.secret_key = secret_key or os.getenv("JWT_SECRET_KEY", "default-secret-change-in-prod")

        # 确保密钥足够长
        if len(self.secret_key) < 32:
            self.secret_key = self._derive_key(self.secret_key)

    def _derive_key(self, weak_key: str) -> bytes:
        """将弱密钥派生为强密钥"""
        salt = b"jwt-salt-2024"
        return hashlib.pbkdf2_hmac(
            'sha256',
            weak_key.encode(),
            salt,
            100000,  # 迭代次数
            dklen=32  # 输出长度
        )

    def create_token(self, payload: Dict[str, Any], expires_minutes: int = 15) -> str:
        """创建JWT令牌"""
        payload.update({
            'exp': datetime.utcnow() + timedelta(minutes=expires_minutes),
            'iat': datetime.utcnow(),
            'iss': 'my-app',
            'alg': 'HS256'
        })

        return jwt.encode(
            payload,
            self.secret_key,
            algorithm='HS256'
        )

    def verify_token(self, token: str) -> Dict[str, Any]:
        """验证JWT令牌"""
        try:
            return jwt.decode(
                token,
                self.secret_key,
                algorithms=['HS256'],
                options={
                    'verify_exp': True,
                    'verify_iss': True,
                    'require': ['exp', 'iat', 'iss']
                }
            )
        except jwt.ExpiredSignatureError:
            raise ValueError("Token已过期")
        except jwt.InvalidTokenError as e:
            raise ValueError(f"无效Token: {e}")

    def manual_hmac_verification(self, token: str):
        """手动HMAC验证（了解原理）"""
        # JWT格式: header.payload.signature
        parts = token.split('.')
        if len(parts) != 3:
            raise ValueError("无效Token格式")

        header_b64, payload_b64, signature_b64 = parts

        # 重新计算签名
        message = f"{header_b64}.{payload_b64}".encode()
        expected_signature = hmac.new(
            self.secret_key.encode() if isinstance(self.secret_key, str) else self.secret_key,
            message,
            hashlib.sha256
        ).digest()

        # Base64URL解码签名
        import base64
        def base64url_decode(input_str):
            padding = 4 - (len(input_str) % 4)
            input_str += '=' * padding
            return base64.urlsafe_b64decode(input_str)

        actual_signature = base64url_decode(signature_b64)

        # 使用恒定时间比较防止时序攻击
        if not hmac.compare_digest(expected_signature, actual_signature):
            raise ValueError("签名验证失败")

    import jwt
from cryptography.hazmat.primitives import serialization
from cryptography.hazmat.primitives.asymmetric import rsa
from cryptography.hazmat.primitives import hashes
from cryptography.hazmat.primitives.asymmetric import padding
import base64
import json
from datetime import datetime, timedelta

class RS256TokenManager:
    """RSA RS256 非对称签名"""

    def __init__(self, private_key_path: str = None, public_key_path: str = None):
        # 生成或加载RSA密钥对
        if private_key_path and public_key_path:
            self.private_key = self._load_private_key(private_key_path)
            self.public_key = self._load_public_key(public_key_path)
        else:
            self.private_key, self.public_key = self._generate_keypair()

    def _generate_keypair(self, key_size: int = 2048):
        """生成RSA密钥对"""
        private_key = rsa.generate_private_key(
            public_exponent=65537,
            key_size=key_size
        )
        public_key = private_key.public_key()
        return private_key, public_key

    def _load_private_key(self, path: str):
        """加载私钥"""
        with open(path, 'rb') as f:
            return serialization.load_pem_private_key(
                f.read(),
                password=None
            )

    def _load_public_key(self, path: str):
        """加载公钥"""
        with open(path, 'rb') as f:
            return serialization.load_pem_public_key(f.read())

    def export_keys(self, private_path: str, public_path: str):
        """导出密钥对到文件"""
        # 导出私钥
        private_pem = self.private_key.private_bytes(
            encoding=serialization.Encoding.PEM,
            format=serialization.PrivateFormat.PKCS8,
            encryption_algorithm=serialization.NoEncryption()
        )
        with open(private_path, 'wb') as f:
            f.write(private_pem)

        # 导出公钥
        public_pem = self.public_key.public_bytes(
            encoding=serialization.Encoding.PEM,
            format=serialization.PublicFormat.SubjectPublicKeyInfo
        )
        with open(public_path, 'wb') as f:
            f.write(public_pem)

    def create_token(self, payload: dict, expires_minutes: int = 60) -> str:
        """使用私钥签名创建Token"""
        payload.update({
            'exp': datetime.utcnow() + timedelta(minutes=expires_minutes),
            'iat': datetime.utcnow(),
            'iss': 'my-app-rsa',
            'alg': 'RS256'
        })

        return jwt.encode(
            payload,
            self.private_key,
            algorithm='RS256'
        )

    def verify_token(self, token: str) -> dict:
        """使用公钥验证Token"""
        try:
            return jwt.decode(
                token,
                self.public_key,
                algorithms=['RS256'],
                options={'verify_exp': True}
            )
        except jwt.ExpiredSignatureError:
            raise ValueError("Token已过期")
        except jwt.InvalidTokenError as e:
            raise ValueError(f"无效Token: {e}")

    def manual_rsa_signature(self, data: bytes):
        """手动RSA签名（了解原理）"""
        # RSA签名流程
        signature = self.private_key.sign(
            data,
            padding.PSS(
                mgf=padding.MGF1(hashes.SHA256()),
                salt_length=padding.PSS.MAX_LENGTH
            ),
            hashes.SHA256()
        )
        return signature

    def manual_rsa_verification(self, data: bytes, signature: bytes):
        """手动RSA验证"""
        try:
            self.public_key.verify(
                signature,
                data,
                padding.PSS(
                    mgf=padding.MGF1(hashes.SHA256()),
                    salt_length=padding.PSS.MAX_LENGTH
                ),
                hashes.SHA256()
            )
            return True
        except Exception:
            return False

    import jwt
from cryptography.hazmat.primitives import serialization
from cryptography.hazmat.primitives.asymmetric import ec
from cryptography.hazmat.primitives import hashes
import base64

class ES256TokenManager:
    """ECDSA ES256 椭圆曲线签名"""

    def __init__(self):
        # 使用P-256曲线（prime256v1）
        self.private_key = ec.generate_private_key(ec.SECP256R1())
        self.public_key = self.private_key.public_key()

    def get_jwk(self):
        """生成JWK（JSON Web Key）格式公钥"""
        # 获取公钥坐标
        public_numbers = self.public_key.public_numbers()

        jwk = {
            "kty": "EC",
            "crv": "P-256",
            "x": base64.urlsafe_b64encode(
                public_numbers.x.to_bytes(32, 'big')
            ).decode().rstrip("="),
            "y": base64.urlsafe_b64encode(
                public_numbers.y.to_bytes(32, 'big')
            ).decode().rstrip("="),
            "use": "sig",
            "alg": "ES256"
        }
        return jwk

    def create_token(self, payload: dict, expires_hours: int = 24) -> str:
        """创建ES256签名Token"""
        payload.update({
            'exp': datetime.utcnow() + timedelta(hours=expires_hours),
            'iat': datetime.utcnow(),
            'alg': 'ES256'
        })

        return jwt.encode(
            payload,
            self.private_key,
            algorithm='ES256'
        )

    def verify_with_jwk(self, token: str, jwk: dict):
        """使用JWK验证Token（适用于分布式验证）"""
        # 从JWK重建公钥
        from cryptography.hazmat.primitives.asymmetric import ec

        x = int.from_bytes(
            base64.urlsafe_b64decode(jwk['x'] + '=='), 'big'
        )
        y = int.from_bytes(
            base64.urlsafe_b64decode(jwk['y'] + '=='), 'big'
        )

        public_key = ec.EllipticCurvePublicNumbers(
            x, y, ec.SECP256R1()
        ).public_key()

        return jwt.decode(token, public_key, algorithms=['ES256'])

        self.public_key = self.private_key.public_key()

        payload.update({
            'exp': datetime.utcnow() + timedelta(days=expires_days),
            'iat': datetime.utcnow(),
            'alg': 'EdDSA'
        })

        # 注意：PyJWT可能需要额外处理EdDSA

            return jwt.encode(
                payload,
                self.private_key,
                algorithm='EdDSA'
            )
        except jwt.exceptions.InvalidAlgorithmError:
            # 如果PyJWT不支持，手动实现
            return self._manual_ed25519_token(payload)

    def _manual_ed25519_token(self, payload: dict) -> str:
        """手动实现Ed25519 Token（当库不支持时）"""
        import hashlib

        # Header
        header = {
            "alg": "EdDSA",
            "typ": "JWT",
            "crv": "Ed25519"
        }

        # Base64URL编码
        def base64url_encode(data):
            if isinstance(data, dict):
                data = json.dumps(data).encode()
            return base64.urlsafe_b64encode(data).decode().rstrip("=")

        header_b64 = base64url_encode(header)
        payload_b64 = base64url_encode(payload)

        # 签名
        message = f"{header_b64}.{payload_b64}".encode()
        signature = self.private_key.sign(message)
        signature_b64 = base64url_encode(signature)

        return f"{header_b64}.{payload_b64}.{signature_b64}"

    def verify_token(self, token: str) -> dict:
        """验证Ed25519 Token"""
        try:
            return jwt.decode(
                token,
                self.public_key,
                algorithms=['EdDSA']
            )
        except:
            # 手动验证
            return self._manual_ed25519_verification(token)

    def _manual_ed25519_verification(self, token: str) -> dict:
        """手动Ed25519验证"""
        parts = token.split('.')
        if len(parts) != 3:
            raise ValueError("无效Token格式")

        header_b64, payload_b64, signature_b64 = parts

        # 验证签名
        message = f"{header_b64}.{payload_b64}".encode()
        signature = base64.urlsafe_b64decode(signature_b64 + '==')

        try:
            self.public_key.verify(signature, message)
        except Exception:
            raise ValueError("签名验证失败")

        # 解码payload
        payload_json = base64.urlsafe_b64decode(payload_b64 + '==')
        payload = json.loads(payload_json)

        # 检查过期时间
        if 'exp' in payload:
            exp_timestamp = payload['exp']
            if datetime.utcnow() > datetime.fromtimestamp(exp_timestamp):
                raise ValueError("Token已过期")

        return payload

    def create_token(self, payload: dict, **kwargs):
        """创建Token"""
        return self.manager.create_token(payload, **kwargs)

    def verify_token(self, token: str):
        """验证Token"""
        return self.manager.verify_token(token)

    def get_public_key_info(self):
        """获取公钥信息（适用于非对称算法）"""
        if hasattr(self.manager, 'get_jwk'):
            return self.manager.get_jwk()
        elif hasattr(self.manager, 'public_key'):
            return {
                'algorithm': self.algorithm,
                'key_type': 'asymmetric'
            }
        else:
            return {
                'algorithm': self.algorithm,
                'key_type': 'symmetric'
            }

    🔐 HMAC HS256 实现示例¶

    HMAC（基于哈希的消息认证码）是对称算法，使用同一个密钥进行签名和验证-1。它的核心在于使用一个密钥与消息结合，通过哈希算法（如SHA-256）生成一个“指纹”来确保消息的完整性和发送方身份-1。

    package main

import (
    "crypto/hmac"
    "crypto/sha256"
    "encoding/base64"
    "fmt"
    "log"
)

// 生成 HMAC 签名 (Base64输出)
func GenerateHMACSignature(message, secret string) (string, error) {
    mac := hmac.New(sha256.New, []byte(secret)) // 创建HMAC对象[citation:1]
    mac.Write([]byte(message)) // 写入消息[citation:1]
    // 计算哈希并编码[citation:1]
    signature := base64.StdEncoding.EncodeToString(mac.Sum(nil))
    return signature, nil
}

// 验证 HMAC 签名
func VerifyHMACSignature(message, signature, secret string) bool {
    expectedSig, err := GenerateHMACSignature(message, secret)
    if err != nil {
        log.Println("生成签名错误:", err)
        return false
    }
    // 使用 hmac.Equal 安全比较，防止时序攻击[citation:1]
    return hmac.Equal([]byte(expectedSig), []byte(signature))
}

func main() {
    secret := "your-secret-key" // 确保密钥足够强且安全
    message := `{"user_id":123,"exp":1733558400}`

    sig, _ := GenerateHMACSignature(message, secret)
    fmt.Printf("签名: %s\n", sig)

    valid := VerifyHMACSignature(message, sig, secret)
    fmt.Printf("验证结果: %v\n", valid)
}

    import "github.com/golang-jwt/jwt/v5"

type MyClaims struct {
    UserID int `json:"user_id"`
    jwt.RegisteredClaims // 嵌入标准声明（exp, iat等）
}

// 2. 生成Token (以RS256为例)
func GenerateRSAToken(privateKey *rsa.PrivateKey, userID int) (string, error) {
    claims := MyClaims{
        UserID: userID,
        RegisteredClaims: jwt.RegisteredClaims{
            ExpiresAt: jwt.NewNumericDate(time.Now().Add(24 * time.Hour)),
            Issuer:    "my-app",
        },
    }
    // 创建并签名Token
    token := jwt.NewWithClaims(jwt.SigningMethodRS256, claims)
    return token.SignedString(privateKey)
}

// 3. 解析并验证Token
func ParseAndVerifyRSAToken(tokenString string, publicKey *rsa.PublicKey) (*MyClaims, error) {
    token, err := jwt.ParseWithClaims(tokenString, &MyClaims{}, func(token *jwt.Token) (interface{}, error) {
        // 验证签名算法
        if _, ok := token.Method.(*jwt.SigningMethodRS256); !ok {
            return nil, fmt.Errorf("非预期的签名算法: %v", token.Header["alg"])
        }
        // 返回用于验证的公钥
        return publicKey, nil // 提供公钥用于验证
    })

    if claims, ok := token.Claims.(*MyClaims); ok && token.Valid {
        return claims, nil
    }
    return nil, err
}

    🔐 为什么ECDSA非常适合这些场景？¶

    相对于RSA，ECDSA有以下优势，非常适合分布式架构：

    • 性能更优：尤其在验证签名的速度上比RSA快很多，这对于需要频繁验证Token的API网关和资源服务器来说至关重要。
    • 密钥更短：一个256位的ECDSA密钥，其安全强度相当于RSA 3072位。这意味着更小的内存占用和网络传输开销。
    • 标准化与支持：ES256（ECDSA using P-256 and SHA-256）是JWT标准算法，所有主流语言库都提供完善支持。
    🔑 公钥管理的正确方式：无需前端参与¶

    公钥（Public Key）用于验证签名，其核心原则是：可以被任何人获取，但绝不用于签名。因此，前端完全不应接触它。

    公钥的获取与分发，在架构上通常通过以下两种安全、标准的方式实现：

    方式一：预置或配置公钥¶

    在微服务启动时，从安全的配置中心、环境变量或文件中加载公钥。适用于服务数量可控、公钥不频繁变更的环境。

    // 资源服务器代码示例：从环境变量加载公钥
publicKeyPEM := os.Getenv("JWT_PUBLIC_KEY")
block, _ := pem.Decode([]byte(publicKeyPEM))
publicKey, _ := x509.ParsePKIXPublicKey(block.Bytes)
// 之后用此 publicKey 验证JWT
    方式二：通过JWKS端点动态发现（最推荐用于SSO/分布式系统）¶

    这是现代SSO（如OAuth 2.0、OpenID Connect）的标准做法。认证服务器提供一个公开的HTTP端点（如 https://auth.example.com/.well-known/jwks.json），返回一个JSON Web密钥集（JWKS），其中包含了用于验证Token的公钥信息。

    工作流程如下：

    1. 前端用户登录，从认证服务器获得一个ECDSA签名的JWT。
    2. 前端访问API时，在Authorization请求头中携带此JWT（仅此而已）。
    3. API网关或资源服务器收到请求： a. 从Token头中提取kid（密钥ID）声明。 b. 向认证服务器的JWKS端点发起请求（通常会有缓存），查找匹配该kid的公钥。 c. 使用获取到的公钥验证JWT签名。
    4. 验证通过后，处理业务请求。

    2. 资源服务器：通过JWKS验证Token:

    import (
    "github.com/golang-jwt/jwt/v5"
    "github.com/MicahParks/keyfunc" // 一个方便的JWKS处理库
)

func verifyTokenWithJWKS(rawToken string, jwksURL string) (*jwt.Token, error) {
    // 1. 从远程JWKS端点创建密钥函数
    options := keyfunc.Options{
        Ctx: ctx,
        RefreshErrorHandler: func(err error) {
            log.Printf("刷新JWKS失败: %v", err)
        },
        RefreshInterval:   time.Hour,
        RefreshRateLimit:  time.Minute * 5,
        RefreshTimeout:    time.Second * 10,
    }
    jwks, err := keyfunc.Get(jwksURL, options)
    if err != nil {
        return nil, err
    }
    defer jwks.EndBackground()

    // 2. 解析并验证Token，库会自动根据`kid`查找正确的公钥
    parsedToken, err := jwt.Parse(rawToken, jwks.Keyfunc)
    if err != nil {
        return nil, err
    }
    return parsedToken, nil
}

    🔧 路径一：坚持HS256，构建密钥分发机制¶

    如果您选择坚持HS256以获得极致性能，必须建立一个中心化且安全的密钥分发系统。以下是几种常见的实施方案：

    方案 工作原理 优点 缺点 适用场景
    专用密钥分发API 由认证服务暴露内部API，其他服务定期调用获取最新密钥。 控制力强，可集成审计、权限。 需实现客户端缓存、失败重试；增加一个需保护的服务。 中大型集群，有专门平台团队。
    配置中心推送 将密钥作为配置项，通过Consul、Etcd、Apollo等推送至各服务。 利用现有设施，服务发现与配置管理一体。 配置中心本身需高可用；需确保配置通道安全。 已成熟使用配置中心的团队。
    服务网格边车 通过Envoy等边车代理下发密钥至本地，业务服务从本地读取。 对业务代码零侵入，语言无关。 引入服务网格复杂度。 已部署或计划部署服务网格的云原生架构。

    关键实施要点（无论采用哪种方案）：

    1. 密钥必须轮换：短期Token（如5分钟）对应的密钥应每小时或每天轮换，长期Token密钥轮换周期更短。
    2. 密钥必须带版本（kid）：每个密钥有唯一ID，Token头中声明kid，验证方据此选择正确密钥。
    3. 支持多版本共存：轮换时，新旧密钥应短暂同时有效，以确保正在传输的Token不会被批量拒绝。
    4. 访问必须认证：获取密钥的请求本身须被认证（例如使用服务间mTLS或ES256签名的服务令牌）。
    // 简化示例：服务从内部API获取并缓存HS256密钥
var currentKey []byte
var keyExpiry time.Time

func getSigningKey(kid string) (interface{}, error) {
    // 如果缓存过期或没有所需kid的密钥，则从中心获取
    if time.Now().After(keyExpiry) || !keyCache.Has(kid) {
        fetchKeyFromCentralAPI()
    }
    return keyCache.Get(kid), nil
}

func verifyToken(tokenString string) {
    token, _ := jwt.Parse(tokenString, func(t *jwt.Token) (interface{}, error) {
        kid := t.Header["kid"].(string)
        return getSigningKey(kid)
    })
    // ... 验证其他声明
}
    🔄 路径二：更换算法，统一使用ES256¶

    如果构建上述分发机制对您的团队来说过于复杂或风险较高，那么更换算法是一个完全合理且稳健的选择。

    直接统一使用ES256（或EdDSA）的好处：

    • 彻底消除密钥分发问题：公钥可以硬编码在配置中、放在环境变量里，或者通过一个无需认证的公开JWKS端点获取。
    • 架构简化：每个服务可以独立验证Token，无需依赖另一个中心化服务，系统容错性更强。
    • 安全性模型清晰：私钥高度集中于认证服务器，与业务逻辑完全分离。

    需要接受的代价：

    • 验证性能下降：ES256的验证速度大约是HS256的 1/10。您需要评估您的网关和资源服务器能否承受此性能开销。
    • Token略长：ES256签名比HS256略长，但通常影响甚微。
    🎯 决策建议：如何选择？¶

    您可以根据以下情况判断：

    坚持HS256，构建分发机制，如果：

    • 您的系统吞吐量极高（如每秒数十万次验证），性能边际至关重要。
    • 您有专门的平台或基础设施团队，能够设计并维护一个高可用、安全的密钥分发服务。
    • 您的微服务架构相对统一且可控（例如，所有服务都用Go/Java编写，易于集成统一的SDK）。

    更换为ES256，如果：

    • 您的团队规模较小，或项目处于早期，希望最大限度地减少运维复杂度。
    • 您的服务是多语言混合架构（Go, Node.js, Python等），难以统一实现复杂的密钥获取逻辑。
    • 您的性能要求尚未触及瓶颈，ES256的验证速度已完全足够。
    • 您需要快速落地，并且希望采用更“标准”和“免维护”的分布式验证方案。

    最终结论： 对于绝大多数微服务场景，统一使用ES256是更简单、更安全、更易维护的选择。只有在性能被明确证明是核心瓶颈，且团队具备相应工程能力时，才值得去构建和维护一套HS256的密钥分发基础设施。这是一个典型的“用工程复杂性换取性能”的权衡。

    五、密钥管理服务¶

    为了构建安全、合规且高可用的密钥管理服务（KMS），需围绕密钥全生命周期搭建技术架构，并结合行业标准与业务场景实现精细化管控。

    密钥管理的安全基石是硬件级隔离与软件级管控的结合，推荐采用 “根密钥 - HSM 托管 + 业务密钥 - KMS 调度” 的分层架构.

    最佳方案遵循 “分层防御、最小权限、密钥与数据分离” 原则。下图展示了一个典型的企业级KMS分层架构：

    🏗️ 分层架构与核心组件¶

    1. 硬件安全层（信任根）¶

    • 核心：使用硬件安全模块（HSM） 或云HSM服务（如AWS CloudHSM、Google Cloud HSM）生成并保护最高层级的根密钥。
    • 职责：根密钥绝不离开HSM，仅用于加密下一层的“密钥加密密钥”。

    2. 核心服务层（密钥全生命周期管理）¶

    • 密钥管理服务：中央服务，管理密钥元数据（ID、策略、状态），处理密钥生成、启用、禁用、轮换、销毁的逻辑。它不直接“看到”明文密钥。
    • 密钥操作代理：唯一能与HSM交互的组件。接收来自KMS的指令，在HSM内部完成实际的加解密、签名操作，将结果返回。可部署在机密计算环境（如Intel SGX Enclave）中。

    3. 应用接入层（安全便捷使用）¶

    • 客户端SDK：为业务服务提供简单API（如 encrypt(dataKeyId, plaintext)），处理与KMS的认证、通信、重试。
    • 职责：确保应用代码永不直接处理明文密钥，只持有密钥ID。

    🔐 密钥生命周期管理实践¶

    1. 生成与存储¶

    • 分层密钥结构：
    • 根密钥：静态，位于HSM，用于加密KEK。
    • 密钥加密密钥：由根密钥加密后存储，用于加密DEK。
    • 数据加密密钥：由KEK加密后存入数据库，用于加密业务数据。每个服务、每个用途甚至每个数据记录都应使用不同的DEK。
    • 存储：所有密钥（除根密钥）均以密文形式存储于常规数据库，但KEK和DEK的密文分开存储。

    2. 分发与使用¶

    • 原则：密钥不离域，密文下行。业务服务只能拿到keyId和密文数据，如需解密或签名，必须通过API调用KMS。

    • 信封加密：大规模数据加密的标准模式。

    // 应用端伪代码：使用信封加密
func encryptData(data []byte, keyId string) (EnvelopedData, error) {
    // 1. 向KMS请求，生成一个临时的数据密钥
    plaintextDEK := kmsClient.GenerateDataKey(keyId)
    // 2. 本地使用这个明文DEK加密业务数据
    ciphertext := aesGCMEncrypt(plaintextDEK, data)
    // 3. 立即在内存中销毁明文DEK，只保存其密文版本
    return EnvelopedData{
        Ciphertext: ciphertext,
        EncryptedDEK: plaintextDEK.encryptedVersion // 由KMS返回
    }, nil
}

    3. 轮换与撤销¶

    • 自动轮换：为每个密钥设置轮换策略（如每年一次）。轮换后，旧密钥仍可用于解密旧数据，新数据一律使用新密钥加密。
    • 密钥版本化：每个轮换周期产生新版本，所有密钥ID包含版本号。
    • 即时撤销：通过更新密钥状态（如设置为Disabled或ScheduledForDeletion）实现。KMS拒绝使用已撤销密钥的请求。

    ⚙️ 关键实现细节¶

    1. 访问控制与审计¶

    • 身份认证：所有请求必须使用强身份（如服务mTLS证书、JWT）。
    • 精细授权：基于属性（ABAC）或角色（RBAC）定义策略，如“服务A只能使用KeyID为app-a-enc-key的密钥进行加密操作”。
    • 不可篡改审计：记录所有密钥操作（谁、何时、对哪个密钥、进行了什么操作、结果）至专用日志，并防止篡改。

    2. 高可用与容灾¶

    • 多区域部署：在多个地理区域部署KMS实例，密钥材料通过HSM的安全同步机制复制。
    • 优雅降级：在设计上，即使KMS短暂不可用，也不应影响现有加密数据的解密（因为DEK已缓存或可离线解密），但新的加密操作会失败。

    ✅ 核心检查清单¶

    一个健壮的KMS应满足：

    • 密钥与数据分离：业务系统无明文密钥。
    • 分层密钥结构：根密钥受HSM保护。
    • 完整的生命周期管理：包括生成、轮换、撤销、销毁。
    • 最小权限访问控制：基于身份的精细授权。
    • 不可否认的审计追踪：所有操作有据可查。
    • 高可用与容灾设计：避免单点故障。
